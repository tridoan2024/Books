Chapter 5: Vector Database Security — The Persistence Layer ⏱ ~25 min read¶

"Your vectors remember everything. The question is who else can access those memories."

Vector databases have become the backbone of modern AI systems. Every retrieval-augmented generation (RAG) pipeline, every semantic search engine, and every recommendation system depends on a persistence layer that stores high-dimensional embeddings alongside their associated metadata. Yet the security posture of these systems remains dangerously immature. Unlike relational databases, which have benefited from decades of hardening, vector databases are often deployed with default configurations, minimal access controls, and no encryption—exposed surfaces waiting to be exploited.

This chapter dissects the security landscape of vector databases. We begin with a comprehensive threat model, move into the emerging science of embedding inversion attacks, cover operational hardening across the major platforms, and conclude with a hands-on penetration testing lab.

5.1 Vector Database Threat Model¶

The first step in securing any system is understanding what can go wrong. Vector databases introduce a unique combination of traditional database risks and novel AI-specific attack surfaces. A proper threat model must address authentication and authorization, multi-tenancy isolation, backup and recovery integrity, and encryption across all states of data.

5.1.1 Authentication and Authorization Failures¶

Most vector databases were designed for developer convenience, not enterprise security. Early versions of Chroma shipped with no authentication at all. Weaviate's default configuration exposes an unauthenticated HTTP endpoint. Pinecone relies on API keys that are frequently hardcoded into application source code.

The authentication threat surface includes:

  • Missing authentication: Services exposed on internal networks without any credential requirement
  • Weak API key management: Static keys shared across teams, committed to repositories, or embedded in client-side code
  • No key rotation: API keys that remain valid indefinitely with no rotation policy
  • Overly permissive service accounts: A single key grants full read/write/admin access
python
# INSECURE: API key hardcoded in application code
import pinecone

pinecone.init(
    api_key="sk-proj-abc123def456",  # Hardcoded secret
    environment="us-west1-gcp"
)

# SECURE: Key loaded from secrets manager with scoped permissions
import os
from azure.keyvault.secrets import SecretClient

vault_client = SecretClient(
    vault_url=os.environ["AZURE_VAULT_URL"],
    credential=DefaultAzureCredential()
)
api_key = vault_client.get_secret("pinecone-read-only-key").value

pinecone.init(api_key=api_key, environment="us-west1-gcp")

Authorization failures are equally prevalent. Most vector database platforms implement coarse-grained access controls—if they implement any at all. The typical failure pattern involves a single API key that grants both read and write access to every namespace in an index. There is no concept of row-level security, no attribute-based access control, and no audit trail of who queried what.

5.1.2 Multi-Tenancy Risks¶

Multi-tenancy in vector databases is often implemented through namespaces, collections, or metadata filtering—none of which were designed as security boundaries.

Consider a SaaS platform that stores customer embeddings in a shared Pinecone index using namespaces:

python
# Tenant A's data
index.upsert(vectors=tenant_a_vectors, namespace="tenant-a")

# Tenant B's data
index.upsert(vectors=tenant_b_vectors, namespace="tenant-b")

# VULNERABILITY: Nothing prevents Tenant A's compromised key
# from querying Tenant B's namespace
results = index.query(
    vector=attack_vector,
    namespace="tenant-b",  # Cross-tenant access
    top_k=100
)

The risks in multi-tenant vector database deployments include:

Risk CategoryDescriptionImpact
Namespace traversalNamespaces are logical, not security boundariesFull cross-tenant data access
Metadata leakageFilter expressions can enumerate tenant metadataCustomer data enumeration
Embedding proximitySimilar tenants share embedding neighborhoodsIndirect information leakage
Resource exhaustionOne tenant's queries degrade others' performanceDenial of service
Index poisoningMalicious vectors injected into shared indexResult manipulation across tenants

A secure multi-tenancy architecture requires dedicated indices per tenant, separate API keys with namespace-level restrictions, and network-level isolation where possible.

5.1.3 Backup, Recovery, and Data Lifecycle¶

Vector databases rarely receive the same backup and recovery discipline applied to transactional databases. This creates several risks:

  • Unencrypted backups: Snapshots stored in object storage without encryption, exposing raw embeddings and metadata
  • No point-in-time recovery: Inability to roll back after an index poisoning attack
  • Orphaned data: Deleted records that persist in backups indefinitely, violating data retention policies
  • Backup credential exposure: Backup processes using long-lived credentials with excessive permissions
yaml
# Example: Weaviate backup configuration with security controls
backup:
  provider: s3
  s3:
    bucket: "weaviate-backups-prod"
    region: "us-east-1"
    sse: "aws:kms"              # Server-side encryption with KMS
    kms_key_id: "arn:aws:kms:us-east-1:123456:key/backup-key"
  schedule: "0 2 * * *"         # Daily at 2 AM
  retention:
    days: 30                    # Automatic deletion after 30 days
    compliance_hold: false
  verification:
    enabled: true               # Verify backup integrity
    sample_queries: 10          # Run 10 sample queries against restored backup

5.1.4 Encryption at Rest and in Transit¶

Encryption for vector databases must cover three states:

  1. In transit: TLS 1.3 for all client-server and inter-node communication
  2. At rest: AES-256 encryption of index files, metadata stores, and write-ahead logs
  3. In use: Protection of embeddings during query processing (emerging area)

Many deployments fail at the basics. Self-hosted Chroma instances frequently run over unencrypted HTTP. Weaviate clusters communicate between nodes without mutual TLS. Pgvector inherits PostgreSQL's encryption capabilities but requires explicit configuration that is often skipped.

bash
# Verify TLS configuration for a Weaviate instance
openssl s_client -connect weaviate.internal:8080 -tls1_3 2>/dev/null | \
  openssl x509 -noout -subject -dates -issuer

# Check for unencrypted Chroma endpoints
nmap -sV -p 8000 chroma.internal --script ssl-enum-ciphers
# If no TLS detected, the endpoint is transmitting embeddings in cleartext

Key Takeaway — Vector Database Threat Model

Vector databases combine traditional database vulnerabilities (weak auth, missing encryption, poor backup hygiene) with AI-specific risks (cross-tenant embedding leakage, namespace traversal, index poisoning). Your threat model must address both categories. Treat namespaces as organizational tools, never as security boundaries.

5.2 Embedding Inversion and Data Extraction¶

Embeddings were once considered safe because they are "just numbers"—high-dimensional vectors with no obvious human-readable content. This assumption is dangerously wrong. A growing body of research demonstrates that embeddings can be inverted to reconstruct the original text, reveal membership in training datasets, and expose personally identifiable information encoded in the vector space.

5.2.1 Text Reconstruction from Embeddings¶

Embedding inversion attacks train a decoder model to map vectors back to their source text. The attacker needs access to the embedding model (or a close approximation) and a corpus of text-embedding pairs to train the inverter.

The seminal work by Morris et al. (2023) demonstrated that sentence embeddings from popular models could be inverted with surprising fidelity. Their Vec2Text approach achieved BLEU scores exceeding 90% on many benchmark datasets—meaning the reconstructed text was nearly identical to the original.

python
import torch
import torch.nn as nn
from transformers import AutoTokenizer, AutoModel

class EmbeddingInverter(nn.Module):
    """
    Demonstration: Train a decoder to reconstruct text from embeddings.
    This illustrates the attack vector — NOT a tool for malicious use.
    """
    def __init__(self, embedding_dim=768, vocab_size=30522, max_seq_len=128):
        super().__init__()
        self.projection = nn.Linear(embedding_dim, 512)
        self.decoder = nn.TransformerDecoder(
            nn.TransformerDecoderLayer(d_model=512, nhead=8, batch_first=True),
            num_layers=6
        )
        self.output_head = nn.Linear(512, vocab_size)
        self.position_encoding = nn.Embedding(max_seq_len, 512)

    def forward(self, embedding, target_ids=None):
        memory = self.projection(embedding).unsqueeze(1)
        positions = torch.arange(
            target_ids.size(1), device=embedding.device
        )
        tgt = self.position_encoding(positions).unsqueeze(0).expand(
            embedding.size(0), -1, -1
        )
        decoded = self.decoder(tgt, memory.expand(-1, tgt.size(1), -1))
        return self.output_head(decoded)


def evaluate_inversion_risk(model_name: str, sample_texts: list[str]) -> dict:
    """Assess how vulnerable an embedding model is to inversion."""
    tokenizer = AutoTokenizer.from_pretrained(model_name)
    model = AutoModel.from_pretrained(model_name)

    embeddings = []
    for text in sample_texts:
        inputs = tokenizer(text, return_tensors="pt", truncation=True)
        with torch.no_grad():
            output = model(**inputs)
            embedding = output.last_hidden_state[:, 0, :]
            embeddings.append(embedding)

    # Measure embedding information density
    stacked = torch.cat(embeddings, dim=0)
    singular_values = torch.linalg.svdvals(stacked)
    effective_rank = (
        singular_values.sum() ** 2 / (singular_values ** 2).sum()
    ).item()

    return {
        "model": model_name,
        "embedding_dim": stacked.shape[1],
        "effective_rank": round(effective_rank, 2),
        "inversion_risk": "HIGH" if effective_rank > 100 else "MEDIUM",
        "recommendation": (
            "Apply dimensionality reduction or noise injection"
            if effective_rank > 100
            else "Monitor for inversion attempts"
        ),
    }

The effectiveness of inversion attacks depends on several factors:

  • Embedding dimensionality: Higher dimensions preserve more information and are easier to invert
  • Model architecture: Contrastive models (e.g., text-embedding-ada-002) retain more invertible information than classification-tuned models
  • Text length: Short texts (under 50 tokens) are significantly easier to reconstruct
  • Vocabulary domain: Specialized domains with limited vocabulary are more vulnerable

5.2.2 Membership Inference Attacks¶

Membership inference attacks determine whether a specific piece of text was used to generate embeddings stored in the database. This is a privacy violation even without reconstructing the text itself—knowing that a medical record or legal document exists in a system reveals sensitive information.

python
import numpy as np
from scipy.spatial.distance import cosine

def membership_inference_attack(
    target_text: str,
    embedding_model,
    vector_db_client,
    collection_name: str,
    threshold: float = 0.05
) -> dict:
    """
    Determine if a specific text has been embedded and stored.
    Uses distance-based analysis to infer membership.
    """
    # Generate embedding for the target text
    target_embedding = embedding_model.encode(target_text)

    # Query the vector database for nearest neighbors
    results = vector_db_client.query(
        collection_name=collection_name,
        query_vector=target_embedding.tolist(),
        top_k=5,
        include_distances=True
    )

    if not results or not results["distances"]:
        return {"is_member": False, "confidence": 0.0}

    min_distance = min(results["distances"][0])

    # A near-zero distance strongly indicates membership
    is_member = min_distance < threshold

    # Generate confidence score based on distance distribution
    distances = np.array(results["distances"][0])
    gap_ratio = distances[1] / max(distances[0], 1e-10) if len(distances) > 1 else float("inf")

    return {
        "is_member": is_member,
        "min_distance": round(float(min_distance), 6),
        "confidence": round(min(gap_ratio / 10.0, 1.0), 3),
        "distance_distribution": {
            "mean": round(float(distances.mean()), 4),
            "std": round(float(distances.std()), 4),
        },
    }

5.2.3 Model Extraction via Query Access¶

An attacker with query access to a vector database can extract a functional copy of the embedding model by submitting crafted inputs and observing the returned vectors. This is known as a model stealing or model extraction attack.

The attack proceeds in stages:

  1. Probing: Submit diverse text inputs and collect the returned embeddings
  2. Training: Use the input-output pairs to train a surrogate model
  3. Refinement: Use active learning to identify regions of the embedding space where the surrogate diverges from the target
python
class ModelExtractionDetector:
    """Detect potential model extraction attempts via query pattern analysis."""

    def __init__(self, window_size: int = 1000, alert_threshold: float = 0.8):
        self.query_log: list[dict] = []
        self.window_size = window_size
        self.alert_threshold = alert_threshold

    def log_query(self, client_id: str, query_vector: list[float],
                  metadata: dict = None):
        self.query_log.append({
            "client_id": client_id,
            "vector": np.array(query_vector),
            "timestamp": metadata.get("timestamp") if metadata else None,
        })

    def analyze_extraction_risk(self, client_id: str) -> dict:
        """Analyze query patterns for signs of model extraction."""
        client_queries = [
            q for q in self.query_log[-self.window_size:]
            if q["client_id"] == client_id
        ]

        if len(client_queries) < 50:
            return {"risk": "LOW", "reason": "Insufficient query volume"}

        vectors = np.array([q["vector"] for q in client_queries])

        # Indicator 1: Systematic coverage of the embedding space
        coverage = self._compute_space_coverage(vectors)

        # Indicator 2: Unusual query rate (burst patterns)
        query_rate = len(client_queries) / max(
            (client_queries[-1]["timestamp"] -
             client_queries[0]["timestamp"]).total_seconds(), 1
        )

        # Indicator 3: Low semantic coherence (random-looking queries)
        coherence = self._compute_pairwise_similarity(vectors)

        risk_score = (
            0.4 * coverage +
            0.3 * min(query_rate / 100, 1.0) +
            0.3 * (1 - coherence)
        )

        return {
            "risk": "HIGH" if risk_score > self.alert_threshold else "MEDIUM"
                    if risk_score > 0.5 else "LOW",
            "risk_score": round(risk_score, 3),
            "indicators": {
                "space_coverage": round(coverage, 3),
                "query_rate_per_sec": round(query_rate, 2),
                "semantic_coherence": round(coherence, 3),
            },
        }

    @staticmethod
    def _compute_space_coverage(vectors: np.ndarray) -> float:
        singular_values = np.linalg.svd(vectors - vectors.mean(axis=0),
                                         compute_uv=False)
        energy = np.cumsum(singular_values ** 2) / (singular_values ** 2).sum()
        return float(np.searchsorted(energy, 0.95) / len(singular_values))

    @staticmethod
    def _compute_pairwise_similarity(vectors: np.ndarray,
                                      sample_size: int = 100) -> float:
        indices = np.random.choice(len(vectors), min(sample_size, len(vectors)),
                                    replace=False)
        sampled = vectors[indices]
        norms = sampled / np.linalg.norm(sampled, axis=1, keepdims=True)
        similarities = norms @ norms.T
        mask = np.triu(np.ones_like(similarities, dtype=bool), k=1)
        return float(similarities[mask].mean())

5.2.4 PII Leakage in Embedding Space¶

Embeddings encode semantic meaning—including personally identifiable information. Names, addresses, social security numbers, and medical conditions embedded alongside document text become queryable through similarity search. An attacker who discovers that the embedding space clusters PII-bearing documents can craft targeted queries to extract sensitive records.

Mitigations for PII in embedding space include:

  • Pre-embedding PII scrubbing: Remove or redact PII before generating embeddings
  • Differential privacy: Add calibrated noise to embeddings (ε-differential privacy)
  • Dimensionality reduction: Project embeddings to lower dimensions, destroying fine-grained PII signals
  • Access-tier separation: Store PII-bearing embeddings in isolated, more heavily controlled collections
python
def apply_differential_privacy(
    embedding: np.ndarray, epsilon: float = 1.0, sensitivity: float = 1.0
) -> np.ndarray:
    """
    Apply the Laplace mechanism to an embedding vector.
    Lower epsilon = stronger privacy, more noise.
    """
    scale = sensitivity / epsilon
    noise = np.random.laplace(loc=0.0, scale=scale, size=embedding.shape)
    noisy_embedding = embedding + noise
    # Re-normalize to maintain unit sphere property
    noisy_embedding = noisy_embedding / np.linalg.norm(noisy_embedding)
    return noisy_embedding

Key Takeaway — Embedding Inversion and Data Extraction

Embeddings are not anonymous. They can be inverted to reconstruct source text, probed to determine membership, and exploited to extract the underlying model. Treat stored embeddings with the same sensitivity classification as the original data they encode. Apply differential privacy, PII scrubbing, and query monitoring as layered defenses.

5.3 Operational Security for Vector Databases¶

Securing a vector database in production requires platform-specific hardening, continuous monitoring, granular access control, and compliance with data protection regulations. This section provides concrete guidance for the four most widely deployed vector database platforms: Pinecone, Weaviate, Chroma, and pgvector.

5.3.1 Hardening Pinecone¶

Pinecone is a fully managed service, which limits infrastructure-level hardening but shifts responsibility to API security, network controls, and organizational policies.

API Key Management

python
# Pinecone key rotation strategy using a secrets manager
import pinecone
from datetime import datetime, timedelta
import boto3

class PineconeKeyRotator:
    """Automated API key rotation for Pinecone deployments."""

    def __init__(self, secrets_manager_client, secret_name: str):
        self.sm = secrets_manager_client
        self.secret_name = secret_name
        self.rotation_interval = timedelta(days=30)

    def get_active_key(self) -> str:
        response = self.sm.get_secret_value(SecretId=self.secret_name)
        secret = json.loads(response["SecretString"])
        return secret["active_key"]

    def rotate_key(self):
        """
        Rotation procedure:
        1. Generate new key via Pinecone console/API
        2. Store new key as pending in secrets manager
        3. Update all consumers to use new key
        4. Verify new key works
        5. Revoke old key
        6. Promote pending key to active
        """
        current = self.get_active_key()
        new_key = self._request_new_pinecone_key()

        self.sm.update_secret(
            SecretId=self.secret_name,
            SecretString=json.dumps({
                "active_key": current,
                "pending_key": new_key,
                "rotation_started": datetime.utcnow().isoformat(),
            })
        )
        return new_key

    def _request_new_pinecone_key(self) -> str:
        # Implementation depends on Pinecone's key management API
        raise NotImplementedError("Integrate with Pinecone org API")

Network Security

  • Restrict Pinecone access to specific IP ranges using the project's network policy
  • Use AWS PrivateLink or GCP Private Service Connect for private connectivity
  • Implement a proxy layer that enforces additional authentication and rate limiting

Namespace Isolation

python
# Enforce namespace isolation at the application layer
class SecureNamespaceProxy:
    """Application-layer enforcement of tenant namespace isolation."""

    def __init__(self, index, tenant_registry: dict):
        self.index = index
        self.tenant_registry = tenant_registry

    def query(self, tenant_id: str, vector: list, top_k: int = 10,
              **kwargs) -> dict:
        allowed_namespaces = self.tenant_registry.get(tenant_id, [])
        namespace = kwargs.get("namespace", "")

        if namespace not in allowed_namespaces:
            raise PermissionError(
                f"Tenant {tenant_id} not authorized for namespace {namespace}"
            )

        results = self.index.query(
            vector=vector, top_k=top_k, namespace=namespace,
            **{k: v for k, v in kwargs.items() if k != "namespace"}
        )

        # Audit log every query
        self._audit_log(tenant_id, namespace, top_k)
        return results

    def _audit_log(self, tenant_id: str, namespace: str, top_k: int):
        log_entry = {
            "event": "vector_query",
            "tenant_id": tenant_id,
            "namespace": namespace,
            "top_k": top_k,
            "timestamp": datetime.utcnow().isoformat(),
        }
        # Send to SIEM / audit log pipeline
        logger.info(json.dumps(log_entry))

5.3.4 Hardening pgvector¶

Pgvector inherits PostgreSQL's mature security model, making it the most hardened option out of the box—but only if PostgreSQL is properly configured.

sql
-- PostgreSQL + pgvector security hardening

-- 1. Create dedicated roles with minimal privileges
CREATE ROLE vector_reader WITH LOGIN PASSWORD 'rotated-secret';
CREATE ROLE vector_writer WITH LOGIN PASSWORD 'rotated-secret';
CREATE ROLE vector_admin  WITH LOGIN PASSWORD 'rotated-secret';

-- 2. Grant granular permissions
GRANT USAGE ON SCHEMA embeddings TO vector_reader, vector_writer;
GRANT SELECT ON ALL TABLES IN SCHEMA embeddings TO vector_reader;
GRANT SELECT, INSERT, UPDATE ON ALL TABLES IN SCHEMA embeddings TO vector_writer;
GRANT ALL PRIVILEGES ON SCHEMA embeddings TO vector_admin;

-- 3. Row-level security for multi-tenancy
ALTER TABLE embeddings.documents ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON embeddings.documents
    FOR ALL
    USING (tenant_id = current_setting('app.current_tenant')::uuid);

-- 4. Audit logging for vector operations
CREATE TABLE embeddings.audit_log (
    id          BIGSERIAL PRIMARY KEY,
    operation   TEXT NOT NULL,
    table_name  TEXT NOT NULL,
    tenant_id   UUID,
    query_vector_hash TEXT,
    result_count INTEGER,
    executed_by TEXT DEFAULT current_user,
    executed_at TIMESTAMPTZ DEFAULT now()
);

-- 5. Create a function to log similarity searches
CREATE OR REPLACE FUNCTION embeddings.audited_similarity_search(
    query_vec   vector(1536),
    tenant_uuid UUID,
    k           INTEGER DEFAULT 10
)
RETURNS TABLE(id UUID, content TEXT, distance FLOAT) AS $$
BEGIN
    -- Set tenant context for RLS
    PERFORM set_config('app.current_tenant', tenant_uuid::text, true);

    -- Log the query
    INSERT INTO embeddings.audit_log (operation, table_name, tenant_id, result_count)
    VALUES ('similarity_search', 'documents', tenant_uuid, k);

    -- Execute the search with RLS enforced
    RETURN QUERY
    SELECT d.id, d.content,
           (d.embedding <=> query_vec)::float AS distance
    FROM embeddings.documents d
    ORDER BY d.embedding <=> query_vec
    LIMIT k;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

5.3.6 GDPR and Data Protection Compliance¶

Vector databases present unique GDPR challenges because embeddings derived from personal data are themselves personal data under the regulation.

Compliance requirements include:

GDPR RightVector Database Implementation
Right to accessMust be able to identify all embeddings derived from a data subject's records
Right to erasureMust delete embeddings, not just source documents—re-indexing may be required
Right to rectificationUpdating source data requires regenerating and replacing embeddings
Data minimizationStore only embeddings necessary for the stated purpose
Purpose limitationEmbeddings generated for search must not be repurposed for profiling
Data portabilityProvide embeddings in a standard, machine-readable format
python
class GDPRCompliantVectorStore:
    """Wrapper enforcing GDPR data subject rights on a vector database."""

    def __init__(self, vector_client, metadata_db):
        self.vectors = vector_client
        self.metadata = metadata_db

    def handle_erasure_request(self, data_subject_id: str) -> dict:
        """Process a GDPR Article 17 erasure request."""
        # Find all embeddings linked to this data subject
        records = self.metadata.find({
            "data_subject_id": data_subject_id
        })

        deleted_ids = []
        for record in records:
            # Delete from vector database
            self.vectors.delete(ids=[record["vector_id"]])
            # Delete metadata
            self.metadata.delete(record["_id"])
            deleted_ids.append(record["vector_id"])

        # Audit trail (retained for compliance, not for data subject)
        self._log_erasure(data_subject_id, deleted_ids)

        return {
            "status": "completed",
            "vectors_deleted": len(deleted_ids),
            "data_subject_id": data_subject_id,
            "timestamp": datetime.utcnow().isoformat(),
        }

    def _log_erasure(self, data_subject_id: str, vector_ids: list):
        """Maintain erasure audit trail as required by GDPR Article 30."""
        self.metadata.insert({
            "event": "erasure_completed",
            "data_subject_id_hash": hashlib.sha256(
                data_subject_id.encode()
            ).hexdigest(),
            "vectors_deleted_count": len(vector_ids),
            "timestamp": datetime.utcnow().isoformat(),
        })

Key Takeaway — Operational Security

Each vector database platform has distinct security characteristics. Pinecone requires API key discipline and application-layer namespace enforcement. Weaviate needs OIDC configuration and inter-node TLS. Chroma demands authentication enablement from day one. Pgvector benefits from PostgreSQL's mature RLS and role system. All platforms require purpose-built monitoring for AI-specific attack patterns and GDPR-compliant data lifecycle management.

# exercise_03_access_controls.py
import hashlib
import hmac
import time
import json
from functools import wraps
from typing import Callable

class VectorDBAccessController:
    """
    Application-layer access control for vector databases
    that lack native RBAC.
    """

    def __init__(self):
        self.policies: dict[str, dict] = {}
        self.audit_log: list[dict] = []

    def register_tenant(self, tenant_id: str, api_key: str,
                         allowed_collections: list[str],
                         permissions: list[str]):
        """Register a tenant with specific collection access."""
        self.policies[api_key] = {
            "tenant_id": tenant_id,
            "allowed_collections": set(allowed_collections),
            "permissions": set(permissions),
            "created_at": time.time(),
        }

    def authorize(self, api_key: str, collection: str,
                  operation: str) -> bool:
        """Check if the API key is authorized for this operation."""
        policy = self.policies.get(api_key)
        if not policy:
            self._log("DENIED", "unknown", collection, operation,
                      "Invalid API key")
            return False

        tenant_id = policy["tenant_id"]

        if collection not in policy["allowed_collections"]:
            self._log("DENIED", tenant_id, collection, operation,
                      "Collection not in allowed list")
            return False

        if operation not in policy["permissions"]:
            self._log("DENIED", tenant_id, collection, operation,
                      "Operation not permitted")
            return False

        self._log("ALLOWED", tenant_id, collection, operation)
        return True

    def enforce(self, api_key: str, collection: str, operation: str):
        """Decorator-style enforcement that raises on denial."""
        if not self.authorize(api_key, collection, operation):
            raise PermissionError(
                f"Access denied: {operation} on {collection}"
            )

    def _log(self, decision: str, tenant_id: str, collection: str,
             operation: str, reason: str = ""):
        entry = {
            "timestamp": time.time(),
            "decision": decision,
            "tenant_id": tenant_id,
            "collection": collection,
            "operation": operation,
            "reason": reason,
        }
        self.audit_log.append(entry)

    def get_security_report(self) -> dict:
        """Generate a security summary from the audit log."""
        denied = [e for e in self.audit_log if e["decision"] == "DENIED"]
        return {
            "total_requests": len(self.audit_log),
            "denied_requests": len(denied),
            "denial_rate": round(len(denied) / max(len(self.audit_log), 1), 3),
            "cross_tenant_attempts": [
                e for e in denied if "Collection not in allowed" in e.get("reason", "")
            ],
            "invalid_key_attempts": [
                e for e in denied if "Invalid API key" in e.get("reason", "")
            ],
        }


def test_access_controls():
    """Verify that access controls properly prevent cross-tenant access."""
    controller = VectorDBAccessController()

    # Register tenants with scoped access
    controller.register_tenant(
        tenant_id="acme-corp",
        api_key="acme-key-001",
        allowed_collections=["acme-corp"],
        permissions=["query", "insert"],
    )
    controller.register_tenant(
        tenant_id="globex-inc",
        api_key="globex-key-001",
        allowed_collections=["globex-inc"],
        permissions=["query"],
    )

    # Test 1: Legitimate access — should succeed

python
# exercise_04_monitoring_dashboard.py
import time
import json
from collections import defaultdict
from dataclasses import dataclass, field
from datetime import datetime, timedelta

@dataclass
class SecurityEvent:
    timestamp: float
    event_type: str
    client_id: str
    collection: str
    severity: str
    details: dict = field(default_factory=dict)

class VectorDBSecurityMonitor:
    """Real-time security monitoring for vector database deployments."""

    def __init__(self, alert_callback=None):
        self.events: list[SecurityEvent] = []
        self.client_profiles: dict[str, dict] = defaultdict(
            lambda: {"query_count": 0, "collections_accessed": set(),
                     "first_seen": None, "last_seen": None}
        )
        self.alert_callback = alert_callback or self._default_alert
        self.thresholds = {
            "max_query_rate": 50,        # queries per minute
            "max_collections": 2,         # collections per client
            "min_coherence": 0.3,         # minimum query similarity
            "max_results_per_minute": 500, # results returned per minute
        }

    def record_query(self, client_id: str, collection: str,
                     query_vector: list[float], results_count: int):
        """Record a query event and check for anomalies."""
        now = time.time()
        profile = self.client_profiles[client_id]
        profile["query_count"] += 1
        profile["collections_accessed"].add(collection)
        profile["last_seen"] = now
        if not profile["first_seen"]:
            profile["first_seen"] = now

        # Check 1: Query rate
        recent = [e for e in self.events
                  if e.client_id == client_id and now - e.timestamp < 60]
        if len(recent) > self.thresholds["max_query_rate"]:
            self._raise_alert(SecurityEvent(
                timestamp=now, event_type="HIGH_QUERY_RATE",
                client_id=client_id, collection=collection,
                severity="WARNING",
                details={"rate": len(recent), "threshold": self.thresholds["max_query_rate"]},
            ))

        # Check 2: Cross-collection access
        if len(profile["collections_accessed"]) > self.thresholds["max_collections"]:
            self._raise_alert(SecurityEvent(
                timestamp=now, event_type="CROSS_COLLECTION_ACCESS",
                client_id=client_id, collection=collection,
                severity="CRITICAL",
                details={"collections": list(profile["collections_accessed"])},
            ))

        # Check 3: Bulk extraction
        if results_count > self.thresholds["max_results_per_minute"]:
            self._raise_alert(SecurityEvent(
                timestamp=now, event_type="BULK_EXTRACTION",
                client_id=client_id, collection=collection,
                severity="CRITICAL",
                details={"results_count": results_count},
            ))

        self.events.append(SecurityEvent(
            timestamp=now, event_type="QUERY",
            client_id=client_id, collection=collection,
            severity="INFO",
            details={"results_count": results_count},
        ))

    def get_dashboard_data(self) -> dict:
        """Generate dashboard summary data."""
        now = time.time()
        last_hour = [e for e in self.events if now - e.timestamp < 3600]
        alerts = [e for e in last_hour if e.severity in ("WARNING", "CRITICAL")]

        return {
            "timestamp": datetime.utcnow().isoformat(),
            "summary": {
                "total_queries_1h": len(last_hour),
                "active_clients": len({e.client_id for e in last_hour}),
                "alerts_1h": len(alerts),
                "critical_alerts": len(
                    [a for a in alerts if a.severity == "CRITICAL"]
                ),
            },
            "top_clients": sorted(
                [
                    {"client": cid, "queries": p["query_count"],
                     "collections": len(p["collections_accessed"])}
                    for cid, p in self.client_profiles.items()
                ],
                key=lambda x: x["queries"], reverse=True
            )[:10],
            "recent_alerts": [
                {"time": datetime.fromtimestamp(a.timestamp).isoformat(),
                 "type": a.event_type, "client": a.client_id,
                 "severity": a.severity, "details": a.details}
                for a in alerts[-20:]
            ],
        }

    def _raise_alert(self, event: SecurityEvent):
        self.events.append(event)
        self.alert_callback(event)

    @staticmethod
    def _default_alert(event: SecurityEvent):
        icon = "🔴" if event.severity == "CRITICAL" else "��"
        print(
            f"{icon} [{event.severity}] {event.event_type} — "
            f"Client: {event.client_id}, Collection: {event.collection}"
        )
        print(f"   Details: {json.dumps(event.details)}")

Vector databases are frequently deployed with critical misconfigurations: no authentication, no tenant isolation, no encryption, and no monitoring. Vector databases are the memory layer of modern AI systems, and their security posture must match the sensitivity of the data they store. This chapter covered the full spectrum of vector database security: from threat modeling through embedding inversion science to platform-specific hardening and hands-on penetration testing.
