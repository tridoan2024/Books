Appendix C: Security Checklists ⏱ ~12 min read¶

These checklists are designed for use during design reviews, deployment readiness assessments, and periodic security audits of agentic AI systems. Each item should be marked as Compliant, Non-Compliant, Not Applicable, or Planned. Assign an owner and target date for any non-compliant items.

C.1 RAG Deployment Security Checklist¶

Document Ingestion and Processing¶

No.Check ItemStatus
1All documents are scanned for embedded prompt injection payloads before ingestion
2Document source provenance is verified and logged for every ingested document
3File type validation rejects unexpected formats (e.g., executables disguised as PDFs)
4Maximum document size limits are enforced at the ingestion endpoint
5Ingestion rate limiting prevents flooding attacks against the vector store
6Document metadata is validated against a schema before storage
7PII detection runs on all documents before embedding and storage
8Documents are classified by sensitivity level during ingestion
9Chunking strategy uses overlapping windows to prevent boundary exploitation
10Ingestion pipeline logs capture full audit trail including source, timestamp, and hash

Vector Store Security¶

No.Check ItemStatus
11Vector database requires authentication for all operations
12Tenant isolation is enforced at the collection or namespace level
13Row-level security policies restrict document access based on user permissions
14Vector store backups are encrypted at rest and tested for restore integrity
15Database network access is restricted to application-tier IP ranges only
16Embedding vectors are not directly exposed through any API endpoint
17Collection deletion requires multi-party approval
18Vector similarity search results include access control filtering before return
19Database audit logs capture all queries with requesting user identity
20Stale document expiration policy is configured and enforced automatically

Query Processing and Retrieval¶

No.Check ItemStatus
21User queries are sanitized to remove embedded instruction patterns
22Query length limits prevent context window overflow attacks
23Retrieved chunk count is capped per query to limit context pollution
24Relevance score threshold filters out low-confidence retrievals
25Query embedding model is version-pinned and integrity-verified
26Cross-collection queries are disabled unless explicitly authorized
27Query logs capture the full retrieval chain for audit and debugging
28Semantic caching does not serve cached results across tenant boundaries
29Query rewriting logic is deterministic and auditable
30Retrieved documents are re-checked for injection patterns before prompt assembly

Prompt Assembly and LLM Interaction¶

No.Check ItemStatus
31System prompt uses clear delimiters separating instructions from retrieved context
32Instruction hierarchy is enforced (system instructions override context content)
33Context window budget allocates fixed space for system instructions vs. retrieved content
34Prompt template is version-controlled and changes require review approval
35Token counting validates that assembled prompts do not exceed model limits
36Retrieved context is placed in a clearly marked section with boundary tokens
37LLM API calls use minimum necessary temperature and token settings
38Response streaming does not bypass output filtering
39Fallback behavior is defined for LLM API failures or timeouts
40Prompt injection canary tokens are embedded in system instructions for detection

Output Filtering and Response Delivery¶

No.Check ItemStatus
41Output filter checks responses for PII before delivery to users
42Source citations are validated against actually retrieved documents
43Hallucination detection compares response claims against retrieved context
44Response length limits prevent resource exhaustion from verbose outputs
45Toxic content classifier evaluates all responses before delivery
46Response metadata includes retrieval source IDs for traceability
47Error responses do not leak internal system details or document content
48Rate limiting is applied per user at the response delivery layer
49Response caching respects tenant boundaries and access control policies
50Full response audit log is maintained with correlation to input query ID

C.2 Multi-Agent System Security Checklist¶

Agent Identity and Access Control¶

No.Check ItemStatus
1Each agent has a unique, verifiable identity (certificate or signed token)
2Agent capabilities are defined in a static manifest reviewed before deployment
3Tool access follows least-privilege: each agent can only invoke tools it requires
4Agent-to-agent communication requires mutual authentication
5Privilege escalation between agents is denied by default
6Agent registration requires approval and is logged in an immutable registry
7Decommissioned agents have credentials revoked immediately
8Agent role assignments are reviewed quarterly and after personnel changes

Inter-Agent Communication¶

No.Check ItemStatus
9All inter-agent messages are validated against an expected schema
10Messages between agents are signed to prevent tampering
11Communication channels are encrypted in transit (TLS 1.3 minimum)
12Message size limits prevent context overflow between agents
13Input sanitization is applied to all received messages before processing
14Agents do not blindly execute instructions received from peer agents
15Message routing prevents direct communication between non-adjacent agents
16Communication audit log captures all inter-agent messages with timestamps

Orchestration and Control Flow¶

No.Check ItemStatus
17Maximum delegation depth is configured and enforced by the coordinator
18Per-task timeout prevents unbounded execution of delegated work
19Circular delegation is detected and terminated automatically
20The coordinator agent cannot be bypassed for task initiation
21Task decomposition follows predefined patterns reviewed by security
22Parallel execution limits prevent resource exhaustion
23Dead letter queues capture failed delegations for analysis
24Coordinator maintains an authoritative task state machine

Shared State and Memory¶

No.Check ItemStatus
25Shared memory is append-only or uses copy-on-write semantics
26Write access to shared state is scoped per agent role
27Shared state changes are versioned with agent identity and timestamp
28Memory poisoning detection alerts on anomalous write patterns
29Shared state is encrypted at rest and in transit
30State cleanup occurs automatically after task completion

Monitoring, Resilience, and Incident Response¶

No.Check ItemStatus
31Each agent emits structured telemetry (traces, metrics, logs)
32Anomaly detection monitors agent behavior against baseline profiles
33Circuit breakers isolate failing agents from the rest of the system
34Bulkhead patterns prevent cascading failures across the agent network
35Emergency termination switch can stop any individual agent or entire system
36Agent crash recovery does not replay potentially malicious inputs
37Incident response runbook covers agent compromise scenarios
38Chaos testing validates agent system resilience quarterly
39Cost anomaly detection alerts on unexpected token usage spikes
40Post-incident review process includes agent interaction trace analysis

C.3 CLI Agent Workstation Security Checklist¶

Agent Configuration and Installation¶

No.Check ItemStatus
1CLI agent is installed from official, verified distribution channels
2Agent binary or package integrity is verified via checksum or signature
3Auto-update is configured to apply security patches promptly
4Agent configuration file permissions restrict read access to the installing user
5Telemetry and data sharing settings are reviewed and configured per policy
6Agent version and configuration are documented in workstation inventory
7Multiple CLI agents on the same workstation are inventoried and access-controlled

File System Access Controls¶

No.Check ItemStatus
8Agent file system access is restricted to project directories only
9Home directory dotfiles (.ssh, .aws, .env) are excluded from agent context
10Credential files and key material directories are explicitly blocked
11Symlink following is disabled or restricted to prevent directory traversal
12File read operations are logged with the full path and requesting context
13File write operations require explicit user confirmation for sensitive paths
14Temporary file creation is restricted to designated directories
15Maximum file size limits prevent the agent from reading very large files into context
16Binary file access is blocked (agent should only read text-based files)
17Git-ignored files and .gitignore patterns are respected by the agent

Command Execution Security¶

No.Check ItemStatus
18Shell command execution requires explicit user approval before running
19Destructive commands (delete, format, drop, force-push) are blocked or require double confirmation
20Network-accessing commands (curl, wget, ssh) are flagged for user review
21Package installation commands are validated against an approved package list
22Command execution occurs in a sandboxed environment where possible
23Command output is size-limited to prevent context flooding
24Background process creation is restricted and monitored
25Environment variable access is filtered to exclude secrets and credentials
26Command execution timeout prevents long-running or hung processes
27Shell history from agent sessions is captured separately for audit

Code Generation and Repository Safety¶

No.Check ItemStatus
28Generated code undergoes static analysis before being written to disk
29Dependency additions are flagged and validated against known-vulnerability databases
30Git commits by the agent are attributed with a machine identity (not human)
31Pre-commit hooks run security checks on all agent-generated changes
32Agent cannot push directly to protected branches
33Generated code diffs are presented for human review before acceptance
34License compliance scanning runs on any new dependencies introduced
35Secret scanning validates no credentials are embedded in generated code
36Agent-generated pull requests require human approval before merge

Network and Cloud Security¶

No.Check ItemStatus
37Agent API calls are routed through an organizational proxy where required
38Outbound network connections are limited to approved API endpoints
39Cloud credential scope follows least-privilege for agent operations
40Session tokens have short TTLs and are not persisted to disk
41Agent does not have access to production cloud environments from workstations
42VPN or network segmentation isolates development workstations from production

Monitoring and Incident Response¶

No.Check ItemStatus
43Agent activity logs are forwarded to central SIEM
44Anomalous agent behavior (excessive file reads, unusual commands) triggers alerts
45Incident response procedures include steps for compromised CLI agent scenarios

C.4 CI/CD Pipeline Agent Security Checklist¶

Pipeline Configuration¶

No.Check ItemStatus
1AI agent steps run in isolated, ephemeral containers with no persistent state
2Agent container images are pinned to specific digests (not mutable tags)
3Pipeline definitions are stored in version control with required review
4Agent-modifiable pipeline steps are separated from security-critical steps
5Pipeline secrets are injected as masked environment variables, never as file mounts
6Agent execution timeout is configured to prevent runaway processes
7Pipeline YAML modifications by agents are blocked via CODEOWNERS rules

Input Handling¶

No.Check ItemStatus
8PR titles, descriptions, and commit messages are sanitized before agent processing
9PR labels and metadata are validated against allowed values
10File content from PRs is treated as untrusted input by the agent
11Fork PRs receive restricted permissions and no access to repository secrets
12Webhook payloads are signature-verified before triggering agent actions
13Maximum input size limits prevent context overflow in agent processing
14Branch name patterns are validated to prevent injection via branch names

Secret Management¶

No.Check ItemStatus
15Pipeline secrets are scoped to specific environments and not globally available
16Agent steps receive only the minimum secrets required for their function
17Secret values are masked in all pipeline logs, including agent output
18Ephemeral credentials with short TTLs are used instead of long-lived tokens
19Secret rotation is automated and does not require pipeline modifications
20OIDC-based authentication replaces stored secrets where supported
21Secret access audit logs are reviewed for anomalous access patterns

Build and Artifact Security¶

No.Check ItemStatus
22Build artifacts are checksummed and signed before storage
23SBOM (Software Bill of Materials) is generated for every build
24Dependency resolution uses lockfiles and verified registries
25Agent-generated code changes trigger the full test suite, not a subset
26Container images built by the pipeline are scanned for vulnerabilities
27Artifact storage requires authentication and supports immutable tags
28Build reproducibility is verified periodically through rebuild testing

Deployment Gates¶

No.Check ItemStatus
29Production deployments require human approval regardless of agent recommendations
30Deployment rollback procedures are tested and can be triggered automatically
31Canary or blue-green deployment strategies limit blast radius of agent errors
32Post-deployment health checks validate application functionality automatically
33Agent cannot modify deployment approval rules or gate configurations

Monitoring and Audit¶

No.Check ItemStatus
34All agent actions within the pipeline are logged with correlation IDs
35Pipeline execution metrics include agent-specific token usage and cost tracking
