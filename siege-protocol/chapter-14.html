Chapter 14: Identity, Authentication, and Authorization for AI Agents ⏱ ~26 min read¶

The proliferation of autonomous AI agents across enterprise environments has introduced a category of identity management challenges that traditional IAM frameworks were never designed to address. When a software agent autonomously negotiates with another agent, invokes a third-party API, or modifies infrastructure state, who is acting? Who is accountable? And how do we enforce boundaries around entities that can spawn sub-agents, delegate credentials, and operate across trust boundaries without human intervention?

This chapter dissects the identity, authentication, and authorization landscape for AI agents. We move beyond surface-level integrations to examine the structural problems—agent impersonation, credential sprawl, capability drift—and provide concrete patterns for building IAM systems that treat agents as first-class security principals.

14.1 The Identity Problem¶

Agents as Digital Identities¶

In classical IAM, identity is anchored to a human user or, at most, a well-defined service account. AI agents shatter this model. An agent is not a user. It is not a traditional service. It is a dynamic, context-dependent entity whose behavior changes based on its prompt, its model weights, its tool integrations, and the task it has been assigned. Yet it must interact with systems that demand a stable, verifiable identity.

The fundamental tension is this: agents are ephemeral in behavior but must be persistent in accountability. A customer-service agent deployed on Monday with access to order history should not silently acquire access to payment instruments by Tuesday because its prompt was updated. Identity must be bound not just to the agent's deployment artifact, but to its operational envelope—the combination of model version, system prompt hash, tool manifest, and authorized scope.

Consider the identity attributes that distinguish an agent from a human principal:

AttributeHuman UserAI Agent
Authentication factorPassword, biometric, MFAAPI key, certificate, workload attestation
Session durationMinutes to hoursSeconds to days (continuous)
Behavioral consistencyVariable, context-dependentDeterministic within prompt scope
Delegation modelExplicit (OAuth consent)Implicit (tool invocation chains)
MultiplicityOne identity per personMany instances per agent definition
AccountabilityLegal personDeploying organization + operator

A robust agent identity token must encode these distinctions. The following JWT structure captures the minimum viable identity claims for an AI agent:

json
{
  "iss": "https://iam.acmecorp.io",
  "sub": "agent:customer-support-v3",
  "aud": "https://api.acmecorp.io",
  "iat": 1719849600,
  "exp": 1719853200,
  "agent_claims": {
    "agent_class": "customer-support",
    "model_id": "gpt-4o-2025-06-01",
    "prompt_hash": "sha256:a3f8b2c1d4e5f6a7b8c9d0e1f2a3b4c5",
    "tool_manifest_hash": "sha256:1a2b3c4d5e6f7a8b9c0d1e2f3a4b5c6d",
    "deployment_id": "deploy-20250701-prod-us-east-1",
    "owner": "team:customer-experience",
    "max_capability_tier": 2,
    "delegation_chain": []
  },
  "scope": "orders:read customer:read tickets:write",
  "jti": "uuid:7f3a1b2c-4d5e-6f7a-8b9c-0d1e2f3a4b5c"
}

The agent_claims block is critical. The prompt_hash and tool_manifest_hash fields bind the identity to a specific behavioral configuration. If either changes, the token must be reissued—ensuring that identity drift triggers re-evaluation of authorization decisions.

Impersonation Risks¶

Agent impersonation is not a theoretical concern. In multi-agent architectures, agents routinely communicate via message passing, shared memory, or API calls. Without strong identity verification at each interaction boundary, a compromised or malicious agent can impersonate a trusted peer.

The attack surface includes:

  1. Token replay: An agent captures another agent's bearer token from shared infrastructure (logs, message queues, debugging endpoints) and presents it as its own.
  2. Prompt injection identity override: An adversarial input instructs an agent to claim a different identity in its outbound communications—effectively social-engineering downstream agents.
  3. Shadow agent registration: An attacker registers a new agent with a name and scope that mimics a legitimate agent, exploiting weak uniqueness constraints in the agent registry.
  4. Delegation chain manipulation: An agent fabricates or extends a delegation chain to claim permissions it was never granted.

Mitigations require cryptographic identity binding. Every agent-to-agent message should carry a signed identity assertion that downstream consumers verify against a trusted identity provider—not against self-reported claims in the message payload.

OAuth Complexities for Agents¶

OAuth 2.0 was designed for user-delegated authorization. Adapting it to autonomous agents introduces friction at multiple levels:

  • Consent: OAuth assumes a human grants consent. Agents operate without interactive consent flows. The client credentials grant partially addresses this, but it conflates the agent's identity with its deployer's service account.
  • Token lifecycle: Standard access tokens assume short sessions with human-initiated refresh. Agents running continuous workloads need token rotation strategies that do not interrupt processing.
  • Scope granularity: OAuth scopes are typically coarse-grained. Agents need fine-grained, context-sensitive scopes that reflect not just what they can access, but under what conditions.
  • Multi-hop delegation: When Agent A calls Agent B, which calls API C on behalf of the original user, standard OAuth token exchange (RFC 8693) can model the delegation—but the complexity grows combinatorially with agent depth.

Multi-Tenant Agent Isolation¶

In platforms hosting agents from multiple tenants—think of an AI agent marketplace or a shared enterprise orchestration layer—identity isolation is paramount. A tenant's agent must never be able to:

  • Access another tenant's data through shared infrastructure side channels
  • Obtain tokens scoped to another tenant's resources
  • Register identity claims that collide with another tenant's namespace

The isolation model must enforce:

Key Takeaway AI agents require a fundamentally richer identity model than human users or traditional service accounts. Agent identity must encode not just who the agent is, but what it is configured to do—binding model version, prompt hash, tool manifest, and deployment context into a cryptographically verifiable identity token. Without this, authorization decisions are made against a phantom—an identity that no longer reflects the agent's actual behavior.

14.2 Authentication Patterns¶

Authentication for AI agents must satisfy constraints that diverge sharply from human-centric flows. Agents cannot enter passwords, approve push notifications, or scan biometrics. Yet their authentication must be at least as strong—arguably stronger—because agents operate at machine speed, can be instantiated at scale, and often hold elevated privileges.

This section catalogs the primary authentication patterns for agent workloads, evaluating each against the criteria of security strength, operational complexity, and suitability for agent-specific scenarios.

API Keys: The Necessary Evil¶

API keys remain the most common authentication mechanism for agent integrations, and the most frequently abused. Their simplicity is their appeal and their liability.

An API key is a shared secret. It provides no cryptographic proof of the caller's identity—only evidence that the caller possesses the secret. For agents, the risks are amplified:

  • Embedding in prompts: Developers routinely inject API keys into agent system prompts or environment variables without rotation policies.
  • Log exposure: Agents that log their HTTP interactions may inadvertently persist API keys in plaintext.
  • Lateral movement: A single leaked API key can grant an attacker the full scope of the agent's permissions.

If API keys must be used, enforce the following controls:

python
import hashlib
import hmac
import time
import secrets
from dataclasses import dataclass, field
from typing import Optional


@dataclass
class AgentAPIKey:
    """Structured API key with embedded metadata and rotation support."""
    key_id: str
    agent_id: str
    tenant_id: str
    created_at: float = field(default_factory=time.time)
    expires_at: Optional[float] = None
    max_requests_per_minute: int = 100
    allowed_scopes: list[str] = field(default_factory=list)
    key_hash: str = ""  # stored; raw key is never persisted

    @staticmethod
    def generate(agent_id: str, tenant_id: str, ttl_seconds: int = 3600,
                 scopes: Optional[list[str]] = None) -> tuple["AgentAPIKey", str]:
        """Generate a new API key. Returns (metadata, raw_key)."""
        raw_key = secrets.token_urlsafe(48)
        key_id = f"ak_{secrets.token_hex(8)}"
        key_hash = hashlib.sha256(raw_key.encode()).hexdigest()

        api_key = AgentAPIKey(
            key_id=key_id,
            agent_id=agent_id,
            tenant_id=tenant_id,
            expires_at=time.time() + ttl_seconds if ttl_seconds else None,
            allowed_scopes=scopes or [],
            key_hash=key_hash,
        )
        return api_key, raw_key

    def validate(self, presented_key: str) -> bool:
        """Validate a presented key against stored hash with timing-safe comparison."""
        if self.expires_at and time.time() > self.expires_at:
            return False
        presented_hash = hashlib.sha256(presented_key.encode()).hexdigest()
        return hmac.compare_digest(self.key_hash, presented_hash)

The critical detail: the raw key is returned exactly once at generation time. Only the SHA-256 hash is stored. Validation uses hmac.compare_digest to prevent timing attacks.

OAuth 2.0 Client Credentials for Agents¶

The OAuth 2.0 client credentials grant (RFC 6749, Section 4.4) is the canonical machine-to-machine authentication flow and the most appropriate OAuth pattern for autonomous agents. The agent authenticates directly with the authorization server using its own credentials—no user delegation involved.

python
import httpx
import time
import jwt
from dataclasses import dataclass


@dataclass
class AgentOAuthClient:
    """OAuth 2.0 client credentials flow for agent authentication."""
    client_id: str
    client_secret: str
    token_endpoint: str
    scopes: list[str]
    _access_token: str | None = None
    _token_expiry: float = 0.0

    async def get_access_token(self) -> str:
        """Obtain or refresh an access token using client credentials grant."""
        if self._access_token and time.time() < self._token_expiry - 30:
            return self._access_token

        async with httpx.AsyncClient() as client:
            response = await client.post(
                self.token_endpoint,
                data={
                    "grant_type": "client_credentials",
                    "client_id": self.client_id,
                    "client_secret": self.client_secret,
                    "scope": " ".join(self.scopes),
                },
                headers={"Content-Type": "application/x-www-form-urlencoded"},
            )
            response.raise_for_status()
            token_data = response.json()

        self._access_token = token_data["access_token"]
        self._token_expiry = time.time() + token_data.get("expires_in", 3600)
        return self._access_token

    async def introspect_agent_token(self, token: str,
                                     jwks_uri: str) -> dict:
        """Validate and decode an agent's JWT access token."""
        async with httpx.AsyncClient() as client:
            jwks_response = await client.get(jwks_uri)
            jwks = jwks_response.json()

        # Decode without verification first to extract the key ID
        unverified_header = jwt.get_unverified_header(token)
        kid = unverified_header.get("kid")

        # Find the matching public key
        signing_key = None
        for key in jwks["keys"]:
            if key["kid"] == kid:
                signing_key = jwt.algorithms.RSAAlgorithm.from_jwk(key)
                break

        if not signing_key:
            raise ValueError(f"No matching key found for kid: {kid}")

        decoded = jwt.decode(
            token,
            signing_key,
            algorithms=["RS256"],
            audience=self.client_id,
            options={"require": ["exp", "iss", "sub", "agent_claims"]},
        )
        return decoded

The 30-second buffer before expiry (time.time() < self._token_expiry - 30) prevents edge cases where a token expires mid-request. The introspection method validates the full JWT chain, including the agent_claims custom claim defined in Section 14.1.

mTLS for Agent-to-Agent Communication¶

When agents communicate directly—bypassing API gateways—mutual TLS (mTLS) provides the strongest authentication guarantee. Both parties present X.509 certificates, and both verify the other's certificate against a trusted certificate authority.

mTLS is particularly valuable in service mesh architectures (Istio, Linkerd) where agent workloads run as sidecar-proxied containers. The mesh automatically handles certificate issuance, rotation, and verification.

For direct agent-to-agent mTLS outside a service mesh:

python
import ssl
import asyncio
from dataclasses import dataclass
from pathlib import Path


@dataclass
class AgentMTLSConfig:
    """mTLS configuration for agent-to-agent authenticated communication."""
    cert_path: Path
    key_path: Path
    ca_bundle_path: Path
    expected_agent_ou: str = "siege-protocol-agents"
    min_tls_version: int = ssl.TLSVersion.TLSv1_3

    def create_ssl_context(self, purpose: ssl.Purpose) -> ssl.SSLContext:
        """Create an SSL context enforcing mTLS with agent-specific validation."""
        ctx = ssl.SSLContext(ssl.PROTOCOL_TLS_CLIENT
                            if purpose == ssl.Purpose.SERVER_AUTH
                            else ssl.PROTOCOL_TLS_SERVER)
        ctx.minimum_version = self.min_tls_version
        ctx.load_cert_chain(certfile=self.cert_path, keyfile=self.key_path)
        ctx.load_verify_locations(cafile=self.ca_bundle_path)
        ctx.verify_mode = ssl.CERT_REQUIRED
        ctx.check_hostname = False  # We validate agent identity via OU/CN
        return ctx

    def validate_peer_certificate(self, peer_cert: dict) -> bool:
        """Validate that the peer certificate belongs to a legitimate agent."""
        subject = dict(x[0] for x in peer_cert.get("subject", ()))
        issuer = dict(x[0] for x in peer_cert.get("issuer", ()))

        # Verify the certificate is issued by our agent CA
        if issuer.get("organizationName") != "Siege Protocol Agent CA":
            return False

        # Verify the agent belongs to the correct organizational unit
        if subject.get("organizationalUnitName") != self.expected_agent_ou:
            return False

        # Extract agent identity from the Common Name
        agent_cn = subject.get("commonName", "")
        if not agent_cn.startswith("agent:"):
            return False

        # Verify Subject Alternative Names include the agent's SPIFFE ID
        san_entries = peer_cert.get("subjectAltName", ())
        spiffe_ids = [
            val for typ, val in san_entries
            if typ == "URI" and val.startswith("spiffe://")
        ]
        if not spiffe_ids:
            return False

        return True

The certificate validation goes beyond standard TLS verification. It checks the organizational unit, the common name prefix (agent:), and requires a SPIFFE (Secure Production Identity Framework for Everyone) URI in the Subject Alternative Name. This layered validation ensures that only certificates issued specifically for agent workloads are accepted.

Short-Lived Tokens and Automatic Rotation¶

Long-lived credentials are the primary vector for agent credential compromise. The mitigation is aggressive: default token lifetimes of 15 minutes or less, with automatic rotation handled by the agent runtime—not the agent logic itself.

python
import asyncio
import time
from typing import Callable, Awaitable


class TokenRotator:
    """Automatic token rotation for agent workloads."""

    def __init__(self, fetch_token: Callable[[], Awaitable[tuple[str, float]]],
                 rotation_margin_seconds: int = 60):
        self._fetch_token = fetch_token
        self._margin = rotation_margin_seconds
        self._current_token: str | None = None
        self._expiry: float = 0.0
        self._lock = asyncio.Lock()
        self._rotation_task: asyncio.Task | None = None

    async def start(self) -> None:
        """Begin the background token rotation loop."""
        self._rotation_task = asyncio.create_task(self._rotation_loop())

    async def _rotation_loop(self) -> None:
        while True:
            async with self._lock:
                self._current_token, self._expiry = await self._fetch_token()
            sleep_duration = max(
                (self._expiry - time.time()) - self._margin, 10
            )
            await asyncio.sleep(sleep_duration)

    async def get_token(self) -> str:
        """Retrieve the current valid token, waiting if rotation is in progress."""
        async with self._lock:
            if self._current_token and time.time() < self._expiry - 5:
                return self._current_token
        # Token expired or not yet available — force refresh
        async with self._lock:
            self._current_token, self._expiry = await self._fetch_token()
            return self._current_token

    async def stop(self) -> None:
        if self._rotation_task:
            self._rotation_task.cancel()

Workload Identity Federation¶

For agents running in cloud environments, workload identity federation eliminates the need for static credentials entirely. The agent's runtime environment (Kubernetes pod, cloud function, VM instance) attests its identity to the cloud provider, which issues short-lived tokens scoped to the workload's declared permissions.

On Google Cloud, this leverages Workload Identity Federation with Kubernetes service accounts. On AWS, it maps to IAM Roles for Service Accounts (IRSA) or EKS Pod Identity. On Azure, it is Workload Identity with federated credentials.

yaml
# Kubernetes: Agent workload identity binding
apiVersion: v1
kind: ServiceAccount
metadata:
  name: customer-support-agent
  namespace: agents-prod
  annotations:
    # GCP Workload Identity
    iam.gke.io/gcp-service-account: agent-cs@acmecorp-prod.iam.gserviceaccount.com
    # AWS IRSA
    eks.amazonaws.com/role-arn: arn:aws:iam::123456789012:role/agent-customer-support
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: customer-support-agent
  namespace: agents-prod
spec:
  template:
    spec:
      serviceAccountName: customer-support-agent
      containers:
        - name: agent
          image: acmecorp/customer-support-agent:v3.2.1
          env:
            - name: AGENT_IDENTITY_PROVIDER
              value: "workload-identity"
            - name: AGENT_TOKEN_AUDIENCE
              value: "https://api.acmecorp.io"
          volumeMounts:
            - name: token-volume
              mountPath: /var/run/secrets/agent-tokens
              readOnly: true
      volumes:
        - name: token-volume
          projected:
            sources:
              - serviceAccountToken:
                  audience: "https://api.acmecorp.io"
                  expirationSeconds: 900
                  path: agent-token

The projected service account token volume provides a Kubernetes-issued JWT that is automatically rotated every 15 minutes. The agent reads this token from the filesystem—no secrets management SDK required—and exchanges it for an application-level access token via token exchange (RFC 8693).

Key Takeaway Agent authentication must be layered: workload identity federation for cloud-native environments, mTLS for agent-to-agent communication, and OAuth 2.0 client credentials for API access. API keys should be treated as a last resort, never embedded in prompts or logs, and always time-bounded. The goal is to eliminate static, long-lived credentials from the agent lifecycle entirely.

14.3 Authorization Models¶

Authentication answers "who is this agent?" Authorization answers "what is this agent allowed to do, right now, in this context?" For AI agents, the second question is dramatically harder than the first—because the context is dynamic, the capabilities are composable, and the consequences of over-permissioning scale with the agent's autonomy.

Role-Based Access Control (RBAC)¶

RBAC remains the foundation of most enterprise authorization systems, and it serves as a reasonable starting point for agent authorization. Agents are assigned roles; roles carry permissions; permissions gate access to resources.

The limitation is granularity. An agent assigned the data-analyst role can read all datasets in its scope. But what if the agent should only read financial data during market hours, and only when processing a request initiated by a human analyst? RBAC cannot express these constraints without an explosion of roles.

A practical RBAC model for agents:

python
from dataclasses import dataclass, field
from enum import Enum


class Permission(Enum):
    ORDERS_READ = "orders:read"
    ORDERS_WRITE = "orders:write"
    CUSTOMER_READ = "customer:read"
    CUSTOMER_PII_READ = "customer:pii:read"
    TICKETS_WRITE = "tickets:write"
    INFRASTRUCTURE_DEPLOY = "infrastructure:deploy"
    MODEL_INVOKE = "model:invoke"
    AGENT_SPAWN = "agent:spawn"


@dataclass
class AgentRole:
    name: str
    permissions: set[Permission]
    max_token_lifetime_seconds: int = 900
    requires_human_approval: list[Permission] = field(default_factory=list)


# Define standard agent roles
AGENT_ROLES: dict[str, AgentRole] = {
    "customer-support-read": AgentRole(
        name="customer-support-read",
        permissions={
            Permission.ORDERS_READ,
            Permission.CUSTOMER_READ,
            Permission.TICKETS_WRITE,
        },
        max_token_lifetime_seconds=900,
    ),
    "customer-support-elevated": AgentRole(
        name="customer-support-elevated",
        permissions={
            Permission.ORDERS_READ,
            Permission.ORDERS_WRITE,
            Permission.CUSTOMER_READ,
            Permission.CUSTOMER_PII_READ,
            Permission.TICKETS_WRITE,
        },
        max_token_lifetime_seconds=300,  # Shorter lifetime for elevated access
        requires_human_approval=[Permission.CUSTOMER_PII_READ],
    ),
    "infrastructure-agent": AgentRole(
        name="infrastructure-agent",
        permissions={
            Permission.INFRASTRUCTURE_DEPLOY,
            Permission.MODEL_INVOKE,
        },
        max_token_lifetime_seconds=600,
        requires_human_approval=[Permission.INFRASTRUCTURE_DEPLOY],
    ),
}

Note the requires_human_approval field. Certain permissions should never be exercisable by an agent without a human confirming the specific action. This is the "human-in-the-loop" gate that prevents autonomous escalation.

Attribute-Based Access Control (ABAC)¶

ABAC evaluates authorization decisions against attributes of the subject (agent), the resource, the action, and the environment. It is the natural evolution of RBAC for agent workloads, where context determines permission.

A production ABAC engine for agents:

python
import time
from dataclasses import dataclass
from typing import Any
from enum import Enum


class Decision(Enum):
    ALLOW = "allow"
    DENY = "deny"
    REQUIRE_APPROVAL = "require_approval"


@dataclass
class ABACRequest:
    """Authorization request with full context for ABAC evaluation."""
    # Subject attributes (the agent)
    agent_id: str
    agent_class: str
    agent_owner: str
    model_id: str
    prompt_hash: str
    delegation_depth: int  # 0 = direct, 1 = delegated once, etc.

    # Resource attributes
    resource_type: str
    resource_id: str
    resource_sensitivity: str  # "public", "internal", "confidential", "restricted"
    resource_tenant: str

    # Action attributes
    action: str  # "read", "write", "delete", "execute"

    # Environment attributes
    request_time: float
    source_ip: str
    is_within_maintenance_window: bool = False
    human_session_linked: bool = False


class ABACPolicyEngine:
    """Attribute-Based Access Control engine for agent authorization."""

    def __init__(self):
        self._policies: list[dict[str, Any]] = []

    def add_policy(self, policy: dict[str, Any]) -> None:
        self._policies.append(policy)

    def evaluate(self, request: ABACRequest) -> Decision:
        """Evaluate all policies against the request. Deny takes precedence."""
        decisions: list[Decision] = []

        for policy in self._policies:
            result = self._evaluate_policy(policy, request)
            if result is not None:
                decisions.append(result)

        # Deny-overrides combination algorithm
        if Decision.DENY in decisions:
            return Decision.DENY
        if Decision.REQUIRE_APPROVAL in decisions:
            return Decision.REQUIRE_APPROVAL
        if Decision.ALLOW in decisions:
            return Decision.ALLOW
        # Default deny if no policy matched
        return Decision.DENY

    def _evaluate_policy(self, policy: dict, request: ABACRequest):
        conditions = policy.get("conditions", {})
        for attr, expected in conditions.items():
            actual = getattr(request, attr, None)
            if callable(expected):
                if not expected(actual):
                    return None
            elif actual != expected:
                return None
        return Decision(policy["decision"])


# --- Build the policy set ---
engine = ABACPolicyEngine()

# Policy 1: Deny all access to restricted resources from delegated agents
engine.add_policy({
    "name": "deny-delegated-restricted",
    "conditions": {
        "resource_sensitivity": "restricted",
        "delegation_depth": lambda d: d > 0,
    },
    "decision": "deny",
})

# Policy 2: Require human approval for write actions on confidential data
engine.add_policy({
    "name": "approval-confidential-write",
    "conditions": {
        "resource_sensitivity": "confidential",
        "action": "write",
    },
    "decision": "require_approval",
})

# Policy 3: Allow read access to internal resources for first-party agents
engine.add_policy({
    "name": "allow-internal-read",
    "conditions": {
        "resource_sensitivity": "internal",
        "action": "read",
        "delegation_depth": lambda d: d == 0,
    },
    "decision": "allow",
})

# Policy 4: Deny access outside business hours for PII resources
engine.add_policy({
    "name": "deny-pii-off-hours",
    "conditions": {
        "resource_sensitivity": "restricted",
        "request_time": lambda t: not (8 <= time.gmtime(t).tm_hour < 18),
    },
    "decision": "deny",
})

The deny-overrides combination algorithm is deliberate. In agent authorization, the consequences of a false positive (granting access that should have been denied) are severe—an autonomous agent will immediately exercise the permission. Deny-overrides ensures that any single policy can veto access.

Capability-Based Security¶

Capability-based security inverts the access control model. Instead of a central policy engine deciding what an agent can do, the agent holds unforgeable tokens (capabilities) that grant specific permissions on specific resources. Presenting a capability is the authorization.

This model is particularly well-suited to agent delegation. When Agent A asks Agent B to perform a task, Agent A passes a scoped capability—not its full credential. Agent B can only do what the capability allows, regardless of its own role or permissions.

python
import hashlib
import json
import time
import secrets
from dataclasses import dataclass


@dataclass
class Capability:
    """An unforgeable capability token granting specific agent permissions."""
    capability_id: str
    resource: str
    actions: list[str]
    constraints: dict  # e.g., {"max_invocations": 5, "valid_until": <timestamp>}
    issued_to: str     # agent ID
    issued_by: str     # issuing agent or system
    parent_capability: str | None  # for delegation tracking
    mac: str = ""      # Message Authentication Code

    @staticmethod
    def issue(resource: str, actions: list[str], constraints: dict,
              issued_to: str, issued_by: str, signing_key: bytes,
              parent: str | None = None) -> "Capability":
        cap_id = f"cap_{secrets.token_hex(16)}"
        cap = Capability(
            capability_id=cap_id,
            resource=resource,
            actions=actions,
            constraints=constraints,
            issued_to=issued_to,
            issued_by=issued_by,
            parent_capability=parent,
        )
        # Compute MAC over all fields
        payload = json.dumps({
            "id": cap.capability_id, "resource": cap.resource,
            "actions": cap.actions, "constraints": cap.constraints,
            "issued_to": cap.issued_to, "issued_by": cap.issued_by,
            "parent": cap.parent_capability,
        }, sort_keys=True).encode()
        cap.mac = hashlib.blake2b(payload, key=signing_key).hexdigest()
        return cap

    def attenuate(self, new_actions: list[str], new_constraints: dict,
                  new_issued_to: str, signing_key: bytes) -> "Capability":
        """Create a delegated capability with reduced permissions."""
        # Attenuation can only remove actions, never add
        if not set(new_actions).issubset(set(self.actions)):
            raise ValueError("Cannot escalate: new actions must be subset")
        return Capability.issue(
            resource=self.resource,
            actions=new_actions,
            constraints={**self.constraints, **new_constraints},
            issued_to=new_issued_to,
            issued_by=self.issued_to,
            signing_key=signing_key,
            parent=self.capability_id,
        )

The attenuate method is the key primitive. An agent can only create a derived capability with fewer actions and tighter constraints than its own. This guarantees that delegation never escalates privileges—a property known as the principle of attenuation.

OPA/Rego Policies for Agent Authorization¶

Open Policy Agent (OPA) with the Rego policy language provides a declarative, externalized authorization framework that decouples policy from application code. For agent systems, OPA serves as the policy decision point (PDP) that evaluates every authorization request.

rego
package siege_protocol.agent_authz

import rego.v1

# Default deny
default allow := false

# Allow if agent role has the required permission and no deny rules trigger
allow if {
    agent_role := data.roles[input.agent.role]
    input.action in agent_role.permissions
    not denied
}

# Deny if the agent is operating beyond its delegation depth limit
denied if {
    input.agent.delegation_depth > data.config.max_delegation_depth
}

# Deny if accessing restricted resources without human linkage
denied if {
    input.resource.sensitivity == "restricted"
    not input.environment.human_session_linked
}

# Deny if the agent's prompt hash doesn't match the registered identity
denied if {
    registered := data.agent_registry[input.agent.agent_id]
    registered.prompt_hash != input.agent.prompt_hash
}

# Deny write operations during read-only maintenance windows
denied if {
    input.action in {"write", "delete", "execute"}
    input.environment.maintenance_mode == true
}

# Require elevated approval for cross-tenant resource access
requires_approval if {
    input.resource.tenant != input.agent.tenant
}

# Compute the authorization decision with full context
decision := result if {
    requires_approval
    result := {
        "allowed": false,
        "reason": "cross_tenant_access_requires_approval",
        "approval_type": "human_in_the_loop",
    }
} else := result if {
    allow
    result := {"allowed": true, "reason": "policy_allow"}
} else := result if {
    result := {"allowed": false, "reason": "default_deny"}
}

This Rego policy encodes several agent-specific rules: delegation depth limits, prompt hash validation (detecting identity drift), maintenance window enforcement, and cross-tenant access gates. The denied rules compose additively—any single denied condition blocks access regardless of role permissions.

Dynamic Authorization Based on Context¶

Static policies are insufficient for agents operating in dynamic environments. An agent's authorization should adapt to:

  • Conversation context: An agent processing a refund request should temporarily gain write access to the refund system—but only for the specific transaction under discussion.
  • Risk signals: If anomaly detection flags an agent's behavior as deviating from its baseline, its permissions should be dynamically reduced.
  • Time-of-day: Financial agents might be restricted to read-only access outside trading hours.
  • Chain of custody: An agent delegated by another agent inherits constraints from the entire delegation chain.

Least-Privilege Enforcement¶

Least privilege for agents is not a suggestion—it is an architectural constraint that must be enforced at the infrastructure level. The pattern is:

  1. Start with zero permissions. An agent begins with no access. Every permission is explicitly granted.
  2. Grant scoped, time-bounded access. Permissions are tied to specific tasks and automatically expire.
  3. Audit continuously. Every permission exercise is logged with full context: agent identity, resource accessed, action taken, policy that allowed it.
  4. Ratchet down automatically. If an agent has not used a permission in N days, revoke it. This is permission hygiene for agents.
yaml
# Agent least-privilege policy specification
apiVersion: policy.siege-protocol.io/v1
kind: AgentLeastPrivilegePolicy
metadata:
  name: auto-revocation
spec:
  rules:
    - name: unused-permission-revocation
      condition:
        permission_unused_days: 14
      action: revoke
      notification: owner
    - name: scope-reduction-on-anomaly
      condition:
        anomaly_score_threshold: 0.85
      action: reduce_to_minimum_scope
      notification: [owner, security_team]
    - name: emergency-revocation
      condition:
        trigger: security_incident
      action: revoke_all
      notification: [owner, security_team, incident_commander]

Key Takeaway Authorization for AI agents demands a layered approach: RBAC for coarse-grained role assignment, ABAC for context-sensitive decisions, capability-based security for safe delegation, and OPA/Rego for externalized policy management. The cardinal rule is deny-by-default with explicit, time-bounded, auditable grants. Any authorization framework that cannot express "this agent may read order data only when processing a human-initiated request during business hours with a delegation depth of zero" is insufficiently expressive for production agent workloads.

Step 1: Agent Identity Provider¶

The IdP is responsible for registering agents, issuing signed JWTs, and maintaining the agent registry. In production, this would be a hardened, HSM-backed service. For this lab, we use RSA key pairs stored in memory.

python
"""agent_idp.py — Agent Identity Provider for the Siege Protocol IAM lab."""

import json
import time
import uuid
import hashlib
from dataclasses import dataclass, field, asdict
from typing import Optional

import jwt  # PyJWT
from cryptography.hazmat.primitives.asymmetric import rsa, padding
from cryptography.hazmat.primitives import serialization, hashes


# --- Key Management ---

class KeyManager:
    """Manages RSA signing keys for JWT issuance."""

    def __init__(self):
        self._private_key = rsa.generate_private_key(
            public_exponent=65537, key_size=2048
        )
        self._public_key = self._private_key.public_key()
        self._kid = f"siege-{uuid.uuid4().hex[:8]}"

    @property
    def kid(self) -> str:
        return self._kid

    @property
    def private_key_pem(self) -> bytes:
        return self._private_key.private_bytes(
            encoding=serialization.Encoding.PEM,
            format=serialization.PrivateFormat.PKCS8,
            encryption_algorithm=serialization.NoEncryption(),
        )

    @property
    def public_key_pem(self) -> bytes:
        return self._public_key.public_bytes(
            encoding=serialization.Encoding.PEM,
            format=serialization.PublicFormat.SubjectPublicKeyInfo,
        )

    def jwks(self) -> dict:
        """Export the public key as a JWKS document."""
        from jwt.algorithms import RSAAlgorithm
        jwk = json.loads(RSAAlgorithm.to_jwk(self._public_key))
        jwk["kid"] = self._kid
        jwk["use"] = "sig"
        jwk["alg"] = "RS256"
        return {"keys": [jwk]}


# --- Agent Registry ---

@dataclass
class RegisteredAgent:
    agent_id: str
    agent_class: str
    owner: str
    tenant_id: str
    model_id: str
    prompt_hash: str
    tool_manifest_hash: str
    allowed_scopes: list[str]
    max_capability_tier: int = 1
    registered_at: float = field(default_factory=time.time)
    is_active: bool = True


class AgentRegistry:
    """In-memory agent registry. Production: use a database with audit logging."""

    def __init__(self):
        self._agents: dict[str, RegisteredAgent] = {}

    def register(self, agent: RegisteredAgent) -> None:
        if agent.agent_id in self._agents:
            raise ValueError(f"Agent {agent.agent_id} already registered")
        self._agents[agent.agent_id] = agent

    def get(self, agent_id: str) -> Optional[RegisteredAgent]:
        return self._agents.get(agent_id)

    def deactivate(self, agent_id: str) -> None:
        if agent_id in self._agents:
            self._agents[agent_id].is_active = False


# --- Token Issuance ---

class AgentIdentityProvider:
    """Issues and manages JWT tokens for registered agents."""

    def __init__(self, issuer: str, key_manager: KeyManager,
                 registry: AgentRegistry):
        self._issuer = issuer
        self._keys = key_manager
        self._registry = registry
        self._revoked_tokens: set[str] = set()

    def issue_token(self, agent_id: str, audience: str,
                    requested_scopes: list[str],
                    ttl_seconds: int = 900) -> str:
        """Issue a signed JWT for a registered agent."""
        agent = self._registry.get(agent_id)
        if not agent:
            raise ValueError(f"Agent {agent_id} not found in registry")
        if not agent.is_active:
            raise ValueError(f"Agent {agent_id} is deactivated")

        # Enforce scope restriction: only grant scopes the agent is allowed
        granted_scopes = [
            s for s in requested_scopes if s in agent.allowed_scopes
        ]
        if not granted_scopes:
            raise ValueError("No valid scopes in request")

        now = time.time()
        token_id = str(uuid.uuid4())

        claims = {
            "iss": self._issuer,
            "sub": f"agent:{agent_id}",
            "aud": audience,
            "iat": int(now),
            "exp": int(now + ttl_seconds),
            "jti": token_id,
            "scope": " ".join(granted_scopes),
            "agent_claims": {
                "agent_class": agent.agent_class,
                "model_id": agent.model_id,
                "prompt_hash": agent.prompt_hash,
                "tool_manifest_hash": agent.tool_manifest_hash,
                "owner": agent.owner,
                "tenant_id": agent.tenant_id,
                "max_capability_tier": agent.max_capability_tier,
                "delegation_chain": [],
            },
        }

        token = jwt.encode(
            claims,
            self._keys.private_key_pem,
            algorithm="RS256",
            headers={"kid": self._keys.kid},
        )
        return token

    def revoke_token(self, jti: str) -> None:
        self._revoked_tokens.add(jti)

    def is_revoked(self, jti: str) -> bool:
        return jti in self._revoked_tokens

Step 2: Token Validation Middleware¶

The middleware intercepts every inbound request, extracts the JWT, validates its signature and claims, and populates the request context with the authenticated agent identity.

python
"""agent_auth_middleware.py — Token validation for agent requests."""

import time
import json
from dataclasses import dataclass
from typing import Optional

import jwt as pyjwt
from jwt.algorithms import RSAAlgorithm


@dataclass
class AuthenticatedAgent:
    """The validated identity of an authenticated agent."""
    agent_id: str
    agent_class: str
    tenant_id: str
    scopes: list[str]
    model_id: str
    prompt_hash: str
    delegation_chain: list[str]
    token_id: str
    token_expiry: float


class TokenValidationError(Exception):
    """Raised when token validation fails."""
    def __init__(self, message: str, error_code: str):
        super().__init__(message)
        self.error_code = error_code


class AgentAuthMiddleware:
    """Validates agent JWTs against the IdP's public keys."""

    def __init__(self, jwks_document: dict, expected_issuer: str,
                 expected_audience: str, revocation_checker=None):
        self._jwks = jwks_document
        self._issuer = expected_issuer
        self._audience = expected_audience
        self._check_revoked = revocation_checker

    def authenticate(self, authorization_header: str) -> AuthenticatedAgent:
        """Validate a Bearer token and return the authenticated agent."""
        if not authorization_header.startswith("Bearer "):
            raise TokenValidationError(
                "Missing Bearer prefix", "invalid_request"
            )

        token = authorization_header[7:]

        # Extract key ID from header
        try:
            unverified = pyjwt.get_unverified_header(token)
        except pyjwt.exceptions.DecodeError:
            raise TokenValidationError("Malformed token", "invalid_token")

        kid = unverified.get("kid")
        signing_key = self._find_key(kid)
        if not signing_key:
            raise TokenValidationError(
                f"Unknown signing key: {kid}", "invalid_token"
            )

        # Full decode with verification
        try:
            claims = pyjwt.decode(
                token,
                signing_key,
                algorithms=["RS256"],
                issuer=self._issuer,
                audience=self._audience,
                options={
                    "require": ["exp", "iss", "sub", "aud", "jti",
                                "agent_claims", "scope"],
                    "verify_exp": True,
                    "verify_iss": True,
                    "verify_aud": True,
                },
            )
        except pyjwt.ExpiredSignatureError:
            raise TokenValidationError("Token expired", "token_expired")
        except pyjwt.InvalidIssuerError:
            raise TokenValidationError("Invalid issuer", "invalid_issuer")
        except pyjwt.InvalidAudienceError:
            raise TokenValidationError("Invalid audience", "invalid_audience")
        except pyjwt.InvalidTokenError as e:
            raise TokenValidationError(f"Invalid token: {e}", "invalid_token")

        # Check revocation
        jti = claims["jti"]
        if self._check_revoked and self._check_revoked(jti):
            raise TokenValidationError("Token revoked", "token_revoked")

        # Validate agent-specific claims
        agent_claims = claims["agent_claims"]
        if not claims["sub"].startswith("agent:"):
            raise TokenValidationError(
                "Subject is not an agent", "invalid_subject"
            )

        return AuthenticatedAgent(
            agent_id=claims["sub"].replace("agent:", ""),
            agent_class=agent_claims["agent_class"],
            tenant_id=agent_claims["tenant_id"],
            scopes=claims["scope"].split(),
            model_id=agent_claims["model_id"],
            prompt_hash=agent_claims["prompt_hash"],
            delegation_chain=agent_claims.get("delegation_chain", []),
            token_id=jti,
            token_expiry=claims["exp"],
        )

    def _find_key(self, kid: str):
        for key_data in self._jwks.get("keys", []):
            if key_data.get("kid") == kid:
                return RSAAlgorithm.from_jwk(key_data)
        return None

Step 3: Policy Authorization Engine¶

The authorization engine consumes the authenticated agent identity and evaluates it against the policy set. This implementation combines RBAC role lookup with ABAC contextual rules.

python
"""agent_authz_engine.py — Combined RBAC + ABAC authorization for agents."""

import time
from dataclasses import dataclass
from enum import Enum
from typing import Any


class AuthzDecision(Enum):
    ALLOW = "allow"
    DENY = "deny"
    REQUIRE_HUMAN_APPROVAL = "require_human_approval"


@dataclass
class AuthzRequest:
    # From authenticated token
    agent_id: str
    agent_class: str
    tenant_id: str
    scopes: list[str]
    prompt_hash: str
    delegation_depth: int

    # From the incoming request
    resource_type: str
    resource_id: str
    resource_tenant: str
    resource_sensitivity: str  # public, internal, confidential, restricted
    action: str               # read, write, delete, execute

    # Environmental context
    request_time: float
    human_session_linked: bool = False


@dataclass
class AuthzResult:
    decision: AuthzDecision
    reason: str
    policy_name: str
    evaluated_at: float = 0.0

    def __post_init__(self):
        if not self.evaluated_at:
            self.evaluated_at = time.time()


class AgentAuthorizationEngine:
    """Production-grade authorization engine combining RBAC and ABAC."""

    def __init__(self, agent_registry_lookup=None):
        self._registry_lookup = agent_registry_lookup

    def authorize(self, request: AuthzRequest) -> AuthzResult:
        """Evaluate authorization for an agent request."""

        # Rule 1: Scope check — the token must contain the required scope
        required_scope = f"{request.resource_type}:{request.action}"
        if required_scope not in request.scopes:
            return AuthzResult(
                decision=AuthzDecision.DENY,
                reason=f"Missing required scope: {required_scope}",
                policy_name="scope-enforcement",
            )

        # Rule 2: Tenant isolation — agents cannot access cross-tenant data
        if request.resource_tenant != request.tenant_id:
            return AuthzResult(
                decision=AuthzDecision.DENY,
                reason="Cross-tenant access denied",
                policy_name="tenant-isolation",
            )

        # Rule 3: Identity drift detection
        if self._registry_lookup:
            registered = self._registry_lookup(request.agent_id)
            if registered and registered.prompt_hash != request.prompt_hash:
                return AuthzResult(
                    decision=AuthzDecision.DENY,
                    reason="Agent prompt hash does not match registry",
                    policy_name="identity-drift-detection",
                )

        # Rule 4: Delegation depth limit
        if request.delegation_depth > 2:
            return AuthzResult(
                decision=AuthzDecision.DENY,
                reason=f"Delegation depth {request.delegation_depth} exceeds max 2",
                policy_name="delegation-depth-limit",
            )

        # Rule 5: Restricted resources require human-in-the-loop
        if request.resource_sensitivity == "restricted":
            if not request.human_session_linked:
                return AuthzResult(
                    decision=AuthzDecision.REQUIRE_HUMAN_APPROVAL,
                    reason="Restricted resource requires human approval",
                    policy_name="restricted-resource-gate",
                )

        # Rule 6: Confidential write operations require human session
        if (request.resource_sensitivity == "confidential"
                and request.action in ("write", "delete")):
            if not request.human_session_linked:
                return AuthzResult(
                    decision=AuthzDecision.REQUIRE_HUMAN_APPROVAL,
                    reason="Confidential write requires human approval",
                    policy_name="confidential-write-gate",
                )

        # All rules passed
        return AuthzResult(
            decision=AuthzDecision.ALLOW,
            reason="All policies satisfied",
            policy_name="default-allow-after-checks",
        )

  3. Workload identity: Deploy the agent in Kubernetes with projected service account tokens and implement the token exchange flow to obtain application-level JWTs.
  4. Anomaly-triggered revocation: Integrate a behavioral anomaly detector that automatically revokes tokens when an agent's API call patterns deviate from its baseline.

Key Takeaway A production agent IAM system requires three tightly integrated components: an Identity Provider that binds agent configuration to cryptographic identity, a validation layer that verifies every request against the IdP's public keys and revocation lists, and a policy engine that makes context-aware authorization decisions. The lab demonstrates that these components are not aspirational—they are implementable with standard libraries and patterns, and they must be deployed before agents are granted access to production resources.

Summary¶

Identity, authentication, and authorization form the security bedrock for AI agent deployments. Without rigorous IAM, every other security control—sandboxing, monitoring, guardrails—operates on an unstable foundation. The patterns in this chapter provide a concrete path from the traditional human-centric IAM model to one that treats agents as first-class security principals with cryptographically verifiable identities, context-aware permissions, and auditable access trails.

In Chapter 15, we turn to the monitoring and observability layer: how to detect when agents violate their authorization boundaries, how to trace actions through multi-agent delegation chains, and how to build the audit infrastructure that regulators and incident responders require.
