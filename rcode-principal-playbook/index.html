rcode Principal Engineer Playbook

Vibe Coding at the Highest Level

You don't write code. You architect systems, make decisions, and orchestrate AI agents to build production-grade software at 10x speed. This is how a principal engineer uses rcode.

The Principal Mindset

A senior engineer tells rcode what to code. A principal engineer tells rcode what to BUILD — then validates the architecture.

You operate at three levels: 1. Vision — What are we building and why 2. Architecture — How should the system be structured 3. Orchestration — Deploy multiple agents to execute in parallel

You rarely write a single function. You design entire systems and let rcode implement them while you review, course-correct, and make judgment calls.

Level 1: System Design Sessions

Kickoff a New System

Don't start with code. Start with architecture:

> I'm building a real-time event processing pipeline.
  Requirements:
  - Ingest 50K events/sec from Kafka
  - Transform and enrich with user data from PostgreSQL
  - Route to 3 downstream services based on event type
  - Dead letter queue for failures
  - Exactly-once delivery semantics

  Design the architecture. Show me the component diagram,
  data flow, and technology choices with tradeoffs.
  Don't write code yet.

Review the design. Push back:

> Why Kafka Streams over Flink? Our team has Flink experience.
  Also, the exactly-once guarantee needs to survive broker failover.
  Revise the design with Flink and show me the failure modes.

Only after the design is solid:

> Good. Now scaffold the project structure.
  Create the directory layout, Cargo.toml, and interface definitions.
  No implementations yet — just the types, traits, and module boundaries.

Architecture Decision Records

> Create an ADR for the decision to use event sourcing over CRUD.
  Include: context, decision, consequences, alternatives considered.
  Save to docs/adr/001-event-sourcing.md

System Boundary Reviews

> Read the entire src/api/ and src/services/ directories.
  Draw the dependency graph between modules.
  Identify any circular dependencies or layering violations.
  Propose fixes that maintain backward compatibility.

Level 2: Multi-Agent Orchestration

This is where principal-level rcode usage separates from senior-level.

The Parallel Build Pattern

Instead of coding sequentially, spawn agents for each module:

> I need to build 4 microservices. Here's the spec for each:

  1. user-service: CRUD for users, JWT auth, PostgreSQL
  2. order-service: order lifecycle, payment integration, event publishing
  3. notification-service: email/SMS/push, template engine, delivery tracking
  4. gateway: rate limiting, auth middleware, request routing

  Use agents to implement all 4 in parallel.
  Each must have: API handlers, service layer, data layer, tests, Dockerfile.
  Shared types go in packages/common/.

The Review-Fix-Verify Pipeline

After multi-agent builds, run the validation pipeline:

> Now do this in sequence:
  1. Run cargo check across all services — fix any compile errors
  2. Run cargo test — fix any failing tests
  3. Review the API contracts between services — are they consistent?
  4. Check for hardcoded values that should be config
  5. Report: what's production-ready and what needs more work

The Codebase Audit

> Audit this entire codebase as a principal engineer would.
  I want:

  1. Architecture assessment — is the structure sound?
  2. Dependency analysis — any risky or outdated deps?
  3. Error handling — are we properly handling failures at every boundary?
  4. Security — input validation, auth checks, secret management
  5. Performance — any obvious N+1 queries, unbounded allocations, or hot paths?
  6. Operational readiness — logging, metrics, health checks, graceful shutdown?
  7. Test coverage — what's well-tested and what's not?

  Rate each area: production-ready / needs-work / critical-gap.
  Show file:line for every finding.

Level 3: Vibe Coding Patterns

Vibe coding means you express INTENT and rcode handles the HOW. But at the principal level, your intents are system-level, not function-level.

Pattern 1: The Vision Prompt

> Build a CLI tool that does for API testing what pytest does for unit testing.
  Users define API test suites in YAML. The tool runs them in parallel,
  validates responses against JSON schemas, generates reports,
  and integrates with CI/CD via exit codes.

  Think about this like you're designing a product, not just writing code.
  Consider: extensibility, plugin system, error reporting UX, performance at scale.

  Design first, then implement.

Pattern 2: The Constraint-Driven Build

> Implement a connection pool with these constraints:
  - Zero unsafe code
  - No allocations in the hot path
  - Configurable min/max connections
  - Health checking with exponential backoff
  - Metrics emission via traits (not hardcoded to any metrics lib)
  - Must pass Tokio's loom testing for concurrency correctness

  I care more about correctness than features. Prove it with tests.

Pattern 3: The Evolution Prompt

> This codebase started as a prototype and now serves 100K users.
  Read the entire src/ directory and identify:

  1. Prototype patterns that don't scale (global state, sync I/O, no error handling)
  2. Missing operational features (no metrics, no structured logging, no circuit breakers)
  3. Architecture debt (god objects, leaky abstractions, missing interfaces)

  Create a migration plan. Prioritize by blast radius.
  Phase 1 should be achievable in a week with zero API changes.

Pattern 4: The Standard-Setter

> Read all existing code in src/. Extract the patterns and conventions being used.
  Now create:

  1. CLAUDE.md with the definitive coding standards for this project
  2. A template for new API endpoints (handler + service + tests + types)
  3. A template for new database migrations
  4. An error handling convention (error types, propagation, user messages)

  These should be so clear that any developer (human or AI) produces consistent code.

Pattern 5: The Cross-Cutting Concern

> Add distributed tracing to every service boundary in this codebase.

  Requirements:
  - OpenTelemetry-compatible spans
  - Trace context propagated across HTTP and message queue calls
  - Every database query gets a span with the SQL template (not values)
  - Every external API call gets a span with URL and status
  - Configurable sampling rate
  - Zero-cost when tracing is disabled

  This must be non-invasive — use middleware/decorators, not manual instrumentation.
  Existing tests must still pass.

Level 4: Production Engineering

Deploy Pipeline Design

> Design the CI/CD pipeline for this project:

  - GitHub Actions
  - Run tests, lint, security scan on every PR
  - Build Docker images on merge to main
  - Deploy to staging automatically
  - Deploy to production with manual approval
  - Rollback on health check failure
  - Canary deploys for the API service

  Create all the workflow YAML files. Use matrix builds where appropriate.

Incident Response Playbooks

> Based on the architecture, create runbooks for:

  1. Database connection pool exhaustion
  2. Kafka consumer lag exceeding 5 minutes
  3. Memory leak in the API service
  4. Certificate expiration
  5. Dependency service outage (graceful degradation)

  Each runbook: detection, impact assessment, immediate mitigation,
  root cause investigation, prevention measures.
  Save to docs/runbooks/

Performance Engineering

> I need to optimize the /search endpoint from 800ms p99 to under 200ms.

  1. Read the full request path: handler → service → DB query → serialization
  2. Identify every allocation, every I/O call, every serialization step
  3. Propose optimizations ranked by impact vs effort
  4. Implement the top 3
  5. Add benchmark tests so we can measure the improvement

Level 5: Knowledge Management

Teach rcode Your Engineering Principles

> remember: I architect for operational excellence first, features second.
  Every service must have: health check, graceful shutdown, structured logging,
  metrics, and circuit breakers before any business logic.

> remember: My design principles are:
  1. Make illegal states unrepresentable (type system enforcement)
  2. Parse, don't validate (convert unstructured to structured at boundaries)
  3. Make it work, make it right, make it fast (in that order)
  4. Every error message must be actionable by the person who reads it

> remember: When I say "production-ready" I mean:
  Tests, error handling, logging, metrics, health checks, graceful shutdown,
  configuration via env vars, Docker-ready, and documented.
  Not just "it compiles."

Project CLAUDE.md at Principal Level

# OrderSystem

## Architecture Principles
- Hexagonal architecture: domain core has zero external dependencies
- All I/O goes through ports (traits), adapters are injected
- Event-driven: services communicate via events, not direct calls
- Configuration: 12-factor app, all config via env vars

## Non-Functional Requirements
- Every endpoint: <200ms p99
- Every service: graceful shutdown within 30 seconds
- Zero data loss: all state changes are event-sourced
- Observability: every request gets a trace ID, every error gets structured context

## Coding Standards
- No unwrap() except in tests
- All public APIs get doc comments with examples
- Error types per module, aggregate at service boundary
- Integration tests use testcontainers, not mocks

## Review Checklist
Before any code is committed:
1. Does it handle all error cases?
2. Is there a test for the happy path AND the failure path?
3. Will this work when the database is slow? When the network drops?
4. Can I deploy this without downtime?
5. Can I roll this back without data loss?

Level 6: The Orchestration Workflow

Here's a complete workflow for building a production system in one session:

Phase 1: Architecture (10 min)

> /model opus
> /effort high
> Design a webhook delivery system. Requirements: [spec].
  Show architecture, data model, API contract, failure handling.

Review. Push back. Iterate until the design is right.

Phase 2: Foundation (5 min)

> Scaffold the project: directory structure, dependencies, 
  core types, trait definitions, error types.
  No implementations — just the skeleton.

Phase 3: Parallel Build (15 min)

> /model sonnet
> Implement all modules in parallel using agents:
  Agent 1: HTTP API layer (handlers, middleware, routing)
  Agent 2: Domain logic (webhook registration, delivery, retry)
  Agent 3: Data layer (PostgreSQL, migrations, queries)
  Agent 4: Worker (queue consumer, delivery executor, circuit breaker)
  Agent 5: Tests (unit + integration test scaffolding)

Phase 4: Integration (10 min)

> Wire everything together. Make cargo check pass.
  Then make cargo test pass. Fix any issues.

Phase 5: Hardening (10 min)

> Now harden for production:
  1. Add structured logging to every boundary
  2. Add metrics (request count, latency, error rate, queue depth)
  3. Add health check endpoint
  4. Add graceful shutdown
  5. Add configuration via env vars
  6. Create Dockerfile

Phase 6: Validation (5 min)

> /model opus
> /effort high
> Review everything we built. 
  Am I missing anything for production? 
  Rate each component: ready / needs-work / gap.

Total: ~55 minutes for a production-grade system.
