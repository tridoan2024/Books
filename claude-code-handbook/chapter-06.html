Chapter 6: Rules – Path-Scoped Coding Standards¶ ~20 min read

A CLAUDE.md that tries to hold every coding standard for every language in your project will bloat to 400 lines. Those 400 lines are injected into every single API call, whether you are editing a Python file or a shell script. You pay for all of them in tokens, every turn, and Claude pays for them in attention – the more instructions it has to juggle, the more likely it is to ignore the ones that matter right now.

Rules solve this problem. They are modular, scoped, and loaded on demand. A Python rule loads when you are working on Python. A shell rule loads when you are editing a bash script. A security rule loads on every session because security is never out of scope. This is not just organization – it is engineering the context window for maximum compliance.

This chapter covers everything you need to know about rules: how they work, how to write them, how scoping works under the hood, and how to avoid the mistakes that make rules useless.

What Rules Are¶

Rules are markdown files that live in .claude/rules/. Each file contains coding standards, conventions, tool preferences, or domain knowledge that Claude Code should follow when generating or reviewing code.

There are two kinds:

Always-on rules have no paths: frontmatter. They load into Claude Code’s context on every session, every turn, regardless of what files you are working with. Use these for cross-cutting concerns that apply everywhere: security practices, testing philosophy, commit conventions.

Path-scoped rules have YAML frontmatter with paths: declarations. They load only when you are working with files that match the specified glob patterns. Use these for language-specific conventions, framework-specific patterns, or domain-specific knowledge that only matters in certain parts of the codebase.

Here is the key mental model: rules are an extension of CLAUDE.md, but with conditional loading. Claude Code merges them into the system prompt alongside CLAUDE.md content, and the model sees them as authoritative project instructions – not suggestions, not hints, but rules to follow.

How Claude Code Decides Which Rules to Load¶

Every time Claude Code assembles a system prompt (which happens before every LLM API call), it evaluates which rules are “in scope.” The process works like this:

  1. Collect in-scope files. Claude Code maintains a set of files that are currently relevant: files you have recently read, files you have recently edited, files mentioned in your conversation, files in your current working directory listing.

  2. Evaluate each rule. For every .md file in .claude/rules/:

  3. If the rule has no paths: frontmatter, it is always in scope. Load it.

  4. If the rule has paths: frontmatter, check each glob pattern against the in-scope files. If any in-scope file matches any pattern, load the rule.

  5. Inject into system prompt. Loaded rules are appended to the system prompt, after CLAUDE.md content and before tool definitions.

This means rules are not statically loaded at session start. They are dynamically evaluated on every turn. If you start a session working on Python files (Python rules load), then switch to editing shell scripts (shell rules load, Python rules may unload if no Python files remain in scope), the rules adapt in real time.

As discussed in Chapter 3, this dynamic loading is part of the system prompt assembly step in the agentic loop. The token cost of rules is paid only when they are in scope, which is why path-scoped rules are dramatically more efficient than stuffing everything into CLAUDE.md.

Rule File Anatomy¶

A rule file is a standard markdown file with optional YAML frontmatter. Here is the complete structure:

markdown
---
paths:
  - "**/*.py"
  - "**/*.pyi"
---

# Python Rules

## Style
- Use type hints on all function signatures
- Use f-strings over .format() or %
- Use pathlib over os.path for file operations
- Use dataclasses or Pydantic for data structures, not raw dicts

## Imports
- Group: stdlib, third-party, local -- separated by blank line
- Use absolute imports. Avoid wildcard imports

## Error Handling
- Catch specific exceptions, never bare `except:`
- Use `logging` module, not print(), for production code

## Tools
- Formatter: `ruff format`
- Linter: `ruff check --fix`
- Type checker: `pyright` or `mypy --strict`

Let’s break down each section.

The Frontmatter¶

The YAML frontmatter block is delimited by --- on its own line. The only recognized field is paths:, which takes a list of glob patterns.

yaml
---
paths:
  - "**/*.py"
  - "**/*.pyi"
---

If the frontmatter is omitted entirely, the rule is always-on. If the frontmatter is present but paths: is empty or missing, the behavior is undefined – do not do this. Either include paths: with at least one pattern, or omit the frontmatter block entirely.

The Content¶

After the frontmatter, the rest of the file is standard markdown. Claude Code injects this content verbatim into the system prompt. You can use any markdown formatting: headers, lists, code blocks, tables, bold, links.

The content is read by the model as authoritative instructions. Write rules the way you would write instructions for a new team member who is technically skilled but unfamiliar with your project’s conventions. Be specific. Be actionable. Do not assume context.

File Location¶

Rules must live in .claude/rules/ at the root of your project. Claude Code scans this directory (non-recursively – subdirectories are not supported as of this writing) for .md files.

The filename does not affect behavior, but use descriptive names: python.md, security.md, testing.md, react-components.md. Your future self will thank you.

Path Scoping Mechanics¶

Path scoping is the mechanism that makes rules efficient. Understanding how glob patterns work and how “in scope” is determined will save you from rules that never load or rules that load when they should not.

Glob Pattern Syntax¶

Rules use standard glob patterns with these wildcards:

Pattern Matches Example
* Any characters within a single directory level *.py matches main.py but not src/main.py
** Any number of directories (including zero) **/*.py matches main.py, src/main.py, src/lib/util.py
? Exactly one character test_?.py matches test_a.py but not test_ab.py
{a,b} Either a or b *.{ts,tsx} matches both .ts and .tsx files

The most common pattern you will use is **/*.ext, which means “any file with this extension, at any depth in the project.” This is almost always what you want for language-specific rules.

Multiple Patterns Per Rule¶

A single rule file can have multiple path patterns. The rule loads if any pattern matches any in-scope file. This is a logical OR:

yaml
---
paths:
  - "**/*.ts"
  - "**/*.tsx"
  - "**/*.js"
  - "**/*.jsx"
---

# JavaScript/TypeScript Rules

This rule loads when you are working with any JavaScript or TypeScript file. You do not need separate rule files for .ts and .tsx – combine them when the rules apply equally.

However, if the rules differ significantly between file types, use separate rule files:

yaml
# .claude/rules/react-components.md
---
paths:
  - "**/*.tsx"
  - "**/*.jsx"
---

# React Component Rules
- Use functional components, not class components
- Extract hooks into custom hooks when reused across 3+ components
yaml
# .claude/rules/node-scripts.md
---
paths:
  - "scripts/**/*.ts"
  - "tools/**/*.ts"
---

# Node Script Rules
- Use commander or yargs for CLI argument parsing
- Always set process.exitCode, never call process.exit() directly

How “In Scope” Files Are Determined¶

This is where many people get confused. A file is “in scope” if it meets any of these conditions:

  • Recently read. You or Claude Code used the Read tool to view the file in the current conversation.
  • Recently edited. You or Claude Code used the Edit or Write tool to modify the file.
  • Mentioned in conversation. You referenced the file by name in your message (e.g., “look at src/auth/login.py”).
  • In the current directory listing. Claude Code includes a snapshot of the project’s file structure in the system prompt. Files visible in this listing are loosely in scope.
  • Part of staged changes. Files that show up in git diff --staged are in scope.

The practical implication: if you open a session and immediately ask “How should I handle errors in Python?”, the Python rule may or may not load depending on whether any .py files are visible in the directory listing or recent git context. But if you say “Read src/main.py” first, the Python rule will definitely load because src/main.py matches **/*.py.

This is by design. Rules load based on what you are actually working on, not what you might hypothetically work on.

When Rules Load vs. When They Do Not¶

Here is a concrete scenario that illustrates the loading behavior:

Turn 1: "Read src/api/routes.py"
         → Python rules load (routes.py matches **/*.py)
         → Security rules load (always-on)

Turn 2: "Now read deploy/run.sh"
         → Shell rules load (run.sh matches **/*.sh)
         → Python rules still loaded (routes.py is still in scope)
         → Security rules still loaded (always-on)

Turn 3: /clear
         → Context reset. All per-turn scope is cleared.

Turn 4: "Read deploy/run.sh"
         → Shell rules load
         → Security rules load
         → Python rules do NOT load (no Python files in scope)

The /clear command resets the conversation context, which clears the in-scope file set. This is one reason Chapter 10 recommends using /clear between unrelated tasks – it keeps the rule set clean and relevant.

Built-In Rule Examples¶

Let’s examine five production-grade rules that cover the most common needs. These are not toy examples – they are rules you can use directly in your own projects.

python.md – Python Coding Standards¶

markdown
---
paths:
  - "**/*.py"
  - "**/*.pyi"
---

# Python Rules

## Style
- Use type hints on all function signatures
- Use f-strings over .format() or %
- Use pathlib over os.path for file operations
- Use dataclasses or Pydantic for data structures, not raw dicts
- Prefer `from __future__ import annotations` for forward references

## Imports
- Group: stdlib, third-party, local -- separated by blank line
- Use absolute imports. Avoid wildcard imports
- Prefer specific imports: `from pathlib import Path` over `import pathlib`

## Error Handling
- Catch specific exceptions, never bare `except:`
- Use `logging` module, not print(), for production code
- Let exceptions propagate unless you can handle them meaningfully

## Tools
- Formatter: `ruff format`
- Linter: `ruff check --fix`
- Type checker: `pyright` or `mypy --strict`
- Package manager: `uv` preferred, `pip` fallback

Notice the specificity. It does not say “write clean Python.” It says “use ruff format” – not black, not autopep8, not yapf. It says “pathlib over os.path” – not “use modern path handling.” Every rule is a concrete instruction that Claude Code can follow without judgment calls.

The “Tools” section is particularly important. Without it, Claude Code might run black for formatting or flake8 for linting, both of which are reasonable defaults but not your defaults. Specifying tool preferences eliminates the “which formatter?” question entirely.

typescript.md – TypeScript Conventions¶

markdown
---
paths:
  - "**/*.ts"
  - "**/*.tsx"
---

# TypeScript Rules

## Language
- Enable strict mode in tsconfig.json
- Prefer `const` over `let`. Never use `var`
- Use async/await over raw Promises or callbacks
- Prefer `interface` over `type` for object shapes (unless union/intersection needed)
- Use `unknown` over `any`. If `any` is truly needed, add a comment explaining why

## Functions
- Explicit return types on exported functions
- Use arrow functions for callbacks: `items.map((x) => x.id)`
- Prefer named exports over default exports

## Error Handling
- Use discriminated unions for error states, not exceptions
- Type-narrow errors in catch blocks: `if (error instanceof SpecificError)`

## React (if applicable)
- Functional components only. No class components
- Props interface named `{ComponentName}Props`
- Hooks before any conditional logic

## Tools
- Formatter: `prettier`
- Linter: `eslint` with `@typescript-eslint/parser`
- Build: `tsup` or `tsc`

security.md – Cross-Cutting Security (Always-On)¶

markdown
# Security Rules

## Code
- NEVER hardcode secrets, tokens, API keys, or passwords
- NEVER commit .env, credentials.json, token.json, *.pem files
- Use environment variables or secret managers for all credentials
- Validate and sanitize ALL external input (user input, API responses, file contents)
- Use parameterized queries -- never string concatenation for SQL/commands

## Dependencies
- Pin dependency versions. Audit with `pip audit` or `npm audit`
- Prefer well-maintained packages with recent commits
- Check for known CVEs before adding new dependencies

## Infrastructure
- Default to least privilege for all permissions
- Encrypt data in transit (HTTPS/TLS). Encrypt at rest for sensitive data
- Never log sensitive data (tokens, passwords, PII)
- Set CORS headers explicitly -- never use wildcard in production

## Review Checklist
- [ ] No secrets in code or config committed to git
- [ ] Input validation at all system boundaries
- [ ] Authentication/authorization checks on all protected endpoints
- [ ] Error messages don't leak internal details
- [ ] Dependencies audited for known vulnerabilities

No paths: frontmatter. This rule loads on every session. Security is never optional and never out of scope.

The checklist at the end is intentional. When Claude Code generates or reviews code, it will reference this checklist as acceptance criteria. Checklists are surprisingly effective in rules because they give the model a concrete verification step.

testing.md – Test Philosophy (Always-On)¶

markdown
# Testing Rules

## Approach
- Write tests for new features and bug fixes
- For bug fixes: write a failing test FIRST, then fix the bug
- Prefer running single tests over full suite for speed during development
- Run full suite before committing

## Python Testing
- Framework: `pytest` with `pytest-asyncio` for async code
- Use fixtures for setup/teardown, not setUp/tearDown methods
- Use `tmp_path` fixture for filesystem tests
- Integration tests hit real services -- never mock databases unless explicitly asked
- Name tests: `test_<what>_<condition>_<expected>`

## Test Structure
tests/
  unit/          -- Fast, isolated, no I/O
  integration/   -- External services, databases
  conftest.py    -- Shared fixtures

## Coverage
- Focus on critical paths, not coverage percentage
- Test edge cases: empty input, None, boundary values, error conditions
- Don't test framework internals or getter/setter boilerplate

This is always-on because testing philosophy applies regardless of language. The Python-specific section could be split into a separate path-scoped rule if your project is multilingual, but for Python-dominant projects, keeping it here avoids an extra file.

shell.md – Shell Script Standards¶

markdown
---
paths:
  - "**/*.sh"
  - "**/*.bash"
---

# Shell Script Rules

## Header
Every script starts with:
#!/usr/bin/env bash
set -euo pipefail

## Style
- Quote all variables: `"${var}"` not `$var`
- Use `[[ ]]` not `[ ]` for conditionals
- Use `$(command)` not backticks
- Use `readonly` for constants
- Use `local` for function variables
- Prefer `printf` over `echo` for portability

## Error Handling
- Check command existence with `command -v`
- Use trap for cleanup: `trap cleanup EXIT`
- Provide meaningful exit codes (0=success, 1=general error, 2=usage error)

## Naming
- Functions: `snake_case`
- Constants: `UPPER_SNAKE_CASE`
- Local variables: `lower_snake_case`
- Files: `kebab-case.sh`

The set -euo pipefail rule is critical. Without it, Claude Code might generate shell scripts that silently swallow errors – a common source of hard-to-debug failures in CI/CD pipelines and automation.

Writing Effective Rules¶

The difference between a rule that Claude Code follows reliably and one it ignores is specificity. Here are the principles that make rules work.

Be Specific and Actionable¶

Bad rule:

markdown
- Write clean, maintainable code

Claude Code cannot act on this. “Clean” and “maintainable” are subjective – the model will fall back to its training defaults, which may not match your standards.

Good rule:

markdown
- Use type hints on all function signatures
- Maximum function length: 30 lines. Extract helper functions beyond that
- No nested callbacks deeper than 2 levels -- use async/await or extract functions

Each of these is a concrete, verifiable instruction. Claude Code can check: “Does this function have type hints? Yes/No.” There is no ambiguity.

Include Tool Preferences¶

Bad:

markdown
- Format code properly

Good:

markdown
- Formatter: `ruff format`
- Linter: `ruff check --fix`
- Run both after editing Python files

Without explicit tool preferences, Claude Code will choose based on its training data. That means it might run black (popular but not your choice), autopep8 (older), or prettier (wrong language). Specifying the tool eliminates this failure mode.

Include Anti-Patterns¶

Rules that say what to do are good. Rules that also say what not to do are better, because they address the model’s most likely mistakes.

markdown
## Do
- Use `pathlib.Path` for all file operations
- Use `tmp_path` fixture for filesystem tests

## Do Not
- Do not use `os.path.join()` -- use `Path / "subdir" / "file.txt"`
- Do not use `open()` with string paths -- use `Path.read_text()` / `Path.write_text()`
- Do not create temp files with `tempfile` in tests -- use the `tmp_path` fixture

The “Do Not” section directly addresses patterns that Claude’s training data would otherwise suggest. os.path.join() is overwhelmingly common in Python codebases, so the model gravitates toward it unless explicitly told not to.

Keep Rules Focused¶

One domain per file. Do not create a rules.md that covers Python, TypeScript, shell, security, and testing. Split them:

.claude/rules/
├── python.md       # Python language conventions
├── typescript.md   # TypeScript language conventions
├── shell.md        # Shell script conventions
├── security.md     # Security practices (always-on)
├── testing.md      # Testing philosophy (always-on)
└── react.md        # React component patterns

Focused rules have two advantages:

  1. Efficient loading. When you are working on a shell script, you get shell rules but not Python rules. This keeps the system prompt lean and focused.

  2. Easier maintenance. When your Python conventions change, you edit one file. When you add React conventions, you add one file. No risk of accidentally breaking unrelated rules.

Show Examples of Good and Bad Patterns¶

For complex rules, inline examples dramatically improve compliance:

markdown
## API Error Responses

Good:
```python
@app.exception_handler(ValidationError)
async def validation_error_handler(request: Request, exc: ValidationError) -> JSONResponse:
    return JSONResponse(
        status_code=422,
        content={"error": "validation_error", "details": exc.errors()},
    )

Bad:

python
@app.exception_handler(Exception)
async def error_handler(request, exc):
    return {"error": str(exc)}  # Leaks internal details, no status code, catches too broadly


The model learns from examples more reliably than from abstract descriptions. When you show the exact pattern you want, Claude Code reproduces it. When you show the exact anti-pattern with a comment explaining why it is wrong, Claude Code avoids it.

## Rules vs. CLAUDE.md -- When to Use Which

This is one of the most common questions, and the answer is straightforward:

| Use Case | Where | Why |
|----------|-------|-----|
| Communication style ("be terse", "no emojis") | `CLAUDE.md` | Applies to every interaction regardless of code |
| Git conventions (commit format, branch naming) | `CLAUDE.md` | Applies to every commit regardless of language |
| Verification workflow ("run tests before committing") | `CLAUDE.md` | Applies to every task |
| Language-specific conventions (type hints, formatting) | `.claude/rules/` with `paths:` | Only relevant when working in that language |
| Framework-specific patterns (React hooks, FastAPI) | `.claude/rules/` with `paths:` | Only relevant in that part of the codebase |
| Security practices | `.claude/rules/` (always-on) | Important enough to warrant a dedicated file, but applies everywhere |
| Domain knowledge (this API uses pagination like X) | `.claude/rules/` with `paths:` | Only relevant when working in that domain |
| One-off project instructions ("we are migrating to v2") | `CLAUDE.md` | Temporary, applies broadly |

The guiding principle: **if it applies to all code in all languages at all times, it goes in `CLAUDE.md`. If it applies to a subset of files, it goes in a path-scoped rule.**

There is a grey area for things like testing philosophy. If your project is exclusively Python, putting `pytest` conventions in `CLAUDE.md` is fine. If your project is multilingual, extract Python testing rules into a path-scoped `python-testing.md` and JavaScript testing rules into `js-testing.md`, and keep only the universal testing philosophy ("test-first for bugs", "run tests before committing") in `CLAUDE.md`.

### Token Budget Considerations

As discussed in Chapter 3, `CLAUDE.md` content is loaded on every single API call. Every line of `CLAUDE.md` costs you tokens on every turn. Rules with `paths:` frontmatter only cost tokens when relevant files are in scope.

For a typical session of 50 turns:

- A 50-line block in `CLAUDE.md` costs: 50 lines x ~3 tokens/line x 50 turns = **7,500 tokens**
- The same 50 lines in a path-scoped rule that is in scope for 15 turns: 50 x 3 x 15 = **2,250 tokens**

Over a day of heavy usage (500+ turns), the savings are substantial. More importantly, a leaner `CLAUDE.md` means the model has fewer instructions competing for attention on any given turn, which improves compliance with the rules that *are* loaded.

## Advanced: Combining Rules with @imports in CLAUDE.md

`CLAUDE.md` supports `@` imports that reference other files. This is how you keep `CLAUDE.md` lean while ensuring rules are discoverable:

```markdown
# CLAUDE.md

## Communication
- Be terse. Lead with the answer
- No emojis. No trailing summaries

## Coding Standards
- Run tests before committing
- Never commit secrets

## Domain Rules
@.claude/rules/python.md
@.claude/rules/security.md
@.claude/rules/testing.md
@.claude/rules/shell.md

The @ import mechanism and path-scoped rule loading are complementary but distinct features:

@ imports are static. They are resolved when CLAUDE.md is loaded and the referenced content is always included in the system prompt. Use @ imports for rules that you want loaded every session regardless of file scope – typically always-on rules like security and testing.

Path-scoped loading is dynamic. Rules with paths: frontmatter are loaded based on which files are in scope on each turn. These do not need @ imports – Claude Code discovers them automatically by scanning .claude/rules/.

So what is the right approach? Consider this pattern:

markdown
# CLAUDE.md

## Core Standards
- [your universal rules here]

## Domain Rules
@.claude/rules/security.md
@.claude/rules/testing.md

Here, security.md and testing.md are @ imported because they are always-on and you want them loaded unconditionally. Language-specific rules like python.md and shell.md are not imported – they are automatically discovered and loaded by the path-scoping mechanism when relevant files come into scope.

If you @ import a rule that also has paths: frontmatter, the content will be included via the import regardless of path matching. The paths: scoping is effectively bypassed for imported rules. This is rarely what you want, so as a general practice: @ import always-on rules, and let path-scoped rules be discovered automatically.

markdown

  - "backend/**/*.py"

# Python Backend Rules

## Style
- Type hints on all function signatures
- f-strings over .format()
- pathlib over os.path
- Pydantic models for request/response schemas

## FastAPI
- Use dependency injection for database sessions
- Use `Annotated[Type, Depends()]` syntax
- Return explicit status codes: `status_code=status.HTTP_201_CREATED`

## Database
- SQLAlchemy 2.0 style (select(), not query())
- Alembic for migrations -- never modify tables directly

## Tools
- Formatter: `ruff format`
- Linter: `ruff check --fix`
- Type checker: `pyright`
- Package manager: `uv`

Note the paths: pattern: backend/**/*.py. This rule loads only when working with Python files inside the backend/ directory. If your project has Python scripts in infra/ that follow different conventions, those scripts will not trigger this rule.

api-conventions.md¶

markdown
---
paths:
  - "backend/src/api/**/*.py"
  - "frontend/src/**/api*.ts"
  - "shared/types/**/*"
---

# API Convention Rules

## REST
- Resources are plural nouns: /users, /orders (not /user, /getOrders)
- Use HTTP methods correctly: GET reads, POST creates, PUT replaces, PATCH updates, DELETE removes
- Return 201 for creation, 204 for deletion, 422 for validation errors

## Response Format
- Envelope: `{"data": ..., "meta": {"page": 1, "total": 100}}`
- Error envelope: `{"error": {"code": "VALIDATION_ERROR", "message": "...", "details": [...]}}`
- Timestamps in ISO 8601 UTC: `2026-04-01T12:00:00Z`

## Versioning
- URL prefix: /api/v1/
- Breaking changes get a new version. Additive changes do not

This rule triggers when working with API route files in the backend, API client files in the frontend, or shared type definitions. It ensures consistent conventions across the stack boundary.

The Result¶

When you work on backend/src/api/routes.py, Claude Code loads: - python-backend.md (matches backend/**/*.py) - api-conventions.md (matches backend/src/api/**/*.py) - security.md (always-on, via @ import) - testing.md (always-on, via @ import)

When you switch to frontend/src/components/LoginForm.tsx: - typescript-frontend.md (matches frontend/**/*.tsx) - security.md (always-on) - testing.md (always-on)

When you edit infra/scripts/deploy.sh: - shell.md (matches **/*.sh) - security.md (always-on) - testing.md (always-on)

Each context gets exactly the rules it needs. No Python rules polluting your TypeScript work. No React conventions cluttering your shell script editing. The system prompt stays focused, and Claude Code’s compliance stays high.

What Most People Get Wrong¶

After reviewing dozens of Claude Code configurations, the same mistakes appear repeatedly. Here is what to avoid.

Mistake 1: Putting Everything in CLAUDE.md¶

The most common mistake. A developer writes a 300-line CLAUDE.md with Python rules, TypeScript rules, SQL conventions, deployment procedures, and team communication standards. Every token of this is injected into every API call. The model has so many instructions that it follows none of them reliably.

Fix: Keep CLAUDE.md under 50 lines. Extract language and domain rules into .claude/rules/ with path scoping.

Mistake 2: Vague, Aspirational Rules¶

“Write performant code.” “Follow best practices.” “Use appropriate design patterns.” These are not rules – they are wishes. The model interprets them according to its training data, which may not match your expectations.

Fix: Every rule should be a concrete instruction that can be verified as followed or not followed. “Use ruff format for Python formatting” is verifiable. “Format code nicely” is not.

Mistake 3: Not Specifying Tools¶

If your rule says “lint the code” but does not say which linter, Claude Code will pick one. It might pick the wrong one. You will spend a turn correcting it, or worse, you will not notice and end up with inconsistent tooling.

Fix: Always specify the exact tool and command. ruff check --fix, not “run the linter.” prettier --write, not “format the file.”

Mistake 4: Overlapping Path Patterns¶

Two rules with overlapping patterns that give contradictory instructions:

yaml
# rule-a.md
---
paths: ["**/*.ts"]
---
# Use semicolons in TypeScript
yaml
# rule-b.md
---
paths: ["**/*.ts", "**/*.tsx"]
---
# Do not use semicolons -- Prettier removes them

Both load when working with .ts files. Claude Code sees contradictory instructions. It will follow one and ignore the other, unpredictably.

Fix: Audit your rules for overlapping patterns. If two rules apply to the same files, ensure their instructions are complementary, not contradictory.

Mistake 5: Rules That Are Too Long¶

A 200-line rule file defeats the purpose of modularity. If your Python rule file covers style, testing, FastAPI, SQLAlchemy, Celery, Redis, and deployment, it is too long.

Fix: Split by subdomain. python-style.md for language conventions, fastapi.md for framework patterns, sqlalchemy.md for database patterns. Each loads only when its relevant files are in scope.

Mistake 6: Forgetting That Rules Are Dynamic¶

A developer writes a path-scoped rule, opens a new session, and asks “What rules are you following?” without first reading any matching files. Claude Code reports that the path-scoped rule is not loaded. The developer concludes rules are broken.

Fix: Rules load based on in-scope files. Read or mention a file that matches the rule’s patterns first. Or use an always-on rule (no frontmatter) if the rule should apply regardless.

Chapter Summary¶

Five takeaways from this chapter:

  1. Rules are modular, conditional extensions of CLAUDE.md. Always-on rules (no frontmatter) load every session. Path-scoped rules (with paths: globs) load only when matching files are in scope. This keeps the system prompt lean and focused.

  2. Specificity is everything. “Use ruff format” works. “Format code properly” does not. Every rule should be a concrete, verifiable instruction – include the tool name, the command, the pattern to follow, and the anti-pattern to avoid.

  3. One domain per file. Keep rules focused. Python conventions in one file, TypeScript in another, security in a third. This maximizes the benefit of path scoping and makes rules easier to maintain.

  4. Understand the loading mechanics. Rules load based on in-scope files (recently read, edited, mentioned). Use @ imports in CLAUDE.md for always-on rules. Let path-scoped rules be discovered automatically. Do not @ import rules that have paths: frontmatter.

  5. Audit for conflicts. When multiple rules apply to the same files, ensure their instructions are complementary. Contradictory rules produce unpredictable behavior – the model will follow one and ignore the other, and you will not know which until you notice the inconsistency in generated code.

Rules are the mechanism that turns Claude Code from a general-purpose coding assistant into your coding assistant – one that knows your formatter, your linter, your conventions, and your anti-patterns. In Chapter 7, we will build on this foundation with skills: reusable AI workflows that teach Claude Code complex, multi-step procedures that go far beyond coding standards.
