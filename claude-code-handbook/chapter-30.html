Chapter 30: Building Multi-Specialized-CLI Systems¶ ~24 min read

Everything in this book has been building toward this chapter.

You have learned to configure Claude Code sessions, write skills and agents, build MCP servers, manage context, and automate workflows. Each of those capabilities is powerful in isolation. But the real unlock comes when you connect multiple specialized Claude Code instances into a coordinated system where each agent has a focused role, its own model, its own tools, and its own operating constraints.

This is the capstone: a multi-CLI architecture where a team of specialized agents collaborates on complex software engineering tasks that no single agent could handle well alone. Not because a single agent lacks the intelligence, but because a single agent lacks the context window, the tool permissions, and the operational focus to do everything at once without degrading quality.

Why Multi-CLI Architecture¶

A single Claude Code session can do remarkable things. But as tasks grow in scope, three constraints bite hard:

Context window saturation. A security audit loads OWASP references, CWE databases, and hundreds of lines of source code. A database migration loads schemas, migration history, and ORM models. Ask one agent to do both and you’ve spent half your context window before any work begins. Specialized agents carry only the context they need.

Tool permission conflicts. Your security agent needs read-only access to production databases. Your deploy agent needs write access to infrastructure. Your test agent needs permission to execute arbitrary code. Granting all permissions to one agent violates least privilege. Separate agents mean separate permission boundaries.

Model cost optimization. Not every task needs Opus. A documentation agent running Haiku at $0.25/million input tokens produces perfectly good README files. A security agent running Opus at $15/million input tokens catches vulnerabilities that cheaper models miss. Multi-CLI lets you right-size the model to the task.

The architecture looks like this:

                    ┌──────────────────────┐
                    │   Human Developer    │
                    └──────────┬───────────┘
                               │
                    ┌──────────▼───────────┐
                    │    Orchestrator CLI   │
                    │   (Sonnet, routing)   │
                    └──────────┬───────────┘
                               │
          ┌────────────────────┼────────────────────┐
          │                    │                     │
┌─────────▼────────┐ ┌────────▼────────┐ ┌─────────▼────────┐
│  SecurityAgent   │ │   DataAgent     │ │   TestAgent      │
│  (Opus, audit)   │ │ (Sonnet, DB)    │ │ (Haiku, runner)  │
└──────────────────┘ └─────────────────┘ └──────────────────┘
          │                    │                     │
┌─────────▼────────┐ ┌────────▼────────┐ ┌─────────▼────────┐
│  DeployAgent     │ │   DocAgent      │ │  ArchitectAgent  │
│ (Sonnet, infra)  │ │ (Haiku, docs)   │ │ (Opus, design)   │
└──────────────────┘ └─────────────────┘ └──────────────────┘

Each box is a separate Claude Code process. Each has its own CLAUDE.md, its own settings.json, its own MCP servers, and its own model. They communicate through a shared workspace on the filesystem and a message bus that the orchestrator manages.

Architecture Patterns¶

Before building, you need to choose how the agents coordinate. There are two fundamental patterns, and a hybrid that works best in practice.

Centralized Orchestration¶

One coordinator receives the task, decomposes it, dispatches subtasks to specialists, collects results, and synthesizes the final output. The specialists never talk to each other directly.

Human → Orchestrator → SecurityAgent → Orchestrator
                     → DataAgent     → Orchestrator
                     → TestAgent     → Orchestrator
                     → Orchestrator  → Human

Advantages: Simple message flow. One place to track progress. Easy to add retry logic. Clear audit trail.

Disadvantages: Orchestrator becomes a bottleneck. If the orchestrator’s context fills up, the whole system degrades. Specialists cannot share insights directly.

Decentralized Peer-to-Peer¶

Agents communicate directly with each other through shared state. Each agent reads from and writes to a shared workspace. No single coordinator.

SecurityAgent ←→ Shared Workspace ←→ DataAgent
     ↕                                    ↕
TestAgent    ←→ Shared Workspace ←→ DeployAgent

Advantages: No single bottleneck. Agents can react to each other’s findings. Scales naturally.

Disadvantages: Coordination is hard. Race conditions on shared state. Difficult to track overall progress. Deadlocks are possible.

Hybrid: Orchestrator with Shared State¶

The practical choice. An orchestrator handles task decomposition, sequencing, and result aggregation. But agents also share a workspace directory where they can read each other’s artifacts — code, test results, audit findings, architecture decisions. The orchestrator does not relay messages between agents; it coordinates the workflow while agents read shared context directly.

Human → Orchestrator → dispatches tasks
                     → agents read/write shared workspace
                     → Orchestrator tracks completion
                     → Orchestrator synthesizes result → Human

This is what we will build.

State Management Across Agents¶

Multi-agent coordination requires shared state. Three approaches, each suited to different needs:

Shared Workspace Directory¶

The simplest and most robust approach. A directory on the filesystem that all agents can read and write. Structure it by purpose:

.multi-cli/
├── config/
│   └── fleet.yaml              # Agent definitions
├── tasks/
│   ├── task-001.json           # Task assignments
│   ├── task-002.json
│   └── task-003.json
├── artifacts/
│   ├── architecture-design.md  # ArchitectAgent output
│   ├── security-audit.md       # SecurityAgent output
│   ├── test-results.json       # TestAgent output
│   └── deploy-plan.md          # DeployAgent output
├── messages/
│   ├── inbox-security.jsonl    # Messages for SecurityAgent
│   ├── inbox-data.jsonl        # Messages for DataAgent
│   └── inbox-orchestrator.jsonl
└── state/
    ├── progress.json           # Overall progress
    └── agent-health.json       # Agent status

Each agent writes its artifacts to artifacts/ and its status updates to state/. The orchestrator reads state/progress.json to track completion. Messages between agents go through messages/inbox-{name}.jsonl — append-only JSONL files that act as simple message queues.

Git Branches¶

For code-producing agents, git branches provide natural isolation. Each agent works on its own branch, and the orchestrator merges results:

main
├── agent/architect-design     # ArchitectAgent's work
├── agent/coder-implement      # CoderAgent's implementation
├── agent/security-audit       # SecurityAgent's findings
└── agent/docs-update          # DocAgent's documentation

The orchestrator creates branches before dispatching tasks and merges them after validation. If an agent’s work fails review, the branch gets discarded without affecting others.

Message Bus with File-Based Queues¶

For real-time coordination, use JSONL files as lightweight message queues. Each line is a JSON message. Agents append to their outbox; the orchestrator (or other agents) read from inboxes.

python
import json
import time
from pathlib import Path
from typing import Any


def send_message(
    inbox_path: Path,
    sender: str,
    msg_type: str,
    payload: dict[str, Any],
) -> None:
    """Append a message to an agent's inbox."""
    message = {
        "timestamp": time.time(),
        "sender": sender,
        "type": msg_type,
        "payload": payload,
    }
    with open(inbox_path, "a") as f:
        f.write(json.dumps(message) + "\n")


def read_messages(inbox_path: Path, since: float = 0.0) -> list[dict[str, Any]]:
    """Read messages from an inbox, optionally filtering by timestamp."""
    if not inbox_path.exists():
        return []
    messages = []
    with open(inbox_path) as f:
        for line in f:
            line = line.strip()
            if not line:
                continue
            msg = json.loads(line)
            if msg["timestamp"] > since:
                messages.append(msg)
    return messages

This is intentionally simple. No Redis, no RabbitMQ, no Kafka. File-based queues are perfectly adequate for a system running six to ten agents on a single machine. You can always graduate to a real message broker if you scale beyond that.

Building the Orchestrator¶

The orchestrator is the brain of the system. It receives a high-level task from the developer, decomposes it into subtasks, dispatches them to specialized agents, monitors progress, and assembles the final result.

Here is a complete implementation.

The Orchestrator Implementation¶

python
#!/usr/bin/env python3
"""
Multi-CLI Orchestrator — Coordinates specialized Claude Code agents.

Usage:
    python orchestrator.py "Add user authentication to the app"
"""
from __future__ import annotations

import json
import subprocess
import sys
import time
import uuid
from dataclasses import dataclass, field
from enum import Enum
from pathlib import Path
from typing import Any

import yaml


# ---------------------------------------------------------------------------
# Data structures
# ---------------------------------------------------------------------------

class TaskStatus(Enum):
    PENDING = "pending"
    RUNNING = "running"
    COMPLETED = "completed"
    FAILED = "failed"
    RETRYING = "retrying"


@dataclass
class AgentConfig:
    name: str
    model: str
    description: str
    tools: list[str] = field(default_factory=list)
    allowed_commands: list[str] = field(default_factory=list)
    skills: list[str] = field(default_factory=list)
    max_retries: int = 2
    timeout_seconds: int = 600


@dataclass
class Task:
    task_id: str
    agent: str
    description: str
    dependencies: list[str] = field(default_factory=list)
    status: TaskStatus = TaskStatus.PENDING
    result: str | None = None
    error: str | None = None
    retries: int = 0
    started_at: float | None = None
    completed_at: float | None = None

    def to_dict(self) -> dict[str, Any]:
        return {
            "task_id": self.task_id,
            "agent": self.agent,
            "description": self.description,
            "dependencies": self.dependencies,
            "status": self.status.value,
            "result": self.result,
            "error": self.error,
            "retries": self.retries,
            "started_at": self.started_at,
            "completed_at": self.completed_at,
        }


@dataclass
class TaskPlan:
    goal: str
    tasks: list[Task]
    created_at: float = field(default_factory=time.time)


# ---------------------------------------------------------------------------
# Fleet loader
# ---------------------------------------------------------------------------

def load_fleet(config_path: Path) -> dict[str, AgentConfig]:
    """Load agent definitions from fleet.yaml."""
    with open(config_path) as f:
        raw = yaml.safe_load(f)

    agents: dict[str, AgentConfig] = {}
    for name, cfg in raw.get("agents", {}).items():
        agents[name] = AgentConfig(
            name=name,
            model=cfg["model"],
            description=cfg["description"],
            tools=cfg.get("tools", []),
            allowed_commands=cfg.get("allowed_commands", []),
            skills=cfg.get("skills", []),
            max_retries=cfg.get("max_retries", 2),
            timeout_seconds=cfg.get("timeout_seconds", 600),
        )
    return agents


# ---------------------------------------------------------------------------
# Task decomposition
# ---------------------------------------------------------------------------

def decompose_task(goal: str) -> TaskPlan:
    """
    Decompose a high-level goal into agent-specific tasks.

    In production, this calls Claude to analyze the goal and produce
    a task graph. Here we show the deterministic decomposition for
    the canonical example: "Add user authentication to the app."
    """
    tasks = [
        Task(
            task_id="arch-001",
            agent="architect",
            description=(
                "Design the authentication architecture: "
                "session management, password hashing, JWT tokens, "
                "database schema for users table. "
                "Write the design to .multi-cli/artifacts/architecture-design.md"
            ),
            dependencies=[],
        ),
        Task(
            task_id="code-001",
            agent="coder",
            description=(
                "Implement the authentication system based on the architecture "
                "design in .multi-cli/artifacts/architecture-design.md. "
                "Create: auth routes, middleware, user model, password hashing. "
                "Write code to src/auth/."
            ),
            dependencies=["arch-001"],
        ),
        Task(
            task_id="test-001",
            agent="tester",
            description=(
                "Write comprehensive tests for the authentication system "
                "in src/auth/. Cover: registration, login, logout, "
                "invalid credentials, token expiry, middleware protection. "
                "Write tests to tests/test_auth.py. Run them and report results "
                "to .multi-cli/artifacts/test-results.json"
            ),
            dependencies=["code-001"],
        ),
        Task(
            task_id="sec-001",
            agent="security",
            description=(
                "Security audit the authentication implementation in src/auth/. "
                "Check: password hashing (bcrypt/argon2), SQL injection, "
                "session fixation, CSRF, token storage, timing attacks. "
                "Write findings to .multi-cli/artifacts/security-audit.md"
            ),
            dependencies=["code-001"],
        ),
        Task(
            task_id="deploy-001",
            agent="deployer",
            description=(
                "Update deployment configuration for the new auth system. "
                "Add environment variables for JWT_SECRET, SESSION_TTL. "
                "Update Dockerfile and docker-compose.yml. "
                "Write deploy plan to .multi-cli/artifacts/deploy-plan.md"
            ),
            dependencies=["code-001", "sec-001"],
        ),
        Task(
            task_id="doc-001",
            agent="documenter",
            description=(
                "Document the authentication system. "
                "Write API docs for auth endpoints, "
                "a getting-started guide for developers, "
                "and update the main README. "
                "Read the implementation in src/auth/ and the architecture "
                "design in .multi-cli/artifacts/architecture-design.md. "
                "Write to .multi-cli/artifacts/auth-documentation.md"
            ),
            dependencies=["code-001", "test-001"],
        ),
    ]

    return TaskPlan(goal=goal, tasks=tasks)


# ---------------------------------------------------------------------------
# Agent execution
# ---------------------------------------------------------------------------

def dispatch_agent(
    agent_config: AgentConfig,
    task: Task,
    workspace: Path,
) -> tuple[bool, str]:
    """
    Dispatch a task to a Claude Code agent as a subprocess.

    Returns (success, output).
    """
    # Build the prompt that the agent will receive
    prompt = build_agent_prompt(agent_config, task, workspace)

    # Construct the claude command
    cmd = [
        "claude",
        "--model", agent_config.model,
        "--print",           # Non-interactive, print output
        "--output-format", "text",
        "--max-turns", "30",
        "--prompt", prompt,
    ]

    # Add allowed tools
    for tool in agent_config.tools:
        cmd.extend(["--allowedTools", tool])

    try:
        result = subprocess.run(
            cmd,
            capture_output=True,
            text=True,
            timeout=agent_config.timeout_seconds,
            cwd=str(workspace),
        )

        if result.returncode == 0:
            return True, result.stdout
        else:
            return False, result.stderr or result.stdout

    except subprocess.TimeoutExpired:
        return False, f"Agent {agent_config.name} timed out after {agent_config.timeout_seconds}s"
    except FileNotFoundError:
        return False, "claude CLI not found. Install Claude Code first."


def build_agent_prompt(
    agent_config: AgentConfig,
    task: Task,
    workspace: Path,
) -> str:
    """Build the complete prompt for an agent, including context from dependencies."""
    # Gather artifacts from completed dependency tasks
    artifacts_dir = workspace / ".multi-cli" / "artifacts"
    context_parts: list[str] = []

    if artifacts_dir.exists():
        for artifact_file in sorted(artifacts_dir.iterdir()):
            if artifact_file.is_file() and artifact_file.suffix in (".md", ".json", ".txt"):
                content = artifact_file.read_text()
                if content.strip():
                    context_parts.append(
                        f"--- Artifact: {artifact_file.name} ---\n{content}"
                    )

    context_block = "\n\n".join(context_parts) if context_parts else "No prior artifacts."

    return f"""You are the {agent_config.name} agent in a multi-agent system.

Your role: {agent_config.description}

## Your Task
{task.description}

## Context from Prior Agents
{context_block}

## Rules
1. Write all output artifacts to the paths specified in your task description.
2. Be thorough. This is production work, not a prototype.
3. If you encounter an issue that blocks your work, write a clear error description
   to .multi-cli/artifacts/{agent_config.name}-error.md and stop.
4. Do not modify files outside your designated area unless the task explicitly requires it.
"""


# ---------------------------------------------------------------------------
# Progress tracking
# ---------------------------------------------------------------------------

class ProgressTracker:
    """Tracks task execution progress and writes state to disk."""

    def __init__(self, workspace: Path, plan: TaskPlan) -> None:
        self.workspace = workspace
        self.plan = plan
        self.state_dir = workspace / ".multi-cli" / "state"
        self.state_dir.mkdir(parents=True, exist_ok=True)
        self._write_progress()

    def get_ready_tasks(self) -> list[Task]:
        """Return tasks whose dependencies are all completed."""
        completed_ids = {
            t.task_id for t in self.plan.tasks
            if t.status == TaskStatus.COMPLETED
        }
        return [
            t for t in self.plan.tasks
            if t.status == TaskStatus.PENDING
            and all(dep in completed_ids for dep in t.dependencies)
        ]

    def mark_running(self, task: Task) -> None:
        task.status = TaskStatus.RUNNING
        task.started_at = time.time()
        self._write_progress()

    def mark_completed(self, task: Task, result: str) -> None:
        task.status = TaskStatus.COMPLETED
        task.result = result
        task.completed_at = time.time()
        self._write_progress()

    def mark_failed(self, task: Task, error: str) -> None:
        task.status = TaskStatus.FAILED
        task.error = error
        task.completed_at = time.time()
        self._write_progress()

    def is_complete(self) -> bool:
        return all(
            t.status in (TaskStatus.COMPLETED, TaskStatus.FAILED)
            for t in self.plan.tasks
        )

    def summary(self) -> str:
        lines = [f"Goal: {self.plan.goal}", ""]
        for t in self.plan.tasks:
            duration = ""
            if t.started_at and t.completed_at:
                duration = f" ({t.completed_at - t.started_at:.1f}s)"
            status_icon = {
                TaskStatus.PENDING: "[ ]",
                TaskStatus.RUNNING: "[~]",
                TaskStatus.COMPLETED: "[x]",
                TaskStatus.FAILED: "[!]",
                TaskStatus.RETRYING: "[r]",
            }[t.status]
            lines.append(f"  {status_icon} {t.task_id} ({t.agent}): {t.description[:60]}...{duration}")
        return "\n".join(lines)

    def _write_progress(self) -> None:
        progress = {
            "goal": self.plan.goal,
            "created_at": self.plan.created_at,
            "tasks": [t.to_dict() for t in self.plan.tasks],
        }
        progress_file = self.state_dir / "progress.json"
        progress_file.write_text(json.dumps(progress, indent=2))

def check_agent_health(agent_config: AgentConfig) -> dict[str, Any]:
    """Verify that an agent's dependencies are available."""
    health: dict[str, Any] = {
        "agent": agent_config.name,
        "model": agent_config.model,
        "status": "healthy",
        "issues": [],
    }

    # Check that claude CLI is available
    try:
        result = subprocess.run(
            ["claude", "--version"],
            capture_output=True,
            text=True,
            timeout=10,
        )
        if result.returncode != 0:
            health["status"] = "degraded"
            health["issues"].append("claude CLI returned non-zero exit code")
    except FileNotFoundError:
        health["status"] = "unhealthy"
        health["issues"].append("claude CLI not found")
    except subprocess.TimeoutExpired:
        health["status"] = "degraded"
        health["issues"].append("claude CLI health check timed out")

    return health

# Main orchestration loop
# ---------------------------------------------------------------------------

def run_orchestrator(goal: str, workspace: Path) -> None:
    """Main orchestration loop: decompose, dispatch, track, aggregate."""
    config_path = workspace / ".multi-cli" / "config" / "fleet.yaml"
    if not config_path.exists():
        print(f"Error: Fleet config not found at {config_path}", file=sys.stderr)
        sys.exit(1)

    # Load fleet configuration
    fleet = load_fleet(config_path)
    print(f"Loaded fleet: {', '.join(fleet.keys())}")

    # Health check all agents
    print("\nHealth checking agents...")
    for name, agent_cfg in fleet.items():
        health = check_agent_health(agent_cfg)
        status = health["status"]
        issues = ", ".join(health["issues"]) if health["issues"] else "none"
        print(f"  {name}: {status} (issues: {issues})")

    # Decompose the goal into tasks
    print(f"\nDecomposing goal: {goal}")
    plan = decompose_task(goal)
    print(f"Created {len(plan.tasks)} tasks")

    # Ensure workspace directories exist
    artifacts_dir = workspace / ".multi-cli" / "artifacts"
    artifacts_dir.mkdir(parents=True, exist_ok=True)

    # Initialize progress tracker
    tracker = ProgressTracker(workspace, plan)
    print(f"\n{tracker.summary()}\n")

    # Main execution loop
    iteration = 0
    max_iterations = len(plan.tasks) * 3  # Safety limit

    while not tracker.is_complete() and iteration < max_iterations:
        iteration += 1
        ready_tasks = tracker.get_ready_tasks()

        if not ready_tasks:
            # Check for deadlock: no ready tasks but not complete
            running = [t for t in plan.tasks if t.status == TaskStatus.RUNNING]
            if not running:
                print("DEADLOCK: No ready tasks and no running tasks.")
                break
            # Wait for running tasks (in a real system, use async)
            time.sleep(2)
            continue

        # Dispatch ready tasks (sequentially here; see async version below)
        for task in ready_tasks:
            agent_cfg = fleet.get(task.agent)
            if not agent_cfg:
                tracker.mark_failed(task, f"Unknown agent: {task.agent}")
                continue

            print(f"Dispatching: {task.task_id} -> {task.agent}")
            tracker.mark_running(task)

            success, output = dispatch_agent(agent_cfg, task, workspace)

            if success:
                tracker.mark_completed(task, output)
                print(f"  Completed: {task.task_id}")
            else:
                # Retry logic
                if task.retries < agent_cfg.max_retries:
                    task.retries += 1
                    task.status = TaskStatus.PENDING  # Re-queue
                    print(f"  Failed: {task.task_id}, retrying ({task.retries}/{agent_cfg.max_retries})")
                else:
                    tracker.mark_failed(task, output)
                    print(f"  Failed permanently: {task.task_id}")

        print(f"\n{tracker.summary()}\n")

    # Final report
    print("=" * 60)
    print("ORCHESTRATION COMPLETE")
    print("=" * 60)
    print(tracker.summary())

    # List produced artifacts
    print("\nArtifacts produced:")
    if artifacts_dir.exists():
        for artifact in sorted(artifacts_dir.iterdir()):
            if artifact.is_file():
                size = artifact.stat().st_size
                print(f"  {artifact.name} ({size:,} bytes)")

    # Check for failures
    failures = [t for t in plan.tasks if t.status == TaskStatus.FAILED]
    if failures:
        print(f"\n{len(failures)} task(s) failed:")
        for t in failures:
            print(f"  {t.task_id}: {t.error[:100] if t.error else 'Unknown error'}")
        sys.exit(1)

# Entry point
# ---------------------------------------------------------------------------

if __name__ == "__main__":
    if len(sys.argv) < 2:
        print("Usage: python orchestrator.py '<goal>'", file=sys.stderr)
        sys.exit(2)

    goal = sys.argv[1]
    workspace = Path.cwd()
    run_orchestrator(goal, workspace)

That is roughly 300 lines of focused orchestration logic. It handles fleet loading, task decomposition, agent dispatch, progress tracking, retry logic, deadlock detection, and result aggregation.

A few design decisions worth noting:

Subprocess-based isolation. Each agent runs as a separate claude subprocess. This gives you process-level isolation — one agent crashing cannot take down another. It also means each agent gets its own context window.

File-based communication. Agents share artifacts through the filesystem, not through the orchestrator’s context window. The orchestrator never reads the full content of an agent’s output. It only tracks success/failure and file paths.

Deterministic decomposition in the example. The decompose_task function is hard-coded for the authentication example. In production, you would call Claude to analyze the goal and produce the task graph dynamically. More on that shortly.

Dynamic Task Decomposition¶

The hard-coded decomposition works for demonstrating the flow, but real systems need Claude to analyze the goal and produce a task plan. Here is how to implement that:

python
def decompose_task_dynamic(
    goal: str,
    fleet: dict[str, AgentConfig],
    workspace: Path,
) -> TaskPlan:
    """Use Claude to decompose a goal into agent tasks."""
    agent_descriptions = "\n".join(
        f"- {name}: {cfg.description}"
        for name, cfg in fleet.items()
    )

    # Scan the workspace for context
    src_files = list(workspace.rglob("*.py"))[:20]  # Sample
    file_list = "\n".join(f"  {f.relative_to(workspace)}" for f in src_files)

    prompt = f"""Decompose this goal into tasks for a multi-agent system.

Goal: {goal}

Available agents:
{agent_descriptions}

Existing project files (sample):
{file_list}

Output a JSON array of tasks. Each task has:
- task_id: unique string (e.g., "arch-001")
- agent: one of the available agent names
- description: detailed description of what the agent should do
- dependencies: list of task_ids that must complete first

Rules:
1. Tasks with no dependencies can run in parallel.
2. Order: design before implementation, implementation before testing/audit.
3. Each task should produce a specific artifact file.
4. Be specific — include file paths, function names, acceptance criteria.

Return ONLY the JSON array, no other text."""

    result = subprocess.run(
        ["claude", "--model", "claude-sonnet-4-20250514", "--print", "--prompt", prompt],
        capture_output=True,
        text=True,
        timeout=120,
        cwd=str(workspace),
    )

    if result.returncode != 0:
        raise RuntimeError(f"Task decomposition failed: {result.stderr}")

    # Parse the JSON task list from Claude's response
    raw_output = result.stdout.strip()
    # Handle markdown code blocks if Claude wraps the JSON
    if "```json" in raw_output:
        raw_output = raw_output.split("```json")[1].split("```")[0].strip()
    elif "```" in raw_output:
        raw_output = raw_output.split("```")[1].split("```")[0].strip()

    task_defs = json.loads(raw_output)

    tasks = [
        Task(
            task_id=td["task_id"],
            agent=td["agent"],
            description=td["description"],
            dependencies=td.get("dependencies", []),
        )
        for td in task_defs
    ]

    return TaskPlan(goal=goal, tasks=tasks)

The key insight: the orchestrator uses a cheaper, faster model (Sonnet) for decomposition because task planning is a well-structured problem. The expensive Opus model is reserved for the agents doing the actual security analysis and architecture design.

Failure Handling and Resilience¶

Production systems fail. Agents time out, models return malformed output, dependencies become unavailable. A robust multi-CLI system handles failures without collapsing.

Retry with Exponential Backoff¶

The basic retry in the orchestrator requeues failed tasks. For production, add exponential backoff:

python
def calculate_backoff(retries: int, base_seconds: float = 2.0) -> float:
    """Exponential backoff with jitter."""
    import random
    delay = base_seconds * (2 ** retries)
    jitter = random.uniform(0, delay * 0.1)
    return min(delay + jitter, 120.0)  # Cap at 2 minutes

Fallback Agents¶

If the primary agent for a task fails repeatedly, fall back to a different agent. A common pattern: if the Haiku-based tester cannot handle a complex test scenario, escalate to Sonnet.

python
FALLBACK_CHAIN: dict[str, list[str]] = {
    "tester": ["coder", "architect"],      # Tester fails → try coder → architect
    "documenter": ["coder"],               # Documenter fails → try coder
    "deployer": ["coder"],                 # Deployer fails → try coder
}

def get_fallback_agent(
    failed_agent: str,
    attempt: int,
    fleet: dict[str, AgentConfig],
) -> AgentConfig | None:
    """Get the next fallback agent for a failed task."""
    chain = FALLBACK_CHAIN.get(failed_agent, [])
    if attempt - 1 < len(chain):
        fallback_name = chain[attempt - 1]
        return fleet.get(fallback_name)
    return None

The Saga Pattern for Rollback¶

When agents produce artifacts that depend on each other, a failure in a later stage may require rolling back earlier work. The saga pattern manages this.

python
@dataclass
class SagaStep:
    task: Task
    compensating_action: str  # Command to undo this step

    def rollback(self, workspace: Path) -> None:
        """Execute the compensating action to undo this step."""
        if self.compensating_action.startswith("git"):
            subprocess.run(
                self.compensating_action.split(),
                cwd=str(workspace),
                capture_output=True,
            )
        elif self.compensating_action.startswith("rm"):
            target = workspace / self.compensating_action.split(" ", 1)[1]
            if target.exists():
                target.unlink()


# Example saga for the authentication feature
saga_steps = [
    SagaStep(
        task=arch_task,
        compensating_action="rm .multi-cli/artifacts/architecture-design.md",
    ),
    SagaStep(
        task=code_task,
        compensating_action="git checkout -- src/auth/",
    ),
    SagaStep(
        task=test_task,
        compensating_action="rm tests/test_auth.py",
    ),
]

If the security audit reveals a fundamental flaw requiring architectural changes, the orchestrator rolls back through the saga steps in reverse order, then re-dispatches from the architecture phase.

Human-in-the-Loop Escalation¶

Some failures require human judgment. Define escalation thresholds:

python
ESCALATION_RULES = {
    "security": {
        "max_retries_before_escalate": 1,
        "escalation_message": "Security audit found critical issues that may require architectural changes.",
    },
    "deployer": {
        "max_retries_before_escalate": 2,
        "escalation_message": "Deployment preparation failed. Infrastructure access may be needed.",

}

def should_escalate(task: Task) -> bool:
    """Determine if a failed task should escalate to a human."""
    rules = ESCALATION_RULES.get(task.agent)
    if not rules:
        return task.retries >= 3  # Default: escalate after 3 failures
    return task.retries >= rules["max_retries_before_escalate"]

def escalate_to_human(task: Task) -> None:
    """Pause execution and notify the developer."""

Deadlock Detection¶

Deadlocks occur when tasks have circular dependencies or when all remaining tasks depend on failed tasks. The orchestrator detects this:

python
def detect_deadlock(plan: TaskPlan) -> str | None:
    """Detect deadlock conditions and return a description, or None."""
    pending = [t for t in plan.tasks if t.status == TaskStatus.PENDING]
    running = [t for t in plan.tasks if t.status == TaskStatus.RUNNING]
    failed = {t.task_id for t in plan.tasks if t.status == TaskStatus.FAILED}

    if not pending and not running:
        return None  # Complete, not deadlocked

    if not running and pending:
        # Check if all pending tasks depend on failed tasks
        for task in pending:
            unmet = [d for d in task.dependencies if d in failed]
            if unmet:
                return (
                    f"Deadlock: {task.task_id} depends on failed tasks: "
                    f"{', '.join(unmet)}"
                )
        # Circular dependency (should not happen with proper decomposition)
        return "Deadlock: Circular dependency detected in task graph"

    return None

@dataclass
class AgentMetrics:
    agent_name: str
    tasks_completed: int = 0
    tasks_failed: int = 0

    estimated_cost_usd: float = 0.0

        return self.tasks_completed / total if total > 0 else 0.0

    def avg_duration(self) -> float:

    output_tokens: int,
) -> float:

    costs = MODEL_COSTS.get(model, {"input": 0.003, "output": 0.015})
    return (input_tokens * costs["input"] + output_tokens * costs["output"]) / 1000

Cost Optimization: Right-Sizing Models¶

The fleet configuration already assigns models by role, but you can go further with dynamic model selection:

python
def select_model_for_task(task: Task, agent_config: AgentConfig) -> str:
    """Dynamically select model based on task complexity."""
    description_length = len(task.description)
    has_security_keywords = any(
        kw in task.description.lower()
        for kw in ["security", "audit", "vulnerability", "authentication"]
    )
    has_architecture_keywords = any(
        kw in task.description.lower()
        for kw in ["design", "architecture", "schema", "contract"]
    )

    # Security and architecture tasks always get Opus
    if has_security_keywords or has_architecture_keywords:
        return "claude-opus-4-20250514"

    # Short, well-defined tasks can use Haiku
    if description_length < 200 and not task.dependencies:
        return "claude-haiku-3-5-20241022"

    # Default to the agent's configured model
    return agent_config.model

This saves money on simple tasks while ensuring critical work gets the best model. In practice, teams report 40-60% cost reduction from right-sizing models compared to running everything on Opus.

Scaling Patterns¶

Horizontal scaling. Run multiple orchestrators for different feature tracks. Each manages its own agent fleet and workspace directory. Use git branches to isolate work.

Vertical scaling. Increase max_turns and timeout_seconds for agents working on larger tasks. Use --max-turns 50 for agents that need to iterate on complex implementations.

Auto-scaling. Monitor queue depth (number of pending tasks) and spin up additional agent instances when demand exceeds capacity:

python
def auto_scale_decision(
    pending_tasks: int,
    running_agents: int,
    max_agents: int = 10,
) -> int:
    """Decide how many additional agents to spawn."""
    if pending_tasks == 0:
        return 0
    desired = min(pending_tasks, max_agents)
    additional = max(0, desired - running_agents)
    return additional

Message Tracing¶

When something goes wrong in a multi-agent system, you need to trace the message flow. Add correlation IDs to every message:

python
import uuid

def create_trace_context(goal: str) -> dict[str, str]:
    """Create a trace context for a complete orchestration run."""
    return {
        "trace_id": str(uuid.uuid4()),
        "goal": goal,
        "started_at": time.strftime("%Y-%m-%dT%H:%M:%SZ", time.gmtime()),
    }

def log_trace_event(
    trace_context: dict[str, str],
    event: str,
    task_id: str | None = None,
    agent: str | None = None,
    details: dict[str, Any] | None = None,
) -> None:
    """Log a trace event for debugging multi-agent flows."""
    entry = {
        "trace_id": trace_context["trace_id"],
        "timestamp": time.time(),
        "event": event,
        "task_id": task_id,
        "agent": agent,
        "details": details or {},
    }
    # Append to trace log
    trace_file = Path(".multi-cli/state/trace.jsonl")
    with open(trace_file, "a") as f:
        f.write(json.dumps(entry) + "\n")

The Future of Multi-CLI Systems¶

Self-Improving Agent Teams¶

The next evolution is agents that improve their own processes. After completing a task, the orchestrator can analyze what worked and what did not, then adjust the fleet configuration:

  • An agent that consistently times out gets its timeout_seconds increased automatically.
  • An agent whose work frequently fails security review gets additional security instructions added to its prompt.
  • An agent that completes tasks in half the allocated time gets reassigned to a cheaper model.

This feedback loop turns the multi-CLI system from a static pipeline into an adaptive one. The system gets better at your specific codebase over time.

Persistent Agent Memory¶

As discussed in Chapter 24 with MCP servers, agents can share structured memory. A security agent that discovers a recurring pattern in your codebase (say, a habit of using string.format() for SQL queries) can write that finding to persistent memory. Future security audits load that memory and check for the known pattern first.

Cross-agent memory creates institutional knowledge that survives individual sessions:

Open Source Ecosystem¶

The multi-CLI pattern is not locked to Claude Code. The orchestrator pattern works with any CLI-based AI agent: GitHub Copilot CLI, Cursor’s agent mode, Aider, or custom agents built on the Claude Agent SDK. The fleet configuration format is model-agnostic — swap model: claude-opus-4-20250514 for model: gpt-4o and the orchestration logic does not change.

Open-source orchestrators are emerging that standardize the task decomposition format, the agent communication protocol, and the artifact schema. The multi-CLI system you build today with Claude Code will plug into tomorrow’s multi-vendor agent ecosystem.

The Convergence¶

Multi-CLI systems are converging with three other trends:

  1. MCP servers give agents access to external tools. Combined with multi-CLI, you get agents that not only write code but also deploy it, monitor it, and respond to production incidents.

  2. Git-based workflows give agents the same collaboration primitives that human developers use. Pull requests, code review, branch protection — these work equally well for agent-authored code.

  3. Continuous integration gives agents a feedback loop. An agent writes code, CI runs tests, the agent reads the results, fixes failures, and pushes again. This loop can run without human intervention for well-defined tasks.

The end state is a development team where humans define goals and review results, and a fleet of specialized agents handles the implementation, testing, security, documentation, and deployment. Not replacing developers, but multiplying their output by handling the tasks that are well-understood, repetitive, and time-consuming.

Chapter Summary¶

  • Multi-CLI systems connect multiple specialized Claude Code instances into a coordinated team, where each agent has a focused role, its own model, its own tools, and its own context window
  • The hybrid orchestrator pattern combines centralized task coordination with shared filesystem state, giving you clear progress tracking without the bottleneck of relaying all data through a single coordinator
  • Fleet configuration in YAML defines each agent’s model, tools, permissions, retry limits, and timeouts — right-sizing the model to the task saves 40-60% on API costs
  • The orchestrator (~300 lines of Python) handles task decomposition, agent dispatch via subprocess, progress tracking, retry logic, deadlock detection, and result aggregation
  • Dependency-aware execution runs independent tasks in parallel (testing and security audit simultaneously) while respecting the task graph’s ordering constraints
  • Failure resilience includes exponential backoff retries, fallback agent chains, the saga pattern for rollback, and human-in-the-loop escalation for tasks that exceed automated recovery
  • Production deployment in CI/CD turns GitHub issues into fully implemented, tested, audited, and documented pull requests — triggered by a label
  • Monitoring and tracing with per-agent metrics, correlation IDs, and JSONL trace logs enable debugging of multi-agent workflows
  • The future points toward self-improving agent teams, persistent cross-agent memory, and an open-source ecosystem where multi-CLI orchestration works across AI vendors
