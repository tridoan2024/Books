Chapter 23: Multi-File Refactoring at Scale¶ ~32 min read

You’re staring at a 2,400-line OrderController.ts that handles order creation, payment processing, inventory checks, email notifications, audit logging, and PDF receipt generation. Everyone on the team agrees it needs to be broken up. Nobody wants to do it. The file has 47 imports and is touched by every PR.

This is where Claude Code becomes the most valuable tool in your arsenal. Multi-file refactoring — the systematic restructuring of code across dozens of files while maintaining correctness — is the task that most dramatically separates Claude Code from other AI tools. It can hold the full dependency graph in context, make coordinated changes across files, and verify correctness at every step.

But “refactor this” is not a strategy. This chapter gives you the strategies.

Why Multi-File Refactoring Is Hard¶

Single-file changes are straightforward: read the file, make the change, run the test. Multi-file refactoring introduces three challenges that compound:

  1. Dependency chains — Changing a function signature in file A requires updating every caller in files B through Z. Miss one and the build breaks.
  2. Ordering sensitivity — Some changes must happen before others. Moving a function to a new module before updating imports creates a window where the code is broken.
  3. Consistency requirements — A rename that’s applied to 18 of 20 files is worse than not renaming at all. Partial refactors create confusion.

Claude Code handles all three because it can search the entire codebase (Grep, Glob), make precise edits across files (Edit with replace_all), and verify results after each step (Bash to run tests). The key is giving it the right strategy.

Strategy 1: Plan Mode for Complex Refactors¶

Plan mode is Claude Code’s read-only exploration mode. When you enter plan mode, Claude can read files, search the codebase, and analyze dependencies — but it cannot write, edit, or execute commands. This constraint forces it to think before acting.

When to Use Plan Mode¶

Use plan mode when: - The refactor affects 5+ files - You’re not sure about the decomposition - Circular dependencies might exist - The change affects public APIs that other teams consume - You want to understand the blast radius before touching anything

The Workflow¶

Step 1: Enter plan mode
Step 2: Claude explores the codebase (read-only)
Step 3: Claude designs a step-by-step refactoring plan
Step 4: You review and approve (or modify) the plan
Step 5: Claude executes the approved plan
Step 6: You verify at each checkpoint

Complete Walkthrough: Planning a Service Extraction¶

Here is the exact interaction, from the prompt you type through the complete plan Claude generates, to partial execution.

Your prompt to enter plan mode:

Enter plan mode.

I want to refactor src/services/order-service.ts (2400 lines) into:
1. OrderCreationService — handles order creation flow
2. PaymentService — processes payments via Stripe
3. InventoryService — stock checks and reservations

The Critical Insight About Plan Mode¶

Plan mode is not just “think before you code.” It’s a forcing function that produces an artifact — the plan — that you can review, modify, and reference throughout execution. Without it, Claude holds the plan in its conversational context, which degrades as the conversation grows. With it, the plan is explicit and reviewable.

Strategy 2: The Worktree Approach¶

For risky refactors where you might need to throw away everything and start over, worktree isolation gives you a full safety net — not just git stash, but a completely separate working directory with its own branch.

How Git Worktrees Work¶

A git worktree is a separate checkout of your repository with its own working directory and branch. Changes in the worktree don’t affect your main working directory until you explicitly merge.

Your main directory:    ~/project/               (branch: main)
Worktree:               ~/project/.claude/worktrees/refactor-auth/
                                                  (branch: refactor-auth)

Both directories share the same .git database, so commits in the worktree are visible from the main directory once you merge. But the working files are completely separate — an experimental change in the worktree cannot corrupt your main branch.

The Complete Worktree Workflow¶

Step 1: Create the worktree

Create a worktree for refactoring the authentication module.

Claude creates a worktree under .claude/worktrees/, creates a new branch, and switches its working context to the worktree directory. From this point, every file read, edit, and bash command operates in the worktree — your main directory is untouched.

Step 2: Do the refactoring work

Refactor src/auth/ to use a strategy pattern for authentication 
providers. Currently we have if/else chains for Google, GitHub, 
and email/password. Each provider should be a separate class 
implementing an AuthProvider interface.

Claude works freely. It creates new files, moves code, updates imports, runs tests. If something breaks, it fixes it. If the approach isn’t working, you have options.

Step 3a: The refactor succeeds

The refactor looks good. All tests pass. Merge it back to main.

Claude exits the worktree, merges the branch into main (or creates a PR if you prefer code review), and removes the worktree directory.

Step 3b: The refactor fails

This approach isn't working. The strategy pattern adds too much 
complexity for three providers. Discard the worktree.

Claude discards the worktree and its branch. Your main directory is exactly as you left it. Zero cleanup required. No git stash pop, no git reset --hard, no “which changes were mine and which were the refactor?”

When to Use Worktrees¶

Scenario Use Worktree? Why
Experimental refactor (might not work) Yes Free rollback
Large refactor spanning multiple sessions Yes Isolates work-in-progress from main
Refactor you want to PR before merging Yes Branch is ready for review
Trying two approaches to compare Yes (two worktrees) Compare side by side
Small, confident refactor (<5 files) No Overhead not worth it
Mechanical rename across 30 files No Low risk, use batch instead

Parallel Worktrees for A/B Refactoring¶

When you’re not sure which approach is better, create two worktrees:

Create two worktrees:
1. refactor-strategy-pattern — refactor auth using the strategy pattern
2. refactor-functional — refactor auth using functional composition

In each worktree, implement the approach and run the full test suite.
Report back: lines of code, test results, and which approach you 
recommend.

Claude creates both worktrees, implements each approach in isolation, and provides a comparison. You keep the better one and discard the other.

Strategy 3: The /batch Command¶

The /batch command is Claude Code’s parallel execution engine. It’s designed for mechanical changes — the same transformation applied to many files simultaneously, using worktrees for isolation and agents for parallelism.

How /batch Works Internally¶

┌─────────────────────────────────────────────┐
│ /batch "Update error format in src/api/"    │
└──────────────────┬──────────────────────────┘
                   │
          ┌────────┴────────┐
          ▼                 ▼
   ┌──────────────┐  ┌──────────────┐
   │ Worktree A   │  │ Worktree B   │
   │ Agent 1      │  │ Agent 2      │
   │ Files 1-15   │  │ Files 16-30  │
   └──────┬───────┘  └──────┬───────┘
          │                 │
          ▼                 ▼
   ┌──────────────────────────────┐
   │ Merge results back           │
   │ Report: 30 files changed     │
   │ Show any conflicts            │
   └──────────────────────────────┘
  1. Claude identifies all affected files using Grep and Glob
  2. Splits them into groups (typically by directory or dependency)
  3. Creates parallel worktrees — one per agent
  4. Each agent applies the transformation independently
  5. Results merge back into your working directory
  6. Any merge conflicts are presented for manual resolution

When to Use /batch vs. Plan Mode¶

Criteria /batch Plan Mode
Same transformation across many files Yes Overkill
Each file can be changed independently Required Not required
Order matters Not supported Supports ordered steps
Changes depend on each other Not supported Handles dependencies
10+ files with the same change Sweet spot Too slow
Complex decomposition with shared state Wrong tool Correct tool

Complete /batch Example¶

You’ve decided to standardize all API error responses across 28 route files:

/batch "In all files matching src/api/routes/**/*.ts, replace 
the old error format:
  res.status(XXX).json({ error: message })
with the new standardized format:
  res.status(XXX).json({ 
    error: { 
      code: ERROR_CODE_CONSTANT, 
      message, 
      timestamp: new Date().toISOString(),
      requestId: req.headers['x-request-id'] 
    } 
  })

Import { ErrorCode } from '@/types/errors' at the top of each file 
if not already imported.

Map each HTTP status to the correct ErrorCode:
- 400 → ErrorCode.BAD_REQUEST
- 401 → ErrorCode.UNAUTHORIZED
- 403 → ErrorCode.FORBIDDEN
- 404 → ErrorCode.NOT_FOUND
- 409 → ErrorCode.CONFLICT
- 422 → ErrorCode.VALIDATION_ERROR
- 500 → ErrorCode.INTERNAL_ERROR"

Claude splits the 28 files across agents. Each agent reads its assigned files, applies the transformation, adds the import if needed, and maps the status codes. Within a few minutes you get a summary:

Batch complete:
  28 files modified
  73 error responses updated
  14 files needed ErrorCode import added
  0 conflicts
  All agents completed successfully

You then run the test suite once to verify everything works.

Manual Batch Pattern¶

If you’re not using /batch, you can achieve similar results with an explicit prompt:

Find all files matching src/routes/**/*.ts that contain 
res.status(4xx).json({ error: ... }).

For each file:
1. Replace the old error format with the new standardized format
2. Import the ErrorResponse type from src/types/errors.ts
3. Use the correct HTTP error code constant

Show me a summary when done: file name, number of changes per file.

Claude uses Grep to find all matching files, reads each one, applies the transformation with Edit, and reports the summary. This is sequential rather than parallel, so it’s slower on large codebases, but it doesn’t require the /batch infrastructure.

Strategy 4: Progressive Refactoring¶

For refactors that can’t be batched (each step depends on the previous), progressive refactoring breaks the work into safe stages where each stage leaves the code working.

When to Use Progressive Refactoring¶

Use this when: - The refactor has dependencies between steps - You want to commit at safe points - The full refactor is too large for a single session - You need to be able to stop and resume - Each intermediate state must be deployable

Example: Migrating from Callbacks to Async/Await¶

This migration can’t be batched because converting a function to async changes its return type, which affects all callers, which might also need to become async.

Step 1: "Add async versions of the 3 database functions in 
src/db/queries.ts. Name them getUser_async, getOrders_async, 
getProducts_async. Keep the old callback versions working."

Run tests. Both old and new versions work.

Step 2: "Update src/services/user-service.ts to use the async 
versions. Make the service methods async. Keep the old callback 
versions of the service methods as deprecated wrappers."

Run tests. User service works with async, other services still use callbacks.

Step 3: "Update src/services/order-service.ts to use async. 
Same pattern: async methods + deprecated callback wrappers."

Run tests.

Step 4: "Update all route handlers in src/routes/ to use the 
async service methods. Remove the callback wrappers."

Run tests.

Step 5: "Remove the old callback database functions. Remove all 
deprecated wrappers. Clean up any unused imports."

Run full test suite.

Each step is independently deployable. If step 3 reveals a problem, you can revert to step 2 without losing steps 1-2’s progress.

The Commit-After-Each-Step Pattern¶

Progressive refactoring pairs naturally with frequent commits:

After each step:
1. Run tests: pytest -x (or npm test)
2. If tests pass: git add the changed files, commit with 
   message describing what this step accomplished
3. If tests fail: fix the failure before proceeding

This way I can git revert any individual step if needed.

Specific Refactoring Patterns¶

Rename Refactoring (Comprehensive)¶

Renaming a type, class, or interface across an entire codebase is the bread-and-butter of multi-file refactoring. Your IDE can handle simple renames, but when the name appears in strings, config files, database queries, documentation, and test fixtures, you need Claude Code’s understanding of context.

Here’s a complete example: renaming UserManager to UserService across a 25-file codebase.

The prompt:

Rename UserManager to UserService everywhere in the codebase.

This includes:
- The class definition and its file name
- All imports (update the import path)
- All type annotations referencing UserManager
- All variable names (userManager → userService, user_manager → user_service)
- All comments and documentation
- All test files and test references
- Config files that reference the class name
- Database seed files that might reference it
- CI/CD pipeline configs

After renaming, grep the entire codebase for "UserManager" and 
"userManager" and "user_manager" to verify zero references remain.

What Claude does, step by step:

  1. Find all references — Claude runs three Grep searches:
Grep("UserManager") → finds class def, imports, type annotations
Grep("userManager") → finds variable names, object properties
Grep("user_manager") → finds snake_case references (config, Python)

This produces a complete list. For a real codebase, it might look like:

src/services/user-manager.ts         — class definition (1 reference)
src/services/index.ts                — barrel export (1 reference)
src/controllers/auth-controller.ts   — import + usage (3 references)
src/controllers/admin-controller.ts  — import + usage (2 references)
src/middleware/auth.ts               — import + usage (2 references)
src/routes/user-routes.ts            — import + usage (4 references)
src/routes/admin-routes.ts           — import + usage (2 references)
src/types/services.ts                — type reference (1 reference)
src/di/container.ts                  — DI registration (3 references)
src/config/services.yaml             — service name (1 reference)

Extract Module Pattern¶

Taking a monolithic file and splitting it into well-organized modules is one of the most common refactoring tasks. Here’s the complete pattern.

Starting point: src/utils/helpers.ts — a 1,200-line “junk drawer” that every file in the project imports from.

The prompt:

Read src/utils/helpers.ts completely. Categorize every exported 
function by responsibility:

- String manipulation (formatName, slugify, truncate, etc.)
- Date/time operations (formatDate, parseTimestamp, getRelativeTime, etc.)
- Validation (isEmail, isUUID, isPhoneNumber, etc.)
- HTTP utilities (parseQueryString, buildUrl, sanitizeHeaders, etc.)
- Crypto (hashPassword, generateToken, verifySignature, etc.)

Then extract each category into its own module:
- src/utils/strings.ts
- src/utils/dates.ts
- src/utils/validation.ts
- src/utils/http.ts
- src/utils/crypto.ts

Update src/utils/index.ts to re-export everything from the new 
modules (backward compatibility). Then find every file that imports 
from helpers.ts and update to import from the specific module instead.

Delete helpers.ts only after all imports are updated and tests pass.

Why backward compatibility matters: The re-export from index.ts means the code is never broken, even mid-refactor. Any file still importing from @/utils continues to work. You can update the imports progressively (or all at once), and the re-exports serve as a safety net until every import is updated.

Claude’s execution:

  1. Read the file, categorize 34 exported functions into 5 groups
  2. Create five new modules, moving functions and their dependencies
  3. Create src/utils/index.ts that re-exports everything:
typescript
// src/utils/index.ts — backward compatibility
export * from './strings';
export * from './dates';
export * from './validation';
export * from './http';
export * from './crypto';
  1. Run tests — all pass (nothing changed from the consumer’s perspective)
  2. Update imports in 42 files: from '@/utils/helpers' → from '@/utils/strings' (or dates, validation, etc., depending on what’s used)
  3. Run tests again — all pass
  4. Delete helpers.ts
  5. Final Grep for helpers to confirm no references remain

The key insight: The re-export barrel file makes this refactor safe. At no point is the code broken. Even if Claude misses an import update, the barrel file catches it. You can verify by deleting the barrel file at the end — if anything breaks, there’s still an import that needs updating.

Handling internal dependencies between functions:

The tricky part of module extraction is when functions depend on each other. If formatName in the strings module calls isValidCharset in the validation module, you’ve created a cross-module dependency that didn’t exist before. Claude handles this by:

  1. Identifying all internal function calls within helpers.ts before splitting
  2. Mapping which calls cross category boundaries
  3. For shared utilities called by multiple categories, creating a src/utils/internal.ts module that both depend on — avoiding circular imports
  4. For functions that belong to one category but are called by another, using the module’s public API (import from the other module) rather than duplicating code
Before extraction, show me a table of cross-category dependencies:
which functions in each category call functions in a different category.
If there are circular dependencies, identify shared utilities that 
should be extracted into a common/internal module first.

This pre-analysis prevents the most common module extraction failure: you split the file, discover circular imports, and have to redesign the module boundaries.

The Python equivalent:

For Python codebases, the same pattern applies with __init__.py as the barrel file:

python
# src/utils/__init__.py — backward compatibility

from .dates import format_date, parse_timestamp, get_relative_time
from .validation import is_email, is_uuid, is_phone_number
from .http import parse_query_string, build_url, sanitize_headers
from .crypto import hash_password, generate_token, verify_signature

Old imports (from src.utils import format_name) keep working. New imports (from src.utils.strings import format_name) are more specific and preferred for new code.

Database Migration Refactoring¶

Database schema changes ripple through every layer of the application. This is the hardest refactoring type because it affects migrations, ORM models, service layer, API types, and tests — all of which must be updated atomically from the database’s perspective but can be staged from the code’s perspective.

Scenario: Splitting a users table’s address text column into structured fields: street, city, state, zip, country.

The prompt:

I need to split the users.address text column into structured 
address fields: street, city, state, zip_code, country.

This affects:
1. Database migration (add new columns, backfill from address, 
   drop address column)
2. ORM model (src/models/user.ts)
3. Service layer (src/services/user-service.ts)
4. API request/response types (src/types/user.ts)
5. Route handlers (src/routes/users.ts)
6. Tests (tests/services/user.test.ts, tests/routes/users.test.ts)

Execute this in order. The migration should be safe for zero-downtime 
deployment:
- Phase 1: Add new columns (nullable), deploy code that writes to both
- Phase 2: Backfill new columns from address
- Phase 3: Make new columns required, remove reads from old column
- Phase 4: Drop address column

Implement Phase 1 and Phase 2 now. We'll do Phase 3 and 4 in a 
follow-up.

Phase 1 execution:

Claude creates the migration:

typescript
// migrations/20260401_split_user_address.ts
import { Knex } from 'knex';

export async function up(knex: Knex): Promise<void> {
  await knex.schema.alterTable('users', (table) => {
    table.string('street', 255).nullable();
    table.string('city', 100).nullable();
    table.string('state', 50).nullable();
    table.string('zip_code', 20).nullable();
    table.string('country', 100).nullable().defaultTo('US');
  });
}

export async function down(knex: Knex): Promise<void> {
  await knex.schema.alterTable('users', (table) => {
    table.dropColumn('street');
    table.dropColumn('city');
    table.dropColumn('state');
    table.dropColumn('zip_code');
    table.dropColumn('country');
  });
}

Updates the ORM model to include both old and new fields:

typescript
// src/models/user.ts
interface User {
  id: string;
  name: string;
  email: string;
  address?: string;        // deprecated — will be removed in Phase 4
  street?: string;         // new
  city?: string;           // new
  state?: string;          // new
  zip_code?: string;       // new
  country?: string;        // new
}

Updates the service layer to write to both old and new fields:

typescript
// src/services/user-service.ts
async function updateUserAddress(
  userId: string,
  address: StructuredAddress
): Promise<void> {
  await db('users').where({ id: userId }).update({
    // Write to new columns
    street: address.street,
    city: address.city,
    state: address.state,
    zip_code: address.zipCode,
    country: address.country,
    // Write to old column for backward compatibility
    address: `${address.street}, ${address.city}, ${address.state} ${address.zipCode}, ${address.country}`,
  });
}

Phase 2 execution:

Claude creates a backfill migration that parses the existing address text field and populates the structured columns:

typescript
// migrations/20260401_backfill_user_address.ts
import { Knex } from 'knex';

export async function up(knex: Knex): Promise<void> {
  const users = await knex('users')
    .whereNotNull('address')
    .whereNull('street');

  for (const user of users) {
    const parsed = parseAddress(user.address);
    await knex('users').where({ id: user.id }).update({
      street: parsed.street,
      city: parsed.city,
      state: parsed.state,
      zip_code: parsed.zipCode,
      country: parsed.country || 'US',
    });
  }
}

function parseAddress(address: string): StructuredAddress {
  // Best-effort parsing of "123 Main St, Springfield, IL 62701, US"
  const parts = address.split(',').map((p) => p.trim());
  const stateZip = (parts[2] || '').split(' ').filter(Boolean);
  return {
    street: parts[0] || '',
    city: parts[1] || '',
    state: stateZip[0] || '',
    zipCode: stateZip[1] || '',
    country: parts[3] || 'US',
  };
}

After both phases, tests pass. The old address column still works (nothing reads exclusively from it), the new columns are populated, and the code writes to both. Phases 3 and 4 can be deployed separately once you’re confident the backfill is correct.

Why this pattern matters: Database migrations are irreversible in production. The phased approach means you can deploy Phase 1, monitor for issues, deploy Phase 2, verify data, and only then proceed to Phase 3 (removing the old column). At every phase, the application works correctly.

Maintaining Consistency¶

Partial refactors are worse than no refactor at all. A codebase where half the error responses use the old format and half use the new format is a codebase where nobody knows which format to follow. These techniques ensure your refactor is complete.

Atomic Commits Per Refactoring Step¶

Each commit should represent a complete logical transformation. Not “updated 3 of 7 files” but “migrated all route handlers to new error format.”

After each refactoring step, before committing:
1. Grep for the OLD pattern — should return zero results
2. Grep for the NEW pattern — should return the expected count
3. Run the test suite
4. Only commit if all three checks pass

If the old pattern still exists in some files, update those files 
before committing. A partial migration should never be committed.

The Completeness Check¶

After any rename or pattern change, always verify completeness:

Verify the refactoring is complete:

1. Grep for the old pattern in ALL file types (not just .ts):
   - Source files (*.ts, *.tsx, *.js)
   - Config files (*.json, *.yaml, *.yml, *.toml)
   - Test files (*.test.ts, *.spec.ts)
   - Documentation (*.md)
   - CI/CD configs (.github/*, Dockerfile, docker-compose.yml)
   - Scripts (*.sh)

2. If any references remain, update them.
3. Show me the final count: zero old references, N new references.

The Barrel File Strategy¶

When splitting a module, create a barrel file (index.ts or init.py) that re-exports everything from the new locations. This provides backward compatibility during the transition:

typescript
// src/utils/index.ts — temporary barrel for backward compatibility
export { formatName, slugify, truncate } from './strings';
export { formatDate, parseTimestamp } from './dates';
export { isEmail, isUUID } from './validation';

Old imports (from '@/utils') keep working. New code uses specific imports (from '@/utils/strings'). Over time, update all old imports to specific ones. When the barrel file has zero importers, delete it.

Lint Rules as Guardrails¶

For long-running refactors that span multiple sessions, add a lint rule to prevent regression:

Add an ESLint rule (or a comment in CLAUDE.md) that prevents 
new code from using the old pattern. For example, after renaming 
UserManager to UserService:

"Do not use UserManager anywhere in new code. 
The class was renamed to UserService."

This prevents other developers (or future Claude Code sessions) from accidentally reintroducing the old pattern.

Rollback Strategies¶

Refactoring goes wrong. A test you didn’t know about breaks. A runtime behavior changes that unit tests don’t cover. A configuration file was missed. Here’s how to recover.

Git Reset to Last Safe Commit¶

If you committed after each step (as recommended), rolling back is trivial:

bash
# See the refactoring commits
git log --oneline -10

# Revert the last step
git revert HEAD

# Or revert to a specific safe point
git revert HEAD~3..HEAD

The key is that each commit is a complete, working state. You’re not reverting to a half-broken intermediate state.

Git Stash for Quick Experiments¶

When you want to try something without committing:

Stash the current changes before trying an alternative approach.
If the alternative works, drop the stash. If not, pop the stash 
to restore the original approach.
bash
git stash push -m "refactor: approach A (WIP)"
# Try approach B...
# If B fails:
git stash pop
# If B works:
git stash drop

Branch Comparison¶

When you’re not sure if the refactor changed behavior:

Compare the behavior of the refactored code against the original.

1. Run the test suite on the current branch (refactored)
2. Show me any test output differences vs. main branch
3. If there are differences, show me the git diff for the 
   files that produce different behavior

Cherry-Pick Recovery¶

When a multi-step refactor partially fails and you want to keep the good steps:

bash
# You're on a branch with steps 1-5
# Steps 1-3 are good, step 4 broke things

# Create a new branch from before the refactor
git checkout main
git checkout -b refactor-take-2

# Cherry-pick only the good steps
git cherry-pick <commit-hash-step-1>
git cherry-pick <commit-hash-step-2>
git cherry-pick <commit-hash-step-3>

# Now redo step 4 with a different approach

The /rewind Escape Hatch¶

If a refactoring step goes sideways mid-execution — Claude is making changes but hasn’t committed and things are breaking:

/rewind

This rolls the conversation back to a previous point. Combined with git checkout . to restore files, you can revert both the conversation state and the file state to any safe checkpoint.

Important: /rewind resets the conversation, not the filesystem. You need to restore files separately:

bash
# Restore all modified files to their last committed state
git checkout .

# Or restore specific files
git checkout -- src/services/order-service.ts

The Diff-Before-Commit Gate¶

Never commit blindly after a refactoring step. Always review what changed:

Before committing, show me:
1. git diff --stat (which files changed and how much)
2. git diff --name-only (just file names, cleaner view)
3. If any UNEXPECTED file appears, show me the diff for 
   that file specifically

I only want to commit files that are part of this refactoring step.

This catches the common issue of Claude making “helpful” changes to files outside the refactoring scope. If you asked Claude to extract a PaymentService and it also “improved” the error handling in three unrelated files, you want to know before committing.

Stage only the files from this refactoring step, not everything:

bash
# Stage only the files from this refactoring step
git add src/services/payment-service.ts src/services/order-service.ts
# NOT git add -A (which would stage everything)

git diff --cached --stat  # Review what you're about to commit

Test Selection for Speed¶

Running the full test suite after every refactoring step is correct but slow. On a large codebase with 2,000 tests, the full suite might take 5 minutes. When you’re doing a 6-step refactor, that’s 30 minutes of waiting.

The faster approach:

After this step, run only the tests related to the files we changed:
- pytest tests/services/test_payment.py -x (stop on first failure)
- pytest tests/integration/test_order_flow.py -x

Save the full suite for after the final step.

The -x flag (pytest) or --bail flag (Jest) stops on the first failure, giving you immediate feedback rather than running 2,000 tests when test #3 already failed.

Large Codebase Techniques¶

Everything above works for codebases under a few hundred files. When you’re working with 1,000+ files, you hit Claude Code’s context window limits and need different strategies.

Context Window Management¶

Claude Code can hold roughly 100-200 files in context before performance degrades. For larger codebases, you need to be strategic about what Claude reads.

The scoped exploration pattern:

I need to rename the ApiResponse type to ServiceResponse across 
the entire codebase. But the codebase has 2,000+ files.

Step 1: Grep for "ApiResponse" and show me ONLY the file paths 
and match counts. Don't read the files yet.

Step 2: Group the files by directory. Show me which directories 
have the most references.

Step 3: Process one directory at a time, starting with 
src/types/ (where the type is defined). After each directory, 
run the type checker to catch any issues.

This approach never loads all 2,000 files into context. Claude processes them in directory-sized batches, keeping the context window focused on the current batch.

The Directory-at-a-Time Pattern¶

For very large refactors, process one directory at a time and commit after each:

Process the rename in this order:
1. src/types/ — rename the type definition
2. src/services/ — update service layer
3. src/controllers/ — update controllers
4. src/routes/ — update routes
5. src/middleware/ — update middleware
6. tests/ — update all tests

After each directory:
- Run tsc --noEmit to check types
- Commit with message "refactor: rename ApiResponse → ServiceResponse in [directory]"
- Use /compact to free context before the next directory

The /compact command between directories is critical. It compresses the conversation history, freeing context window space for the next batch of files. Without it, by directory 4 or 5, Claude’s context is full of previous directories’ content and performance drops.

Chunking with File Lists¶

When a directory itself has too many files:

Here are the first 20 files that need updating:
1. src/api/routes/users.ts
2. src/api/routes/orders.ts
...
20. src/api/routes/webhooks.ts

Update these 20 files. When done, I'll give you the next batch.

This manual chunking ensures Claude never tries to hold more files than it can handle. Each batch is small enough for precise edits and large enough to be efficient.

The Grep-First, Read-Second Pattern¶

Never ask Claude to “read all files matching X” in a large codebase. Instead:

1. Grep for "oldFunction" across the entire codebase — show me 
   only file paths (not content)
2. For each file, show me only the matching lines (not the full file)
3. Based on the matches, I'll tell you which files to read fully 
   and which can be updated with a simple replace_all

Most files in a large rename need only a mechanical replace_all. Only files where the change is more nuanced (e.g., the function definition itself, complex call sites) need a full read.

Pre-Flight Validation¶

Before starting a large refactor, validate that your tools will work:

Before we start the refactor, verify:
1. The test suite passes: npm test (or pytest)
2. The type checker passes: tsc --noEmit (or mypy)
3. The linter passes: eslint . (or ruff check)
4. Git status is clean: no uncommitted changes

If any of these fail, fix them first. I don't want to debug 
pre-existing failures mixed with refactoring failures.

This is the equivalent of a surgeon checking the operating room before the procedure. A clean baseline means every failure during refactoring is caused by the refactoring.

The Type Checker as a Refactoring Safety Net¶

In TypeScript and Python (with mypy/pyright) codebases, the type checker is your most valuable refactoring tool. It catches errors that tests miss — particularly missed call sites and incorrect parameter types.

The workflow:

After each refactoring step, run BOTH:
1. tsc --noEmit (type check without building)
2. npm test (runtime verification)

If the type checker finds errors that tests don't, that tells you 
the test coverage has gaps. Note those gaps for later.

Type checkers catch errors statically, which means they find every call site that’s wrong, not just the ones that happen to be covered by tests. For a function signature change, the type checker is more thorough than the test suite because it checks every reference in the code, not just the paths that tests exercise.

Python example with mypy:

After renaming the parameter, run:
  mypy src/ --strict --no-error-summary

Show me any type errors. Each one is a missed call site that 
needs updating.

Session Spanning for Multi-Day Refactors¶

Some refactors are too large for a single session. When you need to spread work across multiple days:

Before ending this session, create a checkpoint:
1. Commit all completed steps with descriptive messages
2. Create a file called REFACTORING_STATUS.md in the repo root:
   - Which steps are complete
   - Which steps remain
   - Any issues discovered
   - The exact prompt to resume with
3. Push the branch so it's backed up

Next session, I'll say "resume the refactoring" and you should 
read REFACTORING_STATUS.md first.

The status file acts as a handoff document between sessions. Claude doesn’t remember previous sessions (unless using auto-memory), so this explicit artifact ensures continuity.

What Most People Get Wrong¶

1. Trying to Refactor and Add Features Simultaneously¶

The most common mistake. You’re refactoring the payment module and think “while I’m in here, let me add support for Apple Pay.” Now you can’t tell if a test failure is from the refactor or the new feature. Refactoring is behavior-preserving by definition. If you’re changing behavior, that’s a separate commit.

2. Skipping the Test Baseline¶

If you don’t know which tests pass before refactoring, you can’t tell which tests you broke. Run the full suite first. Record the result. Any new failures after refactoring are yours.

3. Making the Refactor Too Large¶

A refactor that touches 50 files in one commit is unreviewable. Break it into steps. Each step should be reviewable in 5 minutes, not 50. As discussed in Chapter 22’s checkpoint pattern, the cost of running tests between steps is trivial compared to the cost of debugging a 50-file change.

4. Not Verifying String References¶

Code refactoring tools (and Claude Code) are excellent at finding code references. They’re less reliable with string references — class names in config files, table names in SQL strings, path references in documentation. Always run a project-wide grep for the old name after the refactor, including non-code files.

5. Forgetting About CI/CD¶

Your GitHub Actions workflow references UserManager in a test step. Your Dockerfile copies user-manager.ts by name. Your docker-compose mounts a volume to the old path. These break silently — the code works locally but fails in CI.

After the refactor, also check:
- .github/workflows/*.yml
- Dockerfile and docker-compose.yml
- Makefile
- package.json scripts
- Any deployment scripts in scripts/ or tools/

6. Not Using /compact Between Refactoring Steps¶

By step 4 of a 6-step refactor, Claude’s context window is full of the code from steps 1-3. It starts making mistakes — missing imports, using old names, forgetting the plan. Use /compact between major steps to free context.

/compact refactoring plan and current step

This keeps the plan and current progress in context while compressing the details of completed steps.

7. Relying on “Claude Will Figure It Out”¶

Claude Code is powerful, but it’s not magic. A vague prompt like “refactor the backend” produces a vague result. The specific strategies in this chapter — plan mode, worktrees, batch, progressive — exist because different refactoring shapes need different approaches. Use the right tool for the job.

What Made This Work¶

  1. Plan mode mapped dependencies before any changes
  2. Extraction order went from least to most dependent (audit → notifications → inventory → payments → controller)
  3. Tests ran after every step — the one failure in step 4 was caught and fixed immediately
  4. Commits at each safe point — any step could be independently reverted
  5. Each step was small enough to reason about but large enough to be meaningful
  6. The controller was reduced last — it depended on everything else being in place

When Not to Refactor with Claude Code¶

Some refactors are better done manually or with traditional tooling:

  • Simple renames in one language — Your IDE’s rename refactoring (F2 in VS Code) is faster and more reliable for a single rename within one language. Use Claude Code when the rename spans config files, documentation, CI/CD, and multiple languages.
  • Formatting-only changes — Run your formatter directly (ruff format, prettier). Claude Code adds nothing here.
  • Import sorting — Use isort or your language’s equivalent. Deterministic tools are better for deterministic problems.
  • Dead code removal — Use coverage tools to identify dead code, then remove manually. Claude Code might misidentify code as dead when it’s used by dynamic imports or reflection.
  • Trivial changes that don’t require understanding — If the transformation can be expressed as a regex find-and-replace, use sed or your editor’s multi-cursor. Claude Code’s value is in changes that require understanding what the code does.

Claude Code shines when refactoring requires understanding — when the transformation depends on knowing what the code does, not just what it looks like.

Chapter Summary¶

Multi-file refactoring at scale is where Claude Code delivers its highest ROI. The strategies in this chapter turn a dreaded, error-prone task into a systematic, verifiable process.

Five key takeaways:

  1. Choose the right strategy for the refactoring shape. Plan mode for complex decompositions. Worktrees for risky experiments. /batch for mechanical changes across many files. Progressive refactoring for dependent, ordered steps. Using the wrong strategy makes easy refactors hard and hard refactors impossible.

  2. Plan before you execute. Plan mode’s read-only constraint forces Claude to understand the dependency graph, identify risks (like that webhook handler calling processRefund directly), and design an extraction order. Skipping the plan is how you end up with circular dependencies at step 4 and have to start over.

  3. Commit at every safe checkpoint. Each commit should be a complete, working state that could be deployed independently. “refactor: extract PaymentService (step 3/5)” is reviewable, revertible, and meaningful. A single 50-file commit is none of those things.

  4. Verify completeness, not just correctness. Tests passing means the code works. Grep returning zero results for the old pattern means the refactor is complete. Both are required. A partial refactor where half the code uses the old pattern and half uses the new pattern is worse than no refactor at all.

  5. Manage the context window on large codebases. Process directories one at a time, use /compact between steps, grep for file paths before reading file contents, and chunk large batches. Claude Code’s accuracy degrades when the context window is full — keeping it focused is your responsibility.
