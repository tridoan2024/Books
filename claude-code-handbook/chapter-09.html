Part III: Tool System Mastery

Chapter 9: The Complete Tool System¶ ~19 min read

You open Claude Code, type “fix the authentication bug in src/auth/login.ts,” and watch it work. It reads the file. Searches for related test files. Edits the broken logic. Runs the tests. Commits the fix. Every one of those actions is a tool call – Claude Code’s mechanism for interacting with the world beyond generating text.

Understanding the tool system is not optional. It is the difference between a user who watches Claude Code fumble through tasks and one who configures it to operate with surgical precision. This chapter is the complete reference for every tool Claude Code has, how it selects between them, and the execution pipeline that governs every tool invocation.

How Tools Work¶

Claude Code operates in an agentic loop: the LLM generates a response, which may include one or more tool calls. The system executes those calls, returns the results, and the LLM generates another response based on the new information. This loop continues until the task is complete.

Tools are Claude Code’s hands. Without them, it can only produce text. With them, it can read your codebase, modify files, run tests, search the web, and spawn child agents to parallelize work.

Every tool follows the same contract:

  1. The LLM outputs a tool_use block specifying the tool name and parameters
  2. The permission system checks whether the call is allowed
  3. PreToolUse hooks fire (can block the call)
  4. The tool executes
  5. PostToolUse hooks fire (can modify results or trigger side effects)
  6. The result is truncated if necessary and returned to the LLM as a ToolResult
LLM returns tool_use → Permission check → PreToolUse hooks →
Execute tool → PostToolUse hooks → Truncate result → Return to LLM

This pipeline applies to every single tool call. No exceptions. Understanding it means understanding exactly where your hooks, permissions, and configurations intercept.

The Complete Tool Inventory¶

Claude Code ships with a rich set of built-in tools. Some you will use constantly. Others exist for specific scenarios. Here is every tool, what it does, and when to reach for it.

1. Read – Read Files from the Filesystem¶

Read loads file contents into Claude Code’s context. It is the most frequently called tool and the foundation of all code understanding.

Key parameters:

Parameter Type Description
file_path string (required) Absolute path to the file
offset integer Line number to start reading from
limit integer Number of lines to read
pages string Page range for PDFs (e.g., “1-5”)

What it reads: - Source code files (any language, any extension) - Images (PNG, JPG, GIF – Claude is multimodal and sees the image visually) - PDF files (requires pages parameter for documents over 10 pages) - Jupyter notebooks (.ipynb – returns all cells with their outputs) - Configuration files, logs, data files – anything on the filesystem

Output format: Line-numbered text using cat -n format, with line numbers starting at 1. This is important because Edit operations reference these line numbers for context.

When to use: Before every Edit. Before analyzing code. Before understanding project structure. When the user references a file.

When NOT to use: To read directories (use ls via Bash). To find files by name (use Glob). To search file contents (use Grep).

Example invocation:

Read(file_path: "/Users/dev/project/src/auth/login.ts")

For a large file where you only need a specific section:

Read(file_path: "/Users/dev/project/src/auth/login.ts", offset: 150, limit: 50)

For a PDF:

Read(file_path: "/Users/dev/docs/spec.pdf", pages: "1-5")

Common mistakes: - Using relative paths. Read requires absolute paths – always. - Reading a directory path instead of a file path. Read only handles files. - Trying to read a large PDF without the pages parameter. It will fail for PDFs over 10 pages. - Forgetting to read before editing. The Edit tool will refuse to operate on a file that has not been read in the current session.

2. Edit – Exact String Replacement¶

Edit modifies existing files by replacing exact strings. It is the primary tool for code changes, and it operates on diffs rather than rewriting entire files.

Key parameters:

Parameter Type Description
file_path string (required) Absolute path to the file
old_string string (required) The exact text to find
new_string string (required) The replacement text
replace_all boolean Replace all occurrences (default: false)

How it works: Edit finds old_string in the file and replaces it with new_string. The match must be exact – same characters, same indentation, same whitespace. If old_string appears more than once and replace_all is false, the edit fails.

When to use: Any time you need to modify an existing file. Changing function logic, updating configuration values, fixing bugs, adding imports.

When NOT to use: Creating new files (use Write). Complete file rewrites (use Write). When you have not read the file first (it will error).

Example – fixing a bug:

Edit(
  file_path: "/Users/dev/project/src/auth/login.ts",
  old_string: "if (password === storedHash) {",
  new_string: "if (await bcrypt.compare(password, storedHash)) {"
)

Example – renaming a variable across a file:

Edit(
  file_path: "/Users/dev/project/src/utils.ts",
  old_string: "userName",
  new_string: "userId",
  replace_all: true
)

Common mistakes: - Not reading the file first. Edit enforces a “read before write” invariant and will error if you skip it. - Indentation mismatch. The old_string must preserve the exact indentation from the file. When you read a file, the output shows line_number\tfile_content – the indentation is everything after the tab following the line number. - Non-unique old_string. If the string appears multiple times and you want to change just one, include more surrounding context to make it unique. - Using Edit when Write is more appropriate. If you are replacing more than 60-70% of a file, Write is cleaner.

3. Write – Create or Rewrite Files¶

Write creates new files or completely overwrites existing ones.

Key parameters:

Parameter Type Description
file_path string (required) Absolute path to the file
content string (required) The complete file content

When to use: Creating new files. Complete rewrites where Edit would require too many individual operations. Generated files (configs, boilerplate).

When NOT to use: Modifying a few lines of an existing file (use Edit – it only transmits the diff). Creating documentation files unless explicitly asked. Overwriting a file you have not read first.

Example – creating a new test file:

Write(
  file_path: "/Users/dev/project/tests/test_login.py",
  content: "import pytest\nfrom src.auth.login import authenticate\n\n\ndef test_valid_credentials():\n    result = authenticate('user@test.com', 'correct-password')\n    assert result.success is True\n"
)

Common mistakes: - Using Write to make a small change to a large file. This sends the entire file content, wasting tokens and risking accidental changes. Use Edit instead. - Not reading an existing file before overwriting it. Write will error if the file exists and you haven’t read it.

4. Glob – Find Files by Pattern¶

Glob finds files matching a pattern. It is fast at any codebase size and returns results sorted by modification time.

Key parameters:

Parameter Type Description
pattern string (required) Glob pattern (e.g., **/*.ts)
path string Directory to search in (defaults to cwd)

Pattern syntax: - * – matches any characters within a single directory level - ** – matches any characters across directory levels (recursive) - ? – matches a single character - {a,b} – matches either a or b

When to use: Finding files by name or extension. Discovering project structure. Locating test files, configs, components.

When NOT to use: Searching file contents (use Grep). Listing directory contents (Glob works, but ls via Bash is simpler for a single directory).

Example – find all TypeScript files:

Glob(pattern: "**/*.ts", path: "/Users/dev/project/src")

Example – find test files for a specific module:

Glob(pattern: "**/auth/**/*.test.{ts,tsx}")

Common mistakes: - Using find in Bash when Glob exists. Glob is faster and its output is optimized for Claude Code’s context. - Forgetting ** for recursive searches. *.ts only matches the current directory; **/*.ts searches all subdirectories.

5. Grep – Search File Contents¶

Grep searches file contents using regular expressions. It is built on ripgrep, which means it is fast, Unicode-aware, and respects .gitignore by default.

Key parameters:

Parameter Type Description
pattern string (required) Regex pattern to search for
path string File or directory to search
output_mode string files_with_matches (default), content, count
glob string Filter by file pattern (e.g., *.py)
type string Filter by file type (e.g., py, ts)
-i boolean Case-insensitive search
-A / -B / -C number Lines after/before/context
multiline boolean Match patterns across line boundaries
head_limit number Max results (default: 250)
offset number Skip first N results

Three output modes:

Mode Returns Use when
files_with_matches File paths only Finding which files contain a pattern
content Matching lines with context Reading actual matches
count Match counts per file Measuring frequency

When to use: Finding function definitions, imports, usages of a variable, TODO comments, error messages, configuration values.

When NOT to use: Finding files by name (use Glob). Reading a file you already know the path to (use Read).

Example – find all usages of a function:

Grep(
  pattern: "authenticateUser\\(",
  type: "ts",
  output_mode: "content",
  -C: 2
)

Example – find files containing a specific import:

Grep(
  pattern: "from.*auth.*import",
  glob: "*.py",
  output_mode: "files_with_matches"
)

Common mistakes: - Using grep or rg in Bash. The Grep tool is optimized for Claude Code’s permission model and output formatting. - Not setting head_limit on broad searches. The default of 250 is usually fine, but a regex like .* on a large codebase will flood the context window. - Forgetting to escape special regex characters. Literal braces need escaping (interface\\{\\} to find interface{}), and parentheses need \\( in patterns.

6. Bash – Execute Shell Commands¶

Bash runs arbitrary shell commands. It is the most powerful tool and the one with the most security implications. It gets its own chapter (Chapter 10), but here is the overview.

Key parameters:

Parameter Type Description
command string (required) Shell command to execute
description string Human-readable description of the command
timeout number Timeout in ms (max 600,000 – 10 min)
run_in_background boolean Run without blocking

When to use: Running tests, git operations, build commands, package management, any command-line operation that does not have a dedicated tool.

When NOT to use: Reading files (use Read). Searching files (use Grep/Glob). Editing files (use Edit). These dedicated tools are faster and produce better-formatted output.

Example:

Bash(command: "cd /Users/dev/project && python -m pytest tests/test_auth.py -v", description: "Run auth tests")

7. Agent – Spawn Subagents¶

Agent creates a child Claude instance for complex or parallel tasks. It protects your main context window from exploration noise and enables parallel execution.

Key parameters:

Parameter Type Description
prompt string (required) Task description for the agent
subagent_type string Agent type: general-purpose, Explore, Plan
model string Override model (sonnet, opus, haiku)
run_in_background boolean Non-blocking execution
isolation string “worktree” for git isolation

When to use: Multi-step investigations, parallel independent tasks, work that would pollute main context. See Chapter 12 for full coverage.

When NOT to use: Simple reads, searches, or operations involving 2-3 files.

8. WebFetch – Fetch Web Content¶

WebFetch retrieves content from a URL and processes it with a small, fast model.

Key parameters:

Parameter Type Description
url string (required) Fully-formed URL to fetch
prompt string (required) What to extract from the page

What it does: Fetches the URL, converts HTML to markdown, processes with a small model using your prompt, and returns the extracted information. Includes a 15-minute cache.

When to use: Reading documentation, extracting data from public web pages, checking API references.

When NOT to use: Authenticated pages (Google Docs, Jira, private GitHub repos). For GitHub, use gh CLI. For authenticated services, use MCP tools that provide authenticated access.

Example:

WebFetch(
  url: "https://docs.python.org/3/library/pathlib.html",
  prompt: "List all the methods on the Path class with brief descriptions"
)

Common mistakes: - Fetching authenticated URLs. WebFetch cannot handle login-protected pages. It will fail silently or return a login page. - Fetching GitHub URLs when gh CLI is available. gh api, gh pr view, gh issue view are faster and more reliable.

9. WebSearch – Search the Web¶

WebSearch performs a web search and returns results with links.

Key parameters:

Parameter Type Description
query string (required) Search query
allowed_domains array Only include results from these domains
blocked_domains array Exclude results from these domains

When to use: Researching current topics, finding documentation, checking for recent updates.

When NOT to use: Searching your local codebase (use Grep).

Example:

WebSearch(query: "Python 3.13 new features 2025", allowed_domains: ["docs.python.org", "realpython.com"])

10. NotebookEdit – Edit Jupyter Notebooks¶

NotebookEdit modifies individual cells in .ipynb files.

Key parameters:

Parameter Type Description
notebook_path string (required) Absolute path to the notebook
new_source string (required) New cell content
cell_id string Cell ID to target
cell_type string code or markdown
edit_mode string replace, insert, or delete

When to use: Modifying data science notebooks, adding analysis cells, fixing notebook code.

When NOT to use: Creating a notebook from scratch (use Write to create the .ipynb JSON structure).

11. AskUserQuestion – Request Clarification¶

AskUserQuestion pauses execution and asks the user for input. Claude Code uses this when it genuinely cannot proceed without information.

When it fires: When the task is ambiguous and Claude Code cannot make a reasonable assumption. For example, “fix it” with no error context and multiple recently changed files.

Design principle: A well-configured Claude Code instance should rarely need this. The auto-execute protocol prioritizes making reasonable assumptions and proceeding. AskUserQuestion is the escape valve, not the default.

12. EnterPlanMode – Switch to Planning¶

EnterPlanMode transitions Claude Code from execution mode to planning mode. In planning mode, Claude Code analyzes and strategizes without making changes.

When to use: Complex multi-file refactors that need a plan before execution. Architecture decisions requiring analysis. When the user says /plan.

When it matters: Planning mode prevents accidental modifications while Claude Code is still figuring out the right approach.

13. Task Management – TaskCreate, TaskList, TaskUpdate, TaskGet¶

These four tools manage a task system for tracking multi-step work.

Tool Purpose
TaskCreate Create a new task with description and steps
TaskList List all active tasks
TaskUpdate Update task status, add subtasks
TaskGet Get details on a specific task

When to use: Complex projects with multiple phases. When the user says /plan and the work spans many files. Tracking progress on long-running tasks.

14. EnterWorktree – Git Isolation¶

EnterWorktree creates an isolated git worktree for experimental work.

What it does: Creates a new branch and working copy inside .claude/worktrees/. The agent operates on this copy without affecting the main working directory.

When to use: Risky refactoring experiments. Testing approaches that might need to be discarded. Parallel development of independent features.

Related: ExitWorktree leaves the worktree, either keeping it (for later) or removing it (clean exit).

15. Skill – Invoke a Skill¶

Skill loads and executes a skill defined in .claude/skills/.

Key parameters:

Parameter Type Description
skill string (required) Skill name (e.g., “code-review”)
args string Optional arguments

When to use: When the user types a slash command like /code-review. When Claude Code detects an intent matching a skill’s description.

Example:

Skill(skill: "fix-issue", args: "42")

16. Cron Management – CronCreate, CronDelete, CronList¶

These tools manage scheduled recurring tasks.

Tool Purpose
CronCreate Schedule a recurring task with interval and prompt
CronDelete Remove a scheduled task
CronList List all active cron tasks

When to use: Monitoring deployments, polling for CI status, recurring checks. When the user says /loop.

Tool Selection Strategy¶

Claude Code does not randomly pick tools. It follows a deliberate selection hierarchy that prioritizes specialized tools over general-purpose ones. Understanding this hierarchy helps you predict what Claude Code will do and configure permissions accordingly.

The Hierarchy¶

Rule 1: Read before Edit. Always.

Every Edit call must be preceded by a Read of the target file in the same session. This is not a suggestion – the Edit tool enforces it and will error if you skip it. This prevents blind edits based on stale assumptions.

Rule 2: Specialized tools over Bash.

Instead of Use
cat file.py in Bash Read(file_path: "file.py")
grep -r "pattern" . in Bash Grep(pattern: "pattern")
find . -name "*.ts" in Bash Glob(pattern: "**/*.ts")
sed -i 's/old/new/' in Bash Edit(old_string: "old", new_string: "new")
echo "content" > file.py in Bash Write(file_path: "file.py", content: "content")

Why? Specialized tools produce output formatted for Claude Code’s context window. They respect permissions more granularly. They have better error handling. And they are faster because they skip shell initialization.

Rule 3: Glob for names, Grep for contents.

Looking for files named *.test.ts? Glob. Looking for files containing describe("Auth? Grep. Different tools for different searches.

Rule 4: Edit for modifications, Write for creation.

Edit sends only the diff. Write sends the entire file. For a one-line change in a 500-line file, Edit uses a fraction of the tokens.

Rule 5: Agent for complexity, direct tools for simplicity.

If the task requires reading 2 files and making an edit, do it directly. If it requires searching through 20 files to understand a pattern before making changes, spawn an Agent to do the exploration.

How Claude Code Decides¶

When Claude Code needs to perform an action, its decision tree looks roughly like this:

Need to access file content?
├── Know the path? → Read
├── Know the name pattern? → Glob → then Read
└── Know content pattern? → Grep → then Read

Need to modify a file?
├── File exists? → Read → Edit
├── New file? → Write
└── Many changes across file? → Read → Write (full rewrite)

Need to run a command?
├── Is there a dedicated tool? → Use it
└── No dedicated tool? → Bash

Need to search broadly?
├── File names? → Glob
├── File contents? → Grep
└── Multi-round search? → Agent with Explore type

Need external information?
├── Public web page? → WebFetch
├── General search? → WebSearch
├── GitHub data? → Bash(gh ...)
└── Authenticated service? → MCP tool

The Tool Execution Pipeline¶

Every tool call passes through the same pipeline. Understanding each stage lets you configure Claude Code precisely.

Stage 1: Permission Check¶

The permission system evaluates the call against your settings:

json
{
  "permissions": {
    "deny": ["Read(./.env)"],
    "allow": ["Bash(git status *)"],
    "ask": []
  }
}

Evaluation order: deny first, then allow, then ask. First match wins. If nothing matches, Claude Code prompts you for permission.

A denied tool call never reaches the hooks. It is stopped immediately, and Claude Code receives an error message explaining the denial.

Stage 2: PreToolUse Hooks¶

If the permission check passes, PreToolUse hooks fire. These hooks receive a JSON payload on stdin describing the tool call:

json
{
  "tool_name": "Bash",
  "tool_input": {
    "command": "rm -rf /tmp/old-cache"
  }
}

A hook that exits with code 2 blocks the call. The hook’s stderr becomes feedback that Claude Code sees. This is how the block-destructive.sh hook prevents dangerous commands – it pattern-matches the command and blocks known-dangerous patterns.

Stage 3: Execution¶

The tool runs. For Bash, this means spawning a shell process. For Read, it means reading the filesystem. For Agent, it means starting a new Claude session. Each tool has its own timeout – Bash defaults to 120 seconds but can be set up to 600 seconds.

Stage 4: PostToolUse Hooks¶

After successful execution, PostToolUse hooks fire. They receive the tool name, input, and output. They cannot block (the action already happened), but they can trigger side effects.

The canonical example is auto-lint-python.sh: after any Edit or Write to a .py file, it runs ruff check --fix and ruff format automatically.

Stage 5: Result Truncation and Return¶

Tool results can be large. A Grep across a big codebase might return thousands of lines. Claude Code truncates results that exceed token limits, keeping the beginning and end to preserve context.

The result goes back to the LLM as a ToolResult containing: - Text content: the tool’s output - Error flag: whether the tool errored - Truncation indicator: whether the output was truncated

ToolResult Format¶

Every tool returns a structured result. When a tool succeeds, Claude Code sees the output text. When it fails, Claude Code sees the error message and adjusts its approach.

Successful Read:
  content: "1\timport express from 'express';\n2\t..."
  is_error: false

Failed Edit (non-unique match):
  content: "Error: old_string appears 3 times in the file. Include more context to make it unique, or use replace_all."
  is_error: true

Truncated Grep:
  content: "[first 200 matches...]\n\n--- output truncated (showing 200 of 1,847 matches) ---"
  is_error: false

How Claude Code handles errors: When a tool returns is_error: true, Claude Code reads the error message, understands what went wrong, and tries a different approach. This is one of the key behaviors of the agentic loop – failure is information, not a dead end. If an Edit fails because the old_string is not unique, Claude Code will read the file again, include more surrounding context, and retry.

Parallel Tool Execution¶

Claude Code can call multiple tools in a single response. This is a critical optimization – it reduces round trips and speeds up complex operations.

When Claude Code parallelizes: - Reading multiple independent files at once - Running Grep and Glob simultaneously for different searches - Spawning multiple Agent instances for independent subtasks

When Claude Code serializes: - Read then Edit (Edit depends on Read output) - Glob then Read (Read depends on Glob finding the files) - Any chain where tool B’s input depends on tool A’s output

Example of parallel execution:

In a single response, Claude Code might output:

Tool call 1: Read(file_path: "/project/src/auth.ts")
Tool call 2: Read(file_path: "/project/src/types.ts")
Tool call 3: Grep(pattern: "AuthService", type: "ts")

All three execute simultaneously. The results return together, and Claude Code proceeds with full knowledge of all three.

Cost implication: Parallel execution does not cost extra tokens for the calls themselves. But the combined results must fit in the context window. Five parallel Reads of large files will consume context fast.

Tool Timeouts¶

Each tool has timeout behavior:

Tool Default Timeout Max Timeout Configurable?
Bash 120,000 ms (2 min) 600,000 ms (10 min) Yes, per-call
Read Fast (filesystem I/O) N/A No
Edit Fast (filesystem I/O) N/A No
Write Fast (filesystem I/O) N/A No
Glob Fast (filesystem scan) N/A No
Grep Fast (ripgrep) N/A No
WebFetch Network-dependent N/A No
Agent Session-dependent N/A No

For Bash commands that take longer than 2 minutes (builds, long test suites), set the timeout explicitly:

Bash(
  command: "python -m pytest tests/ -v --timeout=300",
  timeout: 360000,
  description: "Run full test suite (may take up to 5 minutes)"
)

For commands where you do not need the result immediately:

Bash(
  command: "npm run build",
  run_in_background: true,
  description: "Build the project in the background"
)

Background commands notify Claude Code when they complete. No polling needed.

Tool-Specific Configuration¶

Grep: head_limit and Pagination¶

Grep’s head_limit defaults to 250 entries. This prevents a broad search from flooding the context window. For most searches, 250 is plenty. But if you need more, you can paginate:

# First 250 results
Grep(pattern: "TODO", type: "py", head_limit: 250)

# Next 250 results
Grep(pattern: "TODO", type: "py", head_limit: 250, offset: 250)

Edit: Making old_string Unique¶

When your target string appears multiple times, include surrounding context:

# This might match multiple places:
Edit(old_string: "return null;", new_string: "return undefined;")

# This is unique:
Edit(
  old_string: "  if (!user) {\n    return null;\n  }",
  new_string: "  if (!user) {\n    return undefined;\n  }"
)

Bash: Description Field¶

The description field is not just documentation. It appears in the user’s permission prompt, helping them decide whether to allow the command:

Bash(
  command: "pip install requests",
  description: "Install requests library for HTTP operations"
)

A clear description makes the permission prompt informative rather than cryptic.

Common Tool Patterns¶

These are the patterns you will see Claude Code use repeatedly.

The Read-Understand-Edit Pattern¶

1. Read(file_path: "/project/src/auth.ts")
2. [LLM analyzes the code, identifies the issue]
3. Edit(file_path: "/project/src/auth.ts", old_string: "...", new_string: "...")

This is the most common three-step pattern. Read provides understanding. The LLM reasons about what to change. Edit makes the minimal modification.

The Search-Narrow-Act Pattern¶

1. Grep(pattern: "deprecated.*API", output_mode: "files_with_matches")
2. [LLM identifies the most relevant file]
3. Read(file_path: "/project/src/api/legacy.ts")
4. Edit(...)

Start broad with Grep, narrow to the right file, then act.

The Parallel Investigation Pattern¶

Tool call 1: Grep(pattern: "handleAuth", type: "ts", output_mode: "content")
Tool call 2: Glob(pattern: "**/auth/**/*.ts")
Tool call 3: Read(file_path: "/project/package.json")

Three independent lookups executing simultaneously. The LLM gets all the context it needs in one round trip.

The Delegated Exploration Pattern¶

Agent(
  prompt: "Search the codebase for all places where user sessions are created or destroyed. List each location with file:line and a one-sentence description.",
  subagent_type: "Explore"
)

Offload complex exploration to an agent. The main context stays clean.

The Verify-After-Change Pattern¶

1. Edit(file_path: "/project/src/auth.ts", old_string: "...", new_string: "...")
2. Bash(command: "python -m pytest tests/test_auth.py -v", description: "Verify fix passes tests")

Edit, then immediately verify. Claude Code pairs modifications with verification to catch regressions.

What Most People Get Wrong¶

Mistake 1: Using Bash for everything. New users type “find all Python files” and Claude Code runs find . -name "*.py". The output is unformatted, permissions are checked as a generic Bash command, and PostToolUse hooks for file operations do not fire. Glob does all of this better.

Mistake 2: Editing without reading. Trying to be “efficient” by skipping the Read step. Edit will error. More importantly, editing without reading means editing based on assumptions about the file’s current state. Those assumptions are often wrong.

Mistake 3: Ignoring head_limit on Grep. Running Grep(pattern: "import") on a large codebase without thinking about result volume. The default 250 limit helps, but the real fix is narrowing your search with glob, type, or a more specific pattern.

Mistake 4: Using Write when Edit is appropriate. Write replaces the entire file. If you use it to change one line in a 500-line file, you are transmitting 499 unchanged lines through the context window. Edit sends only the changed portion.

Mistake 5: Not setting Bash timeouts for long operations. A full test suite or a Docker build can take minutes. The default 2-minute timeout will kill it. Set explicit timeouts for known long-running commands.

Summary¶

The tool system is Claude Code’s interface with reality. Every action – reading code, modifying files, running tests, searching the web – flows through the same pipeline: permission check, hooks, execution, hooks, result. The hierarchy is deliberate: specialized tools over Bash, Read before Edit, Glob over find, Grep over grep. Master this system and you control exactly what Claude Code can and cannot do.

Key takeaways: 1. Every tool call passes through the permission-hook-execute-hook pipeline – no exceptions 2. Read before Edit is an enforced invariant, not a guideline 3. Specialized tools (Read, Edit, Glob, Grep) produce better output than their Bash equivalents 4. Parallel tool execution reduces round trips – Claude Code uses it when calls are independent 5. Tool errors are information, not failures – Claude Code reads them and adjusts
