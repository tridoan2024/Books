Chapter 10: Bash Tool and Shell Operations¶ ~17 min read

The Bash tool is a loaded weapon. It can run any command your user account can run – install packages, delete files, push code to production, spin up servers, destroy databases. No other tool in Claude Code’s arsenal has this breadth of power or this potential for damage. That combination makes it both the most useful tool for experienced practitioners and the most dangerous for everyone else.

This chapter covers how the Bash tool actually works, how to secure it, and the patterns that separate productive shell usage from chaos.

What the Bash Tool Can Do¶

The Bash tool executes shell commands on your machine, under your user account, with your permissions. There is no sandbox. There is no container. When Claude Code runs pip install requests, it installs requests into your real environment. When it runs git push origin main, it pushes to your real repository.

Capabilities: - Run any shell command (single or chained) - Access the full filesystem - Access the network - Install and remove packages - Start and stop processes - Execute scripts in any language (Python, Node, Ruby, etc.) - Interact with Docker, Kubernetes, cloud CLIs - Run build systems, test suites, linters

Parameters:

Parameter Type Description
command string (required) The shell command to execute
description string Human-readable description (shown in permission prompt)
timeout number Max execution time in ms (default: 120,000; max: 600,000)
run_in_background boolean Run without blocking (default: false)

The Execution Model¶

Understanding how Bash commands execute is critical because it differs from your normal terminal experience in important ways.

Working Directory Persists, Shell State Does Not¶

The working directory carries over between Bash calls. If one call runs cd /tmp/project, the next call starts in /tmp/project. However, shell state – variables, aliases, functions – does not persist. Each Bash invocation is a fresh shell process.

bash
# Call 1: This works
Bash(command: "cd /Users/dev/project && ls")

# Call 2: Still in /Users/dev/project (cwd persisted)
Bash(command: "ls src/")

# But this FAILS:
# Call 1:
Bash(command: "export MY_VAR=hello")
# Call 2: MY_VAR does not exist here
Bash(command: "echo $MY_VAR")  # Empty output

Why this matters: You cannot set a shell variable in one call and use it in the next. If you need state across commands, chain them in a single call:

bash
Bash(command: "export DB_URL='postgres://localhost/mydb' && python migrate.py")

Environment Initialization¶

Each Bash call inherits your shell profile (.bashrc, .zshrc, etc.). This means: - Your PATH is available - Your shell aliases are loaded (though relying on them is fragile) - Any environment variables set in your profile exist - Tool versions match what you see in your terminal

The shell type is detected from your system – bash or zsh. If your profile sets up pyenv, nvm, or other version managers, those are available.

Command Chaining¶

You have the standard shell operators:

Operator Behavior Use when
&& Run next only if previous succeeds Commands depend on each other
; Run next regardless Independent commands, don’t care about failures
\| Pipe stdout Processing output
bash
# Dependent chain: stop if any step fails
Bash(command: "cd /project && npm install && npm run build && npm test")

# Independent: run both even if first fails
Bash(command: "ruff check src/; mypy src/")

# Pipe: process output
Bash(command: "git log --oneline -20 | head -5")

Important: Do not use newlines to separate commands in a single Bash call. Use &&, ;, or | instead. Newlines inside quoted strings (like heredocs) are fine.

Security Implications¶

The Bash tool runs commands as your user. This means:

Filesystem access: It can read, write, and delete any file your user can. There is no chroot, no filesystem restriction beyond Unix permissions.

Network access: It can make HTTP requests, SSH to remote servers, push to Git remotes, deploy to cloud services.

Package installation: pip install, npm install, cargo add – all modify your real environment. A malicious or buggy package gets installed for real.

Process control: It can start servers, kill processes, modify system services (within your user’s permissions).

The permission system is your primary guardrail. Configure it deliberately.

Permission Patterns¶

The Bash tool’s permission rules use prefix matching with wildcard support. A rule like Bash(git status *) means “allow any Bash command that starts with git status.”

Safe Commands to Allow¶

These commands are read-only or low-risk. Allow them to avoid constant permission prompts:

json
{
  "permissions": {
    "allow": [
      "Bash(git status *)",
      "Bash(git diff *)",
      "Bash(git log *)",
      "Bash(git branch *)",
      "Bash(git stash list *)",
      "Bash(python -m pytest *)",
      "Bash(npm test *)",
      "Bash(cargo test *)",
      "Bash(ruff *)",
      "Bash(prettier *)",
      "Bash(eslint *)",
      "Bash(mypy *)",
      "Bash(pyright *)",
      "Bash(gh *)",
      "Bash(ls *)",
      "Bash(wc *)",
      "Bash(which *)",
      "Bash(cat *)"
    ]
  }
}

Commands to Deny¶

Block patterns that could cause irreversible damage:

json
{
  "permissions": {
    "deny": [
      "Bash(rm -rf /)",
      "Bash(rm -rf ~)",
      "Bash(rm -rf .)",
      "Bash(*DROP TABLE*)",
      "Bash(*DROP DATABASE*)",
      "Bash(*--force*main*)",
      "Bash(*--force*master*)",
      "Bash(:(){ :|:& };:)",
      "Bash(chmod -R 777 *)",
      "Bash(*> /dev/sda*)",
      "Bash(mkfs*)"
    ]
  }
}

The Permissive Local Override¶

For experienced users who trust Claude Code and want zero friction:

json
// .claude/settings.local.json (gitignored, personal only)
{
  "permissions": {
    "allow": ["Bash(*)"]
  }
}

This allows all Bash commands without prompting. Use this only if you have PreToolUse hooks that catch dangerous patterns. The block-destructive.sh hook (covered in Chapter 8) is your safety net here.

Permission Evaluation with Hooks¶

Even with Bash(*) allowed, your PreToolUse hooks still fire. This is the layered security model:

Bash command → Permission check (allow) → PreToolUse hook → Execute
                                           ↑
                                    block-destructive.sh
                                    can still block rm -rf

This is why the recommended setup for power users is Bash(*) in local settings plus block-destructive.sh as a PreToolUse hook. You get zero friction for legitimate commands and a safety net for dangerous ones.

Git Operations via Bash¶

Git is probably the most common category of Bash commands Claude Code runs. Understanding the risk levels helps you configure permissions appropriately.

Safe (Read-Only)¶

These commands never modify your repository:

bash
git status
git diff
git diff --staged
git log --oneline -20
git branch -a
git stash list
git show HEAD
git blame src/auth.ts
git remote -v

Allow all of these freely.

Needs Care (Modifies Local State)¶

These commands change your local repository but are reversible:

bash
git add src/auth.ts          # Stages files
git commit -m "fix: auth"    # Creates a commit (local only)
git checkout -b feat/new     # Creates a branch
git stash                    # Stashes changes
git stash pop                # Restores stashed changes
git merge feature-branch     # Merges locally

Most users allow these. They modify local state, but nothing leaves your machine until you push.

Dangerous (Modifies Remote or Destroys History)¶

These commands can cause permanent damage:

bash
git push origin main         # Pushes to remote (usually fine, but irreversible)
git push --force             # Rewrites remote history
git reset --hard HEAD~3      # Destroys local commits
git clean -fd                # Deletes untracked files
git checkout -- .            # Discards all unstaged changes
git branch -D feature        # Force-deletes a branch

Claude Code’s git safety protocol: - Never uses --force on push without explicit user request - Prefers new commits over amending existing ones - Never skips hooks (--no-verify) - Warns before pushing to main/master - After a failed pre-commit hook, creates a NEW commit (not --amend, which would modify the previous commit)

This protocol is baked into Claude Code’s system instructions. It applies even if you have Bash(*) allowed.

Example: The Safe Git Workflow¶

bash
# Claude Code's typical git workflow:
Bash(command: "git status", description: "Check working tree status")
Bash(command: "git diff", description: "Review changes")
Bash(command: "git add src/auth.ts tests/test_auth.py", description: "Stage specific files")
Bash(command: "git commit -m 'fix: validate token expiry before auth check'", description: "Commit fix")
Bash(command: "git push -u origin fix/token-expiry", description: "Push to feature branch")

Notice: specific files added (not git add .), descriptive commit message, push to a feature branch (not main).

Package Management¶

Claude Code uses Bash for package operations. Each has permission implications.

Python (pip / uv)¶

bash
# Install a package
Bash(command: "pip install requests", description: "Install requests library")

# Install from requirements
Bash(command: "pip install -r requirements.txt", description: "Install project dependencies")

# Using uv (faster)
Bash(command: "uv pip install fastapi", description: "Install FastAPI with uv")

Risk: Packages execute code during installation. A typosquatted package name could run malicious code. Claude Code cannot verify package authenticity – that responsibility is on you.

Node.js (npm / yarn / pnpm)¶

bash
# Install dependencies
Bash(command: "npm install", description: "Install project dependencies")

# Add a specific package
Bash(command: "npm install --save zod", description: "Add zod for schema validation")

# Audit for vulnerabilities
Bash(command: "npm audit", description: "Check for known vulnerabilities")

Rust (cargo)¶

bash
Bash(command: "cargo add serde --features derive", description: "Add serde with derive feature")

Process Management¶

Background Processes¶

The run_in_background parameter lets Claude Code start long-running processes without blocking:

bash
# Start a dev server in the background
Bash(
  command: "npm run dev",
  run_in_background: true,
  description: "Start development server"
)

# Start a database
Bash(
  command: "docker compose up -d postgres",
  run_in_background: true,
  description: "Start PostgreSQL container"
)

When a background process completes (or fails), Claude Code receives a notification. No polling or sleep loops needed.

Reading Background Output¶

After a background process finishes, Claude Code can read its output. For processes that are still running, it can check on them:

bash
# Check if the server is up
Bash(command: "curl -s http://localhost:3000/health", description: "Health check dev server")

Killing Processes¶

bash
# Stop a specific process
Bash(command: "kill $(lsof -t -i:3000)", description: "Kill process on port 3000")

# Stop Docker containers
Bash(command: "docker compose down", description: "Stop all containers")

Testing via Bash¶

Running tests is one of the most frequent Bash operations. Claude Code uses it to verify its changes work.

Running Specific Tests (Fast Feedback)¶

bash
# Python: single test file
Bash(command: "python -m pytest tests/test_auth.py -v", description: "Run auth tests")

# Python: single test function
Bash(command: "python -m pytest tests/test_auth.py::test_login_invalid_password -v", description: "Run specific test")

# JavaScript: single test file
Bash(command: "npx jest tests/auth.test.ts", description: "Run auth tests")

# Rust: single test
Bash(command: "cargo test test_auth_flow -- --nocapture", description: "Run auth flow test")

Running Full Test Suites (Before Commits)¶

Full suites take longer. Set appropriate timeouts:

bash
# Python full suite
Bash(
  command: "python -m pytest tests/ -v --tb=short",
  timeout: 300000,
  description: "Run full test suite (up to 5 min)"
)

# Node full suite
Bash(
  command: "npm test",
  timeout: 300000,
  description: "Run full test suite"
)

Capturing Test Output¶

Claude Code reads the full test output. When tests fail, it sees the assertion errors, stack traces, and failure messages. This is how it identifies what to fix:

FAILED tests/test_auth.py::test_login_expired_token
  AssertionError: Expected 401, got 200

  # Claude Code sees this and knows:
  # 1. The test expects a 401 for expired tokens
  # 2. The code is returning 200 instead
  # 3. The token expiry check is likely missing or broken

Build Systems¶

bash
# Node/TypeScript build
Bash(command: "npm run build", timeout: 180000, description: "Build project")

# Rust build
Bash(command: "cargo build --release", timeout: 300000, description: "Release build")

# Python package build
Bash(command: "python -m build", description: "Build Python package")

# Make
Bash(command: "make all", timeout: 300000, description: "Build everything")

# Docker build
Bash(
  command: "docker build -t myapp:latest .",
  timeout: 600000,
  description: "Build Docker image"
)

Build commands are the primary use case for extended timeouts. A Docker build or a Rust release compilation can take several minutes.

Output Handling¶

Understanding how Claude Code processes Bash output is critical for writing effective commands.

stdout and stderr¶

Bash captures both stdout and stderr. Both are returned to Claude Code as the tool result. This means Claude Code sees error messages, warnings, and diagnostic output – not just the “happy path.”

bash
# Claude Code sees both the lint warnings (stderr) and results (stdout)
Bash(command: "ruff check src/ 2>&1")

In most cases, you do not need to redirect stderr. Let it flow naturally – Claude Code uses error messages to diagnose and correct issues.

Output Truncation¶

When a command produces output that exceeds Claude Code’s result size limit, the output is truncated. Truncation keeps the beginning and end, dropping the middle. This preserves: - The first few hundred lines (usually the start of output or header) - The last few hundred lines (usually summary, totals, final errors)

For commands that produce large output, filter before it reaches Claude Code:

bash
# BAD: Returns ALL test output, likely truncated
Bash(command: "python -m pytest tests/ -v")

# BETTER: Returns only failures
Bash(command: "python -m pytest tests/ -v --tb=short -q")

# BEST for large suites: Return summary only, with failure details
Bash(command: "python -m pytest tests/ --tb=line -q")

Exit Codes¶

Claude Code checks the exit code of every Bash command. Non-zero exit codes mark the result as an error. This triggers Claude Code’s error-handling behavior: it reads the error, forms a hypothesis about what went wrong, and adjusts its approach.

bash
# Exit code 0 → success → Claude Code proceeds
Bash(command: "python -m pytest tests/test_auth.py -v")

# Exit code 1 → test failure → Claude Code reads the failure output and investigates
Bash(command: "python -m pytest tests/test_auth.py -v")
# Output: "FAILED test_login_expired_token - AssertionError..."
# Claude Code: "The test expects 401 for expired tokens but gets 200. Let me check the expiry logic..."

Commands that “fail successfully” (non-zero exit but expected behavior) can confuse Claude Code. Use || true to suppress expected failures:

bash
# grep returns exit code 1 when no matches found -- not really an error
Bash(command: "grep -r 'DEPRECATED' src/ || true")

Binary and Large File Output¶

Some commands produce binary output (images, compiled files). Claude Code cannot meaningfully process binary data. If you need to verify a binary was created, check existence and size:

bash
# Check that the build produced the expected binary
Bash(command: "ls -lh dist/app && file dist/app")

Sandboxing and Safety Modes¶

Claude Code can operate with different safety levels depending on the environment.

The Seatbelt Model¶

In standard operation, the Bash tool has no sandbox. But Claude Code’s safety comes from layers:

Layer 1: Claude Code's built-in safety (refuses destructive commands by default)
Layer 2: Permission system (deny/allow rules in settings.json)
Layer 3: PreToolUse hooks (block-destructive.sh pattern matching)
Layer 4: OS-level permissions (your user account's Unix permissions)

Each layer catches what the previous layer missed. A well-configured setup has all four layers active.

CI/CD Environments¶

When Claude Code runs in CI (covered in detail in Chapter 20), the Bash tool operates differently: - No interactive permission prompts (everything must be pre-allowed or pre-denied) - Typically runs in a container with limited filesystem and network access - Environment variables are set by the CI system - Timeouts may need adjustment for container startup overhead

Audit Trail¶

Every Bash command Claude Code runs is logged in the session. You can review what commands were executed using /export to save the full session transcript. This is valuable for: - Post-incident review (“What did Claude Code run before the failure?”) - Compliance documentation - Debugging unexpected behavior - Learning what commands Claude Code chose for different tasks

Linting and Code Quality via Bash¶

Linting is a frequent Bash use case because lint tools are command-line programs with no dedicated Claude Code tool.

Python¶

bash
# Check for issues
Bash(command: "ruff check src/auth/", description: "Lint Python code")

# Auto-fix issues
Bash(command: "ruff check --fix src/auth/", description: "Auto-fix lint issues")

# Format code
Bash(command: "ruff format src/auth/", description: "Format Python code")

# Type checking
Bash(command: "mypy src/auth/ --strict", description: "Type check auth module")
Bash(command: "pyright src/auth/", description: "Type check with pyright")

JavaScript/TypeScript¶

bash
# ESLint
Bash(command: "npx eslint src/ --fix", description: "Lint and auto-fix JS/TS")

# Prettier
Bash(command: "npx prettier --write 'src/**/*.{ts,tsx}'", description: "Format TypeScript files")

# TypeScript compiler check
Bash(command: "npx tsc --noEmit", description: "Type check TypeScript")

Rust¶

bash
# Clippy (Rust linter)
Bash(command: "cargo clippy -- -D warnings", description: "Lint Rust code")

# Format
Bash(command: "cargo fmt --check", description: "Check Rust formatting")

Combined Quality Check¶

A pattern Claude Code uses before committing:

bash
# Run all quality checks in sequence
Bash(
  command: "ruff check src/ && ruff format --check src/ && mypy src/ --strict && python -m pytest tests/ -q",
  timeout: 180000,
  description: "Full quality check: lint, format, types, tests"
)

If any step fails, && stops the chain and Claude Code sees exactly which step failed.

Advanced Patterns¶

Heredocs for Multi-line Input¶

When Claude Code needs to pass multi-line content to a command, it uses heredocs:

bash
Bash(command: "cat > /tmp/test-config.json << 'EOF'\n{\n  \"database\": \"postgres://localhost/test\",\n  \"port\": 8080,\n  \"debug\": true\n}\nEOF")

This is commonly used for: - Creating temporary configuration files - Writing inline scripts - Passing multi-line SQL queries - Git commit messages with body text

Environment Variables Within a Single Call¶

Since environment variables do not persist across calls, set them inline:

bash
# Set and use in same call
Bash(command: "DATABASE_URL='postgres://localhost/mydb' python -m pytest tests/integration/")

# Multiple variables
Bash(command: "NODE_ENV=production API_KEY=test123 npm run build")

Working with Docker¶

bash
# Run a command in a container
Bash(command: "docker exec -it myapp-db psql -U postgres -c 'SELECT count(*) FROM users;'")

# Compose operations

What NOT to Do with Bash¶

The Bash tool should be your last resort for tasks that have dedicated tools. Here is the rule: if a specialized tool exists, use it.

Do NOT do this Do this instead Why
Bash(command: "cat src/auth.ts") Read(file_path: "src/auth.ts") Read provides line numbers, handles binary/image files, respects read permissions
Bash(command: "grep -r 'TODO' .") Grep(pattern: "TODO") Grep tool uses ripgrep, respects gitignore, has structured output modes
Bash(command: "find . -name '*.ts'") Glob(pattern: "**/*.ts") Glob is faster, sorted by mtime, optimized output
Bash(command: "sed -i 's/old/new/' file.py") Edit(old_string: "old", new_string: "new") Edit is atomic, has read-first safety, triggers PostToolUse hooks
Bash(command: "echo 'content' > new.py") Write(file_path: "new.py", content: "content") Write handles encoding, triggers hooks, tracked in tool history

Why this matters beyond aesthetics:

  1. Permission granularity. You can allow Read(*) and deny Read(./.env). You cannot do that with Bash(cat *) vs Bash(cat .env) because both match Bash(cat *).

  2. Hook integration. PostToolUse hooks fire based on tool name. auto-lint-python.sh triggers on Edit/Write of .py files. It cannot trigger on Bash(sed ...).

  3. Output formatting. Dedicated tools produce output optimized for Claude Code’s context window. Bash output is raw text that may need parsing.

  4. Error handling. Edit’s error “old_string not found” is unambiguous. sed silently does nothing if the pattern does not match.

10 Practical Bash Patterns¶

These are the patterns that come up in real Claude Code sessions.

1. Check Before Acting¶

bash
Bash(command: "test -f package-lock.json && echo 'npm' || (test -f yarn.lock && echo 'yarn' || echo 'unknown')")

Detect the package manager before running install commands.

2. Single Test with Verbose Output¶

bash
Bash(command: "python -m pytest tests/test_auth.py::test_login_expired_token -v --tb=long")

Run one test with full traceback. The fastest feedback loop for bug fixes.

3. Search and Count¶

bash
Bash(command: "git log --oneline --since='1 week ago' | wc -l")

Count commits in the last week. Useful for changelog generation.

4. Safe Branch Cleanup¶

bash
Bash(command: "git branch --merged main | grep -v main | grep -v '\\*'")

List branches that have been merged into main. Never auto-delete – just list.

5. Port Check¶

bash
Bash(command: "lsof -i :3000 -t 2>/dev/null || echo 'Port 3000 is free'")

Check if a port is in use before starting a server.

6. Quick HTTP Check¶

bash
Bash(command: "curl -s -o /dev/null -w '%{http_code}' http://localhost:8080/health")

Get just the HTTP status code. Clean for health checks.

7. Disk Usage for Project¶

bash
Bash(command: "du -sh node_modules/ .git/ dist/ 2>/dev/null")

See where disk space is going.

8. Find Recently Modified Files¶

bash
Bash(command: "find . -name '*.py' -mmin -60 -not -path './venv/*' | head -20")

Files modified in the last hour. Useful for understanding what was just changed. (Note: for finding files by name pattern in general, use Glob.)

9. Inline Python for Data Processing¶

bash
Bash(command: "python3 -c \"import json; data = json.load(open('config.json')); print(json.dumps(data, indent=2))\"")

Quick Python one-liners for data processing tasks that would be awkward in pure shell.

10. Git Commit with Heredoc Body¶

bash
Bash(command: "git commit -m \"$(cat <<'EOF'\nfix: validate token expiry before auth check\n\nThe authentication middleware was accepting expired tokens because\nthe expiry check happened after the signature verification.\nThis fix moves the expiry check to run first.\n\nCo-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>\nEOF\n)\"")

Multi-line commit messages with the heredoc pattern. Claude Code uses this for commits with body text.

What Most People Get Wrong¶

Mistake 1: Sleep loops for polling.

bash
# BAD: Wastes time and tokens
Bash(command: "sleep 10 && curl http://localhost:3000/health")
Bash(command: "sleep 10 && curl http://localhost:3000/health")
Bash(command: "sleep 10 && curl http://localhost:3000/health")

Use run_in_background for the server start, then check when notified. Or use a single check command rather than sleeping first.

Mistake 2: Using interactive flags.

bash
# FAILS: No interactive input available
Bash(command: "git rebase -i HEAD~3")
Bash(command: "git add -i")
Bash(command: "python3")  # Opens interactive REPL

Claude Code cannot interact with prompts. Use non-interactive equivalents or pass input via pipes/heredocs.

Mistake 3: Ignoring timeouts on long builds.

bash
# FAILS after 2 minutes
Bash(command: "cargo build --release")

# WORKS
Bash(command: "cargo build --release", timeout: 600000)

Know your build times and set timeouts accordingly.

Mistake 4: cd without chaining.

bash
# Does nothing useful by itself
Bash(command: "cd /Users/dev/other-project")

# Better: cd and run in one call
Bash(command: "cd /Users/dev/other-project && cargo test")

# Best: use absolute paths
Bash(command: "cargo test --manifest-path /Users/dev/other-project/Cargo.toml")

While cd does persist the working directory, it is cleaner to use absolute paths or chain commands.

Mistake 5: Redirecting stderr to /dev/null unconditionally.

bash
# BAD: Hides errors that Claude Code should see
Bash(command: "npm run build 2>/dev/null")

# GOOD: Let errors flow through
Bash(command: "npm run build")

Claude Code uses error output to diagnose and fix issues. Silencing stderr removes that signal.

Summary¶

The Bash tool is unrestricted power over your machine. It runs as your user, accesses your filesystem, and talks to your network. The permission system and hooks are your guardrails. Configure allow lists for safe commands, deny patterns for destructive ones, and use block-destructive.sh as the catch-all safety net. Prefer dedicated tools (Read, Edit, Grep, Glob) for tasks they handle better. Reserve Bash for git operations, test execution, builds, package management, and the command-line operations that no other tool covers.

Key takeaways: 1. Working directory persists between calls, but shell state (variables, functions) does not 2. Bash runs as your user with full system access – permissions and hooks are your only guardrails 3. Use dedicated tools instead of Bash for file reading, searching, and editing 4. Set explicit timeouts for builds and test suites that exceed 2 minutes 5. Use run_in_background for long-running processes instead of sleep loops
