Chapter 26: Complete Command and Tool Reference¶ ~23 min read

Every previous chapter taught concepts through explanation and example. This chapter is different. It is a reference — the kind of chapter you bookmark and return to when you need the exact name of a command, the precise parameter a tool accepts, or the environment variable that controls a behavior you vaguely remember exists.

The tables here are comprehensive. They cover every built-in command, every tool, the complete configuration schema, and every environment variable that influences Claude Code’s behavior. Where a feature was covered in depth in an earlier chapter, we note the chapter number. Where a feature appears only here, we provide enough detail to use it immediately.

Keep this chapter open in a second tab.

Part 1: Built-in Commands¶

Built-in commands start with / and are entered directly in the Claude Code prompt. They control session behavior, navigation, configuration, and development workflows. Unlike tools (which Claude invokes programmatically), commands are user-initiated actions that modify the session state or produce output.

Session Management Commands¶

These commands control your current conversation session — clearing context, viewing costs, and managing session state.

Command Arguments Description
/help None Display all available commands with brief descriptions. The starting point when you forget a command name.
/clear None Reset the conversation context entirely. Use between unrelated tasks to prevent context pollution. Does not affect files on disk — only the in-memory conversation. Covered in Chapter 7.
/compact [focus-topic] Compress the conversation context, optionally preserving information related to a specific topic. Example: /compact authentication retains auth-related context while discarding the rest. Essential when approaching context limits. Covered in Chapter 7.
/context None Display a visual breakdown of current context window usage — how many tokens are consumed by system prompt, conversation history, tool results, and available capacity. Use this to decide whether to /compact or /clear.
/cost None Show the cost of the current session in dollars. Breaks down input tokens, output tokens, and cache hits. Useful for monitoring spend during long sessions.
/tokens None Display raw token counts for the current session — input, output, cache read, cache write. More granular than /cost.
/stats None Session statistics dashboard: token usage, cost, tool call counts, duration, and model information. Run at end of day for a summary.

Navigation Commands¶

These commands let you move through conversation history, resume previous sessions, and export transcripts.

Command Arguments Description
/rewind None Roll back the conversation to a previous point. Claude presents numbered checkpoints — pick one to restore. Everything after that point is discarded. Use when a line of investigation went wrong.
/resume None Resume the most recent conversation. When you restart Claude Code, your previous session is available for continuation.
/sessions None List all saved conversation sessions with timestamps and summaries. Navigate between sessions or resume any previous one.
/export [filename] Export the current conversation to a file. Default format is markdown. Example: /export debug-session.md. Useful for creating records of debugging sessions or architectural discussions.

Configuration Commands¶

Runtime configuration — change models, reasoning depth, and settings without editing files.

Command Arguments Description
/model [model-name] Switch the active model. Options include opus, sonnet, haiku, or full model IDs like claude-sonnet-4-6. Without arguments, displays the current model. Change persists for the session. Covered in Chapter 3.
/effort low \| medium \| high \| max Set reasoning depth. low = fast, shallow responses. high = deep analysis (default in most professional setups). max = extended thinking with maximum token budget. Covered in Chapter 7.
/fast None Toggle fast mode on/off. When enabled, Claude produces shorter responses with less deliberation. Same model, faster output. Good for simple file edits and quick questions.
/config None Open the interactive settings editor. View and modify permissions, hooks, model preferences, and other configuration without editing JSON files directly.

Development Commands¶

Commands that integrate with your development workflow — viewing diffs, creating commits, and planning multi-step work.

Command Arguments Description
/diff None Display uncommitted changes in an interactive diff viewer. Shows staged and unstaged modifications. Equivalent to a visual git diff within the Claude Code interface.
/commit None Create a git commit with Claude-generated message. Claude analyzes staged changes, drafts a commit message following conventional commit format, and includes the Co-Authored-By attribution line. Covered in Chapter 8.
/plan None Enter plan mode. Claude creates a structured plan before executing any changes. Useful for multi-file refactors or complex features where you want to review the approach before code is modified. Exit with /exitplan. Covered in Chapter 11.

Task Management Commands¶

Commands for managing background and parallel tasks. Covered in depth in Chapter 14.

Command Arguments Description
/tasks None List all active, pending, and completed tasks in the current session. Shows task IDs, status, and summaries.
/branch [name] Create a conversation branch for experimental work. If the experiment fails, discard the branch. If it succeeds, continue from it.

Advanced Commands¶

Power-user commands for parallel operations, recurring tasks, side conversations, and diagnostics.

Command Arguments Description
/batch "instruction" Execute an instruction across multiple files in parallel using isolated worktrees. Example: /batch "Update all API endpoints to return standardized error responses". Claude spawns parallel agents, each working in its own worktree. Covered in Chapter 14.
/loop [interval] [command] Run a command on a recurring interval. Example: /loop 5m check if CI pipeline passed. Default interval is 10 minutes. Useful for monitoring deployments, waiting for CI, or polling external services.
/btw [question] Ask a side question without contaminating the main conversation context. Example: /btw what does the --no-verify flag do?. Claude answers in an isolated context, then returns to the main thread.
/simplify None Review recently changed code for quality — identifies redundancy, unnecessary complexity, and opportunities for reuse. Runs automatically in some configurations.
/doctor None Diagnose your Claude Code installation. Checks: API key validity, model access, MCP server connectivity, hook execution, settings syntax, and environment variables. Run this first when something is not working.
/skills None List all available skills (both built-in and custom) with their trigger descriptions. Shows which skills are user-invocable vs. auto-activating.
/agents None List all configured subagents with their model, tools, and descriptions. Shows which agents are available for delegation.

Keyboard Shortcuts¶

These work in the Claude Code terminal interface at any time.

Shortcut Action
Esc Cancel the current generation. Claude stops producing output immediately. Partial output is retained in context.
Ctrl+C Interrupt the current operation. If Claude is running a tool (e.g., a long bash command), this cancels the tool execution.
Tab Autocomplete file paths after @ mentions. Also autocompletes slash commands after /.
/ + Tab Browse all available slash commands and skills in an autocomplete menu.
Up Arrow Cycle through previous prompts (prompt history).
Ctrl+L Clear the terminal display (does not clear conversation context — use /clear for that).
Ctrl+D Exit Claude Code (end the session).

Part 2: All Tools Reference¶

Tools are the functions Claude invokes programmatically to interact with your system. Unlike commands (which you type), tools are called by Claude in response to your requests. Each tool has defined parameters, return values, and restrictions.

The tools below are listed in functional groups. For each tool, we document: what it does, its parameters, what it returns, important notes, and restrictions.

File System Tools¶

Read¶

Reads file contents from the local filesystem.

Field Detail
Parameters file_path (string, required): Absolute path to the file. offset (integer, optional): Line number to start reading from. limit (integer, optional): Number of lines to read. pages (string, optional): Page range for PDFs, e.g., "1-5".
Returns File contents with line numbers (cat -n format). For images, returns visual content. For PDFs, returns text content of specified pages. For notebooks, returns all cells with outputs.
Notes Can read any file type: text, images (PNG, JPG), PDFs, Jupyter notebooks (.ipynb). Default reads up to 2000 lines. For large PDFs (>10 pages), the pages parameter is required. Cannot read directories — use ls via Bash for that.
Restrictions Files listed in settings.json deny rules (e.g., .env, credentials.json) are blocked. Path must be absolute, not relative.

Edit¶

Performs exact string replacements in files.

Field Detail
Parameters file_path (string, required): Absolute path. old_string (string, required): Exact text to find. new_string (string, required): Replacement text. replace_all (boolean, optional, default false): Replace all occurrences instead of just the first.
Returns Confirmation of the edit with before/after context.
Notes The old_string must be unique in the file unless replace_all is true. If it matches multiple locations and replace_all is false, the edit fails with an error. Always preserves exact indentation. The file must have been Read at least once in the conversation before editing.
Restrictions Cannot create new files — use Write for that. Fails if old_string is not found or is not unique (without replace_all).

Write¶

Creates or completely overwrites a file.

Field Detail
Parameters file_path (string, required): Absolute path. content (string, required): Complete file content.
Returns Confirmation that the file was written.
Notes Overwrites existing files entirely. For existing files, the file must have been Read first — Write will fail otherwise. Prefer Edit for modifications to existing files (sends only the diff, more efficient).
Restrictions Path must be absolute. The requirement to Read before Write prevents accidental overwrites.

NotebookEdit¶

Edits Jupyter notebook (.ipynb) cells.

Field Detail
Parameters notebook_path (string, required): Absolute path to .ipynb file. new_source (string, required): New cell content. cell_id (string, optional): Cell ID for positioning inserts. cell_type (string, optional): "code" or "markdown". edit_mode (string, optional): "replace", "insert", or "delete".
Returns Confirmation of the notebook edit.
Notes Cell numbering is 0-indexed. When inserting, the new cell appears after the cell specified by cell_id. When edit_mode is "insert", cell_type is required.
Restrictions Path must be absolute. Only operates on .ipynb files.

Search Tools¶

Glob¶

Fast file pattern matching.

Field Detail
Parameters pattern (string, required): Glob pattern, e.g., "**/*.py", "src/**/*.ts". path (string, optional): Directory to search in. Defaults to current working directory.
Returns List of matching file paths, sorted by modification time (most recent first).
Notes Works with any codebase size. Use for finding files by name or extension. For finding files by content, use Grep instead.
Restrictions None beyond filesystem permissions.

Grep¶

Content search powered by ripgrep.

Field Detail
Parameters pattern (string, required): Regex pattern. path (string, optional): File or directory to search. output_mode (string, optional): "files_with_matches" (default), "content", or "count". glob (string, optional): File filter, e.g., "*.js". type (string, optional): File type filter, e.g., "py", "js". -A, -B, -C (integer, optional): Lines after, before, or around matches. -i (boolean, optional): Case-insensitive. -n (boolean, optional): Show line numbers (default true). multiline (boolean, optional): Match across line boundaries. head_limit (integer, optional): Limit output entries (default 250). offset (integer, optional): Skip first N entries.
Returns Depends on output_mode: file paths, matching lines with context, or match counts.
Notes Uses ripgrep syntax, not grep. Literal braces need escaping: interface\\{\\} to find interface{}. For cross-line patterns, enable multiline: true. Prefer type over glob for standard file types (more efficient).
Restrictions The head_limit defaults to 250 to prevent context overflow. Pass 0 for unlimited (use sparingly).

Execution Tools¶

Bash¶

Executes shell commands.

Field Detail
Parameters command (string, required): The shell command to execute. description (string, optional): Human-readable description of what the command does. timeout (integer, optional): Timeout in milliseconds (max 600,000 = 10 minutes). Default is 120,000 (2 minutes). run_in_background (boolean, optional): Run asynchronously.
Returns Command stdout and stderr. Exit code information on failure.
Notes Working directory resets between calls — always use absolute paths. Shell state (variables, aliases) does not persist between calls. When run_in_background is true, the result is delivered asynchronously. Use && to chain dependent commands; use ; when order matters but failure does not.
Restrictions Subject to permission rules in settings.json (allow/deny lists). PreToolUse hooks fire before execution and can block commands (exit code 2). Commands matching deny patterns are rejected.

Agent¶

Spawns a subagent for complex, multi-step tasks.

Field Detail
Parameters prompt (string, required): The task description for the subagent. model (string, optional): Override the subagent’s model. allowed-tools (list, optional): Restrict which tools the subagent can use.
Returns The subagent’s complete response after it finishes its work.
Notes Subagents run in forked context — they cannot see or modify the parent conversation’s state. Useful for investigation tasks where you want to keep your main context clean. Custom agents defined in .claude/agents/ are available for specialized delegation. Covered in Chapter 13.
Restrictions Subagents inherit permission rules from settings. They cannot access tools not in their allowed-tools list.

Web Tools¶

WebFetch¶

Fetches and processes web page content.

Field Detail
Parameters url (string, required): Fully-formed URL (HTTP auto-upgrades to HTTPS). prompt (string, required): Instruction for processing the fetched content.
Returns AI-processed summary of the page content based on the prompt.
Notes Converts HTML to markdown before processing. Includes a 15-minute cache for repeated requests. If the URL redirects to a different host, returns the redirect URL instead of following it — make a second request with the new URL. Content may be summarized if very large.
Restrictions Fails for authenticated or private URLs (Google Docs, Confluence, Jira, private GitHub). For GitHub, use gh CLI via Bash instead. For authenticated sites, use MCP tools or browser automation.

WebSearch¶

Performs web searches and returns results.

Field Detail
Parameters query (string, required): Search query (minimum 2 characters). allowed_domains (list, optional): Only include results from these domains. blocked_domains (list, optional): Exclude results from these domains.
Returns Search results with titles, URLs, and snippets formatted as markdown.
Notes Results include links as markdown hyperlinks. Domain filtering supports both inclusion and exclusion. When using this tool, sources should be cited in the response.
Restrictions Web search availability may be region-dependent.

User Interaction Tools¶

AskUserQuestion¶

Prompts the user for input during execution.

Field Detail
Parameters question (string, required): The question to ask the user.
Returns The user’s text response.
Notes Use sparingly. The auto-execute philosophy (Chapter 5) prefers Claude to make reasonable decisions rather than asking. Reserve for genuinely ambiguous situations where proceeding without clarification would likely waste significant effort.
Restrictions None, but overuse degrades the user experience.

Exits planning mode and returns to normal execution.

Field Detail

Returns Task ID and confirmation.

TaskGet¶

Gets detailed status of a specific task.

Field Detail
Parameters task_id (string, required): The task identifier.
Returns Full task details: status, output, errors, duration.
Notes Use after TaskList to inspect a specific task’s results.
Restrictions Task must exist in the current session.

TaskUpdate¶

Updates an existing task’s parameters.

Field Detail
Parameters task_id (string, required): The task identifier. status (string, optional): New status. notes (string, optional): Additional context or instructions.
Returns Confirmation of the update.
Notes Can be used to cancel a running task or update priority.

Worktree Tools¶

EnterWorktree¶

Creates an isolated git worktree for safe experimentation.

Field Detail
Parameters name (string, optional): Name for the worktree (kebab-case, max 64 characters). Auto-generated if omitted.
Returns Confirmation with worktree path and branch name.
Notes Creates a new worktree inside .claude/worktrees/ with a new branch based on HEAD. The session’s working directory switches to the worktree. All file operations happen in isolation — the main working tree is untouched. Used by /batch internally. Covered in Chapter 14.
Restrictions Must be in a git repository. Cannot enter a worktree if already in one. Only use when the user explicitly asks for a worktree.

ExitWorktree¶

Leaves a worktree session and returns to the original directory.

Field Detail
Parameters action (string, required): "keep" (leave worktree on disk) or "remove" (delete worktree and branch). discard_changes (boolean, optional): Required when removing a worktree with uncommitted changes.
Returns Confirmation of exit. If kept, returns the worktree path for later access.
Notes Only operates on worktrees created by EnterWorktree in the current session. With "keep", the worktree remains on disk for future use. With "remove", the worktree directory and branch are deleted. Refuses to remove worktrees with uncommitted changes unless discard_changes is true.
Restrictions No-op if not in a worktree session. Will not touch manually created worktrees or worktrees from previous sessions.

Creates a recurring scheduled task.

Field Detail

Returns Cron job ID and confirmation with next run time.
Notes Tasks execute in the background. Each execution gets its own context. Useful for monitoring, periodic health checks, or recurring maintenance tasks.
Restrictions Maximum number of concurrent cron jobs may be limited.

CronDelete¶

Deletes a scheduled task.

Field Detail
Parameters cron_id (string, required): The cron job identifier from CronCreate.
Returns Confirmation of deletion.
Notes Stops all future executions. Currently running executions complete normally.
Restrictions Cron job must exist.

CronList¶

Lists all scheduled tasks.

Field Detail
Parameters None.
Returns Table of cron jobs with IDs, schedules, names, status, and last/next run times.
Notes Shows both active and paused cron jobs.
Restrictions None.

Skill Tool¶

Skill¶

Invokes a registered skill within the conversation.

Field Detail
Parameters skill (string, required): Skill name, e.g., "commit", "review-pr", "fix-issue". Can use fully qualified names like "my-extension:skill-name". args (string, optional): Arguments to pass to the skill.
Returns The skill’s output after execution.
Notes Skills are loaded from .claude/skills/*/SKILL.md. When a user types /skill-name, this tool is invoked internally. Skills can also be triggered automatically based on their description field matching the user’s intent. Covered in Chapter 10.

Part 3: Configuration Reference¶

settings.json Complete Schema¶

The settings.json file (located at .claude/settings.json) controls permissions, hooks, and behavior. A .claude/settings.local.json file provides personal overrides that are not committed to git. Settings precedence: Local > Project > User > Managed.

Top-Level Fields¶

Field Type Description
permissions object Allow and deny rules for tool access.
permissions.allow string[] Glob patterns for auto-approved tool calls. Example: "Bash(git diff *)".
permissions.deny string[] Glob patterns for blocked tool calls. Example: "Read(./.env)". Evaluated before allow rules.
hooks object Maps lifecycle events to hook scripts.
hooks.SessionStart HookDef[] Hooks that run when a session begins.
hooks.PreToolUse HookDef[] Hooks that run before a tool executes. Can block (exit 2).
hooks.PostToolUse HookDef[] Hooks that run after a tool executes successfully.
hooks.Notification HookDef[] Hooks that run when Claude sends a notification.
hooks.UserPromptSubmit HookDef[] Hooks that run when the user submits a prompt.
hooks.Stop HookDef[] Hooks that run before Claude finishes responding.
hooks.SubagentStop HookDef[] Hooks that run when a subagent finishes.
hooks.TaskCompleted HookDef[] Hooks that run when a background task completes.
effortLevel string Default reasoning depth: "low", "medium", "high", or "max".
model string Default model ID, e.g., "claude-sonnet-4-6".
includeGitInstructions boolean Include git workflow instructions in system prompt. Default true.
autoMemoryEnabled boolean Enable auto-memory across sessions. Default false.
attribution.commit string Co-author line added to git commits.
attribution.pr string Attribution line added to PR descriptions.
disableAllHooks boolean Disable all hooks (useful for debugging). Default false.

HookDef Object¶

Field Type Description
matcher string Tool name this hook applies to, e.g., "Bash", "Edit", "Write". Omit for event-level hooks (SessionStart, Notification).
hooks HookConfig[] Array of hook configurations.

HookConfig Object¶

Field Type Description
type string Always "command".
command string Shell command or script path to execute.
timeout integer Maximum execution time in milliseconds. Default 10,000.
statusMessage string Message displayed while the hook runs.

Permission Rule Syntax¶

Permission rules use the format ToolName(pattern):

Pattern Meaning
Bash(git diff *) Allow any bash command starting with git diff
Bash(python -m pytest *) Allow running pytest
Read(./.env) Match reading the .env file
Read(./secrets/*) Match reading any file in secrets/
Bash(*) Match any bash command (use in local settings for unrestricted access)

Evaluation order: deny rules are checked first. First matching rule wins. If no rule matches, Claude asks for permission.

SKILL.md Frontmatter Fields¶

Skills are defined in .claude/skills/[name]/SKILL.md. The YAML frontmatter controls behavior. Covered in Chapter 10.

Field Type Required Description
name string Yes The slash command name. Becomes /name in the interface.
description string Yes When Claude should auto-load this skill. Include trigger phrases for reliable auto-activation.
argument-hint string No Shown in autocomplete. Example: "[issue-number]".
allowed-tools string No Comma-separated list of tools the skill can use without permission prompts. Example: "Read, Grep, Bash, Glob".
model string No Override model for this skill: "opus", "sonnet", "haiku", or full model ID.
effort string No Reasoning depth: "low", "medium", "high", "max".
context string No "fork" runs the skill in an isolated subagent context. Keeps main context clean.
agent string No Which subagent type to use when context: fork. Example: "general-purpose".
user-invocable boolean No If false, hidden from the / menu. Only Claude can trigger it. Default true.
disable-model-invocation boolean No If true, only the user can trigger this skill via slash command — Claude cannot auto-invoke it. Default false.
paths string[] No Glob patterns for auto-activation. When the user works on files matching these patterns, the skill loads automatically. Example: ["src/api/**", "**/routes/**"].

Dynamic Context Injection¶

Within SKILL.md content, the !`command` syntax executes shell commands before content reaches Claude:

markdown
## Context
- PR diff: !`gh pr diff $ARGUMENTS`
- Changed files: !`gh pr diff $ARGUMENTS --name-only`

The $ARGUMENTS variable contains whatever the user typed after the slash command.

Agent Definition Frontmatter Fields¶

Agents are defined in .claude/agents/[name].md. Covered in Chapter 13.

Field Type Required Description
name string Yes Agent identifier. Used for delegation references.
description string Yes When Claude should delegate to this agent. Include specific task types and trigger phrases.
tools string Yes Comma-separated tools the agent can access. Example: "Read, Grep, Glob, Bash".
model string No Model override: "opus" for critical analysis, "sonnet" for balanced work, "haiku" for fast tasks.
effort string No Reasoning depth: "low", "medium", "high", "max".
memory string No Memory scope: "user" (global), "project" (per-project), "local" (gitignored).
maxTurns integer No Maximum number of tool-use turns the agent can take before returning. Prevents runaway agents.
isolation string No "worktree" runs the agent in an isolated git worktree. Changes do not affect the main working tree until explicitly merged.

Rule Frontmatter Fields¶

Rules are defined in .claude/rules/[name].md. Covered in Chapter 9.

Field Type Required Description
paths string[] No Glob patterns that activate this rule. Example: ["**/*.py", "**/*.pyi"]. When a file matching the pattern is involved in the conversation, the rule loads automatically.

Rules without paths frontmatter are always loaded. Rules with paths are loaded only when relevant files are being worked on.

Rule Import Syntax¶

In CLAUDE.md, rules can be imported with the @ prefix:

markdown
@.claude/rules/python.md
@.claude/rules/security.md

Imported rules load every session regardless of their paths frontmatter. Path-scoped rules that are not @-imported load conditionally based on file context.

Model & Behavior Variables¶

Variable Description Default
