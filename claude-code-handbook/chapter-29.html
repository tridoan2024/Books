Chapter 29: CLI-to-CLI Protocol — Agent-to-Agent Communication¶ ~24 min read

Every revolution in computing follows the same arc. First, we build powerful standalone systems. Then we connect them. Mainframes became networks. Monoliths became microservices. Databases became distributed systems. In every case, the protocol that connected the pieces mattered more than the pieces themselves. HTTP mattered more than any single web server. gRPC mattered more than any single microservice.

We are at that inflection point with AI agents.

Today, Claude Code is a formidable standalone agent — it reads code, writes code, searches the web, executes commands, and reasons about complex systems. But it works alone. When a security audit agent finds a critical vulnerability, it cannot hand that finding to a deployment agent that blocks the release. When a research agent discovers that a dependency has a known CVE, it cannot tell the code-review agent to flag every import of that package. When a debugging agent isolates a root cause across three services, it cannot dispatch fix agents to each service in parallel.

This chapter introduces the CLI-to-CLI Protocol (C2C) — a communication standard that lets AI CLI agents discover each other, negotiate capabilities, delegate tasks, and coordinate work. It is not a theoretical exercise. By the end of this chapter, you will have a working reference implementation and concrete patterns for building multi-agent systems that talk to each other over the same pipes and sockets that Unix processes have used for fifty years.

The Vision: CLI Agents That Talk to Each Other¶

The Isolation Problem¶

Consider a realistic development scenario. You are preparing a release. The work involves:

  1. A security audit of all changed files
  2. A dependency vulnerability scan
  3. Performance regression testing on three critical endpoints
  4. Documentation updates for new API endpoints
  5. A changelog generated from commit history
  6. A pre-deployment readiness check

Today, you run these as sequential Claude Code sessions. Each session starts from scratch, builds its own context, does its work, and produces output that you — the human — must manually feed into the next step. You are the message bus. You are the integration layer. You are the bottleneck.

This is exactly where server-side computing was in 2005. Individual services were powerful, but they communicated through humans copying data between systems. Then came service-oriented architecture, then microservices, then event-driven systems. The services got smaller and more specialized, but the protocols between them got richer and more standardized.

AI agents need the same evolution.

Why Existing Protocols Fall Short¶

You might ask: don’t we already have protocols for this? MCP connects Claude Code to external systems. Google’s Agent-to-Agent (A2A) protocol addresses multi-agent communication. Why build something new?

The answer lies in what each protocol optimizes for.

MCP (Model Context Protocol) connects a single AI model to external tools and data sources. It is a client-server protocol where the AI is always the client and external systems are always the servers. MCP has no concept of peer-to-peer communication. An MCP server cannot initiate a conversation with the AI — it can only respond to requests. This makes MCP excellent for tool use but inadequate for agent coordination, where either party might need to initiate, delegate, or interrupt.

Google A2A addresses agent-to-agent communication but targets cloud-native, HTTP-based agents with Agent Cards, JSON-RPC over HTTP, and server-sent events. It assumes agents are long-running web services with discoverable URLs. CLI agents are different — they are ephemeral processes, they communicate over stdio, they live on developer machines, and they need to work without network access. A2A’s web-service assumptions create unnecessary overhead for local agent coordination.

What C2C optimizes for:

Property MCP A2A C2C
Communication model Client-server Peer-to-peer (HTTP) Peer-to-peer (stdio/socket)
Agent lifecycle Server always running Long-running service Ephemeral or persistent
Transport stdio (one direction) HTTP/SSE stdio, Unix socket, filesystem
Network required No Yes No
Initiation Client only Either party Either party
CLI-native Partially No Yes
Works offline Yes No Yes
Discovery Config-based Agent Cards (HTTP) Local registry + capability query

C2C takes the Unix philosophy — small programs that do one thing well, communicating through text streams — and applies it to AI agents. It is designed for the world where you have a security agent, a code-review agent, a debugging agent, and a deployment agent, all running as CLI processes on your machine, coordinating through pipes and sockets.

The Microservices Analogy¶

If you understand microservices architecture, you already understand the C2C vision. Map the concepts:

Microservices Concept C2C Equivalent
Service CLI agent (Claude Code session with specific role)
API gateway Orchestrator agent or human developer
Service mesh C2C transport layer (sockets, pipes)
Service registry Local agent registry (~/.c2c/agents.json)
Health check capability_query message
Circuit breaker Timeout + fallback in task_request
Saga pattern Multi-step handoff chain with compensation
Event bus Shared filesystem or pub/sub socket

The key insight: just as microservices decompose a monolith into specialized services that communicate through well-defined APIs, C2C decomposes a monolithic AI session into specialized agents that communicate through well-defined messages.

Designing the CLI-to-CLI Protocol (C2C)¶

Design Principles¶

C2C follows five principles, each drawn from decades of protocol design experience:

1. Unix Philosophy. Agents are processes. They communicate through text streams. They can be composed with pipes. security-agent | deploy-agent should be a valid expression, just as grep | sort | uniq is valid today.

2. Text-First (JSON over stdio). Every message is a single line of JSON written to stdout or a socket. No binary encoding, no protocol buffers, no XML. JSON is human-readable, debuggable with cat and jq, and supported by every programming language. When something goes wrong, you can read the messages.

3. Stateful Conversations. Unlike REST APIs, C2C conversations maintain state. A task_request creates a conversation context. Subsequent status_update messages refer back to it via correlation_id. This enables long-running tasks with progress reporting, unlike fire-and-forget protocols.

4. Authenticated and Auditable. Every message carries a signature. Every message is logged. In a world where AI agents can modify code, deploy services, and access production data, you need to know exactly which agent said what, when, and why. The audit trail is not optional.

5. Observable. Every message carries a trace_id that follows the entire operation across all agents involved. You can reconstruct the full decision chain: “The security agent found CVE-2026-1234, which triggered the dependency agent to check for patches, which triggered the deploy agent to block the release.” Distributed tracing for AI agents.

The C2C Message Format¶

Every C2C message follows this schema:

json
{
  "protocol": "c2c/1.0",
  "id": "msg_a1b2c3d4",
  "from": {
    "agent": "security-auditor",
    "session": "sess_abc123",
    "capabilities": ["code-analysis", "cve-lookup", "owasp-audit"]
  },
  "to": {
    "agent": "devops-deployer",
    "session": "sess_def456"
  },
  "type": "task_request",
  "payload": {
    "task": "block_deployment",
    "context": {
      "reason": "Critical vulnerability CVE-2026-1234 in auth module",
      "severity": "critical",
      "files_affected": ["src/auth/login.py", "src/auth/oauth.py"],
      "evidence": {
        "cwe": "CWE-287",
        "description": "Improper authentication allows bypass via crafted JWT"
      }
    },
    "priority": "critical",
    "timeout_ms": 30000
  },
  "metadata": {
    "correlation_id": "corr_xyz789",
    "trace_id": "trace_release_v2.1.0",
    "timestamp": "2026-04-01T14:30:00Z",
    "signature": "sha256:e3b0c44298fc1c149afbf4c8996fb924...",
    "reply_to": null,
    "ttl_ms": 60000
  }
}

Let us walk through every field.

protocol — Version string. Agents check this first and reject messages with incompatible versions. Semantic versioning: c2c/1.0 is compatible with c2c/1.x but not c2c/2.0.

id — Unique message identifier. Used for deduplication and reply targeting. Format: msg_ prefix followed by a random alphanumeric string.

from — The sending agent’s identity. Includes the agent’s role name, session identifier (unique per process), and declared capabilities. Capabilities are how agents discover what each other can do without requiring a central registry.

to — The target agent. Can be a specific session (unicast) or just an agent role (anycast — any agent with that role can handle it). When session is null, the transport layer routes to the first available agent with the matching role.

type — One of six message types (detailed below).

payload — Type-specific content. For task_request, this contains the task name, context, priority, and timeout. For task_result, it contains the outcome and any artifacts.

metadata — Cross-cutting concerns. correlation_id links related messages into a conversation. trace_id links related conversations into an operation. signature is an HMAC-SHA256 of the message body for authentication. reply_to references the id of the message being replied to. ttl_ms sets an expiration — messages older than their TTL are dropped.

The Six Message Types¶

C2C defines six message types. This number is deliberate — enough to model real coordination patterns, few enough to implement in an afternoon.

1. capability_query — “What can you do?”

An agent asks another agent (or broadcasts to all agents) what capabilities they support. This is service discovery.

json
{
  "type": "capability_query",
  "payload": {
    "query": "Can you perform dependency vulnerability scanning?",
    "required_capabilities": ["cve-lookup", "dependency-tree"]
  }
}

The response is a capability_query with reply_to set:

json
{
  "type": "capability_query",
  "payload": {
    "capabilities": ["cve-lookup", "dependency-tree", "license-audit", "sbom-generation"],
    "status": "available",
    "load": 0.3
  },
  "metadata": {
    "reply_to": "msg_original_query_id"
  }
}

2. task_request — “Do this work.”

One agent delegates a task to another. This is the core coordination primitive.

json
{
  "type": "task_request",
  "payload": {
    "task": "audit_dependencies",
    "context": {
      "project_path": "/Users/dev/myapp",
      "focus": "production dependencies only",
      "ignore": ["dev-dependencies", "test-fixtures"]
    },
    "priority": "high",
    "timeout_ms": 120000,
    "callback": "task_result"
  }
}

The priority field accepts four levels: critical (blocks other work), high (should be next), normal (queued), low (background). Agents use priority to order their task queues.

3. task_result — “Here’s what I found.”

The response to a task_request. Contains the outcome, any artifacts produced, and status.

json
{
  "type": "task_result",
  "payload": {
    "status": "completed",
    "result": {
      "vulnerabilities_found": 3,
      "critical": 1,
      "high": 2,
      "details": [
        {
          "package": "jsonwebtoken@8.5.1",
          "cve": "CVE-2026-1234",
          "severity": "critical",
          "fix": "upgrade to jsonwebtoken@9.0.0"
        }
      ]
    },
    "artifacts": [
      {"type": "file", "path": "/tmp/c2c/audit-report.json"},
      {"type": "inline", "format": "markdown", "content": "## Audit Summary\n..."}
    ],
    "duration_ms": 45000
  },
  "metadata": {
    "reply_to": "msg_original_task_request_id"
  }
}

The status field accepts: completed, failed, partial (some results available, others errored), rejected (agent cannot or will not perform this task), and timeout (agent exceeded its own deadline).

4. status_update — “Here’s where I am.”

Progress reporting for long-running tasks. Enables the requesting agent (or a human observer) to monitor work without polling.

json
{
  "type": "status_update",
  "payload": {
    "progress": 0.65,
    "phase": "scanning_transitive_dependencies",
    "message": "Analyzed 847 of 1,302 packages",
    "estimated_remaining_ms": 25000
  }
}

5. handoff — “You take over from here.”

One agent transfers ownership of a task to another. This is different from task_request — in a handoff, the original agent is done with its part and passes accumulated context to the next agent in a chain.

json
{
  "type": "handoff",
  "payload": {
    "task": "deploy_with_fixes",
    "accumulated_context": {
      "original_task": "release_v2.1.0",
      "security_audit": "passed_with_fixes",
      "fixes_applied": [
        {"file": "src/auth/login.py", "commit": "abc123"},
        {"file": "package.json", "commit": "def456"}
      ],
      "tests": "all_passing",
      "changelog": "generated at /tmp/c2c/changelog.md"
    },
    "chain_position": 4,
    "chain_total": 5,
    "compensation": {
      "on_failure": "revert_commits",
      "commits_to_revert": ["abc123", "def456"]
    }
  }
}

The compensation field is critical. It tells the receiving agent what to undo if the task fails — the saga pattern applied to agent coordination.

6. negotiate — “Let’s agree on terms.”

Before committing to a task, agents can negotiate parameters: resource limits, output format, priority, deadlines, and which capabilities will be used.

json
{
  "type": "negotiate",
  "payload": {
    "proposal": {
      "max_files_to_analyze": 50,
      "output_format": "sarif",
      "model_to_use": "sonnet",
      "max_cost_usd": 0.50
    },
    "constraints": {
      "must_complete_before": "2026-04-01T15:00:00Z",
      "must_not_modify_files": true
    }
  }
}

Negotiation is optional but valuable for resource-constrained environments. If you have a limited API budget, you want agents to agree on cost limits before starting expensive analysis.

Transport Mechanisms¶

C2C is transport-agnostic. The message format stays the same; only the delivery mechanism changes. Four transports are defined, ordered from simplest to most capable:

1. stdio pipes — The simplest transport. One agent’s stdout connects to another’s stdin.

bash
security-agent --c2c | deploy-agent --c2c

Limitation: unidirectional. The deploy agent cannot send messages back to the security agent. Use for simple pipelines where data flows one way.

2. Unix domain sockets — Bidirectional, local, fast. Each agent listens on a socket file.

/tmp/c2c/security-auditor.sock
/tmp/c2c/devops-deployer.sock
/tmp/c2c/code-reviewer.sock

Agents connect to each other’s sockets to send messages. This is the recommended transport for local multi-agent systems. No network overhead, no firewall issues, no port conflicts.

3. Shared filesystem — The most resilient transport. Agents write messages as JSON files to a shared directory.

/tmp/c2c/mailbox/
├── security-auditor/
│   ├── inbox/
│   │   └── msg_001.json
│   └── outbox/
│       └── msg_002.json
├── devops-deployer/
│   ├── inbox/
│   └── outbox/
└── .registry.json

Each agent watches its inbox directory for new files (using fswatch, inotifywait, or polling). This transport survives agent crashes — messages persist on disk. It also works across machines via network filesystems, though latency is higher.

4. HTTP/WebSocket — For distributed agents across machines or in CI/CD pipelines. Follows the same message format, wrapped in HTTP POST or WebSocket frames. This is where C2C bridges to A2A — if you need to coordinate with cloud-hosted agents, upgrade from sockets to HTTP.

Transport Selection Guide¶

Scenario Transport Why
Two agents, simple pipeline stdio pipes Zero setup, Unix-native
3+ agents, local machine Unix sockets Bidirectional, fast, no config
Agents that crash/restart Shared filesystem Messages survive restarts
Cross-machine coordination HTTP/WebSocket Network-capable
CI/CD pipeline agents HTTP or filesystem Persistent, auditable

Implementing C2C in Practice¶

Reference Implementation: Python¶

The following is a complete, runnable C2C agent framework in Python. It handles message serialization, transport over Unix sockets, capability registration, and task dispatch.

python
"""C2C Protocol — Reference implementation in Python."""
from __future__ import annotations

import asyncio
import json
import hashlib
import hmac
import os
import time
import uuid
from dataclasses import dataclass, field, asdict
from enum import Enum
from pathlib import Path
from typing import Any, Callable, Awaitable


class MessageType(str, Enum):
    CAPABILITY_QUERY = "capability_query"
    TASK_REQUEST = "task_request"
    TASK_RESULT = "task_result"
    STATUS_UPDATE = "status_update"
    HANDOFF = "handoff"
    NEGOTIATE = "negotiate"


class Priority(str, Enum):
    CRITICAL = "critical"
    HIGH = "high"
    NORMAL = "normal"
    LOW = "low"


@dataclass
class AgentIdentity:
    agent: str
    session: str = field(default_factory=lambda: f"sess_{uuid.uuid4().hex[:12]}")
    capabilities: list[str] = field(default_factory=list)


@dataclass
class C2CMessage:
    protocol: str = "c2c/1.0"
    id: str = field(default_factory=lambda: f"msg_{uuid.uuid4().hex[:12]}")
    from_agent: AgentIdentity | None = None
    to_agent: AgentIdentity | None = None
    type: MessageType = MessageType.TASK_REQUEST
    payload: dict[str, Any] = field(default_factory=dict)
    metadata: dict[str, Any] = field(default_factory=dict)

    def __post_init__(self) -> None:
        if "timestamp" not in self.metadata:
            self.metadata["timestamp"] = time.strftime("%Y-%m-%dT%H:%M:%SZ", time.gmtime())
        if "correlation_id" not in self.metadata:
            self.metadata["correlation_id"] = f"corr_{uuid.uuid4().hex[:12]}"

    def to_json(self) -> str:
        data = {
            "protocol": self.protocol,
            "id": self.id,
            "from": asdict(self.from_agent) if self.from_agent else None,
            "to": asdict(self.to_agent) if self.to_agent else None,
            "type": self.type.value,
            "payload": self.payload,
            "metadata": self.metadata,
        }
        return json.dumps(data, separators=(",", ":"))

    @classmethod
    def from_json(cls, raw: str) -> C2CMessage:
        data = json.loads(raw)
        msg = cls(
            protocol=data["protocol"],
            id=data["id"],
            from_agent=AgentIdentity(**data["from"]) if data.get("from") else None,
            to_agent=AgentIdentity(**data["to"]) if data.get("to") else None,
            type=MessageType(data["type"]),
            payload=data.get("payload", {}),
            metadata=data.get("metadata", {}),
        )
        return msg

    def sign(self, secret: str) -> None:
        body = json.dumps(self.payload, sort_keys=True, separators=(",", ":"))
        sig = hmac.new(secret.encode(), body.encode(), hashlib.sha256).hexdigest()
        self.metadata["signature"] = f"sha256:{sig}"

    def verify(self, secret: str) -> bool:
        expected_sig = self.metadata.get("signature", "")
        body = json.dumps(self.payload, sort_keys=True, separators=(",", ":"))
        computed = hmac.new(secret.encode(), body.encode(), hashlib.sha256).hexdigest()
        return hmac.compare_digest(expected_sig, f"sha256:{computed}")


TaskHandler = Callable[[C2CMessage], Awaitable[C2CMessage]]


class C2CAgent:
    """A CLI agent that speaks the C2C protocol over Unix domain sockets."""

    def __init__(
        self,
        name: str,
        capabilities: list[str],
        secret: str = "",
        socket_dir: str = "/tmp/c2c",
    ) -> None:
        self.identity = AgentIdentity(agent=name, capabilities=capabilities)
        self.secret = secret or os.environ.get("C2C_SECRET", "dev-secret")
        self.socket_dir = Path(socket_dir)
        self.socket_path = self.socket_dir / f"{name}.sock"
        self.handlers: dict[str, TaskHandler] = {}
        self.message_log: list[C2CMessage] = []
        self._server: asyncio.Server | None = None

    def on_task(self, task_name: str) -> Callable[[TaskHandler], TaskHandler]:
        """Decorator to register a handler for a specific task type."""
        def decorator(fn: TaskHandler) -> TaskHandler:
            self.handlers[task_name] = fn
            return fn
        return decorator

    async def start(self) -> None:
        """Start listening for incoming C2C messages on a Unix socket."""
        self.socket_dir.mkdir(parents=True, exist_ok=True)
        if self.socket_path.exists():
            self.socket_path.unlink()

        self._server = await asyncio.start_unix_server(
            self._handle_connection, path=str(self.socket_path)
        )
        self._register()
        print(f"[C2C] {self.identity.agent} listening on {self.socket_path}")

    async def stop(self) -> None:
        """Gracefully shut down the agent."""
        if self._server:
            self._server.close()
            await self._server.wait_closed()
        if self.socket_path.exists():
            self.socket_path.unlink()
        self._deregister()

    def _register(self) -> None:
        """Register this agent in the local C2C registry."""
        registry_path = self.socket_dir / "registry.json"
        registry: dict[str, Any] = {}
        if registry_path.exists():
            registry = json.loads(registry_path.read_text())
        registry[self.identity.agent] = {
            "session": self.identity.session,
            "capabilities": self.identity.capabilities,
            "socket": str(self.socket_path),
            "pid": os.getpid(),
            "started_at": time.strftime("%Y-%m-%dT%H:%M:%SZ", time.gmtime()),
        }
        registry_path.write_text(json.dumps(registry, indent=2))

    def _deregister(self) -> None:
        """Remove this agent from the local registry."""
        registry_path = self.socket_dir / "registry.json"
        if registry_path.exists():
            registry = json.loads(registry_path.read_text())
            registry.pop(self.identity.agent, None)
            registry_path.write_text(json.dumps(registry, indent=2))

    async def _handle_connection(
        self, reader: asyncio.StreamReader, writer: asyncio.StreamWriter
    ) -> None:
        """Handle an incoming connection from another agent."""
        try:
            data = await asyncio.wait_for(reader.readline(), timeout=30.0)
            if not data:
                return

            msg = C2CMessage.from_json(data.decode().strip())
            self.message_log.append(msg)

            if self.secret and not msg.verify(self.secret):
                response = self._error_response(msg, "signature_invalid")
            elif msg.type == MessageType.CAPABILITY_QUERY:
                response = self._handle_capability_query(msg)
            elif msg.type == MessageType.TASK_REQUEST:
                response = await self._handle_task_request(msg)
            elif msg.type == MessageType.HANDOFF:
                response = await self._handle_task_request(msg)
            else:
                response = self._error_response(msg, "unsupported_type")

            response.sign(self.secret)
            self.message_log.append(response)
            writer.write((response.to_json() + "\n").encode())
            await writer.drain()
        except asyncio.TimeoutError:
            pass
        except Exception as exc:
            error_msg = self._make_message(
                MessageType.TASK_RESULT,
                payload={"status": "failed", "error": str(exc)},
            )
            writer.write((error_msg.to_json() + "\n").encode())
            await writer.drain()
        finally:
            writer.close()
            await writer.wait_closed()

    def _handle_capability_query(self, msg: C2CMessage) -> C2CMessage:
        required = msg.payload.get("required_capabilities", [])
        supported = [c for c in required if c in self.identity.capabilities]
        return self._make_message(
            MessageType.CAPABILITY_QUERY,
            payload={
                "capabilities": self.identity.capabilities,
                "matched": supported,
                "status": "available" if len(supported) == len(required) else "partial",
            },
            reply_to=msg.id,
            correlation_id=msg.metadata.get("correlation_id"),
        )

    async def _handle_task_request(self, msg: C2CMessage) -> C2CMessage:
        task_name = msg.payload.get("task", "")
        handler = self.handlers.get(task_name)
        if not handler:
            return self._error_response(msg, f"no_handler_for_{task_name}")

        try:
            return await handler(msg)
        except Exception as exc:
            return self._make_message(
                MessageType.TASK_RESULT,
                payload={"status": "failed", "error": str(exc), "task": task_name},
                reply_to=msg.id,
                correlation_id=msg.metadata.get("correlation_id"),
            )

    def _make_message(
        self,
        msg_type: MessageType,
        payload: dict[str, Any],
        reply_to: str | None = None,
        correlation_id: str | None = None,
        to_agent: AgentIdentity | None = None,
    ) -> C2CMessage:
        metadata: dict[str, Any] = {}
        if reply_to:
            metadata["reply_to"] = reply_to
        if correlation_id:
            metadata["correlation_id"] = correlation_id
        return C2CMessage(
            from_agent=self.identity,
            to_agent=to_agent,
            type=msg_type,
            payload=payload,
            metadata=metadata,
        )

    def _error_response(self, original: C2CMessage, error: str) -> C2CMessage:
        return self._make_message(
            MessageType.TASK_RESULT,
            payload={"status": "rejected", "error": error},
            reply_to=original.id,
            correlation_id=original.metadata.get("correlation_id"),
        )

    async def send(self, target_agent: str, message: C2CMessage) -> C2CMessage | None:
        """Send a message to another agent and wait for the response."""
        registry_path = self.socket_dir / "registry.json"
        if not registry_path.exists():
            raise ConnectionError("No C2C registry found")

        registry = json.loads(registry_path.read_text())
        if target_agent not in registry:
            raise ConnectionError(f"Agent '{target_agent}' not registered")

        target_socket = registry[target_agent]["socket"]
        message.from_agent = self.identity
        message.to_agent = AgentIdentity(
            agent=target_agent,
            session=registry[target_agent]["session"],
        )
        message.sign(self.secret)

        reader, writer = await asyncio.open_unix_connection(target_socket)
        try:
            writer.write((message.to_json() + "\n").encode())
            await writer.drain()

            timeout_ms = message.payload.get("timeout_ms", 30000)
            data = await asyncio.wait_for(
                reader.readline(), timeout=timeout_ms / 1000
            )
            if data:
                response = C2CMessage.from_json(data.decode().strip())
                self.message_log.append(response)
                return response
            return None
        finally:
            writer.close()
            await writer.wait_closed()

This is approximately 200 lines of production-quality Python. Let us look at what it provides:

  • C2CMessage — Full message serialization, deserialization, HMAC signing, and verification.
  • C2CAgent — A complete agent that listens on a Unix socket, registers itself in a local registry, dispatches tasks to handlers, and sends messages to other agents.
  • @agent.on_task("task_name") — Decorator-based handler registration, similar to Flask route decorators.
  • Registry — A shared JSON file at /tmp/c2c/registry.json that all agents use for discovery.

Here is how you build a security auditor agent using this framework:

python
"""Security auditor agent — scans code for vulnerabilities."""
import asyncio
from c2c import C2CAgent, C2CMessage, MessageType

agent = C2CAgent(
    name="security-auditor",
    capabilities=["code-analysis", "cve-lookup", "owasp-audit"],
)

@agent.on_task("scan_for_vulnerabilities")
async def scan(msg: C2CMessage) -> C2CMessage:
    project_path = msg.payload["context"]["project_path"]
    # ... perform actual scanning logic here ...
    return agent._make_message(
        MessageType.TASK_RESULT,
        payload={
            "status": "completed",
            "result": {
                "vulnerabilities_found": 2,
                "critical": 1,
                "details": [{"file": "auth.py", "line": 42, "cwe": "CWE-287"}],
            },
        },
        reply_to=msg.id,
        correlation_id=msg.metadata.get("correlation_id"),
    )

async def main() -> None:
    await agent.start()
    try:
        await asyncio.Event().wait()  # Run until interrupted
    finally:
        await agent.stop()

if __name__ == "__main__":
    asyncio.run(main())

Reference Implementation: TypeScript¶

The same framework in TypeScript, for teams that prefer Node.js:

typescript
import { createServer, createConnection } from "net";
import { randomUUID } from "crypto";
import { createHmac, timingSafeEqual } from "crypto";
import { readFileSync, writeFileSync, mkdirSync, existsSync, unlinkSync } from "fs";
import { join } from "path";

// --- Types ---

type MessageType =
  | "capability_query"
  | "task_request"
  | "task_result"
  | "status_update"
  | "handoff"
  | "negotiate";

interface AgentIdentity {
  agent: string;
  session: string;
  capabilities: string[];
}

interface C2CMessage {
  protocol: "c2c/1.0";
  id: string;
  from: AgentIdentity | null;
  to: AgentIdentity | null;
  type: MessageType;
  payload: Record<string, unknown>;
  metadata: Record<string, unknown>;
}

type TaskHandler = (msg: C2CMessage) => Promise<C2CMessage>;

// --- Message Helpers ---

function createMessage(
  fromAgent: AgentIdentity,
  type: MessageType,
  payload: Record<string, unknown>,
  opts: { replyTo?: string; correlationId?: string; toAgent?: AgentIdentity } = {}
): C2CMessage {
  return {
    protocol: "c2c/1.0",
    id: `msg_${randomUUID().replace(/-/g, "").slice(0, 12)}`,
    from: fromAgent,
    to: opts.toAgent ?? null,
    type,
    payload,
    metadata: {
      timestamp: new Date().toISOString(),
      correlation_id: opts.correlationId ?? `corr_${randomUUID().replace(/-/g, "").slice(0, 12)}`,
      ...(opts.replyTo ? { reply_to: opts.replyTo } : {}),
    },
  };
}

function signMessage(msg: C2CMessage, secret: string): void {
  const body = JSON.stringify(msg.payload, Object.keys(msg.payload).sort());
  const sig = createHmac("sha256", secret).update(body).digest("hex");
  msg.metadata.signature = `sha256:${sig}`;
}

function verifyMessage(msg: C2CMessage, secret: string): boolean {
  const expected = msg.metadata.signature as string;
  if (!expected) return false;
  const body = JSON.stringify(msg.payload, Object.keys(msg.payload).sort());
  const computed = `sha256:${createHmac("sha256", secret).update(body).digest("hex")}`;
  return timingSafeEqual(Buffer.from(expected), Buffer.from(computed));
}

// --- Agent ---

class C2CAgent {
  readonly identity: AgentIdentity;
  private readonly secret: string;
  private readonly socketDir: string;
  private readonly socketPath: string;
  private readonly handlers = new Map<string, TaskHandler>();
  private server: ReturnType<typeof createServer> | null = null;

  constructor(name: string, capabilities: string[], secret?: string, socketDir?: string) {
    this.identity = {
      agent: name,
      session: `sess_${randomUUID().replace(/-/g, "").slice(0, 12)}`,
      capabilities,
    };
    this.secret = secret ?? process.env.C2C_SECRET ?? "dev-secret";
    this.socketDir = socketDir ?? "/tmp/c2c";
    this.socketPath = join(this.socketDir, `${name}.sock`);
  }

  onTask(taskName: string, handler: TaskHandler): void {
    this.handlers.set(taskName, handler);
  }

  async start(): Promise<void> {
    mkdirSync(this.socketDir, { recursive: true });
    if (existsSync(this.socketPath)) unlinkSync(this.socketPath);

    return new Promise((resolve) => {
      this.server = createServer((socket) => {
        let buffer = "";
        socket.on("data", async (chunk) => {
          buffer += chunk.toString();
          const newlineIdx = buffer.indexOf("\n");
          if (newlineIdx === -1) return;

          const raw = buffer.slice(0, newlineIdx);
          buffer = buffer.slice(newlineIdx + 1);

          try {
            const msg: C2CMessage = JSON.parse(raw);
            const response = await this.dispatch(msg);
            signMessage(response, this.secret);
            socket.write(JSON.stringify(response) + "\n");
          } catch (err) {
            const errMsg = createMessage(this.identity, "task_result", {
              status: "failed",
              error: String(err),
            });
            socket.write(JSON.stringify(errMsg) + "\n");
          }
          socket.end();
        });
      });

      this.server.listen(this.socketPath, () => {
        this.register();
        console.log(`[C2C] ${this.identity.agent} listening on ${this.socketPath}`);
        resolve();
      });
    });
  }

  async stop(): Promise<void> {
    this.server?.close();
    if (existsSync(this.socketPath)) unlinkSync(this.socketPath);
    this.deregister();
  }

  async send(targetAgent: string, msg: C2CMessage): Promise<C2CMessage> {
    const registryPath = join(this.socketDir, "registry.json");
    const registry = JSON.parse(readFileSync(registryPath, "utf-8"));
    const target = registry[targetAgent];
    if (!target) throw new Error(`Agent '${targetAgent}' not registered`);

    msg.from = this.identity;
    msg.to = { agent: targetAgent, session: target.session, capabilities: [] };
    signMessage(msg, this.secret);

    return new Promise((resolve, reject) => {
      const socket = createConnection(target.socket, () => {
        socket.write(JSON.stringify(msg) + "\n");
      });
      let buffer = "";
      socket.on("data", (chunk) => {
        buffer += chunk.toString();
        const idx = buffer.indexOf("\n");
        if (idx !== -1) {
          resolve(JSON.parse(buffer.slice(0, idx)));
          socket.end();
        }
      });
      socket.on("error", reject);
      setTimeout(() => reject(new Error("C2C send timeout")), 30_000);
    });
  }

  private async dispatch(msg: C2CMessage): Promise<C2CMessage> {
    if (this.secret && !verifyMessage(msg, this.secret)) {
      return createMessage(this.identity, "task_result", {
        status: "rejected",
        error: "signature_invalid",
      }, { replyTo: msg.id });
    }

    if (msg.type === "capability_query") {
      const required = (msg.payload.required_capabilities as string[]) ?? [];
      const matched = required.filter((c) => this.identity.capabilities.includes(c));
      return createMessage(this.identity, "capability_query", {
        capabilities: this.identity.capabilities,
        matched,
        status: matched.length === required.length ? "available" : "partial",
      }, { replyTo: msg.id, correlationId: msg.metadata.correlation_id as string });
    }

    if (msg.type === "task_request" || msg.type === "handoff") {
      const taskName = msg.payload.task as string;
      const handler = this.handlers.get(taskName);
      if (!handler) {
        return createMessage(this.identity, "task_result", {
          status: "rejected",
          error: `no_handler_for_${taskName}`,
        }, { replyTo: msg.id });
      }
      return handler(msg);
    }

    return createMessage(this.identity, "task_result", {
      status: "rejected",
      error: "unsupported_type",
    }, { replyTo: msg.id });
  }

  private register(): void {
    const registryPath = join(this.socketDir, "registry.json");
    const registry = existsSync(registryPath)
      ? JSON.parse(readFileSync(registryPath, "utf-8"))
      : {};
    registry[this.identity.agent] = {
      session: this.identity.session,
      capabilities: this.identity.capabilities,
      socket: this.socketPath,
      pid: process.pid,
      started_at: new Date().toISOString(),
    };
    writeFileSync(registryPath, JSON.stringify(registry, null, 2));
  }

  private deregister(): void {
    const registryPath = join(this.socketDir, "registry.json");
    if (!existsSync(registryPath)) return;
    const registry = JSON.parse(readFileSync(registryPath, "utf-8"));
    delete registry[this.identity.agent];
    writeFileSync(registryPath, JSON.stringify(registry, null, 2));
  }
}

export { C2CAgent, C2CMessage, createMessage, signMessage, verifyMessage };

Integrating C2C with Claude Code via MCP Bridge¶

The most practical path to C2C today is building an MCP server that acts as a bridge. Claude Code already speaks MCP. If your MCP server speaks C2C on the other side, Claude Code gains agent-to-agent communication without any modifications to Claude Code itself.

python
"""MCP-to-C2C bridge — lets Claude Code send C2C messages as MCP tool calls."""
from __future__ import annotations

import asyncio
import json
from pathlib import Path

from mcp.server.lowlevel import Server
from mcp.types import Tool, TextContent

from c2c import C2CAgent, C2CMessage, MessageType

app = Server("c2c-bridge")
bridge_agent = C2CAgent(name="claude-code-bridge", capabilities=["mcp-bridge"])

@app.list_tools()
async def list_tools() -> list[Tool]:
    return [
        Tool(
            name="c2c_discover",
            description="Discover available C2C agents and their capabilities",
            inputSchema={
                "type": "object",
                "properties": {},
                "required": [],
            },
        ),
        Tool(
            name="c2c_send_task",
            description="Send a task request to another C2C agent",
            inputSchema={
                "type": "object",
                "properties": {
                    "target_agent": {"type": "string", "description": "Name of the target agent"},
                    "task": {"type": "string", "description": "Task name to execute"},
                    "context": {"type": "object", "description": "Task context and parameters"},
                    "priority": {"type": "string", "enum": ["critical", "high", "normal", "low"]},
                },
                "required": ["target_agent", "task"],
            },
        ),
    ]

@app.call_tool()
async def call_tool(name: str, arguments: dict) -> list[TextContent]:
    if name == "c2c_discover":
        registry_path = Path("/tmp/c2c/registry.json")
        if not registry_path.exists():
            return [TextContent(type="text", text="No C2C agents registered.")]
        registry = json.loads(registry_path.read_text())
        return [TextContent(type="text", text=json.dumps(registry, indent=2))]

    if name == "c2c_send_task":
        msg = C2CMessage(
            type=MessageType.TASK_REQUEST,
            payload={
                "task": arguments["task"],
                "context": arguments.get("context", {}),
                "priority": arguments.get("priority", "normal"),
                "timeout_ms": 60000,
            },
        )
        response = await bridge_agent.send(arguments["target_agent"], msg)
        if response:
            return [TextContent(type="text", text=json.dumps(response.payload, indent=2))]
        return [TextContent(type="text", text="No response from agent.")]

    return [TextContent(type="text", text=f"Unknown tool: {name}")]

Protocol Patterns¶

Six coordination patterns emerge from the C2C primitives. Each solves a different class of multi-agent problem.

1. Request-Response¶

The simplest pattern. One agent asks another to perform a task and waits for the result.

┌─────────────┐  task_request   ┌──────────────┐
│ Code Review  │ ──────────────►│ Security Scan │
│    Agent     │◄────────────── │    Agent      │
└─────────────┘  task_result    └──────────────┘

Use when: one agent needs a specific capability it does not have. The code review agent cannot scan for CVEs, so it asks the security agent.

python
# Code review agent delegates CVE scanning
msg = C2CMessage(
    type=MessageType.TASK_REQUEST,
    payload={
        "task": "scan_for_cves",
        "context": {"files": ["requirements.txt", "package-lock.json"]},
        "priority": "high",
    },
)
result = await review_agent.send("security-auditor", msg)
cves = result.payload["result"]["vulnerabilities"]

2. Pipeline¶

Tasks flow through a chain of agents, each adding value. The output of one becomes the input of the next, using handoff messages.

┌──────────┐  handoff   ┌──────────┐  handoff   ┌──────────┐  handoff   ┌──────────┐
│ Security │ ──────────►│ Code     │ ──────────►│ Test     │ ──────────►│ Deploy   │
│  Audit   │            │  Fix     │            │  Runner  │            │  Agent   │
└──────────┘            └──────────┘            └──────────┘            └──────────┘

Use when: a workflow has sequential stages where each agent specializes in one phase. The release pipeline is the canonical example.

3. Fan-Out / Fan-In¶

One orchestrator agent dispatches tasks to multiple agents in parallel, then aggregates the results.

                          ┌──────────────┐
                     ┌───►│ Security     │───┐
                     │    │ Auditor      │   │
┌─────────────┐      │    └──────────────┘   │    ┌─────────────┐
│ Orchestrator │──────┤                       ├───►│ Orchestrator│
│              │      │    ┌──────────────┐   │    │ (aggregate) │
└─────────────┘      ├───►│ Dep Auditor  │───┤    └─────────────┘
                     │    └──────────────┘   │
                     │    ┌──────────────┐   │
                     └───►│ Perf Tester  │───┘
                          └──────────────┘

Use when: multiple independent analyses can run simultaneously. This is the most common pattern for pre-release validation.

python
# Fan-out: dispatch three tasks in parallel
tasks = [
    ("security-auditor", "scan_for_vulnerabilities"),
    ("dep-auditor", "audit_dependencies"),
    ("perf-tester", "run_benchmarks"),
]
coros = []
for target, task in tasks:
    msg = C2CMessage(
        type=MessageType.TASK_REQUEST,
        payload={"task": task, "context": {"project": "/app"}, "priority": "high"},
        metadata={"trace_id": "trace_release_v3"},
    )
    coros.append(orchestrator.send(target, msg))

# Fan-in: wait for all results
results = await asyncio.gather(*coros, return_exceptions=True)
all_passed = all(
    r.payload.get("status") == "completed"
    for r in results
    if isinstance(r, C2CMessage)
)

4. Saga (Compensating Transactions)¶

A multi-step workflow where each step can be undone if a later step fails. The handoff message’s compensation field carries rollback instructions.

Step 1: Apply security fix    → compensation: revert commit abc123
Step 2: Update dependencies   → compensation: revert commit def456
Step 3: Run integration tests → (fails!)
Step 4: Execute compensations → revert def456, revert abc123

Use when: a multi-agent workflow modifies shared state (code, deployments, databases) and needs to be atomic — either everything succeeds or everything rolls back.

5. Event-Driven¶

Agents publish events to a shared channel. Other agents subscribe to events they care about and react asynchronously. Implemented via the shared filesystem transport with a pub/sub directory structure.

/tmp/c2c/events/
├── vulnerability_found/
│   ├── subscriber_deploy_agent.json
│   └── subscriber_notification_agent.json
├── tests_passed/
│   └── subscriber_deploy_agent.json
└── deploy_complete/
    └── subscriber_monitoring_agent.json

Use when: agents need loose coupling. The security agent publishes “vulnerability found” without knowing or caring which agents react to it.

6. Consensus¶

Multiple agents analyze the same task independently, then compare results. An arbiter agent resolves disagreements. This pattern increases reliability for critical decisions.

┌──────────┐
│ Agent A   │──── analysis ────┐
└──────────┘                   │    ┌───────────┐
┌──────────┐                   ├───►│  Arbiter  │──► Final Decision
│ Agent B   │──── analysis ────┤    └───────────┘
└──────────┘                   │
┌──────────┐                   │
│ Agent C   │──── analysis ────┘
└──────────┘

Use when: the decision is high-stakes (production deployment, security assessment) and a single agent’s opinion is not sufficient. Three security agents independently auditing the same code, with a consensus threshold of 2/3 to approve.

Security and Governance¶

AI agents that coordinate autonomously introduce a new class of security challenges. A compromised agent in a C2C network could instruct other agents to deploy malicious code, exfiltrate data, or disable security controls. The security model must be rigorous.

Zero-Trust Between Agents¶

Every C2C agent treats every other agent as potentially compromised. This means:

1. Message Authentication. Every message is HMAC-SHA256 signed. The shared secret is distributed out-of-band (environment variable, secret manager, key file). Unsigned or incorrectly signed messages are rejected at the transport layer before reaching any handler.

2. Capability-Based Authorization. Agents declare what they can do. Agents verify that a requesting agent has the right to request a specific task. A code-review agent should not be able to request deployment. Authorization rules are declared in a policy file:

json
{
  "policies": [
    {
      "from": "security-auditor",
      "allowed_tasks": ["block_deployment", "request_review"],
      "denied_tasks": ["deploy", "delete_branch"]
    },
    {
      "from": "deploy-agent",
      "allowed_tasks": ["deploy", "rollback"],
      "requires_approval_from": ["security-auditor"]
    }
  ]
}

3. Input Validation. Every handler validates its input before processing. A task request with a project_path that traverses directories (../../../etc/passwd) is rejected. A context field containing shell metacharacters is sanitized or rejected.

Message Signing Deep Dive¶

The signing mechanism is straightforward but must be implemented correctly:

python
import hmac
import hashlib
import json

def sign_payload(payload: dict, secret: str) -> str:
    """Produce a deterministic HMAC-SHA256 signature of the payload."""
    # Canonical JSON: sorted keys, no whitespace
    canonical = json.dumps(payload, sort_keys=True, separators=(",", ":"))
    return hmac.new(
        secret.encode("utf-8"),
        canonical.encode("utf-8"),
        hashlib.sha256,
    ).hexdigest()

def verify_payload(payload: dict, signature: str, secret: str) -> bool:
    """Verify a signature using constant-time comparison."""
    expected = sign_payload(payload, secret)
    return hmac.compare_digest(signature, expected)

Critical implementation details:

  • Canonical JSON. Keys must be sorted and whitespace must be consistent. Otherwise the same payload produces different signatures on different machines.
  • Constant-time comparison. Use hmac.compare_digest, never ==. String equality comparison leaks timing information that can be used to forge signatures byte by byte.
  • Secret rotation. Support multiple active secrets during rotation periods. Accept messages signed with either the current or previous secret.

Rate Limiting¶

Without rate limiting, a compromised or malfunctioning agent can flood other agents with requests, creating a denial-of-service condition. Each agent enforces:

python
from collections import defaultdict
import time

class RateLimiter:
    def __init__(self, max_requests: int = 100, window_seconds: int = 60) -> None:
        self.max_requests = max_requests
        self.window = window_seconds
        self.requests: dict[str, list[float]] = defaultdict(list)

    def allow(self, agent_name: str) -> bool:
        now = time.monotonic()
        # Prune old entries
        self.requests[agent_name] = [
            t for t in self.requests[agent_name]
            if now - t < self.window
        ]
        if len(self.requests[agent_name]) >= self.max_requests:
            return False
        self.requests[agent_name].append(now)
        return True

Audit Logging¶

Every C2C message — sent and received — is logged to an append-only audit file. This log is critical for incident response (“which agent told the deploy agent to push to production at 3 AM?”) and for debugging coordination failures.

python
import json
from pathlib import Path
from datetime import datetime, timezone

class AuditLog:
    def __init__(self, log_dir: str = "/tmp/c2c/audit") -> None:
        self.log_dir = Path(log_dir)
        self.log_dir.mkdir(parents=True, exist_ok=True)
        today = datetime.now(timezone.utc).strftime("%Y-%m-%d")
        self.log_file = self.log_dir / f"c2c-audit-{today}.jsonl"

    def record(self, direction: str, msg: dict) -> None:
        entry = {
            "timestamp": datetime.now(timezone.utc).isoformat(),
            "direction": direction,  # "sent" or "received"
            "message_id": msg.get("id"),
            "from": msg.get("from", {}).get("agent"),
            "to": msg.get("to", {}).get("agent"),
            "type": msg.get("type"),
            "correlation_id": msg.get("metadata", {}).get("correlation_id"),
            "trace_id": msg.get("metadata", {}).get("trace_id"),
            "status": msg.get("payload", {}).get("status"),
        }
        with open(self.log_file, "a") as f:
            f.write(json.dumps(entry) + "\n")

The audit log format is JSONL (one JSON object per line), which is trivially parseable with jq:

bash
# All messages in the release trace
jq 'select(.trace_id == "trace_release_v3")' /tmp/c2c/audit/c2c-audit-2026-04-01.jsonl

# All rejected messages (potential attacks)
jq 'select(.status == "rejected")' /tmp/c2c/audit/c2c-audit-2026-04-01.jsonl

# Message flow for a specific correlation
jq 'select(.correlation_id == "corr_abc123")' /tmp/c2c/audit/c2c-audit-2026-04-01.jsonl

Sandboxing¶

Each C2C agent should run in a sandboxed environment that limits its blast radius:

Control Implementation
Filesystem Agent can only read/write within its designated project directory
Network Agent can only connect to its own socket and the registry
Processes Agent cannot spawn arbitrary subprocesses
Resources CPU and memory limits via cgroups or container constraints
Time Maximum execution time per task, enforced by the framework

In practice, the simplest sandboxing approach for local development is running each agent in a separate container or using macOS sandbox profiles:

bash
# Run a C2C agent in a container with limited access
docker run --rm \
  -v /tmp/c2c:/tmp/c2c \
  -v /path/to/project:/project:ro \
  --network none \
  --memory 512m \
  --cpus 1 \
  c2c-security-agent

The --network none flag is critical. A security auditor agent does not need network access to scan local code. If it is compromised, it cannot exfiltrate data.

Where C2C Meets Claude Code Today¶

The C2C protocol is forward-looking, but pieces of it are already implementable within Claude Code’s current architecture. Here is the practical roadmap:

Today (fully implementable): - MCP bridge server that routes tasks to external agents - Shared filesystem transport for agent coordination - Claude Code subagents (as described in Chapter 6) as lightweight C2C participants - Audit logging of all inter-agent communication

Near-term (requires minor tooling): - Unix socket transport with proper agent lifecycle management - Local agent registry with health checking - Rate limiting and capability-based authorization

Future (requires Claude Code changes): - Native C2C support in Claude Code’s agent framework - Built-in agent discovery and negotiation - First-class support for the saga pattern with automatic compensation

The key insight is that you do not need to wait for native support. The MCP bridge pattern lets you build C2C-capable systems today, using Claude Code as one participant in a larger multi-agent workflow. Start with the shared filesystem transport — it is the most resilient and easiest to debug. Graduate to Unix sockets when you need lower latency. Consider HTTP only when you need cross-machine coordination.

Chapter Summary¶

The CLI-to-CLI Protocol addresses the fundamental limitation of today’s AI agents: they work alone. By defining a simple, Unix-native message format with six message types (capability_query, task_request, task_result, status_update, handoff, negotiate), four transport mechanisms (stdio, Unix sockets, filesystem, HTTP), and a zero-trust security model, C2C enables the transition from monolithic AI sessions to specialized, coordinating agent systems.

The protocol’s power lies not in any single message type but in the patterns that emerge from their composition — pipelines, fan-out/fan-in, sagas, event-driven reactions, and consensus. These are the same patterns that transformed web development from monoliths to microservices, now applied to AI agents.

The reference implementations in this chapter — approximately 200 lines each in Python and TypeScript — are production-ready starting points. The MCP bridge integration means you can start building C2C-capable systems today, within Claude Code’s existing architecture, without waiting for native protocol support.

We are at the beginning of the multi-agent era. The agents are ready. The protocols are defined. The question is no longer whether AI agents will coordinate — it is how well they will coordinate, and which protocols will become the standards. C2C is our answer: simple enough to implement in an afternoon, powerful enough to orchestrate a fleet of specialized agents, and secure enough to trust with production workloads.

In the next chapter, we will build on C2C to construct complete multi-agent workflows — the orchestration layer that turns independent agents into a coordinated team.
