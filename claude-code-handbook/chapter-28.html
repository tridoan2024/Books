Part VIII: Specialized CLI Systems

Chapter 28: Specializing Claude Code at the Source Code Level¶ ~22 min read

Every chapter in this book so far has worked within the boundaries that Claude Code gives you. CLAUDE.md files, skills, rules, hooks, agents, MCP servers — these are the official extension points. They are powerful. For most teams, they are enough.

But some teams hit a wall.

You need a tool that participates in the agentic loop differently than any built-in tool can. You need permission semantics that don’t map to allow/deny lists. You need to strip away 80% of Claude Code’s capabilities and ship a focused product to a non-engineering team. You need telemetry, billing gates, or compliance logic wired into the core execution path.

When you reach that wall, there is one more layer: forking and modifying Claude Code’s TypeScript source code itself.

This chapter is not about fine-tuning models. It is not about writing better CLAUDE.md files. It is about reading, understanding, and modifying the actual TypeScript codebase that powers Claude Code — building specialized CLI tools that share its DNA but serve entirely different purposes.

The Five Customization Layers¶

Before you touch source code, understand the full spectrum of customization available to you. Each layer goes deeper. Each layer trades convenience for control.

Layer 1: CLAUDE.md (Zero Code)¶

The project instruction file. Loaded into context at session start. Controls communication style, coding standards, workflow preferences. You write natural language and Claude follows it.

What it can do: Change how Claude talks, what conventions it follows, which patterns it prefers. Reference architecture docs, import rules.

What it cannot do: Change what tools exist, modify permission logic, alter the execution loop, add new capabilities.

Sufficient when: Your team wants consistent behavior from stock Claude Code across a project.

Layer 2: Skills + Rules + Hooks (Configuration)¶

Skills package reusable workflows as slash commands. Rules enforce path-scoped coding standards. Hooks run shell scripts at lifecycle events — blocking destructive commands, auto-linting on save, scanning for secrets before commits.

What it can do: Create repeatable workflows, enforce guardrails, automate pre/post actions on tool calls, gate specific operations.

What it cannot do: Add new tool types, modify how the agentic loop decides what to do next, change the permission model fundamentally.

Sufficient when: You need automated enforcement, repeatable workflows, and lifecycle-event automation — but the existing tools and loop behavior work for you.

Layer 3: Custom Agents (Agent Definitions)¶

Agent markdown files in .claude/agents/ define specialized sub-instances with their own model, tool restrictions, and system prompts. Claude delegates to them for specific task types.

What it can do: Create specialists with constrained tool access, different models, focused system prompts. Run parallel investigations.

What it cannot do: Add tools that don’t already exist. Change how delegation works at the protocol level. Modify the agent execution engine.

Sufficient when: You need role-based specialization — a security reviewer on Opus, a doc writer on Sonnet — using existing tools.

Layer 4: MCP Servers (External Capabilities)¶

As discussed in Chapter 24, MCP servers add entirely new tools via JSON-RPC over stdio. You write a server in any language, register it in settings, and Claude Code treats its tools like built-in ones.

What it can do: Add arbitrary new capabilities — database queries, API integrations, custom analysis. Tools participate in Claude’s planning like any built-in tool.

What it cannot do: Modify the core loop’s decision-making. Change how permissions are evaluated. Alter the system prompt injection. Control billing or cost gates. Remove or replace built-in tools.

Sufficient when: You need new capabilities exposed to Claude, and the standard tool invocation pattern (Claude decides to call → tool executes → result returns) works.

Layer 5: Source Code Modification (This Chapter)¶

Fork the Claude Code TypeScript repository. Modify the tool registry, the agentic loop, the permission engine, the CLI entry point, the system prompt assembly. Build and distribute a custom binary.

What it can do: Anything. Add tools that interact with the loop differently. Implement cost gates, compliance checks, telemetry. Strip features. Rebrand. Ship a focused product.

What it costs: You own a fork. You must merge upstream changes. You need TypeScript expertise and a build/distribution pipeline.

When you need it:

  • Your tool needs to influence the loop’s next decision (not just return a result)
  • You need permission semantics beyond allow/deny (role-based, time-based, budget-based)
  • You are building a product on top of Claude Code for a specific audience
  • You need to remove capabilities (compliance requirement: no web access, no file writes outside a sandbox)
  • You need deep telemetry or billing integration in the execution path

The rest of this chapter lives in Layer 5.

Claude Code is a TypeScript application built on Node.js. The open-source repository contains roughly 500K+ lines of TypeScript, organized into a clear modular structure. Understanding this structure is prerequisite to modifying it safely.

The Tool Registry Pattern¶

Every tool in Claude Code — built-in and MCP-provided — goes through a central registry. This is the single most important pattern to understand for customization.

typescript
// src/tools/registry.ts (simplified)

export interface ToolDefinition {
  name: string;
  description: string;
  parameters: JsonSchema;
  execute: (params: ToolInput, context: ExecutionContext) => Promise<ToolResult>;
  isAvailable?: (context: ExecutionContext) => boolean;
  metadata?: ToolMetadata;
}

class ToolRegistry {
  private tools: Map<string, ToolDefinition> = new Map();

  register(tool: ToolDefinition): void {
    if (this.tools.has(tool.name)) {
      throw new Error(`Tool "${tool.name}" is already registered`);
    }
    this.tools.set(tool.name, tool);
  }

  get(name: string): ToolDefinition | undefined {
    return this.tools.get(name);
  }

  list(context: ExecutionContext): ToolDefinition[] {
    return Array.from(this.tools.values())
      .filter(tool => !tool.isAvailable || tool.isAvailable(context));
  }
}

export const toolRegistry = new ToolRegistry();

Key observations:

  • ToolDefinition is the interface every tool implements. Name, description, JSON Schema parameters, and an async execute function.
  • ExecutionContext carries session state — current directory, permissions, settings, conversation history.
  • isAvailable is an optional gate. A tool can hide itself based on context (e.g., a database tool only appears when a connection string is configured).
  • toolRegistry is a singleton. All tools register here at startup.

When Claude decides to use a tool, the engine calls toolRegistry.get(toolName), checks permissions, then calls tool.execute(params, context).

The Agentic Loop¶

The loop is the core of Claude Code. It is what makes Claude Code an agent rather than a chatbot.

typescript
// src/engine/AgentLoop.ts (simplified)

class AgentLoop {
  async run(userMessage: string, context: ExecutionContext): Promise<void> {
    // Add user message to conversation
    context.conversation.addUser(userMessage);

    while (true) {
      // 1. Send conversation to Claude API
      const response = await this.callClaude(context);

      // 2. If Claude responded with text only, we're done
      if (!response.hasToolCalls()) {
        context.conversation.addAssistant(response.text);
        break;
      }

      // 3. Process each tool call
      for (const toolCall of response.toolCalls) {
        // 3a. Check permissions
        const permitted = await this.permissionEngine.check(
          toolCall, context
        );
        if (!permitted) {
          context.conversation.addToolResult(toolCall.id, {
            error: "Permission denied"
          });
          continue;
        }

        // 3b. Run PreToolUse hooks
        const hookResult = await this.hooks.runPre(toolCall, context);
        if (hookResult.blocked) {
          context.conversation.addToolResult(toolCall.id, {
            error: hookResult.reason
          });
          continue;
        }

        // 3c. Execute the tool
        const result = await this.toolExecutor.execute(toolCall, context);

        // 3d. Run PostToolUse hooks
        await this.hooks.runPost(toolCall, result, context);

        // 3e. Add result to conversation
        context.conversation.addToolResult(toolCall.id, result);
      }

      // 4. Loop back — Claude sees the tool results and decides next action
    }
  }
}

The loop is simple in structure but rich in extension points:

  1. Before calling Claude — you can modify the context, inject system prompts, add constraints
  2. After Claude responds but before tool execution — permission checks, hook execution, custom gates
  3. During tool execution — the tool itself runs
  4. After tool execution — post-hooks, telemetry, result transformation
  5. Loop continuation decision — Claude decides whether to call more tools or respond with text

Every modification in this chapter targets one of these five points.

Adding Custom Tools to the Registry¶

The most common source-level modification is adding tools that cannot be implemented as MCP servers — tools that need access to internal engine state, tools that need to influence the loop’s behavior, or tools that need to be deeply integrated for performance reasons.

Tool Interface Deep Dive¶

A complete tool implementation follows this structure:

typescript
// src/tools/DatabaseQueryTool.ts

import { ToolDefinition, ToolInput, ToolResult } from "./registry";
import { ExecutionContext } from "../engine/types";
import { Pool } from "pg";

let pool: Pool | null = null;

function getPool(context: ExecutionContext): Pool {
  if (!pool) {
    const connectionString = context.env.DATABASE_URL;
    if (!connectionString) {
      throw new Error(
        "DATABASE_URL environment variable is required for database queries"
      );
    }
    pool = new Pool({
      connectionString,
      max: 5,
      idleTimeoutMillis: 30000,
      connectionTimeoutMillis: 5000,
    });
  }
  return pool;
}

export const DatabaseQueryTool: ToolDefinition = {
  name: "DatabaseQuery",

  description:
    "Execute a read-only SQL query against the project's PostgreSQL database. " +
    "Use this to inspect schema, check data, debug issues, or answer questions " +
    "about application state. Only SELECT queries are permitted.",

  parameters: {
    type: "object",
    properties: {
      query: {
        type: "string",
        description:
          "The SQL query to execute. Must be a SELECT statement. " +
          "Use LIMIT to avoid returning excessive rows.",
      },
      maxRows: {
        type: "number",
        description: "Maximum rows to return. Defaults to 100.",
        default: 100,
      },
    },
    required: ["query"],
  },

  isAvailable(context: ExecutionContext): boolean {
    return !!context.env.DATABASE_URL;
  },

  async execute(
    params: ToolInput,
    context: ExecutionContext
  ): Promise<ToolResult> {
    const query = params.query as string;
    const maxRows = (params.maxRows as number) || 100;

    // Safety: reject non-SELECT queries
    const normalized = query.trim().toUpperCase();
    if (
      !normalized.startsWith("SELECT") &&
      !normalized.startsWith("WITH") &&
      !normalized.startsWith("EXPLAIN")
    ) {
      return {
        error:
          "Only SELECT, WITH (CTE), and EXPLAIN queries are permitted. " +
          "This tool is read-only.",
      };
    }

    // Safety: reject dangerous patterns
    const dangerous = [
      /;\s*(DROP|DELETE|UPDATE|INSERT|ALTER|CREATE|TRUNCATE)/i,
      /INTO\s+OUTFILE/i,
      /LOAD_FILE/i,
    ];
    for (const pattern of dangerous) {
      if (pattern.test(query)) {
        return {
          error: "Query contains disallowed SQL operations.",
        };
      }
    }

    try {
      const db = getPool(context);
      const result = await db.query(`${query} LIMIT ${maxRows + 1}`);

      const truncated = result.rows.length > maxRows;
      const rows = truncated ? result.rows.slice(0, maxRows) : result.rows;

      return {
        content: JSON.stringify(
          {
            columns: result.fields.map((f) => f.name),
            rows,
            rowCount: rows.length,
            truncated,
            ...(truncated
              ? { note: `Results truncated to ${maxRows} rows` }
              : {}),
          },
          null,
          2
        ),
      };
    } catch (err) {
      const message = err instanceof Error ? err.message : String(err);
      return {
        error: `Query failed: ${message}`,
      };
    }
  },
};

Second Example: A Jira Integration Tool¶

Here is a tool that creates Jira tickets directly from Claude’s tool-calling interface, without shelling out to curl:

typescript
// src/tools/JiraTicketTool.ts

import { ToolDefinition, ToolInput, ToolResult } from "./registry";
import { ExecutionContext } from "../engine/types";

export const JiraTicketTool: ToolDefinition = {
  name: "JiraTicket",

  description:
    "Create a Jira ticket in the project's Jira instance. Use this when " +
    "you've identified a bug, security issue, or task that needs tracking. " +
    "Returns the ticket key (e.g., PROJ-123) and URL.",

  parameters: {
    type: "object",
    properties: {
      project: {
        type: "string",
        description: "Jira project key (e.g., 'PROJ', 'SECOPS')",
      },
      summary: {
        type: "string",
        description: "Ticket title. Keep it concise and actionable.",
      },
      description: {
        type: "string",
        description:
          "Detailed description in Jira wiki markup or plain text.",
      },
      issueType: {
        type: "string",
        enum: ["Bug", "Task", "Story", "Security"],
        description: "Issue type. Defaults to 'Task'.",
        default: "Task",
      },
      priority: {
        type: "string",
        enum: ["Critical", "High", "Medium", "Low"],
        description: "Priority level. Defaults to 'Medium'.",
        default: "Medium",
      },
      labels: {
        type: "array",
        items: { type: "string" },
        description: "Labels to apply to the ticket.",
      },
    },
    required: ["project", "summary", "description"],
  },

  isAvailable(context: ExecutionContext): boolean {
    return !!context.env.JIRA_BASE_URL && !!context.env.JIRA_API_TOKEN;
  },

  async execute(
    params: ToolInput,
    context: ExecutionContext
  ): Promise<ToolResult> {
    const baseUrl = context.env.JIRA_BASE_URL;
    const token = context.env.JIRA_API_TOKEN;
    const email = context.env.JIRA_EMAIL;

    if (!baseUrl || !token || !email) {
      return {
        error:
          "JIRA_BASE_URL, JIRA_API_TOKEN, and JIRA_EMAIL environment " +
          "variables are required.",
      };
    }

    const payload = {
      fields: {
        project: { key: params.project as string },
        summary: params.summary as string,
        description: params.description as string,
        issuetype: { name: (params.issueType as string) || "Task" },
        priority: { name: (params.priority as string) || "Medium" },
        ...(params.labels
          ? { labels: params.labels as string[] }
          : {}),
      },
    };

    try {
      const response = await fetch(`${baseUrl}/rest/api/2/issue`, {
        method: "POST",
        headers: {
          "Content-Type": "application/json",
          Authorization: `Basic ${Buffer.from(`${email}:${token}`).toString("base64")}`,
        },
        body: JSON.stringify(payload),
      });

      if (!response.ok) {
        const errorBody = await response.text();
        return {
          error: `Jira API error (${response.status}): ${errorBody}`,
        };
      }

      const result = await response.json();
      return {
        content: JSON.stringify(
          {
            key: result.key,
            url: `${baseUrl}/browse/${result.key}`,
            summary: params.summary,
            status: "Created",
          },
          null,
          2
        ),
      };
    } catch (err) {
      const message = err instanceof Error ? err.message : String(err);
      return { error: `Failed to create Jira ticket: ${message}` };
    }
  },
};

Why Not MCP?¶

A fair question: both of these examples could be MCP servers. The reasons to implement them as built-in tools instead:

Factor MCP Server Built-in Tool
Access to engine internals No Yes
Startup latency Separate process spawn None (loaded with binary)
Distribution Must ship separately Bundled in your fork
Connection pooling Manages own lifecycle Shares process lifecycle
Error handling JSON-RPC serialization Native TypeScript exceptions
Testing Integration tests only Unit tests with mocked context

If the tool is a core part of your specialized CLI and needs to be fast, reliable, and deeply integrated — build it in. If it is a pluggable capability that different users might swap out — use MCP.

Modifying the Agentic Loop¶

Adding tools is the straightforward case. The more powerful modifications happen inside the loop itself — changing how Claude Code decides what to do, adding gates between tool calls, injecting logic that the model never sees.

Adding a Cost Gate¶

Real-world scenario: your team uses Claude Code with an Anthropic API key, and you need to prevent runaway costs. You want the agent to pause and ask for confirmation after spending more than a configured amount in a single session.

typescript
// src/engine/CostGate.ts

export interface CostGateConfig {
  maxSessionCostUsd: number;
  warningThresholdPercent: number; // warn at this % of max
}

export class CostGate {
  private totalInputTokens: number = 0;
  private totalOutputTokens: number = 0;
  private readonly config: CostGateConfig;

  // Pricing per million tokens (update as pricing changes)
  private static readonly INPUT_COST_PER_M = 15.0;   // Opus input
  private static readonly OUTPUT_COST_PER_M = 75.0;  // Opus output

  constructor(config: CostGateConfig) {
    this.config = config;
  }

  recordUsage(inputTokens: number, outputTokens: number): void {
    this.totalInputTokens += inputTokens;
    this.totalOutputTokens += outputTokens;
  }

  getCurrentCostUsd(): number {
    const inputCost =
      (this.totalInputTokens / 1_000_000) * CostGate.INPUT_COST_PER_M;
    const outputCost =
      (this.totalOutputTokens / 1_000_000) * CostGate.OUTPUT_COST_PER_M;
    return inputCost + outputCost;
  }

  shouldWarn(): boolean {
    const threshold =
      this.config.maxSessionCostUsd *
      (this.config.warningThresholdPercent / 100);
    return this.getCurrentCostUsd() >= threshold;
  }

  shouldBlock(): boolean {
    return this.getCurrentCostUsd() >= this.config.maxSessionCostUsd;
  }

  getSummary(): string {
    const cost = this.getCurrentCostUsd().toFixed(4);
    const max = this.config.maxSessionCostUsd.toFixed(2);
    return (
      `Session cost: $${cost} / $${max} ` +
      `(${this.totalInputTokens.toLocaleString()} input, ` +
      `${this.totalOutputTokens.toLocaleString()} output tokens)`
    );
  }
}

Now wire it into the agentic loop:

typescript
// src/engine/AgentLoop.ts (modified)

class AgentLoop {
  private costGate: CostGate;

  constructor(config: LoopConfig) {
    // ... existing constructor ...
    this.costGate = new CostGate({
      maxSessionCostUsd: config.maxSessionCostUsd ?? 10.0,
      warningThresholdPercent: config.warningThresholdPercent ?? 80,
    });
  }

  async run(userMessage: string, context: ExecutionContext): Promise<void> {
    context.conversation.addUser(userMessage);

    while (true) {
      // Cost check BEFORE calling Claude API
      if (this.costGate.shouldBlock()) {
        context.conversation.addAssistant(
          `Session cost limit reached. ${this.costGate.getSummary()}. ` +
          `Use /cost-override to continue or start a new session.`
        );
        break;
      }

      if (this.costGate.shouldWarn()) {
        // Inject a system message that Claude sees
        context.conversation.addSystem(
          `WARNING: ${this.costGate.getSummary()}. ` +
          `Approaching session cost limit. Prefer concise responses.`
        );
      }

      const response = await this.callClaude(context);

      // Record token usage from API response
      this.costGate.recordUsage(
        response.usage.inputTokens,
        response.usage.outputTokens
      );

      // ... rest of the loop (tool execution, etc.) ...
    }
  }
}

This is something you cannot do with hooks, skills, or MCP servers. The cost gate needs to:

  1. Read token counts from the Claude API response (internal engine data)
  2. Inject system messages into the conversation (modify context)
  3. Break the loop (alter control flow)
  4. Run before the API call, not after a tool call (lifecycle point that hooks don’t cover)

Adding Domain-Specific Decision Points¶

Another common pattern: injecting logic between tool calls that steers Claude’s behavior without Claude knowing about it.

Example: in a security-focused fork, you want every file write to be preceded by an automatic security scan. Not a PostToolUse hook (which runs after the write) but a PreToolUse injection that adds the scan result to context so Claude can reconsider:

Custom Permission Modes¶

Claude Code’s default permission model is a list-based system: allow rules, deny rules, and ask-the-user for everything else. This works well for individual developers but breaks down in team and enterprise scenarios.

The Permission Engine¶

export type PermissionDecision = "allow" | "deny" | "ask";

export interface PermissionRule {
  pattern: string;   // e.g., "Bash(git diff *)"
  decision: PermissionDecision;
}

class PermissionEngine {
  private rules: PermissionRule[];

  evaluate(toolCall: ToolCall, context: ExecutionContext): PermissionDecision {
    // Deny rules first (highest priority)
    for (const rule of this.rules.filter((r) => r.decision === "deny")) {
      if (this.matches(toolCall, rule.pattern)) return "deny";
    }

    // Allow rules
    for (const rule of this.rules.filter((r) => r.decision === "allow")) {
      if (this.matches(toolCall, rule.pattern)) return "allow";
    }

    // Default: ask user
    return "ask";
  }

Role-Based Permission Mode¶

For a team deployment, you might need three distinct roles:

typescript
// src/permissions/RoleBasedPermissions.ts

export type Role = "developer" | "reviewer" | "analyst";

interface RoleConfig {
  allowedTools: string[];
  deniedTools: string[];
  bashPatterns: {
    allow: string[];
    deny: string[];
  };
  maxFileWritesPerSession: number;
  canCommit: boolean;
  canPush: boolean;
}

const ROLE_CONFIGS: Record<Role, RoleConfig> = {
  developer: {
    allowedTools: [
      "Read", "Edit", "Write", "Bash", "Grep", "Glob",
      "WebFetch", "WebSearch", "NotebookEdit",
    ],
    deniedTools: [],
    bashPatterns: {
      allow: ["*"],
      deny: [
        "rm -rf /",
        "git push --force *main*",
        "DROP TABLE",
      ],
    },
    maxFileWritesPerSession: Infinity,
    canCommit: true,
    canPush: true,
  },

  reviewer: {
    allowedTools: [
      "Read", "Grep", "Glob", "Bash", "WebFetch",
    ],
    deniedTools: ["Edit", "Write", "NotebookEdit"],
    bashPatterns: {
      allow: [
        "git diff *", "git log *", "git show *", "git status",
        "gh pr *", "gh issue *",
        "ruff check *", "mypy *", "pytest *",
      ],
      deny: ["git commit *", "git push *", "rm *", "mv *"],
    },
    maxFileWritesPerSession: 0,
    canCommit: false,
    canPush: false,
  },

  analyst: {
    allowedTools: ["Read", "Grep", "Glob"],
    deniedTools: [
      "Edit", "Write", "Bash", "NotebookEdit",
      "WebFetch", "WebSearch",
    ],
    bashPatterns: {
      allow: [],
      deny: ["*"],
    },
    maxFileWritesPerSession: 0,
    canCommit: false,
    canPush: false,
  },
};

export class RoleBasedPermissionEngine {
  private role: Role;
  private config: RoleConfig;
  private fileWriteCount: number = 0;

  constructor(role: Role) {
    this.role = role;
    this.config = ROLE_CONFIGS[role];
  }

  evaluate(
    toolCall: ToolCall,
    _context: ExecutionContext
  ): PermissionDecision {
    const toolName = toolCall.name;

    // Explicit deny
    if (this.config.deniedTools.includes(toolName)) {
      return "deny";
    }

    // Tool not in allowed list
    if (!this.config.allowedTools.includes(toolName)) {
      return "deny";
    }

    // Bash: check command patterns
    if (toolName === "Bash") {
      const cmd = toolCall.params.command as string;
      return this.evaluateBashCommand(cmd);
    }

    // Write limit check
    if (toolName === "Edit" || toolName === "Write") {
      if (this.fileWriteCount >= this.config.maxFileWritesPerSession) {
        return "deny";
      }
      this.fileWriteCount++;
    }

    return "allow";
  }

  private evaluateBashCommand(cmd: string): PermissionDecision {
    // Check deny patterns first
    for (const pattern of this.config.bashPatterns.deny) {
      if (this.globMatch(cmd, pattern)) return "deny";
    }

    // Check allow patterns
    for (const pattern of this.config.bashPatterns.allow) {
      if (this.globMatch(cmd, pattern)) return "allow";
    }

    return "deny"; // Default deny for bash in role-based mode
  }

  private globMatch(value: string, pattern: string): boolean {
    const regex = new RegExp(
      "^" + pattern.replace(/\*/g, ".*").replace(/\?/g, ".") + "$"
    );
    return regex.test(value);
  }
}

This is fundamentally different from what hooks can achieve. Hooks can block individual commands after the model has already decided to use them (and the model sees the rejection). Role-based permissions filter the tool list before Claude even sees it — Claude in analyst mode never knows that Edit, Write, or Bash exist.

// SastScan — runs Semgrep or CodeQL on changed files
export const SastScanTool: ToolDefinition = {
  name: "SastScan",
  description:
    "Run static analysis (Semgrep) on the specified file or directory. " +
    "Returns findings with CWE references and severity.",
  parameters: {
    type: "object",
    properties: {
      target: { type: "string", description: "File or directory to scan" },
      rulesets: {
        type: "array",
        items: { type: "string" },
        description: "Semgrep rulesets. Defaults to ['p/security-audit']",
      },
    },
    required: ["target"],
  },
  async execute(params, context) {
    const target = params.target as string;
    const rulesets = (params.rulesets as string[]) || ["p/security-audit"];

    const { stdout, stderr } = await execAsync(
      `semgrep --config ${rulesets.join(" --config ")} ` +
      `--json --quiet "${target}"`,
      { cwd: context.workingDirectory }
    );

    const results = JSON.parse(stdout);
    return {
      content: JSON.stringify({
        findings: results.results.map((r: any) => ({
          rule: r.check_id,
          severity: r.extra.severity,
          message: r.extra.message,
          file: r.path,
          line: r.start.line,
          cwe: r.extra.metadata?.cwe || [],
        })),
        summary: {
          total: results.results.length,
          bySeverity: countBySeverity(results.results),
        },
      }, null, 2),
    };
  },
};

The dry-run gate pattern:

typescript
// Every infrastructure mutation goes through plan-first
const INFRA_MUTATING_COMMANDS = [
  /terraform apply/,
  /kubectl (delete|apply|patch|scale)/,
  /aws .* (create|delete|update|put|remove)/,
  /helm (install|upgrade|uninstall)/,
];

for (const toolCall of response.toolCalls) {
  if (toolCall.name === "Bash") {
    const cmd = toolCall.params.command as string;
    const isMutating = INFRA_MUTATING_COMMANDS.some((p) => p.test(cmd));

    if (isMutating && !context.flags.applyMode) {
      // Convert to dry-run automatically
      let dryRunCmd = cmd;
      if (/terraform apply/.test(cmd)) {
        dryRunCmd = cmd.replace("terraform apply", "terraform plan");
      } else if (/kubectl apply/.test(cmd)) {
        dryRunCmd = cmd + " --dry-run=client";
      }

      const planResult = await this.toolExecutor.execute(
        { ...toolCall, params: { command: dryRunCmd } },
        context
      );

      context.conversation.addToolResult(toolCall.id, {
        content:
          `DRY RUN (--apply flag not set):\n${planResult.content}\n\n` +
          `To execute for real, restart with: infraagent --apply`,
      });
      continue;
    }
  }
}

Keeping Your Fork Healthy¶

Source-level forking creates a maintenance burden. Minimize it:

  1. Isolate changes. Keep all custom code in clearly named files (src/tools/custom/, src/engine/gates/). Avoid modifying existing files when you can extend them instead.

  2. Use the extension points. Even within the source code, prefer registering new tools and injecting into the loop over rewriting core logic.

  3. Pin upstream versions. Don’t merge upstream on every commit. Pick stable release tags and merge those.

  4. Test your custom code separately. Your custom tools and gates should have their own test files. Run them in CI alongside the upstream test suite.

  5. Document divergence. Maintain a FORK.md at the root that lists every file you have modified and why. This makes merge conflict resolution predictable.

When to Stay at Layer 4¶

Before you fork, honestly evaluate whether you need Layer 5. Here is a decision framework:

You Need… Layer 4 (MCP) Handles It? Layer 5 Required?
New tool that queries an API Yes No
New tool that reads a database Yes No
Tool that needs engine internals No Yes
Custom permission model No Yes
Cost gate in the loop No Yes
Remove built-in tools No Yes
Rebrand the CLI No Yes
Pre-tool-call security scanning with loop control Partially (hooks) Yes, for full control
Ship a product to non-engineers No Yes
Add telemetry to every API call No Yes
Compliance: air-gapped, no web tools No Yes

The honest answer for most teams: MCP servers and hooks cover 90% of customization needs. The remaining 10% — products, compliance, deep loop control — is where source modification earns its cost.

Chapter Summary¶

Claude Code’s customization model is a stack of five layers, each trading convenience for control. Most teams never need to go past Layer 4 (MCP servers). But when you need to change how the agent thinks, decides, and operates — not just what tools it has — Layer 5 is the only option.

The key modifications covered in this chapter:

  • Custom tools in the registry — implement the ToolDefinition interface, register at startup, use isAvailable for conditional exposure
  • Agentic loop modifications — inject gates (cost, security, dry-run) at the five extension points in the loop
  • Permission engine changes — replace the allow/deny list model with role-based permissions that filter tool visibility before the model sees them
  • Specialized CLI products — combine custom tools, loop modifications, permission modes, and system prompt injection to build focused products (SecureCode, DataOps, InfraAgent)
  • Fork maintenance — isolate changes, document divergence, pin upstream versions, test custom code separately

The next chapters in this part will build on this foundation. Chapter 29 covers building multi-agent orchestration systems where multiple specialized CLIs collaborate on complex tasks, and Chapter 30 addresses testing, monitoring, and operating these systems in production.

The source code is your final degree of freedom. Use it when you must, not when you can.
