Chapter 11: File Operations at Scale¶ ~17 min read

You inherit a codebase with 3,000 files. A library you depend on renamed its main export. You need to find every import, update each one, verify nothing broke, and commit the result. This is not a hypothetical – it is Tuesday.

Claude Code’s file operation tools – Read, Edit, Write, Glob, and Grep – are purpose-built for this kind of work. Each tool does one thing well, and together they handle operations ranging from reading a single file to refactoring across an entire monorepo. This chapter covers each tool in depth, the patterns that make them powerful at scale, and the recipes you will reach for repeatedly.

The File Operation Toolkit¶

Five tools, five jobs:

Tool Job Analogy
Read Load file contents into context Opening a file in your editor
Edit Change specific text in a file Find-and-replace
Write Create a file or rewrite one completely Save As / overwrite
Glob Find files by name pattern File explorer search
Grep Find files by content IDE “Find in Files”

These tools share a design philosophy: be specialized, be fast, produce output optimized for Claude Code’s context window. They are not wrappers around shell commands. They are first-class tools with their own permission handling, hook integration, and error reporting.

Read: Loading Files Into Context¶

Read is the most-called tool in any Claude Code session. It loads file contents, line-numbered, into the conversation context where Claude Code can analyze them.

What Read Handles¶

Read is not limited to source code:

File Type Behavior
Source code (any language) Returns line-numbered text
Plain text, config files Returns line-numbered text
Images (PNG, JPG, GIF, etc.) Renders visually (Claude is multimodal)
PDFs Extracts text, requires pages param for >10 pages
Jupyter notebooks (.ipynb) Returns all cells with outputs
Binary files Returns error or raw representation

Output Format¶

Read returns content in cat -n format:

1   import express from 'express';
2   import { authenticate } from './auth';
3   
4   const app = express();
5   
6   app.post('/login', async (req, res) => {
7     const { email, password } = req.body;
8     const result = await authenticate(email, password);
9     res.json(result);
10  });

The format is: line_number + tab character + file content. This is important for Edit operations – the indentation you see after the tab is the exact indentation in the file.

Partial File Reading¶

For large files, use offset and limit to read specific sections:

Read(file_path: "/project/src/database.ts", offset: 200, limit: 50)

This reads lines 200-249. Use this when: - A file exceeds 2,000 lines (Read’s default limit) - You only need a specific function or section - You want to conserve context window tokens

Default behavior: Without offset and limit, Read returns up to 2,000 lines from the beginning. Files shorter than 2,000 lines are read completely.

Reading Images¶

Claude Code is multimodal. When you Read an image file, Claude sees the actual image:

Read(file_path: "/project/docs/architecture-diagram.png")

Claude Code can then describe the diagram, identify components, spot inconsistencies with the code, or suggest updates. This works with screenshots, mockups, diagrams, and any standard image format.

Reading PDFs¶

PDFs require the pages parameter for documents over 10 pages:

# Short PDF (10 pages or fewer) -- no pages param needed
Read(file_path: "/project/docs/api-spec.pdf")

# Long PDF -- specify page range (max 20 pages per call)
Read(file_path: "/project/docs/full-spec.pdf", pages: "1-10")
Read(file_path: "/project/docs/full-spec.pdf", pages: "11-20")

Reading Jupyter Notebooks¶

Notebooks return all cells with their types, source code, and outputs:

Read(file_path: "/project/analysis/data-exploration.ipynb")

The output includes: - Cell type (code or markdown) - Cell source code - Cell outputs (text, tables, error traces) - Cell execution order

The Read-Before-Edit Invariant¶

This is the most important rule about Read: you must Read a file before you can Edit it. The Edit tool enforces this and will return an error if you try to edit a file that has not been read in the current session.

Why? Because editing without reading means editing based on assumptions. Maybe the file changed since you last looked at it. Maybe it has a different structure than you expected. Maybe the line you want to change was already changed by someone else. Reading first eliminates these failure modes.

Pattern you will see constantly:

1. Read(file_path: "/project/src/auth.ts")       # Load the file
2. [Claude Code analyzes the content]              # Understand it
3. Edit(file_path: "/project/src/auth.ts", ...)   # Modify it

This three-step sequence – read, understand, edit – is the fundamental building block of all file modification workflows.

Edit: Precision Modifications¶

Edit replaces exact strings in files. It is the workhorse of code modification because it transmits only the change, not the entire file.

How Matching Works¶

Edit performs an exact string match on old_string. “Exact” means character-for-character identical, including: - Whitespace (spaces, tabs) - Indentation - Line endings - Blank lines

If the exact string appears once in the file, the replacement succeeds. If it appears zero times, Edit errors with “old_string not found.” If it appears more than once and replace_all is false, Edit errors with a message about multiple matches.

Making old_string Unique¶

The most common Edit failure is a non-unique old_string. The fix is always the same: include more surrounding context.

Non-unique (likely fails):

Edit(
  file_path: "/project/src/utils.ts",
  old_string: "return null;",
  new_string: "return undefined;"
)

The string return null; probably appears in multiple functions.

Unique (succeeds):

Edit(
  file_path: "/project/src/utils.ts",
  old_string: "function findUser(id: string): User | null {\n  const user = cache.get(id);\n  if (!user) {\n    return null;\n  }",
  new_string: "function findUser(id: string): User | undefined {\n  const user = cache.get(id);\n  if (!user) {\n    return undefined;\n  }"
)

By including the function signature and surrounding lines, the match becomes unique.

Preserving Indentation¶

The most subtle Edit mistake is wrong indentation. When Read shows you:

15      if (!user) {
16        return null;
17      }

The actual file content starts after the tab following the line number. Line 16’s content is return null; (six spaces of indentation). Your old_string must include those six spaces exactly.

Wrong (will fail):

Edit(old_string: "return null;", new_string: "return undefined;")

Right:

Edit(old_string: "      return null;", new_string: "      return undefined;")

Or, better, include surrounding lines for uniqueness and let the indentation match naturally:

Edit(
  old_string: "    if (!user) {\n      return null;\n    }",
  new_string: "    if (!user) {\n      return undefined;\n    }"
)

replace_all for Bulk Changes¶

When you intentionally want to change every occurrence, use replace_all: true:

Edit(
  file_path: "/project/src/auth.ts",
  old_string: "UserName",
  new_string: "UserId",
  replace_all: true
)

This renames every UserName to UserId in the file. Use this for variable renames, import path updates, and consistent terminology changes.

Caution: replace_all is literal. If you rename User to Account, it will also change UserService to AccountService and getUser to getAccount. Be specific enough to avoid unintended replacements.

Multi-Part Edits¶

To make multiple distinct changes to a file, use separate Edit calls:

# Edit 1: Fix the import
Edit(
  file_path: "/project/src/auth.ts",
  old_string: "import { hash } from 'crypto';",
  new_string: "import { hash, compare } from 'crypto';"
)

# Edit 2: Fix the function (later in the same file)
Edit(
  file_path: "/project/src/auth.ts",
  old_string: "if (password === storedHash) {",
  new_string: "if (await compare(password, storedHash)) {"
)

These execute sequentially because the second edit depends on the file state after the first. Claude Code handles this correctly – each Edit updates the file, and subsequent Edits operate on the updated content.

Edit vs. Write: When to Choose Which¶

Scenario Tool Reason
Change 1-5 specific sections Edit Sends only the diffs, saves tokens
Change > 60% of the file Write Full rewrite is simpler than many Edits
Create a new file Write Edit cannot create files
Rename a variable across one file Edit with replace_all Surgical, handles multiple occurrences
Generate a new config file Write No existing content to edit
Fix a single bug Edit Minimal change, easy to review

Write: Creating and Rewriting Files¶

Write creates new files or overwrites existing ones completely.

Creating New Files¶

Write(
  file_path: "/project/tests/test_auth.py",
  content: "import pytest\nfrom src.auth import authenticate, AuthError\n\n\nclass TestAuthenticate:\n    def test_valid_credentials(self, mock_db):\n        result = authenticate('user@test.com', 'correct-pass')\n        assert result.success is True\n        assert result.user_id is not None\n\n    def test_invalid_password(self, mock_db):\n        with pytest.raises(AuthError, match='invalid credentials'):\n            authenticate('user@test.com', 'wrong-pass')\n\n    def test_nonexistent_user(self, mock_db):\n        with pytest.raises(AuthError, match='user not found'):\n            authenticate('nobody@test.com', 'any-pass')\n"
)

Rewriting Existing Files¶

When a file needs fundamental restructuring – not just line-level changes but a complete rewrite of its structure – Write is cleaner than multiple Edits:

# Read the file first (required for existing files)
Read(file_path: "/project/config/database.yml")

# Rewrite with new structure
Write(
  file_path: "/project/config/database.yml",
  content: "# Database Configuration\n\ndevelopment:\n  adapter: postgresql\n  host: localhost\n  port: 5432\n  database: myapp_dev\n  pool: 5\n\ntest:\n  adapter: postgresql\n  host: localhost\n  port: 5432\n  database: myapp_test\n  pool: 2\n\nproduction:\n  adapter: postgresql\n  host: ${DATABASE_HOST}\n  port: ${DATABASE_PORT}\n  database: ${DATABASE_NAME}\n  pool: ${DATABASE_POOL}\n  ssl: true\n"
)

When Write Is the Wrong Choice¶

  • Small changes to large files. Rewriting a 500-line file to change 3 lines wastes tokens and makes the change harder to review. Use Edit.
  • Files you have not read. Write errors if the file exists and you have not Read it. This prevents accidental overwrites.
  • Documentation files. Unless the user explicitly asks, Claude Code avoids creating .md or README files.

Glob: Finding Files by Pattern¶

Glob searches for files by name pattern. It is fast at any scale and returns results sorted by modification time (most recent first).

Pattern Syntax¶

Pattern Matches Example
* Any characters in one directory level *.ts matches app.ts but not src/app.ts
** Any characters across all directory levels **/*.ts matches src/app.ts, src/lib/utils.ts
? One character test?.py matches test1.py, testA.py
{a,b} Either a or b *.{ts,tsx} matches .ts and .tsx files
[abc] One of the listed characters test[123].py matches test1.py

Common Glob Patterns¶

# All TypeScript files in a project
Glob(pattern: "**/*.ts", path: "/project")

# Test files only
Glob(pattern: "**/*.test.{ts,tsx,js,jsx}", path: "/project")

# Config files at project root
Glob(pattern: "*.{json,yml,yaml,toml}", path: "/project")

# React components in a specific directory
Glob(pattern: "src/components/**/*.tsx", path: "/project")

# Python files excluding virtual environment
Glob(pattern: "**/*.py", path: "/project/src")

# All files in a specific directory (not recursive)
Glob(pattern: "src/auth/*", path: "/project")

# Migration files (numbered)
Glob(pattern: "**/migrations/*.sql", path: "/project")

Glob’s Sort Order¶

Results come back sorted by modification time, most recent first. This is useful when you want to find the most recently changed files:

# What was edited recently?
Glob(pattern: "src/**/*.py", path: "/project")
# First result = most recently modified Python file

Glob vs. ls vs. find¶

Task Best tool Why
Find files by name pattern Glob Optimized output, sorted by mtime
List one directory’s contents Bash(ls …) Simpler for a flat listing
Find files by complex criteria (size, permissions) Bash(find …) Glob cannot filter by metadata
Find files by content Grep Glob only matches file names

Grep: Searching File Contents¶

Grep is built on ripgrep, the fastest file content searcher available. It searches your codebase, respects .gitignore, handles Unicode, and produces structured output.

Three Output Modes¶

files_with_matches (default)¶

Returns just the file paths that contain the pattern:

Grep(pattern: "authenticateUser", type: "ts")

Output:

/project/src/auth/login.ts
/project/src/middleware/auth.ts
/project/tests/test_auth.ts

Use this when you want to know which files to investigate.

content¶

Returns matching lines with optional context:

Grep(
  pattern: "authenticateUser",
  type: "ts",
  output_mode: "content",
  -C: 3
)

Output:

/project/src/auth/login.ts
10-
11-export async function authenticateUser(
12:  email: string,
13:  password: string
14-): Promise<AuthResult> {
15-  const user = await findByEmail(email);

Use this when you need to see the actual code around matches.

count¶

Returns match counts per file:

Grep(pattern: "TODO", output_mode: "count")

Output:

/project/src/auth/login.ts:3
/project/src/api/routes.ts:7
/project/tests/fixtures.ts:1

Use this for measuring technical debt, usage frequency, or deciding which files to prioritize.

Filtering Results¶

Grep offers two filtering mechanisms:

By file pattern (glob):

Grep(pattern: "import.*React", glob: "*.tsx")

By file type:

Grep(pattern: "import.*React", type: "tsx")

The type parameter uses ripgrep’s built-in type definitions, which handle associated extensions automatically. type: "py" matches .py and .pyi files. type: "ts" matches .ts and .tsx.

Regex Patterns¶

Grep supports full regex syntax. Some common patterns:

# Function definitions
Grep(pattern: "^(export )?(async )?function \\w+", type: "ts", output_mode: "content")

# Class definitions
Grep(pattern: "^class \\w+", type: "py", output_mode: "content")

# Import statements for a specific module
Grep(pattern: "from ['\"]express['\"]", type: "ts")

# TODO with assignee
Grep(pattern: "TODO\\(\\w+\\):", output_mode: "content")

# Environment variable usage
Grep(pattern: "process\\.env\\.", type: "ts", output_mode: "content")

# SQL injection risk (string concatenation in queries)
Grep(pattern: "query.*\\+.*\\$|query.*\\+.*req\\.", type: "ts", output_mode: "content")

Case-Insensitive Search¶

Grep(pattern: "authentication", -i: true, type: "ts")

Matches Authentication, AUTHENTICATION, authentication, and any other casing.

Multiline Patterns¶

By default, patterns match within single lines. For patterns that span lines, enable multiline mode:

Grep(
  pattern: "interface User \\{[\\s\\S]*?email:",
  type: "ts",
  multiline: true,
  output_mode: "content"
)

This finds interface User { blocks that contain an email: field, even if they span many lines.

Controlling Result Volume¶

head_limit (default: 250) caps the number of results:

# Only show first 10 matches
Grep(pattern: "console\\.log", type: "ts", head_limit: 10, output_mode: "content")

offset for pagination:

# Skip first 250, show next 250
Grep(pattern: "import", type: "py", offset: 250, head_limit: 250)

Why head_limit matters: A broad pattern on a large codebase can return thousands of matches. Without head_limit, all those matches go into the context window, consuming tokens and potentially pushing out important earlier context.

Common Grep Mistakes¶

Forgetting to escape regex metacharacters:

# WRONG: Trying to find "Array<string>"
Grep(pattern: "Array<string>")

# RIGHT: Escape the angle brackets
Grep(pattern: "Array<string>")
# Actually, < and > are not regex metacharacters in ripgrep, so this works.
# But for braces:
# WRONG:
Grep(pattern: "interface{}")
# RIGHT:
Grep(pattern: "interface\\{\\}")

Using Grep when Glob is appropriate:

# WRONG: Using Grep to find files named "config"
Grep(pattern: "config", output_mode: "files_with_matches")
# This finds files CONTAINING "config", not files NAMED "config"

# RIGHT:
Glob(pattern: "**/config.*")

Not using type/glob filters:

# SLOW: Searches everything including node_modules, .git, binaries
Grep(pattern: "handleAuth")

# FAST: Searches only TypeScript files, skips gitignored directories
Grep(pattern: "handleAuth", type: "ts")

NotebookEdit: Modifying Jupyter Notebooks¶

Jupyter notebooks are structured JSON files containing an array of cells. While you can Read a notebook to see all cells, modifying individual cells requires the NotebookEdit tool.

Cell Operations¶

NotebookEdit supports three operations:

Operation edit_mode What It Does
Replace replace Replace a cell’s content
Insert insert Add a new cell after a specified cell
Delete delete Remove a cell

Replacing a Cell¶

# First, Read the notebook to see cell IDs and content
Read(file_path: "/project/analysis/exploration.ipynb")

# Replace a code cell's content
NotebookEdit(
  notebook_path: "/project/analysis/exploration.ipynb",
  cell_id: "abc123",
  new_source: "import pandas as pd\nimport matplotlib.pyplot as plt\n\ndf = pd.read_csv('data/sales.csv')\ndf.describe()"
)

Inserting a New Cell¶

# Insert a markdown cell after an existing cell
NotebookEdit(
  notebook_path: "/project/analysis/exploration.ipynb",
  cell_id: "abc123",
  edit_mode: "insert",
  cell_type: "markdown",
  new_source: "## Analysis Results\n\nThe following chart shows the distribution of sales by region."
)

Deleting a Cell¶

NotebookEdit(
  notebook_path: "/project/analysis/exploration.ipynb",
  cell_id: "def456",
  edit_mode: "delete",
  new_source: ""
)

When to Use NotebookEdit vs. Write¶

Use NotebookEdit when modifying individual cells in an existing notebook. Use Write when creating a notebook from scratch (you would write the entire .ipynb JSON structure). The .ipynb format is JSON, so a Write-based creation looks like:

Write(
  file_path: "/project/analysis/new_notebook.ipynb",
  content: "{\"cells\": [{\"cell_type\": \"code\", \"source\": [\"import pandas as pd\"], \"metadata\": {}, \"outputs\": []}], \"metadata\": {\"kernelspec\": {\"display_name\": \"Python 3\", \"language\": \"python\", \"name\": \"python3\"}}, \"nbformat\": 4, \"nbformat_minor\": 5}"
)

Performance Considerations¶

File operations have performance characteristics worth understanding, especially in large projects.

Read Performance¶

Read is filesystem I/O. For local files, it is effectively instant. The bottleneck is not the read itself but the context window cost of the content. A 2,000-line file might be 50-80K tokens – a significant chunk of the 200K context window.

Performance tips: - Use offset/limit for targeted reads of large files - Read small files completely, large files selectively - After /compact, previously read files are no longer in context and must be re-read

Edit Performance¶

Edit is a find-and-replace operation on disk. It is fast regardless of file size. The performance consideration is token cost: old_string and new_string both consume tokens in the tool call.

Performance tips: - For small changes, Edit is always more efficient than Write - For replace_all operations across a very large file, the operation itself is fast but the file re-read needed to verify the change costs tokens

Glob Performance¶

Glob uses optimized filesystem scanning. It is fast even in repositories with millions of files because it uses the same efficient traversal as ripgrep, skipping .gitignore entries and common non-source directories.

Grep Performance¶

Grep inherits ripgrep’s performance. Ripgrep is often faster than IDE search because it: - Respects .gitignore (skips node_modules, dist, .git) - Uses memory mapping for large files - Parallelizes across CPU cores - Uses SIMD instructions for pattern matching

The type parameter makes searches faster because ripgrep can skip files with non-matching extensions before reading them.

The Tool Interaction Matrix¶

Understanding which tools work together and in what order:

Glob ─── finds files ──→ Read ─── loads content ──→ Edit ─── modifies content
  │                        │                           │
  │                        └── provides understanding──┘
  │                                                    
Grep ─── finds content ─→ Read ─── loads full file ──→ Edit
  │                        │
  │                        └── context for unique old_string
  │
  └── head_limit ─→ prevents context exhaustion

The most important flow: Search (Glob/Grep) then Read then Edit. Each step narrows the scope for the next.

Large Codebase Patterns¶

When working with codebases containing thousands of files, the order and combination of tool calls matters significantly.

The Funnel Pattern¶

Start broad, narrow progressively:

Step 1: Glob to understand structure
  Glob(pattern: "src/**/*.ts", path: "/project")
  → 847 files

Step 2: Grep to find relevant files
  Grep(pattern: "PaymentService", type: "ts", output_mode: "files_with_matches")
  → 12 files

Step 3: Grep with content to understand usage
  Grep(pattern: "PaymentService", type: "ts", output_mode: "content", -C: 5, head_limit: 50)
  → See how it's used in context

Step 4: Read the key files
  Read(file_path: "/project/src/services/payment.ts")
  Read(file_path: "/project/src/api/checkout.ts")

Step 5: Edit what needs changing
  Edit(file_path: "/project/src/services/payment.ts", ...)

Each step reduces the scope. You never read 847 files – you narrow to the 2-3 that matter.

The Parallel Read Pattern¶

When you know which files to read, read them all at once:

# These execute in parallel (independent reads)
Read(file_path: "/project/src/auth/login.ts")
Read(file_path: "/project/src/auth/register.ts")
Read(file_path: "/project/src/auth/types.ts")
Read(file_path: "/project/src/auth/middleware.ts")

Four files loaded in one round trip. Claude Code can then analyze them together and make coordinated edits.

The Agent Delegation Pattern¶

When exploration scope is too broad for direct tool calls:

Agent(
  prompt: "Search the entire codebase for all places where database connections are opened but might not be properly closed. Check for: missing finally blocks, missing context managers, connections opened in try without finally. Report each location as file:line with a one-sentence description.",
  subagent_type: "Explore"
)

The agent does the multi-round search, reads files, follows leads, and reports back. Your main context stays clean.

Avoiding Context Window Exhaustion¶

Every file you Read, every Grep result, every Bash output consumes context window tokens. On a large codebase, careless reads can exhaust your context fast.

Strategies:

  1. Read sections, not files. For a 2,000-line file where you only care about one function, use offset and limit: Read(file_path: "/project/src/massive-file.ts", offset: 340, limit: 60)

  2. Use files_with_matches before content. First find which files match, then read only the relevant ones: Grep(pattern: "deprecated", type: "ts") # Just file paths # Then Read only the files that matter

  3. Set head_limit on Grep. If you only need a few examples: Grep(pattern: "TODO", type: "py", output_mode: "content", head_limit: 10)

  4. Delegate broad searches to Agent. An Agent has its own context window. Its exploration noise stays in its context, and only the summary comes back to you.

Multi-File Editing Patterns¶

Sequential Edits (Dependent Changes)¶

When editing file B depends on what you changed in file A:

# Step 1: Read both files
Read(file_path: "/project/src/types.ts")
Read(file_path: "/project/src/auth.ts")

# Step 2: Edit types first (auth.ts depends on these types)
Edit(
  file_path: "/project/src/types.ts",
  old_string: "interface User {\n  name: string;\n}",
  new_string: "interface User {\n  name: string;\n  email: string;\n}"
)

# Step 3: Now edit auth to use the new field
Edit(
  file_path: "/project/src/auth.ts",
  old_string: "const greeting = `Hello, ${user.name}`;",
  new_string: "const greeting = `Hello, ${user.name} (${user.email})`;"
)

Parallel Edits (Independent Changes)¶

When changes in different files are independent:

# Both edits in one response (execute in parallel)
Edit(
  file_path: "/project/src/config/dev.ts",
  old_string: "port: 3000",
  new_string: "port: 8080"
)

Edit(
  file_path: "/project/src/config/test.ts",
  old_string: "port: 3000",
  new_string: "port: 8081"
)

The Rename Across a Project¶

Renaming a function, class, or variable across multiple files:

# Step 1: Find all files using the old name
Grep(pattern: "fetchUserData", type: "ts", output_mode: "files_with_matches")

# Step 2: Read each file, then edit
# File 1
Read(file_path: "/project/src/api/users.ts")
Edit(file_path: "/project/src/api/users.ts", old_string: "fetchUserData", new_string: "getUserById", replace_all: true)

# File 2
Read(file_path: "/project/src/components/UserProfile.tsx")
Edit(file_path: "/project/src/components/UserProfile.tsx", old_string: "fetchUserData", new_string: "getUserById", replace_all: true)

# File 3
Read(file_path: "/project/tests/api.test.ts")
Edit(file_path: "/project/tests/api.test.ts", old_string: "fetchUserData", new_string: "getUserById", replace_all: true)

For a rename that touches many files (10+), consider using the Agent tool or the /batch command to parallelize.

10 File Operation Recipes¶

1. Find and Read All Test Files for a Module¶

Glob(pattern: "**/auth/**/*.test.{ts,tsx}")
# Then Read each result

2. Count Lines of Code by Language¶

Bash(command: "find /project/src -name '*.ts' | xargs wc -l | tail -1")
Bash(command: "find /project/src -name '*.py' | xargs wc -l | tail -1")

(This is a valid Bash use case – there is no dedicated tool for line counting.)

3. Find All Exported Functions in a Directory¶

Grep(
  pattern: "^export (async )?function \\w+",
  path: "/project/src/api",
  type: "ts",
  output_mode: "content"
)

4. Find Dead Code (Exported but Never Imported)¶

# Step 1: Find all exports
Grep(pattern: "^export ", path: "/project/src", type: "ts", output_mode: "content", head_limit: 100)

# Step 2: For each export, search for imports
Grep(pattern: "import.*specificFunction", type: "ts", output_mode: "count")
# If count is 0, it might be dead code

5. Update a Version Number Everywhere¶

Grep(pattern: "1\\.2\\.3", output_mode: "files_with_matches")
# For each file:
Read(file_path: "...")
Edit(file_path: "...", old_string: "1.2.3", new_string: "1.3.0", replace_all: true)

6. Add a License Header to All Source Files¶

Glob(pattern: "src/**/*.ts")
# For each file:
Read(file_path: "...")
Edit(
  file_path: "...",
  old_string: "[first line of file]",
  new_string: "// Copyright 2026 MyCompany. All rights reserved.\n// SPDX-License-Identifier: MIT\n\n[first line of file]"
)

7. Find All TODO Comments with Context¶

Grep(
  pattern: "TODO|FIXME|HACK|XXX",
  output_mode: "content",
  -C: 1,
  head_limit: 50
)

8. Find Configuration Inconsistencies¶

# Compare dev and production configs
Read(file_path: "/project/config/development.json")
Read(file_path: "/project/config/production.json")
# Claude Code analyzes differences and flags inconsistencies

9. Extract All API Endpoints¶

Grep(
  pattern: "(app|router)\\.(get|post|put|patch|delete)\\(",
  type: "ts",
  output_mode: "content",
  -B: 1,
  -A: 3

10. Find Potential Security Issues¶

# Hardcoded secrets
Grep(pattern: "(password|secret|key|token)\\s*=\\s*['\"][^'\"]{8,}", -i: true, output_mode: "content")

# SQL injection risks
Grep(pattern: "query.*\\$\\{|query.*\\+\\s*req", type: "ts", output_mode: "content")

# Eval usage
Grep(pattern: "\\beval\\(", output_mode: "content")

What Most People Get Wrong¶

Mistake 1: Reading entire large files when you need one section.

A 3,000-line file read without offset/limit consumes significant context tokens. If you know the function starts around line 450, read just that section:

Read(file_path: "/project/src/massive.ts", offset: 440, limit: 40)

Mistake 2: Grep without type or glob filters.

On a large codebase, an unfiltered Grep searches everything – including node_modules, .git, build output, and binary files. Always filter:

# SLOW and noisy
Grep(pattern: "handleAuth")

# FAST and precise
Grep(pattern: "handleAuth", type: "ts")

Mistake 3: Using Write for small changes.

Every time you Write an existing file, you send the entire file content through the context window. For a 500-line file where you change one line, that is 499 wasted lines. Edit sends only the change.

Mistake 4: Not using replace_all when renaming.

Editing the same string three times in three separate Edit calls when one replace_all: true would handle all of them:

# THREE calls for one rename (wasteful)
Edit(old_string: "oldName", new_string: "newName")  # First occurrence
Edit(old_string: "oldName", new_string: "newName")  # Second occurrence
Edit(old_string: "oldName", new_string: "newName")  # Third occurrence

# ONE call (correct)
Edit(old_string: "oldName", new_string: "newName", replace_all: true)

Mistake 5: Forgetting Glob sorts by modification time.

When Glob returns 200 files, the first ones are the most recently modified. If you are looking for files that were just changed (maybe by a teammate, or by Claude Code in a previous step), they are at the top.

Summary¶

The five file operation tools – Read, Edit, Write, Glob, Grep – are designed to work together. Glob finds files by name. Grep finds files by content. Read loads them into context. Edit modifies them precisely. Write creates new ones. The patterns scale from single-file edits to project-wide refactoring when you combine them correctly: funnel from broad to narrow, parallelize independent operations, delegate exploration to agents, and always read before you edit.

Key takeaways: 1. Read before Edit is enforced, not optional – the Edit tool errors if you skip it 2. Edit sends only the diff; Write sends the entire file – prefer Edit for modifications 3. Glob for names, Grep for contents – never confuse the two 4. Filter Grep with type or glob to avoid searching irrelevant files 5. For large codebases, use the funnel pattern: Glob/Grep to narrow, Read the relevant files, Edit the specific lines
