Chapter 20: CI/CD Integration¶ ~32 min read

Claude Code is not just a development tool. It runs in CI/CD pipelines, reviews pull requests, generates release notes, and validates deployments — all without a human in the loop. This chapter covers how to integrate Claude Code into your automated workflows: GitHub Actions, Azure DevOps Pipelines, pre-commit hooks, automated PR reviews, deployment verification, and cost management.

The key principle: use Claude Code in CI the same way you use it locally, but with tighter cost controls and appropriate model selection. Haiku for simple tasks, Sonnet for reviews, Opus reserved for security audits. The second principle: never deploy Claude Code in CI without a budget ceiling and a kill switch. LLM costs scale with usage, and a runaway pipeline can burn through hundreds of dollars before anyone notices.

As we discussed in Chapter 19, test-driven development with Claude Code changes how fast you ship code. This chapter covers what happens after you commit — how Claude Code participates in the automated gauntlet between git push and production.

Headless Mode: Running Claude Code Without a Terminal¶

Before diving into pipeline configurations, you need to understand how Claude Code operates in CI environments. Locally, Claude Code runs in an interactive terminal UI (TUI). In CI, there is no terminal. You need headless mode.

The –print Flag¶

The simplest way to run Claude Code non-interactively is the --print flag:

bash
claude --print "Review this code for security issues" \
  --model claude-sonnet-4-6 \
  --output-format json

The --print flag sends a single prompt, prints the response to stdout, and exits. No TUI, no interactive session, no follow-up turns. This is what you want for most CI tasks.

The –output-format Flag¶

In CI, you almost always want structured output for downstream processing:

bash
# Plain text (default) --- good for PR comments
claude --print "Summarize these changes" --output-format text

# JSON --- good for parsing in subsequent pipeline steps
claude --print "Classify this PR" --output-format json

# Stream JSON --- good for real-time processing
claude --print "Review this diff" --output-format stream-json

Environment Variables for CI¶

Claude Code reads several environment variables that are essential for CI operation:

Variable Purpose Required
ANTHROPIC_API_KEY Authentication with Anthropic API Yes
CLAUDE_MODEL Override the default model No
CLAUDE_CODE_MAX_TURNS Limit agentic loop iterations No
CLAUDE_CODE_MAX_TOOL_USES Limit total tool calls No
CLAUDE_CODE_HEADLESS Force headless mode (no TUI) No
DISABLE_PROMPT_CACHING Disable prompt caching No
CI Standard CI indicator (auto-set by most CI systems) Auto

A typical CI step sets these as environment variables:

yaml
- name: Run Claude Code
  run: |
    claude --print "Review the changes in this PR" \
      --model claude-sonnet-4-6 \
      --output-format json \
      --max-turns 5
  env:
    ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
    CLAUDE_CODE_MAX_TURNS: "5"

Max Turns: The Safety Net¶

In agentic mode, Claude Code can loop — reading files, running commands, iterating on results. Locally, you monitor this interactively. In CI, a runaway loop means runaway costs.

Always set --max-turns in CI:

bash
# For simple classification tasks: 1-2 turns
claude --print "Classify this PR as bug/feature/chore" --max-turns 1

# For code reviews that need to read files: 3-5 turns
claude --print "Review src/auth/ for security issues" --max-turns 5

# For complex analysis: 10-15 turns (with timeout)
claude --print "Audit the entire API surface" --max-turns 15

Each turn is an LLM call plus any tool executions. A 5-turn review with Sonnet costs roughly $0.15-0.30. A 15-turn audit with Opus could cost $3-5. Know your numbers.

Installing Claude Code in CI¶

For the official GitHub Action, installation is handled automatically. For direct CLI usage in any CI system:

yaml
- name: Install Claude Code
  run: npm install -g @anthropic-ai/claude-code

- name: Verify installation
  run: claude --version

For Docker-based CI, add it to your image:

dockerfile
FROM node:20-slim
RUN npm install -g @anthropic-ai/claude-code

GitHub Actions Integration¶

The Official GitHub Action¶

Anthropic provides claude-code-action, the official GitHub Action for running Claude Code in CI. It supports PR reviews, automated fixes, and arbitrary Claude Code prompts.

Basic setup:

yaml
# .github/workflows/claude-review.yml

name: Claude Code Review

on:
  pull_request:
    types: [opened, synchronize]

permissions:
  contents: read
  pull-requests: write

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          prompt: |
            Review this PR for:
            1. Security vulnerabilities (check for injection, auth issues, secrets)
            2. Logic bugs that could cause production issues
            3. Performance regressions

            Focus on what could break. Skip style nitpicks.

            For each finding, provide:
            - Severity (CRITICAL / IMPORTANT / SUGGESTION)
            - File and line number
            - Description
            - Suggested fix

This runs on every PR open and update. Claude reads the diff, reviews it, and posts findings as a PR comment.

Configuring the Action¶

The claude-code-action supports several configuration options:

yaml
- uses: anthropics/claude-code-action@v1
  with:
    # Required: API key
    anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}

    # The prompt to run
    prompt: "Review this PR for security issues"

    # Model selection (default: claude-sonnet-4-6)
    model: "claude-sonnet-4-6"

    # Maximum tokens for the response
    max_tokens: 4096

    # Whether to post results as PR comment
    post_comment: true

    # Effort level: low, medium, high
    effort: "medium"

    # Timeout in seconds
    timeout: 300

    # Working directory
    working_directory: "."

The Full CI Pipeline: Lint, Test, Type-Check, Review, Build, Deploy¶

Most teams need more than just a Claude Code review. Here is a complete GitHub Actions workflow that covers every stage of a professional CI pipeline:

yaml
# .github/workflows/ci.yml

name: CI Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

permissions:
  contents: read
  pull-requests: write
  checks: write

env:
  NODE_VERSION: '20'
  PYTHON_VERSION: '3.12'

jobs:
  # ── Stage 1: Parallel Quality Gates ───────────────────────
  lint:
    name: Lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: ${{ env.PYTHON_VERSION }}

      - name: Install linting tools
        run: pip install ruff

      - name: Ruff check
        run: ruff check src/ --output-format github

      - name: Ruff format check
        run: ruff format src/ --check --diff

  type-check:
    name: Type Check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: ${{ env.PYTHON_VERSION }}

      - name: Install dependencies
        run: |
          pip install -e ".[dev]"
          pip install pyright

      - name: Run pyright
        run: pyright src/

  test:
    name: Test
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16-alpine
        env:
          POSTGRES_USER: test
          POSTGRES_PASSWORD: test
          POSTGRES_DB: testdb
        ports: ['5432:5432']
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: ${{ env.PYTHON_VERSION }}

      - name: Install dependencies
        run: pip install -e ".[dev]"

      - name: Run tests with coverage
        run: |
          python -m pytest tests/ \
            --cov=src \
            --cov-report=xml \
            --cov-report=term-missing \
            -v
        env:
          DATABASE_URL: postgresql://test:test@localhost:5432/testdb

      - uses: actions/upload-artifact@v4
        with:
          name: coverage
          path: coverage.xml

  security-scan:
    name: Security Scan
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: ${{ env.PYTHON_VERSION }}

      - name: Install audit tools
        run: pip install pip-audit safety

      - name: Pip audit
        run: pip-audit -r requirements.txt --strict

      - name: Safety check
        run: safety check -r requirements.txt
        continue-on-error: true

      - name: Secret scan
        run: |
          # Scan for common secret patterns
          if grep -rEn \
            '(AKIA[0-9A-Z]{16}|ghp_[0-9a-zA-Z]{36}|sk-[a-zA-Z0-9]{48}|-----BEGIN.*PRIVATE KEY-----)' \
            --include="*.py" --include="*.js" --include="*.ts" --include="*.yaml" --include="*.yml" \
            src/ ; then
            echo "::error::Potential secrets found in source code!"
            exit 1
          fi

  # ── Stage 2: Claude Code Review (PRs only) ─────────────
  claude-review:
    name: Claude Code Review
    runs-on: ubuntu-latest
    if: github.event_name == 'pull_request'
    needs: [lint, type-check, test, security-scan]
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          model: "claude-sonnet-4-6"
          effort: "medium"
          timeout: 300
          prompt: |
            Review this PR. Focus on what could break in production.

            ## Check for:
            - Logic bugs and unhandled edge cases
            - Security vulnerabilities (OWASP Top 10)
            - Missing error handling
            - Performance regressions
            - Missing test coverage for new code paths
            - Breaking API changes

            ## Format:
            **[CRITICAL]** `file:line` --- finding
            **[IMPORTANT]** `file:line` --- finding
            **[SUGGESTION]** `file:line` --- finding

            Skip style issues (linter handles those).

  # ── Stage 3: Build (main branch only) ──────────────────
  build:
    name: Build
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    needs: [lint, type-check, test, security-scan]
    steps:
      - uses: actions/checkout@v4

      - name: Build Docker image
        run: |
          docker build \
            --tag app:${{ github.sha }} \
            --tag app:latest \
            --build-arg GIT_SHA=${{ github.sha }} \
            -f Dockerfile .

      - name: Push to registry
        run: |
          echo "${{ secrets.REGISTRY_PASSWORD }}" | \
            docker login ghcr.io -u ${{ github.actor }} --password-stdin
          docker tag app:${{ github.sha }} ghcr.io/${{ github.repository }}:${{ github.sha }}
          docker push ghcr.io/${{ github.repository }}:${{ github.sha }}

Notice the structure: lint, type-check, test, and security scan run in parallel (Stage 1). The Claude Code review only runs after all four pass (Stage 2). Build only runs on the main branch after quality gates pass (Stage 3). This means Claude Code never reviews code that fails basic checks — no point reviewing code that does not compile or pass tests.

1. PR Security Review¶

yaml
name: Security Review

on:
  pull_request:
    paths:
      - 'src/auth/**'
      - 'src/middleware/**'
      - 'src/routes/**'
      - '**.env*'

jobs:
  security-review:
    runs-on: ubuntu-latest

            Perform a security audit on this PR. Check against OWASP Top 10.

            Specifically look for:
            - SQL injection or command injection
            - Authentication/authorization bypasses
            - Secrets or credentials in code
            - Insecure data exposure
            - Missing input validation
            - CORS misconfigurations

            | Severity | CWE | Finding | File:Line | Recommended Fix |

2. Automated Label Assignment¶

yaml
name: Auto-Label PR

on:
  pull_request:
    types: [opened]

permissions:
  pull-requests: write

jobs:
  label:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - id: classify
        uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          model: "claude-haiku-3-5"
          prompt: |
            Classify this PR. Output ONLY a JSON array of label names.
            Available labels: bug-fix, feature, refactor, docs, tests, 
            security, breaking-change, dependencies, performance

            Example: ["feature", "tests"]

      - name: Apply labels
        uses: actions/github-script@v7
        with:
          script: |
            const labels = JSON.parse('${{ steps.classify.outputs.response }}');
            await github.rest.issues.addLabels({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: context.issue.number,
              labels: labels,
            });

Uses Haiku (cheapest model) because label classification is a simple task.

4. Automated PR Review Comment with Inline Annotations¶

This is the pattern teams actually want: Claude Code reviews the PR and posts findings as inline comments on specific lines, not just a wall of text:

yaml
name: Claude Inline Review

on:
  pull_request:
    types: [opened, synchronize]

permissions:
  contents: read
  pull-requests: write

jobs:
  inline-review:
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Get diff
        id: diff
        run: |
          git diff origin/${{ github.base_ref }}...HEAD > /tmp/pr-diff.txt
          echo "diff_file=/tmp/pr-diff.txt" >> $GITHUB_OUTPUT

      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code

      - name: Review and generate annotations
        id: review
        run: |
          claude --print "$(cat <<'PROMPT'
          Read the git diff for this PR. For each issue you find, output a JSON array.
          Each item must have: file, line, severity, message, suggestion.

          Example:
          [
            {
              "file": "src/auth/login.py",
              "line": 42,
              "severity": "CRITICAL",
              "message": "SQL injection via unsanitized user input",
              "suggestion": "Use parameterized query: cursor.execute('SELECT * FROM users WHERE email = %s', (email,))"
            }
          ]

          Only output the JSON array. No other text.
          PROMPT
          )" --model claude-sonnet-4-6 --output-format text > /tmp/review.json
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}

      - name: Post inline comments
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const findings = JSON.parse(fs.readFileSync('/tmp/review.json', 'utf8'));

            for (const finding of findings) {
              const emoji = finding.severity === 'CRITICAL' ? '🔴' :
                            finding.severity === 'IMPORTANT' ? '🟡' : '🟢';

              await github.rest.pulls.createReviewComment({
                owner: context.repo.owner,
                repo: context.repo.repo,
                pull_number: context.issue.number,
                body: `**${emoji} ${finding.severity}**: ${finding.message}\n\n**Suggestion:** ${finding.suggestion}`,
                commit_id: context.payload.pull_request.head.sha,
                path: finding.file,
                line: finding.line,
                side: 'RIGHT',
              });
            }

This is more complex than the basic action but produces much better developer experience. Instead of scrolling through a comment looking for what applies to which file, developers see findings right next to the code that triggered them.

Pre-Commit Hooks¶

Pre-commit hooks run before git commit completes. They are your last line of defense against bad commits.

Secret Scanning¶

This hook scans staged files for credentials before allowing the commit:

bash
#!/usr/bin/env bash
# .claude/hooks/secret-scan.sh
set -euo pipefail

INPUT=$(cat)
TOOL=$(echo "${INPUT}" | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_name',''))")

if [[ "${TOOL}" != "Bash" ]]; then
    exit 0
fi

COMMAND=$(echo "${INPUT}" | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_input',{}).get('command',''))")

# Only check git commit commands
if [[ "${COMMAND}" != *"git commit"* ]]; then
    exit 0
fi

# Get staged file contents
STAGED_DIFF=$(git diff --cached --diff-filter=ACM)

# Pattern matching for common secrets
PATTERNS=(
    'AKIA[0-9A-Z]{16}'                    # AWS Access Key
    'ghp_[0-9a-zA-Z]{36}'                 # GitHub Personal Token
    'sk-[a-zA-Z0-9]{48}'                  # OpenAI/Anthropic API Key
    'xox[baprs]-[0-9a-zA-Z-]+'            # Slack Token
    'password\s*=\s*["\x27][^"\x27]+'      # Hardcoded passwords
    'PRIVATE KEY-----'                     # Private keys
)

for pattern in "${PATTERNS[@]}"; do
    if echo "${STAGED_DIFF}" | grep -qE "${pattern}"; then
        cat >&2 <<BLOCK
{"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"deny","permissionDecisionReason":"BLOCKED: Potential secret detected in staged files. Pattern matched: ${pattern}. Remove the secret before committing."}}
BLOCK
        exit 2
    fi
done

exit 0

Commit Message Validation¶

Ensure commit messages follow conventional commits format:

bash
#!/usr/bin/env bash
# .claude/hooks/commit-msg-check.sh
set -euo pipefail

INPUT=$(cat)
COMMAND=$(echo "${INPUT}" | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_input',{}).get('command',''))")

# Only check git commit commands
if [[ "${COMMAND}" != *"git commit"* ]]; then
    exit 0
fi

# Extract commit message from the command
MSG=$(echo "${COMMAND}" | grep -oP '(?<=-m\s")[^"]+' || echo "")
if [[ -z "${MSG}" ]]; then
    exit 0
fi

# Validate conventional commit format
VALID_PREFIXES="feat|fix|docs|style|refactor|perf|test|build|ci|chore|revert"
if ! echo "${MSG}" | grep -qE "^(${VALID_PREFIXES})(\(.+\))?!?:"; then
    cat >&2 <<BLOCK
{"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"deny","permissionDecisionReason":"Commit message must follow conventional commits format: type(scope): description. Valid types: ${VALID_PREFIXES}"}}
BLOCK
    exit 2
fi

exit 0

Git Hooks That Integrate Claude Code Analysis¶

Beyond Claude Code’s own hook system, you can use standard git hooks to invoke Claude Code before commits. This is useful when developers use Claude Code locally and want pre-commit intelligence:

bash
#!/usr/bin/env bash
# .git/hooks/pre-commit
set -euo pipefail

# Skip in CI (CI systems have their own review pipelines)
if [[ -n "${CI:-}" ]]; then
    exit 0
fi

# Skip if Claude Code is not installed
if ! command -v claude >/dev/null 2>&1; then
    exit 0
fi

# Get staged diff
STAGED_DIFF=$(git diff --cached --diff-filter=ACM)

if [[ -z "${STAGED_DIFF}" ]]; then
    exit 0
fi

# Quick security scan using Claude Code (Haiku for speed)
echo "Running Claude Code pre-commit security scan..."
RESULT=$(echo "${STAGED_DIFF}" | claude --print \
    "Scan this diff for security issues. If you find any CRITICAL issues (secrets, injection, auth bypass), output them. If no critical issues, output: PASS" \
    --model claude-haiku-3-5 \
    --max-turns 1 \
    2>/dev/null || echo "SKIP")

if [[ "${RESULT}" == "SKIP" ]]; then
    # Claude Code not available or errored --- allow commit
    exit 0
fi

if [[ "${RESULT}" != *"PASS"* ]]; then
    echo ""
    echo "=== Claude Code Security Findings ==="
    echo "${RESULT}"
    echo ""
    echo "Fix the issues above or use --no-verify to skip this check."
    exit 1
fi

exit 0

Automated PR Reviews¶

The Review Pipeline¶

A comprehensive PR review pipeline uses multiple Claude Code invocations, each focused on a different aspect:

yaml
# .github/workflows/pr-review.yml

name: PR Review Pipeline

on:
  pull_request:
    types: [opened, synchronize]

permissions:
  contents: read
  pull-requests: write

jobs:
  # Job 1: Quick classification (cheap, fast)
  classify:
    runs-on: ubuntu-latest
    outputs:
      risk_level: ${{ steps.classify.outputs.risk_level }}
      labels: ${{ steps.classify.outputs.labels }}
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - id: classify
        uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          model: "claude-haiku-3-5"
          prompt: |
            Classify this PR. Output JSON only:
            {
              "risk_level": "low|medium|high|critical",
              "labels": ["label1", "label2"],
              "needs_security_review": true/false,
              "breaking_changes": true/false
            }

            Risk levels:
            - low: docs, tests, minor refactors
            - medium: new features, dependency updates
            - high: auth changes, API changes, database migrations
            - critical: security fixes, infrastructure changes

  # Job 2: Code quality review (always runs)
  code-review:
    runs-on: ubuntu-latest
    needs: classify
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          model: "claude-sonnet-4-6"
          prompt: |
            Review this PR for code quality.
            Risk level: ${{ needs.classify.outputs.risk_level }}

            Check for:
            - Logic bugs
            - Error handling gaps
            - Missing tests for new code
            - Performance issues
            - API contract changes

            Format findings as:
            **[SEVERITY]** `file:line` --- description

            Skip style issues. Focus on production impact.

  # Job 3: Security review (only for high/critical risk)
  security-review:
    runs-on: ubuntu-latest
    needs: classify
    if: needs.classify.outputs.risk_level == 'high' || needs.classify.outputs.risk_level == 'critical'
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          model: "claude-sonnet-4-6"
          effort: "high"
          prompt: |
            Security audit for this high-risk PR.

            Check against OWASP Top 10. Additionally check for:
            - Authentication/authorization bypass
            - Injection (SQL, command, template)
            - Secrets in code
            - Insecure deserialization
            - Missing rate limiting
            - CORS issues

            Output:
            | Severity | CWE | Finding | Location | Fix |

This pipeline is cost-efficient:

  • Classification uses Haiku (~$0.01 per PR)
  • Code review uses Sonnet (~$0.15 per PR)
  • Security review only runs for high-risk PRs using Sonnet with high effort (~$0.30 per PR)

Total cost for a low-risk PR: ~$0.16. For a high-risk PR: ~$0.46. At 50 PRs per week, that is $8-23 per week for automated, consistent reviews.

Running Skills in CI: /security-audit and /code-review¶

If you have Claude Code skills configured in your repository (see Chapter 21 on custom agents and intent routing), you can invoke them directly in CI. This gives you the same structured analysis in CI that you get locally:

            --model claude-sonnet-4-6 \

        if: always()

        with:
          script: |
            const fs = require('fs');
            const audit = fs.readFileSync('/tmp/security-audit.txt', 'utf8');

            await github.rest.issues.createComment({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: context.issue.number,
              body: `## 🔒 Security Audit Results\n\n${audit}\n\n---\n*Automated by Claude Code /security-audit*`,

Lint Fix and Commit¶

When linting fails in CI, Claude Code can fix the issues and push the correction:

yaml
name: Auto-Fix Lint

on:
  pull_request:
    types: [opened, synchronize]

permissions:
  contents: write

jobs:
  lint-fix:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          ref: ${{ github.head_ref }}
          token: ${{ secrets.GITHUB_TOKEN }}

      - uses: actions/setup-python@v5
        with:
          python-version: '3.12'

      - name: Install tools
        run: pip install ruff

      - name: Check for lint issues
        id: lint
        run: |
          if ! ruff check src/ --output-format json > lint-results.json 2>&1; then
            echo "has_issues=true" >> $GITHUB_OUTPUT
          else
            echo "has_issues=false" >> $GITHUB_OUTPUT
          fi

      - name: Auto-fix
        if: steps.lint.outputs.has_issues == 'true'
        run: |
          ruff check src/ --fix
          ruff format src/

      - name: Commit fixes
        if: steps.lint.outputs.has_issues == 'true'
        run: |
          git config user.name "Claude Code Bot"
          git config user.email "claude-code-bot@noreply.github.com"
          git add -A
          git diff --cached --quiet || git commit -m "style: auto-fix lint issues

          Co-Authored-By: Claude Code <noreply@anthropic.com>"
          git push

Deployment Pipelines¶

Staging to Production with Claude Code Verification¶

The most valuable CI/CD integration is not just reviewing code — it is verifying deployments. Claude Code can act as a deployment gatekeeper, checking that what you are about to ship matches what was approved in the PR.

Here is a full staging-to-production pipeline with Claude Code verification at each transition:

yaml
# .github/workflows/deploy.yml

name: Deploy Pipeline

on:
  push:
    branches: [main]

permissions:
  contents: write
  deployments: write
  issues: write

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  # ── Build ──────────────────────────────────────────────
  build:
    runs-on: ubuntu-latest
    outputs:
      image_tag: ${{ steps.meta.outputs.tags }}
      image_digest: ${{ steps.build.outputs.digest }}
    steps:
      - uses: actions/checkout@v4

      - name: Docker meta
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=sha,prefix=

      - name: Build and push
        id: build
        uses: docker/build-push-action@v6
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}

  # ── Deploy to Staging ──────────────────────────────────
  deploy-staging:
    runs-on: ubuntu-latest
    needs: build
    environment: staging
    steps:
      - uses: actions/checkout@v4

      - name: Deploy to staging
        run: |
          kubectl set image deployment/app \
            app=${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }} \
            --namespace staging
          kubectl rollout status deployment/app --namespace staging --timeout=300s

      - name: Run smoke tests
        run: |
          # Wait for deployment to stabilize
          sleep 10
          # Hit health endpoint
          HTTP_STATUS=$(curl -s -o /dev/null -w "%{http_code}" https://staging.example.com/health)
          if [[ "${HTTP_STATUS}" != "200" ]]; then
            echo "Health check failed with status ${HTTP_STATUS}"
            exit 1
          fi

  # ── Claude Code Staging Verification ───────────────────
  verify-staging:
    runs-on: ubuntu-latest
    needs: deploy-staging
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          model: "claude-sonnet-4-6"
          prompt: |
            Pre-production deployment check for commit ${{ github.sha }}.

            1. Read the git log for this commit and associated PRs
            2. Identify what changed (features, fixes, migrations)
            3. Assess deployment risk:
               - Does this include database migrations?
               - Does this change API contracts?
               - Does this modify authentication/authorization?
               - Does this update infrastructure config?
            4. Generate a deployment summary

            Output:
            ## Deployment Summary
            - **Commit:** ${{ github.sha }}
            - **Risk Level:** LOW / MEDIUM / HIGH
            - **Changes:** [bullet list]
            - **Requires Rollback Plan:** YES / NO
            - **Recommended:** PROCEED / HOLD FOR MANUAL REVIEW

  # ── Deploy to Production (Blue-Green) ──────────────────
  deploy-production:
    runs-on: ubuntu-latest
    needs: verify-staging
    environment: production
    steps:
      - uses: actions/checkout@v4

      - name: Deploy green instance
        id: green
        run: |
          # Deploy new version as "green"
          kubectl set image deployment/app-green \
            app=${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }} \
            --namespace production
          kubectl rollout status deployment/app-green --namespace production --timeout=300s

      - name: Smoke test green
        run: |
          HTTP_STATUS=$(curl -s -o /dev/null -w "%{http_code}" https://green.example.com/health)
          if [[ "${HTTP_STATUS}" != "200" ]]; then
            echo "Green health check failed"
            exit 1
          fi

      - name: Switch traffic to green
        run: |
          # Update service selector to point to green
          kubectl patch service app-service \
            -p '{"spec":{"selector":{"version":"green"}}}' \
            --namespace production

      - name: Verify production
        run: |
          sleep 5
          HTTP_STATUS=$(curl -s -o /dev/null -w "%{http_code}" https://app.example.com/health)
          if [[ "${HTTP_STATUS}" != "200" ]]; then
            echo "Production health check failed --- initiating rollback"
            kubectl patch service app-service \
              -p '{"spec":{"selector":{"version":"blue"}}}' \
              --namespace production
            exit 1
          fi

      - name: Promote green to blue
        run: |
          # Tag current green as the new blue for next deployment
          kubectl set image deployment/app-blue \
            app=${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }} \
            --namespace production

  # ── Post-Deploy: Release Notes ─────────────────────────
  release-notes:
    runs-on: ubuntu-latest
    needs: deploy-production
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          model: "claude-sonnet-4-6"
          prompt: |
            Generate release notes for the deployment of ${{ github.sha }}.

            Read the git log since the last deployment tag.
            Write user-facing release notes:

            ## What's New
            - Feature descriptions

            ## Bug Fixes
            - Fix descriptions

            ## Security
            - Security-related changes

            Keep it concise. Write for users, not developers.

Caching Strategies¶

Claude Code in CI does not have built-in result caching, but you can implement it at the pipeline level. The idea: if the diff has not changed since the last review, skip the review.

yaml
name: Cached Review

on:
  pull_request:
    types: [opened, synchronize]

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Compute diff hash
        id: diff_hash
        run: |
          HASH=$(git diff origin/${{ github.base_ref }}...HEAD | sha256sum | cut -d' ' -f1)
          echo "hash=${HASH}" >> $GITHUB_OUTPUT

      - name: Check cache
        id: cache
        uses: actions/cache@v4
        with:
          path: /tmp/claude-review-cache
          key: claude-review-${{ steps.diff_hash.outputs.hash }}

      - name: Run review (only if cache miss)
        if: steps.cache.outputs.cache-hit != 'true'
        uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          model: "claude-sonnet-4-6"
          prompt: "Review this PR for bugs and security issues"

      - name: Save to cache
        if: steps.cache.outputs.cache-hit != 'true'
        run: |
          echo "${{ steps.review.outputs.response }}" > /tmp/claude-review-cache/result.txt

This avoids re-reviewing force-pushed branches where the only change is a rebase (same diff, different commits).

Conditional Execution¶

Skip expensive reviews for trivial changes:

yaml
jobs:
  should-review:
    runs-on: ubuntu-latest
    outputs:
      skip: ${{ steps.check.outputs.skip }}
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - id: check
        run: |
          # Skip review for docs-only, test-only, or config-only changes
          CHANGED=$(git diff --name-only origin/main...HEAD)
          if echo "$CHANGED" | grep -qvE '\.(md|txt|yml|yaml|json)$'; then
            echo "skip=false" >> $GITHUB_OUTPUT
          else
            echo "skip=true" >> $GITHUB_OUTPUT
          fi

  review:
    needs: should-review
    if: needs.should-review.outputs.skip == 'false'
    # ... review job

When NOT to Use Claude Code in CI¶

Not every CI task benefits from an LLM. Claude Code adds latency (30-120 seconds per invocation) and cost. Use traditional tools when they are sufficient:

Task Use Claude Code? Better Alternative
Formatting checks No ruff format --check, prettier --check
Static type checking No pyright, mypy, tsc --noEmit
Dependency vulnerability scan No pip-audit, npm audit, safety check
Test execution No pytest, jest, go test
Docker image build No docker build
Secret scanning (regex) No gitleaks, trufflehog
License compliance No license-checker, pip-licenses
Code coverage threshold No coverage, istanbul
PR classification Yes No good alternative for nuanced classification
Code review for logic bugs Yes Static analysis catches only a subset
Security audit beyond regex Yes LLM reasoning finds logic-level vulnerabilities
Breaking change analysis Yes Requires understanding of API semantics
Release note generation Yes Git log alone produces developer-facing notes, not user-facing

The rule of thumb: if a deterministic tool can do the job, use the deterministic tool. Claude Code is for tasks that require judgment, reasoning, or natural language understanding.

The Complete Production Pipeline¶

Here is a production-grade CI/CD pipeline that integrates Claude Code at every stage. This is the full workflow a professional team would use:

yaml
# .github/workflows/pipeline.yml

name: CI/CD Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

permissions:
  contents: write
  pull-requests: write
  issues: write

env:
  NODE_VERSION: '20'
  PYTHON_VERSION: '3.12'

jobs:
  # ── Stage 1: Build and Test ─────────────────────────────
  test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16-alpine
        env:
          POSTGRES_USER: test
          POSTGRES_PASSWORD: test
          POSTGRES_DB: test
        ports: ['5432:5432']
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5

    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: pnpm

      - run: pnpm install --frozen-lockfile
      - run: pnpm exec prisma migrate deploy
        env:
          DATABASE_URL: postgresql://test:test@localhost:5432/test
      - run: pnpm test -- --coverage
        env:
          DATABASE_URL: postgresql://test:test@localhost:5432/test
          JWT_SECRET: ci-test-secret

      - uses: actions/upload-artifact@v4
        with:
          name: coverage
          path: coverage/

  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: pnpm

      - run: pnpm install --frozen-lockfile
      - run: pnpm lint
      - run: pnpm exec tsc --noEmit

  # ── Stage 2: Claude Code Review (PRs only) ─────────────
  claude-review:
    runs-on: ubuntu-latest
    if: github.event_name == 'pull_request'
    needs: [test, lint]
    timeout-minutes: 10
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          model: "claude-sonnet-4-6"
          effort: "medium"
          timeout: 300
          prompt: |
            Review this PR comprehensively:

            ## Code Quality
            - Logic bugs
            - Error handling gaps
            - Missing edge cases
            - Performance concerns

            ## Security
            - OWASP Top 10 violations
            - Auth/authz issues
            - Input validation
            - Secrets in code

            ## Testing
            - Are new code paths tested?
            - Are edge cases covered?

            ## Breaking Changes
            - API contract changes
            - Database schema changes

            Format each finding as:
            **[CRITICAL/IMPORTANT/SUGGESTION]** `file:line` --- description

  # ── Stage 3: Build (main branch only) ──────────────────
  build:
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    needs: [test, lint]
    steps:
      - uses: actions/checkout@v4

      - name: Build Docker images
        run: |
          docker build -f packages/api/Dockerfile -t app-api:${{ github.sha }} .
          docker build -f packages/web/Dockerfile -t app-web:${{ github.sha }} .

      - name: Push to registry
        run: |
          echo "${{ secrets.REGISTRY_PASSWORD }}" | \
            docker login ghcr.io -u ${{ github.actor }} --password-stdin
          docker tag app-api:${{ github.sha }} ghcr.io/${{ github.repository }}-api:${{ github.sha }}
          docker push ghcr.io/${{ github.repository }}-api:${{ github.sha }}

  # ── Stage 4: Deploy ────────────────────────────────────
  deploy:
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    needs: [build]
    environment: production
    steps:
      - uses: actions/checkout@v4

      - name: Deploy
        run: |
          # Your deployment commands here
          echo "Deploying ${{ github.sha }}"

  # ── Stage 5: Post-Deploy Verification ──────────────────
  verify:
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    needs: [deploy]
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          model: "claude-haiku-3-5"
          prompt: |
            Generate release notes for commit ${{ github.sha }}.
            Read the git log for this commit and any PR associated with it.
            Write user-facing release notes categorized as:
            Added, Changed, Fixed, Security.

Pipeline Flow¶

The pipeline follows this sequence:

PR opened
  |---> Tests run (parallel with lint)
  |---> Lint runs (parallel with tests)
  |---> Claude reviews PR (after tests + lint pass)
  |---> Human reviews (informed by Claude's analysis)
  |---> Merge to main
          |---> Tests run again
          |---> Build Docker images
          |---> Deploy to production
          |---> Claude generates release notes

Claude Code participates at two points: reviewing the PR before merge, and generating release notes after deploy. The human reviewer sees Claude’s analysis alongside the diff, making their review faster and more thorough.

API Key Security¶

Your Anthropic API key in CI is a production secret. Treat it accordingly:

  1. Never print it in logs. GitHub Actions masks secrets automatically, but verify by checking workflow logs after the first run.
  2. Use environment-scoped secrets. GitHub allows you to restrict secrets to specific environments (staging, production). Create separate API keys per environment.
  3. Rotate regularly. Create a new API key quarterly and update the secret.
  4. Set spending limits. On the Anthropic console, set a monthly spending cap for the API key used in CI. This is your financial kill switch.
  5. Use a dedicated API key for CI. Do not share your personal development API key with CI. When the CI key hits its budget, your local development is unaffected.

Timeout Configuration¶

CI jobs should have timeouts to prevent runaway costs:

yaml
jobs:
  review:
    runs-on: ubuntu-latest
    timeout-minutes: 10    # Kill the entire job after 10 minutes
    steps:
      - uses: anthropics/claude-code-action@v1
        with:
          timeout: 300     # 5 minute timeout for Claude specifically

Always set both: timeout-minutes on the job (GitHub-level kill switch) and timeout on the action (Claude Code-level limit). Belt and suspenders.

Troubleshooting CI Integration¶

Common Failure Modes¶

Claude Code returns empty or truncated output. This usually means the timeout was too short or max_tokens was too low. Increase timeout to 600 seconds and max_tokens to 8192 for complex reviews. Check the workflow logs for timeout errors.

The GitHub Action fails with authentication errors. Verify the ANTHROPIC_API_KEY secret is set correctly. Go to Settings > Secrets > Actions, edit the secret, and paste the key again. Common mistake: trailing whitespace in the secret value.

Claude Code reviews are not appearing as PR comments. Check the workflow’s permissions block. You need pull-requests: write for the action to post comments. Also verify that the workflow runs on pull_request events, not push events — push events do not have an associated PR to comment on.

Reviews are inconsistent between runs. LLMs are non-deterministic. The same diff can produce different findings on different runs. This is expected behavior, not a bug. If you need consistency, pin the model version and use effort: "high" which tends to produce more stable output due to deeper reasoning.

CI costs spiking unexpectedly. Check for: (1) missing timeout-minutes on jobs, (2) missing conditional execution for docs-only PRs, (3) a workflow running on push AND pull_request (double review), (4) Dependabot/Renovate creating many dependency PRs that all trigger reviews.

Azure DevOps pipeline cannot find Claude Code. The CLI is installed per-step, not globally. Each step in Azure DevOps runs in a fresh shell context. Either install Claude Code in a setup step that other steps depend on, or install it in each step that needs it.

What Most People Get Wrong¶

Running Opus in CI for every PR. Opus is powerful but expensive. A code review with Opus costs roughly 5-10x what Sonnet costs. Reserve Opus for security audits on high-risk PRs. Use Sonnet for standard reviews and Haiku for classification and simple tasks. A team that switches from Opus-everywhere to tiered model selection typically reduces CI LLM costs by 70-80%.

No timeout on Claude Code CI steps. Without a timeout, a runaway prompt can consume your entire API budget. Always set timeout on the action and timeout-minutes on the job. A single stuck review with Opus can cost $10+ if it loops for 15 minutes.

Reviewing every file on every PR. If only documentation files changed, you do not need a security review. Use path filters and conditional execution to skip expensive reviews for trivial changes. The should-review pattern shown earlier saves hundreds of dollars per month on active repositories.

Ignoring CI costs until the bill arrives. Track your Claude Code CI spend weekly. A team of 10 engineers doing 5 PRs per day with Sonnet reviews costs roughly $150/week. That is reasonable. But add Opus security reviews on every PR and it jumps to $1,000/week. Choose your battles.

Using Claude Code for tasks deterministic tools handle better. Formatting? Use ruff or prettier. Type checking? Use pyright or tsc. Dependency vulnerabilities? Use pip-audit or npm audit. These tools are faster, cheaper, and more reliable for their specific tasks. Claude Code is for the tasks that require judgment — logic bug detection, security reasoning, breaking change analysis, and natural language generation.

Not setting max-turns. In interactive mode, you watch Claude work and can interrupt it. In CI, nobody is watching. A 50-turn agentic loop that reads every file in your repository, runs tests, reads results, and iterates is technically working as designed — but it is also costing $5-15 per PR. Set --max-turns to the minimum needed for each task.

Treating Claude Code CI output as authoritative. Claude Code finds real bugs. It also produces false positives. Never gate deployments solely on Claude Code output. Use it as an additional signal alongside tests, linting, and type checking. A human reviewer should have final say on critical findings. The best practice: Claude Code posts findings as a PR comment, the human reviewer triages them alongside the rest of the review.

Running the same prompt for every language and framework. A Python project needs different review instructions than a TypeScript project. A database migration needs different scrutiny than a UI component. Customize your prompts per file path and change type. The path-filtered workflow pattern (triggering different reviews for src/auth/** vs src/ui/** vs prisma/**) costs the same to maintain and produces far better results.

Chapter Summary¶

  1. Claude Code runs natively in CI/CD pipelines. The official GitHub Action (anthropics/claude-code-action) handles PR reviews, classification, release notes, and arbitrary prompts. For Azure DevOps and other CI systems, install the CLI and use --print mode with --output-format json. Always set --max-turns and timeouts.

  2. Match the model to the task, ruthlessly. Haiku for classification (~$0.01/run), Sonnet for code reviews (~$0.15/run), Opus only for critical security audits (~$1.00/run). A tiered pipeline that classifies first, reviews second, and security-audits only high-risk PRs keeps costs at $8-25/week for active teams.

  3. Layer your pipeline: deterministic tools first, Claude Code second. Run linting, type checking, tests, and static security scanning before Claude Code touches the PR. This means Claude never reviews code that fails basic checks, saving money and producing better results because it focuses on the issues that deterministic tools cannot catch.

  4. Pre-commit hooks and git hooks are your local defense. Secret scanning, commit message validation, and quick Claude Code security scans catch issues before they reach CI. Wire them in .claude/settings.json for Claude Code hooks or .githooks/ for standard git hooks.

  5. Cost management is not optional — it is operational. Set budget alerts, use conditional execution to skip reviews for trivial changes, cache results for unchanged diffs, use dedicated CI API keys with spending limits, and review your weekly spend. The difference between a well-managed Claude Code CI setup ($100/month) and an unmanaged one ($2,000/month) is entirely in the configuration, not the technology.
