Part VI: Advanced Mastery

Chapter 21: Custom Agents and Intent Routing¶ ~19 min read

You open Claude Code and type three words: “audit the auth.” Within seconds, an Opus-powered security reviewer agent is dissecting your authentication module, checking for OWASP Top 10 vulnerabilities, tracing injection paths, and generating a severity-ranked findings table with CWE references. You didn’t invoke a slash command. You didn’t specify a model. You didn’t even say “security.” Claude read your intent, matched it to the right specialist, and delegated.

This is what custom agents and intent routing unlock: a Claude Code session that behaves like a team of specialists, each with the right model, the right tools, and the right expertise for the task at hand. In this chapter, you’ll build that team from scratch.

What Agents Are¶

An agent in Claude Code is a specialized sub-instance of Claude with its own model, tool access, system prompt, and behavioral constraints. Think of agents as roles on a development team. Your main Claude Code session is the tech lead — it understands the full picture and delegates to specialists when the task demands focused expertise.

Agents differ from the main session in three critical ways:

  1. Isolated context — each agent runs in a forked context, so its work doesn’t pollute your main conversation
  2. Constrained tools — you control exactly which tools the agent can use
  3. Model override — a security audit can use Opus for deep reasoning while a documentation task uses Sonnet for speed

Agents live as markdown files in .claude/agents/. When Claude encounters a task matching an agent’s description, it can spawn that agent automatically.

The Agent Definition Format¶

Every agent is a single markdown file in .claude/agents/ with YAML frontmatter followed by a system prompt. Here is the anatomy:

markdown
---
name: security-reviewer
description: Reviews code for security vulnerabilities. Use when user mentions "security", "vulnerability", "audit", "OWASP", "CVE", "injection".
tools: Read, Grep, Glob, Bash
model: opus
effort: high
memory: project
---

You are a senior security engineer specializing in application security.

When reviewing code:
1. Check OWASP Top 10 vulnerabilities
2. Look for injection points (SQL, command, XSS)
3. Verify authentication and authorization logic
4. Check for secrets, hardcoded credentials
5. Assess data exposure risks

Output format:
| Severity | CWE | Finding | Location | Fix |
|----------|-----|---------|----------|-----|

The frontmatter is structured configuration. The body after the --- delimiter is the system prompt — free-form instructions that shape how the agent thinks and responds.

Frontmatter Fields — Complete Reference¶

Every field controls a specific aspect of agent behavior:

name¶

The identifier Claude uses to reference this agent. Keep it lowercase with hyphens. This is how you explicitly invoke it (“use the security-reviewer agent”) and how Claude references it in delegation decisions.

yaml
name: security-reviewer

description¶

This is the single most important field. Claude reads every agent’s description to decide whether to delegate a task. If your description doesn’t contain the right trigger words, the agent never gets used.

yaml
# Good: specific trigger words that Claude can match
description: Reviews code for security vulnerabilities. Use when user mentions "security", "vulnerability", "audit", "OWASP", "CVE", "injection", "XSS", "auth bypass", "secrets in code".

# Bad: vague, no trigger words
description: Helps with code issues.

Write descriptions as if you’re telling a dispatcher: “Send this specialist when someone says X, Y, or Z.”

tools¶

Restricts which tools the agent can access. This is a security boundary — a documentation agent has no business running Bash commands.

yaml
# Full access for investigation
tools: Read, Grep, Glob, Bash

# Read-only for analysis
tools: Read, Grep, Glob

# With web access for research
tools: Read, Grep, Glob, WebFetch, WebSearch

Available tools: Read, Write, Edit, Bash, Grep, Glob, WebFetch, WebSearch, NotebookEdit, Agent. Omit tools the agent shouldn’t have.

model¶

Override the model for this specific agent. This is where cost optimization meets quality requirements.

yaml
model: opus    # Deep analysis, complex reasoning, security audits
model: sonnet  # Balanced — code review, documentation, research
model: haiku   # Fast, cheap — simple lookups, formatting, quick checks

Use Opus for tasks where mistakes are costly (security, debugging production issues). Use Sonnet for everyday work. Use Haiku for mechanical tasks like formatting or simple lookups.

effort¶

Controls reasoning depth. Higher effort means more internal deliberation before responding.

yaml
effort: low    # Quick, surface-level
effort: medium # Standard analysis
effort: high   # Deep reasoning, considers edge cases
effort: max    # Maximum deliberation (most tokens consumed)

maxTurns¶

Limits how many agentic loop iterations the agent can take. Without this, an agent investigating a complex bug might run for dozens of turns, consuming significant tokens.

yaml
maxTurns: 10   # Stop after 10 tool-use cycles
maxTurns: 5    # Quick investigation, report back
maxTurns: 25   # Extended deep dive

memory¶

Controls where the agent persists insights across sessions.

yaml
memory: user     # Global memory — shared across all projects
memory: project  # Per-project memory — scoped to current project
memory: local    # Gitignored memory — private to this machine

A security reviewer with memory: project remembers past audit findings for this project. A personal productivity agent with memory: local keeps notes that never leave your machine.

isolation¶

Determines whether the agent operates in the same git state as your main session or in an isolated worktree.

yaml
isolation: worktree  # Agent gets its own git worktree (separate branch)

Worktree isolation is essential for agents that modify code — their changes don’t affect your working directory until explicitly merged. This is how /batch achieves parallel file modifications safely.

Five Complete Agent Definitions¶

Here are five production-ready agents that form a professional development team.

1. Security Reviewer¶

markdown
---
name: security-reviewer
description: Reviews code for security vulnerabilities, injection flaws, authentication bugs, and secret exposure. Use when user mentions "security", "vulnerability", "audit", "OWASP", "CVE", "CWE", "injection", "XSS", "CSRF", "auth bypass", "secrets", "pentest", or "hardening".
tools: Read, Grep, Glob, Bash
model: opus
effort: high
memory: project
---

You are a senior application security engineer with deep expertise in OWASP Top 10, CWE classifications, and secure coding practices.

## Review Protocol

For every file or module you review:

1. **Input Validation** — Trace all external input (HTTP params, headers, body, file uploads, environment variables). Verify sanitization at every trust boundary.
2. **Injection Analysis** — Check for SQL injection, command injection, XSS (reflected, stored, DOM-based), template injection, LDAP injection, path traversal.
3. **Authentication & Authorization** — Verify session management, token handling, password storage (bcrypt/argon2, not MD5/SHA), privilege escalation paths, IDOR vulnerabilities.
4. **Secrets & Configuration** — Scan for hardcoded credentials, API keys, connection strings, private keys. Check that secrets come from environment variables or secret managers.
5. **Data Exposure** — Identify PII leakage in logs, error messages, API responses. Check that sensitive fields are excluded from serialization.
6. **Cryptography** — Verify use of strong algorithms (AES-256, RSA-2048+), proper IV/nonce handling, no ECB mode, no custom crypto.
7. **Dependencies** — Flag known vulnerable packages if visible in imports or config files.

## Output Format

Present findings as a severity-ranked table:

| Severity | CWE | Finding | Location | Fix |
|----------|-----|---------|----------|-----|

Severity levels: CRITICAL, HIGH, MEDIUM, LOW, INFO.

For CRITICAL and HIGH findings, provide complete fix code — not just a description. For MEDIUM and below, a one-line fix description suffices.

End with a summary: total findings by severity, overall risk assessment, and the top 3 actions to take immediately.

2. Code Reviewer¶

markdown
---
name: code-reviewer
description: Reviews code for bugs, performance issues, readability problems, and best practice violations. Use when user says "review", "check this code", "code review", "find bugs", "look at this", or "is this good".
tools: Read, Grep, Glob
model: sonnet
effort: high
memory: project
---

You are a senior software engineer conducting thorough code reviews. Your priorities, in order:

1. **Correctness** — Logic errors, off-by-one, null/undefined handling, race conditions, resource leaks
2. **Security** — Flag obvious issues but defer deep security analysis to the security-reviewer agent
3. **Performance** — N+1 queries, unnecessary allocations, missing indexes, inefficient algorithms
4. **Readability** — Naming, function length, nesting depth, dead code, inconsistent patterns
5. **Maintainability** — Test coverage gaps, missing error handling, tight coupling

## Review Protocol

1. Read the entire file or changeset before commenting
2. Understand the purpose and context of the code
3. Check the surrounding code for patterns to follow
4. Identify issues in priority order (correctness first, style last)

## Output Format

**[CRITICAL]** `file.py:42` — Description of the bug and how to fix it

**[IMPORTANT]** `file.py:87` — Description of the performance issue

**[SUGGESTION]** `file.py:103` — Description of the readability improvement

Skip nitpicks. Skip formatting issues that a linter handles. Focus on what breaks in production or causes maintenance pain.

End with: "N issues found: X critical, Y important, Z suggestions."

3. Debugger¶

markdown
---
name: debugger
description: Investigates bugs, errors, and unexpected behavior through systematic root cause analysis. Use when user mentions "bug", "error", "broken", "not working", "fails", "crash", "traceback", "stack trace", "exception", or "unexpected behavior".
tools: Read, Grep, Glob, Bash
model: opus
effort: high
maxTurns: 20
memory: project
---

You are a senior debugging specialist. Your method is systematic root cause analysis — not guessing.

## Investigation Protocol

1. **Reproduce** — Understand the exact symptoms. What was expected vs. what happened?
2. **Hypothesize** — Form 2-3 hypotheses about root cause, ranked by likelihood.
3. **Investigate** — For each hypothesis, search the codebase for evidence. Read relevant files. Check git blame for recent changes. Run targeted tests if available.
4. **Narrow** — Eliminate hypotheses with evidence until one remains.
5. **Root Cause** — Identify the exact file, line, and condition that causes the bug.
6. **Fix** — Write a minimal fix that addresses the root cause without side effects.
7. **Verify** — Run tests to confirm the fix works and nothing else broke.

## Output Format

### Root Cause
`file.py:42` — Exact description of what goes wrong and why.

### Evidence
- [What you found that confirms this]
- [What you ruled out and why]

### Fix
```diff
- old code
+ new code

Prevention¶

How to prevent this class of bug in the future (linting rule, type annotation, test pattern).

Never guess. If you can’t determine root cause with available evidence, say what you’ve ruled out and what additional information you need.


### 4. Researcher

```markdown
---
name: researcher
description: Performs deep technical research using web search and documentation. Use when user says "research", "what is", "latest", "compare", "should I use", "investigate", "look into", "find out about", or "what are the options for".
tools: Read, Grep, Glob, WebFetch, WebSearch
model: sonnet
effort: high
maxTurns: 15
memory: user
---

You are a senior technical researcher. Your job is to find accurate, current information and synthesize it into actionable intelligence.

## Research Protocol

1. **Define the question** — What specifically does the user need to know?
2. **Search broadly** — Use WebSearch for current information. Use WebFetch for specific documentation pages. Cross-reference multiple sources.
3. **Verify accuracy** — Check dates, version numbers, and claims against official documentation. Never present a single source's claim as fact.
4. **Synthesize** — Combine findings into a clear, actionable answer.

## Output Format

### Summary
2-3 sentences answering the core question.

### Key Findings
- Specific, sourced findings as bullet points
- Include version numbers, dates, metrics where relevant

### Comparison (if comparing options)
| Criteria | Option A | Option B | Option C |
|----------|----------|----------|----------|

### Recommendation
What to use and why, with trade-offs noted.

### Sources
- [Source Title](URL) — what this source contributed

Never show failed searches, 404 errors, or "I couldn't find" messages. Only present the information you did find. If a search yields nothing useful, try different search terms before reporting back.

5. Doc Writer¶

markdown
---
name: doc-writer
description: Writes technical documentation including READMEs, API docs, ADRs, runbooks, and inline documentation. Use when user says "document", "write docs", "README", "ADR", "runbook", "API docs", "update documentation", or "document this".
tools: Read, Grep, Glob
model: sonnet
effort: medium
memory: project
---

You are a senior technical writer who understands that good documentation is code — it needs the same care, consistency, and maintenance.

## Documentation Protocol

1. **Read the code** — Understand what you're documenting by reading the actual implementation, not just function names.
2. **Read existing docs** — Match the existing style, format, and conventions. If the project uses JSDoc, write JSDoc. If ADRs use a specific template, follow it.
3. **Write for the reader** — The reader is a senior engineer who is new to this codebase. They understand programming concepts but not your project's specific decisions.
4. **Be accurate** — Every code reference, file path, and command must be verified against the actual codebase.

## Document Types

### README
- What it does (one paragraph)
- Quick start (< 5 steps to running)
- Architecture overview (if complex)
- Configuration reference
- Development setup

### ADR (Architecture Decision Record)
- Title: ADR-NNNN: Decision Title
- Status: Proposed / Accepted / Deprecated
- Context: What problem are we solving?
- Decision: What we chose and why
- Consequences: Trade-offs accepted

### Runbook
- When to use this runbook
- Prerequisites
- Step-by-step procedure (numbered, exact commands)
- Verification (how to confirm each step worked)
- Rollback procedure
- Escalation path

### API Documentation
- Endpoint: method + path
- Purpose: one sentence
- Request: headers, params, body (with example)
- Response: status codes, body (with example)
- Error responses: all possible errors

Never generate placeholder text. Every section either has real content or doesn't exist.

How Agents Get Delegated To¶

There are three paths from your prompt to agent execution.

Automatic Delegation¶

This is the default and most common path. Claude reads every agent’s description field, compares it against your prompt, and delegates when the match is strong.

You: "Check if the payment module has any SQL injection vulnerabilities"

Claude's internal reasoning:
- Keywords: "SQL injection", "vulnerabilities"
- Matches: security-reviewer description
- Action: delegate to security-reviewer agent

→ Security reviewer agent spawns with Opus model
→ Reads payment module files
→ Returns severity-ranked findings
→ Main session presents results to you

The quality of automatic delegation depends entirely on the description field. If you find an agent isn’t being used when it should be, add more trigger words to its description. If it’s being used when it shouldn’t be, make the description more specific.

Explicit Delegation¶

You can directly invoke an agent by name:

You: "Use the debugger agent to investigate why the user sync job fails silently"

Claude spawns the debugger agent regardless of keyword matching. This is useful when you know which specialist you want and don’t want to rely on automatic matching.

Delegation via Skills¶

Skills can specify an agent to handle their execution:

yaml
---
name: security-audit
description: Comprehensive security audit of the codebase
context: fork
agent: security-reviewer
---

When you run /security-audit, Claude forks context and delegates to the security-reviewer agent. This combines the invocability of skills (slash commands) with the specialization of agents (model override, tool restrictions, custom prompt).

Agent vs. Skill — Decision Matrix¶

Agents and skills serve different purposes. Here’s when to use which:

Criteria Agent Skill
Fixed, repeatable steps No — agents exercise judgment Yes — skills follow a script
Deep reasoning required Yes — use Opus agents Less common
Specific model needed Yes — each agent has its own model Possible via frontmatter
User-invocable with / Not directly Yes — /skill-name
Long investigation Yes — agents explore freely No — skills execute steps
Mechanical consistency No — agents may vary Yes — same steps every time
Context isolation Built-in (forked context) Optional (context: fork)
Cost control maxTurns limits Predictable step count

Rule of thumb: If the task has clear steps that don’t change (generate changelog, run deploy check, scaffold project), use a skill. If the task requires exploration, judgment, and domain expertise (security audit, debugging, code review), use an agent.

Intent Routing — Auto-Detecting What You Want¶

Intent routing is the system that turns “audit the auth” into a security review without you specifying the workflow. It’s the mechanism behind Claude Code’s auto-execute behavior.

The Routing Table¶

Intent routing works through a mapping of trigger keywords to workflows. You define this mapping in your CLAUDE.md or in a dedicated file that CLAUDE.md imports:

markdown
## Auto-Execute Protocol

| User says (examples) | Route to |
|---|---|
| mentions bug, error, stack trace | **Debug** → hypothesize, investigate, fix |
| mentions review, check, "look at" | **Code review** → severity-ranked findings |
| mentions security, vulnerability, OWASP | **Security audit** → CWE table, fix code |
| mentions research, "what is", compare | **Research** → multi-source synthesis |
| mentions refactor, clean up, simplify | **Refactor** → preserve behavior, small steps |
| mentions deploy, ship, release | **Deploy check** → full readiness validation |
| mentions performance, slow, optimize | **Perf profile** → bottleneck analysis |

Building Your Own Routing Table¶

Start with the workflows your team uses daily, then map natural language to each one:

markdown
| Trigger Words | Route To | Agent | Model |
|---------------|----------|-------|-------|
| bug, error, broken, crash, traceback | Debug workflow | debugger | opus |
| review, check, "how's this" | Code review | code-reviewer | sonnet |
| security, vuln, OWASP, inject, XSS | Security audit | security-reviewer | opus |
| research, "what is", compare, latest | Research | researcher | sonnet |
| docs, README, document, ADR | Documentation | doc-writer | sonnet |
| slow, perf, optimize, bottleneck | Performance | — (inline) | — |
| deploy, ship, release, "ready?" | Deploy check | — (inline) | — |
| refactor, clean, simplify | Refactoring | — (inline) | — |

Not every workflow needs an agent. Simple workflows like deploy checks can run inline in the main session. Reserve agents for tasks that benefit from model overrides, tool restrictions, or domain-specific prompts.

The Auto-Execute Protocol¶

The routing table is just data. The auto-execute protocol is the instruction that tells Claude to use it. Place this at the top of your CLAUDE.md:

markdown
## Auto-Execute Protocol

**Step 1 — Decode Intent** (silent)
Parse the user's prompt. Identify: WHAT they want, WHAT target, HOW deep.

**Step 2 — Route to Workflow** (silent)
Match intent to the routing table. Select the matching workflow.

**Step 3 — Expand & Execute** (visible to user)
Execute the full workflow. Deliver the final result. Never ask
"would you like me to..." — just do it.

Three rules make this work:

  1. Silent routing — The user never sees the decode/route steps. They see only the result.
  2. Full execution — Once routed, Claude executes the complete workflow, not just the first step.
  3. One question maximum — If Claude truly can’t proceed (ambiguous target), it asks one focused question, then executes on the answer.

This protocol eliminates the most common complaint about AI assistants: the endless back-and-forth of “would you like me to do X?” Just do X.

Database Agent¶

markdown
---
name: database-expert
description: Expert in our PostgreSQL database schema, query optimization, and migrations. Use when user mentions "database", "query", "migration", "schema", "SQL", "index", "slow query", or "table".
tools: Read, Grep, Glob, Bash
model: sonnet
effort: high
memory: project
---

You are the database expert for this project. You know:

- Our schema is defined in `db/migrations/` (Alembic migrations)
- The current schema is documented in `db/schema.sql`
- We use PostgreSQL 16 with TimescaleDB extension
- Connection pooling via PgBouncer (max 100 connections)

- Always check for missing indexes on WHERE/JOIN columns

2. Arrange: set up minimal state using fixtures
3. Act: call the function/endpoint under test

2. Read the implementation to find the mismatch
3. Determine if the test or the implementation is wrong
4. Fix the correct one

Multi-Agent Orchestration¶

Single agents are useful. Multiple agents working together are transformative. Here are the orchestration patterns.

Parallel Review¶

Run multiple specialists simultaneously on the same code:

You: "Review the auth module for security AND code quality"

Claude spawns both the security-reviewer (Opus) and code-reviewer (Sonnet) agents in parallel. Each examines the auth module through its lens. Results return independently, and Claude synthesizes them into a unified review.

This is faster than sequential review (both agents work concurrently) and produces deeper analysis (each agent focuses on its specialty without distraction).

Pipeline Orchestration¶

Agents feed their output into the next stage:

You: "Research the best approach to rate limiting, then implement it"

Stage 1: researcher agent investigates rate limiting approaches, compares token bucket vs. sliding window vs. fixed window, and recommends an approach.

Stage 2: Main session (or an implementation agent) takes the recommendation and implements it using the chosen approach.

The key is that stage 2 benefits from stage 1’s research without the user having to relay information between steps.

Debate Pattern¶

Two agents review the same code and their findings are compared:

You: "Have the security-reviewer and code-reviewer both examine the payment processing module, then synthesize their findings"

Both agents independently analyze the code. Where they agree, confidence is high. Where they disagree, Claude highlights the tension and presents both perspectives. This surfaces blind spots that a single reviewer might miss.

Agent Communication Patterns¶

How agents interact with the main session and each other:

One-Shot Delegation¶

The simplest pattern. Claude spawns an agent, waits for the result, and presents it.

Main session → spawns agent → agent investigates → agent returns findings → main session presents to user

This is the default for most tasks. The agent runs to completion (or until maxTurns), and its findings become part of the main conversation.

Background Agents¶

For long-running investigations, agents can run in the background while you continue working:

You: "In the background, have the security agent audit all files in src/api/"
You: [continue working on something else]
[Later: agent finishes, results appear]

The run_in_background capability lets agents work asynchronously. This is ideal for comprehensive audits, deep research, or large-scale code analysis that would otherwise block your workflow.

Result Aggregation¶

When multiple agents return results, the main session aggregates them:

Main session:
├── Spawns security-reviewer → returns 3 critical, 5 medium findings
├── Spawns code-reviewer → returns 1 critical, 8 suggestions
└── Aggregates: deduplicates, ranks by severity, presents unified report

The main session handles synthesis — merging overlapping findings, resolving conflicting recommendations, and presenting a coherent summary.

Cost Management for Agents¶

Each agent is a separate API session with its own token consumption. Without cost awareness, agents can become expensive.

Model Selection Per Agent¶

This is the single biggest cost lever. Assign models based on task criticality:

Task Model Why
Security audit Opus Mistakes here cost millions
Debugging production Opus Complex reasoning required
Code review Sonnet Good balance of quality/cost
Research Sonnet Web synthesis doesn’t need Opus
Documentation Sonnet Adequate for prose generation
Deploy checks Haiku Mechanical checks, no judgment
Formatting Haiku Trivial transformations

Turn Limits¶

Set maxTurns to prevent runaway investigations:

yaml
# Quick check — 5 turns is enough
maxTurns: 5

# Standard investigation — 10-15 turns
maxTurns: 15

# Deep dive — 20-25 turns max
maxTurns: 25

Without maxTurns, an Opus agent investigating a complex bug could run for 40+ turns, consuming significant tokens. Set limits based on the task’s expected complexity.

Monitoring¶

Use /cost during your session to check cumulative spend across all agents. If you notice an agent consuming disproportionate tokens, consider:

  • Reducing its maxTurns
  • Switching to a cheaper model
  • Narrowing its scope via the system prompt
  • Using a skill with fixed steps instead

Cost Estimation Rules of Thumb¶

  • Opus agent, 10 turns, reading ~5 files: approximately $0.50-1.00
  • Sonnet agent, 10 turns, reading ~5 files: approximately $0.10-0.25
  • Haiku agent, 5 turns, reading ~3 files: approximately $0.01-0.05
  • Parallel agents multiply these costs

Practical Tips for Effective Agents¶

Writing Good Descriptions¶

The description field is your agent’s resume. Claude reads it to decide delegation. Include:

  • What the agent does — one sentence summary
  • Trigger words — exact words the user might say
  • Negative triggers — “Do NOT use for general code review” to prevent false matches
yaml
# Effective description
description: >
  Investigates database performance issues including slow queries,
  missing indexes, N+1 patterns, and connection pool exhaustion.
  Use when user mentions "slow query", "database performance",
  "N+1", "query optimization", "missing index", "connection pool",
  or "database bottleneck". Do NOT use for schema design or migrations.

Keeping Agent Prompts Focused¶

An agent that tries to do everything does nothing well. Each agent should have:

  • A single area of expertise
  • A clear protocol (numbered steps)
  • A specific output format
  • Explicit boundaries on what it does NOT do

Testing Agents¶

Before relying on an agent in production work, test it:

  1. Ask Claude: “When would you use the security-reviewer agent?” — it will quote the description
  2. Give a prompt that should trigger the agent — verify it delegates
  3. Give a prompt that should NOT trigger the agent — verify it doesn’t
  4. Check the agent’s output quality on a known file

Evolving Agents Over Time¶

Agents improve with iteration. After using an agent:

  • Add trigger words it missed to the description
  • Remove trigger words that caused false matches
  • Refine the protocol based on what the agent got wrong
  • Adjust maxTurns based on actual usage patterns
  • Consider switching models if quality is too low or cost too high

Chapter Summary¶

  • Agents are specialized Claude instances with their own model, tools, and system prompt, defined as markdown files in .claude/agents/
  • The description field is the most important part of an agent definition — it determines when Claude delegates to the agent
  • Five core agents (security reviewer, code reviewer, debugger, researcher, doc writer) cover most development needs
  • Intent routing maps trigger words to workflows, enabling Claude to auto-detect what you want from short prompts
  • Domain-specific agents encode your project’s unique knowledge (database schema, test frameworks, deployment procedures)
  • Multi-agent orchestration patterns (parallel, pipeline, debate) combine specialists for deeper analysis
  • Cost management through model selection, turn limits, and monitoring keeps agents economical
  • The auto-execute protocol (decode, route, execute) eliminates back-and-forth by having Claude silently route to the right workflow and execute it fully
