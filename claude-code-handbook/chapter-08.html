Chapter 8: Hooks – Automation Guardrails¶ ~20 min read

A developer on your team runs rm -rf . inside a Claude Code session. Not maliciously – they told Claude to “clean up the temp files” and Claude interpreted that as a recursive delete from the current directory. Without hooks, the command executes before anyone blinks. With a PreToolUse hook, a twelve-line shell script catches the pattern, returns exit code 2, and the operation never touches the filesystem. Claude gets feedback explaining why the command was blocked and reformulates a safer approach.

That is what hooks do. They are shell scripts – or webhooks, or LLM prompts, or subagent invocations – that fire at specific points in Claude Code’s execution lifecycle. They can observe, they can transform, and most critically, they can block. Hooks are the difference between trusting an AI agent with good intentions and trusting an AI agent with enforced boundaries.

This chapter covers every hook lifecycle event, the JSON protocol hooks use to communicate, the four hook types available, and five production-ready hook implementations you can drop into your project today. By the end, you will have the knowledge to build custom hooks that enforce your team’s specific guardrails.

The Hook Lifecycle¶

Claude Code exposes seven lifecycle events where hooks can execute. Each event has different capabilities, different timing, and different access to context. Understanding when each fires – and what it can do – is the foundation for everything else in this chapter.

SessionStart¶

Fires once when a Claude Code session begins, before any user interaction occurs.

Timing: After Claude Code loads CLAUDE.md, settings, and rules, but before the first prompt is processed.

Can block: No. SessionStart hooks run for side effects only.

Use cases: Setting environment variables, initializing project state, logging session start times, checking prerequisites (is Docker running? Is the VPN connected?).

What it receives: Session metadata – session ID, project path, working directory.

Session begins
    │
    ▼
┌─────────────────────┐
│ Load CLAUDE.md       │
│ Load settings.json   │
│ Load rules           │
├─────────────────────┤
│ SessionStart hooks   │ ◄── Fire here
└─────────┬───────────┘
          │
          ▼
  Waiting for first prompt

PreToolUse¶

Fires before any tool executes. This is the most powerful hook event because it can block execution entirely.

Timing: After Claude decides to use a tool, after the permission system evaluates (deny/ask/allow), but before the tool actually runs.

Can block: Yes. Exit code 2 blocks the tool call. The stderr output is fed back to Claude as an explanation.

Use cases: Blocking dangerous commands, scanning for secrets before commits, enforcing naming conventions, rate-limiting API calls, requiring confirmation for destructive operations.

What it receives: The tool name and its full input – for Bash tools, that means the exact command string.

This is the event you will wire most often. Every safety guardrail in this chapter uses PreToolUse.

PostToolUse¶

Fires after a tool executes successfully.

Timing: Immediately after the tool completes with a successful result, before the result is sent back to the LLM for the next iteration of the agentic loop.

Can block: No. The tool has already executed. PostToolUse hooks run for side effects.

Use cases: Auto-formatting code after edits, running linters, logging tool usage, updating dashboards, sending notifications for specific operations.

What it receives: The tool name, input, and the tool’s output/result.

Notification¶

Fires when Claude Code sends a notification to the user – typically when a long-running task completes or when Claude needs attention.

Timing: When the notification event is emitted.

Can block: No.

Use cases: Desktop notifications with sound, Slack/Teams messages, logging.

What it receives: The notification title and message.

Stop¶

Fires when Claude finishes its response and is about to return control to the user.

Timing: After the final LLM turn completes, before the response is displayed.

Can block: Yes. A Stop hook returning exit code 2 forces Claude to continue working – effectively acting as a quality gate. The stderr feedback tells Claude what is missing or wrong with its response.

Use cases: Verifying that Claude’s output meets quality criteria, checking that tests were run before claiming a fix is complete, ensuring documentation was updated alongside code changes.

What it receives: The full assistant response and conversation context.

Other Events¶

Claude Code also exposes UserPromptSubmit (validate user input before processing), SubagentStop (verify subagent output), TaskCompleted (post-completion checks), and FileChanged (react to filesystem changes). These follow the same patterns as the events above – some can block, some are observation-only. The five events covered in detail above handle 95% of real-world hook needs.

Multiple PreToolUse hooks execute sequentially, in the order they appear in settings.json. If any hook returns exit code 2, subsequent hooks do not run – the tool call is blocked immediately. This is a fail-fast design: the first objection wins.

Hook Wiring in settings.json¶

Hooks live as scripts in your project (typically .claude/hooks/), but they only fire when wired into settings.json. The wiring configuration tells Claude Code which scripts to run, when to run them, and what tool calls they apply to.

Basic Structure¶

json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "./.claude/hooks/block-destructive.sh",
            "timeout": 5000,
            "statusMessage": "Safety check..."
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "./.claude/hooks/auto-lint-python.sh",
            "timeout": 10000,
            "statusMessage": "Auto-formatting..."
          }
        ]
      }
    ],
    "SessionStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "./.claude/hooks/session-setup.sh",
            "timeout": 5000
          }
        ]
      }
    ],
    "Notification": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "./.claude/hooks/notify-macos.sh",
            "timeout": 3000
          }
        ]
      }
    ]
  }
}

The Matcher Field¶

The matcher field determines which tool calls trigger the hook. It accepts a tool name or a pipe-separated list of tool names:

Matcher Matches
"Bash" Any Bash tool call
"Edit" Any Edit tool call
"Write" Any Write tool call
"Edit\|Write" Edit or Write tool calls
"Bash\|Edit\|Write" Bash, Edit, or Write tool calls
(omitted) All tool calls

When you omit the matcher entirely (as in the SessionStart and Notification examples above), the hook fires for every event of that type regardless of tool. For PreToolUse and PostToolUse, you almost always want a matcher – otherwise your secret-scanning hook would fire on every Read, Grep, and Glob call, not just Bash commands.

Configuration Fields¶

Each hook entry accepts these fields:

Field Required Description
type Yes Hook type: "command", "http", "prompt", or "agent"
command Yes (for command type) Path to the script, relative to project root
timeout No Maximum execution time in milliseconds (default: 60000)
statusMessage No Message shown in the UI while the hook runs

The timeout matters more than you think. A PreToolUse hook that hangs blocks the entire agentic loop. Set aggressive timeouts – 5000ms for safety checks, 10000ms for linting. If your hook needs more than 10 seconds, it is doing too much.

The JSON Stdin Protocol¶

Every hook receives context as a JSON object on standard input. The structure varies by event type, but the core pattern is consistent.

What Hooks Receive¶

For PreToolUse and PostToolUse on a Bash tool:

json
{
  "event": "PreToolUse",
  "tool_name": "Bash",
  "tool_input": {
    "command": "rm -rf ./build",
    "description": "Clean build directory",
    "timeout": 120000
  },
  "session_id": "abc123-def456"
}

For PreToolUse on an Edit tool:

json
{
  "event": "PreToolUse",
  "tool_name": "Edit",
  "tool_input": {
    "file_path": "/Users/dev/project/src/auth.py",
    "old_string": "password = request.body",
    "new_string": "password = sanitize(request.body)"
  },
  "session_id": "abc123-def456"
}

For Notification:

json
{
  "event": "Notification",
  "title": "Task Complete",
  "message": "Finished refactoring auth module. 3 files changed, all tests pass.",
  "session_id": "abc123-def456"
}

Reading JSON in Bash¶

The standard pattern for parsing JSON stdin in a hook script:

bash
#!/usr/bin/env bash
set -euo pipefail

# Read all stdin into a variable
INPUT=$(cat)

# Extract fields using Python's json module (available everywhere)
TOOL_NAME=$(echo "${INPUT}" | python3 -c "
import sys, json
data = json.load(sys.stdin)
print(data.get('tool_name', ''))
")

COMMAND=$(echo "${INPUT}" | python3 -c "
import sys, json
data = json.load(sys.stdin)
print(data.get('tool_input', {}).get('command', ''))
")

Why python3 -c instead of jq? Because python3 is installed on every macOS and most Linux systems. jq is not. For hooks that run on every tool call, you want zero dependency assumptions.

If your team standardizes on jq (and you should if you are writing many hooks):

bash
COMMAND=$(echo "${INPUT}" | jq -r '.tool_input.command // ""')

Exit Codes¶

This is the contract between your hook and Claude Code:

Exit Code Meaning Effect
0 Allow / success Tool call proceeds (PreToolUse) or hook completed (PostToolUse)
2 Block Tool call is blocked. Stderr is sent to Claude as feedback
Any other Non-blocking error Logged, but execution continues. Hook is treated as if it did not object

Exit code 2 is special. It is the only code that blocks execution. Exit code 1 (the standard error code) does not block – it is treated as “the hook itself had a problem” and the tool call proceeds. This is a deliberate design choice: a buggy hook should not halt your workflow.

Providing Feedback to Claude¶

When a PreToolUse hook blocks with exit code 2, the message on stderr is sent directly to Claude as context. Claude uses this feedback to understand why the command was blocked and to formulate an alternative.

Write clear, actionable feedback:

bash
# Bad: Claude learns nothing
echo "Blocked." >&2
exit 2

# Good: Claude knows exactly what went wrong and what to do instead
echo "BLOCKED: 'rm -rf .' would delete the entire project directory. Use 'rm -rf ./build' to clean only the build directory, or list specific files to remove." >&2
exit 2

The structured JSON feedback format for the permission system:

bash
echo '{"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"deny","permissionDecisionReason":"Blocked: rm -rf with wildcard targets project root. Specify exact directory."}}' >&2
exit 2

Both formats work. The plain text format is simpler and sufficient for most hooks. The JSON format integrates more cleanly with Claude Code’s internal permission system.

The Four Hook Types¶

command – Shell Scripts¶

The workhorse. A command hook runs a shell script as a subprocess, passing context on stdin and reading the exit code.

json
{
  "type": "command",
  "command": "./.claude/hooks/my-hook.sh",
  "timeout": 5000
}

The command field is relative to the project root. The script must be executable (chmod +x). It runs with the same user permissions as Claude Code itself.

Use command hooks for: everything that does not require an HTTP call, LLM evaluation, or subagent orchestration. That covers about 90% of hooks.

http – Webhook URLs¶

An http hook sends the JSON context as a POST request to a URL and interprets the HTTP status code as the hook result.

json
{
  "type": "http",
  "url": "https://your-server.com/hooks/pre-tool-check",
  "timeout": 5000
}
HTTP Status Meaning
200 Allow
403 Block (response body = feedback to Claude)
Other Non-blocking error

Use http hooks for: centralized policy enforcement across a team, logging to external systems, integration with security platforms. A team lead can run a policy server that all developers’ Claude Code instances call before executing commands.

prompt – LLM Evaluation¶

A prompt hook sends a prompt to Claude (or another configured model) for evaluation. The model’s response determines whether the action proceeds.

json
{
  "type": "prompt",
  "prompt": "Review this command for safety: {{tool_input.command}}. Is it destructive? Reply YES or NO with explanation."
}

Use prompt hooks for: nuanced safety checks that regex cannot handle. A regex can catch rm -rf /, but it takes an LLM to evaluate whether find . -name '*.log' -delete is dangerous in the context of the current project.

The tradeoff is latency. A prompt hook adds an LLM API call to every matched tool invocation. Use sparingly.

agent – Subagent Invocation¶

An agent hook spawns a Claude Code subagent with its own context, tools, and instructions to evaluate the action.

json
{
  "type": "agent",
  "agent": "./.claude/agents/safety-reviewer.md",
  "timeout": 30000
}

Use agent hooks for: complex multi-step evaluations – the agent can read files, search the codebase, and make informed judgments about whether an action is safe. This is the most powerful hook type and the most expensive.

In practice, most teams use command hooks for 90% of their guardrails, http hooks for centralized team policies, and only reach for prompt or agent hooks when the safety check genuinely requires LLM reasoning.

Five Production-Ready Hook Examples¶

1. block-destructive.sh – Preventing Catastrophic Commands¶

This hook catches the commands that end careers: recursive deletes at root, dropping production databases, force-pushing to main.

bash
#!/usr/bin/env bash
set -euo pipefail

# Read JSON input from stdin
INPUT=$(cat)

# Extract the command being executed
COMMAND=$(echo "${INPUT}" | python3 -c "
import sys, json
data = json.load(sys.stdin)
print(data.get('tool_input', {}).get('command', ''))
")

# Skip if no command (not a Bash tool call)
if [[ -z "${COMMAND}" ]]; then
    exit 0
fi

# Patterns that should NEVER execute
# Each pattern: regex | human-readable description
readonly BLOCKED_PATTERNS=(
    "rm -rf /|Recursive delete from root"
    "rm -rf \.|Recursive delete from current directory"
    "rm -rf ~|Recursive delete of home directory"
    "rm -rf \*|Recursive delete with wildcard"
    "DROP TABLE|SQL table deletion"
    "DROP DATABASE|SQL database deletion"
    "TRUNCATE TABLE|SQL table truncation"
    "git push.*--force.*main|Force push to main branch"
    "git push.*--force.*master|Force push to master branch"
    "git push.*-f.*main|Force push to main branch"
    "git push.*-f.*master|Force push to master branch"
    "chmod -R 777|World-writable permissions"
    "mkfs\.|Format filesystem"
    "dd if=.*of=/dev/|Direct disk write"
    "> /dev/sda|Direct disk overwrite"
    ":(){ :|:& };:|Fork bomb"
)

for pattern_entry in "${BLOCKED_PATTERNS[@]}"; do
    IFS='|' read -r pattern description <<< "${pattern_entry}"
    if echo "${COMMAND}" | grep -qE "${pattern}"; then
        cat >&2 << EOF
BLOCKED: ${description}

Command: ${COMMAND}
Pattern matched: ${pattern}

This command was blocked by the safety hook because it could cause
irreversible damage. If you need to perform this operation, be more
specific about the target (e.g., 'rm -rf ./build' instead of 'rm -rf .').
EOF
        exit 2
    fi
done

exit 0

Wire it:

json
"PreToolUse": [
  {
    "matcher": "Bash",
    "hooks": [
      {
        "type": "command",
        "command": "./.claude/hooks/block-destructive.sh",
        "timeout": 5000,
        "statusMessage": "Safety check..."
      }
    ]
  }
]

2. secret-scan.sh – Catching Secrets Before They Commit¶

This hook fires on git commit commands and scans staged files for patterns that look like secrets.

bash
#!/usr/bin/env bash
set -euo pipefail

INPUT=$(cat)

COMMAND=$(echo "${INPUT}" | python3 -c "
import sys, json
data = json.load(sys.stdin)
print(data.get('tool_input', {}).get('command', ''))
")

# Only scan on git commit commands
if ! echo "${COMMAND}" | grep -qE "^git commit"; then
    exit 0
fi

# Get staged files
STAGED_FILES=$(git diff --cached --name-only 2>/dev/null || true)

if [[ -z "${STAGED_FILES}" ]]; then
    exit 0
fi

# Secret patterns to scan for
readonly SECRET_PATTERNS=(
    "AKIA[0-9A-Z]{16}|AWS Access Key ID"
    "aws_secret_access_key\s*=|AWS Secret Key assignment"
    "ghp_[0-9a-zA-Z]{36}|GitHub Personal Access Token"
    "github_pat_[0-9a-zA-Z_]{82}|GitHub Fine-Grained PAT"
    "sk-[0-9a-zA-Z]{48}|OpenAI/Anthropic API Key"
    "xox[bporas]-[0-9a-zA-Z-]{10,}|Slack Token"
    "-----BEGIN (RSA |EC |DSA )?PRIVATE KEY|Private Key"
    "password\s*=\s*['\"][^'\"]{8,}['\"]|Hardcoded password"
    "ANTHROPIC_API_KEY\s*=\s*['\"]sk-|Anthropic API Key"
)

FOUND_SECRETS=0
FINDINGS=""

while IFS= read -r file; do
    # Skip binary files and deleted files
    if [[ ! -f "${file}" ]]; then
        continue
    fi

    for pattern_entry in "${SECRET_PATTERNS[@]}"; do
        IFS='|' read -r pattern description <<< "${pattern_entry}"
        MATCHES=$(grep -nE "${pattern}" "${file}" 2>/dev/null || true)
        if [[ -n "${MATCHES}" ]]; then
            FOUND_SECRETS=$((FOUND_SECRETS + 1))
            FINDINGS="${FINDINGS}\n  [${description}] ${file}:\n${MATCHES}\n"
        fi
    done
done <<< "${STAGED_FILES}"

if [[ ${FOUND_SECRETS} -gt 0 ]]; then
    cat >&2 << EOF
BLOCKED: Potential secrets detected in staged files.

Found ${FOUND_SECRETS} potential secret(s):
$(echo -e "${FINDINGS}")

Remove these secrets before committing. Use environment variables
or a secrets manager instead of hardcoding credentials.

To inspect staged changes: git diff --cached
To unstage a file: git reset HEAD <file>
EOF
    exit 2
fi

exit 0

3. auto-lint-python.sh – Automatic Formatting After Edits¶

This hook runs ruff after any Python file is edited or written, ensuring code is always formatted and linted.

bash
#!/usr/bin/env bash
set -euo pipefail

INPUT=$(cat)

# Extract the file path from the tool input
FILE_PATH=$(echo "${INPUT}" | python3 -c "
import sys, json
data = json.load(sys.stdin)
tool_input = data.get('tool_input', {})
# Edit tool uses file_path, Write tool uses file_path
print(tool_input.get('file_path', ''))
")

# Only process Python files
if [[ ! "${FILE_PATH}" == *.py ]]; then
    exit 0
fi

# Verify the file exists
if [[ ! -f "${FILE_PATH}" ]]; then
    exit 0
fi

# Check if ruff is available
if ! command -v ruff >/dev/null 2>&1; then
    exit 0
fi

# Run ruff check with auto-fix (suppress output unless there are errors)
ruff check --fix --quiet "${FILE_PATH}" 2>/dev/null || true

# Run ruff format
ruff format --quiet "${FILE_PATH}" 2>/dev/null || true

exit 0

Wire it on PostToolUse with an Edit|Write matcher:

json
"PostToolUse": [
  {
    "matcher": "Edit|Write",
    "hooks": [
      {
        "type": "command",
        "command": "./.claude/hooks/auto-lint-python.sh",
        "timeout": 10000,
        "statusMessage": "Auto-formatting..."
      }
    ]
  }
]

The key design choice: this hook exits 0 even if ruff fails. A linting failure should never block the workflow – it is a convenience, not a guardrail. The || true after each ruff command ensures the hook never exits non-zero due to unfixable lint issues.

5. session-setup.sh – Environment Initialization¶

This hook runs once at session start, setting environment variables and verifying prerequisites.

bash
#!/usr/bin/env bash
set -euo pipefail

# Prevent Python from creating __pycache__ directories
export PYTHONDONTWRITEBYTECODE=1

# Ensure local bin is in PATH
export PATH="${HOME}/.local/bin:${HOME}/bin:${PATH}"

# Verify critical tools are available
MISSING_TOOLS=()

for tool in git python3 node ruff; do
    if ! command -v "${tool}" >/dev/null 2>&1; then
        MISSING_TOOLS+=("${tool}")
    fi
done

if [[ ${#MISSING_TOOLS[@]} -gt 0 ]]; then
    echo "Warning: Missing tools: ${MISSING_TOOLS[*]}" >&2
    echo "Some features may not work correctly." >&2
fi

# Log session start (optional)
echo "[$(date -u +"%Y-%m-%dT%H:%M:%SZ")] Session started in $(pwd)" >> "${HOME}/.claude/session.log" 2>/dev/null || true

exit 0

  }
]

Building a Custom Hook – Step by Step¶

You need a hook that enforces a project rule: no one should add a new Python dependency without also updating the requirements.txt pinned versions. Here is how to build it from scratch.

Step 1: Write the Script¶

Create .claude/hooks/check-deps.sh:

bash
#!/usr/bin/env bash
set -euo pipefail

INPUT=$(cat)

COMMAND=$(echo "${INPUT}" | python3 -c "
import sys, json
data = json.load(sys.stdin)
print(data.get('tool_input', {}).get('command', ''))
")

# Only check pip install commands
if ! echo "${COMMAND}" | grep -qE "pip install|uv add|uv pip install"; then
    exit 0
fi

# Extract the package name (simplified -- handles 'pip install requests')
PACKAGE=$(echo "${COMMAND}" | grep -oE "(pip install|uv add|uv pip install)\s+\S+" | awk '{print $NF}')

if [[ -z "${PACKAGE}" ]]; then
    exit 0
fi

# Check if requirements.txt exists and if the package is already pinned
if [[ -f "requirements.txt" ]]; then
    if ! grep -qi "^${PACKAGE}[=>~!]" requirements.txt; then
        cat >&2 << EOF
REMINDER: You are installing '${PACKAGE}' which is not yet in requirements.txt.

After installing, please add it with a pinned version:
  pip freeze | grep -i ${PACKAGE} >> requirements.txt

Or better, use: pip install ${PACKAGE} && pip freeze | grep -i ${PACKAGE} >> requirements.txt
EOF
        # Exit 0, not 2 -- this is a reminder, not a block
        exit 0
    fi
fi

exit 0

Notice this hook exits 0 even when the package is not pinned. It provides a reminder, not a block. Whether to block or remind is a judgment call for each hook. Blocking pip installs would disrupt the workflow too aggressively – a reminder is sufficient.

Step 2: Make It Executable¶

bash
chmod +x .claude/hooks/check-deps.sh

Forgetting this step is the number one reason hooks silently fail. Claude Code will try to execute the script, get a permission denied error, treat it as a non-blocking error (exit code != 2), and move on. Your hook simply never fires. No error message, no warning, no indication that anything went wrong.

Step 3: Wire It in settings.json¶

json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "./.claude/hooks/check-deps.sh",
            "timeout": 5000,
            "statusMessage": "Checking dependencies..."
          }
        ]
      }
    ]
  }
}

Step 4: Test It¶

Test your hook manually before relying on it in a session:

bash
echo '{"tool_input":{"command":"pip install requests"}}' | ./.claude/hooks/check-deps.sh
echo "Exit code: $?"
bash
echo '{"tool_input":{"command":"git status"}}' | ./.claude/hooks/check-deps.sh
echo "Exit code: $?"

The first command should print the reminder. The second should exit silently with code 0. If either behaves unexpectedly, fix the script before wiring it.

Step 5: Test It Inside Claude Code¶

Start a Claude Code session and ask it to install a package:

Install the httpx library for HTTP requests

You should see the statusMessage (“Checking dependencies…”) appear briefly, followed by the reminder in Claude’s context. Claude will likely respond by both installing the package and adding it to requirements.txt – which is exactly the behavior you wanted.

Advanced Hook Patterns¶

Chaining Multiple Hooks¶

You can wire multiple hooks to the same event. They execute sequentially:

json
"PreToolUse": [
  {
    "matcher": "Bash",
    "hooks": [
      {
        "type": "command",
        "command": "./.claude/hooks/block-destructive.sh",
        "timeout": 5000,
        "statusMessage": "Safety check..."
      },
      {
        "type": "command",
        "command": "./.claude/hooks/secret-scan.sh",
        "timeout": 5000,
        "statusMessage": "Secret scan..."
      },
      {
        "type": "command",
        "command": "./.claude/hooks/check-deps.sh",
        "timeout": 5000,
        "statusMessage": "Dependency check..."
      }
    ]
  }
]

Execution order is top-to-bottom. If block-destructive.sh returns exit 2, secret-scan.sh and check-deps.sh never run. Design your hook chain with the most critical checks first.

Conditional Execution with Matchers¶

Different hooks for different tools:

json
"PreToolUse": [
  {
    "matcher": "Bash",
    "hooks": [
      {
        "type": "command",
        "command": "./.claude/hooks/block-destructive.sh",
        "timeout": 5000
      }
    ]
  },
  {
    "matcher": "Edit|Write",
    "hooks": [
      {
        "type": "command",
        "command": "./.claude/hooks/check-file-size.sh",
        "timeout": 5000
      }
    ]
  }
]

This runs the destructive command blocker only on Bash calls, and a file size checker only on Edit/Write calls. No wasted cycles.

Hook Debugging¶

When a hook does not behave as expected, debug it in isolation:

bash
# Test with exact JSON Claude Code would send
echo '{"event":"PreToolUse","tool_name":"Bash","tool_input":{"command":"rm -rf ."}}' | \
  ./.claude/hooks/block-destructive.sh 2>&1
echo "Exit: $?"
bash
# Test with a command that should pass
echo '{"event":"PreToolUse","tool_name":"Bash","tool_input":{"command":"ls -la"}}' | \
  ./.claude/hooks/block-destructive.sh 2>&1
echo "Exit: $?"

Common debugging issues:

  1. Hook does not fire. Check: Is it executable? Is it wired in settings.json? Is the matcher correct? Is the JSON valid?

  2. Hook fires but does not block. Check: Are you returning exit code 2 (not 1)? Is the regex pattern correct? Is set -e causing an early exit before your blocking logic?

  3. Hook blocks everything. Check: Is there an early exit 2 before your conditional logic? Is the regex too broad?

  4. Hook produces no stderr feedback. Check: Are you writing to stderr (>&2), not stdout?

The Stop Hook as a Quality Gate¶

The Stop hook is underused but powerful. Here is an example that checks whether Claude ran tests before claiming a bug fix is complete:

bash
#!/usr/bin/env bash
set -euo pipefail

INPUT=$(cat)

# Extract the assistant's response
RESPONSE=$(echo "${INPUT}" | python3 -c "
import sys, json
data = json.load(sys.stdin)
print(data.get('assistant_response', ''))
")

# If the response mentions a fix but no test execution evidence
if echo "${RESPONSE}" | grep -qi "fix\|fixed\|resolved\|patched"; then
    if ! echo "${RESPONSE}" | grep -qi "test.*pass\|tests.*pass\|pytest\|npm test\|test suite"; then
        echo "Quality gate: You mentioned fixing a bug but did not show test results. Please run the relevant tests to verify the fix." >&2
        exit 2
    fi
fi

exit 0

When this hook returns exit 2, Claude does not stop – it continues working, using the stderr feedback as guidance. It will run the tests and include the results before trying to finish again.

Environment Variable Hooks¶

Sometimes a hook’s behavior should vary by environment. Use environment variables:

bash
#!/usr/bin/env bash
set -euo pipefail

# Allow overriding strict mode for CI environments
if [[ "${CLAUDE_HOOKS_STRICT:-true}" == "false" ]]; then
    exit 0
fi

# ... normal hook logic

In development: hooks run normally. In CI: export CLAUDE_HOOKS_STRICT=false to relax them.

What Most People Get Wrong¶

1. Using exit code 1 to block. Exit code 1 means “the hook crashed.” Exit code 2 means “block this action.” Every week, someone files a bug report saying “my hook doesn’t block anything” because they wrote exit 1 instead of exit 2. The contract is explicit: 2 blocks, everything else does not.

2. Forgetting chmod +x. The script exists. The wiring is correct. The matcher is right. But the hook never fires. Claude Code cannot execute a script without execute permissions. It treats the permission error as a non-blocking failure and continues silently. Add chmod +x to your hook setup checklist and to your workspace validation script.

3. Overly broad matchers. Wiring a heavy hook to fire on every tool call – Read, Grep, Glob, Edit, Write, Bash – adds latency to every single step of the agentic loop. A ten-turn task with 30 tool calls and a 500ms hook means 15 seconds of pure hook overhead. Use specific matchers.

4. Hooks that are too clever. A hook that parses SQL queries, evaluates their safety against the database schema, and checks row counts before allowing execution is impressive engineering and terrible in practice. It will be fragile, slow, and impossible for your teammates to debug. Start with simple regex patterns. Add complexity only when the simple approach demonstrably fails.

5. Not testing hooks in isolation. Hooks run inside Claude Code’s process pipeline. If your hook has a bug, the debugging cycle is: start Claude Code session, trigger the hook, observe the behavior, read the logs, edit the script, restart the session. That is a five-minute cycle. Testing with echo '...' | ./hook.sh is a five-second cycle. Always test outside Claude Code first.

6. Ignoring timeout. A hook that makes an HTTP call to a slow server with no timeout will hang the entire agentic loop for the default 60 seconds. Set explicit timeouts in both the settings.json configuration and within your script logic (using timeout command or similar).

7. Writing blocking hooks for non-critical checks. Reserve exit code 2 for genuinely dangerous operations – things that would lose data, leak secrets, or break production. Style violations, missing documentation, and naming convention issues should be reminders (exit 0 with stderr output), not blocks (exit 2). If you block too aggressively, developers will disable your hooks entirely.

Chapter Summary¶

Five things to remember from this chapter:

  1. Hooks are lifecycle interceptors. They fire at specific points in Claude Code’s execution loop – PreToolUse (before, can block), PostToolUse (after, observe only), SessionStart (once, setup), Notification (alerts), and Stop (quality gate). The execution order is: permission check, then PreToolUse hooks sequentially, then tool execution, then PostToolUse hooks.

  2. Exit code 2 is the blocking signal. Not exit 1, not exit 255 – exit code 2. Anything written to stderr when exiting with code 2 becomes feedback that Claude uses to reformulate its approach. Write clear, actionable messages.

  3. Hooks receive JSON on stdin. Every hook gets tool_name, tool_input, session_id, and event-specific fields. Parse with python3 -c for zero-dependency reliability or jq for cleanliness. Test by piping JSON manually.

  4. Four types serve different needs. command (shell scripts) handles 90% of cases. http (webhooks) enables centralized team policies. prompt (LLM evaluation) catches nuanced dangers that regex misses. agent (subagent) provides deep, multi-step evaluation.

  5. Start simple, stay simple. A twelve-line script that catches rm -rf / prevents more disasters than a thousand-line policy engine that no one understands. Build hooks that your least experienced team member can read, debug, and modify. Complexity is the enemy of guardrails that actually get used.

As discussed in Chapter 3, hooks fire after the permission system has already evaluated the tool call. They are a second layer of defense, not a replacement for proper permission configuration. In Chapter 5, we covered how settings.json permissions provide the first gate. Hooks provide the second. Together, they form a defense-in-depth strategy for autonomous AI tool use.
