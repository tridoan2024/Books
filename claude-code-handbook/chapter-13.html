Part IV: Context Engineering

Chapter 13: Token Economics and Context Windows¶ ~17 min read

You open Claude Code on a Monday morning, feed it a 1,500-line Python module, ask it to refactor the authentication layer, and thirty minutes later it responds with something nonsensical about a database migration you discussed on Friday. What happened? You ran out of context. The conversation history from your previous task bled into the new one, Claude’s understanding of your codebase degraded, and you burned $12 in API costs getting confused output.

This chapter is about understanding the single most important constraint in working with Claude Code: the context window. Every professional who uses this tool daily needs to internalize token economics the way a systems engineer internalizes memory allocation. Context is a finite resource. Managing it well is the difference between a $5 productive session and a $50 frustrating one.

What the Context Window Actually Is¶

The context window is the total amount of text Claude can “see” at one time. Think of it as working memory. Everything Claude needs to understand your request – your system prompt, the conversation history, files it has read, tool results, its own previous responses – must fit inside this window. Once the window fills up, older content gets compressed or dropped.

Every model in the Claude family has a defined context window size:

Model Context Window Approximate Words Approximate Pages
Claude Opus 4 200K tokens ~150,000 words ~375 pages
Claude Sonnet 4 200K tokens ~150,000 words ~375 pages
Claude Haiku 3.5 200K tokens ~150,000 words ~375 pages

That sounds enormous – 375 pages of text. But in practice, you never get all 200K tokens for your actual work. The system prompt, tool definitions, and Claude’s own response all compete for the same space. Understanding where those tokens go is the key to working effectively.

What a Token Is¶

A token is the fundamental unit Claude uses to process text. It is not a word, not a character, and not a line of code. It is a chunk of text determined by the model’s tokenizer. Here are the practical conversion factors:

  • 1 token is roughly 4 characters of English text
  • 1 token is roughly 0.75 words
  • 1 word is roughly 1.3 tokens
  • 100 tokens is roughly 75 words

For code, tokens are less predictable. Variable names, punctuation, indentation, and language syntax all affect token count differently. Some practical benchmarks:

Content Approximate Tokens
A typical source file (500 lines) 2,000 - 4,000 tokens
A large source file (2,000 lines) 8,000 - 16,000 tokens
The Claude Code system prompt 10,000 - 30,000 tokens
A 200-word user message ~270 tokens
A detailed tool result (file listing, grep output) 500 - 5,000 tokens
A full test suite output (100 tests) 3,000 - 8,000 tokens

Here is what 200K tokens means in practical terms:

200,000 tokens = ~150,000 words = ~375 pages of text
                = ~50 typical source files read end-to-end
                = ~25 large source files
                = ~4 hours of dense conversation

This seems like a lot, but it gets consumed faster than you expect. A single Read of a large file followed by a Bash command that dumps 500 lines of test output can consume 20,000 tokens in two tool calls. That is 10% of your total window, gone in seconds.

Token Budgeting: Where the 200K Actually Goes¶

Here is the critical insight most people miss: you do not get 200K tokens for your messages. The context window is shared across everything Claude needs to function. Here is how Claude Code allocates the window:

┌─────────────────────────────────────────────────┐
│              200K Token Context Window            │
├─────────────────────────────────────────────────┤
│  System Prompt        ~25%    (~50K tokens)      │
│  ├── CLAUDE.md + @imported rules                 │
│  ├── Git context (recent commits, branch info)   │
│  ├── Active skill instructions                   │
│  └── Memory files (MEMORY.md + loaded memories)  │
├─────────────────────────────────────────────────┤
│  Tool Definitions     ~15%    (~30K tokens)      │
│  ├── JSON schema for every available tool        │
│  ├── MCP server tool definitions                 │
│  └── Skill-provided tool instructions            │
├─────────────────────────────────────────────────┤
│  Conversation History ~50%    (~100K tokens)     │
│  ├── Your messages                               │
│  ├── Claude's responses                          │
│  ├── Tool calls and results                      │
│  └── File contents that were read                │
├─────────────────────────────────────────────────┤
│  Response Reserve     ~10%    (~20K tokens)      │
│  └── Space for Claude's next response            │
└─────────────────────────────────────────────────┘

That means your actual working space for conversation and tool results is roughly 100K tokens – half the advertised window. And that 100K fills up fast.

The System Prompt Tax¶

Every Claude Code session starts by loading the system prompt. This includes:

  1. CLAUDE.md and all @imported rule files
  2. Tool definitions – every tool available to Claude gets a JSON schema describing its parameters, constraints, and usage
  3. Git context – current branch, recent commits, repository state
  4. Memory files – auto-memory index and any relevant memory files
  5. Active skill instructions – if a skill is loaded, its SKILL.md adds to the prompt

A lean CLAUDE.md with five rule files might cost 5,000 tokens. But add ten MCP servers with 50 custom tools, and your tool definitions alone could consume 30,000 tokens. Add a detailed SKILL.md for the currently active skill, and your system prompt can easily hit 50,000 tokens before you type a single message.

This is why the guidance from Chapter 4 about keeping CLAUDE.md concise matters for performance, not just readability.

What Consumes the Most Tokens¶

In order from most to least expensive:

1. Large file reads. A single Read of a 2,000-line file consumes 8,000-16,000 tokens. Read five files to understand a feature, and you have spent 40,000-80,000 tokens – nearly half your conversation budget.

# This single command can consume 15K tokens:
Read /src/services/authentication.py    # 2,100 lines → ~10K tokens

# Better: read only what you need
Read /src/services/authentication.py offset=450 limit=50  # ~200 tokens

2. Bash command output. Test suites, build logs, and directory listings produce enormous output. A failing test suite with verbose output can generate 10,000+ tokens.

bash
# This can produce 5,000+ tokens of output:
python -m pytest tests/ -v

# Better: run the specific test you care about:
python -m pytest tests/test_auth.py::test_login_invalid_password -v

3. Accumulated tool results. Every tool call adds its result to the conversation history. In a 30-turn session, even modest tool results accumulate to tens of thousands of tokens.

4. System prompt bloat. Too many MCP servers, oversized CLAUDE.md, or verbose skill files inflate the baseline cost of every single turn.

5. Previous messages in long sessions. Your messages and Claude’s responses accumulate. A detailed explanation Claude gave on turn 5 still occupies tokens on turn 50.

How Tokens Work in Code vs. Prose¶

The 4-characters-per-token rule of thumb holds for English prose but breaks down in code. Understanding why helps you estimate context consumption more accurately.

Whitespace and indentation are tokenized. A deeply nested Python function with four levels of indentation burns tokens on spaces that carry no semantic meaning. Here is a concrete example:

python
# This function uses ~45 tokens:
def greet(name: str) -> str:
    return f"Hello, {name}!"

# This equivalent function uses ~120 tokens (indentation + boilerplate):
class GreetingService:
    def __init__(self, prefix: str = "Hello") -> None:
        self.prefix = prefix

    def greet(self, name: str) -> str:
        if name is None:
            raise ValueError("Name cannot be None")
        return f"{self.prefix}, {name}!"

Language affects token density. Python is relatively token-efficient because of its minimal syntax. Languages with verbose syntax like Java or XML consume more tokens per line of functionality:

Language Tokens per 100 Lines (Typical)
Python 400 - 600
TypeScript 500 - 800
Java 600 - 1,000
Go 500 - 700
HTML/XML 300 - 500
JSON (data) 200 - 400
YAML 250 - 450

Comments are expensive. A 20-line docstring explaining a function costs the same tokens as 20 lines of code. When reading files into context, heavily-commented code consumes significantly more tokens than sparse code. This is another reason to use offset and limit when reading – skip the module-level docstring and jump straight to the function you need.

Import blocks are cheap but repetitive. A standard Python import section of 15 lines costs only ~60-100 tokens, but if you read 10 files that all have similar imports, that is 600-1,000 tokens of near-identical content adding no new information.

Estimating Token Cost Before Acting¶

Before asking Claude to read a large file or run a verbose command, develop the habit of estimating the token cost:

Lines of code / 4 ≈ tokens (rough estimate for Python/TypeScript)
Lines of output / 3 ≈ tokens (rough estimate for command output)

If a file is 2,000 lines: ~500 tokens minimum, likely 2,000-4,000 in practice. If a test suite produces 800 lines of output: ~250-2,500 tokens depending on verbosity. These rough estimates help you decide whether to read the whole file or just a section.

Token Pricing¶

Every token you send to or receive from Claude costs money. The pricing differs significantly across models:

Model Input ($/MTok) Output ($/MTok) Cache Read ($/MTok)
Claude Opus 4 $15.00 $75.00 $1.50
Claude Sonnet 4 $3.00 $15.00 $0.30
Claude Haiku 3.5 $0.80 $4.00 $0.08

MTok = million tokens

A few things jump out immediately:

  • Output tokens cost 5x input tokens. Claude’s responses are five times more expensive than what you send. Long, verbose responses are the biggest cost driver.
  • Opus costs 5x Sonnet, which costs 3.75x Haiku. Model selection has a massive impact on your bill.
  • Cache reads are 10x cheaper than fresh input. This is where prompt caching (next section) becomes critical.

Let us put real numbers on common operations:

Reading a 10K-token file on Opus:
  Input cost: 10,000 / 1,000,000 * $15 = $0.15

Claude writing a 2K-token response on Opus:
  Output cost: 2,000 / 1,000,000 * $75 = $0.15

A single turn (10K input + 2K output) on Opus: $0.30

Same turn on Sonnet: $0.06
Same turn on Haiku: $0.016

Over a 50-turn session, these costs compound. Model selection is the single biggest lever you have for cost control.

Prompt Caching¶

Prompt caching is Claude Code’s most important cost optimization feature, and it works automatically. Here is how it works:

The system prompt – CLAUDE.md, rules, tool definitions, memory – is the same on every turn of a conversation. Without caching, Claude would re-process all 50,000 tokens of system prompt on every single turn. With caching, Claude processes it once and reads it from cache on subsequent turns.

Turn 1 (cold cache):
  System prompt (50K tokens): 50,000 / 1,000,000 * $15 = $0.75 (Opus)

Turn 2+ (cached):
  System prompt (50K tokens): 50,000 / 1,000,000 * $1.50 = $0.075 (Opus)

Savings: 90% on system prompt costs after first turn

The cache has a TTL (time-to-live) of 5 minutes. If you pause for more than 5 minutes between turns, the cache expires and the next turn pays full price again. This means:

  • Rapid back-and-forth is cheaper per turn than slow, deliberate prompting
  • Stepping away for lunch resets the cache – your next turn is expensive again
  • The first turn of every session always pays full price

In practice, prompt caching reduces the average cost of a multi-turn session by 40-60% compared to what it would cost without caching.

Extended Thinking Tokens¶

When Claude reasons about a complex problem, it uses extended thinking – an internal chain-of-thought that happens before the visible response. These thinking tokens are not shown to you by default, but they still cost money.

Extended thinking is controlled by the /effort command:

Effort Level Thinking Tokens When to Use
/effort low ~1,000 Simple edits, formatting, quick lookups
/effort medium ~5,000 Standard coding tasks, reviews
/effort high ~10,000+ Complex debugging, architecture, security audits
ultrathink Maximum One-off deep reasoning for a single prompt

The ultrathink keyword is special: add it to any prompt to get maximum reasoning depth for that one turn only, without changing the persistent effort setting.

# Standard prompt
Fix the race condition in worker.py

# Maximum reasoning for this one turn
ultrathink Fix the race condition in worker.py

Extended thinking tokens are billed at input rates, not output rates. On Opus, 10K thinking tokens cost $0.15, which is modest. But on long sessions with high effort, thinking tokens can add up to 20-30% of your total cost.

Cost optimization tip: Use /effort low for simple tasks like formatting, renaming, or small edits. Reserve /effort high for debugging, security analysis, and architectural decisions. The quality difference is real – low effort produces shallower reasoning – so match effort to task complexity.

Context Window Management Strategies¶

These are the habits that separate professionals from amateurs. Each one directly affects both the quality of Claude’s output and your costs.

1. /clear Between Unrelated Tasks¶

This is the single most impactful habit you can develop. When you finish one task and start another, run /clear. It resets the conversation history, freeing up the entire 100K conversation budget.

# BAD: Everything in one session
You: Fix the login bug in auth.py
Claude: [fixes bug, reads 5 files, runs tests]
You: Now write me a blog post about our new feature
Claude: [confused, tries to relate blog post to auth bug]

# GOOD: Clean context between tasks
You: Fix the login bug in auth.py
Claude: [fixes bug, reads 5 files, runs tests]
You: /clear
You: Write me a blog post about our new feature
Claude: [focused, clean context, better output]

Without /clear, the context from your auth bug investigation – the five files Claude read, the test output, the discussion about error handling – is still taking up space and potentially confusing Claude’s response to your completely unrelated blog post request.

2. /compact for Long Sessions¶

When you are deep in a single task and running low on context, /compact summarizes older messages while preserving the focus area:

/compact authentication    # Summarize everything, keep auth-related details

After compaction, old messages are replaced with a summary. You lose granular detail from earlier in the session, but you free up tokens for continued work. Think of it as moving data from RAM to disk – it is still there conceptually, but access is lossy.

3. Use the Agent Tool for Investigation¶

When Claude needs to investigate something – searching through many files, reading documentation, exploring architecture – delegate to a subagent. The subagent runs in its own context, does its investigation, and returns only a summary to your main session.

# BAD: Investigation fills your context
You: Figure out how the payment system works
Claude: [reads 20 files, context now 80% full]
You: Now refactor the payment module
Claude: [barely enough context left for the actual work]

# GOOD: Delegate investigation
You: Use a subagent to investigate how the payment system works,
     then summarize the architecture for me
Claude: [spawns agent, agent reads 20 files, returns 500-token summary]
You: Now refactor the payment module
Claude: [has the summary AND plenty of context for the refactoring]

As discussed in Chapter 12, the Agent tool is the primary mechanism for keeping your main context clean during research-heavy tasks.

4. Read Specific Line Ranges¶

Never read an entire file when you only need a function:

# Costly: 2000 lines, ~8K tokens
Read /src/services/payment.py

# Efficient: 30 lines, ~150 tokens
Read /src/services/payment.py offset=142 limit=30

Use Grep first to find the relevant lines, then Read with an offset:

# Find the function
Grep: pattern="def process_payment" path="/src/services/payment.py"
# Result: line 142

# Read just that function
Read /src/services/payment.py offset=140 limit=40

5. Limit Search Results¶

The Grep tool has a head_limit parameter that defaults to 250 results. For exploratory searches, you rarely need more than 10-20 results:

# Default: up to 250 results, could be thousands of lines
Grep: pattern="TODO" path="/src"

# Better: just show me the first 10
Grep: pattern="TODO" path="/src" head_limit=10

6. Suppress Unnecessary Bash Output¶

Build logs, dependency installation output, and verbose test runs generate enormous amounts of tokens that rarely contain useful information:

bash
# Bad: dumps entire npm install output into context
npm install

# Better: suppress output, only show errors
npm install 2>&1 | tail -5

# Bad: full verbose test output
python -m pytest tests/ -v --tb=long

# Better: summary only
python -m pytest tests/ -q

What Happens When You Hit the Limit¶

When the conversation approaches the 200K token limit, Claude Code takes action automatically:

  1. Auto-compaction triggers. Older messages are summarized to free space. This happens transparently – you will not see a notification, but you might notice Claude losing awareness of earlier discussion details.

  2. Response quality degrades. As context fills, Claude’s attention to older content becomes less reliable. Instructions from early in the session may be followed less accurately.

  3. Eventually, the session stalls. If compaction cannot free enough space, Claude may be unable to generate a useful response. At this point, you need to /clear or start a fresh session.

The practical takeaway: Sessions are not meant to last forever. For most work, a focused 20-40 turn session is optimal. If you find yourself at 100+ turns, you are almost certainly carrying dead context from earlier tasks.

Cost Optimization Patterns¶

Here are six patterns ordered from highest to lowest impact:

1. Start with Sonnet, Escalate to Opus When Needed¶

Sonnet is 5x cheaper than Opus and handles most coding tasks well. Use Opus for tasks that require deep reasoning: complex debugging, security audits, architectural decisions, nuanced code review.

# Daily workflow
/model sonnet          # Default for most work

# Escalate for hard problems
/model opus
ultrathink Debug why the distributed lock fails under contention

# Drop back after
/model sonnet

2. Match Effort to Task Complexity¶

/effort low            # Formatting, renaming, simple edits
/effort medium         # Standard feature work, test writing
/effort high           # Debugging, security, architecture

The difference between low and high effort can be 5-10x in thinking tokens per turn. Over a 50-turn session, that adds up.

3. /clear Aggressively¶

Every /clear saves you money on subsequent turns. The cached system prompt reprices at cache-read rates (10x cheaper), and you stop paying to re-read irrelevant conversation history.

4. Delegate to Agents with Appropriate Models¶

Use the Agent tool with model-appropriate delegation:

# Expensive: Opus agent for simple research
Agent: [model=opus] "What version of React is in package.json?"

# Cheap: Haiku or Sonnet agent for simple lookups
Agent: [model=haiku] "What version of React is in package.json?"

5. Read Files Strategically¶

Before reading a file, ask: do I need the whole thing, or just a section? Use Grep to find relevant lines, then Read with offset and limit.

6. Suppress Verbose Output¶

Pipe build and test output through tail, head, or grep to keep only the relevant lines. A full build log can consume 10,000+ tokens that add zero value to the conversation.

Here is what different work patterns actually cost with Opus 4 as the primary model:

A Worked Example: Bug Fix Session¶

Let us trace the token flow through a real bug fix to make these costs concrete. You are using Opus 4.

Turn 1: You type "Fix the race condition in session.py:89"
  System prompt (cached): 45K tokens × $1.50/MTok = $0.067
  Your message: 20 tokens × $15/MTok = $0.0003
  Claude reads session.py (800 lines): ~3,200 tokens input
  Claude's response (analysis + plan): ~500 tokens output × $75/MTok = $0.037
  Turn cost: ~$0.15

Turn 2: Claude reads two related files
  History replay (cached portions): ~5K tokens
  File reads: ~6,000 tokens input
  Response (identifies root cause): ~800 tokens output = $0.06
  Turn cost: ~$0.20

Turn 3: Claude writes a failing test
  History replay: ~12K tokens
  Writes test file: ~400 tokens output = $0.03
  Turn cost: ~$0.22

Turn 4: Claude implements the fix
  History replay: ~15K tokens
  Edits session.py: ~300 tokens output = $0.022
  Turn cost: ~$0.25

Turn 5: Claude runs pytest
  History replay: ~17K tokens
  Bash output (test results): ~500 tokens
  Response (confirms fix): ~200 tokens output = $0.015
  Turn cost: ~$0.28

Total session: ~$1.10 across 5 turns

The key observation: input costs dominate because the conversation history replays on every turn. By turn 5, Claude is re-reading 17K tokens of history. If this were an unfocused session with 50K tokens of irrelevant prior context, that same 5-turn bug fix would cost $3-4 instead of $1.10.

Cost reduction with Sonnet: Divide the Opus estimates by roughly 4-5x. A full workday on Sonnet typically costs $10-$35.

Cost reduction with effort management: Using /effort low for simple tasks and /effort high only when needed typically saves 20-30% compared to running high effort all day.

The Context Engineering Mindset¶

The term context engineering describes the discipline of deliberately managing what information Claude has access to at any given moment. It is not just about saving money – it is about getting better output.

Consider an analogy: if you walked into a meeting and dumped 375 pages of documents on the table, then asked a question, you would get a worse answer than if you brought just the three relevant pages. Claude works the same way. More context is not always better context. Relevant context is better context.

The context engineering mindset has three principles:

1. Load what you need, not what you have. Before reading a file, ask: does Claude need this to do the task? If the task is “fix the login bug,” Claude probably needs auth.py and test_auth.py, not the entire src/ directory.

2. Clear what you do not need. After finishing a task, /clear. After investigation, /compact. Do not let dead context accumulate and dilute the signal.

3. Budget before you build. For complex tasks, estimate the context cost before starting. If you need to read 30 files to understand a system, delegate that to a subagent. If you need to run a full test suite, capture only the failures.

This mindset connects directly to the CLAUDE.md design principles from Chapter 4 (keep it lean), the skill design from Chapter 7 (focused instructions), and the agent patterns from Chapter 12 (delegation for context isolation). Context engineering is not a single technique – it is the thread that ties professional Claude Code usage together.

What Most People Get Wrong¶

“I’ll just read everything so Claude has full context.” Reading 20 files to give Claude “the whole picture” fills 60% of your context window with content Claude may never reference. It also costs $3-5 in input tokens on Opus. Read surgically.

“I’ve been in this session for 3 hours, it should remember everything.” After auto-compaction kicks in, Claude’s recall of early-session content is lossy at best. If you need details from earlier, re-read the relevant file or re-state the key facts.

“Opus is better, so I should always use Opus.” Opus is better at reasoning, not at everything. For straightforward code generation, test writing, formatting, and simple edits, Sonnet produces equivalent output at 20% of the cost.

“I don’t need to worry about tokens, my company pays for it.” Even if cost is not a concern, context window saturation is. A full context window produces worse output regardless of who is paying.

Chapter Summary¶

  1. The 200K context window is shared across system prompt (~25%), tool definitions (~15%), conversation history (~50%), and response reserve (~10%). Your working budget is roughly 100K tokens.

  2. Token costs are asymmetric. Output costs 5x input. Opus costs 5x Sonnet. Cache reads cost 10x less than fresh input. These ratios define your optimization strategy.

  3. The biggest token consumers are large file reads, verbose bash output, and accumulated tool results. Manage these three and you control your costs.

  4. /clear is your most important habit. Use it between every unrelated task. It frees context, reduces costs, and improves output quality.

  5. Context engineering is the meta-skill of professional Claude Code usage. Load what you need, clear what you do not, and budget before you build.
