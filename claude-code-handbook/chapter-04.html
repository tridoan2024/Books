Part II: The Professional Configuration Stack

Chapter 4: CLAUDE.md — Engineering Your Project Brain¶ ~22 min read

You just installed Claude Code on a new project. You run claude in the terminal. Claude reads your code, suggests a fix, and gets it wrong — it used os.path instead of pathlib, committed with a merge commit instead of a rebase, and wrote a test with unittest instead of pytest. None of these are Claude’s fault. You never told it how you work.

That is what CLAUDE.md fixes. It is a plain markdown file at the root of your project that Claude Code reads at the start of every single session. Think of it as the onboarding document you would write for a new hire who is brilliant but knows nothing about your team’s conventions.

This chapter covers how to write one that actually works — not a wish list of aspirational rules, but a precisely engineered document that changes Claude Code’s behavior on every prompt.

The Single Most Important File¶

In Chapter 3, we walked through how Claude Code assembles its system prompt. Step 2 of that assembly is loading CLAUDE.md content. Every instruction you put in this file becomes part of the system prompt that shapes every response, every tool call, every code generation for the entire session.

This makes CLAUDE.md the highest-leverage configuration file in your entire Claude Code setup. One line added here changes behavior across hundreds of interactions. One line removed creates hundreds of micro-mistakes that compound into hours of cleanup.

Here is the mental model: Claude Code is a senior engineer who just joined your team. Without CLAUDE.md, this engineer writes perfectly valid code that follows no team conventions. With a well-crafted CLAUDE.md, they write code that looks like it came from someone who has been on the team for a year.

The difference between “valid code” and “team code” is entirely in this file.

Where It Lives¶

The most common placement is at the root of your project:

my-project/
├── CLAUDE.md          ← Claude reads this every session
├── .claude/
│   ├── settings.json
│   └── rules/
├── src/
├── tests/
└── package.json

When Claude Code starts a session, it looks for CLAUDE.md in the current working directory and its parents. If it finds one, the content is injected into the system prompt near the top — after Claude Code’s own base instructions but before tool definitions and conversation history. This placement matters: instructions that appear earlier in the system prompt carry more weight during generation.

The CLAUDE.md Hierarchy¶

Claude Code does not look for CLAUDE.md in just one place. It searches a hierarchy of locations and merges them all into the system prompt. Understanding this hierarchy is essential because it determines which instructions apply and how conflicts resolve.

The Four Levels¶

~/.claude/CLAUDE.md                          ← Global (personal preferences)
~/projects/CLAUDE.md                         ← Workspace (shared team defaults)
~/projects/my-app/CLAUDE.md                  ← Project (project-specific rules)
~/projects/my-app/src/CLAUDE.md              ← Local (directory-specific overrides)

Level 1: Global (~/.claude/CLAUDE.md)

This file lives in your home directory and applies to every project you work on. Put your personal preferences here — the things that are true regardless of what project you are in.

markdown
# Global Preferences

## Communication
- Be terse. Lead with the answer, not the reasoning
- No emojis unless requested
- Reference code as `file_path:line_number`

## Personal Standards
- Always use type hints in Python
- Prefer const over let in TypeScript
- Run tests before committing

This is your personal style. It follows you from project to project. It is not committed to any repository — it lives in your home directory and is yours alone.

Level 2: Workspace (~/projects/CLAUDE.md)

If you keep multiple related projects in a parent directory — a monorepo workspace, a company directory, a client folder — you can place a CLAUDE.md at that level. It applies to every project within the workspace.

markdown
# Acme Corp Workspace

## Team Standards
- All commits must reference a Jira ticket: "ACME-123: description"
- Use conventional commits: feat:, fix:, docs:, refactor:
- All PRs require at least one reviewer
- Never commit directly to main — always use feature branches

## Infrastructure
- CI/CD: GitHub Actions
- Secrets: AWS Secrets Manager (never .env files in production)
- Deployment: Kubernetes via ArgoCD

This level is useful when your organization has standards that span multiple repositories. A consulting firm might have ~/clients/acme/CLAUDE.md with Acme’s conventions, while ~/clients/beta/CLAUDE.md has Beta Corp’s different standards.

Level 3: Project (~/projects/my-app/CLAUDE.md)

This is the most common and most important level. It sits at the root of a specific repository and contains everything Claude Code needs to know about that project. Most teams will have only this level plus maybe a global level.

markdown
# Project: Inventory API

FastAPI service managing warehouse inventory. PostgreSQL + SQLAlchemy.
Deployed to AWS ECS behind an ALB. Used by 3 internal services.

## Coding Standards
- Python 3.12+. Type hints on all signatures
- Pydantic v2 for request/response models
- SQLAlchemy 2.0 style (select() not query())
- Format: ruff format. Lint: ruff check --fix

## Git Workflow
- Branch: feat/, fix/, docs/, refactor/
- Commits: imperative mood, reference ticket (INV-xxx)
- Run pytest -x --tb=short before committing

## Verification
- Write a failing test first when fixing bugs
- Full suite before PR: pytest --cov=src

This file is committed to git. Every team member — and every Claude Code session — gets the same instructions.

Level 4: Local (~/projects/my-app/src/api/CLAUDE.md)

You can place CLAUDE.md files in subdirectories to provide directory-specific context. This is powerful for large projects where different directories have different conventions.

markdown
# src/api/ Context

This directory contains the REST API layer.
- Every endpoint must have an OpenAPI docstring
- Use dependency injection for database sessions
- Rate limiting is handled by middleware — do not add per-endpoint rate limits
- All responses use the StandardResponse envelope: {data, error, meta}

Local CLAUDE.md files are especially useful in monorepos where one directory is Python, another is TypeScript, and a third is Go. Each directory can have its own context document.

How Merging Works¶

All levels are included in the system prompt. They do not replace each other — they concatenate. Claude Code sees instructions from every level that applies to the current working context.

When instructions conflict, there is no formal “override” mechanism with explicit precedence rules. Instead, Claude Code follows the general principle that more specific instructions take priority. A project-level CLAUDE.md saying “use tabs” will override a global CLAUDE.md saying “use spaces” because the project-level instruction is more contextually relevant.

In practice, you can reinforce priority by using the IMPORTANT: prefix:

markdown
# Project CLAUDE.md
- IMPORTANT: Use tabs for indentation in this project (overrides personal preference)
Level Scope Typical Content Shared via Git?
Global All projects Personal preferences, communication style No
Workspace All projects in a directory Organization standards, CI/CD conventions Sometimes
Project One repository Tech stack, coding standards, architecture Yes
Local One directory Module-specific context, API conventions Yes

Practical recommendation: Most professionals need only two levels — global (personal preferences) and project (project-specific rules). Add workspace and local levels only when you have a concrete reason. Over-engineering the hierarchy creates maintenance burden for little benefit.

Structure Best Practices¶

Anthropic’s official guidance is to keep CLAUDE.md under 200 lines. In practice, the best ones I have seen are between 30 and 80 lines, with @imports handling everything else. Here is why, and how to structure it.

Why Brevity Matters¶

As discussed in Chapter 3, CLAUDE.md content is injected into the system prompt on every API call. A 200-line file consumes roughly 600-800 tokens per call. Over a 50-turn session, that is 30,000-40,000 tokens spent just on your instructions — before Claude Code even reads your code or generates a response.

But the cost is not just financial. There is a cognitive effect: the more rules you include, the less reliably any individual rule gets followed. A 40-line CLAUDE.md where every line is critical will be followed more consistently than a 400-line document where important rules are buried between nice-to-haves. Language models have attention patterns, and dense, focused instructions receive more attention than sprawling ones.

The Golden Structure¶

Every effective CLAUDE.md I have seen follows this general pattern:

markdown
# Project: [Name]

[One-line description of what this project is and does]

## Communication
[How Claude should talk to you — 3-5 lines max]

## Coding Standards
[Language-specific rules — only what Claude would get WRONG without guidance]

## Git Workflow
[Branch naming, commit format, pre-commit requirements]

## Verification
[What to run before considering work done]

## Key Integrations
[CI/CD, MCP servers, external services Claude needs to know about]

## Domain Rules
[@imports to detailed rule files]

## Architecture
[@import to architecture doc]

This is not a template to follow blindly — it is a pattern that works because it mirrors how information is consumed. Communication rules come first because they affect every response. Coding standards come next because they affect every code generation. Verification comes near the end because it applies only when work is being finalized.

Most Important Rules First¶

Claude Code reads CLAUDE.md top-down. Instructions at the top of the file have more influence than instructions at the bottom. This is not a quirk — it is how language models process context. Earlier tokens in the system prompt carry more weight during generation.

Put your most critical rules first. If “never commit secrets” is your top priority, it belongs in the first section, not buried after formatting preferences.

markdown
# Project: Banking API

## Critical Rules
- NEVER commit secrets, tokens, API keys, or passwords
- NEVER use string concatenation in SQL queries
- ALL external input must be validated at system boundaries

## Communication
- Be terse. Lead with the answer
...

For projects with security sensitivity — financial applications, healthcare systems, anything handling PII — the first section should always be security constraints.

The “IMPORTANT:” Prefix¶

When you have a rule that must not be ignored under any circumstances, prefix it with IMPORTANT:. This is not a convention I invented — the model recognizes this prefix as a signal to prioritize that instruction above surrounding text.

markdown
## Coding Standards
- Python: type hints, f-strings, pathlib, PEP 8
- IMPORTANT: Never use bare except: clauses. Always catch specific exceptions
- IMPORTANT: Never commit .env files or any file containing secrets

Use this sparingly. If everything is IMPORTANT:, the signal degrades and nothing gets prioritized. Reserve it for rules where violation would cause real damage — security constraints, data integrity rules, things that break production.

The @import System¶

The @import system is what makes a lean CLAUDE.md possible. Instead of cramming every coding standard, security rule, and testing convention into one file, you reference external files that Claude Code loads as needed.

Basic Syntax¶

markdown
## Domain Rules
@.claude/rules/python.md
@.claude/rules/security.md
@.claude/rules/testing.md

## Architecture
@docs/architecture.md

Each @ reference tells Claude Code to load that file’s content into the system prompt alongside CLAUDE.md. The syntax is simple: the @ symbol followed by a file path, on its own line.

How Imports Are Resolved¶

Paths are resolved relative to the CLAUDE.md file that contains them. If your CLAUDE.md is at ~/projects/my-app/CLAUDE.md, then:

  • @.claude/rules/python.md resolves to ~/projects/my-app/.claude/rules/python.md
  • @docs/architecture.md resolves to ~/projects/my-app/docs/architecture.md

You cannot use absolute paths or paths that escape the project directory. This is a security measure — CLAUDE.md files committed to git should not reference files outside the repository. An import like @/etc/passwd or @../../other-project/secrets.md will not work.

If an imported file does not exist, it is silently skipped. This means you can reference optional files without breaking the session.

Always-On vs. Path-Scoped Imports¶

Here is where imports become truly powerful. Not all imported files need to load on every session. Rule files can declare paths: frontmatter that makes them load only when you are working with matching files.

An import referenced in CLAUDE.md:

markdown
@.claude/rules/security.md    ← Always loaded (no paths: frontmatter in the file)
@.claude/rules/python.md      ← Loaded only when working with .py files

The scoping is controlled by the imported file’s frontmatter, not by the @ reference itself. Here is how python.md declares its scope:

yaml
---
paths:
  - "**/*.py"
  - "**/*.pyi"
---

# Python Rules
- Use type hints on all function signatures
- Use f-strings over .format() or %
- Use pathlib over os.path
- Use dataclasses or Pydantic for data structures, not raw dicts

When you open or edit a .py file, Claude Code sees the path match and loads python.md into context. When you are working exclusively on .ts files, python.md does not load. This keeps the system prompt lean — you only pay the token cost for rules that are relevant to what you are actually doing.

A security rules file without paths: frontmatter loads every session regardless of what files are in scope:

markdown
# Security Rules
- NEVER hardcode secrets, tokens, API keys, or passwords
- Validate and sanitize ALL external input
- Use parameterized queries — never string concatenation for SQL

No frontmatter means always-on. This is correct for security rules — they apply regardless of language or directory.

Import Strategy¶

Import Type Use When Example
Always-on Rule applies to every file type Security rules, git conventions
Path-scoped Rule applies to one language or directory Python rules, shell rules, API conventions
Architecture import Claude needs system context for decisions @docs/architecture.md
API docs import Working on API layer frequently @docs/api-reference.md

A typical setup has 2-3 always-on imports (security, testing, git conventions) plus 2-5 path-scoped imports (one per language in the project). This gives you comprehensive coverage while keeping any individual session’s token cost low.

The Token Math¶

Consider a project with five rule files, each averaging 40 lines (~120 tokens):

Strategy Tokens per turn 50-turn session cost
All rules inline in CLAUDE.md 600 tokens 30,000 tokens
All rules as always-on imports 600 tokens 30,000 tokens
2 always-on + 3 path-scoped imports 240-360 tokens 12,000-18,000 tokens

Path-scoped imports cut your token cost by 40-60% compared to loading everything every turn. Over weeks of development, this adds up to millions of tokens saved.

What to Include¶

The right content for CLAUDE.md passes a simple test: would Claude Code get this wrong without being told? If the answer is yes, include it. If the answer is no — if Claude could figure it out by reading your code — leave it out.

Communication Preferences¶

Claude Code defaults to a helpful, verbose style with explanations before answers. Most senior engineers want something different.

markdown
## Communication
- Be terse. Lead with the answer, not the reasoning
- No emojis unless requested
- Reference code as `file_path:line_number`
- No trailing summaries — I read the diff
- Never show failed intermediate tool calls — only final results

This is one of the highest-impact sections. Without it, you get paragraphs of preamble before every answer. With it, you get direct results. Five lines that save you from reading hundreds of unnecessary words per session.

Coding Standards¶

Only include rules that Claude would not infer from reading your existing code. If your entire codebase uses pathlib, Claude will probably use pathlib without being told. But if you have a strong preference that contradicts common patterns or defaults, state it explicitly.

markdown
## Coding Standards
- Python: type hints, f-strings, pathlib, PEP 8. Format with ruff
- TypeScript: strict mode, const, async/await
- Shell: set -euo pipefail, quote all variables
- No speculative abstractions. Three similar lines > premature abstraction
- IMPORTANT: Never commit secrets. Validate at system boundaries only

The “no speculative abstractions” rule is a good example of something that must be stated explicitly. Claude’s training data is full of patterns that extract functions after two repetitions. If your team prefers concrete repetition over premature abstraction, you must say so.

Git Conventions¶

Every team has different conventions. Claude Code does not know yours unless you tell it.

markdown
## Git Workflow
- Branch: feat/, fix/, docs/, refactor/ prefixes
- Commits: imperative mood, focus on WHY not WHAT
- Run tests before committing
- Never force-push or amend without permission

Without this section, Claude Code will use reasonable defaults — but “reasonable” might mean conventional commits when your team uses a different format, or squash commits when your team uses rebase.

Verification Requirements¶

This section transforms Claude Code from “writes code and calls it done” to “writes code, runs tests, checks types, and lints before declaring success.” The difference in output quality is dramatic.

markdown
## Verification
- Always verify your work: run tests, check types, lint
- Prefer running single tests over full suite for speed
- When fixing bugs, write a failing test first

Without explicit verification instructions, Claude Code will sometimes skip testing, especially on small changes. With these instructions, it consistently runs your test suite.

Key Integrations¶

If Claude Code needs to interact with external systems, it needs to know they exist and how to use them.

markdown
## Key Integrations
- CI/CD: GitHub Actions (workflows in .github/workflows/)
- MCP Servers: copilot-mem (persistent memory)
- Issue tracker: GitHub Issues (use gh CLI)
- Package manager: uv (not pip)

The “use gh CLI” instruction is particularly important. Without it, Claude Code might try to use the GitHub API directly or parse web pages. With it, Claude Code uses the fast, authenticated gh tool that is already configured on your machine.

Domain Terminology¶

If your project uses domain-specific terms that have different meanings in common usage, define them. Claude Code will use your definitions in code, comments, and commit messages.

markdown
## Domain Terms
- "tenant" = organization-level account, not individual user
- "workspace" = a tenant's isolated environment
- "artifact" = any build output (container image, binary, package)
- "promotion" = moving an artifact from staging to production

Without this section, Claude Code might confuse “tenant” with “user,” “workspace” with “directory,” or “promotion” with a marketing concept. Domain terminology is surprisingly high-impact for projects with specialized vocabularies — fintech, healthcare, logistics, DevOps.

What NOT to Include¶

Equally important is knowing what to leave out. Every unnecessary line in CLAUDE.md dilutes the impact of the necessary ones and costs tokens on every turn.

Code Patterns Derivable from Reading the Code¶

If your codebase consistently uses a pattern, Claude will pick it up from reading the files. You do not need to state what is already evident.

Bad (unnecessary):

markdown
- Use React functional components with hooks
- Import React at the top of every file
- Use className instead of class

All of these are standard React conventions that Claude knows. And if your codebase uses them, Claude will see them when reading your files.

Include only what contradicts expectations. If your React project uses class components intentionally (legacy codebase, performance reasons), then you need to say “use class components, not functional components” because that contradicts what Claude would do by default.

Git History¶

Do not paste recent changes or commit logs into CLAUDE.md. Claude Code already has access to git log and recent commit messages as part of its system prompt assembly (see Chapter 3).

Bad:

markdown
## Recent Changes
- 2026-03-28: Added user authentication
- 2026-03-29: Fixed login bug
- 2026-03-30: Refactored database layer

This is stale the moment you write it and wastes tokens every session. Claude Code gets fresh git context automatically.

Ephemeral Task Details¶

CLAUDE.md is for permanent project knowledge. Tasks, plans, and to-do lists belong in separate files or in your issue tracker.

Bad:

markdown
## Current Tasks
- [ ] Fix the login timeout bug
- [ ] Add rate limiting to /api/users
- [ ] Update the README

Use GitHub Issues, a dedicated TODO.md file, or Claude Code’s plan mode (/plan) for this. Putting tasks in CLAUDE.md means they consume tokens long after they are completed.

Entire API Documentation¶

If you have a 500-line API reference, do not paste it into CLAUDE.md. The token cost is enormous and most of it is irrelevant to any given task.

Better approaches: - Put it in docs/api-reference.md and @import it with path scoping - Create a skill with a references/ folder that loads only when working on API code - Let Claude Code read the OpenAPI spec file directly when it needs API details

Environment-Specific Configuration¶

Configuration that varies between machines — ports, paths, database URLs, local tool versions — belongs in settings.json or environment variables, not in CLAUDE.md.

Bad:

markdown
## Local Setup
- Database runs on port 5433
- Redis on port 6380
- API server on port 8080

This is different on every developer’s machine and does not belong in a file committed to git. Use .claude/settings.local.json (which is gitignored) for machine-specific configuration.

Real-World Examples¶

Theory is useful. Working examples are better. Here are three CLAUDE.md files for different project types, each demonstrating the principles from this chapter.

Example 1: Python REST API¶

markdown
# Project: Inventory API

FastAPI service managing warehouse inventory. PostgreSQL + SQLAlchemy.
Deployed to AWS ECS behind an ALB. Used by 3 internal services and 1 mobile app.

## Communication
- Be terse. Lead with the answer
- Reference code as `file_path:line_number`
- No emojis. No trailing summaries

## Coding Standards
- Python 3.12+. Type hints on all signatures
- Pydantic v2 for request/response models
- SQLAlchemy 2.0 style (select() not query())
- Use pathlib, f-strings, dataclasses
- Format: ruff format. Lint: ruff check --fix
- IMPORTANT: Never use string interpolation in SQL. Always parameterized queries

## Git Workflow
- Branch: feat/, fix/, docs/, refactor/
- Commits: imperative mood, reference ticket (INV-xxx)
- Run pytest -x --tb=short before committing

## Verification
- Write a failing test first when fixing bugs
- Run single test files during development: pytest tests/test_specific.py
- Full suite before PR: pytest --cov=src

## Key Integrations
- CI: GitHub Actions (.github/workflows/ci.yml)
- Secrets: AWS Secrets Manager (never .env in production)
- Monitoring: Datadog APM
- Package manager: uv

## Domain Rules
@.claude/rules/python.md
@.claude/rules/security.md
@.claude/rules/testing.md

## Architecture
@docs/architecture.md

47 lines. Notice the one-line project description at the top — Claude now knows the tech stack, deployment target, and consumers without reading any code. The IMPORTANT: rule on SQL injection is there because it is a critical security constraint that must never be violated.

Example 2: React + TypeScript Frontend¶

markdown
# Project: Dashboard UI

React 19 + TypeScript frontend for the analytics dashboard.
Vite build. TanStack Query for data fetching. Deployed to CloudFront.

## Communication
- Concise. Show the code, not the explanation
- Never show intermediate errors

## Coding Standards
- TypeScript strict mode. No any types. No @ts-ignore
- React: functional components only. No class components
- State: TanStack Query for server state, Zustand for client state
- Styling: Tailwind CSS. No inline styles. No CSS modules
- IMPORTANT: All user input must be sanitized before rendering (XSS prevention)

## Git Workflow
- Branch: feat/, fix/, refactor/
- Commits: conventional commits (feat:, fix:, chore:)
- Run tsc --noEmit && vitest run before committing

## Verification
- Type check: tsc --noEmit
- Test: vitest run (single file: vitest run src/components/Button.test.tsx)
- Lint: eslint . --fix
- Build: vite build (must succeed before PR)

## Component Conventions
- One component per file. File name matches component name
- Props interface exported and named {ComponentName}Props
- Test file co-located: Button.tsx + Button.test.tsx
- Shared hooks in src/hooks/ prefixed with use

## Domain Rules
@.claude/rules/typescript.md
@.claude/rules/security.md

## Architecture
@docs/frontend-architecture.md

42 lines. The “Component Conventions” section is especially important here — without it, Claude Code might put test files in a separate __tests__ directory, use different naming conventions, or inline props instead of exporting a named interface. These are team conventions that cannot be inferred from the framework itself.

Example 3: Monorepo with Multiple Services¶

markdown
# Project: Platform Monorepo

4 services: auth, billing, notifications, gateway.
Turborepo for builds. Docker Compose for local dev. Kubernetes for production.

## Communication
- Terse. Lead with answers
- Always specify which service you are modifying

## Coding Standards
- Backend (services/auth, services/billing): Python 3.12, FastAPI, SQLAlchemy 2.0
- Gateway (services/gateway): Go 1.22, chi router
- Frontend (apps/dashboard): React 19, TypeScript strict
- Shared (packages/): TypeScript for shared types, Python for shared utilities
- IMPORTANT: Changes to packages/ affect all consumers. Run full test suite

## Git Workflow
- Branch: {service}/feat/, {service}/fix/ (e.g., auth/feat/oauth-provider)
- Commits: scope required — feat(auth):, fix(billing):, refactor(gateway):
- Run affected tests only: turbo test --filter={changed-service}

## Verification
- Single service: cd services/{name} && pytest -x
- Affected services: turbo test --filter=...{changed-package}
- Full suite before merge to main: turbo test
- Type check all TS: turbo typecheck

## Inter-Service Communication
- Sync: gRPC between services. Proto files in proto/
- Async: RabbitMQ. Event schemas in schemas/events/
- IMPORTANT: Never call a service's database directly. Always go through its API

## Domain Rules
@.claude/rules/python.md
@.claude/rules/security.md
@.claude/rules/testing.md
@services/gateway/.claude/rules/golang.md

## Architecture
@docs/architecture.md

46 lines. The monorepo example demonstrates several important patterns. First, the scoped branch naming (auth/feat/...) prevents confusion about which service a branch targets. Second, the inter-service communication section prevents a common mistake — directly querying another service’s database instead of using the API boundary. Third, the IMPORTANT: rule about packages/ changes ensures Claude runs the full test suite when modifying shared code, because a change there can break any consumer.

What Most People Get Wrong¶

After reviewing dozens of CLAUDE.md files from professional teams and open-source projects, the same mistakes appear repeatedly. Here are the most common ones and how to avoid them.

Mistake 1: Writing an Essay Instead of Instructions¶

CLAUDE.md is not documentation. It is not a README. It is not a design document. It is a set of instructions. Every sentence should change Claude Code’s behavior. If a sentence is informational but does not affect what Claude Code does, it does not belong here.

Bad:

markdown
## Background
This project was started in 2024 as a replacement for the legacy
inventory system. It was originally built with Django but we migrated
to FastAPI in early 2025 for performance reasons. The team consists
of 4 backend engineers and 2 frontend developers. We follow agile
methodology with two-week sprints...

Good:

markdown
## Overview
FastAPI inventory service. Migrated from Django — do not introduce Django patterns.

One line. The only piece of that entire paragraph that changes Claude Code’s behavior is “do not introduce Django patterns.” Everything else is noise.

Mistake 2: Duplicating What the Code Already Says¶

If your linter config enforces 4-space indentation and your entire codebase uses it, you do not need to tell Claude to use 4-space indentation. Claude reads the code. It will follow existing patterns.

Only include rules that Claude would violate without being told. The test is simple: remove the rule, start a fresh session, ask Claude to write code, and see if it gets it right. If it does, the rule is unnecessary.

Mistake 3: Too Many Rules, Not Enough Priority¶

When everything is a rule, nothing is a rule. A CLAUDE.md with 50 bullet points of equal weight means Claude Code will follow most of them most of the time — but the critical ones (like “never commit secrets”) get the same priority as the trivial ones (like “prefer single quotes”).

Fix: Use hierarchy. Group by importance. Mark critical rules with IMPORTANT:. Keep the total count under 20-25 actionable items across your main CLAUDE.md, and push the rest into @imported rule files.

Mistake 4: Never Updating It¶

CLAUDE.md should evolve with your project. When you notice Claude Code making the same mistake repeatedly, that is a signal to add a rule. When you switch from Jest to Vitest, update the testing section. When you adopt a new convention, add it.

A practical cadence: review your CLAUDE.md once a month, or whenever you notice a pattern of repeated mistakes. Some teams add “update CLAUDE.md” to their definition of done when they change tooling or conventions.

Mistake 5: Not Using @imports¶

A CLAUDE.md that is 300 lines long because it includes full Python rules, security rules, and testing rules inline is paying a massive token cost on every API call. Move those into separate files with @imports. Better yet, make them path-scoped so they only load when relevant.

The natural progression:

Month 1:  Everything in CLAUDE.md (150 lines, always loaded)
Month 2:  Extract rules to @imports (CLAUDE.md: 50 lines + 3 rule files)
Month 3:  Add paths: frontmatter to rules (only load when relevant)

This is not premature optimization — it is basic engineering. You would not put your entire test suite in one file. Do not put your entire configuration in one file either.

Mistake 6: Conflicting Rules Across Levels¶

If your global CLAUDE.md says “use spaces for indentation” and your project CLAUDE.md says “use tabs for indentation,” you will get inconsistent behavior. Claude Code sees both instructions and has to make a judgment about which to follow.

Fix: Use global CLAUDE.md only for universal preferences that never conflict with project standards — communication style, personal workflow habits. Keep all coding standards at the project level where they belong.

Mistake 7: Including Secrets or Environment Configuration¶

I have seen CLAUDE.md files committed to git with database connection strings, API keys, and internal hostnames. This is a security incident waiting to happen. CLAUDE.md is committed to your repository. Anyone with repo access — including future employees, contractors, and anyone who forks a public repo — can read it.

Secrets belong in environment variables, secret managers, or .claude/settings.local.json (which is gitignored). Sensitive internal URLs belong in .claude/settings.local.json as well.

Mistake 8: Forgetting the Verification Section¶

The most common omission, and one of the highest-impact sections. Without explicit verification instructions, Claude Code writes code and considers the task complete. With verification instructions, it runs tests, checks types, and lints before declaring success.

markdown
## Verification
- Run tests before committing
- Run type checker (pyright or tsc --noEmit)
- Run linter (ruff check or eslint)
- When fixing bugs, write a failing test first

Four lines. They transform Claude Code from a code generator into a code generator that validates its own output. If you add only one section to your CLAUDE.md today, make it this one.

Testing Your CLAUDE.md¶

After writing or updating your CLAUDE.md, verify it works. Do not trust it blindly — test it the same way you would test code.

The Five-Minute Smoke Test¶

  1. Start a fresh session — run /clear or open a new terminal
  2. Ask Claude to write a small function in your primary language. Check: Does it follow your coding standards? (naming, types, imports, formatting)
  3. Ask Claude to commit the change. Check: Does it follow your git conventions? (branch name, commit message format)
  4. Ask Claude to fix a trivial bug. Check: Does it write a test first? (if your verification section says to)
  5. Ask Claude what the coding standards are. Check: Can it recite your rules back to you?

If any of these fail, your CLAUDE.md needs adjustment. Either the rule is not clear enough, it is buried too deep in the file, or it conflicts with another rule.

The Contradiction Test¶

Ask Claude Code directly: “Are there any contradictions in my project instructions?” Claude will identify conflicting rules between your CLAUDE.md levels or between inline rules and imported rules. This is especially useful after merging configuration changes from multiple team members.

Chapter Summary¶

Five takeaways for engineering your CLAUDE.md:

  1. CLAUDE.md is loaded every session. It is the single highest-leverage file in your Claude Code setup. Every line changes behavior across hundreds of interactions. Treat it as infrastructure, not documentation — engineer it with the same rigor you apply to your CI/CD pipeline.

  2. Use the hierarchy intentionally. Global for personal preferences, project for coding standards, local for directory-specific context. Most teams need only global + project. Do not over-engineer with four levels unless you have a concrete reason.

  3. Brevity is engineering, not laziness. Keep the main file under 80 lines with @imports for detailed rules. Use path-scoped frontmatter so rules load only when relevant. Shorter files get followed more reliably and cost fewer tokens per API call.

  4. Structure for priority. Critical rules go first. Use the IMPORTANT: prefix sparingly but deliberately. Communication preferences and security constraints before formatting preferences. Every line must pass the test: “Would Claude get this wrong without being told?”

  5. Evolve it. Review your CLAUDE.md when you notice repeated mistakes. Add rules for what Claude gets wrong, remove rules for what it gets right. A stale CLAUDE.md is worse than no CLAUDE.md — it wastes tokens on instructions that no longer apply and may conflict with your current practices.
