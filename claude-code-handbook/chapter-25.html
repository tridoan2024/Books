Part VII: Case Studies and Reference

Chapter 25: Case Study — SaaS Application from Zero to Production¶ ~26 min read

Everything in this book has been presented as individual techniques: how to scaffold a project, how to test, how to deploy, how to review code with subagents. This chapter weaves every technique into a single continuous narrative. We are going to build InvoiceFlow — a fully functional invoicing SaaS for freelancers — entirely with Claude Code, from the first mkdir to a production deployment on Vercel. No IDE. No manual coding. Just a developer, a terminal, and a conversation.

This is not a tutorial toy. InvoiceFlow handles authentication, a relational database, payment processing via Stripe, PDF generation, email notifications through Resend, and a real-time analytics dashboard. It is the kind of application that a two-person team would typically spend four to six weeks building. We will finish it in seven days, using approximately 150 prompts, generating around 15,000 lines of code with 75 percent test coverage.

Every prompt shown in this chapter was actually used. The mistakes were real. The twelve /rewind commands were necessary. The lessons cost real hours before they crystallized into principles.

The Application: InvoiceFlow¶

InvoiceFlow targets a specific pain point: freelancers who track invoices in spreadsheets, chase payments via email, and spend too much time formatting PDFs. The value proposition is direct — create a professional invoice in under sixty seconds, send it with a payment link, get paid.

Tech Stack¶

Layer Technology Why
Frontend Next.js 14 (App Router) Server components, streaming, file-based routing
ORM Prisma Type-safe queries, schema-as-code, migrations
Database PostgreSQL (Neon) Serverless Postgres, branching for dev environments
Payments Stripe Checkout Sessions + webhooks for payment tracking

Auth NextAuth.js v5 OAuth + email magic links with Prisma adapter

Deployment Vercel Zero-config Next.js hosting with edge functions

Feature Set¶

The completed application includes:

  • Authentication: Google OAuth and email magic links via NextAuth.js v5
  • Client Management: Full CRUD for clients with contact details and payment terms
  • Invoice Builder: Dynamic line-item editor with real-time tax calculation, discounts, and notes
  • Status Workflow: Draft -> Sent -> Viewed -> Paid -> Overdue with automatic transitions and validation
  • Payment Integration: Stripe Checkout for card payments, webhook-driven status updates, payment records
  • PDF Generation: Branded invoice PDFs, downloadable and attachable to outbound emails
  • Email Notifications: Invoice sent, payment received, and overdue reminders via Resend
  • Dashboard: Revenue charts, outstanding balance, invoice aging, client breakdown using Recharts

The CLAUDE.md File¶

This is the step most developers skip, and it costs them hours by day three. We create a project-specific CLAUDE.md encoding every architectural decision that matters:

Create a CLAUDE.md for this project. It's an invoicing SaaS using
Next.js 14 App Router, Prisma, PostgreSQL, Stripe, Resend. Include
coding standards, the data model overview, and API conventions.
Enforce server components by default, Zod for all validation,
and server actions for mutations.

Claude Code produces:

markdown
# InvoiceFlow

Invoicing SaaS for freelancers. Next.js 14 App Router + Prisma +
PostgreSQL + Stripe + Resend.

## Architecture
- Server Components by default. Client Components only for interactivity.
- Server Actions for all mutations (no API routes for form submissions).
- API routes only for webhooks (Stripe) and file generation (PDF).
- Zod schemas validate all input at the boundary. Share schemas
  between client and server.

## Coding Standards
- TypeScript strict mode. No `any` types.
- Prisma for all database access. No raw SQL unless performance requires it.
- Use `@/` path alias for all imports.
- Error handling: try/catch in Server Actions, return `{ error: string }`
  not throw.
- Components: one component per file, named export matching filename.

## Data Model
- User -> has many Clients -> has many Invoices -> has many LineItems
- Invoice statuses: DRAFT, SENT, VIEWED, PAID, OVERDUE
- All monetary values stored as integers (cents) to avoid floating point.

## Conventions
- Route groups: (auth) for login/register, (dashboard) for protected pages.
- Middleware handles auth redirect. No auth checks in individual pages.
- Stripe webhook verifies signature before processing.
- Email templates are React components in src/emails/.

That file pays for itself within hours. Every subsequent prompt inherits the knowledge that monetary values are in cents, that server components are the default, and that mutations go through server actions. Without it, you would re-specify these decisions in every prompt — or worse, Claude Code would make inconsistent choices across files.

Prisma Schema¶

The data model is the skeleton of the entire application. Getting it right on day one prevents migration headaches later:

Design the Prisma schema for InvoiceFlow. Users have clients, clients
have invoices, invoices have line items. Include invoice status enum
(DRAFT, SENT, VIEWED, PAID, OVERDUE), payment tracking fields,
and a settings model for branding (logo URL, company name, address).
Store all money as Int (cents). Add proper indexes for queries
we'll need: invoices by status, invoices by due date, invoices by client.

Claude Code generates a comprehensive schema:

prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

enum InvoiceStatus {
  DRAFT
  SENT
  VIEWED
  PAID
  OVERDUE
}

model User {
  id            String    @id @default(cuid())
  name          String?
  email         String    @unique
  emailVerified DateTime?
  image         String?
  accounts      Account[]
  sessions      Session[]
  clients       Client[]
  invoices      Invoice[]
  settings      UserSettings?
  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @updatedAt
}

model UserSettings {
  id          String  @id @default(cuid())
  userId      String  @unique
  user        User    @relation(fields: [userId], references: [id],
                      onDelete: Cascade)
  companyName String?
  address     String?
  logoUrl     String?
  taxRate     Int     @default(0) // basis points (1000 = 10%)
  currency    String  @default("USD")
  paymentTerms Int    @default(30) // days
}

model Client {
  id        String    @id @default(cuid())
  userId    String
  user      User      @relation(fields: [userId], references: [id],
                      onDelete: Cascade)
  name      String
  email     String
  company   String?
  address   String?
  phone     String?
  notes     String?
  invoices  Invoice[]
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt

  @@index([userId])
}

model Invoice {
  id            String        @id @default(cuid())
  userId        String
  user          User          @relation(fields: [userId], references: [id],
                              onDelete: Cascade)
  clientId      String
  client        Client        @relation(fields: [clientId], references: [id])
  number        String
  status        InvoiceStatus @default(DRAFT)
  issueDate     DateTime      @default(now())
  dueDate       DateTime
  subtotal      Int           @default(0)
  taxAmount     Int           @default(0)
  discount      Int           @default(0)
  total         Int           @default(0)
  notes         String?
  lineItems     LineItem[]
  payments      Payment[]
  viewedAt      DateTime?
  sentAt        DateTime?
  paidAt        DateTime?
  stripeSessionId String?
  createdAt     DateTime      @default(now())
  updatedAt     DateTime      @updatedAt

  @@unique([userId, number])
  @@index([userId, status])
  @@index([userId, dueDate])
  @@index([clientId])
}

model LineItem {
  id          String  @id @default(cuid())
  invoiceId   String
  invoice     Invoice @relation(fields: [invoiceId], references: [id],
                      onDelete: Cascade)
  description String
  quantity    Int     @default(1)
  unitPrice   Int     // cents
  amount      Int     // cents (quantity * unitPrice)
  sortOrder   Int     @default(0)
}

model Payment {
  id              String   @id @default(cuid())
  invoiceId       String
  invoice         Invoice  @relation(fields: [invoiceId], references: [id])
  amount          Int      // cents
  stripePaymentId String?  @unique
  createdAt       DateTime @default(now())
}

Two design decisions worth noting. First, the @@unique([userId, number]) constraint ensures invoice numbers are unique per user, not globally — freelancers want “INV-001” for their first invoice, not “INV-847291.” Second, all four monetary fields on Invoice (subtotal, taxAmount, discount, total) are stored as integers in cents. This came from the CLAUDE.md rule and prevents the floating-point rounding errors that plague every invoicing system built with DECIMAL or FLOAT.

We run the initial migration:

Run prisma migrate dev with the name "init" and then generate the client.
Verify it compiles.

Day one ends with a compiling project, a clean schema, and a CLAUDE.md that will guide every future prompt. Total: approximately 15 prompts, two hours.

Day 2: Authentication and Users¶

Authentication blocks everything. If users cannot log in, nothing else matters. We tackle it on day two while the project is still simple enough that auth configuration issues are easy to isolate.

NextAuth.js v5 Setup¶

Implement NextAuth.js v5 with two providers: Google OAuth and email
magic links (using Resend as the email transport). Use the Prisma
adapter. Configure middleware to protect all (dashboard) routes.
The login page should be at /login with a clean design showing
both sign-in options.

Claude Code generates the auth configuration in src/lib/auth.ts:

typescript
import NextAuth from "next-auth";
import { PrismaAdapter } from "@auth/prisma-adapter";
import Google from "next-auth/providers/google";
import Resend from "next-auth/providers/resend";
import { prisma } from "@/lib/db";

export const { handlers, auth, signIn, signOut } = NextAuth({
  adapter: PrismaAdapter(prisma),
  providers: [
    Google({
      clientId: process.env.GOOGLE_CLIENT_ID!,
      clientSecret: process.env.GOOGLE_CLIENT_SECRET!,
    }),
    Resend({
      from: process.env.EMAIL_FROM!,
      apiKey: process.env.RESEND_API_KEY!,
    }),
  ],
  pages: {
    signIn: "/login",
  },
  callbacks: {
    session({ session, user }) {
      session.user.id = user.id;
      return session;
    },
  },
});

The middleware in src/middleware.ts protects every route under the (dashboard) group:

typescript
import { auth } from "@/lib/auth";

export default auth((req) => {
  if (!req.auth && req.nextUrl.pathname.startsWith("/dashboard")) {
    const loginUrl = new URL("/login", req.url);
    loginUrl.searchParams.set("callbackUrl", req.nextUrl.pathname);
    return Response.redirect(loginUrl);
  }
});

export const config = {
  matcher: ["/((?!api|_next/static|_next/image|favicon.ico).*)"],
};

The First Rewind¶

The login page renders, but clicking “Sign in with Google” throws OAUTH_CALLBACK_ERROR. Rather than debugging manually, we hand the error to Claude Code:

Getting OAUTH_CALLBACK_ERROR when clicking Google sign-in.
Here's the console output: [pasted error]

Claude Code identifies the issue immediately: the NextAuth route handler file at src/app/api/auth/[...nextauth]/route.ts exists but is empty — a scaffolding oversight. The fix is two lines:

typescript
import { handlers } from "@/lib/auth";
export const { GET, POST } = handlers;

This was not a /rewind moment — it was a targeted fix. But the first real /rewind arrives twenty minutes later. Claude Code generates a session callback that breaks TypeScript types for the Session type. The type chain through NextAuth v5’s generics is complex, and the generated code creates a circular inference problem. Rather than spend fifteen minutes patching type assertions, we rewind:

/rewind

Implement the NextAuth session callback. The session object needs
user.id available. Extend the Session type in types/next-auth.d.ts
to include id on the user object. Create the type declaration file first.

The second attempt produces clean, type-safe code. The key difference: specifying “create the type declaration file first” forced Claude Code to handle the type extension before writing the callback that depends on it. Lesson learned — when the type system fights you, rewind and specify the dependency order explicitly.

User Settings¶

With auth working, we add the settings page:

Create a settings page at (dashboard)/settings/page.tsx. Server component
that loads current UserSettings from Prisma. A client component form
for editing company name, address, logo URL, default tax rate, currency,
and payment terms. Use a server action for the save mutation. Validate
with Zod. Create the settings automatically when a user first visits
(upsert pattern).

The upsert pattern is important — rather than forcing users through a setup wizard, we create default settings on first visit and let them edit at any time. Claude Code implements this as a findOrCreate in the server component.

Day two ends with working authentication, user profiles, and settings. Total: approximately 25 prompts (cumulative 40), four hours. Two of those hours were spent on the Google OAuth configuration in the Google Cloud Console, which Claude Code cannot do for you — it is a manual browser task.

Day 3: Core Features — Clients and Invoices¶

Day three is the heaviest. Clients and invoices represent the core domain, and the interactions between line items, calculations, status transitions, and PDF generation must all work together coherently.

Client Management¶

Build the client management pages. List page with search and pagination
at (dashboard)/clients/page.tsx. Detail page at (dashboard)/clients/[id]/page.tsx
showing client info and their invoice history. New/edit form as a modal
using a client component. Server actions for create, update, delete.
Zod validation. Confirm dialog before delete.

Claude Code generates five files: list page, detail page, client form component, server actions, and Zod schema. The list page uses server-side search params for filtering, which means no client-side state management and instant results on page load:

typescript
// src/app/(dashboard)/clients/page.tsx
import { prisma } from "@/lib/db";
import { auth } from "@/lib/auth";
import { ClientList } from "@/components/clients/client-list";

export default async function ClientsPage({
  searchParams,
}: {
  searchParams: { q?: string; page?: string };
}) {
  const session = await auth();
  const page = Number(searchParams.page) || 1;
  const query = searchParams.q || "";

  const [clients, total] = await Promise.all([
    prisma.client.findMany({
      where: {
        userId: session!.user.id,
        ...(query && {
          OR: [
            { name: { contains: query, mode: "insensitive" } },
            { email: { contains: query, mode: "insensitive" } },
            { company: { contains: query, mode: "insensitive" } },
          ],
        }),
      },
      include: { _count: { select: { invoices: true } } },
      orderBy: { name: "asc" },
      skip: (page - 1) * 20,
      take: 20,
    }),
    prisma.client.count({
      where: { userId: session!.user.id },
    }),
  ]);

  return <ClientList clients={clients} total={total} page={page} />;
}

Invoice Builder¶

The invoice builder is the most complex UI component in the application. It needs a dynamic line-item editor with real-time total calculation, and the calculation logic must be shared with the server action and PDF generator:

Build the invoice creation and editing pages. The invoice form needs:
- Client selector (dropdown of user's clients)
- Invoice number (auto-generated, editable)
- Issue date and due date pickers
- Dynamic line items: add/remove rows, each with description,
  quantity, unit price. Calculate line total, subtotal, tax, discount,
  and grand total in real time.
- Notes textarea
- Save as draft, or send immediately
Use a client component for the form. Store the calculation logic
in a shared util so we can reuse it for PDF generation.

This single prompt produces the most code of any prompt in the entire project — approximately 400 lines across the form component, the calculation utility, and the server actions. The shared calculation module is the linchpin:

typescript
// src/lib/invoice-calculations.ts
export interface LineItemInput {
  description: string;
  quantity: number;
  unitPrice: number; // cents
}

export function calculateInvoiceTotals(
  lineItems: LineItemInput[],
  taxRateBasisPoints: number,
  discountCents: number
) {
  const subtotal = lineItems.reduce(
    (sum, item) => sum + item.quantity * item.unitPrice,
    0
  );
  const taxAmount = Math.round((subtotal * taxRateBasisPoints) / 10000);
  const total = subtotal + taxAmount - discountCents;

  return {
    subtotal,
    taxAmount,
    discount: discountCents,
    total: Math.max(0, total),
    lineItems: lineItems.map((item) => ({
      ...item,
      amount: item.quantity * item.unitPrice,
    })),
  };
}

This function is called in three places: the React form component (for live totals), the server action (for validated persistence), and the PDF generator (for rendering). Because the CLAUDE.md specifies “money in cents,” all three callsites agree on the data format without any conversion layers.

Status Workflow¶

The invoice status machine needs explicit transitions with validation rules:

Implement the invoice status workflow as a state machine.
Valid transitions:
  DRAFT -> SENT (requires at least one line item, valid client email)
  SENT -> VIEWED (triggered when client opens invoice link)
  SENT -> PAID (triggered by Stripe webhook)
  VIEWED -> PAID (triggered by Stripe webhook)
  SENT -> OVERDUE (triggered by scheduled check when past due date)
  VIEWED -> OVERDUE (triggered by scheduled check when past due date)
  OVERDUE -> PAID (triggered by Stripe webhook)
Create a transitionInvoiceStatus server action that enforces these rules.

Claude Code generates a clean state machine with a transition map:

typescript
const VALID_TRANSITIONS: Record<InvoiceStatus, InvoiceStatus[]> = {
  DRAFT: ["SENT"],
  SENT: ["VIEWED", "PAID", "OVERDUE"],
  VIEWED: ["PAID", "OVERDUE"],
  PAID: [],
  OVERDUE: ["PAID"],
};

export async function transitionInvoiceStatus(
  invoiceId: string,
  newStatus: InvoiceStatus
): Promise<{ error?: string }> {
  const invoice = await prisma.invoice.findUnique({
    where: { id: invoiceId },
    include: { lineItems: true, client: true },
  });

  if (!invoice) return { error: "Invoice not found" };

  const allowed = VALID_TRANSITIONS[invoice.status];
  if (!allowed.includes(newStatus)) {
    return {
      error: `Cannot transition from ${invoice.status} to ${newStatus}`,
    };
  }

  if (newStatus === "SENT" && invoice.lineItems.length === 0) {
    return { error: "Invoice must have at least one line item" };
  }

  const timestamps: Partial<Record<string, Date>> = {};
  if (newStatus === "SENT") timestamps.sentAt = new Date();
  if (newStatus === "VIEWED") timestamps.viewedAt = new Date();
  if (newStatus === "PAID") timestamps.paidAt = new Date();

  await prisma.invoice.update({
    where: { id: invoiceId },
    data: { status: newStatus, ...timestamps },
  });

  return {};
}

The PAID state has an empty transitions array — it is a terminal state. The OVERDUE state permits only PAID, because overdue invoices can still be paid. These constraints prevent invalid state transitions that would confuse the dashboard metrics and break the email notification logic.

Stripe Checkout¶

Implement Stripe payment integration for invoices. When an invoice
is sent, generate a Stripe Checkout Session URL with the invoice
total and line items. Store the session ID on the invoice. Create
a public invoice page at /invoice/[id] (outside auth) that shows
invoice details and a "Pay Now" button linking to Stripe Checkout.
On successful payment, redirect to /invoice/[id]/paid. Handle
the checkout.session.completed webhook to transition invoice status
to PAID and create a Payment record.

The webhook handler is the most security-sensitive code in the project. It must verify the Stripe signature before processing any data, and the database operations must be atomic:

typescript
// src/app/api/stripe/webhook/route.ts
import { headers } from "next/headers";
import Stripe from "stripe";
import { prisma } from "@/lib/db";

const stripe = new Stripe(process.env.STRIPE_SECRET_KEY!);

export async function POST(req: Request) {
  const body = await req.text();
  const signature = headers().get("stripe-signature")!;

  let event: Stripe.Event;
  try {
    event = stripe.webhooks.constructEvent(
      body,
      signature,
      process.env.STRIPE_WEBHOOK_SECRET!
    );
  } catch (err) {
    return new Response("Invalid signature", { status: 400 });
  }

  if (event.type === "checkout.session.completed") {
    const session = event.data.object as Stripe.Checkout.Session;
    const invoiceId = session.metadata?.invoiceId;

    if (invoiceId) {
      await prisma.$transaction([
        prisma.invoice.update({
          where: { id: invoiceId },
          data: {
            status: "PAID",
            paidAt: new Date(),
          },
        }),
        prisma.payment.create({
          data: {
            invoiceId,
            amount: session.amount_total ?? 0,
            stripePaymentId: session.payment_intent as string,
          },
        }),
      ]);
    }
  }

  return new Response("OK", { status: 200 });
}

Two things to note. First, constructEvent verifies the Stripe signature before any business logic runs — this was in the CLAUDE.md conventions (“Stripe webhook verifies signature before processing”), so Claude Code included it without explicit prompting. Second, the status update and payment creation run in a $transaction because they must succeed or fail together. A paid invoice without a payment record, or vice versa, would corrupt the dashboard analytics.

Lessons Learned¶

Six lessons from building InvoiceFlow that generalize to any Claude Code project.

1. The CLAUDE.md Pays for Itself by Day Two¶

Without the CLAUDE.md, every prompt after Day 1 would need to re-specify “server components by default,” “money in cents,” “Zod for validation,” and “server actions for mutations.” With it, Claude Code just knows. The fifteen minutes spent writing the CLAUDE.md saved hours of repeated context.

The single convention that saved the most time: “All monetary values stored as integers (cents).” Without this, Claude Code would have used decimal in some files and integer in others, creating subtle rounding bugs that only surface in production edge cases — a freelancer sends an invoice for $1,333.33 and the cents don’t round-trip correctly.

2. Rewind Is Not Failure — It Is Navigation¶

Twelve rewinds across 150 prompts is an 8 percent rewind rate. Each rewind cost approximately two minutes of discarded work. The alternative — debugging and patching structurally flawed generated code — would have taken ten to thirty minutes per instance.

The pattern: if Claude Code generates code with a structural problem (wrong architecture, wrong type approach, wrong abstraction), rewind and re-prompt with more specificity. If the code has a minor bug (missing import, off-by-one error), fix it directly. The distinction is structural versus superficial.

3. One Feature per Prompt for Core Logic, Batch for Boilerplate¶

Day 3’s invoice builder was a single large prompt because line items, calculations, and form state are deeply interdependent. Splitting it into separate prompts would have produced code needing significant refactoring to integrate.

Conversely, Day 5’s loading states and empty states were batched in a single prompt because they are independent UI components with no shared logic.

The rule: if components share state or data flow, prompt them together. If they are independent, batch them in one prompt to save round-trips.

4. Test After the Feature Stabilizes, Not During Development¶

We wrote tests on Day 6, not alongside each feature. This is a pragmatic departure from the test-first approach discussed in Chapter 14. For a greenfield project built with Claude Code, early code changes frequently as the architecture solidifies. Tests written on Day 2 would have been rewritten three times by Day 5.

The compromise: write state transition tests early (they verify pure logic that will not change) but defer integration and E2E tests until the UI stabilizes. This is pragmatic, not ideal.

5. Security Audit Catches What You Forget¶

The /security-audit on Day 6 found three real issues that would have been embarrassing in production. The missing rate limiting on the public invoice page was an oversight born from focusing on functionality. The missing ownership check on the PDF route was a classic authorization flaw. These are the kinds of bugs that developers miss because they test happy paths.

Run the security audit before every deployment. As discussed in Chapter 19, the audit skill checks against OWASP Top 10 patterns and catches issues that manual review misses.

6. Environment Variables Are the Last Mile Problem¶

Both deployment failures on Day 7 — Prisma binary targets and environment variable configuration — represent a category of bugs that only surface in production. The code works perfectly locally where the environment is implicit. Production exposes every assumption the developer made.

The fix: create .env.example on Day 1, not Day 7. Document every variable with where to get its value. Run /deploy-check on Day 5 to surface issues early, not just before the final deployment.
