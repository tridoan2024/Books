Chapter 2: Setup and Configuration Mastery¶ ~18 min read

A junior engineer installs Claude Code with npm install -g and starts prompting. A senior engineer installs Claude Code, configures the provider, sets environment variables, wires up shell integration, creates a project configuration, and verifies the entire stack before writing a single prompt. The junior gets an AI chatbot. The senior gets an engineering system.

This chapter gets your installation right. Not “it runs” right – professionally configured right.

Installation Methods¶

npm (Recommended)¶

Claude Code is distributed as an npm package. You need Node.js 18+ installed.

bash
npm install -g @anthropic-ai/claude-code

Verify the installation:

bash
claude --version

You should see output like claude-code 1.x.x. If you get command not found, your npm global bin directory is not in your PATH. Fix it:

bash
# Find where npm puts global binaries
npm config get prefix

# Add to your shell profile (~/.zshrc or ~/.bashrc)
export PATH="$(npm config get prefix)/bin:$PATH"

Version Management¶

Pin to a specific version when stability matters (CI environments, team standardization):

bash
npm install -g @anthropic-ai/claude-code@1.0.32

Update to the latest:

bash
npm update -g @anthropic-ai/claude-code

Check for updates without installing:

bash
npm outdated -g @anthropic-ai/claude-code

For teams that need to coordinate versions, add the version to your project documentation or use a setup script:

bash
#!/usr/bin/env bash
set -euo pipefail

readonly REQUIRED_VERSION="1.0.32"
CURRENT_VERSION=$(claude --version 2>/dev/null | grep -oE '[0-9]+\.[0-9]+\.[0-9]+' || echo "none")

if [[ "${CURRENT_VERSION}" != "${REQUIRED_VERSION}" ]]; then
    echo "Installing Claude Code v${REQUIRED_VERSION}..."
    npm install -g "@anthropic-ai/claude-code@${REQUIRED_VERSION}"
fi

First Launch: What Happens¶

The first time you run claude, several things happen in sequence:

bash
$ claude

  1. API key prompt. If ANTHROPIC_API_KEY is not set, Claude Code asks for it interactively and stores it in your system keychain (macOS) or credential store (Linux).

  2. Directory creation. ~/.claude/ is created with default configuration files.

  3. Project detection. If you are in a git repository, Claude Code looks for .claude/ directory and CLAUDE.md files.

  4. System prompt assembly. Claude Code builds the system prompt from your configuration (more on this in Chapter 3).

  5. REPL. You get an interactive prompt. Type natural language, get results.

First launch output looks like this:

$ claude

  Claude Code v1.0.32
  Model: claude-sonnet-4-20250514

  Tips:
  - /help for available commands
  - /model to switch models
  - /doctor to diagnose issues
  - @ to reference files

  >

Environment Setup¶

API Keys and Providers¶

Claude Code supports four provider backends. You only need one.

Option 1: Anthropic Direct (default)

bash
export ANTHROPIC_API_KEY="sk-ant-api03-..."

Add this to your shell profile (~/.zshrc or ~/.bashrc) for persistence:

bash
echo 'export ANTHROPIC_API_KEY="sk-ant-api03-..."' >> ~/.zshrc

Option 2: AWS Bedrock

For organizations that route through AWS. Bedrock is common in enterprises that require data to stay within their AWS account and compliance boundary:

bash
export CLAUDE_CODE_USE_BEDROCK=1
export AWS_REGION="us-east-1"
export AWS_ACCESS_KEY_ID="AKIA..."
export AWS_SECRET_ACCESS_KEY="..."

Bedrock uses your standard AWS credential chain, so IAM roles, SSO, and credential files all work:

bash
# Using AWS SSO (recommended for organizations)
aws sso login --profile my-profile
export AWS_PROFILE="my-profile"
export CLAUDE_CODE_USE_BEDROCK=1

Bedrock model IDs differ from direct API. Claude Code handles the mapping automatically – you still use /model opus and it resolves to the correct Bedrock model ARN.

Option 3: Google Vertex AI

For organizations on Google Cloud Platform:

bash
export CLAUDE_CODE_USE_VERTEX=1
export CLOUD_ML_REGION="us-east5"
export ANTHROPIC_VERTEX_PROJECT_ID="my-gcp-project"

Vertex AI uses Application Default Credentials:

bash
gcloud auth application-default login

Like Bedrock, Vertex has its own model identifiers, but Claude Code handles the translation. You configure the provider once and use the same /model opus and /model sonnet commands regardless of backend.

Option 4: OpenAI-Compatible Endpoints

For custom proxies or alternative providers:

bash
export CLAUDE_CODE_USE_THIRD_PARTY=1
export ANTHROPIC_BASE_URL="https://my-proxy.example.com/v1"
export ANTHROPIC_API_KEY="my-proxy-key"

A professional configuration for power users looks like this in settings.json:

json
{
  "env": {
    "CLAUDE_CODE_MAX_OUTPUT_TOKENS": "128000",
    "CLAUDE_CODE_MAX_CONTEXT_TOKENS": "1000000",
    "MAX_THINKING_TOKENS": "100000",
    "CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY": "8",
    "CLAUDE_CODE_MAX_RETRIES": "5",
    "CLAUDE_CODE_IDLE_THRESHOLD_MINUTES": "180",
    "DISABLE_TELEMETRY": "1",
    "PYTHONDONTWRITEBYTECODE": "1"
  }
}

Key insight: CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY set to 8 allows Claude Code to execute up to eight tool calls in parallel. This dramatically speeds up operations like searching multiple files or running multiple commands. The default of 1 is conservative; if your machine can handle it, raise it.

Model Selection: Opus vs. Sonnet vs. Haiku¶

Claude Code supports three model tiers. Choosing the right one per task is a genuine engineering decision.

Model Strengths Weaknesses Cost (Input/Output per MTok)
Opus Deep reasoning, complex architecture, security analysis, multi-file refactoring Slower, expensive $15 / $75
Sonnet Balanced speed/quality, code generation, reviews, most daily tasks Less depth than Opus on complex reasoning $3 / $15
Haiku Fast responses, simple queries, formatting, boilerplate Misses nuance, weaker on complex code $0.25 / $1.25

The practical rule: Use Sonnet as your default. Switch to Opus for security reviews, complex debugging, architectural decisions, and multi-file refactoring. Use Haiku for quick questions, documentation formatting, and simple boilerplate generation.

Switch models during a session:

/model opus          # Deep reasoning for the next task
/model sonnet        # Back to balanced
/model haiku         # Quick answers

Override the default model globally:

bash
export ANTHROPIC_MODEL="claude-opus-4-20250514"

Or per-skill in SKILL.md frontmatter:

yaml
---
name: security-audit
model: opus
---

This is powerful. Your security audit skill uses Opus ($15/MTok) for depth, while your code formatting skill uses Haiku ($0.25/MTok) for speed. You pay for intelligence where it matters.

Shell Integration¶

Bash/Zsh Completion¶

Claude Code supports tab completion for commands, file paths, and arguments. Enable it:

bash
# For zsh (add to ~/.zshrc)
eval "$(claude completion zsh)"

# For bash (add to ~/.bashrc)
eval "$(claude completion bash)"

After sourcing your profile, you get:

bash
$ claude /      # Tab shows all slash commands
$ claude @src/  # Tab shows files in src/

Terminal Requirements¶

Claude Code works best in terminals that support:

  • Unicode. For rendering special characters and Mermaid diagrams.
  • 256-color or TrueColor. For syntax highlighting in diffs and code blocks.
  • Minimum 80 columns wide. For readable output. 120+ recommended.

Terminals known to work well: iTerm2 (macOS), WezTerm, Alacritty, Kitty, Windows Terminal, VS Code integrated terminal. The default macOS Terminal.app works but has limited color support.

Non-Interactive Mode¶

Claude Code runs in headless mode for scripting and CI/CD:

bash
# Single prompt, get output, exit
claude -p "What is the purpose of src/auth/middleware.py?"

# Pipe input
echo "Review this for security issues" | claude -p --stdin

# JSON output for programmatic consumption
claude -p "List all Python files" --output-format json

# Stream output as it generates
claude -p "Explain this codebase" --output-format stream-json

Non-interactive mode is essential for CI integration (Chapter 20) and building automation scripts that use Claude Code as a component.

Command-Line Flags Reference¶

Claude Code accepts several flags that control its behavior at launch:

Flag Purpose Example
--version Print version and exit claude --version
-p Non-interactive single prompt claude -p "explain main.py"
--resume Resume previous session claude --resume
--output-format Output format: text, json, stream-json claude -p "..." --output-format json
--stdin Read additional input from stdin cat error.log \| claude -p "explain" --stdin
--model Override model for this session claude --model opus
--allowedTools Restrict available tools claude --allowedTools Read,Grep,Glob
--no-cache Bypass prompt cache claude --no-cache
--verbose Extra debugging output claude --verbose

These flags compose. For example, a CI script might use:

bash
claude -p "Run the test suite and report failures" \
  --output-format json \
  --model sonnet \
  --allowedTools Bash,Read,Grep,Glob

This runs a single prompt, outputs JSON (for programmatic parsing), uses Sonnet (cheaper for CI), and restricts tools to read-only plus Bash (so it can run tests but not edit files).

Directory Structure: What Gets Created Where¶

Understanding where Claude Code stores its data prevents confusion later.

Global Configuration: ~/.claude/¶

~/.claude/
├── settings.json           # Your global user settings
├── credentials.json        # Encrypted API keys (auto-managed)
├── projects/               # Per-project auto-memory
│   └── <project-hash>/
│       └── MEMORY.md       # What Claude remembers about this project
├── sessions/               # Session history (SQLite)
└── todos/                  # Task tracking

~/.claude/settings.json contains your user-level settings. These apply to every project unless overridden. Example:

json
{
  "permissions": {
    "allow": [
      "Bash(git *)",
      "Bash(npm *)"
    ]
  },
  "model": "claude-sonnet-4-20250514",
  "effortLevel": "high"
}

Project Configuration: .claude/¶

your-project/
├── .claude/
│   ├── settings.json       # Shared project settings (committed to git)
│   ├── settings.local.json # Personal overrides (gitignored)
│   ├── rules/              # Path-scoped coding standards
│   ├── skills/             # Reusable workflow definitions
│   ├── agents/             # Specialized subagent definitions
│   └── hooks/              # Automation shell scripts
├── CLAUDE.md               # Project brain (committed to git)
└── ... your code ...

The settings cascade: Settings are merged in this order, with later values overriding earlier ones:

Managed (enterprise) → User (~/.claude/) → Project (.claude/) → Local (.claude/settings.local.json)

This means your team commits .claude/settings.json with safe defaults and shared hooks, while individual engineers override with personal permissions in settings.local.json (which is .gitignored).

What to Commit vs. What to Gitignore¶

File Commit? Why
CLAUDE.md Yes Team-shared project instructions
.claude/settings.json Yes Shared hooks, permissions, env vars
.claude/settings.local.json No Personal overrides (add to .gitignore)
.claude/rules/*.md Yes Team coding standards
.claude/skills/*/SKILL.md Yes Team workflow definitions
.claude/agents/*.md Yes Team subagent definitions
.claude/hooks/*.sh Yes Team automation scripts

Add to your .gitignore:

.claude/settings.local.json

IDE Integration¶

VS Code Extension¶

Claude Code has a VS Code extension that connects the terminal-based agent to your editor:

bash
# Install the extension
code --install-extension anthropic.claude-code

The extension provides:

  • Inline diff view for Claude Code’s edits (accept/reject per hunk)
  • Status bar showing current model, token usage, session status
  • Terminal integration for the Claude Code REPL
  • File decorations marking files Claude Code has modified

Using Claude Code primarily through VS Code is a valid workflow, especially if you want visual diffs. But the terminal-native workflow is faster for power users who configure skills and hooks.

JetBrains Plugin¶

A JetBrains plugin provides similar functionality for IntelliJ, PyCharm, WebStorm, and other JetBrains IDEs.

Terminal-Native Workflow (Recommended for This Book)¶

This book assumes you are working in the terminal. The terminal-native workflow is:

  1. Open a terminal in your project directory
  2. Run claude
  3. Type prompts, get results
  4. Use /commands for built-in operations
  5. Exit with /exit or Ctrl+D

Every example in this book works in the terminal. IDE-specific features are bonuses, not requirements.

Multi-Model Setup¶

A professional setup uses different models for different tasks. Here is how to configure that.

Per-Session Model Switching¶

During a session, switch models with the /model command:

> /model opus
Switched to claude-opus-4-20250514

> Analyze the security implications of our auth middleware
[deep analysis using Opus]

> /model sonnet
Switched to claude-sonnet-4-20250514

> Add a docstring to this function
[quick task using Sonnet]

Per-Skill Model Overrides¶

In your skill’s SKILL.md frontmatter, specify which model to use:

yaml
---
name: security-audit
description: Deep security analysis...
model: opus
effort: high
---
yaml
---
name: scaffold
description: Bootstrap a new project...
model: sonnet
effort: medium
---

This means /security-audit automatically uses Opus (the most capable model for security reasoning), while /scaffold uses Sonnet (fast enough for generating boilerplate). You never think about model selection during the workflow – it is baked into the configuration.

Per-Agent Model Selection¶

Agents (Chapter 6) can specify their model:

yaml
---
name: security-reviewer
model: opus
tools: Read, Grep, Glob, Bash
---
yaml
---
name: code-reviewer
model: sonnet
tools: Read, Grep, Glob
---

When Claude Code delegates to the security-reviewer agent, it uses Opus. When it delegates to code-reviewer, it uses Sonnet. Parallel agent delegation means both can run simultaneously, each on the optimal model.

Cost Management¶

With different models at different price points, monitoring cost matters:

bash
# Check current session usage
/stats

# Output:
#   Session tokens: 45,231 input / 12,456 output
#   Estimated cost: $0.87
#   Duration: 23 minutes
#   Tool calls: 14

A rough monthly cost guide:

Usage Pattern Monthly Cost
Light (1-2 hours/day, mostly Sonnet) $50-100
Moderate (4 hours/day, Sonnet + occasional Opus) $200-400
Heavy (8+ hours/day, frequent Opus for security/architecture) $500-1000
Enterprise (team of 5, mixed models) $2000-5000

The Anthropic Max plan ($100-200/month) provides a fixed-price alternative to API usage. It includes generous limits for individual use and is worth considering if your monthly API costs consistently exceed $150.

Essential Built-In Commands¶

Claude Code ships with dozens of built-in commands. Here are the ones you should know from day one, before building any custom skills:

Session Management¶

Command What It Does
/clear Wipe conversation context and start fresh. Use between unrelated tasks
/compact [topic] Summarize conversation to free context. Optionally keep topic-specific content
/exit or Ctrl+D End the session
/resume Resume the most recent session
/context Show how much context is used (tokens, percentage)
/rewind Roll back to an earlier conversation state

Model and Reasoning Control¶

Command What It Does
/model [name] Switch model: opus, sonnet, haiku
/effort [level] Set reasoning depth: low, medium, high, max
/fast Toggle fast output mode
/thinking Show/hide Claude’s extended thinking

The effort level is a surprisingly powerful control. At low, Claude Code answers quickly with minimal reasoning – good for “what does this function do?” questions. At high (the default in a professional setup), it reasons deeply before responding – better for debugging, architecture, and security analysis. At max, it engages extended thinking with large thinking budgets – use for the hardest problems.

> /effort low
> What does the @app decorator do in Flask?
[Quick, direct answer in 2 seconds]

> /effort high
> Why is our authentication middleware leaking sessions under load?
[Deep analysis with hypothesis formation, evidence search, root cause]

Git Operations¶

Command What It Does
/diff Show uncommitted changes with syntax highlighting
/commit Create a commit with auto-generated message
/pr-comments [N] Show comments on PR #N

Exploration¶

Command What It Does
/btw [question] Side question that does not pollute the main context
/stats Show session usage: tokens, cost, duration, tool calls
/export [file] Save the entire conversation to a file
/help List all available commands

Pro tip: /btw is underused. If you are deep in a debugging session and want to ask “what does __slots__ do in Python?” without contaminating your debug context, /btw is the answer. The question and response exist in a side channel.

Effort Level Configuration¶

The effort level deserves its own section because it directly impacts quality, speed, and cost.

Set the default effort in settings.json:

json
{
  "effortLevel": "high"
}

Override per-skill:

yaml
---
name: security-audit
effort: high
---
yaml
---
name: scaffold
effort: medium
---

The thinking budget also matters. Extended thinking gives Claude more “internal scratch space” to reason through complex problems. Configure it:

json
{
  "env": {
    "MAX_THINKING_TOKENS": "100000"
  },
  "alwaysThinkingEnabled": true
}

With alwaysThinkingEnabled, you see Claude’s thinking process in real-time – useful for understanding why it made certain choices. With thinking disabled, you only see the final output – cleaner but less transparent.

Setting Use Case
effort: low, thinking off Quick questions, boilerplate, formatting
effort: high, thinking on Debugging, architecture, security analysis
effort: max, thinking on + high budget Complex multi-file refactoring, novel problem solving

First Professional Session Walkthrough¶

Let’s put it all together. You have a fresh project, and you want to set it up for professional Claude Code usage from day one.

Step 1: Clone Your Repo¶

bash
git clone https://github.com/your-org/your-project.git
cd your-project

Step 2: Create the Configuration Structure¶

bash
mkdir -p .claude/rules .claude/skills .claude/hooks .claude/agents

Step 3: Create CLAUDE.md¶

This is the single most important file. Start minimal and iterate:

bash
cat > CLAUDE.md << 'EOF'
# Project: Your Project Name

## Communication
- Be terse. Lead with the answer
- Reference code as `file:line`
- No emojis

## Coding Standards
- Python: type hints, f-strings, pathlib, PEP 8
- Tests: pytest, test-first for bug fixes
- Formatting: ruff

## Git
- Branch prefixes: feat/, fix/, docs/, refactor/
- Commits: imperative mood, explain WHY

## Verification
- Run tests before committing
- Lint with ruff check
EOF

Step 4: Create Project Settings¶

bash
cat > .claude/settings.json << 'EOF'
{
  "$schema": "https://json.schemastore.org/claude-code-settings.json",
  "permissions": {
    "allow": [
      "Bash(git *)",
      "Bash(python -m pytest *)",
      "Bash(ruff *)"
    ],
    "deny": [
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./*credentials*)",
      "Read(./*secret*)"
    ]
  },
  "env": {
    "PYTHONDONTWRITEBYTECODE": "1"
  },
  "effortLevel": "high",
  "includeGitInstructions": true
}
EOF

Step 5: Create a Python Rule¶

bash
cat > .claude/rules/python.md << 'EOF'

paths:
  - "**/*.py"
  - "**/*.pyi"
---

# Python Rules

- Type hints on all function signatures
- f-strings over .format() or %
- pathlib over os.path
- Use dataclasses or Pydantic for data structures
- Prefer `from __future__ import annotations`
EOF

> Review src/main.py

Claude Code reads src/main.py, applies your coding standards from CLAUDE.md and .claude/rules/python.md, and produces severity-ranked findings with file:line references. No extra instructions needed – the configuration handles it.

Step 8: Verify the Configuration Stack¶

Run a quick smoke test to confirm everything works together:

> Write a simple Python function in src/utils/helpers.py
  that validates email addresses

If any of these are missing, your configuration has a gap. Check the file paths in your rules’ paths: frontmatter and verify CLAUDE.md is in the project root.

What Most People Get Wrong in Setup¶

Using Default Permissions Forever¶

The default permission mode asks for confirmation on nearly every tool call. After a week of clicking “Allow” on git diff, pytest, and ruff hundreds of times, most engineers either switch to AllowAll (too permissive) or just tolerate the interruptions (too slow).

The professional approach: explicitly allow the safe, frequently-used commands and explicitly deny the dangerous ones. Chapter 5 covers the permission system in depth, but even a basic allow list transforms the experience:

json
{
  "permissions": {
    "allow": [
      "Bash(git *)",
      "Bash(python -m pytest *)",
      "Bash(ruff *)",
      "Read(*)",
      "Glob(*)",
      "Grep(*)"
    ],
    "deny": [
      "Read(./.env)",
      "Bash(rm -rf *)"
    ]
  }
}

Not Setting Up a settings.local.json¶

Your team’s settings.json should be conservative – safe defaults for everyone. Your personal settings.local.json can be more permissive because you know your own habits and risk tolerance:

json
{
  "permissions": {
    "allow": [
      "Bash(*)",
      "Edit(*)",
      "Write(*)"
    ]
  }
}

This lets you work without permission prompts while the team config stays safe. Remember to add .claude/settings.local.json to .gitignore.

Ignoring the env Configuration¶

The "env" block in settings.json sets environment variables for every Claude Code session. Most engineers do not configure it. The professional setup uses it to tune concurrency, output limits, and tool behavior:

json
{
  "env": {
    "CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY": "8",
    "CLAUDE_CODE_MAX_OUTPUT_TOKENS": "128000",
    "MAX_THINKING_TOKENS": "100000",
    "PYTHONDONTWRITEBYTECODE": "1"
  }
}

These four settings alone make Claude Code noticeably faster (parallel tool execution), capable of longer outputs (128K vs 16K default), better at complex reasoning (100K thinking tokens), and cleaner (no .pyc files cluttering your project).

Verifying Installation: The Doctor¶

The /doctor command is your diagnostic tool. Run it whenever something feels off:

> /doctor

It checks: - Claude Code version and update availability - API connectivity (can it reach the provider?) - Model availability (is the selected model accessible?) - Configuration files (are they valid JSON/YAML?) - Hook executability (are hook scripts marked executable?) - MCP server connectivity (if configured) - Disk space for session storage

If /doctor reports issues, fix them before doing anything else. A misconfigured setup produces inconsistent results that are hard to debug.

Troubleshooting Common Setup Issues¶

“command not found: claude”¶

Your npm global bin is not in your PATH.

bash
# Find the bin directory
npm config get prefix
# Output: /usr/local/lib (or similar)

# The binary is at <prefix>/bin/claude
# Add to PATH in your shell profile
export PATH="$(npm config get prefix)/bin:$PATH"

“API key not found”¶

Claude Code looks for the key in this order: 1. ANTHROPIC_API_KEY environment variable 2. System keychain (macOS Keychain, Linux credential store) 3. Interactive prompt

If the environment variable is set but Claude Code cannot find it, check for typos and ensure it is exported (not just assigned):

bash
# Wrong (not exported):
ANTHROPIC_API_KEY="sk-ant-..."

# Right (exported):
export ANTHROPIC_API_KEY="sk-ant-..."

“Model not available”¶

If you get errors about model availability:

bash
# Check which models your API key can access
claude -p "What model are you?" --output-format json

Opus requires a separate permission tier on some API plans. If Opus is unavailable, Sonnet is the fallback. Set it explicitly:

bash
export ANTHROPIC_MODEL="claude-sonnet-4-20250514"

“Permission denied” on Hook Scripts¶

Hook scripts must be executable:

bash
chmod +x .claude/hooks/*.sh

CLAUDE.md Not Loading¶

The file must be named exactly CLAUDE.md (case-sensitive) and placed in the project root (the directory where you run claude). It can also be placed in parent directories for global rules:

~/CLAUDE.md                    # Global (applies to all projects)
~/projects/CLAUDE.md           # Workspace-level
~/projects/my-app/CLAUDE.md    # Project-level (highest priority)

Settings Not Merging Correctly¶

Settings merge in order: Managed > User > Project > Local. To debug which settings are active:

bash
# Validate JSON syntax
python3 -c "import json; json.load(open('.claude/settings.json'))"
python3 -c "import json; json.load(open('.claude/settings.local.json'))"

If a setting from your project settings.json is not taking effect, check if settings.local.json is overriding it. Local settings always win.

Slow Response Times¶

If Claude Code feels sluggish:

  1. Check your network. API calls go to api.anthropic.com (or your Bedrock/Vertex endpoint). High latency to Anthropic’s servers directly impacts response time. A quick test: curl -o /dev/null -s -w "%{time_total}\n" https://api.anthropic.com – anything under 200ms is fine, over 500ms is a problem.
  2. Reduce context. Large projects with many files in the root directory slow down initial context assembly. Use /clear between unrelated tasks.
  3. Use Sonnet for quick tasks. Opus is slower (first-token latency ~1.5s vs ~0.7s for Sonnet). Switch to Sonnet for simple edits and questions. As we’ll see in Chapter 3, model choice directly impacts response time.
  4. Increase concurrency. Set CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY to 4 or 8 to parallelize tool execution. This is one of the highest-impact performance settings – when Claude Code needs to read 5 files, it reads them simultaneously instead of sequentially.
json
{
  "env": {
    "CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY": "8"
  }
}

What Most People Get Wrong: The Setup Checklist¶

Before moving on, verify your setup against this checklist. These are the items that most engineers skip and later regret:

  • [ ] API key set in shell profile (not just the current session)
  • [ ] CLAUDE.md exists in project root with coding standards
  • [ ] .claude/settings.json has explicit allow and deny permissions
  • [ ] .claude/settings.local.json exists and is in .gitignore
  • [ ] At least one path-scoped rule (for your primary language)
  • [ ] .claude/ directory structure created (rules, skills, hooks, agents)
  • [ ] CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY set to 4 or higher
  • [ ] MAX_THINKING_TOKENS set to 10000 or higher
  • [ ] /doctor runs clean with no warnings
  • [ ] You have run one real task and verified configuration applied

If all ten items are checked, you have a professional setup. If fewer than five are checked, you are leaving significant capability on the table.

Key Takeaways¶

  1. Configure your provider early. Whether Anthropic direct, Bedrock, or Vertex, set the environment variables in your shell profile so they persist across sessions.

  2. The settings cascade matters. Managed > User > Project > Local. Commit .claude/settings.json for team standards; use settings.local.json for personal overrides. Gitignore the local file.

  3. Match models to tasks. Opus for security and architecture, Sonnet for daily work, Haiku for quick queries. Configure this per-skill so it is automatic.

  4. Set up the configuration structure immediately. CLAUDE.md, .claude/settings.json, .claude/rules/, .claude/skills/, .claude/hooks/, .claude/agents/. Even if they are mostly empty, the structure enables growth.

  5. Use /doctor liberally. It catches configuration issues before they manifest as inconsistent behavior. Run it after any setup change.
