Chapter 14: Memory, Sessions, and Persistence¶ ~18 min read

You spend forty-five minutes teaching Claude Code about your project’s authentication architecture – the custom JWT rotation scheme, the Redis session store, the edge cases around token refresh during deployments. The session produces excellent results. You close the terminal. The next morning, you open Claude Code and ask a follow-up question about the same authentication system. Claude has no idea what you are talking about.

This is the persistence problem. By default, Claude Code sessions are ephemeral. When the session ends, the conversation is gone. But Anthropic built three mechanisms to bridge this gap: auto-memory, session storage, and plan files. Understanding how each works – and when to use which – is the difference between starting fresh every morning and having Claude Code that actually learns from working with you.

The Three Persistence Mechanisms¶

Before diving into details, here is the landscape:

Mechanism What Persists Where Stored Lifespan Loaded Automatically
Auto-memory Facts, preferences, corrections ~/.claude/projects/<hash>/memory/ Permanent (until deleted) Yes, every session
Sessions Full conversation history SQLite database Until explicitly deleted No, must /resume
Plan files Structured implementation plans .claude/plans/ Until deleted No, loaded on demand

Each serves a different purpose. Auto-memory captures what Claude should always know. Sessions capture what happened in a conversation. Plans capture what we decided to do and why.

Auto-Memory: Claude’s Long-Term Knowledge¶

The auto-memory system is Claude Code’s mechanism for retaining important information across sessions. When you correct Claude, state a preference, or share project context that should persist, Claude writes it to a memory file that gets loaded at the start of every future session.

Where Memory Lives¶

Memory files are stored at:

~/.claude/projects/<project-hash>/memory/
├── MEMORY.md                          # Index file (loaded every session)
├── user_preferences.md                # Your personal preferences
├── project_architecture.md            # Project-specific context
├── feedback_no_force_push.md          # A correction you gave
└── reference_api_conventions.md       # Domain knowledge

The <project-hash> is derived from your project’s absolute path, so memories are scoped per project. Claude working on /projects/webapp has different memories than Claude working on /projects/cli-tool.

MEMORY.md – The Index¶

MEMORY.md is the master index file. It is loaded into the system prompt at the start of every session. It contains one-line summaries of each memory file with links:

markdown
# Memory Index

- [user_preferences.md](user_preferences.md) -- Prefers terse output, uses ruff, type hints required
- [project_architecture.md](project_architecture.md) -- Django 5.1, PostgreSQL, Redis cache, Celery workers
- [feedback_no_force_push.md](feedback_no_force_push.md) -- Never force-push to main; all changes via PR
- [reference_api_conventions.md](reference_api_conventions.md) -- REST API follows JSON:API spec, snake_case fields

Because MEMORY.md is loaded into the system prompt, it counts toward your system prompt token budget (as discussed in Chapter 13). A bloated MEMORY.md with 50 entries adds significant overhead. Keep it under 200 lines – more on hygiene later.

Memory File Anatomy¶

Each memory file follows a consistent structure with YAML frontmatter:

markdown
---
name: No force push to main
description: Team rule -- all changes go through PRs, never push directly to main
type: feedback
---

Never force-push to main or push directly to main branch.

**Why:** Production deploys from main automatically. A bad push = instant outage.
**How to apply:** Always create a branch, always create a PR, always wait for CI.

The frontmatter fields:

Field Purpose
name Short identifier for the memory
description One-line summary (appears in MEMORY.md index)
type Category: user, feedback, project, reference

Memory Types¶

user type – Information about you as an engineer:

markdown
---
name: User profile
description: Senior engineer, security focus, prefers concise output
type: user
---

- Senior security engineer at a mid-stage startup
- Expert in Python, TypeScript, and infrastructure
- Prefers concise output -- lead with the answer
- Uses vim keybindings
- Timezone: US Pacific

feedback type – Corrections and behavioral adjustments:

markdown
---
name: Always run tests before committing
description: Correction -- never commit without running the test suite first
type: feedback
---

Always run `python -m pytest tests/` before creating a git commit.
Previous behavior: committed code without running tests.
Corrected behavior: run tests, fix failures, then commit.

project type – Project-specific technical context:

markdown
---
name: Authentication architecture
description: JWT + Redis session store with 15-minute rotation
type: project
---

## Authentication System
- JWT tokens with 15-minute expiry
- Refresh tokens stored in Redis (7-day TTL)
- Token rotation: new access token issued on each refresh
- Edge case: during deployments, accept tokens up to 30 min old (grace period)
- Auth middleware: src/middleware/auth.py
- Token service: src/services/token_service.py

reference type – Domain knowledge and conventions:

markdown
---
name: API conventions
description: REST API follows JSON:API spec with snake_case fields
type: reference
---

## API Conventions
- All endpoints return JSON:API format
- Field names: snake_case (not camelCase)
- Pagination: cursor-based, not offset
- Error format: { "errors": [{ "code": "...", "detail": "..." }] }
- Versioning: URL-based (/api/v2/)

When Claude Auto-Saves Memories¶

Claude creates memories automatically in these situations:

  1. You correct Claude’s behavior. “Don’t use print statements, use logging.” Claude saves this as a feedback memory so it applies the correction in all future sessions.

  2. You state a preference. “I prefer Pydantic over dataclasses for API models.” Claude saves this as a user preference.

  3. You share project context that would be lost. “The payment service uses Stripe webhooks on /api/webhooks/stripe.” Claude saves this as a project memory.

  4. Claude discovers something important during work. After investigating your codebase, Claude might save an architecture note about how your modules connect.

Auto-save is not always perfect. Sometimes Claude saves too aggressively (trivial details) or not aggressively enough (missing important context). You can manage this explicitly.

Manual Memory Management¶

Asking Claude to remember something:

Remember: our staging environment uses a separate Stripe test key,
stored in AWS Secrets Manager under /staging/stripe/api-key

Claude creates a memory file and adds it to MEMORY.md.

Asking Claude to forget something:

Forget the memory about our old deployment process -- we switched to Kubernetes last month

Claude removes the memory file and its MEMORY.md entry.

Reviewing what Claude remembers:

What do you remember about this project?

Claude lists its loaded memories. You can also read MEMORY.md directly:

bash
cat ~/.claude/projects/*/memory/MEMORY.md

Memory Hygiene¶

Over time, memories accumulate. Some become outdated. Others are too granular to be worth the token cost. Here are the hygiene practices:

Keep MEMORY.md under 200 lines. Each memory entry in MEMORY.md adds to your system prompt cost. At 200 lines, MEMORY.md alone consumes roughly 800-1,200 tokens – manageable. At 500 lines, you are spending 2,000-3,000 tokens on memory before you type a word.

Consolidate related memories. Five separate memories about API conventions should be one comprehensive reference_api_conventions.md.

Delete outdated memories. If you migrated from Django to FastAPI, the Django-specific memories are not just useless – they are actively harmful, causing Claude to suggest Django patterns in a FastAPI codebase.

Audit quarterly. Once per quarter, read through MEMORY.md and ask: is each of these still true? Is each worth the token cost?

Review all memories for this project. Flag any that are outdated
or could be consolidated.

Memory Conflict Resolution¶

What happens when a memory contradicts your current instructions? Consider this scenario:

markdown
# Memory (saved 3 months ago):
---
name: Database ORM
description: Project uses SQLAlchemy for all database operations
type: project
---
Always use SQLAlchemy for database queries.

But you just migrated to Tortoise ORM. If you ask Claude to write a database query, the memory says SQLAlchemy while the actual codebase uses Tortoise. Claude handles this with a simple priority rule: current context beats stored memory. If Claude reads your models and sees Tortoise ORM, it will use Tortoise despite the memory. But this creates confusion and wasted reasoning – Claude spends thinking tokens reconciling the conflict.

The fix is proactive: update or delete outdated memories when your stack changes.

Forget the memory about SQLAlchemy. We migrated to Tortoise ORM.
Remember: this project uses Tortoise ORM with asyncpg for PostgreSQL.

Cross-Project vs. Project-Scoped Memory¶

Memory files are scoped to the project directory path. This means:

  • Memories saved while working in /projects/webapp are only loaded in /projects/webapp
  • Working in /projects/cli-tool has its own separate memory store
  • Personal preferences (like “I prefer terse output”) need to be saved in each project separately, OR stored in the global user memory

The global memory at ~/.claude/memory/ applies across all projects. Use it for universal preferences:

# Global preference (applies everywhere)
Remember globally: I prefer concise responses. Lead with the answer,
skip the preamble.

# Project preference (applies only here)
Remember: this project uses Celery for background jobs with Redis
as the broker.

The distinction matters. Your coding style preferences belong in global memory. Your project’s database choice belongs in project memory. Putting project details in global memory pollutes every session for every project.

Session Management¶

While auto-memory captures key facts, sessions capture the full conversation history. Every Claude Code session is automatically saved to a SQLite database on your machine.

Session Storage¶

Sessions are stored locally. Each session includes:

  • Complete message history (your messages and Claude’s responses)
  • All tool calls and their results
  • Files that were read
  • Metadata: start time, duration, model used, token count

Listing and Resuming Sessions¶

/sessions                    # List recent sessions with timestamps and summaries

Output looks like:

Recent sessions:
  #1  2h ago   "Fixed auth bug in login.py" (34 turns, opus)
  #2  5h ago   "Refactored payment module" (28 turns, sonnet)
  #3  1d ago   "Wrote chapter 12 of the handbook" (15 turns, opus)
  #4  2d ago   "Set up CI/CD pipeline" (42 turns, sonnet)

To continue a previous session:

/resume 1                    # Resume the auth bug session

Resuming a session loads the full conversation history back into context. This means:

  • Claude remembers everything from that session
  • The context window fills up with the old conversation
  • You pick up exactly where you left off

When to resume vs. start fresh: Resume when you are continuing the exact same task (debugging a tricky issue across multiple sittings, for example). Start fresh when the context from the old session would be noise. In practice, starting fresh with auto-memory providing background knowledge is the better choice 80% of the time.

Exporting Sessions¶

/export session-auth-debug.md    # Save conversation as readable markdown

Exported sessions are useful for:

  • Creating documentation from a debugging session
  • Sharing a Claude Code workflow with teammates
  • Archiving a particularly productive session as a reference
  • Post-mortem analysis of what went wrong

Session Search¶

When you have dozens of past sessions, finding the right one to resume matters:

/sessions --search "payment refactor"    # Search session summaries

Claude Code indexes session summaries so you can search by topic, file names mentioned, or keywords from the conversation. This is particularly useful when you are picking up work from days or weeks ago and cannot remember which session contained the relevant context.

Session Size and Resumption Costs¶

A critical detail most people overlook: resuming a session replays its entire history into your context window. A 50-turn session with heavy file reads might contain 80,000 tokens of history. Resuming it immediately consumes 80,000 tokens – 40% of your total window – before you type a single new message.

This has cost implications too. On Opus, resuming an 80K-token session costs:

80,000 tokens * $15 / 1,000,000 = $1.20 (first turn after resume)
80,000 tokens * $1.50 / 1,000,000 = $0.12 (subsequent turns, cached)

For long sessions, it is often cheaper to start fresh and re-read the specific files you need than to resume and pay for 80K tokens of mostly-irrelevant history.

Rule of thumb: Resume sessions under 20 turns. For longer sessions, start fresh and let auto-memory bridge the gap.

Session Branching¶

Claude Code supports session branching – creating a fork of the current session to try an experimental approach without contaminating your main line of work:

/branch experiment-new-auth     # Fork the session here

If the experiment works, you continue on the branch. If it fails, the original session is uncontaminated. This is conceptually similar to git branching but for conversations.

Practical use cases for session branching:

  • Risky refactoring: Branch before attempting a large change. If it breaks everything, discard the branch.
  • Alternative approaches: Branch to try approach A, then go back and try approach B, and compare results.
  • Exploratory investigation: Branch to investigate a hypothesis without cluttering the main debugging session.

Context Compaction¶

As discussed in Chapter 13, the context window is finite. Context compaction is the mechanism for freeing space in a long-running session by summarizing older messages.

Manual Compaction¶

/compact                        # Summarize everything older, keep recent turns
/compact authentication         # Summarize everything except auth-related content
/compact --keep-last 10         # Keep the last 10 turns detailed, summarize the rest

The /compact command with a focus topic is particularly powerful. When you run /compact authentication, Claude:

  1. Reads through the entire conversation history
  2. Identifies messages related to authentication
  3. Preserves those messages in full detail
  4. Summarizes everything else into a brief recap

The result is a smaller context window that retains the information you actually need.

Automatic Compaction¶

When the context window approaches its limit, Claude Code triggers automatic compaction. You will not see a notification – it happens silently between turns. The symptoms:

  • Claude seems to “forget” details from early in the session
  • Responses that should reference earlier work miss the context
  • The /context command shows a drop in usage (older messages were summarized)

Automatic compaction is a last resort. It means you have been in the session too long without managing your context. The professional approach is to compact manually or /clear before you hit the automatic threshold.

Quality Implications¶

Compaction is lossy. When older messages are summarized:

  • Specific code details are lost. Claude will not remember the exact line numbers it suggested earlier.
  • Nuanced reasoning is flattened. A 500-word explanation of why a particular approach was chosen becomes a one-line summary.
  • Tool results disappear. The contents of files Claude read earlier are no longer in context.

This is why /clear between tasks is almost always better than running one marathon session. A fresh session with relevant files re-read is higher quality than a compacted session with summarized ghosts of old context.

Strategic Compaction¶

The best time to compact is before starting a complex sub-task within a longer session:

# You've been debugging for 20 minutes, found the root cause.
# Before starting the fix:
/compact root-cause-analysis

# Now Claude has a clean context with the root cause preserved,
# and plenty of room for the implementation work.

Plan Mode¶

For complex tasks that affect multiple files or involve architectural decisions, Claude Code offers plan mode – a structured approach where Claude investigates and plans before writing any code.

Entering Plan Mode¶

/plan                           # Enter plan mode

Or use the EnterPlanMode command:

Plan the migration from SQLAlchemy to async SQLAlchemy across the project

What Happens in Plan Mode¶

In plan mode, Claude operates in read-only. It can:

  • Read files
  • Search the codebase
  • Analyze architecture
  • Ask clarifying questions

It cannot:

  • Edit files
  • Write new files
  • Run commands that modify state
  • Create commits

The output is a structured plan document:

markdown
# Migration Plan: SQLAlchemy to Async SQLAlchemy

## Scope
- 14 files affected across src/models/, src/services/, src/api/
- 3 migration scripts needed
- Test suite changes in 8 files

## Step 1: Update Dependencies
- Replace sqlalchemy with sqlalchemy[asyncio] in pyproject.toml
- Add asyncpg (async PostgreSQL driver)
- Remove psycopg2-binary

## Step 2: Convert Session Management
- File: src/db/session.py
- Change: Replace Session with AsyncSession
- Change: Replace create_engine with create_async_engine
...

## Step 3: Convert Models
[detailed per-file changes]

## Step 4: Convert Service Layer
[detailed per-file changes]

## Step 5: Update Tests
[detailed per-file changes]

## Risks
- Async context propagation in middleware (medium risk)
- Background tasks currently use sync sessions (needs Celery async support)

## Estimated effort: 3-4 hours with Claude Code assistance

Approving and Executing Plans¶

After Claude presents the plan, you review it:

Looks good, but skip step 5 for now -- I'll update tests separately.
Execute the plan.

Claude exits plan mode and begins implementing the approved steps. The plan remains in context as a reference, keeping Claude aligned with the agreed-upon approach.

When to Use Plan Mode¶

  • Multi-file changes affecting 5+ files
  • Architectural decisions where the approach matters more than the code
  • Risky operations where you want to review before any writes happen
  • Complex refactoring where the order of changes matters
  • Onboarding to a new codebase – “Plan how you would add a notification system”

Plan Storage¶

Plans are saved to .claude/plans/:

.claude/plans/
├── 2026-03-15-async-migration.md
├── 2026-03-20-auth-refactor.md
└── 2026-04-01-api-v2-design.md

These plans serve as lightweight Architecture Decision Records. They capture not just what was done, but why it was done that way.

The /clear Discipline¶

If you take one thing from this chapter, take this: /clear between unrelated tasks is the most important productivity habit in Claude Code.

Why /clear Matters¶

When you work on Task A and then switch to Task B without clearing:

  1. Context bleed. Claude’s responses to Task B are influenced by Task A’s context. It might suggest patterns from Task A, reference files from Task A, or make assumptions based on Task A’s discussion.

  2. Token waste. Every turn of Task B pays to re-read Task A’s conversation history. If Task A consumed 40K tokens of context, you are paying input costs on those 40K tokens for every single turn of Task B.

  3. Reduced working space. Those 40K tokens of Task A context reduce your available window for Task B. Complex work in Task B might trigger premature auto-compaction.

The Anti-Pattern¶

9:00 AM  Fix login bug in auth.py
9:30 AM  Write unit tests for the payment module
10:00 AM Review PR #142
10:30 AM Debug the CI pipeline failure
11:00 AM Write API documentation

Five unrelated tasks, one continuous session. By 11 AM, the context window is saturated with fragments of all five tasks. Claude is confused, expensive, and producing mediocre output.

The Pattern¶

9:00 AM  Fix login bug in auth.py
9:25 AM  /clear
9:25 AM  Write unit tests for the payment module
9:55 AM  /clear
9:55 AM  Review PR #142
10:15 AM /clear
10:15 AM Debug the CI pipeline failure
10:45 AM /clear
10:45 AM Write API documentation

Five focused sessions. Each task gets Claude’s full attention and full context window. Auto-memory carries forward anything important. Total cost is lower because each session starts with a clean, cached system prompt.

When NOT to /clear¶

Do not clear when you are still working on the same task. If you are debugging an issue, you want Claude to remember the hypotheses it explored, the files it read, and the approaches it tried. Clearing mid-task forces you to re-establish all that context.

The rule is simple: same task, keep the context. New task, clear the context.

The /btw Command¶

Sometimes you need a quick answer to a side question without derailing your current task. The /btw command handles this:

# You're deep in refactoring auth.py
/btw What does the @lru_cache decorator do with a maxsize of None?

Claude answers the side question without incorporating it into the main conversation context. The answer appears, but the follow-up turns continue as if the side question never happened.

Use /btw for:

  • Quick syntax lookups mid-task
  • “What does this error code mean?” questions
  • Checking a fact without changing Claude’s focus
  • “How do I do X in bash again?” utility questions

The /rewind Command¶

When Claude goes down a wrong path – suggests a bad approach, makes an incorrect fix, or misunderstands your intent – /rewind undoes the last exchange:

You: Refactor auth.py to use a class-based approach
Claude: [rewrites entire file as a class hierarchy]
You: /rewind
# Claude's class rewrite is removed from context
You: Refactor auth.py to extract the validation logic into a separate function
Claude: [makes the targeted change you actually wanted]

Multiple /rewind commands go further back:

/rewind          # Undo last exchange
/rewind          # Undo the one before that
/rewind          # Go back three exchanges

This is cheaper and faster than trying to correct Claude’s course with additional prompts. Each correction prompt adds to context and cost. /rewind removes the mistake entirely.

Here is how all these mechanisms work together in a typical day:

# Auto-memory loads automatically:
#   - Your preferences (terse output, type hints, ruff)
#   - Project architecture (Django, PostgreSQL, Redis)

The Interplay Between Persistence Mechanisms¶

The three mechanisms are not isolated – they work together. Understanding their interactions makes the system more powerful.

Memory Informs Session Quality¶

When you start a fresh session, auto-memory provides the baseline knowledge that would otherwise require 10-15 turns of explanation. Without memory:

Turn 1: "We use Django with PostgreSQL"
Turn 2: "Our API follows JSON:API spec"
Turn 3: "Never force-push to main"
Turn 4: "I prefer type hints and ruff"
Turn 5: NOW you can start actual work

With memory, all of that context loads automatically. Turn 1 is your actual task. Over a year of daily use, this saves hundreds of turns and significant cost.

Plans Bridge Sessions¶

A plan created in one session persists on disk. When you start a new session the next day, you can reference it:

Continue implementing the async migration plan from yesterday.
We finished steps 1-3, pick up at step 4.

Claude reads the plan file from .claude/plans/, understands the full scope, and knows exactly where you left off – even without resuming the original session. This is more efficient than resuming because:

  1. The plan file is typically 500-2,000 tokens (vs. 50K+ for a full session)
  2. Claude has a clear roadmap instead of having to re-derive intent from old messages
  3. The context window is clean for the actual implementation work

When Each Mechanism Fails¶

Auto-memory fails when the information changes frequently. Do not save volatile data (current sprint goals, this week’s deadline, today’s branch name) as memories – they become stale immediately.

Sessions fail for long-running tasks. If a task spans three days, you do not want to resume a 3-day-old session. Use plan files to capture the state, auto-memory to capture the learnings, and start fresh each day.

Plans fail for tasks that evolve significantly. If the plan said “migrate to async SQLAlchemy” but you discovered halfway through that Tortoise ORM is a better fit, the plan is now misleading. Update or replace it.

The professional approach: use all three in combination. Plans for structure, memory for knowledge, and fresh sessions for execution. Do not rely on any single mechanism.

What Most People Get Wrong¶

“I never need /clear because my sessions are short.” Even short sessions accumulate context from tool results. If your “short” session reads 10 files and runs 5 bash commands, it might consume 50K tokens. The next task starts with half its budget gone.

“I’ll just tell Claude what to remember.” While explicit memory saves work, they are not a substitute for Claude’s auto-memory. Let Claude save what it thinks is important, and supplement with explicit saves for information Claude might not recognize as worth keeping.

“Plan mode is too slow – I’ll just start coding.” For small changes, yes, skip plan mode. But for anything touching 5+ files, the ten minutes spent planning saves an hour of rework when Claude’s approach diverges from what you wanted at step 3 of 8.

“I resumed my session from last week, so Claude should know everything.” Resuming loads the conversation history, but the context window still has a limit. A 200-turn session from last week will auto-compact heavily when resumed, losing most of the detail. It is often better to start fresh and let auto-memory provide the persistent knowledge.

“More memories are better.” Each memory file adds tokens to every session’s system prompt. Twenty memory files might add 5,000 tokens to your baseline cost. Be selective. Save what is truly persistent and useful, not every detail from every debugging session.

Chapter Summary¶

  1. Three persistence mechanisms serve different purposes: auto-memory for permanent knowledge (loaded every session), sessions for conversation history (resumed on demand), and plans for structured implementation decisions (loaded on demand).

  2. Auto-memory is your most valuable persistence tool. It bridges sessions automatically, teaching Claude about your preferences, your project, and your corrections. Keep MEMORY.md under 200 lines and audit it quarterly.

  3. Session management is about discipline. /clear between tasks, /compact when context gets heavy, /rewind when Claude goes wrong, /btw for side questions. These four commands are your context control panel.

  4. Plan mode prevents expensive mistakes. For complex multi-file changes, the upfront cost of planning is always less than the cost of correcting a wrong approach mid-implementation.

  5. The persistence workflow is: auto-memory for knowledge, /clear for hygiene, /compact for maintenance, plan mode for complex work. Master this cycle and Claude Code becomes a tool that genuinely learns from working with you.
