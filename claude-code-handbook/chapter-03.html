Chapter 3: Architecture Deep Dive – How Claude Code Actually Works¶ ~21 min read

You are about to write your first hook. It fires on PreToolUse and checks the command against a blocklist. But you are unsure: does the hook see the command before permission checks or after? Does it get the raw command string or a JSON object? What exit code blocks execution?

These are not trivia questions. They are the difference between a hook that works and one that silently fails. And you cannot answer them without understanding the architecture.

This chapter is the map. Every configuration decision in Part II – how you structure CLAUDE.md, what permissions to set, which lifecycle events to hook – flows from understanding what happens between your keystroke and Claude Code’s response.

The Agentic Loop¶

This is the single most important concept in Claude Code. Everything else is implementation detail.

When you type a prompt and press Enter, Claude Code enters a loop:

Let’s walk through each step with concrete detail.

Step 1: System Prompt Assembly¶

Before the LLM sees your message, Claude Code builds a system prompt. This is not a simple template – it is a dynamically assembled document that can exceed 10,000 tokens. Here is what gets included, in order:

  1. Base system prompt. Claude Code’s core instructions: how to use tools, how to format output, general behavior guidelines. You do not control this.

  2. CLAUDE.md content. Hierarchically merged. If you have a ~/CLAUDE.md (global), a ~/projects/CLAUDE.md (workspace), and ~/projects/my-app/CLAUDE.md (project), all three are included with project-level taking highest priority.

  3. Matching rules. Claude Code checks which files are “in scope” (recently read, recently edited, mentioned in the conversation) and loads rules whose paths: frontmatter matches those files.

  4. Active skill instructions. If a skill is triggered (by /command or auto-detection), its SKILL.md content is injected.

  5. Git context. Current branch, recent commit messages, staged changes, dirty files. This helps Claude Code understand the project’s current state.

  6. Directory listing. A snapshot of the project’s file structure so Claude Code knows what files exist without having to call Glob first.

  7. Memory context. Auto-memory entries from ~/.claude/projects/<hash>/MEMORY.md – things Claude Code has learned about your project in previous sessions.

  8. Tool definitions. JSON schema definitions for every available tool: parameters, descriptions, constraints.

Here is an approximate token budget for a typical system prompt:

Component Typical Tokens % of Budget
Base system prompt 3,000-5,000 15-25%
CLAUDE.md + rules 500-2,000 2-10%
Active skill 200-800 1-4%
Git context 200-500 1-2%
Directory listing 200-1,000 1-5%
Memory 100-500 0.5-2%
Tool definitions 3,000-5,000 15-25%
Total system prompt 7,000-15,000 35-75%

The remaining context window is for your conversation (messages, tool results, thinking).

Why this matters for configuration: A 200-line CLAUDE.md consumes roughly 600 tokens on every single API call. Those tokens are paid for repeatedly, turn after turn. A 42-line CLAUDE.md with @imports to rules that load only when relevant is dramatically more efficient. As we’ll see in Chapter 4, brevity in CLAUDE.md is not laziness – it is engineering.

Step 2: LLM API Call¶

With the system prompt assembled, Claude Code sends the full conversation (system + message history + new user message) to the Claude API. The response streams back as Server-Sent Events (SSE).

The streaming format looks like this on the wire:

event: content_block_start
data: {"type":"content_block_start","index":0,"content_block":{"type":"thinking","thinking":""}}

event: content_block_delta
data: {"type":"content_block_delta","index":0,"delta":{"type":"thinking_delta","thinking":"Let me analyze..."}}

event: content_block_stop
data: {"type":"content_block_stop","index":0}

event: content_block_start
data: {"type":"content_block_start","index":1,"content_block":{"type":"text","text":""}}

event: content_block_delta
data: {"type":"content_block_delta","index":1,"delta":{"type":"text_delta","text":"I'll read the file"}}

event: content_block_start
data: {"type":"content_block_start","index":2,"content_block":{"type":"tool_use","id":"toolu_01...","name":"Read","input":{}}}

event: content_block_delta
data: {"type":"content_block_delta","index":2,"delta":{"type":"input_json_delta","partial_json":"{\"file_path\":\"/src/main.py\"}"}}

Each response can contain multiple content blocks:

Block Type Purpose
thinking Extended thinking (reasoning, planning) – visible with /thinking or when alwaysThinkingEnabled is true
text Human-readable text response shown to the user
tool_use A tool call that Claude Code will execute

A single response might contain: one thinking block (internal reasoning), one text block (“Let me investigate…”), and three tool_use blocks (Read + Grep + Bash). Claude Code processes all of them.

Step 3: Response Parsing¶

Claude Code parses the streamed content blocks and makes a decision:

  • If stop_reason is end_turn: The response is complete. Display it and wait for the next user message.
  • If stop_reason is tool_use: There are tool calls to execute. Enter the tool execution pipeline.

This is what makes Claude Code agentic. The LLM itself decides whether to respond with text (done) or request tool execution (continue working). Claude Code does not make this decision – the model does.

Step 4: Permission Check¶

Before executing any tool call, Claude Code evaluates permissions. The cascade is:

deny rules → ask rules → allow rules → default behavior

First match wins. Here is how it evaluates a Bash(git push origin main) call:

  1. Check deny rules. Is there a deny pattern matching Bash(git push *)? If yes, block immediately. Claude Code receives an error message that the tool was denied.
  2. Check ask rules. Is there an explicit ask pattern? If yes, prompt the user for approval.
  3. Check allow rules. Is there an allow pattern matching Bash(git *)? If yes, execute without asking.
  4. Default. If no rule matches, the behavior depends on the permission mode (Default asks for most things, AllowAll permits everything).

Permission patterns use simple glob matching:

json
{
  "permissions": {
    "allow": [
      "Bash(git *)",           // Any git command
      "Bash(python -m pytest *)", // pytest only
      "Read(*)"                // Read any file
    ],
    "deny": [
      "Read(./.env)",          // Never read .env
      "Bash(rm -rf *)"        // Never rm -rf
    ]
  }
}

The pattern format is ToolName(argument_pattern). For Bash, the argument is the command string. For Read, it is the file path. For Edit, it is the file path.

Practical implication: If you allow Bash(git *), Claude Code can run git push --force without asking. If that concerns you, be more specific:

json
"allow": [
  "Bash(git diff *)",
  "Bash(git status *)",
  "Bash(git log *)",
  "Bash(git add *)",
  "Bash(git commit *)"
]

And explicitly deny the dangerous ones:

json
"deny": [
  "Bash(git push --force *)",
  "Bash(git push * --force *)",
  "Bash(git reset --hard *)"
]

Step 5: PreToolUse Hooks¶

If the tool passes permission checks, PreToolUse hooks fire. These are shell scripts that receive the tool call details via stdin as JSON:

json
{
  "tool_name": "Bash",
  "tool_input": {
    "command": "rm -rf ./tmp/cache"
  }
}

The hook script reads this JSON, applies its logic, and exits:

Exit Code Meaning
0 Allow the tool to proceed
2 Block the tool. Stderr output becomes feedback to Claude Code
Other Non-blocking error (logged, execution continues)

Here is a real PreToolUse hook that blocks destructive commands:

bash
#!/usr/bin/env bash
set -euo pipefail

INPUT=$(cat)
TOOL_NAME=$(echo "${INPUT}" | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_name',''))" 2>/dev/null || echo "")

if [[ "${TOOL_NAME}" != "Bash" ]]; then
    exit 0
fi

COMMAND=$(echo "${INPUT}" | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_input',{}).get('command',''))" 2>/dev/null || echo "")

# Destructive patterns
BLOCKED_PATTERNS=(
    "rm -rf /"
    "rm -rf ~"
    "rm -rf \."
    "DROP TABLE"
    "DROP DATABASE"
    "git push --force main"
    "git push --force origin main"
    "git push origin main --force"
)

for pattern in "${BLOCKED_PATTERNS[@]}"; do
    if [[ "${COMMAND}" == *"${pattern}"* ]]; then
        cat >&2 << HEREDOC
{"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"deny","permissionDecisionReason":"BLOCKED: Command matches destructive pattern '${pattern}'. This action could cause irreversible damage."}}
HEREDOC
        exit 2
    fi
done

exit 0

Hooks are wired in settings.json:

json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "./.claude/hooks/block-destructive.sh",
            "timeout": 5000,
            "statusMessage": "Safety check..."
          }
        ]
      }
    ]
  }
}

The matcher field determines which tool triggers the hook. "Bash" means only Bash tool calls fire this hook. An empty matcher "" fires on every tool call.

Step 6: Tool Execution¶

The tool executes with the provided input and a configured timeout. Here is what happens for each major tool:

Bash: - Spawns a child process with the command - Captures stdout and stderr - Respects the timeout (default 120 seconds, configurable up to 600 seconds) - Working directory is the project root - Environment variables from settings.json "env" are applied

Read: - Opens the file, reads content - Returns content with line numbers (cat -n format) - For large files: truncates with a warning - For binary files: returns a description or renders (images)

Edit: - Performs exact string replacement in the file - The old_string must match exactly one location - Returns the result (success or failure + reason)

Write: - Overwrites the file entirely - Creates the file if it does not exist - Creates parent directories if needed

Grep: - Runs ripgrep under the hood - Returns matching lines, file paths, or counts depending on output mode - Respects .gitignore patterns

Agent: - Spawns a child Claude Code process (a subagent) - The subagent has its own conversation context - Can run in the same directory or an isolated worktree - Results are returned to the parent conversation

Step 7: PostToolUse Hooks¶

After the tool executes successfully, PostToolUse hooks fire. These cannot block (the action already happened), but they can perform follow-up actions.

The canonical example: auto-linting Python files after every edit.

bash
#!/usr/bin/env bash
set -euo pipefail

INPUT=$(cat)
TOOL_NAME=$(echo "${INPUT}" | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_name',''))" 2>/dev/null || echo "")

if [[ "${TOOL_NAME}" != "Edit" && "${TOOL_NAME}" != "Write" ]]; then
    exit 0
fi

FILE_PATH=$(echo "${INPUT}" | python3 -c "import sys,json; d=json.load(sys.stdin).get('tool_input',{}); print(d.get('file_path',d.get('path','')))" 2>/dev/null || echo "")

if [[ "${FILE_PATH}" == *.py ]]; then
    ruff check --fix "${FILE_PATH}" 2>/dev/null || true
    ruff format "${FILE_PATH}" 2>/dev/null || true
fi

exit 0

Wired as:

json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "./.claude/hooks/auto-lint-python.sh",
            "timeout": 10000,
            "statusMessage": "Auto-linting..."
          }
        ]
      }
    ]
  }
}

Step 8: Result Truncation¶

Tool results can be large. A Grep across a large codebase might return thousands of lines. A Read on a 10,000-line file returns all of it. Claude Code handles this:

  1. Result size check. If the result exceeds the truncation threshold (roughly 50K characters), it is truncated.
  2. Truncation message. A note is appended: “Output truncated. N lines omitted.”
  3. The truncated result is what Claude sees. The LLM works with incomplete information and may need to make additional, more targeted tool calls.

Why this matters: If you configure a Grep with no head_limit, it might return 10,000 matches. Claude Code truncates to fit context, possibly losing the relevant result. Always use targeted searches:

// Bad: returns everything
Grep(pattern="error", path="./")

// Good: targeted, limited
Grep(pattern="AuthenticationError", path="./src/auth/", head_limit=20)

Step 9: Append and Loop¶

The tool result is appended to the conversation as a tool_result message, and the loop returns to Step 2 (LLM API call). The LLM now has:

  • The original system prompt
  • The user’s message
  • Its previous response (text + tool calls)
  • The tool results

It decides what to do next: call more tools (loop continues) or respond with text (loop ends).

A typical task might loop 3-8 times:

Turn Action
1 User: “Fix the auth bug”
2 Claude: Grep for auth-related files
3 Claude: Read the main auth file
4 Claude: Read the test file
5 Claude: Edit the auth file (fix the bug)
6 Claude: Bash(pytest tests/test_auth.py)
7 Claude: Text response (“Fixed the bug, here’s what I changed…”)

Max turns: Claude Code has a configurable turn limit to prevent runaway loops. If the model keeps calling tools without converging, the loop stops and the user is warned. The default is generous (around 100 turns) but can be controlled with CLAUDE_CODE_MAX_TURNS.

The Context Window¶

The context window is the total amount of text (measured in tokens) that can be in a single conversation. For Claude models, this is approximately 200K tokens.

What Fills the Context¶

┌─────────────────────────────────────────────────┐
│              200K Token Window                   │
│                                                  │
│  ┌──────────────────────────────────────────┐    │
│  │ System Prompt: ~10K tokens               │    │
│  │ (CLAUDE.md + rules + tools + git + memory)│    │
│  └──────────────────────────────────────────┘    │
│                                                  │
│  ┌──────────────────────────────────────────┐    │
│  │ Conversation History: ~150K tokens       │    │
│  │ (messages + tool calls + tool results)   │    │
│  │                                          │    │
│  │ Turn 1: User message (50 tokens)         │    │
│  │ Turn 1: Claude response (200 tokens)     │    │
│  │ Turn 1: Tool result - Read (3000 tokens) │    │
│  │ Turn 2: Claude response (150 tokens)     │    │
│  │ Turn 2: Tool result - Grep (500 tokens)  │    │
│  │ ...                                      │    │
│  └──────────────────────────────────────────┘    │
│                                                  │
│  ┌──────────────────────────────────────────┐    │
│  │ Reserve for Response: ~40K tokens        │    │
│  │ (Claude needs room to generate output)   │    │
│  └──────────────────────────────────────────┘    │
│                                                  │
└─────────────────────────────────────────────────┘

Approximate budget:

Component Tokens Percentage
System prompt 7K-15K 4-8%
Conversation history 100K-150K 50-75%
Response reserve 30K-50K 15-25%

The conversation history is where tokens accumulate fast. Every Read of a 500-line Python file adds roughly 2,000 tokens. Every Grep result adds hundreds. Every Bash output adds the full stdout. After 20-30 tool calls, you have consumed tens of thousands of tokens in tool results alone.

Context Compaction¶

When the context approaches the window limit, Claude Code performs compaction. This is an automatic process that summarizes older parts of the conversation to free up space.

Here is what compaction does:

  1. Identifies old turns. Turns from early in the conversation that are unlikely to be relevant to the current task.
  2. Summarizes them. Replaces verbose tool results and messages with a brief summary: “Earlier, I read src/auth/login.py and found a token validation bug on line 47.”
  3. Keeps recent turns intact. The last N turns remain uncompacted to preserve current context.
  4. Preserves key information. File paths, error messages, and decisions are retained even in compacted form.

You can also trigger compaction manually:

/compact               # Compact everything
/compact security      # Compact but preserve security-related context

The /clear discipline: Between unrelated tasks, use /clear to reset the conversation entirely. This is more effective than compaction for task switching because it starts with a fresh context window:

> /clear
Context cleared. Starting fresh.

> [new task here]

If you are fixing a bug in auth, then switching to writing documentation for the API, /clear between them. The auth context is irrelevant to the docs task and wastes tokens.

The Permission System in Depth¶

Permissions have four layers, evaluated in order:

Managed → User → Project → Local

The Four Layers¶

Managed (enterprise): Set by your organization’s IT. You cannot override these. They are delivered by the enterprise deployment and typically deny dangerous operations globally.

User (~/.claude/settings.json): Your personal global settings. Apply to every project.

Project (.claude/settings.json): Committed to git. Apply to everyone on the team.

Local (.claude/settings.local.json): Personal overrides for this project. Gitignored. Apply only to you.

Permission Modes¶

Claude Code supports several permission modes that set the baseline behavior:

Mode Behavior
Default Allow known-safe tools (Read, Glob, Grep). Ask for Bash, Write, Edit
AllowAll Allow everything without asking (use with hooks for safety)
Plan Claude Code can only read and analyze, never modify files or run commands
DenyAll Block all tool use (for testing system prompt assembly)

In practice, professional setups use the Default mode with explicit allow and deny lists:

json
{
  "permissions": {
    "allow": [
      "Bash(git diff *)",
      "Bash(git status)",
      "Bash(git log *)",
      "Bash(python -m pytest *)",
      "Bash(ruff *)",
      "Read(*)",
      "Glob(*)",
      "Grep(*)"
    ],
    "deny": [
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./**/*.pem)",
      "Bash(rm -rf *)",
      "Bash(git push --force *)"
    ]
  }
}

Pattern Matching Details¶

Permission patterns use glob-style matching:

Pattern Matches
Bash(git *) Any bash command starting with git
Read(./.env) Exactly the file ./.env
Read(./.env.*) ./.env.local, ./.env.production, etc.
Read(./**/*.pem) Any .pem file in any subdirectory
Bash(*) Any bash command (use in settings.local.json for personal use)

The * wildcard matches any characters. Paths use ./ prefix for project-relative paths.

Session Lifecycle¶

Session Creation¶

Every claude invocation creates a session. Sessions are identified by a unique ID and stored in SQLite:

~/.claude/sessions/
└── sessions.db

The session stores: - Session ID and creation timestamp - Full conversation history (messages, tool calls, results) - System prompt at creation time - Model used - Token consumption per turn

Resuming Sessions¶

You can resume a previous session:

bash
# Resume the most recent session
claude --resume

# Resume a specific session
claude --resume <session-id>

# List recent sessions
claude sessions list

Resuming restores the full conversation context, so you can continue exactly where you left off. This is useful when you need to step away and return to a complex task.

Non-Interactive Sessions¶

For automation and scripting, run Claude Code non-interactively:

bash
# Single prompt
claude -p "Explain the architecture of src/auth/"

# With specific output format
claude -p "List all TODO comments" --output-format json

# Piped input
cat error.log | claude -p "What caused this error?" --stdin

Non-interactive sessions create a session record just like interactive ones, so they appear in session history.

Subagent Architecture¶

When Claude Code calls the Agent tool, it spawns a child process – a subagent. Understanding how subagents work is critical for multi-agent patterns (Chapter 12 and Chapter 21).

How Agent Spawning Works¶

┌─────────────────────────────────────────┐
│ Parent Claude Code Session              │
│                                         │
│  User: "Review code quality AND security│
│         of src/auth/"                   │
│                                         │
│  ┌───────────────┐  ┌────────────────┐  │
│  │ Agent Tool     │  │ Agent Tool     │  │
│  │ call #1        │  │ call #2        │  │
│  │                │  │                │  │
│  │ Task: "Review  │  │ Task: "Review  │  │
│  │ code quality"  │  │ security"      │  │
│  └───────┬───────┘  └───────┬────────┘  │
│          │                  │           │
└──────────┼──────────────────┼───────────┘
           │                  │
     ┌─────▼─────┐     ┌─────▼──────┐
     │ Subagent 1 │     │ Subagent 2  │
     │ (Sonnet)   │     │ (Opus)      │
     │            │     │             │
     │ Tools:     │     │ Tools:      │
     │ Read, Grep │     │ Read, Grep  │
     │ Glob       │     │ Glob, Bash  │
     │            │     │             │
     │ Returns:   │     │ Returns:    │
     │ Findings   │     │ CWE table   │
     └────────────┘     └─────────────┘

Each subagent: - Gets its own conversation context. It does not see the parent’s full history. - Receives a task description. The parent passes a prompt explaining what to do. - Has restricted tools. The parent can limit which tools the subagent can use. - Returns a result. The subagent’s final text response is returned as a tool_result to the parent. - Can use a different model. Defined by agent configuration in .claude/agents/.

Isolation Modes¶

Subagents can run in two isolation modes:

Shared (default): The subagent operates in the same project directory. It can read and edit the same files as the parent. This is fine for read-only tasks (reviews, analysis) but risky for tasks that make changes (two agents editing the same file simultaneously).

Worktree: The subagent operates in an isolated git worktree. It has its own copy of the codebase and can make changes without affecting the parent’s working directory. Changes are merged back when the subagent completes. This is safer for parallel modifications.

yaml
# In .claude/agents/security-reviewer.md
---
name: security-reviewer
model: opus
isolation: worktree
---

Agent Communication¶

Subagents do not communicate with each other directly. They report back to the parent, which synthesizes results. The pattern is:

  1. Parent spawns Agent A and Agent B in parallel
  2. Agent A returns code quality findings
  3. Agent B returns security findings
  4. Parent combines both into a unified review

This is a fan-out/fan-in pattern. As we’ll see in Chapter 21, you can build sophisticated multi-agent workflows by configuring different agents for different aspects of a task.

Streaming Architecture¶

Claude Code uses streaming to display responses in real-time rather than waiting for the complete response. Here is how the streaming pipeline works:

Claude API  ──SSE──►  Stream Parser  ──►  Content Assembler  ──►  Terminal Renderer
                           │                     │
                           │                     ├── Text blocks → display immediately
                           │                     ├── Thinking blocks → display if enabled
                           │                     └── Tool_use blocks → queue for execution
                           │
                           └── Connection monitoring (watchdog)

Server-Sent Events (SSE): The Claude API streams response tokens as SSE events. Each event contains a delta (partial content). Claude Code reassembles these deltas into complete content blocks.

Content block types:

Block Processing
thinking Displayed in real-time if thinking is enabled. Otherwise buffered silently
text Displayed character-by-character as it arrives
tool_use Accumulated until complete, then sent to the tool execution pipeline

Reconnection: If the stream drops (network hiccup, server timeout), Claude Code can detect the break and either retry the request or inform the user. The CLAUDE_ENABLE_STREAM_WATCHDOG environment variable enables a watchdog that monitors for stalled streams.

Why streaming matters for configuration: When Claude Code streams a text response, you see it in real-time. But when it streams a tool_use block, you see a status indicator (“Reading file…” or “Running command…”) until the tool completes. If your hooks add latency (e.g., a 5-second timeout on every Bash command), that latency is visible to the user as a delay. Keep hooks fast.

All Hook Lifecycle Events¶

We covered PreToolUse and PostToolUse in detail. Here is the complete set of lifecycle events you can hook into:

Event When Can Block? Use Case
SessionStart Session begins No Set environment variables, load context
PreToolUse Before any tool executes Yes (exit 2) Block dangerous commands, validate inputs
PostToolUse After tool succeeds No Auto-lint, log actions, notify
Notification Alert event occurs No Desktop notifications, Slack alerts
UserPromptSubmit Before user prompt is processed Yes Input validation, prompt rewriting
Stop Before Claude finishes responding Yes Quality gates, output validation
SubagentStop Before subagent returns Yes Verify subagent output quality
TaskCompleted After a task completes No Cleanup, reporting
FileChanged File system change detected No Trigger rebuilds, update caches

The most commonly used events are PreToolUse (safety), PostToolUse (quality), and Notification (awareness). The others are available for advanced workflows that we’ll build in Chapter 8.

Hook JSON Protocol¶

All hooks receive JSON on stdin. The structure varies by event:

PreToolUse / PostToolUse:

json
{
  "tool_name": "Bash",
  "tool_input": {
    "command": "git commit -m 'fix: auth bug'"
  }
}

SessionStart:

json
{
  "session_id": "sess_abc123",
  "project_path": "/Users/you/projects/my-app",
  "model": "claude-sonnet-4-20250514"
}

Notification:

json
{
  "message": "Task completed: fixed 3 bugs in src/auth/",
  "level": "info"
}

Hooks return output on stderr (for feedback to Claude Code) and use exit codes for control flow.

Performance Characteristics¶

Understanding where time goes helps you optimize your workflow.

Anatomy of a Typical Turn¶

User presses Enter
    │
    ├── System prompt assembly: ~50ms
    │   (read CLAUDE.md, match rules, build tool schemas)
    │
    ├── Network to API: ~100-300ms
    │   (depends on distance to Anthropic data center)
    │
    ├── First token latency: ~500-2000ms
    │   (LLM "thinking time" before first output)
    │   Opus: ~1500ms, Sonnet: ~700ms, Haiku: ~300ms
    │
    ├── Token generation: ~50 tokens/sec (Opus), ~100 tokens/sec (Sonnet)
    │   (a 500-token response takes ~5-10 seconds)
    │
    ├── Tool execution: varies
    │   Read file: ~10ms
    │   Grep codebase: ~50-200ms
    │   Bash command: ~100ms-120s (depends on command)
    │   Agent spawn: ~3-10s (new LLM call)
    │
    ├── Hook execution: ~50-200ms per hook
    │   (PreToolUse + PostToolUse per tool call)
    │
    └── Rendering: ~10ms
        (format and display output)

Where Time Goes in a Bug Fix¶

A typical bug fix session (6 turns, 12 tool calls):

Phase Time %
LLM thinking (6 API calls) 15s 38%
Token generation (6 responses) 12s 30%
Tool execution (12 calls) 8s 20%
Network overhead 3s 8%
Hooks (12 Pre + 12 Post) 1.5s 4%
Total ~40s 100%

LLM thinking and token generation dominate. This is why model selection matters: Opus takes 2x longer per turn than Sonnet. For a bug fix that requires 6 turns, that is the difference between 40 seconds (Sonnet) and 70 seconds (Opus).

Cost Per Operation¶

Operation Input Tokens Output Tokens Cost (Sonnet) Cost (Opus)
Read a 500-line file ~2,000 ~100 $0.007 $0.037
Grep search ~500 ~200 $0.004 $0.022
Code review (1 file) ~5,000 ~1,000 $0.030 $0.150
Bug fix (6 turns) ~30,000 ~5,000 $0.165 $0.825
Architecture analysis ~50,000 ~10,000 $0.300 $1.500

These are approximate. Actual costs depend on file sizes, conversation length, and how many turns the model takes.

Putting It Together: A Complete Request Trace¶

Let’s trace a complete request through the architecture. You type:

> Fix the off-by-one error in src/parser.py line 42

Turn 1:

  1. System prompt assembly (~50ms): CLAUDE.md loaded (42 lines, ~150 tokens). Python rule matched (parser.py is *.py). Git context: on branch fix/parser-bug, 1 modified file.

  2. API call (~700ms first token): Full context sent. System prompt (~8K tokens) + user message (~30 tokens).

  3. Response: Claude returns thinking (“Let me read the file to understand the bug”) + tool_use(Read, file_path=”/src/parser.py”).

  4. Permission check (~1ms): Read(*) is in the allow list. Allowed.

  5. PreToolUse hooks (~0ms): No PreToolUse hooks configured for Read.

  6. Tool execution (~10ms): File read, 200 lines, ~800 tokens.

  7. PostToolUse hooks (~0ms): No PostToolUse hooks for Read.

  8. Result appended: Conversation now has system + user message + Claude response + file content.

Turn 2:

  1. API call (~700ms): Claude sees the file content, identifies the bug, returns thinking + text(“Found the issue…”) + tool_use(Edit, old_string=”for i in range(len(items))”, new_string=”for i in range(len(items) - 1)”).

  2. Permission check: Edit is allowed by default.

  3. PreToolUse: No hooks for Edit.

  4. Edit execution (~5ms): String replacement successful.

  5. PostToolUse (~200ms): auto-lint-python.sh fires, runs ruff check --fix and ruff format on parser.py.

Turn 3:

  1. API call: Claude decides to verify the fix, returns tool_use(Bash, command=”python -m pytest tests/test_parser.py -x”).

  2. Permission check: Bash(python -m pytest *) is in the allow list.

  3. PreToolUse (~100ms): block-destructive.sh fires, checks “python -m pytest…” against blocklist. No match. Exit 0.

  4. Bash execution (~3s): Tests run. 5 passed, 0 failed.

  5. PostToolUse: No hooks for Bash.

Turn 4:

  1. API call: Claude sees tests passed, returns text only: “Fixed the off-by-one error in src/parser.py:42. The loop was iterating one element too far. All 5 tests pass.”

  2. stop_reason: end_turn: Loop ends.

Total time: ~8 seconds. Total cost (Sonnet): ~$0.05. Total tool calls: 3.

That is the architecture in action. Every configuration choice – the Python rule that loaded, the permission that allowed the test command, the hook that checked for destructive patterns, the auto-lint that ran after the edit – served a purpose. None of it required your intervention during the task.

Key Takeaways¶

  1. The agentic loop is the core. Claude Code repeatedly calls the LLM, executes tools, and feeds results back until the task is complete. Understanding this loop is understanding Claude Code.

  2. System prompt assembly is expensive and dynamic. CLAUDE.md, rules, skills, git context, memory, and tool schemas all contribute. Keep CLAUDE.md lean and use path-scoped rules to control what loads.

  3. Permissions cascade: deny beats ask beats allow. First match wins. Be specific in allow patterns and explicit in deny patterns. The four-layer cascade (managed, user, project, local) gives you team-safe defaults with personal flexibility.

  4. Hooks are your automation layer. PreToolUse blocks dangerous actions (exit code 2). PostToolUse automates follow-up (linting, notifications). Both receive JSON on stdin and must be fast (under 5 seconds).

  5. Context is finite and expensive. Every tool result consumes tokens. Every turn adds to the conversation. Use /clear between tasks, target searches precisely, and let compaction handle long sessions.

  6. Model selection directly impacts speed and cost. Opus is 2x slower and 5x more expensive than Sonnet. Use it where depth matters (security, architecture). Use Sonnet for everything else. Configure per-skill overrides so the choice is automatic.
