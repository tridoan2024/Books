Chapter 24: MCP Servers — Extending Claude Code’s Reach¶ ~17 min read

Claude Code ships with a powerful set of built-in tools: file reading, code search, bash execution, web fetching. For most development tasks, these are sufficient. But the moment you need to query your production database, post a message to Slack, create a Jira ticket, or interact with any system that requires authentication and domain-specific operations, you hit a wall.

Model Context Protocol (MCP) is Anthropic’s answer to that wall. It’s an open protocol that lets you connect Claude Code to any external system through a standardized interface. Think of MCP as USB for AI — a universal plug that works with any capability, from any provider, in any language.

This chapter covers everything from understanding the protocol to building production MCP servers to integrating the existing ecosystem into your Claude Code workflow.

What MCP Is¶

MCP is a JSON-RPC-based protocol that defines how an AI model (the client) communicates with external systems (the servers). The client sends requests (“call this tool”, “read this resource”), and the server responds with results.

The core architecture:

┌─────────────┐    JSON-RPC over stdio    ┌─────────────────┐
│ Claude Code  │ ◄──────────────────────► │   MCP Server     │
│ (MCP Client) │    stdin/stdout          │ (Your Service)   │
└─────────────┘                           └────────┬────────┘
                                                   │
                                          ┌────────▼────────┐
                                          │ External System  │
                                          │ (DB, API, etc.)  │
                                          └─────────────────┘

Claude Code is already an MCP client. When you register an MCP server in your settings, Claude Code spawns the server process, sends tool calls to it via stdin, and reads results from stdout. From Claude’s perspective, MCP tools look and work exactly like built-in tools.

Why MCP Matters¶

Without MCP, accessing external systems means writing bash commands:

bash
# Query a database
psql -h localhost -U app -d mydb -c "SELECT * FROM users WHERE active = true"

# Post to Slack
curl -X POST https://slack.com/api/chat.postMessage \
  -H "Authorization: Bearer $SLACK_TOKEN" \
  -d '{"channel":"#deploys","text":"Deploy complete"}'

This works, but it’s fragile, insecure (credentials in command strings), and limited (Claude has to parse text output). MCP provides:

  1. Structured input/output — JSON parameters in, JSON results out. No text parsing.
  2. Authentication isolation — Credentials stay in the server process, never in Claude’s context.
  3. Type safety — Each tool declares its parameter schema. Claude knows what to pass.
  4. Discoverability — Claude can list available tools and their descriptions, just like built-in tools.
  5. Reusability — Build a server once, use it in every project.

MCP Transport: stdio¶

MCP servers communicate via stdio (standard input/output). This is the simplest possible transport:

  • Claude Code spawns the server as a child process
  • Sends JSON-RPC messages to the server’s stdin
  • Reads JSON-RPC responses from the server’s stdout
  • Server’s stderr is used for logging (not protocol messages)

This design means MCP servers can be written in any language that can read stdin and write stdout — Python, TypeScript, Go, Rust, shell scripts, anything.

The protocol uses JSON-RPC 2.0 message format:

json
// Client → Server (tool call)
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "query_database",
    "arguments": {
      "sql": "SELECT count(*) FROM orders WHERE status = 'pending'"
    }
  }
}

// Server → Client (result)
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "{ \"count\": 42 }"
      }
    ]
  }
}

Registering MCP Servers¶

MCP servers are registered in your Claude Code settings. There are two places to register them:

Project-Level Registration (.claude/settings.json)¶

For servers that are project-specific — your project’s database, your project’s API:

json
{
  "mcpServers": {
    "project-db": {
      "command": "python3",
      "args": ["mcp_servers/database.py"],
      "env": {
        "DATABASE_URL": "postgresql://app:password@localhost:5432/mydb"
      }
    }
  }
}

User-Level Registration (~/.claude/settings.json)¶

For servers you want available across all projects — Slack, Jira, personal tools:

json
{
  "mcpServers": {
    "slack": {
      "command": "npx",
      "args": ["-y", "@anthropic/mcp-server-slack"],
      "env": {
        "SLACK_BOT_TOKEN": "${SLACK_BOT_TOKEN}"
      }
    },
    "memory": {
      "command": "python3",
      "args": ["/Users/you/tools/mcp-memory/server.py"]
    }
  }
}

Registration Fields¶

Field Required Description
command Yes The executable to run
args No Command-line arguments
env No Environment variables passed to the server process
cwd No Working directory for the server process

Environment variables support ${VAR_NAME} syntax to reference your shell environment variables. This keeps secrets out of the config file:

json
{
  "env": {
    "API_KEY": "${MY_API_KEY}"
  }
}

The server process inherits the value of MY_API_KEY from your shell environment.

How MCP Tools Appear in Claude Code¶

Once registered, MCP server tools appear alongside built-in tools with a namespaced format:

mcp__servername__toolname

For example, if you register a server named project-db that exposes a tool called query, Claude sees it as mcp__project-db__query. Claude can use it just like Bash or Read:

Claude: I'll query the database to check the order count.
[Uses tool: mcp__project-db__query with arguments: {"sql": "SELECT count(*) FROM orders"}]
Result: {"count": 1547}

Permission rules in settings.json apply to MCP tools the same way they apply to built-in tools:

json
{
  "permissions": {
    "allow": [
      "mcp__project-db__query(*)"
    ],
    "deny": [
      "mcp__project-db__execute_ddl(*)"
    ]
  }
}

Building an MCP Server from Scratch (Python)¶

Let’s build a complete, working MCP server that provides database query capabilities. This server connects to a SQLite database and exposes tools for querying and listing tables.

The Complete Server¶

python
#!/usr/bin/env python3
"""MCP server for SQLite database access."""

from __future__ import annotations

import json
import sqlite3
import sys
from pathlib import Path
from typing import Any


def read_message() -> dict[str, Any] | None:
    """Read a JSON-RPC message from stdin."""
    line = sys.stdin.readline()
    if not line:
        return None
    return json.loads(line.strip())


def write_message(message: dict[str, Any]) -> None:
    """Write a JSON-RPC message to stdout."""
    sys.stdout.write(json.dumps(message) + "\n")
    sys.stdout.flush()


def make_response(request_id: int | str, result: Any) -> dict[str, Any]:
    """Create a JSON-RPC success response."""
    return {
        "jsonrpc": "2.0",
        "id": request_id,
        "result": result,
    }


def make_error(
    request_id: int | str, code: int, message: str
) -> dict[str, Any]:
    """Create a JSON-RPC error response."""
    return {
        "jsonrpc": "2.0",
        "id": request_id,
        "error": {"code": code, "message": message},
    }


# --- Tool Definitions ---

TOOLS = [
    {
        "name": "query",
        "description": (
            "Execute a read-only SQL query against the database. "
            "Returns results as a list of row objects. "
            "Only SELECT statements are allowed."
        ),
        "inputSchema": {
            "type": "object",
            "properties": {
                "sql": {
                    "type": "string",
                    "description": "The SQL SELECT query to execute",
                },
                "params": {
                    "type": "array",
                    "items": {"type": "string"},
                    "description": "Optional query parameters for parameterized queries",
                    "default": [],
                },
            },
            "required": ["sql"],
        },
    },
    {
        "name": "list_tables",
        "description": (
            "List all tables in the database with their column names and types."
        ),
        "inputSchema": {
            "type": "object",
            "properties": {},
        },
    },
    {
        "name": "describe_table",
        "description": "Show the schema for a specific table.",
        "inputSchema": {
            "type": "object",
            "properties": {
                "table_name": {
                    "type": "string",
                    "description": "Name of the table to describe",
                },
            },
            "required": ["table_name"],
        },
    },
]


# --- Tool Implementations ---

class DatabaseServer:
    """MCP server providing SQLite database access."""

    def __init__(self, db_path: str) -> None:
        self.db_path = db_path
        self.conn: sqlite3.Connection | None = None

    def _get_connection(self) -> sqlite3.Connection:
        if self.conn is None:
            self.conn = sqlite3.connect(self.db_path)
            self.conn.row_factory = sqlite3.Row
        return self.conn

    def handle_query(self, arguments: dict[str, Any]) -> str:
        """Execute a read-only SQL query."""
        sql = arguments["sql"].strip()
        params = arguments.get("params", [])

        # Safety: only allow SELECT statements
        if not sql.upper().startswith("SELECT"):
            return json.dumps({"error": "Only SELECT queries are allowed"})

        conn = self._get_connection()
        try:
            cursor = conn.execute(sql, params)
            columns = [desc[0] for desc in cursor.description]
            rows = [dict(zip(columns, row)) for row in cursor.fetchall()]
            return json.dumps({"columns": columns, "rows": rows, "count": len(rows)})
        except sqlite3.Error as e:
            return json.dumps({"error": str(e)})

    def handle_list_tables(self, arguments: dict[str, Any]) -> str:
        """List all tables in the database."""
        conn = self._get_connection()
        cursor = conn.execute(
            "SELECT name FROM sqlite_master WHERE type='table' ORDER BY name"
        )
        tables = [row[0] for row in cursor.fetchall()]
        return json.dumps({"tables": tables})

    def handle_describe_table(self, arguments: dict[str, Any]) -> str:
        """Describe a specific table's schema."""
        table_name = arguments["table_name"]

        # Prevent SQL injection in table name
        if not table_name.isidentifier():
            return json.dumps({"error": "Invalid table name"})

        conn = self._get_connection()
        cursor = conn.execute(f"PRAGMA table_info({table_name})")
        columns = [
            {
                "name": row[1],
                "type": row[2],
                "nullable": not row[3],
                "primary_key": bool(row[5]),
            }
            for row in cursor.fetchall()
        ]
        return json.dumps({"table": table_name, "columns": columns})

    def handle_request(self, request: dict[str, Any]) -> dict[str, Any]:
        """Route a JSON-RPC request to the appropriate handler."""
        method = request.get("method", "")
        request_id = request.get("id")
        params = request.get("params", {})

        if method == "initialize":
            return make_response(request_id, {
                "protocolVersion": "2024-11-05",
                "capabilities": {"tools": {}},
                "serverInfo": {
                    "name": "sqlite-database",
                    "version": "1.0.0",
                },
            })

        if method == "notifications/initialized":
            # This is a notification, no response needed
            return None

        if method == "tools/list":
            return make_response(request_id, {"tools": TOOLS})

        if method == "tools/call":
            tool_name = params.get("name", "")
            arguments = params.get("arguments", {})

            handlers = {
                "query": self.handle_query,
                "list_tables": self.handle_list_tables,
                "describe_table": self.handle_describe_table,
            }

            handler = handlers.get(tool_name)
            if handler is None:
                return make_error(request_id, -32601, f"Unknown tool: {tool_name}")

            try:
                result_text = handler(arguments)
                return make_response(request_id, {
                    "content": [{"type": "text", "text": result_text}],
                })
            except Exception as e:
                return make_response(request_id, {
                    "content": [{"type": "text", "text": json.dumps({"error": str(e)})}],
                    "isError": True,
                })

        return make_error(request_id, -32601, f"Unknown method: {method}")

    def run(self) -> None:
        """Main loop: read requests from stdin, write responses to stdout."""
        sys.stderr.write(f"MCP SQLite server starting, db: {self.db_path}\n")

        while True:
            request = read_message()
            if request is None:
                break

            response = self.handle_request(request)
            if response is not None:
                write_message(response)

        if self.conn:
            self.conn.close()


if __name__ == "__main__":
    import os

    db_path = os.environ.get("DATABASE_PATH", "database.db")
    if not Path(db_path).exists():
        sys.stderr.write(f"Database not found: {db_path}\n")
        sys.exit(1)

    server = DatabaseServer(db_path)
    server.run()

Building an MCP Server (TypeScript/Node.js)¶

For TypeScript servers, you can use the official MCP SDK. Here’s a Jira integration server:

typescript
#!/usr/bin/env node
/**
 * MCP server for Jira integration.
 * Provides tools to read issues, create issues, and add comments.
 */

import * as readline from "readline";

const JIRA_BASE_URL = process.env.JIRA_BASE_URL || "";
const JIRA_EMAIL = process.env.JIRA_EMAIL || "";
const JIRA_API_TOKEN = process.env.JIRA_API_TOKEN || "";

const AUTH_HEADER =
  "Basic " + Buffer.from(`${JIRA_EMAIL}:${JIRA_API_TOKEN}`).toString("base64");

interface JsonRpcRequest {
  jsonrpc: string;
  id?: number | string;
  method: string;
  params?: Record<string, unknown>;
}

interface JsonRpcResponse {
  jsonrpc: string;
  id: number | string;
  result?: unknown;
  error?: { code: number; message: string };
}

const TOOLS = [
  {
    name: "get_issue",
    description:
      "Get details of a Jira issue by key (e.g., PROJ-123). Returns summary, description, status, assignee, and comments.",
    inputSchema: {
      type: "object",
      properties: {
        issue_key: {
          type: "string",
          description: "The Jira issue key (e.g., PROJ-123)",
        },
      },
      required: ["issue_key"],
    },
  },
  {
    name: "search_issues",
    description:
      "Search Jira issues using JQL (Jira Query Language). Returns matching issues with key, summary, status, and assignee.",
    inputSchema: {
      type: "object",
      properties: {
        jql: {
          type: "string",
          description: 'JQL query (e.g., "project = PROJ AND status = Open")',
        },
        max_results: {
          type: "number",
          description: "Maximum number of results (default: 10)",
          default: 10,
        },
      },
      required: ["jql"],
    },
  },
  {
    name: "add_comment",
    description: "Add a comment to a Jira issue.",
    inputSchema: {
      type: "object",
      properties: {
        issue_key: {
          type: "string",
          description: "The Jira issue key",
        },
        body: {
          type: "string",
          description: "The comment text",
        },
      },
      required: ["issue_key", "body"],
    },
  },
];

async function jiraFetch(
  path: string,
  options: RequestInit = {}
): Promise<unknown> {
  const url = `${JIRA_BASE_URL}/rest/api/3${path}`;
  const response = await fetch(url, {
    ...options,
    headers: {
      Authorization: AUTH_HEADER,
      "Content-Type": "application/json",
      Accept: "application/json",
      ...(options.headers || {}),
    },
  });

  if (!response.ok) {
    throw new Error(`Jira API error: ${response.status} ${response.statusText}`);
  }
  return response.json();
}

async function handleGetIssue(args: Record<string, unknown>): Promise<string> {
  const issueKey = args.issue_key as string;
  const data = (await jiraFetch(`/issue/${issueKey}`)) as Record<string, any>;

  const result = {
    key: data.key,
    summary: data.fields.summary,
    description: data.fields.description,
    status: data.fields.status?.name,
    assignee: data.fields.assignee?.displayName || "Unassigned",
    priority: data.fields.priority?.name,
    created: data.fields.created,
    updated: data.fields.updated,
    labels: data.fields.labels,
  };

  return JSON.stringify(result, null, 2);
}

async function handleSearchIssues(
  args: Record<string, unknown>
): Promise<string> {
  const jql = args.jql as string;
  const maxResults = (args.max_results as number) || 10;

  const data = (await jiraFetch(
    `/search?jql=${encodeURIComponent(jql)}&maxResults=${maxResults}`
  )) as Record<string, any>;

  const issues = data.issues.map((issue: any) => ({
    key: issue.key,
    summary: issue.fields.summary,
    status: issue.fields.status?.name,
    assignee: issue.fields.assignee?.displayName || "Unassigned",
    priority: issue.fields.priority?.name,
  }));

  return JSON.stringify({ total: data.total, issues }, null, 2);
}

async function handleAddComment(
  args: Record<string, unknown>
): Promise<string> {
  const issueKey = args.issue_key as string;
  const body = args.body as string;

  await jiraFetch(`/issue/${issueKey}/comment`, {
    method: "POST",
    body: JSON.stringify({
      body: {
        type: "doc",
        version: 1,
        content: [
          {
            type: "paragraph",
            content: [{ type: "text", text: body }],
          },
        ],
      },
    }),
  });

  return JSON.stringify({ success: true, message: `Comment added to ${issueKey}` });
}

async function handleRequest(
  request: JsonRpcRequest
): Promise<JsonRpcResponse | null> {
  const { method, id, params = {} } = request;

  if (method === "initialize") {
    return {
      jsonrpc: "2.0",
      id: id!,
      result: {
        protocolVersion: "2024-11-05",
        capabilities: { tools: {} },
        serverInfo: { name: "jira", version: "1.0.0" },
      },
    };
  }

  if (method === "notifications/initialized") {
    return null; // Notification, no response
  }

  if (method === "tools/list") {
    return {
      jsonrpc: "2.0",
      id: id!,
      result: { tools: TOOLS },
    };
  }

  if (method === "tools/call") {
    const toolName = (params as any).name as string;
    const args = (params as any).arguments || {};

    const handlers: Record<
      string,
      (args: Record<string, unknown>) => Promise<string>
    > = {
      get_issue: handleGetIssue,
      search_issues: handleSearchIssues,
      add_comment: handleAddComment,
    };

    const handler = handlers[toolName];
    if (!handler) {
      return {
        jsonrpc: "2.0",
        id: id!,
        error: { code: -32601, message: `Unknown tool: ${toolName}` },
      };
    }

    try {
      const resultText = await handler(args);
      return {
        jsonrpc: "2.0",
        id: id!,
        result: {
          content: [{ type: "text", text: resultText }],
        },
      };
    } catch (e: any) {
      return {
        jsonrpc: "2.0",
        id: id!,
        result: {
          content: [{ type: "text", text: JSON.stringify({ error: e.message }) }],
          isError: true,
        },
      };
    }
  }

  return {
    jsonrpc: "2.0",
    id: id!,
    error: { code: -32601, message: `Unknown method: ${method}` },
  };
}

// Main loop
const rl = readline.createInterface({ input: process.stdin });

rl.on("line", async (line: string) => {
  try {
    const request = JSON.parse(line) as JsonRpcRequest;
    const response = await handleRequest(request);
    if (response) {
      process.stdout.write(JSON.stringify(response) + "\n");
    }
  } catch (e: any) {
    process.stderr.write(`Error processing request: ${e.message}\n`);
  }
});

MCP Resources vs. Tools¶

MCP defines two types of capabilities that servers can expose:

Tools¶

Tools are actions Claude can take. They have parameters, perform an operation, and return a result. Tools are the primary way Claude interacts with external systems.

json
{
  "name": "create_ticket",
  "description": "Create a new Jira ticket",
  "inputSchema": {
    "type": "object",
    "properties": {
      "project": { "type": "string" },
      "summary": { "type": "string" },
      "description": { "type": "string" }
    },
    "required": ["project", "summary"]
  }
}

Resources¶

Resources are data Claude can read. They represent static or semi-static information — database schemas, configuration files, documentation, API specs. Resources are identified by URIs.

json
{
  "uri": "schema://tables",
  "name": "Database Schema",
  "description": "Current database table definitions",
  "mimeType": "application/json"
}

Claude can request resources to load context before taking action. For example, a database MCP server might expose the schema as a resource (so Claude understands the tables) and queries as tools (so Claude can fetch data).

When to Use Resources vs. Tools¶

Use Case Resource or Tool
Database schema Resource — static, read for context
Query results Tool — dynamic, depends on parameters
API documentation Resource — reference material
Create/update/delete operations Tool — action with side effects
Configuration values Resource — read once, use many times
Search with parameters Tool — dynamic query

Caching for Performance¶

MCP servers that query external APIs benefit from caching. A simple time-based cache prevents redundant API calls:

python
import time
from typing import Any

class Cache:
    """Simple TTL cache for MCP server responses."""

    def __init__(self, ttl_seconds: int = 300) -> None:
        self.ttl = ttl_seconds
        self._store: dict[str, tuple[float, Any]] = {}

    def get(self, key: str) -> Any | None:
        if key in self._store:
            timestamp, value = self._store[key]
            if time.time() - timestamp < self.ttl:
                return value
            del self._store[key]
        return None

    def set(self, key: str, value: Any) -> None:
        self._store[key] = (time.time(), value)


# Usage in tool handler
cache = Cache(ttl_seconds=60)

def handle_query(arguments: dict[str, Any]) -> str:
    cache_key = arguments["sql"]
    cached = cache.get(cache_key)
    if cached is not None:
        return cached

    result = execute_query(arguments["sql"])
    cache.set(cache_key, result)
    return result

Error Handling and Timeouts¶

Robust MCP servers handle failures gracefully:

python
import signal
from contextlib import contextmanager
from typing import Generator


@contextmanager
def timeout(seconds: int) -> Generator[None, None, None]:
    """Context manager that raises TimeoutError after N seconds."""
    def handler(signum: int, frame: Any) -> None:
        raise TimeoutError(f"Operation timed out after {seconds}s")

    old_handler = signal.signal(signal.SIGALRM, handler)
    signal.alarm(seconds)
    try:
        yield
    finally:
        signal.alarm(0)
        signal.signal(signal.SIGALRM, old_handler)


def handle_query(arguments: dict[str, Any]) -> str:
    """Execute a query with timeout and error handling."""
    try:
        with timeout(30):
            result = execute_query(arguments["sql"])
            return json.dumps(result)
    except TimeoutError:
        return json.dumps({"error": "Query timed out after 30 seconds"})
    except sqlite3.OperationalError as e:
        return json.dumps({"error": f"SQL error: {e}"})
    except Exception as e:
        return json.dumps({"error": f"Unexpected error: {type(e).__name__}: {e}"})

Always return errors as structured JSON in the tool result — don’t let exceptions crash the server process.

Security Considerations¶

MCP servers run with your user permissions. A malicious or poorly-written MCP server has the same access as any process you run. Treat MCP server code with the same security scrutiny as any code that handles credentials.

Secrets Management¶

Never hardcode credentials in MCP server source code. Use environment variables:

json
{
  "env": {
    "DATABASE_URL": "${DATABASE_URL}",
    "API_TOKEN": "${API_TOKEN}"
  }
}

The ${VAR} syntax reads from your shell environment at server startup. The values never appear in the settings file or in Claude’s context.

Input Validation¶

Every tool handler must validate input before acting on it:

python
def handle_query(arguments: dict[str, Any]) -> str:
    sql = arguments.get("sql", "").strip()

    # Reject empty queries
    if not sql:
        return json.dumps({"error": "SQL query is required"})

    # Only allow SELECT (read-only)
    if not sql.upper().startswith("SELECT"):
        return json.dumps({"error": "Only SELECT queries are allowed"})

    # Block dangerous keywords even in SELECT
    dangerous = ["DROP", "DELETE", "INSERT", "UPDATE", "ALTER", "EXEC"]
    upper_sql = sql.upper()
    for keyword in dangerous:
        if keyword in upper_sql:
            return json.dumps({"error": f"Forbidden keyword: {keyword}"})

    return execute_query(sql)

Permission Rules¶

Apply Claude Code’s permission system to MCP tools:

json
{
  "permissions": {
    "allow": [
      "mcp__project-db__query(*)",
      "mcp__project-db__list_tables(*)",
      "mcp__project-db__describe_table(*)"
    ],
    "deny": [
      "mcp__project-db__execute_ddl(*)",
      "mcp__project-db__drop_table(*)"
    ]
  }
}

Even if the server exposes a dangerous tool, the deny rule prevents Claude from using it.

Auditing Third-Party MCP Servers¶

Before registering a community MCP server:

  1. Read the source code — What does it do with your credentials?
  2. Check the npm/PyPI package — Is it what the repository claims?
  3. Verify the author — Is this a maintained, trusted project?
  4. Pin the version — Don’t use npx -y package@latest in production
  5. Test in isolation — Run it standalone before giving Claude access

Debugging MCP Servers¶

When an MCP tool fails, Claude reports the error. But the error message alone often isn’t enough to diagnose the issue.

Common Issues and Fixes¶

Server won’t start:

bash
# Check if the command works standalone
python3 mcp_servers/database.py
# Check stderr output for error messages

Stdio buffering: Python buffers stdout by default. If the server seems to hang, ensure you’re flushing after every write:

python
sys.stdout.write(json.dumps(response) + "\n")
sys.stdout.flush()  # Critical — without this, messages may buffer

Or run Python with unbuffered stdout:

json
{
  "command": "python3",
  "args": ["-u", "mcp_servers/database.py"]
}

JSON parsing errors: Each message must be a single line of JSON. No pretty-printing, no multi-line output to stdout. Use stderr for logging:

python
# Wrong — this goes to stdout (protocol channel)
print("Debug: processing query")

# Right — this goes to stderr (logging channel)
sys.stderr.write("Debug: processing query\n")

Environment variables not set: Verify that the ${VAR} references in your settings resolve correctly:

bash
echo $DATABASE_URL  # Should print the connection string
echo $JIRA_API_TOKEN  # Should print the token

Testing the Full Pipeline¶

Test an MCP server end-to-end by simulating what Claude Code does:

bash
#!/usr/bin/env bash
set -euo pipefail

# Start the server
DATABASE_PATH=./data/app.db python3 -u mcp_servers/database.py &
SERVER_PID=$!

# Send initialize
echo '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{}}' > /proc/$SERVER_PID/fd/0

# Send tool list request
echo '{"jsonrpc":"2.0","id":2,"method":"tools/list","params":{}}' > /proc/$SERVER_PID/fd/0

# Send a query
echo '{"jsonrpc":"2.0","id":3,"method":"tools/call","params":{"name":"query","arguments":{"sql":"SELECT count(*) FROM users"}}}' > /proc/$SERVER_PID/fd/0

# Clean up
kill $SERVER_PID

Chapter Summary¶

  • MCP (Model Context Protocol) is an open protocol that connects Claude Code to external systems through a standardized JSON-RPC interface over stdio
  • MCP servers are registered in settings.json with a command, args, and env — Claude Code spawns the process and communicates via stdin/stdout
  • MCP tools appear as mcp__servername__toolname and work identically to built-in tools, including permission rules
  • Building MCP servers requires handling four JSON-RPC methods: initialize, notifications/initialized, tools/list, and tools/call
  • The MCP ecosystem includes official servers (SQLite, filesystem, GitHub, Slack) and community servers (Notion, Linear, PostgreSQL, MongoDB, and more)
  • Security fundamentals: use environment variables for credentials, validate all input, apply permission deny rules, audit tool calls, and review third-party server code before registering
  • Multi-server setups let Claude orchestrate across databases, messaging, and project management in a single workflow
