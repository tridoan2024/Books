Chapter 15: Prompt Engineering for Code Generation¶ ~18 min read

A junior engineer types “fix the bug” into Claude Code and spends thirty minutes clarifying what bug, which file, what the expected behavior should be. A senior engineer types “fix the null pointer in src/auth/validator.py:142 – the user object can be None when the session expires mid-request” and gets a working fix in one turn. Same tool. Same model. Wildly different outcomes.

The difference is not “prompting” – it is prompt engineering: the systematic practice of crafting inputs that consistently produce correct, complete, first-try results from an AI coding agent. In a chat interface, a mediocre prompt wastes a few seconds. In Claude Code, a mediocre prompt wastes minutes of tool execution, dollars of API costs, and thousands of tokens of context window. Your prompt determines not just what Claude writes, but which tools it selects, which files it reads, which approaches it considers, and when it stops.

This chapter covers the five-layer prompt stack, twenty battle-tested patterns, the anti-patterns that burn money, and the meta-skill of knowing when to stop prompting and start configuring.

Prompting vs. Prompt Engineering¶

Prompting is typing what you want and hoping for the best:

fix the login bug

Prompt engineering is systematic input design that controls the output:

Fix the race condition in src/auth/session.py:89 where concurrent
requests can both pass the token-expiry check before either invalidates
the token. Write a failing test first, then fix with a Redis-based
distributed lock. Run the test to confirm.

The second prompt is not just “more detailed.” It does five things the first prompt cannot:

  1. Locates the target (file and line number)
  2. Describes the root cause (race condition, concurrent requests)
  3. Specifies the approach (Redis distributed lock)
  4. Defines the workflow (test first, then fix)
  5. Sets the stopping condition (run the test to confirm)

Claude Code amplifies the difference between good and bad prompts because it makes autonomous decisions. With a vague prompt, Claude decides which files to read, which approach to take, and when to stop. With a precise prompt, you make those decisions and Claude executes them.

The Prompt Engineering Stack¶

When you type a message into Claude Code, your text is just one of five layers that influence Claude’s behavior. Understanding all five layers is essential because they interact – and sometimes conflict.

┌──────────────────────────────────────────┐
│  Layer 5: Memory Layer (persistent)       │
│  Auto-memory, preferences, corrections    │
├──────────────────────────────────────────┤
│  Layer 4: Message Layer (ephemeral)       │
│  Your actual typed prompt                 │
├──────────────────────────────────────────┤
│  Layer 3: Context Layer (accumulated)     │
│  Files read, search results, tool output  │
├──────────────────────────────────────────┤
│  Layer 2: Skill Layer (per-workflow)      │
│  Active skill's SKILL.md instructions     │
├──────────────────────────────────────────┤
│  Layer 1: System Layer (persistent)       │
│  CLAUDE.md, rules, settings, tool defs    │
└──────────────────────────────────────────┘

Layer 1: System Layer¶

This is CLAUDE.md, your rule files, settings.json, and tool definitions. It loads every session and defines Claude’s baseline behavior: coding standards, communication style, permission boundaries. As discussed in Chapter 4, this is the most cost-effective place to put instructions that apply universally.

Layer 2: Skill Layer¶

When a skill activates (either via /slash-command or auto-detection), its SKILL.md loads into context. This provides workflow-specific instructions: “When doing a code review, check security first, then logic, then performance.” Skills are the mid-tier of the stack – more specific than system instructions, more persistent than a single message.

Layer 3: Context Layer¶

Everything Claude has read or received during the session: file contents, grep results, bash output, previous tool calls. This layer is accumulated – it grows with every turn. It is also the most expensive layer in terms of tokens, as discussed in Chapter 13.

Layer 4: Message Layer¶

Your actual typed prompt. This is the most visible layer but often the least impactful, because layers 1-3 already define most of Claude’s behavior. A well-configured system means your messages can be short and still produce excellent results.

Layer 5: Memory Layer¶

Auto-memory facts and preferences from previous sessions. These load into the system prompt and influence Claude’s defaults: “User prefers Pydantic over dataclasses,” “Never force-push to main,” “Project uses Redis for caching.”

Why the Stack Matters¶

When your prompt says one thing but your CLAUDE.md says another, Claude follows CLAUDE.md. When a memory says “use dataclasses” but you type “use Pydantic for this model,” your message wins for this turn but the memory persists for future sessions.

Understanding the stack helps you place instructions at the right level:

Instruction Type Best Layer
“Always use type hints in Python” System (CLAUDE.md or rule)
“When reviewing code, check security first” Skill (SKILL.md)
“Read auth.py before fixing this bug” Message (your prompt)
“I prefer terse output” Memory (auto-saved or explicit)

The highest-leverage prompt engineering happens at layers 1 and 2, not layer 4. But since most engineers interact primarily at layer 4, the rest of this chapter focuses on message-layer patterns.

The 20 Prompt Patterns¶

These patterns are drawn from thousands of hours of professional Claude Code usage. Each one addresses a specific failure mode of unstructured prompting.

Pattern 1: The Constraint Pattern¶

When to use: You want Claude to work within specific boundaries – no external dependencies, specific language version, particular library choices.

Example:

Implement a rate limiter for our API endpoints. Constraints:
- Use only Python stdlib (no redis, no external packages)
- Must be thread-safe (we run with gunicorn workers)
- Token bucket algorithm
- Configuration via environment variables

Why it works: Without constraints, Claude makes its own technology choices. It might pull in Redis, add a new pip dependency, or choose an algorithm that does not fit your architecture. Constraints eliminate entire branches of Claude’s decision tree.

Pro tip: Negative constraints (“do NOT use X”) are even more powerful than positive ones. Claude tends to reach for popular solutions; telling it what to avoid forces creative alternatives.

Pattern 2: The Architecture-First Pattern¶

When to use: Before implementing anything complex, you want Claude to show you the design.

Example:

Before writing any code, show me the architecture for a webhook
processing system:
- Component diagram (Mermaid)
- Data flow for incoming webhooks
- How retries and dead-letter queue work
- Which files you'll create and why

Why it works: Claude’s first instinct is to start coding. This pattern forces it to think before writing, catching architectural mistakes before they become 500-line implementations you have to throw away.

Pro tip: Combine with plan mode (/plan) for even more structured output.

Pattern 3: The Test-First Pattern¶

When to use: Any bug fix or feature implementation where correctness matters more than speed.

Example:

Write failing tests for these requirements first:
1. Users with expired subscriptions cannot access premium endpoints
2. Grace period of 3 days after expiry still allows access
3. Admin users bypass subscription checks entirely

Then implement the middleware to make all tests pass.

Why it works: Tests constrain Claude’s implementation. Without tests, Claude might implement something that “looks right” but misses edge cases. With tests defined first, the implementation must satisfy concrete, verifiable requirements.

Pro tip: Include edge cases in the test specification. Claude tends to implement the happy path well but miss boundary conditions unless you explicitly name them.

Pattern 4: The Example-Driven Pattern¶

When to use: When the output format or style matters and is hard to describe in words.

Example:

Here's an existing API endpoint that follows our conventions:

```python
@router.post("/users", response_model=UserResponse, status_code=201)
async def create_user(
    payload: CreateUserRequest,
    db: AsyncSession = Depends(get_db),
    current_user: User = Depends(require_admin),
) -> UserResponse:
    """Create a new user account."""
    user = await user_service.create(db, payload)
    return UserResponse.from_orm(user)

Now create endpoints for: - POST /teams (create team) - GET /teams/{team_id} (get team by ID) - PUT /teams/{team_id} (update team)

Follow exactly the same patterns: dependency injection, response models, service layer delegation.


**Why it works:** An example communicates more than a page of specifications. Claude pattern-matches against examples extremely well. The code it produces will mirror the structure, naming, and patterns of your example.

**Pro tip:** Provide examples of what you DO want, not just what you do not want. Positive examples are more effective than negative instructions.

### Pattern 5: The Progressive Refinement Pattern

**When to use:** When you are exploring a design space and do not know the final shape upfront.

**Example:**

Start with the simplest possible implementation of a task queue. Just an in-memory list with add/get/complete operations. No persistence, no retries, no priority – bare minimum.


Then iterate:

Now add Redis persistence. Tasks survive process restarts.


Then:

Now add retry logic. Failed tasks retry 3 times with exponential backoff.


**Why it works:** Each increment is small enough for Claude to get right on the first try. Building up gradually also lets you steer the design at each stage rather than discovering problems after a large implementation.

### Pattern 6: The Chain-of-Files Pattern

**When to use:** When Claude needs context from multiple files before making a change.

**Example:**

Read these files first, then fix the bug: 1. src/models/user.py (the User model) 2. src/services/auth.py (the authentication service) 3. src/middleware/session.py (the session middleware) 4. tests/test_auth.py (existing auth tests)

The bug: session middleware doesn’t check if the user is soft-deleted before allowing access.


**Why it works:** Claude reads files in the order you specify, building up context before attempting the fix. Without this pattern, Claude might read one file, attempt a fix, realize it needs another file, read it, revise the fix, and repeat -- wasting turns and tokens.

**Pro tip:** Order the files from most fundamental (models, schemas) to most applied (middleware, routes). This mirrors how a human engineer would study the code.

### Pattern 7: The Negative Constraint Pattern

**When to use:** When Claude's default behavior adds things you do not want.

**Example:**

Fix the null pointer exception in parser.py:67. Do NOT: - Add logging - Add type hints to existing code - Refactor surrounding functions - Add error handling beyond the specific fix Only change what’s necessary to prevent the null pointer.


**Why it works:** Claude has a tendency toward "while I'm here" improvements -- adding type hints, restructuring code, adding logging. This is usually helpful but sometimes destructive (especially in large codebases where unrelated changes create merge conflicts or break other team members' work). Negative constraints keep Claude focused.

### Pattern 8: The Role Pattern

**When to use:** When you want Claude to adopt a specific perspective for analysis or review.

**Example:**

Review src/api/payment.py as a senior security engineer who has seen PCI compliance audits fail. Focus exclusively on: - Data exposure (are card numbers ever logged?) - Input validation (can malformed requests cause unexpected behavior?) - Authentication gaps (are all endpoints properly protected?)


**Why it works:** The role frames Claude's evaluation criteria. A "security engineer" review produces different findings than a "performance engineer" review, even on the same code. The role also suppresses irrelevant findings -- a security reviewer does not comment on variable naming.

### Pattern 9: The Output Format Pattern

**When to use:** When you need structured output for further processing or consistent presentation.

**Example:**

Audit all TODO comments in src/. Output as a markdown table:

| File | Line | TODO Text | Priority (P1/P2/P3) | Suggested Action |


**Why it works:** Without format specification, Claude outputs free-form text that you then have to restructure. Specifying the format upfront gets usable output in one turn.

**Pro tip:** For data you need to process programmatically, ask for JSON output explicitly.

### Pattern 10: The Diff Pattern

**When to use:** Before allowing Claude to modify files, you want to preview the changes.

**Example:**

Show me what you would change in auth.py to fix the session timeout bug. Don’t make any edits yet – just show the diff as a code block.


**Why it works:** This gives you a review gate before Claude touches files. Particularly useful for large files where a wrong edit is expensive to undo, or when you want to understand Claude's approach before committing to it.

### Pattern 11: The Scope Limiter

**When to use:** When you want Claude to modify specific files or functions and nothing else.

**Example:**

Only modify the validate_token() function in src/auth/tokens.py. Do not change any other function in that file. Do not modify any other file.


**Why it works:** Without scope limits, Claude may "helpfully" update callers of the function, adjust related tests, or refactor neighboring code. In a team environment, touching unexpected files creates surprise diffs in pull requests.

### Pattern 12: The Reference Pattern

**When to use:** When your codebase has an established pattern and you want new code to match it.

**Example:**

Create a new service for notifications following the exact same pattern as src/services/email_service.py: - Same class structure - Same error handling approach - Same logging pattern - Same test structure (see tests/test_email_service.py)


**Why it works:** Consistency across a codebase is more valuable than any individual file being "optimal." The reference pattern ensures Claude produces code that looks like it was written by the same team, not by an AI with different style preferences.

### Pattern 13: The Explain-First Pattern

**When to use:** When you need to understand code before changing it.

**Example:**

Before suggesting any changes, explain how the request pipeline works in src/middleware/. Walk me through a request from entry to response, noting each middleware function and what it does.


**Why it works:** Claude's explanation reveals its understanding of the code. If the explanation is wrong, you can correct it before Claude makes changes based on a flawed mental model. This is especially valuable in unfamiliar codebases.

### Pattern 14: The Batch Pattern

**When to use:** When the same change needs to be applied across many files.

**Example:**

In every file matching src/api/*/.py, replace: from src.db import get_session with: from src.db import get_async_session

Then update every function that calls get_session() to use get_async_session() with await.


**Why it works:** Claude handles repetitive transformations across files well, but only if you define the transformation precisely. Vague batch instructions ("update all the database calls") lead to inconsistent changes across files.

**Pro tip:** For large batches (20+ files), consider using the `/batch` command, which runs parallel worktrees for each change.

### Pattern 15: The Checkpoint Pattern

**When to use:** Multi-step implementations where you want to verify correctness at each stage.

**Example:**

Implement the caching layer in three steps. After each step, run the relevant tests and stop if any fail:

Step 1: Add Redis cache client in src/cache/client.py Run: pytest tests/test_cache_client.py

Step 2: Add cache decorator in src/cache/decorators.py Run: pytest tests/test_cache_decorators.py

Step 3: Apply decorator to the 5 most expensive queries Run: pytest tests/ -k “cache”


**Why it works:** Without checkpoints, Claude plows through all three steps. If step 1 has a bug, steps 2 and 3 are built on a broken foundation. Checkpoints catch errors early, before they compound.

### Pattern 16: The Multi-Agent Pattern

**When to use:** When a task has independent sub-problems that benefit from parallel investigation.

**Example:**

Use subagents to investigate these three things in parallel: 1. How does our current auth middleware handle token refresh? 2. What rate limiting exists on our API endpoints? 3. What logging/monitoring covers our auth flow?

Then synthesize the findings into a security assessment.


**Why it works:** Each subagent gets its own context window, reads its own set of files, and returns a focused summary. Your main context stays clean for the synthesis step. As discussed in Chapter 12, this is the primary pattern for context-efficient investigation.

### Pattern 17: The Plan-Then-Execute Pattern

**When to use:** Any change affecting more than three files.

**Example:**

Plan first, don’t code yet. I need to add audit logging to every service that modifies user data. Show me: 1. Which services are affected 2. What the audit log schema should be 3. Where the logging calls go in each service 4. How to test it

Wait for my approval before implementing.


**Why it works:** This pattern prevents the most expensive mistake in Claude Code: implementing the wrong approach across multiple files and then having to undo it all. Ten minutes of planning saves an hour of rework.

### Pattern 18: The Minimal Change Pattern

**When to use:** Bug fixes, hotfixes, or any change where minimizing diff size matters.

**Example:**

Fix the off-by-one error in pagination. Make the smallest possible change. Do not refactor, do not add comments, do not improve surrounding code. I need a clean, reviewable one-line diff.


**Why it works:** In production environments, small diffs are easier to review, easier to revert, and less likely to introduce new bugs. Claude's tendency to improve "while it's in there" works against you in hotfix scenarios.

### Pattern 19: The Context-Loading Pattern

**When to use:** Complex tasks where Claude needs broad context before starting.

**Example:**

Before doing anything, read: 1. README.md (project overview) 2. docs/architecture.md (system design) 3. The last 10 git commits (recent changes) 4. .env.example (configuration shape)

Then: add a new background job that processes webhook events.


**Why it works:** Front-loading context prevents Claude from making assumptions. Without reading the architecture doc, Claude might design the background job in a way that conflicts with your existing job processing system.

**Pro tip:** Be selective. Loading too much context wastes tokens (Chapter 13). Load only what is directly relevant to the task.

### Pattern 20: The Escape Hatch Pattern

**When to use:** When there is a decision point where Claude's choice could go wrong.

**Example:**

Implement the file upload endpoint. If you’re unsure about: - Maximum file size (ask me) - Supported file types (ask me) - Storage backend (ask me) Do not guess – ask before proceeding. For everything else, use your best judgment.


**Why it works:** Claude's default mode is to make all decisions autonomously. For most decisions (variable naming, error handling style), this is fine. For domain-specific decisions (business rules, infrastructure choices), Claude's guesses are often wrong. The escape hatch tells Claude exactly where its judgment is not trusted.

## Anti-Patterns: What NOT to Do

### Anti-Pattern 1: The Vague Prompt

Bad¶

make it better

Claude changes random things: renames variables, adds comments,¶

restructures code – none of which you wanted¶


**Fix:** Be specific about what "better" means. Faster? More readable? More secure? More testable?

### Anti-Pattern 2: The Novel

Bad¶

I need you to fix the authentication system. The problem is that when users log in, sometimes their session expires too quickly. I think it might be related to the token refresh logic, but I’m not sure. Also, we’ve been having issues with the Redis connection pool, and I wonder if that’s related. Last week, Sarah mentioned that the staging environment had a similar issue but with different symptoms. The error we’re seeing in production is a 401 on the /api/dashboard endpoint, but only for users who logged in more than 2 hours ago…


**Fix:** Lead with the actionable information:

Good¶

Fix: 401 errors on /api/dashboard for sessions older than 2 hours. Likely cause: token refresh in src/auth/tokens.py. Check Redis connection pool timeouts as a contributing factor.


### Anti-Pattern 3: The Missing Context

Bad¶

fix the bug in the handler

Claude: “Which handler? Which bug? Let me search the entire codebase…”¶

5 tool calls later, it finds the wrong handler¶


**Fix:** Always include the file path, and ideally the line number or function name.

### Anti-Pattern 4: The Overload

Bad¶

Fix the login bug, then add rate limiting to all endpoints, then write tests for the payment module, then review the PR from yesterday, and also update the README with the new API endpoints.


**Fix:** One task per prompt. Use `/clear` between tasks. Claude prioritizes the first request and may lose quality on later ones, or attempt all five simultaneously and do none well.

### Anti-Pattern 5: The Context Bleed

Bad: continuing a long session without clearing¶

Turn 1-20: Debugging auth system (reads 15 files) Turn 21: “Now write a haiku about programming”

Claude’s haiku somehow references JWT tokens¶


**Fix:** `/clear` between unrelated tasks. Always.

## Advanced: Intent Routing via Prompt Structure

As introduced in Chapter 4, Claude Code's auto-execute protocol routes short prompts to specific workflows based on structure:

| Prompt Shape | Detected Intent | Workflow |
|-------------|----------------|---------|
| `filename.py` | Code review | Read file, severity-ranked findings |
| `#42` | Fix GitHub issue | Read issue, investigate, fix, test, PR |
| `PR 123` | PR summary | Load diff, risk assessment, checklist |
| `[stack trace]` | Debug | Hypothesize, investigate, root-cause, fix |
| `"this is slow"` | Performance | Profile, find bottlenecks, optimize |

This works because CLAUDE.md contains an intent routing table (detailed in `tools/prompts/intent-map.md`) that maps prompt patterns to execution workflows. Claude reads this routing table as part of its system prompt and applies it to every message.

### How It Works Under the Hood

When you type `login.py`, Claude's reasoning chain is:

1. "The user typed just a filename. Per my routing table, a bare filename maps to code review."
2. "The code review workflow says: read the file, check security, then logic, then performance, then readability."
3. "Output format: severity-ranked findings with file:line references."

The prompt you typed was two words. The actual instructions Claude followed came from layers 1 and 2 of the prompt stack (CLAUDE.md and the code-review skill). This is prompt engineering at the system level -- you invested time once in configuring the routing, and now every short prompt gets expanded automatically.

### Building Your Own Intent Routes

You can extend the routing table for your domain. Add entries to your CLAUDE.md or intent-map.md:

```markdown
## custom-routes
| User says | Route to |
|---|---|
| mentions "migration" | Database migration workflow |
| mentions "onboard" or "new hire" | Onboarding documentation |
| mentions "incident" or "outage" | Incident response runbook |

Then define the execution steps for each route, either in the intent map or as a dedicated skill.

The Meta-Skill: When to Prompt vs. When to Configure¶

The most important prompt engineering insight is knowing when to stop prompting and start configuring. Here is the rule:

If you prompt the same instruction three times, it belongs in configuration.

You keep typing… Move it to…
“Always use type hints” A rule file (.claude/rules/python.md)
“Check security first when reviewing code” A skill (.claude/skills/code-review/SKILL.md)
“Don’t allow rm -rf commands” A hook (.claude/hooks/block-destructive.sh)
“I prefer Pydantic over dataclasses” Auto-memory (ask Claude to remember)
“Start by reading the architecture doc” CLAUDE.md

Configuration is compiled prompts. You write the instruction once, and it applies to every future session automatically. The ROI compounds: a rule you write today saves you from typing the same instruction hundreds of times over the next year.

The Configuration Hierarchy¶

As you identify repeated prompts, place them at the right level:

Most persistent, broadest scope
─────────────────────────────
  CLAUDE.md              ← Universal instructions, every session
  Rules (.claude/rules/) ← Language/domain-specific, path-scoped
  Skills (.claude/skills/) ← Workflow-specific, invoked per task
  Auto-memory            ← Personal preferences, project facts
  Your message           ← This task, this moment
─────────────────────────────
Most ephemeral, narrowest scope

Move instructions up this hierarchy as they prove their value. A prompt that works well in a message becomes a memory. A memory that applies broadly becomes a rule. A rule that defines a workflow becomes a skill.

Measuring Prompt Quality¶

How do you know if your prompts are good? Four metrics:

1. First-Try Success Rate¶

Did Claude do what you wanted on the first attempt? If you frequently need to /rewind or send correction messages, your prompts are not specific enough. Track this informally: aim for 80%+ first-try success on routine tasks.

2. Tool Selection Accuracy¶

Did Claude choose the right tools? If you asked for a code review and Claude started editing files instead of analyzing them, the prompt did not convey the intent clearly. Watch for unnecessary tool calls – they waste tokens and indicate prompt ambiguity.

3. Appropriate Stopping¶

Did Claude stop at the right point? Two failure modes:

  • Under-doing: Claude fixes the bug but does not run the tests you expected
  • Over-doing: Claude fixes the bug, then refactors the surrounding code, then updates the README, then modifies the CI config

If Claude consistently over-does, add scope limits. If it under-does, add explicit completion criteria.

4. Correction Frequency¶

How often do you need to send follow-up messages correcting Claude’s approach? Each correction costs tokens and indicates a gap between your intent and Claude’s interpretation. High correction frequency on a specific type of task means you should create a skill or rule for it.

The Feedback Loop¶

Write prompt → Observe result → Identify gap → Improve prompt
     ↑                                              │
     └──────── If repeated 3x, move to config ──────┘

This cycle is the core practice of prompt engineering. Every interaction with Claude Code is an opportunity to refine your prompting and configuration. Over weeks and months, your system becomes increasingly tuned to your workflow, your codebase, and your preferences. The tool gets better not because the model improves, but because your prompt engineering stack improves.

What Most People Get Wrong¶

“Longer prompts are better prompts.” Length is not quality. A 10-word prompt with a file path and a clear action verb outperforms a 200-word prompt with background context Claude does not need. Lead with the action, add context only as needed.

“I should explain how the code works.” Claude can read the code. Do not explain what the code does – explain what you want done to it. Your prompt should be intent, not information.

“I need to be polite to get better results.” “Please kindly review this code if you wouldn’t mind” does not produce different results than “Review auth.py.” Skip the social niceties. Every pleasantry is a token that could be a constraint or specification.

“I’ll put everything in my message.” If an instruction applies to more than this one task, it does not belong in your message. Put it in CLAUDE.md, a rule, a skill, or auto-memory. Your message should contain only what is unique to this specific request.

“The prompt is the most important part.” Your message (layer 4) is actually the least important layer of the prompt stack. A well-configured system (layers 1-3 and 5) means your messages can be minimal. Invest your engineering effort in configuration, not in crafting the perfect single prompt.

Chapter Summary¶

  1. Prompt engineering is systematic input design, not just “writing better prompts.” It controls Claude’s tool selection, file choices, approach, and stopping behavior.

  2. The prompt stack has five layers: system, skill, context, message, and memory. Your typed message is just layer 4 – and often the least impactful. Invest in configuring layers 1-3 for the highest leverage.

  3. Twenty patterns cover 90% of professional use cases. The most impactful are: Constraint (bound the solution space), Test-First (verify before implementing), Chain-of-Files (load context deliberately), Negative Constraint (prevent unwanted changes), and Minimal Change (keep diffs reviewable).

  4. The meta-skill is knowing when to stop prompting and start configuring. If you type the same instruction three times, move it to a rule, skill, hook, or memory. Configuration is compiled prompts – write once, apply forever.

  5. Measure prompt quality by first-try success rate, tool selection accuracy, appropriate stopping behavior, and correction frequency. Use these metrics to continuously improve both your prompts and your configuration stack.
