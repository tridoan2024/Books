Chapter 27: Troubleshooting, FAQ, and Expert Tips¶ ~18 min read

Every tool has failure modes. Claude Code is no exception. The difference between a frustrated user and a productive one is knowing where to look when something breaks, having quick answers to recurring questions, and accumulating the small tricks that compound into dramatic efficiency gains.

This chapter is your reference guide for the rough edges. Part 1 walks through the ten most common issues with a strict Symptom-Cause-Fix format. Part 2 answers the twenty questions that come up in every team onboarding. Part 3 delivers twenty-five expert tips — the kind of knowledge that normally takes months of daily use to discover.

Part 1: Troubleshooting¶

Each issue follows the same structure: what you observe, why it happens, and how to fix it. No preamble.

Issue 1: “Claude Ignores My CLAUDE.md Rules”¶

Symptom: You added rules to CLAUDE.md — commit message format, coding standards, output preferences — but Claude doesn’t follow them. It formats code differently than specified, skips your preferred commit style, or ignores explicit instructions.

Cause: The most common culprit is file length. Once CLAUDE.md exceeds roughly 200 lines, Claude’s adherence to individual rules drops measurably. Rules buried below line 100 get less attention than rules at the top. Another cause: vague phrasing. “Try to use type hints” is weaker than “ALWAYS use type hints on every function signature.”

Fix:

  1. Keep CLAUDE.md under 100 lines. Move language-specific rules to .claude/rules/ files with paths: frontmatter so they load only when relevant.

  2. Put your most critical rules in the first 30 lines. Claude reads top-down, and early instructions carry more weight.

  3. Use the IMPORTANT: prefix for rules that must never be violated:

markdown
IMPORTANT: Never commit files matching *.env, *.pem, or credentials.json.
  1. Use @imports for domain rules instead of inlining them:
markdown
## Domain Rules
@.claude/rules/python.md
@.claude/rules/security.md
  1. Validate by asking Claude directly: “What rules from CLAUDE.md apply to this task?” If it can’t recite them, they’re buried too deep or phrased too weakly.

Issue 2: “Skill Doesn’t Trigger Automatically”¶

Symptom: You say “review this code for security issues” but Claude doesn’t load the security-audit skill. It gives a generic response instead of the structured OWASP/CWE output you built.

Cause: Claude decides whether to auto-load a skill based on its description field in the SKILL.md frontmatter. If the description doesn’t contain phrases that match your natural language, Claude won’t make the connection.

Fix:

  1. Open the skill’s SKILL.md and expand the description with more trigger phrases:
yaml
---
name: security-audit
description: >
  Performs security analysis on code. Use when user says "security audit",
  "check for vulnerabilities", "is this secure", "OWASP check", "find
  security bugs", "pentest this", "hardening review", "CVE scan",
  "injection check", "auth bypass", "security review".
---
  1. If you want the skill to trigger ONLY via /security-audit and never automatically, add:
yaml
disable-model-invocation: true
  1. Test by asking Claude: “When would you use the security-audit skill?” It will quote the description back. If it doesn’t mention your trigger phrases, they’re missing.

Issue 3: “Hook Blocks Everything”¶

Symptom: The block-destructive.sh hook rejects legitimate commands. You try to run rm -rf dist/ to clean a build directory, and the hook kills it. Or git push gets blocked because the regex matches too broadly.

Cause: Regex patterns in the hook are too greedy. A pattern like rm -rf catches every recursive force-delete, including safe ones. A pattern like git push.*force catches git push --force-with-lease, which is the safe variant.

Fix:

  1. Test the hook directly with the command that’s being blocked:
bash
echo '{"tool_input":{"command":"rm -rf dist/"}}' | .claude/hooks/block-destructive.sh
echo $?  # 0 = allowed, 2 = blocked
  1. Tighten the regex. Instead of blocking all rm -rf, block only rm -rf / (root) and rm -rf . (current directory):
bash
# Too broad — blocks legitimate cleanup
if [[ "${COMMAND}" =~ rm[[:space:]]+-rf ]]; then

# Better — blocks only catastrophic patterns
if [[ "${COMMAND}" =~ rm[[:space:]]+-rf[[:space:]]+(/|\.)[[:space:]]*$ ]]; then
  1. Add an allowlist for known-safe patterns that your workflow needs:
bash
# Allow build directory cleanup
if [[ "${COMMAND}" =~ rm[[:space:]]+-rf[[:space:]]+(dist|build|node_modules|__pycache__|\.pytest_cache) ]]; then
  exit 0
fi
  1. As a temporary bypass, disable hooks in settings.local.json:
json
{ "disableAllHooks": true }

Remember to re-enable them.

Issue 4: “Context Window Fills Up Too Fast”¶

Symptom: Mid-conversation, Claude starts losing track of earlier instructions. Responses become less accurate. You see warnings about context usage, or Claude asks questions you already answered.

Cause: Context is a finite resource. Common drains: reading large files without line limits, pasting entire stack traces or log dumps, accumulating tool call outputs from long debugging sessions, or working on many unrelated tasks in a single session.

Fix:

  1. Use /clear between unrelated tasks. This is the single highest-impact habit. Each new task should start with a clean context.

  2. Use /compact [focus-topic] when you’re mid-task and running low. The focus topic tells Claude what to preserve:

/compact authentication

This compresses the conversation history while keeping authentication-related context intact.

  1. Read files with limits instead of dumping entire files into context:
Read lines 50-120 of src/auth/middleware.py

  1. Use subagents for investigative work. When Claude delegates to a subagent, the subagent’s full investigation stays in its own context. Only the summary returns to your main conversation.

  2. Check context usage with /context to see how much headroom you have before hitting the limit.

Issue 5: “Claude Makes Wrong Tool Choices”¶

Symptom: You ask Claude to search for a pattern in the codebase and it reads files one by one instead of using Grep. Or you ask it to edit a file and it rewrites the entire file instead of making a targeted edit.

Cause: Ambiguous prompts. “Find where we handle auth” could mean search the codebase (Grep), read a specific file (Read), or explain the architecture (analysis). Claude guesses, and sometimes guesses wrong.

Fix:

  1. Be explicit about the tool you want:
# Vague — Claude guesses
"Find the auth handler"

# Explicit — Claude uses Grep
"Search the codebase for functions that handle authentication"

# Explicit — Claude uses Read
"Read src/auth/handler.py and explain the login flow"
  1. If Claude picks the wrong tool mid-conversation, correct it once. Claude learns from in-session corrections:
"Don't read each file individually. Use Grep to search across the codebase."
  1. In skills that execute multi-step workflows, specify the tools in the SKILL.md instructions:
markdown
## Step 1: Search
Use Grep to find all files containing the error pattern.

## Step 2: Analyze
Read the top 3 matching files to identify root cause.

Issue 6: “Settings Not Taking Effect”¶

Symptom: You changed a setting in settings.json — added a permission, modified a hook, changed the effort level — but Claude Code behaves as if nothing changed.

Cause: Three possibilities. First, invalid JSON: a missing comma or unclosed brace silently invalidates the entire file. Second, precedence: settings.local.json overrides settings.json, and a conflicting local setting wins. Third, the setting name is wrong — JSON keys are case-sensitive and must match the schema exactly.

Fix:

  1. Validate JSON syntax:
bash
python3 -c "import json; json.load(open('.claude/settings.json')); print('Valid')"
python3 -c "import json; json.load(open('.claude/settings.local.json')); print('Valid')"

  1. Check for conflicts between local and project settings. The precedence order is: Managed (enterprise) > Local > Project > User. If settings.local.json has "disableAllHooks": true, no hooks from settings.json will fire.

  2. Run the workspace validator:

bash
bash tools/scripts/validate-workspace.sh
  1. Check the schema for correct key names: https://json.schemastore.org/claude-code-settings.json

Issue 7: “MCP Server Fails to Connect”¶

Symptom: You registered an MCP server in settings, but Claude Code can’t find its tools. You see errors like “server failed to start” or “command not found” in the logs.

Cause: MCP servers run as child processes spawned by Claude Code. If the server’s executable isn’t in the PATH, or if the server writes debug output to stdout (contaminating the JSON-RPC channel), or if the server crashes on startup, the connection fails silently.

Fix:

  1. Verify the command exists and runs:
bash
# If your MCP config says: "command": "node", "args": ["server.js"]
node server.js  # Does it start without errors?

# If it says: "command": "python3", "args": ["-m", "my_mcp_server"]
python3 -m my_mcp_server  # Does it start?
  1. Check that the server only writes JSON-RPC to stdout. Any print() or console.log() in the server’s main path will corrupt the protocol. Use stderr for debug logging:
python
# Wrong — breaks MCP
print("Server started")

# Right — stderr doesn't interfere
import sys
print("Server started", file=sys.stderr)
  1. Use absolute paths in the MCP configuration to avoid PATH issues:
json
{
  "mcpServers": {
    "my-server": {
      "command": "/usr/local/bin/node",
      "args": ["/Users/me/mcp-servers/my-server/index.js"]
    }
  }
}
  1. Check for version mismatches. The MCP SDK evolves rapidly. Ensure your server uses a compatible version of @modelcontextprotocol/sdk or mcp (Python).

Issue 8: “Agent Tool Returns Unhelpful Results”¶

Symptom: You delegate to a subagent — the security reviewer or debugger — and the response is shallow, generic, or misses the point entirely.

Cause: Subagents inherit the prompt you give them, plus their system prompt from .claude/agents/. If your delegation prompt is vague (“check this code”), the agent lacks direction. Additionally, agents have limited tool access defined in their frontmatter, and may not have the tools needed for your specific task.

Fix:

  1. Be specific in your delegation:
# Vague — agent flounders
"Use the security-reviewer agent to check the code"

# Specific — agent delivers
"Use the security-reviewer agent to audit src/auth/oauth.py
for OWASP Top 10 issues, specifically injection and broken
access control. Check all user inputs that reach the database."
  1. Verify the agent has the right tools. If your security reviewer needs to run commands but only has Read, Grep, Glob, it can’t execute dynamic analysis:
yaml
# In .claude/agents/security-reviewer.md
tools: Read, Grep, Glob, Bash  # Add Bash if needed
  1. Add maxTurns to prevent agents from spinning without progress:
yaml
maxTurns: 15  # Stop after 15 tool calls
  1. If the agent’s system prompt is too generic, add domain-specific instructions. The more context the agent has about what “good” looks like, the better its output.

Issue 9: “Claude Code Is Slow”¶

Symptom: Responses take 15-30 seconds. Tool calls feel sluggish. The overall interaction pace drags compared to what you’ve seen in demos or other sessions.

Cause: Three main factors. First, the Opus model is inherently slower than Sonnet — it’s doing more reasoning. Second, large context windows (approaching 200K tokens) increase latency on every turn. Third, complex tool chains with many sequential calls add round-trip time.

Fix:

  1. Switch to Sonnet for simple tasks:
/model sonnet

Use Opus for complex reasoning (debugging, security audits, architecture). Use Sonnet for straightforward tasks (file edits, simple searches, formatting).

  1. Use /effort low for trivial tasks:
/effort low
"Rename the variable from `usr` to `user` in auth.py"

  1. Use /fast to enable streaming optimizations without changing models.

  2. Reduce context size. /clear between tasks. /compact mid-task. Read files with line ranges instead of dumping entire files.

  3. Parallelize where possible. “Review security AND performance” spawns two agents simultaneously, faster than sequential reviews.

Issue 10: “Permission Denied Errors”¶

Symptom: Claude Code refuses to run a command or read a file, saying it’s not permitted. You didn’t intend to block that action.

Cause: The deny list in settings.json is too broad. A rule like Bash(rm *) blocks every command starting with rm, including safe ones. Or Read(./config/*) blocks all config files when you only meant to protect secrets.

Fix:

  1. Check what’s in the deny list:
bash
python3 -c "
import json
s = json.load(open('.claude/settings.json'))
for d in s.get('permissions', {}).get('deny', []):
    print(d)
"
  1. Use narrow, specific deny rules:
json
{
  "deny": [
    "Read(./.env)",
    "Read(./.env.production)",
    "Read(./secrets/*)",
    "Read(./credentials.json)"
  ]
}

  1. Remember the evaluation order: deny > ask > allow. A deny rule always wins over an allow rule for the same pattern.

  2. Use settings.local.json to override for your personal workflow without affecting the team:

json
{
  "permissions": {
    "allow": ["Bash(*)"]
  }
}

Part 2: Frequently Asked Questions¶

Twenty questions that come up in every onboarding, every team Slack channel, every “getting started” session. Concise answers.

Q1: How much does Claude Code cost per day?¶

It depends on usage, but a typical professional developer using Claude Code with the Opus model for 4-6 hours of active coding spends $15-40/day. Heavy users doing complex debugging, security audits, and multi-agent workflows can hit $50-80/day. Sonnet usage runs roughly 5-7x cheaper than Opus for the same token volume. Use /stats to see your actual usage.

Q2: Do I need my own API key?¶

If you’re using the Claude Code subscription plan (Max), your usage is included. If you’re using the API-key mode, yes — you need an Anthropic API key set as ANTHROPIC_API_KEY in your environment. Teams can also use centralized billing through Anthropic’s admin console.

Q3: Can I use Claude Code with AWS Bedrock or Google Vertex?¶

Yes. Set the appropriate environment variables:

bash
# For Bedrock
export CLAUDE_CODE_USE_BEDROCK=1
export AWS_REGION=us-east-1

# For Vertex
export CLAUDE_CODE_USE_VERTEX=1
export CLOUD_ML_REGION=us-central1
export ANTHROPIC_VERTEX_PROJECT_ID=my-project

Claude Code detects these and routes requests through your cloud provider instead of Anthropic’s API directly.

Q4: Does Claude Code have internet access?¶

Yes. Claude Code can use WebSearch and WebFetch tools to search the web and fetch URLs. It can also run curl, wget, or any network command via Bash. MCP servers can connect to external APIs. However, fetching authenticated pages (behind login walls) requires either an MCP server with credentials or the dev-browser tool for browser-based authentication.

Q5: How do I share my Claude Code setup with my team?¶

Commit these files to your repository:

  • .claude/settings.json — Shared permissions, hooks, guardrails
  • .claude/rules/ — Coding standards
  • .claude/skills/ — Workflow definitions
  • .claude/agents/ — Subagent configurations
  • CLAUDE.md — Project instructions

Keep .claude/settings.local.json in .gitignore for personal overrides. When a teammate clones the repo, Claude Code automatically picks up the shared configuration.

Q6: Can Claude Code edit multiple files at once?¶

Yes. Claude can edit as many files as needed in a single turn. For large-scale changes across many files, use /batch which spawns parallel worktrees — each making changes independently, then merging results. As discussed in Chapter 22, this is the fastest path for cross-cutting refactors.

Q7: How do I prevent Claude from reading sensitive files?¶

Add deny rules to .claude/settings.json:

json
"deny": [
  "Read(./.env)",
  "Read(./.env.*)",
  "Read(./secrets/*)",
  "Read(./**/*.pem)",
  "Read(./credentials.json)"
]

Deny rules are evaluated first and cannot be overridden by allow rules.

Q8: What’s the maximum file size Claude Code can read?¶

Claude Code can read files of any size, but it loads content into the context window, which has a finite token limit. Practically, files over 10,000 lines should be read with line ranges (offset and limit parameters). For binary files, images, and PDFs, the tools handle conversion automatically, but large PDFs should be read in page ranges of 20 or fewer.

Q9: Can I use Claude Code in CI/CD pipelines?¶

Yes. Claude Code runs in headless mode when there’s no TTY attached. Use the --print flag for non-interactive execution:

bash
claude -p "Review the diff in this PR and output a summary" --output-format json

Set CLAUDE_CODE_HEADLESS=1 for pipeline environments. This is commonly used for automated code review, changelog generation, and pre-merge security checks.

Q10: How do I understand why Claude made a particular decision?¶

Ask it. Claude Code retains the full conversation context. Simply ask: “Why did you choose that approach?” or “What alternatives did you consider?” For more systematic tracking, use a decisions.md file (see Expert Tip 9) or the memory system’s save_decision function.

Q11: Can Claude Code work offline?¶

No. Claude Code requires an internet connection to communicate with Anthropic’s API (or Bedrock/Vertex). The models run in the cloud. Local tools (file reads, grep, bash) work locally, but the reasoning engine itself is remote. There is no offline mode.

Q12: How do I reset Claude Code to factory defaults?¶

Remove your configuration files:

bash
rm -rf .claude/          # Project-level config
rm -rf ~/.claude/         # User-level config (careful — this includes memory)

For a lighter reset, just delete settings.local.json to remove personal overrides while keeping team config intact.

Q13: Can I use custom or fine-tuned models?¶

Claude Code supports model selection among Anthropic’s model family (Opus, Sonnet, Haiku). You can switch models per-session (/model sonnet), per-skill (in SKILL.md frontmatter), or per-agent (in agent frontmatter). Custom fine-tuned models are not currently supported through Claude Code.

Q14: How do I export a session?¶

Use the /export command:

/export session-debug-auth.txt

This saves the full conversation — your prompts, Claude’s responses, and tool outputs — to a text file. Useful for post-mortems, documentation, or sharing debugging sessions with teammates.

Q15: What data gets sent to Anthropic?¶

Your prompts, file contents that Claude reads, tool outputs, and conversation history are sent to Anthropic’s API for processing. Anthropic’s data retention policy applies — on the API tier, data is not used for training by default. Check your contract and Anthropic’s current privacy policy for specifics. For sensitive environments, use Bedrock or Vertex to keep data within your cloud account.

Q16: Can Claude Code run destructive commands?¶

By default, Claude Code asks for confirmation before running any command not in the allow list. The block-destructive.sh hook (Chapter 19) adds a hard block on patterns like rm -rf /, DROP TABLE, and git push --force main. You can further restrict or relax these in settings, but the defaults are designed to prevent irreversible mistakes.

Q17: How do I undo something Claude did?¶

For file changes: Claude Code edits files on disk. Use git diff to see what changed and git checkout -- <file> to revert. For commits: git reset HEAD~1 undoes the last commit while keeping changes staged. For multi-step changes: /rewind rolls the conversation back to an earlier state, though it doesn’t automatically revert file changes on disk.

Q18: Can I restrict Claude Code to specific directories?¶

Not directly through a single setting, but you can achieve this through deny rules:

json
"deny": [
  "Read(/etc/*)",
  "Read(/var/*)",
  "Bash(cd /etc*)",
  "Bash(cd /var*)"
]

For project isolation, use worktrees (Chapter 22) which confine Claude to a specific branch and directory.

Q19: What’s the cheapest way to use Claude Code?¶

Use Sonnet as your default model (/model sonnet). Use /effort low for simple tasks. Use /clear between tasks to avoid context bloat (large contexts cost more per turn). Use /compact to compress mid-session. Reserve Opus for complex reasoning tasks where the quality difference justifies the cost. A Sonnet-heavy workflow can run 5-7x cheaper than all-Opus.

Q20: What’s the difference between /fast and /effort low?¶

/fast enables streaming and output optimizations without reducing reasoning depth. Claude still thinks thoroughly but delivers results faster through output pipeline improvements. /effort low actually reduces reasoning depth — Claude spends less time deliberating, which is appropriate for trivial tasks (renaming variables, simple formatting) but produces shallower analysis for complex problems. Use /fast when you want speed without sacrificing quality. Use /effort low when the task genuinely doesn’t require deep thought.

Part 3: Expert Tips¶

Twenty-five tips from daily use. Each one earns its place.

Tip 1: Use /btw for side questions. When you’re mid-debugging and want to ask “what does this regex do?” without polluting your main context, prefix with /btw. Claude answers the side question in a fork that doesn’t affect your primary conversation flow.

Tip 2: Warm-start hooks with a session-setup script. The SessionStart hook runs once when Claude Code launches. Use it to set environment variables, check tool availability, and print a welcome message. This ensures every session starts in a known-good state:

bash
#!/usr/bin/env bash
set -euo pipefail
export PYTHONDONTWRITEBYTECODE=1
export RUFF_CACHE_DIR="/tmp/ruff-cache"
command -v ruff >/dev/null 2>&1 || echo "Warning: ruff not installed" >&2

Tip 3: Build a /today skill. Create a skill that fetches your GitHub notifications, open PRs, assigned issues, and calendar (via MCP) in one command. Start every morning with /today instead of checking five different apps.

Tip 4: Use CronCreate patterns for polling. The /loop command runs a prompt on a recurring interval. Use it for deployment monitoring:

/loop 5m "Check if the deploy to staging succeeded using gh run list"

Tip 5: Build a /standup skill. A skill that reads your git log from the last 24 hours, summarizes what you did, checks your open PRs, and formats it for your team standup. Five seconds instead of five minutes of context-switching.

Tip 6: Use worktree isolation for risky experiments. When Claude is about to make sweeping changes, ask it to work in a worktree. The changes happen on a separate branch in a separate directory. If they fail, you discard the worktree with zero impact on your main branch. As covered in Chapter 22, this is essential for /batch operations.

Tip 7: Auto-format with PostToolUse hooks. The auto-lint-python.sh hook runs ruff after every Python file edit. Build equivalents for your stack:

json
{
  "matcher": "Edit|Write",
  "hooks": [{
    "type": "command",
    "command": "./.claude/hooks/auto-format-ts.sh",
    "timeout": 10000
  }]
}

This guarantees every file Claude touches is formatted correctly, without Claude spending tokens on formatting.

Tip 8: Use /compact with a focus topic. Don’t just run /compact — tell it what to keep:

/compact authentication flow

Claude compresses the conversation but preserves everything related to authentication. Without a focus topic, it guesses what’s important and sometimes guesses wrong.

Tip 9: Maintain a decisions.md file. For long-running projects, keep a file where Claude logs architectural decisions:

markdown
## 2026-03-15: Chose JWT over session cookies
**Context:** Need stateless auth for microservices
**Decision:** JWT with 15min expiry, refresh tokens in httpOnly cookies
**Trade-offs:** Revocation is harder, but avoids shared session store

Reference it in CLAUDE.md so Claude reads it every session and maintains consistency.

Tip 10: Mix models in a single session. Start with Sonnet for file exploration and understanding, then switch to Opus for the actual fix:

/model sonnet
"Read through src/auth/ and map out the authentication flow"

/model opus
"Now fix the race condition in the token refresh logic"

You get Sonnet’s speed for the exploratory phase and Opus’s reasoning depth for the surgical fix.

Tip 11: Use MCP servers for internal tools. If your team has internal APIs — deployment systems, feature flags, incident management — wrap them in MCP servers. Claude Code can then deploy, toggle flags, and check incident status without you writing bash commands or remembering API endpoints.

Tip 12: Deny rules for production credentials. Beyond denying .env files, deny access to production credential paths specifically:

json
"deny": [
  "Read(./.env.production)",
  "Read(./config/prod/*)",
  "Bash(*--env production*)",
  "Bash(*PROD_*)"
]

This creates a hard barrier even if you accidentally ask Claude to “check the production config.”

Tip 13: Use /batch with worktrees for cross-cutting changes. When you need to update 50 files with the same pattern change — renaming an API field, updating error handling, migrating to a new library — /batch parallelizes across worktrees. Each worktree handles a subset, and results merge automatically. A change that takes 30 minutes sequentially completes in 5.

Tip 14: Leverage auto-memory for cross-session continuity. Claude Code’s memory system persists insights across sessions. When you solve a tricky bug, the root cause and fix are remembered. Next time a similar issue appears, Claude recalls the pattern. Verify memory is enabled:

json
"autoMemoryEnabled": true

You can also manually save critical facts with the memory MCP tools.

Tip 15: Set maxTurns on agents to prevent runaway investigations. Without a limit, a subagent can burn through tokens chasing dead ends. Set a reasonable cap:

yaml
---
name: debugger
maxTurns: 20
---

If the agent can’t solve it in 20 turns, it reports what it found and returns control. You can then redirect with better context.

Tip 16: Use context: fork on expensive skills. Skills with context: fork run in an isolated subagent context. The main conversation only receives the final output, not every intermediate tool call. This keeps your primary context clean and reduces token consumption for investigation-heavy workflows.

Tip 17: Pin skill models for consistency. If your security-audit skill should always use Opus (because Sonnet misses subtle vulnerabilities), pin it in the frontmatter:

yaml
---
name: security-audit
model: opus
effort: high
---

This overrides whatever model the main session is using, ensuring critical workflows always get the reasoning depth they need.

Tip 18: Use dynamic context injection in skills. The !`command` syntax in SKILL.md runs shell commands before the skill loads, injecting their output as context:

markdown
## Current state
- Branch: !`git branch --show-current`
- Recent commits: !`git log --oneline -5`
- Failing tests: !`python -m pytest --tb=no -q 2>&1 | tail -5`

Claude starts with all this context loaded, saving 3-4 tool calls per invocation.

Tip 19: Use UserPromptSubmit hooks for input validation. If your team has conventions about how prompts should be structured (issue numbers must include #, file paths must be absolute), validate at the prompt level:

json
{
  "UserPromptSubmit": [{
    "hooks": [{
      "type": "command",
      "command": "./.claude/hooks/validate-prompt.sh"
    }]
  }]
}

The hook can reject malformed prompts before they waste a model turn.

Tip 20: Create project-specific .claude/ configs. Every repository can have its own .claude/ directory with rules, skills, and settings tailored to that project. When you cd into the repo, Claude Code loads that project’s configuration automatically. A Python microservice, a React frontend, and an infrastructure repo can each have completely different rules.

Tip 21: Use the Stop hook as a quality gate. The Stop event fires before Claude finalizes its response. Use it to verify that outputs meet your standards — for example, checking that code review findings include file:line references, or that commit messages follow conventional commits format.

Tip 22: Leverage --output-format json for toolchain integration. When using Claude Code in scripts or CI/CD, JSON output is parseable by downstream tools:

bash
result=$(claude -p "List all TODO comments in src/" --output-format json)
echo "$result" | jq '.todos[] | .file + ":" + .line'

This lets you build Claude Code into automated workflows where its output feeds into other systems.

Tip 23: Use paths: frontmatter to auto-activate skills. Instead of typing /api-conventions every time you edit an API file, make the skill auto-activate:

yaml
---
name: api-conventions
paths:
  - "src/api/**"
  - "**/routes/**"
  - "**/endpoints/**"
user-invocable: false
---

Claude loads the skill automatically when you touch matching files. Zero friction.

Tip 24: Keep a runbooks/ directory for operational procedures. Document your team’s recurring procedures — deployments, incident response, database migrations — as runbooks. Reference them in CLAUDE.md:

markdown
@docs/runbooks/deploy.md
@docs/runbooks/incident-response.md

When someone says “deploy to staging,” Claude follows the documented procedure instead of improvising.

Tip 25: Audit your setup quarterly. Run the workspace validator every few months:

bash
bash tools/scripts/validate-workspace.sh

Check for: skills that nobody uses (remove them to reduce cognitive load), hooks that fire too often (tighten matchers), rules that contradict each other (merge or prioritize), and deny rules that are too broad (relax to match actual threats). A lean configuration is a fast configuration.
