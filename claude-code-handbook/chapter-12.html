Chapter 12: Agent Tool and Multi-Agent Patterns¶ ~18 min read

You need to refactor an authentication system that spans 47 files across three services. Reading all 47 files in your main session would consume half your context window before you write a single line of code. Or you need a security review and a performance audit at the same time, and each requires deep investigation. These are the problems the Agent tool solves.

The Agent tool spawns a child Claude instance – a subagent – that operates in its own context window with its own tool access. It explores, analyzes, or implements independently, then reports back. The main session stays clean. And when you launch multiple agents in parallel, you get concurrent workstreams executing simultaneously.

This chapter covers how the Agent tool works, when to use it, how to write effective agent prompts, and the multi-agent patterns that handle complex engineering tasks.

What the Agent Tool Does¶

When Claude Code calls the Agent tool, it creates a new Claude session – a separate instance with its own context window, its own tool calls, and its own reasoning. The parent session provides a prompt describing the task. The child session executes it and returns a result.

The flow:

Main session                    Subagent session
     │                               │
     ├── Agent(prompt: "...")  ──────>│
     │                               ├── Grep(...)
     │   [main waits or               ├── Read(...)
     │    continues other work]        ├── Read(...)
     │                               ├── Grep(...)
     │                               ├── [analyzes findings]
     │<──────── result ──────────────┤
     │                               └── [session ends]
     ├── [uses result to proceed]
     │

The subagent has full tool access (or restricted access, depending on its type). It can Read files, run Grep searches, execute Bash commands, even spawn its own subagents. When it finishes, its final response becomes the Agent tool’s result in the parent session.

Why This Matters¶

Context protection. Every file Read, every Grep result, every Bash output consumes tokens in your context window. Exploration often requires reading 10-20 files to find the 2-3 that matter. An agent does that exploration in its own context. Only the summary comes back to the main session.

Parallel execution. You can launch multiple agents simultaneously for independent tasks. A security review agent and a performance review agent running in parallel complete in the time of the slower one, not the sum of both.

Specialized configuration. Different agents can use different models. Use Opus for security analysis, Sonnet for code quality checks, Haiku for simple file searches. Each task gets the right model at the right cost.

Agent Types¶

The subagent_type parameter configures the agent’s capabilities and restrictions.

general-purpose¶

The default type. Full tool access. Use for any task that requires reading, writing, running commands, or making changes.

Agent(
  prompt: "Refactor the PaymentService class in src/services/payment.ts to use the Strategy pattern. Create separate strategy files for each payment provider. Write tests for the new structure.",
  subagent_type: "general-purpose"
)

Tools available: Read, Edit, Write, Glob, Grep, Bash, WebFetch, WebSearch, NotebookEdit, Agent (can spawn sub-subagents)

Explore¶

A read-only agent optimized for codebase investigation. No write tools – it cannot modify files or run destructive commands. This makes it safe to let loose on a codebase.

Agent(
  prompt: "Map the entire authentication flow from the login endpoint to the session creation. Trace every function call, identify all database queries, and note any external service calls. Report as: file:line → function → description.",
  subagent_type: "Explore"
)

Tools available: Read, Glob, Grep, Bash (read-only commands)

When to use Explore: When you need to understand code without changing it. Tracing call chains, mapping dependencies, finding patterns, answering questions about the codebase.

Plan¶

An architecture and planning agent. Like Explore, it has no write tools, but it is oriented toward producing plans, architectures, and strategies rather than raw code findings.

Agent(
  prompt: "Design the database migration strategy for splitting the monolithic users table into separate users, profiles, and preferences tables. Consider: data migration approach, backward compatibility, rollback plan, and timeline.",
  subagent_type: "Plan"
)

Tools available: Read, Glob, Grep

claude-code-guide¶

A specialized agent that answers questions about Claude Code itself. It has access to internal documentation and can explain features, configuration, and behavior.

Agent(
  prompt: "How do PreToolUse hooks work? What JSON format do they receive on stdin? What exit codes control behavior?",
  subagent_type: "claude-code-guide"
)

statusline-setup¶

A utility agent for configuring Claude Code’s status line display. Rarely used directly.

When to Use Agent¶

Use Agent When:¶

The task requires extensive exploration. If you need to search through many files, follow call chains, or investigate a pattern across the codebase, an agent keeps that exploration noise out of your main context.

Agent(
  prompt: "Find all places in the codebase where we create database connections. For each: (1) Is the connection properly closed? (2) Is there a timeout set? (3) Is there error handling? Report as a table: file:line | connection type | properly closed | has timeout | has error handling.",
  subagent_type: "Explore"
)

Multiple independent tasks can run in parallel. If tasks A and B do not depend on each other, launch both as agents and get results simultaneously.

You need a different model for a subtask. Security reviews benefit from Opus’s deeper reasoning. Simple file searches work fine with Haiku. Agents let you match the model to the task.

Investigation would pollute main context. Reading 15 files to understand one bug fills your context with code that is irrelevant after the investigation. An agent absorbs that cost.

Do NOT Use Agent When:¶

The task involves 1-3 files. Reading two files and making an edit is faster done directly. The overhead of spawning an agent (new session, context initialization) is not worth it for simple operations.

The result is a simple lookup. “What line is the authenticate function defined on?” – just use Grep directly. Do not spawn an agent for a single search.

You need tight back-and-forth control. Agents execute autonomously. You give them a prompt, they execute, they return. If the task requires iterative human feedback at each step, do it in the main session.

The task requires sequential human decisions. “Read this file, show me the options, I’ll pick one, then proceed” – this is a main-session workflow, not an agent workflow.

Agent Parameters¶

Parameter Type Description Default
prompt string (required) Complete task description –
subagent_type string Agent type “general-purpose”
model string Override model Parent’s model
run_in_background boolean Non-blocking execution false
isolation string “worktree” for git isolation none

The prompt Parameter¶

This is the most important parameter. The agent sees your prompt as its primary instruction. Everything it needs to know must be in this prompt or discoverable through its tools.

The model Parameter¶

Override the model for cost or capability optimization:

# Use Opus for security-critical analysis
Agent(
  prompt: "Audit src/auth/ for OWASP Top 10 vulnerabilities...",
  model: "opus"
)

# Use Haiku for simple file-finding tasks
Agent(
  prompt: "List all Python files that import the requests library",
  model: "haiku"
)

The run_in_background Parameter¶

When true, the agent runs without blocking the main session. The main session can continue working while the agent executes. When the agent finishes, the result arrives as a notification.

# Start a background investigation
Agent(
  prompt: "Map all external API calls in the codebase...",
  subagent_type: "Explore",
  run_in_background: true
)

# Main session continues immediately -- can do other work
# Agent result arrives when it finishes

Use background execution when: - The agent task is independent of your current work - The task is long-running and you want to continue working - You are launching multiple agents and do not need all results at once

Use foreground execution (default) when: - You need the result before proceeding - The agent’s findings determine your next action

The isolation Parameter¶

Setting isolation: "worktree" creates a temporary git worktree for the agent. The agent gets its own copy of the repository and can make changes without affecting your working directory.

Agent(
  prompt: "Refactor the database layer to use the repository pattern. Make all necessary changes, including tests.",
  isolation: "worktree"
)

The worktree is created inside .claude/worktrees/ with a new branch based on the current HEAD. After the agent finishes: - You can review the changes in the worktree - Merge the branch if the changes look good - Discard the worktree if the approach was wrong

Use worktree isolation for: - Experimental refactoring you might discard - Risky changes that could break the codebase - Parallel development of independent features - “What if” explorations

Writing Effective Agent Prompts¶

The quality of an agent’s output depends entirely on the quality of its prompt. An agent cannot ask you clarifying questions mid-execution. It cannot read your mind about context you have but have not shared. Everything must be in the prompt.

Include WHAT and WHY¶

# BAD: No context, no goal
Agent(prompt: "Look at the auth module")

# GOOD: Clear task with purpose
Agent(
  prompt: "Review src/auth/ for security vulnerabilities. We recently had a report of session fixation attacks. Check: (1) How are session IDs generated? (2) Are sessions invalidated on login? (3) Are session IDs rotated after privilege changes? Report findings with CWE references and severity ratings."
)

Include Known Context¶

If you already know relevant facts, include them. Do not make the agent rediscover what you already know.

# BAD: Agent must rediscover everything
Agent(prompt: "Fix the bug where users can't log in")

# GOOD: Agent starts with your knowledge
Agent(
  prompt: "Fix the login failure bug. Known facts: (1) The error is in src/auth/login.ts around line 45. (2) The bcrypt.compare call is receiving the hash and password in the wrong order. (3) The test in tests/test_auth.py::test_valid_login is failing with 'invalid credentials'. Write the fix and verify the test passes."
)

Specify the Desired Output Format¶

Agent(
  prompt: "Audit the dependency list in package.json. For each dependency, check: (1) Latest version available. (2) Any known CVEs. (3) Whether our version is pinned. Output a markdown table: | Package | Current | Latest | Pinned? | CVEs | Action |"
)

Do Not Delegate Understanding¶

# BAD: "Based on your findings" assumes shared context
Agent(prompt: "Investigate the auth module. Based on your findings, fix whatever is wrong.")

# GOOD: Separate investigation from action, or be specific
Agent(
  prompt: "Investigate all authentication paths in src/auth/. Report: (1) List every authentication method (password, OAuth, API key, etc.). (2) For each, trace the flow from request to session creation. (3) Identify any path where authentication can be bypassed. DO NOT make any changes -- just report findings.",
  subagent_type: "Explore"
)
# Then in main session, use findings to make targeted fixes

Set Boundaries¶

Agent(
  prompt: "Search for all uses of the deprecated `fetchData` function. Only search in src/ -- ignore node_modules, dist, and test directories. Report as a list of file:line entries. Do not modify any files.",
  subagent_type: "Explore"
)

Parallel Agent Patterns¶

Pattern 1: Parallel Code Review¶

Launch multiple reviewers simultaneously, each focused on a different concern:

# Security review (Opus for depth)
Agent(
  prompt: "Security review of src/api/. Check for: injection, auth bypass, data exposure, SSRF. Output: severity | CWE | file:line | description | fix.",
  model: "opus",
  run_in_background: true
)

# Performance review (Sonnet for speed/cost balance)
Agent(
  prompt: "Performance review of src/api/. Check for: N+1 queries, missing indexes, unbounded result sets, blocking I/O in async code. Output: impact | file:line | description | fix.",
  model: "sonnet",
  run_in_background: true
)

# Code quality review (Sonnet)
Agent(
  prompt: "Code quality review of src/api/. Check for: dead code, duplicated logic, unclear naming, missing error handling, type safety gaps. Output: priority | file:line | description | suggestion.",
  model: "sonnet",
  run_in_background: true
)

Three reviews running simultaneously. Each agent has its own context window, its own model, and its own focus area. Results arrive as each finishes.

Pattern 2: Investigation Then Fix¶

Use an Explore agent to investigate, then fix in the main session:

# Step 1: Investigation (agent)
Agent(
  prompt: "The /api/users endpoint is returning 500 errors intermittently. Investigate: (1) Read the route handler in src/api/users.ts. (2) Check the database query in src/db/users.ts. (3) Look at error logs if available. (4) Trace the full request path. Report: root cause, affected file:line, and recommended fix.",
  subagent_type: "Explore"
)

# Step 2: Fix (main session, using agent's findings)
Read(file_path: "/project/src/db/users.ts")
Edit(file_path: "/project/src/db/users.ts", old_string: "...", new_string: "...")
Bash(command: "python -m pytest tests/test_users.py -v")

The investigation agent reads 10+ files and traces the call chain. Only its summary enters the main context. The fix is then precise and targeted.

Pattern 3: Multi-File Refactor with Worktrees¶

For a large refactoring that touches multiple subsystems:

# Agent 1: Refactor auth subsystem
Agent(
  prompt: "Refactor src/auth/ to use the new TokenService interface. Update all functions to use dependency injection. Write tests.",
  isolation: "worktree",
  run_in_background: true
)

# Agent 2: Refactor payments subsystem
Agent(
  prompt: "Refactor src/payments/ to use the new TokenService interface. Update all functions to use dependency injection. Write tests.",
  isolation: "worktree",
  run_in_background: true
)

# Agent 3: Update API layer
Agent(
  prompt: "Update src/api/ to pass the TokenService to auth and payment handlers. Wire up dependency injection at the route level. Write integration tests.",
  isolation: "worktree",
  run_in_background: true
)

Each agent works in its own git worktree. No conflicts. When all three finish, you review each branch and merge them in order.

Pattern 4: Research Then Implement¶

# Step 1: Research (background agent)
Agent(
  prompt: "Research the best rate-limiting library for Express.js in 2026. Compare: express-rate-limit, rate-limiter-flexible, and @nestjs/throttler (if applicable to Express). Check: maintenance status, features, performance benchmarks. Recommend one with rationale.",
  subagent_type: "Explore",
  run_in_background: true
)

# While research runs, prepare the codebase
Read(file_path: "/project/src/api/middleware/index.ts")
# ... main session continues working

# When research result arrives, implement the recommendation

Communicating with Running Agents¶

Claude Code supports sending messages to previously spawned agents using the SendMessage capability. This allows you to provide additional context or redirect an agent that is already running.

The key requirement: the agent must still be active (not yet returned its final response). Once an agent has completed, you would need to spawn a new one.

This is useful when: - You realize the agent needs additional context you forgot to include - The agent’s intermediate output (visible in streaming) suggests it is going in the wrong direction - You want to narrow the scope after seeing initial findings

Custom Agents from .claude/agents/¶

As discussed in Chapter 6 and Chapter 8, Claude Code supports custom agent definitions in .claude/agents/. These are markdown files with YAML frontmatter that define specialized agents. The key distinction from ad-hoc Agent tool calls: custom agents have a persistent identity, a model preference, tool restrictions, and a system prompt.

When Custom Agents Beat Ad-Hoc Agents¶

Scenario Use Why
Recurring task (weekly security review) Custom agent Same prompt, same model, same tools every time
One-off investigation Ad-hoc Agent call Not worth creating a persistent definition
Team-shared expertise (company-specific patterns) Custom agent Checked into git, shared via settings
Specialized model requirement (always use Opus) Custom agent Model is defined in frontmatter, not forgotten

Example: A Database Migration Agent¶

markdown
---
name: migration-expert
description: Expert in database migrations. Use when creating or reviewing
  schema changes, migration scripts, or data transformations.
tools: Read, Grep, Glob, Bash
model: sonnet
effort: high
---

You are a database migration expert. When invoked:

1. Read the current schema (check src/db/schema/ or migrations/)
2. Understand the requested change
3. Generate a migration script that:
   - Is reversible (include both up and down)
   - Handles large tables with zero-downtime patterns
   - Includes data backfill if needed
   - Has appropriate indexes
4. Generate a test for the migration
5. Verify the migration runs against the test database

Always check for:
- Foreign key constraints that might block the migration
- Large table concerns (> 1M rows: use batched operations)
- Nullable vs NOT NULL with defaults
- Index creation concurrency (CREATE INDEX CONCURRENTLY in PostgreSQL)

Claude Code delegates to this agent whenever it encounters migration-related tasks, using the right model, the right tools, and the right approach every time.

Agent Composition: Agents Calling Agents¶

A general-purpose agent can spawn its own subagents. This creates a hierarchy:

Main session
├── Agent: "Review this PR for security and quality"
│   ├── Sub-agent (Opus): Security review
│   └── Sub-agent (Sonnet): Code quality review
└── [Main session continues after results arrive]

This is powerful but expensive. Each layer multiplies token costs. Use agent composition for genuinely complex tasks where each level adds meaningful specialization.

Worktree Workflows in Depth¶

The worktree isolation feature deserves deeper exploration because it enables workflows that are impossible (or risky) without it.

What Happens When You Use isolation: “worktree”¶

  1. Claude Code creates a new directory under .claude/worktrees/ with a unique name
  2. A new git branch is created based on the current HEAD
  3. A git worktree is checked out at that branch
  4. The agent’s working directory is set to the worktree
  5. All file operations (Read, Edit, Write, Bash) operate on the worktree copy
  6. The original working directory is untouched

The Experiment-and-Merge Pattern¶

# Step 1: Agent experiments in a worktree
Agent(
  prompt: "Attempt to migrate the Express.js server to Hono. Update route handlers, middleware, error handling, and tests. This is experimental -- try your best but it may not be feasible.",
  isolation: "worktree",
  run_in_background: true
)

# Step 2: Review the result
# If the agent succeeded, a new branch exists with the changes
Bash(command: "git diff main..worktree-branch-name --stat")

# Step 3: Merge if it looks good
Bash(command: "git merge worktree-branch-name")

# Or discard if the experiment failed
Bash(command: "git worktree remove .claude/worktrees/the-worktree")
Bash(command: "git branch -D worktree-branch-name")

Parallel Feature Development¶

Multiple agents, each in their own worktree, developing different features simultaneously:

Agent(
  prompt: "Implement the user notification system in src/notifications/. Create: service, routes, tests, database migration.",
  isolation: "worktree",
  run_in_background: true
)

Agent(
  prompt: "Implement the audit logging system in src/audit/. Create: logger, middleware, storage adapter, tests.",
  isolation: "worktree",
  run_in_background: true
)

Both agents work independently. Neither’s changes affect the other or your working directory. When both finish, review and merge each branch.

Worktree Limitations¶

  • Cost: Creating a worktree is a git checkout operation plus whatever file operations the agent performs. It is fast but not free.
  • Merge conflicts: If two worktree agents modify the same files, you will have merge conflicts when combining their branches.
  • Disk space: Each worktree is a near-full copy of the working tree (though git shares the object database). Large repos consume significant space.
  • Agent context: The agent in a worktree does not have your main session’s context. It starts fresh with only its prompt.

MCP: Extending Claude Code with External Tools¶

Model Context Protocol (MCP) is the standard protocol for connecting Claude Code to external tools and data sources. While the built-in tools cover filesystem, shell, and web operations, MCP lets you add tools for anything else – databases, APIs, custom services, specialized workflows.

What MCP Is¶

MCP is a JSON-RPC protocol over stdio. An MCP server is a program that: 1. Accepts JSON-RPC messages on stdin 2. Exposes tools (functions Claude Code can call) 3. Exposes resources (data Claude Code can read) 4. Returns results on stdout

When you register an MCP server, its tools appear alongside Claude Code’s built-in tools. Claude Code can call them the same way it calls Read or Bash.

Registering MCP Servers¶

In your settings, configure MCP servers:

json
{
  "mcpServers": {
    "my-database": {
      "command": "node",
      "args": ["/path/to/db-mcp-server/index.js"],
      "env": {
        "DATABASE_URL": "postgres://localhost/mydb"
      }
    },
    "my-api": {
      "command": "python",
      "args": ["/path/to/api-mcp-server/main.py"],
      "env": {
        "API_KEY": "${API_KEY}"
      }
    }
  }
}

Each server runs as a child process. Claude Code communicates with it via stdin/stdout using the MCP protocol.

What MCP Tools Look Like¶

Once registered, MCP tools appear to Claude Code like built-in tools. If your database MCP server exposes a query tool, Claude Code can call it:

mcp__my-database__query(sql: "SELECT count(*) FROM users WHERE created_at > '2026-01-01'")

The naming convention is mcp__<server-name>__<tool-name>.

Common MCP Servers¶

Server Purpose
Database MCP Run SQL queries against your database
Memory MCP Persistent memory across sessions
GitHub MCP Enhanced GitHub operations (beyond gh CLI)
Slack MCP Read/post messages to Slack channels
Browser MCP Authenticated browser operations
Custom API MCP Interface with internal company APIs

Building a Simple MCP Server¶

A minimal MCP server in Python:

python
#!/usr/bin/env python3
"""Minimal MCP server that provides a 'greet' tool."""
import json
import sys


def handle_request(request: dict) -> dict:
    method = request.get("method", "")

    if method == "initialize":
        return {
            "protocolVersion": "2024-11-05",
            "capabilities": {"tools": {}},
            "serverInfo": {"name": "example-server", "version": "1.0.0"},
        }

    if method == "tools/list":
        return {
            "tools": [
                {
                    "name": "greet",
                    "description": "Generate a greeting for a person",
                    "inputSchema": {
                        "type": "object",
                        "properties": {
                            "name": {"type": "string", "description": "Person's name"}
                        },
                        "required": ["name"],
                    },
                }
            ]
        }

    if method == "tools/call":
        tool_name = request["params"]["name"]
        args = request["params"].get("arguments", {})
        if tool_name == "greet":
            name = args.get("name", "World")
            return {"content": [{"type": "text", "text": f"Hello, {name}!"}]}

    return {"error": {"code": -32601, "message": f"Unknown method: {method}"}}


def main() -> None:
    for line in sys.stdin:
        line = line.strip()
        if not line:
            continue
        request = json.loads(line)
        response = handle_request(request)
        response["jsonrpc"] = "2.0"
        response["id"] = request.get("id")
        print(json.dumps(response), flush=True)


if __name__ == "__main__":
    main()

Register it:

json
{
  "mcpServers": {
    "example": {
      "command": "python3",
      "args": ["/path/to/example_server.py"]
    }
  }
}

Claude Code can now call mcp__example__greet(name: "Alice"). Building production MCP servers is covered in detail in Chapter 24.

MCP vs. Bash¶

When should you use an MCP server instead of just running commands via Bash?

Scenario Use Why
One-off database query Bash(psql -c "...") Simple, no setup needed
Frequent database queries Database MCP Persistent connection, schema awareness
Read a web page WebFetch Built-in, no setup
Authenticated API with complex logic Custom MCP Handles auth, retries, pagination
Call an internal tool MCP Typed parameters, proper error handling

MCP servers make sense when you repeatedly need to interact with an external system in a structured way. For one-off operations, Bash is simpler.

Cost Implications¶

Multi-agent workflows multiply costs. Each agent is a separate API session with its own token consumption.

Cost Factors¶

Factor Impact
Number of agents Multiplied by number of parallel agents
Model per agent Opus costs more than Sonnet costs more than Haiku
Agent exploration depth More tool calls = more tokens
Context size per agent Larger contexts = more input tokens

Cost Optimization Strategies¶

Match model to task complexity:

# Expensive but necessary: security audit needs deep reasoning
Agent(prompt: "Security audit...", model: "opus")

# Cost-effective: simple file search
Agent(prompt: "Find all files importing lodash...", model: "haiku")

# Balanced: code quality review
Agent(prompt: "Code quality review...", model: "sonnet")

Scope agent prompts tightly: An agent that knows exactly where to look spends fewer tokens exploring.

# EXPENSIVE: Broad scope, agent must search the entire codebase
Agent(prompt: "Find the bug")

# CHEAP: Narrow scope, agent reads 2-3 files
Agent(prompt: "The bug is in src/auth/login.ts around line 45. The bcrypt.compare arguments are reversed. Verify and fix.")

Use Explore agents for investigation, not general-purpose: Explore agents have no write tools, so they consume fewer tokens on tool calls and make fewer mistakes that require correction.

Monitor costs: Use the /cost command to see your token spend. After a multi-agent workflow, check how much each agent consumed to calibrate future decisions.

Cost Comparison Example¶

A typical code review workflow:

Approach Estimated Cost
Single-agent review (main session, Opus) $0.50-1.50
Three parallel agents (Sonnet for each) $0.30-0.90
Three parallel agents (Opus for security, Sonnet for quality and perf) $0.40-1.20

Parallel Sonnet agents can be cheaper AND faster than a single Opus agent doing everything sequentially.

Real-World Workflow: End-to-End Bug Investigation¶

Here is a complete multi-agent workflow for a production bug:

# Step 1: User reports "API returns 500 on /api/orders for some users"

# Step 2: Launch investigation agent
Agent(
  prompt: "Investigate 500 errors on /api/orders. (1) Read the route handler. (2) Trace the database query. (3) Check for any user-specific logic that could fail. (4) Look at error handling. Report: suspected root cause, affected file:line, and what data conditions trigger the failure.",
  subagent_type: "Explore"
)

# Step 3: Agent reports back:
# "Root cause: src/db/orders.ts:78 - the query joins on user.org_id,
#  but some users have null org_id. The JOIN fails silently, and the
#  null result causes a TypeError at src/api/orders.ts:23 when
#  accessing result.rows.length."

# Step 4: Main session writes failing test
Write(
  file_path: "/project/tests/test_orders.py",
  content: "..."  # Test with a user that has null org_id
)

# Step 5: Verify test fails
Bash(command: "python -m pytest tests/test_orders.py::test_orders_null_org -v")

# Step 6: Fix the bug
Read(file_path: "/project/src/db/orders.ts")
Edit(
  file_path: "/project/src/db/orders.ts",
  old_string: "JOIN organizations ON users.org_id = organizations.id",
  new_string: "LEFT JOIN organizations ON users.org_id = organizations.id"
)

# Step 7: Add null check
Read(file_path: "/project/src/api/orders.ts")
Edit(
  file_path: "/project/src/api/orders.ts",
  old_string: "const count = result.rows.length;",
  new_string: "const count = result?.rows?.length ?? 0;"
)

# Step 8: Verify test passes
Bash(command: "python -m pytest tests/test_orders.py -v")

The agent did the expensive exploration (reading route handlers, tracing database queries, understanding the data model). The main session stayed focused: write test, fix bug, verify.

What Most People Get Wrong¶

Mistake 1: Using agents for simple tasks.

Spawning an agent to read one file and report back is like calling a meeting to ask one question. Just Read the file directly.

Mistake 2: Vague agent prompts.

“Look at the code and tell me what you think” produces vague results. Be specific about what to check, where to look, and how to format the output.

Mistake 3: Not including known context.

If you already know the bug is in src/auth/login.ts near line 45, tell the agent. Do not make it rediscover what you already know. Agent tokens are real money.

Mistake 4: Foreground agents for independent tasks.

If you launch three foreground agents sequentially, you wait for each one to finish before starting the next. Use run_in_background: true for independent tasks and let them run in parallel.

Mistake 5: Not using worktree isolation for risky changes.

An agent that modifies files in your working directory can break things you are working on. For experimental refactoring, always use isolation: "worktree".

Summary¶

The Agent tool is Claude Code’s mechanism for scaling beyond a single context window. Subagents explore codebases without polluting your context, execute in parallel for independent tasks, and use appropriate models for each job. The key is knowing when to delegate (complex exploration, parallel work, risky experiments) and when not to (simple reads, lookups, 2-3 file operations). Write agent prompts like briefings to a colleague: include what you know, be specific about what you need, and define the output format. MCP extends the tool system beyond built-in capabilities, letting you connect Claude Code to databases, APIs, and custom tools through a standard protocol.

Key takeaways: 1. Agents protect your main context from exploration noise – use them for investigation, reserve the main session for targeted action 2. Parallel agents reduce wall-clock time for independent tasks – launch them simultaneously with run_in_background: true 3. Match models to tasks: Opus for security/complexity, Sonnet for general work, Haiku for simple searches 4. Write agent prompts with full context, specific scope, and defined output format 5. MCP servers extend Claude Code’s tool system with custom tools – use them for repeated interactions with external systems
