Part V: Building Real Applications

Chapter 16: Project Architecture Patterns¶ ~15 min read

You can hand Claude Code a messy codebase and it will do its best. But “its best” in a well-structured project is categorically different from “its best” in a pile of spaghetti. This chapter is about engineering your project so that Claude Code operates at maximum effectiveness — before you write a single line of application code.

The difference is not subtle. A project with a clear CLAUDE.md, consistent directory structure, and architecture documentation lets Claude Code make accurate changes across dozens of files with minimal correction. A project without these guardrails produces code that works but drifts — inconsistent patterns, misplaced files, missed conventions. Over weeks, the drift compounds into technical debt that takes longer to fix than it would have taken to set up properly.

The CLAUDE.md-First Approach¶

Most engineers start a project by writing code. With Claude Code, you start by writing CLAUDE.md.

This is not documentation for humans. It is configuration for your AI collaborator. Think of it as the project brain — the document that tells Claude Code how to think about your codebase, what conventions to follow, and what mistakes to avoid.

The CLAUDE.md-first approach inverts the traditional workflow:

  1. Define communication rules (how Claude should talk to you)
  2. Define coding standards (language conventions, formatting, patterns)
  3. Define architecture (where things go, how components connect)
  4. Define verification (how to run tests, lint, type-check)
  5. Then start building

Here is a minimal but effective CLAUDE.md for a new TypeScript API project:

markdown
# Project: TaskFlow API

REST API for task management. TypeScript, Express, PostgreSQL, Prisma ORM.

## Communication
- Be terse. Lead with the answer
- Reference code as `file_path:line_number`
- No emojis

## Coding Standards
- TypeScript strict mode everywhere
- Async/await, never raw Promises or callbacks
- Zod for all request validation
- Prisma for all database access (never raw SQL)
- Error responses follow RFC 7807 Problem Details format

## Architecture
- src/routes/ — Express route handlers (thin: validate, call service, respond)
- src/services/ — Business logic (all complexity lives here)
- src/middleware/ — Auth, error handling, request logging
- src/prisma/ — Schema, migrations, seed data
- src/types/ — Shared TypeScript types and Zod schemas

## Verification
- Run tests: `npm test`
- Type check: `npx tsc --noEmit`
- Lint: `npx eslint src/`
- Run single test: `npx vitest run src/routes/__tests__/boards.test.ts`

## Rules
@.claude/rules/typescript.md
@.claude/rules/api-conventions.md

## Architecture
@docs/architecture.md

Notice what is not here: no lengthy explanations, no API documentation, no code patterns that Claude can infer from reading the code. The CLAUDE.md is a compass, not an encyclopedia. As discussed in Chapter 4, keeping it under 200 lines ensures the instructions get followed reliably.

When to Write CLAUDE.md¶

Write it before git init. Seriously. The first file in your repository should be CLAUDE.md, not README.md. README is for humans who visit your GitHub page. CLAUDE.md is for the AI that will write most of your code.

If you are retrofitting Claude Code onto an existing project, write the CLAUDE.md by reading the codebase first:

Read the codebase structure and conventions, then draft a CLAUDE.md
that captures the coding standards, architecture, and verification commands.

Claude Code will analyze your project and produce a first draft. Edit it for accuracy, remove anything redundant, and commit.

Directory Conventions That Help Claude Navigate¶

Claude Code reads your directory structure to understand where things belong. Consistent, predictable directory layouts let Claude place new files correctly without you specifying the path every time.

Flat vs. Deep Hierarchies¶

Flat structures (fewer levels of nesting) are easier for Claude Code to navigate. When Claude uses the Glob tool to find files, a flat structure means fewer directories to search and clearer file names.

Deep structures (many levels of nesting) provide more organization but require Claude to know the full path or search recursively. This works fine for large codebases but adds friction for small ones.

Structure Best For Claude Code Impact
src/routes/boards.ts Small to medium projects Claude finds files instantly
src/modules/boards/routes/index.ts Large monoliths with 50+ modules Claude needs architecture docs or more context
packages/api/src/routes/boards.ts Monorepos Claude needs per-package CLAUDE.md

The rule of thumb: use the shallowest structure that keeps related files together. Three levels deep is the sweet spot for most projects.

The Standard Layout¶

Regardless of language or framework, these directory names are nearly universal and Claude Code recognizes them immediately:

project/
├── CLAUDE.md              # AI project brain
├── .claude/               # Claude Code configuration
│   ├── settings.json
│   └── rules/
├── src/                   # Source code
├── tests/                 # Test files
├── docs/                  # Documentation
├── scripts/               # Build/deploy/utility scripts
├── config/                # Configuration files
└── .github/               # CI/CD workflows

Claude Code has been trained on millions of repositories using this layout. When you follow it, Claude places new files correctly by default. When you deviate, you need to document the deviation in CLAUDE.md.

When using frameworks, add a one-liner to CLAUDE.md:

markdown
## Framework
Next.js 15 with App Router. Follow standard App Router conventions for file-based routing.

That single line prevents Claude from generating Pages Router code, creating getServerSideProps, or making other version-specific mistakes.

Project Types and Their Ideal Claude Code Setups¶

Different project types need different configurations. Here are five common patterns with complete setup details.

1. Web Application (React/Next.js + API)¶

This is the most common project type. The CLAUDE.md focuses on component patterns and API conventions.

markdown
# Project: TaskFlow

Full-stack task management app. Next.js 15 (App Router), TypeScript, Prisma, PostgreSQL.

## Coding Standards
- TypeScript strict mode
- Server Components by default, 'use client' only when needed
- Zod for all form/API validation
- Prisma for database access
- Tailwind CSS for styling (no CSS modules)

## Architecture
- src/app/ — Pages and API routes (file-based routing)
- src/components/ — Reusable UI components (one component per file)
- src/lib/ — Server utilities, database client, auth helpers
- src/actions/ — Server actions for mutations
- src/types/ — Shared types

## Rules
@.claude/rules/typescript.md
@.claude/rules/react.md

Rules setup:

markdown
<!-- .claude/rules/typescript.md -->
---
paths:
  - "**/*.ts"
  - "**/*.tsx"
---

# TypeScript Rules
- Strict mode. No `any` types.
- Use `interface` for object shapes, `type` for unions/intersections
- Prefer `const` assertions for literal types
- Use `satisfies` operator for type-safe object literals
- Named exports only (no default exports except pages)
markdown
<!-- .claude/rules/react.md -->
---
paths:
  - "**/*.tsx"
---

# React Rules
- Functional components only
- Hooks at the top of the component
- Props interface named `{ComponentName}Props`
- Co-locate component tests: `ComponentName.test.tsx` next to `ComponentName.tsx`
- Server Components by default — add 'use client' only for interactivity

Custom skills:

A /component skill that scaffolds a new React component:

markdown
<!-- .claude/skills/component/SKILL.md -->
---
name: component
description: Scaffold a new React component with types, tests, and story.
argument-hint: ComponentName
---

# Create Component

Given a component name:
1. Create `src/components/{Name}/{Name}.tsx` with props interface and component
2. Create `src/components/{Name}/{Name}.test.tsx` with basic render test
3. Export from `src/components/{Name}/index.ts`
4. Use Server Component by default unless user specifies client

2. CLI Tool (Python)¶

CLI tools are text-in, text-out. Claude Code excels here.

markdown
# Project: git-analytics

CLI tool for analyzing git repository statistics. Python 3.12, Click, Rich.

## Coding Standards
- Type hints on all functions
- Click for argument parsing (not argparse)
- Rich for terminal output (tables, progress bars, colors)
- Pathlib for all file operations

## Architecture
- src/git_analytics/cli.py — Click command group (entry point)
- src/git_analytics/commands/ — One file per command
- src/git_analytics/core/ — Git analysis logic (no Click dependency)
- src/git_analytics/formatters/ — Output formatting (Rich tables, JSON, CSV)

## Verification
- Tests: `pytest tests/ -v`
- Lint: `ruff check src/`
- Format: `ruff format src/`
- Type check: `pyright src/`
- Build: `python -m build`

## Rules
@.claude/rules/python.md
@.claude/rules/cli.md

The CLI-specific rule:

markdown
<!-- .claude/rules/cli.md -->
---
paths:
  - "src/**/commands/**"
  - "src/**/cli.py"
---

# CLI Rules
- Every command has: --help text, --json flag for machine output, --verbose flag
- Exit codes: 0=success, 1=error, 2=usage error
- Errors go to stderr, data goes to stdout
- Support piping: detect if stdout is a TTY, skip Rich formatting if not

3. Library/Package¶

Libraries have stricter requirements around public API stability and documentation.

markdown
# Project: fastcache

High-performance caching library for Python. Pure Python, zero dependencies.

## Coding Standards
- Public API in src/fastcache/__init__.py — never change signatures without discussion
- All public functions have docstrings with Args/Returns/Raises
- Type hints required on all public and private functions
- Use __all__ to control exports

## Rules
@.claude/rules/python.md
@.claude/rules/api-stability.md

The API stability rule is critical for libraries:

markdown
<!-- .claude/rules/api-stability.md -->

# API Stability Rules
- NEVER remove or rename a public function/class/method
- NEVER change the type signature of a public function
- To deprecate: add @deprecated decorator with version and alternative
- New public API requires: docstring, type hints, at least 2 tests, changelog entry
- Breaking changes require a major version bump and migration guide

4. Monorepo¶

Monorepos need layered CLAUDE.md files: one at the root for cross-package rules, one per package for specific contexts.

monorepo/
├── CLAUDE.md                    # Cross-package rules
├── packages/
│   ├── api/
│   │   ├── CLAUDE.md            # API-specific rules
│   │   └── src/
│   ├── web/
│   │   ├── CLAUDE.md            # Frontend-specific rules
│   │   └── src/
│   └── shared/
│       ├── CLAUDE.md            # Shared library rules
│       └── src/
└── .claude/
    ├── settings.json
    └── rules/

Root CLAUDE.md:

markdown
# Project: TaskFlow Monorepo

Monorepo with pnpm workspaces. Three packages: api (Express), web (Next.js), shared (types + utils).

## Cross-Package Rules
- Shared types live in packages/shared/src/types/
- Never duplicate types between packages
- Import shared code via @taskflow/shared (not relative paths)
- All packages use TypeScript strict mode
- Run from root: `pnpm test` (all), `pnpm --filter api test` (one package)

## Package Contexts
@packages/api/CLAUDE.md
@packages/web/CLAUDE.md
@packages/shared/CLAUDE.md

Each package has its own focused CLAUDE.md. The packages/api/CLAUDE.md:

markdown
# Package: @taskflow/api

Express.js REST API. TypeScript, Prisma, PostgreSQL.

## Architecture
- src/routes/ — Route handlers (thin)
- src/services/ — Business logic
- src/middleware/ — Auth, validation, error handling

## Verification
- Tests: `pnpm --filter api test`
- Lint: `pnpm --filter api lint`

5. Microservices¶

Microservices benefit from a template approach: every service follows the same structure.

markdown
# Project: TaskFlow Platform

Microservices architecture. Each service follows the standard template.

## Service Template
Every service in services/ follows this structure:
- src/server.ts — HTTP server setup
- src/routes/ — Route handlers
- src/services/ — Business logic
- src/events/ — Event producers/consumers (Kafka)
- src/config.ts — Environment configuration
- Dockerfile — Multi-stage build
- docker-compose.yml — Local development

## Cross-Service Rules
- Services communicate via Kafka events and HTTP (never direct DB access)
- Every service has its own database (no shared databases)
- All services use the same error response format (RFC 7807)
- Health check endpoint: GET /health
- OpenAPI spec: src/openapi.yaml

## Services
@services/users/CLAUDE.md
@services/boards/CLAUDE.md
@services/notifications/CLAUDE.md

Architecture Documentation for Claude¶

Claude Code does not just follow instructions — it uses architecture documentation to make decisions. When you document your architecture, Claude can:

  • Determine which files a change affects (impact analysis)
  • Understand data flow to avoid breaking dependencies
  • Follow established patterns when adding new features
  • Catch architectural violations before they happen

The Architecture Doc¶

Create docs/architecture.md and import it from CLAUDE.md:

markdown
<!-- docs/architecture.md -->

# Architecture Overview

## System Diagram

```mermaid
graph TB
    Client[React SPA] --> API[Express API]
    API --> Auth[Auth Middleware]
    Auth --> JWT[JWT Verification]
    API --> DB[(PostgreSQL)]
    API --> Cache[(Redis)]
    API --> WS[WebSocket Server]
    WS --> Client
    API --> Queue[Bull Queue]
    Queue --> Email[Email Worker]
    Queue --> Notify[Notification Worker]

Component Responsibilities¶

Component Directory Responsibility
API Server src/routes/, src/services/ HTTP request handling, business logic
Auth src/middleware/auth.ts JWT verification, role-based access
Database src/prisma/ PostgreSQL via Prisma ORM
Cache src/lib/cache.ts Redis for session and query caching
WebSocket src/ws/ Real-time board updates
Workers src/workers/ Background jobs (email, notifications)

Data Flow¶

  1. Client sends HTTP request with JWT in Authorization header
  2. Auth middleware verifies JWT, attaches user to request
  3. Route handler validates request body with Zod
  4. Service layer executes business logic
  5. Prisma handles database operations
  6. Response sent to client
  7. If mutation: WebSocket broadcast to connected clients

API Contracts¶

  • All responses: { data: T } for success, { error: ProblemDetails } for failure
  • Pagination: ?page=1&limit=20 → { data: T[], meta: { total, page, limit } }
  • Auth: Bearer token in Authorization header
  • WebSocket: JSON messages with { type, payload } structure

Claude Code references this documentation when making decisions. If you ask "Add a notification when a card is moved," Claude reads the architecture doc, sees the Queue/Worker pattern, and implements it correctly --- producing a Bull queue job rather than a synchronous email send.

### Architecture Decision Records

**Architecture Decision Records (ADRs)** capture the *why* behind architectural choices. Claude Code reads these to avoid undoing past decisions.

```markdown
<!-- docs/decisions/0003-use-prisma-over-raw-sql.md -->

# ADR-0003: Use Prisma ORM Over Raw SQL

## Status
Accepted

## Context
We need a database access layer. Options: raw SQL (pg library), query builder (Knex), or ORM (Prisma, TypeORM).

## Decision
Use Prisma.

## Rationale
- Type-safe queries generated from schema
- Migration system built in
- Schema-as-code (prisma.schema) is the single source of truth
- Team familiar with Prisma from previous projects

## Consequences
- Cannot use advanced PostgreSQL features not supported by Prisma (rare)
- Must use Prisma Client for all DB access (no raw SQL escape hatches)
- Schema changes require `prisma migrate dev`

When Claude encounters a situation where raw SQL would be tempting, it reads this ADR and sticks with Prisma. This prevents architectural drift.

The “Claude-Ready” Project Checklist¶

Before you start building features, verify your project meets these requirements. Each item directly impacts Claude Code’s effectiveness.

# Item Why It Matters
1 CLAUDE.md exists with communication + coding standards Claude follows your conventions from day one
2 .claude/rules/ has language-specific rules Claude applies the right rules to the right files
3 .claude/settings.json has permissions + hooks Safety guardrails are active, auto-formatting works
4 docs/architecture.md exists Claude understands the system before modifying it
5 Tests exist and pass Claude verifies changes work, catches regressions
6 .gitignore is complete Claude does not commit artifacts, secrets, or build output
7 README.md has setup instructions Claude can set up the dev environment if needed
8 Linter is configured PostToolUse hooks can auto-format code
9 CI pipeline exists Changes are validated automatically on push
10 ADRs document key decisions Claude respects past architectural choices

Run through this checklist for every new project. It takes 15 minutes and saves hours of correction later.

Customizing Scaffolding¶

You can create a custom /scaffold skill that matches your team’s standards. Create .claude/skills/scaffold/SKILL.md with your specific project template:

markdown
---
name: scaffold
description: Bootstrap a new project with our team's standard structure.
argument-hint: [type] [name]
allowed-tools: Read, Write, Bash
---

# Scaffold New Project

## For Python projects:
1. Use the directory structure from our standard template
2. Include: CLAUDE.md, .claude/settings.json, .claude/rules/python.md
3. pyproject.toml with: ruff, pytest, pyright configured
4. GitHub Actions CI with: test, lint, type-check jobs
5. Pre-populated .gitignore for Python
6. First test that imports the package and verifies it loads
7. docs/architecture.md with placeholder sections

## For TypeScript projects:
1. Use our standard Next.js or Express template
2. Include: CLAUDE.md, .claude/settings.json, .claude/rules/typescript.md
3. tsconfig.json with strict mode
4. ESLint + Prettier configured
5. Vitest for testing
6. GitHub Actions CI

This skill encodes your team’s conventions so every new project starts from the same foundation.

Evolving Architecture with Claude Code¶

Projects change over time. Features get added, patterns evolve, and sometimes the original architecture no longer fits. Claude Code helps you evolve your architecture safely.

ADRs as Living Documentation¶

When you make an architectural change, create an ADR before implementing:

We need to switch from REST to GraphQL for the board queries because
of the N+1 query problem with nested resources. Create an ADR documenting
this decision, then implement the change.

Claude Code creates the ADR first, then implements the change in accordance with the decision. Future sessions read the ADR and understand why GraphQL is used for boards but REST is used for auth endpoints.

Refactoring with Plan Mode¶

For large architectural changes, use /plan mode to review the approach before executing:

/plan
Refactor the API from flat route handlers to a service layer pattern.
Currently all business logic is in src/routes/. We need to extract it
to src/services/ and keep routes as thin request/response handlers.

Claude Code produces a detailed plan: which files change, what moves where, what the new structure looks like, and what risks exist. You review the plan, adjust it, then execute.

Migration Strategies Using Worktrees¶

For high-risk architectural changes, use worktrees to isolate the work:

Create a worktree for the service-layer migration. Move boards logic first
as a proof of concept.

Claude Code creates an isolated git worktree, implements the change for one module, and you can verify it works before applying the pattern to the rest of the codebase. If the approach fails, you discard the worktree with zero impact on your main branch.

When Architecture Outgrows CLAUDE.md¶

As your project grows, the CLAUDE.md and architecture docs need to grow with it. Signs that your architecture documentation needs updating:

  • Claude keeps putting files in the wrong directories
  • Claude generates patterns inconsistent with recent changes
  • New team members (human or AI) make the same mistakes
  • You find yourself correcting Claude on the same conventions repeatedly

When this happens, update the documentation immediately. The cost of a 5-minute CLAUDE.md update is trivial compared to repeatedly correcting architectural drift across dozens of files.

What Most People Get Wrong¶

Over-documenting in CLAUDE.md. The instinct is to write everything Claude needs to know. But Claude reads your code. It sees your imports, your patterns, your test structure. CLAUDE.md should contain what Claude cannot infer from the code: conventions, preferences, and decisions. If Claude can see that you use Prisma by reading package.json, you do not need to say “we use Prisma” in CLAUDE.md. But you do need to say “never use raw SQL” because that is a constraint Claude cannot infer.

Ignoring framework conventions. Claude Code knows Next.js App Router conventions. If you follow them, Claude generates correct code automatically. If you deviate without documenting the deviation, Claude generates code for the standard convention and you spend time correcting it.

Skipping architecture docs for small projects. Even a 3-file project benefits from a one-paragraph architecture note in CLAUDE.md. “Routes handle HTTP, services handle logic, models handle data.” That one sentence prevents Claude from putting business logic in route handlers.

No tests in the scaffold. A project without tests is a project where Claude cannot verify its changes. Always include at least one test in your scaffold, even if it is just an import smoke test. Claude uses test results as a feedback signal.

Chapter Summary¶

  1. Start with CLAUDE.md, not code. Engineer your project brain before writing a single line of application code. This upfront investment pays dividends throughout development.

  2. Directory structure matters. Use consistent, shallow directory layouts that follow framework conventions. Claude Code navigates predictable structures faster and more accurately.

  3. Match your project type to the right setup. Web apps, CLI tools, libraries, monorepos, and microservices each need different CLAUDE.md structures, rules, and skills.

  4. Architecture documentation is not optional. Component diagrams, data flow docs, and ADRs give Claude Code the context it needs for accurate impact analysis and consistent pattern application.

  5. Use the Claude-ready checklist. Ten items, fifteen minutes. Every project. The discipline of a proper setup eliminates hours of correction downstream.
