Chapter 19: Test-Driven Development with Claude Code¶ ~15 min read

Test-driven development has always been a discipline. You write the test first, watch it fail, write the minimum code to make it pass, then refactor. The problem was never the philosophy — it was the friction. Writing comprehensive tests takes time. Waiting for the red-green-refactor cycle burns minutes. TDD advocates preach its benefits, but most teams abandon it under deadline pressure because the upfront cost feels too high.

Claude Code changes the economics. Claude writes tests in seconds. Claude implements code to pass them in seconds. The friction that made TDD impractical for everyday work is gone. What remains is the discipline — and now, the discipline is cheap.

Why TDD Is More Powerful with Claude Code¶

The TDD loop with Claude Code looks like this:

  1. You describe the behavior (one sentence)
  2. Claude writes the failing test (5 seconds)
  3. You run the test — it fails (confirms the test is meaningful)
  4. Claude writes the implementation (10 seconds)
  5. You run the test — it passes (confirms the implementation is correct)
  6. Claude refactors if needed (5 seconds)

Total elapsed time: under a minute for a complete feature cycle. Without Claude Code, step 2 alone takes 5-10 minutes for a non-trivial test.

The deeper insight: you focus on WHAT to test, Claude handles HOW. Your expertise lies in knowing which behaviors matter, which edge cases break production, and which invariants must hold. Claude’s strength is translating those requirements into syntactically correct, comprehensive test code. This is a force multiplier.

The Feedback Loop¶

Without Claude Code:

Think about what to test (2 min)
→ Write test code (8 min)
→ Run test, see it fail (30 sec)
→ Write implementation (15 min)
→ Run test, debug, iterate (5 min)
→ Refactor (5 min)
Total: ~35 minutes per feature

With Claude Code:

Describe the behavior (30 sec)
→ Claude writes test (5 sec)
→ Run test, see it fail (30 sec)
→ Claude writes implementation (10 sec)
→ Run test, see it pass (30 sec)
→ Claude refactors (5 sec)
Total: ~2 minutes per feature

That is not a marginal improvement. It is a category change. TDD goes from “expensive discipline we aspire to” to “natural way we build software.”

The Claude Code TDD Workflow¶

Here is the workflow applied to a real feature: password reset via email.

Step 1: Describe the Behavior¶

Users should be able to request a password reset. The system generates a
time-limited token, sends an email, and allows the user to set a new
password using that token. Tokens expire after 1 hour. Used tokens
cannot be reused.

Step 2: Claude Writes the Failing Test¶

Write a failing test for the password reset feature. Cover:
1. Requesting a reset generates a token
2. Valid token allows password change
3. Expired token is rejected
4. Used token cannot be reused
5. Invalid token returns an error

Claude produces:

from datetime import datetime, timedelta, timezone
from unittest.mock import AsyncMock

import pytest
from httpx import ASGITransport, AsyncClient

from app.main import app
from app.models.user import User
from tests.helpers import create_test_user, get_db


@pytest.fixture

    transport = ASGITransport(app=app)
    async with AsyncClient(transport=transport, base_url="http://test") as ac:
        yield ac

async def test_request_reset_generates_token(
    client: AsyncClient, test_user: User
) -> None:
    """Requesting a password reset should create a reset token."""
    response = await client.post(
        "/api/auth/password-reset/request",
        json={"email": "user@example.com"},
    )
    assert response.status_code == 200

    # Verify token was created in the database
    async with get_db() as db:
        token_record = await db.fetchone(
            "SELECT * FROM password_reset_tokens WHERE user_id = ?",
            (test_user.id,),
        )
    assert token_record is not None
    assert token_record["expires_at"] > datetime.now(timezone.utc).isoformat()


@pytest.mark.asyncio
async def test_valid_token_allows_password_change(
    client: AsyncClient, test_user: User
) -> None:
    """A valid, unexpired token should allow password change."""
    # Request reset
    await client.post(
        "/api/auth/password-reset/request",
        json={"email": "user@example.com"},
    )

    # Get the token from DB
    async with get_db() as db:
        row = await db.fetchone(
            "SELECT token FROM password_reset_tokens WHERE user_id = ?",
            (test_user.id,),
        )
    token = row["token"]

    # Reset password using the token
    response = await client.post(
        "/api/auth/password-reset/confirm",
        json={"token": token, "new_password": "NewPassword456!"},
    )
    assert response.status_code == 200

    # Verify login works with new password
    login_response = await client.post(
        "/api/auth/login",
        json={"email": "user@example.com", "password": "NewPassword456!"},
    )
    assert login_response.status_code == 200

async def test_expired_token_is_rejected(
    client: AsyncClient, test_user: User
) -> None:
    """An expired token should be rejected."""
    # Insert an expired token directly
    expired_time = (datetime.now(timezone.utc) - timedelta(hours=2)).isoformat()
    async with get_db() as db:
        await db.execute(
            """INSERT INTO password_reset_tokens (user_id, token, expires_at)
               VALUES (?, 'expired-token-123', ?)""",
            (test_user.id, expired_time),
        )

    response = await client.post(
        "/api/auth/password-reset/confirm",
        json={"token": "expired-token-123", "new_password": "NewPassword456!"},
    )
    assert response.status_code == 400
    assert "expired" in response.json()["detail"].lower()


@pytest.mark.asyncio
async def test_used_token_cannot_be_reused(

        json={"email": "user@example.com"},

            name:    "empty string",
            ua:      "",
            wantErr: true,
        },

            assert.Equal(t, tt.wantBrowser, result.Browser)
        })

# tests/fakes.py

from app.models import User
from app.repositories.base import UserRepository


class FakeUserRepository(UserRepository):
    def __init__(self) -> None:
        self.users: dict[str, User] = {}

    async def get_by_id(self, user_id: str) -> User | None:
        return self.users.get(user_id)

    async def get_by_email(self, email: str) -> User | None:
        for user in self.users.values():
            if user.email == email:
                return user
        return None

    async def create(self, email: str, name: str, password_hash: str) -> User:
        user = User(id=f"fake-{len(self.users)}", email=email, name=name, password_hash=password_hash)
        self.users[user.id] = user
        return user

Integration Tests vs. Unit Tests¶

The rule of thumb: mock what you do not own, test against what you do own.

Dependency Testing Strategy

set -euo pipefail

INPUT=$(cat)
TOOL=$(echo "${INPUT}" | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_name',''))")
FILE=$(echo "${INPUT}" | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_input',{}).get('file_path',''))")

    exit 0
fi

if [[ "${FILE}" != *.py ]]; then
    exit 0
fi

# Find matching test file

if [[ -f "${TEST_FILE}" ]]; then
    echo "Running tests: ${TEST_FILE}" >&2
    python -m pytest "${TEST_FILE}" -x -q 2>&1 | tail -5 >&2
fi

# src/middleware/rate_limit.py

from __future__ import annotations

import time
from typing import Callable

from fastapi import Request, Response
from starlette.middleware.base import BaseHTTPMiddleware
from starlette.responses import JSONResponse

from app.lib.redis import redis_client


class RateLimitMiddleware(BaseHTTPMiddleware):
    """Sliding window rate limiter using Redis."""

    def __init__(self, app: Callable, limit: int = 100, window: int = 60) -> None:
        super().__init__(app)
        self.limit = limit
        self.window = window

    async def dispatch(self, request: Request, call_next: Callable) -> Response:
        # Skip rate limiting for unauthenticated requests
        user_id = getattr(request.state, "user_id", None)
        if user_id is None:
            return await call_next(request)

        key = f"rate_limit:{user_id}"
        now = time.time()
        window_start = now - self.window

        pipe = redis_client.pipeline()
        pipe.zremrangebyscore(key, 0, window_start)
        pipe.zadd(key, {str(now): now})
        pipe.zcard(key)
        pipe.expire(key, self.window)
        results = await pipe.execute()

        request_count = results[2]
        remaining = max(0, self.limit - request_count)

        if request_count > self.limit:
            oldest = await redis_client.zrange(key, 0, 0, withscores=True)
            if oldest:
                retry_after = int(self.window - (now - oldest[0][1]))
            else:
                retry_after = self.window

            return JSONResponse(
                status_code=429,
                content={"detail": "Rate limit exceeded"},
                headers={
                    "Retry-After": str(max(1, retry_after)),
                    "X-RateLimit-Limit": str(self.limit),
                    "X-RateLimit-Remaining": "0",
                },
            )

        response = await call_next(request)
        response.headers["X-RateLimit-Limit"] = str(self.limit)
        response.headers["X-RateLimit-Remaining"] = str(remaining)
        return response

What Most People Get Wrong¶

Writing tests after the implementation. If you implement first, the tests tend to test the implementation rather than the behavior. Tests written first test the contract — what should happen — which catches regressions even when the implementation changes.

Testing too much at once. One test should verify one behavior. If your test has 15 assertions, it is probably testing 5 different things. Break it up. Claude generates focused tests when you give focused descriptions.

Over-mocking. Mock external services you do not control (Stripe, AWS, SendGrid). Do not mock your own database, your own cache, or your own queue. Integration tests that hit real infrastructure catch real bugs.

Chapter Summary¶

  1. Claude Code makes TDD economically viable. The friction that made TDD impractical — writing tests, implementing code to pass them — is eliminated. The cycle drops from 35 minutes to 2 minutes.

  2. Focus on WHAT to test, let Claude handle HOW. Your expertise is knowing which behaviors matter. Claude’s expertise is writing comprehensive, correct test code. This division of labor is the force multiplier.

  3. Use parametrized tests for input validation. Describe the categories of inputs. Claude enumerates dozens of specific cases you would not have thought of.

  4. Integration tests over mocks for your own infrastructure. Hit the real database, the real cache, the real queue. Mock only external services. This catches real bugs that mocked tests miss.
