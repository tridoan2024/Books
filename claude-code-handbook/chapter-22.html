Chapter 22: Prompt Patterns That 10x Your Output¶ ~17 min read

The difference between a junior and senior Claude Code user isn’t the features they know — it’s the prompts they write. A junior types “make a login page.” A senior types “Read src/auth/login.ts and src/components/LoginForm.tsx. Then add rate limiting to the login endpoint: max 5 attempts per IP per 15 minutes, using the existing Redis connection in src/lib/redis.ts. Write a test first.”

Both get results. One gets the right result on the first try.

This chapter catalogs 20 prompt patterns that consistently produce better output from Claude Code. Each pattern includes when to use it, a complete example, what Claude does in response, a pro tip, and the anti-pattern to avoid. These aren’t theoretical — they’re distilled from hundreds of hours of professional Claude Code usage.

Pattern 1: The Constraint Pattern¶

When to use: You want a focused solution without unnecessary complexity. Claude’s default tendency is to reach for external libraries, add comprehensive error handling, and build for extensibility. Constraints channel that energy.

Example prompt:

Using only the Python standard library, implement a URL shortener 
that stores mappings in a JSON file. No classes — only functions. 
Keep it under 80 lines.

What Claude does: Strips away the instinct to import Flask, set up SQLAlchemy, and build a REST API. Instead, it produces a minimal, elegant solution using http.server, json, and hashlib.

Pro tip: Stack constraints for precision. “No new dependencies, no classes, under 100 lines, Python 3.11+” gives Claude four guardrails that collectively force a specific kind of solution.

Anti-pattern: Giving no constraints and then complaining about over-engineering. If you don’t want a class hierarchy, say so upfront.

Variations: - “Without adding any new dependencies…” - “In a single file, under 200 lines…” - “Using only functions, no OOP…” - “Compatible with Python 3.9+ standard library only…”

Pattern 2: The Architecture-First Pattern¶

When to use: Before any complex feature or system. Forces Claude to think about structure before writing code, preventing the common failure mode of building a perfectly-coded component that doesn’t fit the architecture.

Example prompt:

Before writing any code, show me:
1. Component diagram (which modules/files we need)
2. Data flow (how a request moves through the system)
3. File structure (what goes where)
4. Interface contracts (function signatures between components)

Then I'll approve and you can implement.

The feature: real-time notification system using WebSockets, 
supporting email and in-app notification channels.

What Claude does: Produces a Mermaid diagram, maps the data flow from event trigger through notification dispatch, lays out the file structure, and defines the interfaces between components. Implementation only begins after you approve the design.

Pro tip: Combine with /plan mode. Enter plan mode, use this pattern, review the plan, then let Claude execute. The plan persists as a reference throughout implementation.

Anti-pattern: “Build me a notification system” — Claude dives into code immediately, makes architectural choices you’d disagree with, and you spend more time undoing than building.

Pattern 3: The Test-First Pattern¶

When to use: Any new feature or bug fix. Tests define the specification in executable form. Claude naturally follows the red-green-refactor cycle when you prompt this way.

Example prompt:

Write failing tests for a password validator with these rules:
- Minimum 12 characters
- At least one uppercase, one lowercase, one digit, one special char
- Cannot contain the username
- Cannot be in the common passwords list (load from passwords.txt)

Then implement the validator to make all tests pass.

What Claude does: Writes 8-12 test cases covering valid passwords, each individual rule violation, edge cases (empty string, None, unicode), and the username-in-password check. Then implements the validator, runs the tests, and iterates until they all pass.

Pro tip: Be specific about edge cases in your test prompt. “Include tests for: empty string, None input, unicode characters, passwords that are exactly 12 characters” guides Claude to test boundaries you care about.

Anti-pattern: “Build a password validator and add tests” — the tests end up testing what the implementation does rather than what it should do, missing edge cases entirely.

Pattern 4: The Chain-of-Files Pattern¶

When to use: When the change depends on understanding multiple files. Loading context before requesting changes dramatically reduces hallucination — Claude can’t invent function signatures that don’t exist when it’s already read the actual code.

Example prompt:

Read these files in order:
1. src/models/user.py (the User model)
2. src/services/auth.py (the authentication service) 
3. src/middleware/rate_limit.py (the existing rate limiter)
4. src/routes/auth_routes.py (the auth endpoints)

Then add a login attempt tracker that:
- Records failed login attempts per IP in Redis
- Locks the account after 5 failures in 10 minutes
- Integrates with the existing rate limiter middleware
- Returns a 429 with retry-after header when locked

What Claude does: Reads all four files, understands the existing patterns (how models are defined, how services call repositories, how middleware chains), and implements the feature using the same patterns. The result fits seamlessly into the codebase because Claude has the full picture.

Pro tip: Order matters. Read the foundational files first (models, types) and the integration points last (routes, controllers). This mirrors how a human developer would onboard to the code.

Anti-pattern: “Add a login attempt tracker to the auth system” without loading any context. Claude guesses at your User model fields, invents service method names, and produces code that requires extensive editing.

Advanced variation:

Read all files matching src/auth/**/*.py, then add session 
invalidation when a user changes their password.

Pattern 5: The Progressive Refinement Pattern¶

When to use: Building anything with multiple layers of complexity. Each turn adds one concern to working code, rather than trying to handle everything at once.

Example prompt sequence:

Turn 1: "Build a basic REST API for a todo app with FastAPI. 
         Just CRUD endpoints, in-memory storage. No auth, 
         no validation, no error handling."

Turn 2: "Add Pydantic validation models for all endpoints. 
         Title: 1-200 chars, required. Description: optional, 
         max 1000 chars. Status: enum (pending, in_progress, done)."

Turn 3: "Replace in-memory storage with SQLAlchemy + PostgreSQL. 
         Add Alembic migration for the todos table."

Turn 4: "Add JWT authentication. Protected endpoints require 
         a valid token. Add a /auth/login endpoint."

Turn 5: "Add pagination (cursor-based) and filtering by status 
         to the GET /todos endpoint."

What Claude does: Each turn builds on verified working code. Turn 2 can’t break the CRUD logic because it only adds validation. Turn 3 replaces storage without touching the API contract. Each layer is independently testable.

Pro tip: Run tests between turns. “Run the tests before we continue” after each layer catches regressions immediately.

Anti-pattern: “Build a full-featured todo API with FastAPI, PostgreSQL, JWT auth, pagination, filtering, rate limiting, caching, and WebSocket notifications.” Claude produces 500 lines with subtle bugs in the interaction between features that are painful to debug.

Pattern 6: The Example-Driven Pattern¶

When to use: When you need consistency with existing code. Examples are worth a thousand words of description. Claude is exceptional at pattern replication — give it one good example and it’ll produce ten more in the same style.

Example prompt:

Here's how our UserService is structured:

```python
class UserService:
    def __init__(self, db: Database, cache: RedisCache):
        self._db = db
        self._cache = cache

    async def get_by_id(self, user_id: UUID) -> User | None:
        cached = await self._cache.get(f"user:{user_id}")
        if cached:
            return User.model_validate_json(cached)
        user = await self._db.users.find_one({"id": user_id})
        if user:
            await self._cache.set(f"user:{user_id}", user.model_dump_json(), ttl=300)
        return user

    async def create(self, data: CreateUserRequest) -> User:
        user = User(id=uuid4(), **data.model_dump())
        await self._db.users.insert_one(user.model_dump())
        await self._cache.delete(f"users:list:*")
        return user

Create an OrderService following the exact same patterns: - Same constructor signature pattern (db + cache) - Same cache-aside pattern for reads - Same cache invalidation pattern for writes - Async throughout - Methods: get_by_id, list_by_user, create, update_status, cancel


**What Claude does:** Produces an OrderService that looks like it was written by the same developer as UserService. Same naming conventions, same caching pattern, same error handling approach, same type hint style.

**Pro tip:** Point at a file instead of pasting: "Follow the exact same patterns used in `src/services/user_service.py`." This works because Claude reads the file and extracts the patterns itself.

**Anti-pattern:** Describing the patterns you want in prose. "Use dependency injection, implement caching, make it async" is ambiguous — showing is unambiguous.

## Pattern 7: The Negative Constraint Pattern

**When to use:** When you want Claude to do less. By default, Claude adds error handling, input validation, logging, type annotations, docstrings, and comments. Sometimes you just want the core logic.

**Example prompt:**

Implement the sorting algorithm. Do NOT add: - Error handling or input validation - Comments or docstrings - Type annotations - Logging - Edge case handling

Just the raw algorithm, nothing else.


**What Claude does:** Produces the minimal implementation without any of its default additions. This is useful for prototyping, for understanding an algorithm's core logic, or for situations where the surrounding code already handles these concerns.

**Pro tip:** This pattern is especially powerful for minimal diffs. "Fix only the null check on line 42. Don't add error handling, don't improve the surrounding code, don't change formatting."

**Anti-pattern:** Asking for minimal changes without negative constraints, then being frustrated when Claude "helpfully" refactors three surrounding functions.

## Pattern 8: The Scope Limiter

**When to use:** When you need a precise, surgical change. Claude's tendency toward thoroughness means it often modifies more files or functions than intended. Scope limiters draw hard boundaries.

**Example prompt:**

Only modify the render() function in src/components/Dashboard.tsx. Do not change any other function in this file. Do not modify any other file. Add a loading skeleton that shows while data is being fetched.


**What Claude does:** Modifies exactly the `render()` function, adds the loading skeleton, and touches nothing else. No "while I'm here, I'll also..." improvements.

**Pro tip:** Combine with file-level scoping: "Only modify files in `src/auth/`. Do not touch `src/models/` or `src/routes/`."

**Anti-pattern:** "Add a loading state to the Dashboard" — Claude might modify the Dashboard component, the data fetching hook, the API client, and the route configuration, when all you wanted was a spinner in one function.

## Pattern 9: The Batch Pattern

**When to use:** Applying the same transformation to many files. Instead of asking Claude to modify files one at a time, describe the pattern once and let it apply everywhere.

**Example prompt:**

Apply this change to all files matching src/api/routes/*.ts:

Replace the old error response format: res.status(400).json({ error: message })

With the new standardized format: res.status(400).json({ error: { code: “BAD_REQUEST”, message, timestamp: new Date().toISOString() } })

Do all files, show me the count when done.


**What Claude does:** Uses Grep to find all matching files, reads each one, applies the transformation consistently, and reports the total count of changes.

**Pro tip:** For large-scale changes across 20+ files, use the `/batch` command which runs parallel agents in worktrees. Each agent handles a subset of files, and results merge automatically.

**Anti-pattern:** Asking Claude to change one file, then saying "now do the same to the next file" twenty times. Batch it.

## Pattern 10: The Checkpoint Pattern

**When to use:** Multi-step changes where regressions are possible. Embedding test checkpoints in your prompt ensures Claude catches problems immediately rather than stacking broken changes.

**Example prompt:**

Refactor the payment processing module. After EACH change: 1. Run pytest tests/test_payments.py 2. If any test fails, stop immediately and show me the failure 3. Only proceed to the next change if all tests pass

Changes needed: - Extract card validation into its own function - Extract receipt generation into its own function
- Move both to a new payments/utils.py module - Update all imports


**What Claude does:** Makes the first extraction, runs tests, confirms they pass. Makes the second extraction, runs tests, confirms they pass. Moves to the new module, runs tests, catches a broken import, fixes it, runs tests again, confirms they pass.

**Pro tip:** Specify which test file or test function to run, not the full suite. Running `pytest tests/test_payments.py -x` (stop on first failure) is faster than the full suite and still catches regressions in the affected area.

**Anti-pattern:** "Refactor this, then run all the tests at the end." If something breaks in step 2 of 5, you've wasted the work in steps 3-5.

## Pattern 11: The Explain-Then-Change Pattern

**When to use:** Before modifying code you don't fully understand, or when you want Claude to demonstrate understanding before touching anything. This prevents the failure mode where Claude confidently modifies code it misunderstands.

**Example prompt:**

First, explain how the authentication middleware chain works. Walk me through a request’s journey from hitting the /api/users endpoint to getting a response. Reference file:line for each step.

Then, and only then, suggest improvements to the session refresh logic.


**What Claude does:** Reads the middleware stack, traces a request through each layer (CORS check, rate limit, JWT validation, session refresh, route handler), and explains each step with file:line references. Only after demonstrating understanding does it propose changes to the session refresh logic — changes that fit the existing architecture because Claude actually understands it.

**Pro tip:** This is your quality gate for unfamiliar code. If Claude's explanation reveals a misunderstanding, correct it before allowing changes.

**Anti-pattern:** "Improve the auth middleware" on a codebase Claude hasn't read. It makes assumptions, implements changes based on those assumptions, and creates bugs.

## Pattern 12: The Reference Pattern

**When to use:** When consistency with existing code is non-negotiable. Instead of describing the desired patterns, point Claude at a canonical example.

**Example prompt:**

Follow the exact same patterns used in src/services/user-service.ts for the new src/services/order-service.ts. Same directory structure, same naming conventions, same error handling approach, same test file structure.


**What Claude does:** Reads `user-service.ts` and its test file, extracts every pattern (how errors are thrown, how database calls are wrapped, how responses are shaped, how tests are structured), and replicates them precisely in the new service.

**Pro tip:** If you have a service that's your best example and one that's your worst, explicitly say: "Follow user-service.ts, NOT legacy-user-service.ts."

**Anti-pattern:** Writing a long description of your coding patterns when you could just point at a file that demonstrates them.

## Pattern 13: The Diff Preview Pattern

**When to use:** Before committing to changes that are hard to undo. Database migrations, configuration changes, infrastructure modifications — anything where "just undo it" isn't trivial.

**Example prompt:**

Show me what you would change as a unified diff before making any edits. I want to review the changes before they’re applied.

Proposed change: add soft delete to the User model and update all queries to filter deleted users.


**What Claude does:** Produces a complete diff showing every file and line it would change, without actually making any changes. You review, request modifications, and only then say "apply these changes."

**Pro tip:** For large changes, ask for the diff in sections: "Show me the model changes first, then the query changes, then the test changes."

**Anti-pattern:** Letting Claude make database migration changes without previewing them. A bad migration on production is a bad day.

## Pattern 14: The Multi-Agent Research Pattern

**When to use:** When investigating a codebase issue that spans multiple dimensions — security, performance, correctness — and you want each dimension examined by a specialist.

**Example prompt:**

Use parallel agents to investigate the session management code: 1. Security agent: audit for session fixation, hijacking, cookie security 2. Code review agent: check for logic errors, race conditions, resource leaks 3. Performance agent: profile session lookup latency, check for unnecessary database hits

Synthesize all findings into a single prioritized report.


**What Claude does:** Spawns three agents (or handles the three concerns sequentially if agents aren't configured). Each examines session management through its lens. The main session merges findings, deduplicates, and presents a unified report ranked by severity.

**Pro tip:** This works even without custom agents — Claude can adopt different analytical perspectives within a single session. But dedicated agents with model overrides (Opus for security, Sonnet for code quality) produce stronger results.

**Anti-pattern:** "Review the session code" with no specification of what to look for. Claude produces a generic review that's shallow across all dimensions.

## Pattern 15: The Plan-Execute Pattern

**When to use:** Complex changes affecting 5+ files. Planning upfront prevents the "I'm 80% done and just realized I need to restructure everything" problem.

**Example prompt:**

Enter plan mode. I need to refactor the payment processing module from a monolithic service into separate modules: - PaymentGateway (Stripe/PayPal abstraction) - InvoiceService (invoice generation and storage)
- ReceiptService (receipt formatting and delivery) - PaymentOrchestrator (coordinates the flow)

Explore the current codebase, identify all dependencies, and create a step-by-step migration plan. Don’t write any code yet.


**What Claude does:** In plan mode, Claude can only read and search — not write. It maps the current payment module's internal structure, traces all external dependencies (who calls what, what imports what), identifies the safest decomposition order, and produces a numbered migration plan. Each step is small enough to be independently verifiable.

**Pro tip:** After approving the plan, say "Execute steps 1-3" rather than "Execute the plan." This gives you checkpoints to verify progress.

**Anti-pattern:** Asking Claude to "refactor the payment module" with no planning step. It charges ahead, gets tangled in circular dependencies at step 4, and has to restart.

## Pattern 16: The Context-Loading Pattern

**When to use:** At the start of a session when Claude has no context about your project, or before asking about architecture-level questions that require broad understanding.

**Example prompt:**

Read these files to understand the project: 1. README.md 2. docs/architecture.md 3. src/app.ts (entry point) 4. package.json (dependencies)

Then show me the last 10 commits with: git log –oneline -10

Now, explain the current state of the project and suggest the three highest-impact improvements.


**What Claude does:** Loads the foundational context, reviews recent development activity, and provides recommendations grounded in actual project reality rather than generic advice.

**Pro tip:** Create a "context bundle" for your project — a standard set of files that Claude should read at the start of any complex task. Document it in your CLAUDE.md: "For architecture questions, always read: [list of files]."

**Anti-pattern:** Asking architecture questions without loading any context. "What should I refactor?" when Claude doesn't know your project is like asking a consultant for advice before they've seen the codebase.

## Pattern 17: The Minimal Change Pattern

**When to use:** Bug fixes where you need a surgical correction without any collateral changes. This is the opposite of "make it better" — it's "make it work without touching anything else."

**Example prompt:**

Fix only the null pointer exception on line 47 of src/utils/parser.ts.

Rules: - Do not refactor any surrounding code - Do not add new tests (I’ll add those separately) - Do not change any imports - Do not modify any other function - Do not change formatting or whitespace

One line change, maximum.


**What Claude does:** Adds a null check (or whatever the minimal fix is) to the exact line specified. Nothing else changes. The diff is one or two lines.

**Pro tip:** Use this in production hotfix scenarios where you need the smallest possible change to minimize deployment risk.

**Anti-pattern:** "Fix the parser" — Claude reads the parser, finds the null pointer bug, but also refactors the error handling, improves the type annotations, and reorganizes the imports. Your "hotfix" becomes a "minor rewrite."

## Pattern 18: The Role Pattern

**When to use:** When you need Claude to evaluate code from a specific professional perspective. Different roles catch different problems.

**Example prompt:**

You are a senior database administrator who has managed PostgreSQL clusters handling 100K+ queries per second.

Review this migration file for: - Data safety during the migration - Impact on running queries (locks, blocking) - Rollback safety if something goes wrong - Index impact on write performance - Whether this should run during maintenance window or can be applied live


**What Claude does:** Evaluates the migration not as a general-purpose AI but as someone with specific DBA expertise. It flags that an `ALTER TABLE ADD COLUMN` with a default value will rewrite the entire table in PostgreSQL < 11, recommends adding the column without a default and backfilling in batches, and suggests the migration should be applied during low-traffic hours.

**Pro tip:** Combine roles with the explain pattern: "As a senior SRE, explain what would happen to our system during a traffic spike given this architecture."

**Anti-pattern:** Using the role pattern for simple tasks. "As a senior engineer, add a print statement" adds nothing — the role pattern is for leveraging domain-specific reasoning.

## Pattern 19: The Iterative Search Pattern

**When to use:** When you need to understand how a feature works across multiple files and layers. Instead of a broad "find everything related to X," guide the exploration layer by layer.

**Example prompt:**

I need to understand how user permissions work in this codebase.

Step 1: Find where permissions are defined (models, enums, constants) Step 2: Find where permissions are checked (middleware, decorators, guards) Step 3: Find where permissions are assigned (user creation, role changes) Step 4: Trace a specific request: when a user hits DELETE /api/projects/123, what permission checks happen, in what order?

Show me file:line for each discovery.


**What Claude does:** Follows the guided exploration, building up understanding layer by layer. By step 4, it has a complete mental model of the permission system and can trace a specific request through every check.

**Pro tip:** This is how senior engineers onboard to unfamiliar code. They don't search "permissions" and read every result — they trace the concept layer by layer. Teach Claude to do the same.

**Anti-pattern:** "Find all permission-related code" — returns 50 file matches with no structure. You learn that permissions exist everywhere but don't understand how they connect.

## Pattern 20: The Escape Hatch Pattern

**When to use:** When the cost of a mistake is high. Database migrations, infrastructure changes, third-party API integrations, financial calculations — anywhere Claude's confidence might exceed its accuracy.

**Example prompt:**

Implement the Stripe webhook handler for payment events.

IMPORTANT: If you’re unsure about any of the following, ask me before writing code: - The Stripe API version we’re using - Whether we process refunds synchronously or async - The retry behavior for failed webhook deliveries - Which events we need to handle vs. can ignore

Do NOT guess on any of these — wrong assumptions about payment processing can cost real money.


**What Claude does:** Implements what it's confident about (webhook signature verification, basic event routing) and explicitly asks about the four items where assumptions could be dangerous.

**Pro tip:** Use escape hatches for domain-specific knowledge that Claude can't derive from your code. Stripe API versions, business rules, regulatory requirements — these are areas where guessing is dangerous.

**Anti-pattern:** Never giving Claude escape hatches and then being surprised when it confidently implements a wrong assumption about your business logic.

## Combining Patterns

The real power comes from combining patterns. Here are three compound patterns that handle complex, real-world scenarios:

### The Full Feature Pattern (Patterns 2 + 3 + 5 + 10)

Architecture-first + Test-first + Progressive refinement + Checkpoints

“Before writing any code, show me the architecture for a real-time notification system.

[Approve architecture]

Write failing tests for the WebSocket connection handler. Then implement to make them pass. Run tests — confirm they pass.

Write failing tests for the notification dispatch service. Then implement. Run tests.

Write failing tests for the email notification channel. Then implement. Run tests.

Write integration tests for the full flow. Run them.”


### The Safe Refactor Pattern (Patterns 4 + 11 + 13 + 10)

Chain-of-files + Explain-then-change + Diff preview + Checkpoint

“Read all files in src/payments/. Explain how the payment flow works end to end.

[Verify understanding]

Show me a diff of the proposed refactoring before making changes.

[Approve diff]

Apply changes. After each file modification, run pytest tests/test_payments.py -x. Stop on any failure.”


### The Research-to-Implementation Pattern (Patterns 14 + 2 + 6 + 3)

Multi-agent research + Architecture-first + Example-driven + Test-first

“Research the best approaches for rate limiting in FastAPI. Compare token bucket vs. sliding window vs. fixed window.

[Review research findings]

Before implementing, show me the architecture: where the middleware sits, how it integrates with our existing middleware chain, what the configuration looks like.

Follow the same patterns used in src/middleware/auth.py for the middleware structure.

Write failing tests first, then implement.” ```

What Most People Get Wrong¶

Over-prompting: Writing a 500-word prompt for a simple task. “Add a null check on line 42” doesn’t need three paragraphs of context.

Under-prompting for complex tasks: Typing “build the payment system” and expecting production-ready code in one turn. Complex systems need the progressive refinement pattern.

Not loading context: Asking Claude to modify code it hasn’t read. Always chain-of-files before modification.

Ignoring the checkpoint pattern: Making five changes before running tests. The first change broke something, and now you can’t tell which one.

Using the wrong model for the task: Asking Haiku to do a security audit. Asking Opus to rename a variable. Match model to task complexity.

Chapter Summary¶

  • Constraints force focused solutions — specify what you don’t want as clearly as what you do
  • Architecture-first prevents building the wrong thing well — think before coding
  • Test-first defines the spec before the implementation — tests are the source of truth
  • Chain-of-files loads context before requesting changes — Claude can’t hallucinate what it’s already read
  • Progressive refinement builds complexity in verified layers — each turn adds one concern to working code
  • Combine patterns for complex tasks — the full feature pattern, safe refactor pattern, and research-to-implementation pattern handle real-world complexity
  • The single most important habit: load context before making changes
