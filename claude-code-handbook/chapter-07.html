Chapter 7: Skills – Building Reusable AI Workflows¶ ~20 min read

You have used Claude Code enough to notice a pattern. Every code review follows the same steps: read the file, check security, check logic, check performance, output findings with severity and line numbers. Every bug fix follows the same arc: read the issue, search the codebase, write a test, implement the fix, run tests, commit, create a PR. Every deployment check walks through the same checklist: git clean, no secrets, dependencies audited, tests pass, build succeeds.

You keep repeating the same instructions. You keep getting slightly different results depending on how you phrase the prompt that day. Sometimes Claude remembers to check security first. Sometimes it forgets. Sometimes it includes line numbers. Sometimes it does not.

Skills solve this permanently. A skill is a packaged set of instructions that teaches Claude Code a specific workflow, exactly once, and ensures it executes that workflow the same way every time you invoke it. You write the instructions in a markdown file. Claude Code loads them into the system prompt when triggered. The workflow becomes deterministic, repeatable, and shareable across your team.

This chapter is the complete guide to building, configuring, and deploying skills. By the end of it, you will have built three production-grade skills from scratch and understood every configuration option available.

What Skills Are¶

A skill is a markdown file named SKILL.md that lives in a specific directory structure. When activated, its contents are injected into Claude Code’s system prompt – the invisible instructions that govern how the model behaves. This means a skill does not run code. It teaches Claude Code how to approach a category of work.

There are two locations for skills:

Location Scope Use Case
.claude/skills/<name>/SKILL.md Project-specific Workflows tied to this codebase
~/.config/claude/skills/<name>/SKILL.md Global (all projects) Personal workflows you use everywhere

Project skills travel with the repository. Global skills travel with you.

How Skills Get Activated¶

Three activation mechanisms:

  1. Slash command. You type /skill-name in the REPL. Claude Code finds the matching skill directory, loads SKILL.md into context, and follows the instructions.

  2. Auto-trigger. Claude Code reads the description field in the skill’s frontmatter and decides whether the current user prompt matches. If you type “review this code for security issues” and your security-audit skill’s description includes “security”, “vulnerability”, and “audit” as trigger phrases, Claude Code loads it automatically.

  3. Path-based auto-activation. The skill’s frontmatter specifies file glob patterns. When you work on files matching those patterns, the skill loads automatically. An api-conventions skill with paths: ["src/api/**"] activates whenever you edit API route handlers.

These mechanisms are not mutually exclusive. A single skill can be triggered by all three.

The system prompt injection is the critical detail. Skills are not plugins or scripts. They are instructions that reshape Claude Code’s behavior for the duration of that task. This is why well-written skills produce consistent results – the model receives the exact same instructions every time.

The SKILL.md Anatomy¶

Every skill is a single markdown file with two parts: YAML frontmatter that configures how the skill behaves, and a markdown body that contains the actual instructions Claude Code follows.

Here is a complete example with every available frontmatter field:

yaml
---
name: fix-issue
description: Fix a GitHub issue end-to-end. Use when user says "fix issue",
  "fix #N", "work on issue", or provides a GitHub issue number.
argument-hint: [issue-number]
allowed-tools: Read, Grep, Glob, Bash, Edit, Write
model: opus
effort: high
context: fork
agent: general-purpose
user-invocable: true
disable-model-invocation: false
paths:
  - "src/**"
---

# Fix Issue Workflow

## Instructions
1. Read the GitHub issue: `gh issue view $ARGUMENTS`
2. Analyze the issue description and comments
3. Search the codebase for relevant files
4. Write a failing test that demonstrates the bug
5. Implement the minimal fix
6. Run the test suite
7. Commit with message: `fix: description (fixes #$ARGUMENTS)`
8. Create a pull request linking the issue

Let us examine every frontmatter field in detail.

name¶

yaml
name: fix-issue

The name becomes the slash command. Setting name: fix-issue means the skill is invoked with /fix-issue. Names must be kebab-case (lowercase, hyphens, no spaces). The name also determines the directory name: .claude/skills/fix-issue/SKILL.md.

Choose names that are short and unambiguous. /review is better than /perform-code-quality-review. /scaffold is better than /create-new-project-from-template. Your future self will type this hundreds of times.

description¶

yaml
description: Fix a GitHub issue end-to-end. Use when user says "fix issue",
  "fix #N", "work on issue", or provides a GitHub issue number.

The description serves double duty. For humans, it appears in the / autocomplete menu so you can scan what skills are available. For Claude Code, it is the primary signal for auto-triggering.

When you type a natural language prompt without a slash command, Claude Code reads every skill’s description and decides whether to load one. If your description says “Use when user says ‘security’, ‘vulnerability’, ‘audit’, ‘OWASP’, ‘pentest’, or ‘CVE’” and the user types “check this for vulnerabilities,” the skill activates.

The most common skill problem is a vague description. If your skill does not trigger automatically when it should, the fix is almost always improving the description with more trigger phrases.

Bad:

yaml
description: Reviews code.

Good:

yaml
description: Reviews code for bugs, security issues, performance problems,
  and readability. Use when user says "review", "check my code", "code review",
  "find bugs", "audit this file", "look at this code", "is this good".

argument-hint¶

yaml
argument-hint: [issue-number]

Shown in the autocomplete tooltip when you type /fix-issue and press Tab. It is purely cosmetic – a hint to the user about what argument the skill expects. It does not validate or parse the argument.

Common patterns:

yaml
argument-hint: [issue-number]        # /fix-issue 42
argument-hint: [PR-number]           # /pr-summary 123
argument-hint: [type] [name]         # /scaffold python my-api
argument-hint: [v1..v2]              # /changelog v1.0..v2.0

The actual argument text is available inside the skill body as $ARGUMENTS. More on this in the dynamic context injection section.

allowed-tools¶

yaml
allowed-tools: Read, Grep, Glob, Bash, Edit, Write

This field lists tools that the skill can use without triggering permission prompts. Normally, Claude Code checks your settings.json permission rules before every tool call. If a skill declares allowed-tools: Bash, then Bash commands executed within that skill’s context bypass the permission check.

This is a convenience and a speed optimization. Without it, a complex skill like fix-issue would pause for permission approval on every git commit, every pytest run, every gh pr create. With it, the skill runs uninterrupted.

Security note: allowed-tools grants broad permission. Listing Bash means the skill can run any shell command. For skills shared across a team, be deliberate about what you allow. A code-review skill needs Read, Grep, Glob – it does not need Bash or Write.

model¶

yaml
model: opus

Overrides the default model for this skill. Available values: opus, sonnet, haiku.

Use this strategically:

Model When to Use Example Skills
opus Complex reasoning, security analysis, debugging fix-issue, security-audit, threat-model
sonnet Balanced speed and quality, most workflows code-review, refactor, pr-summary
haiku Fast, simple, high-volume tasks changelog, git-cleanup

A security audit that misses a vulnerability because you used a cheaper model is not a cost saving. A changelog generation that takes 30 seconds instead of 5 because you used Opus is not a quality improvement. Match the model to the cognitive demand of the task.

effort¶

yaml
effort: high

Controls the reasoning depth. Available values: low, medium, high, max.

  • low: Quick pattern matching, simple transformations. Good for formatting, renaming, simple lookups.
  • medium: Standard reasoning. Good for code review, documentation, straightforward implementation.
  • high: Deep analysis, multi-step reasoning, consideration of edge cases. Good for debugging, security audit, complex implementation.
  • max: Extended thinking with maximum token allocation. Good for threat modeling, architecture analysis, difficult debugging.

Combined with model, this gives you fine-grained control:

yaml
# Fast and cheap: simple categorization
model: haiku
effort: low

# Balanced: standard development work
model: sonnet
effort: medium

# Deep analysis: security and debugging
model: opus
effort: high

context: fork¶

yaml
context: fork

When set to fork, the skill runs in an isolated subagent context. This means it gets its own conversation context, separate from your main conversation. The subagent executes the skill’s workflow, returns the result, and its context is discarded.

Why this matters: in a long conversation where you have been discussing architecture, debugging, and planning, the main context is full of information. If you trigger a code-review skill, all that prior context is noise – it wastes tokens and can confuse the model. context: fork gives the skill a clean slate.

When to use context: fork: - Skills that are self-contained (fix-issue, scaffold) - Skills where prior conversation context would be distracting - Long-running skills that generate a lot of intermediate output

When NOT to use context: fork: - Skills that need to reference the current conversation (e.g., “review the code we just discussed”) - Quick skills where the forking overhead is not worth it

agent¶

yaml
agent: general-purpose

When context: fork is set, agent specifies which type of subagent to spawn. The value corresponds to agent definitions in .claude/agents/. If you have a custom security-reviewer agent defined, you could set agent: security-reviewer and the skill would run with that agent’s system prompt and model configuration layered on top.

user-invocable¶

yaml
user-invocable: true

Controls whether the skill appears in the / autocomplete menu. Default is true.

Set to false for skills that should only be triggered automatically (via description matching or path activation), not by direct user command. Example: an api-conventions skill that auto-loads when working in API files but is not useful as a standalone command.

disable-model-invocation¶

yaml
disable-model-invocation: false

When true, Claude Code will never auto-trigger this skill based on the description. Only explicit /command invocation works.

Use this for skills with side effects (deployment, destructive operations) or skills whose descriptions might overlap with common conversation topics. You do not want a /deploy skill auto-triggering because someone said “let’s deploy this approach to the codebase.”

yaml
# Safe: only manual trigger
name: deploy
description: Deploy the application to production
disable-model-invocation: true

paths¶

yaml
paths:
  - "src/api/**"
  - "src/routes/**"

Glob patterns that trigger automatic skill activation. When you open, edit, or reference a file matching any of these patterns, the skill loads into context automatically.

This is powerful for convention enforcement. An api-conventions skill with the paths above ensures that every time you touch an API route handler, the conventions are loaded – without you remembering to invoke anything.

yaml
# Auto-load Python testing conventions when editing tests
paths:
  - "tests/**/*.py"
  - "**/test_*.py"

Caveat: Path-activated skills load silently. If you have too many, they consume context window space without you realizing it. Keep path-activated skills focused and concise.

Dynamic Context Injection¶

Skills can execute shell commands before their content reaches Claude Code’s context. This is done with the !`command` syntax – backtick-wrapped commands prefixed with an exclamation mark.

The !command`` Syntax¶

Inside a skill body, any line containing !`command` is executed as a shell command during skill loading. The command’s output replaces the !`command` expression in the final prompt.

Here is a real-world example from a pr-summary skill:

markdown
---
name: pr-summary
description: Summarize a pull request with risk assessment
argument-hint: [PR-number]
allowed-tools: Read, Grep, Bash
---

# PR Summary

## Context
The PR diff:
!`gh pr diff $ARGUMENTS`

Changed files:
!`gh pr diff $ARGUMENTS --name-only`

PR details:
!`gh pr view $ARGUMENTS`

## Instructions
1. Analyze the diff for: breaking changes, security implications,
   performance impact, test coverage
2. Output a summary table, risk assessment, and review checklist

When you run /pr-summary 45, here is what happens:

  1. $ARGUMENTS is substituted with 45
  2. gh pr diff 45 executes and its output is injected into the prompt
  3. gh pr diff 45 --name-only executes and its output is injected
  4. gh pr view 45 executes and its output is injected
  5. Claude Code receives the fully-populated skill with all the PR data already in context

The model never has to call tools to fetch this data. It arrives pre-loaded. This saves tool call round-trips and ensures the model has everything it needs to begin analysis immediately.

$ARGUMENTS Substitution¶

$ARGUMENTS is replaced with whatever text the user provided after the slash command:

Invocation $ARGUMENTS Value
/fix-issue 42 42
/scaffold python my-api python my-api
/changelog v1.0..v2.0 v1.0..v2.0
/code-review (empty string)

The substitution happens before command execution, so gh pr diff $ARGUMENTS becomes gh pr diff 42 before the shell runs it.

When to Use Dynamic Injection vs. Tool Calls¶

Scenario Use Dynamic Injection Use Tool Calls
Data is always needed for this skill Yes No
Data is sometimes needed, depending on context No Yes
Data is large (hundreds of lines) Carefully – it consumes context Yes (model reads on demand)
Speed is critical Yes (eliminates round-trips) No
Data requires user interaction or judgment No Yes

Dynamic injection is a pre-loading optimization. If a skill always needs the PR diff, pre-load it. If a skill sometimes needs to read additional files based on what it finds, let the model decide with tool calls.

Warning: Dynamic injection output counts against your context window. Injecting a 2,000-line diff means 2,000 lines of your context budget are consumed before the model even starts reasoning. For large diffs, consider injecting only the file list and letting the model read specific files as needed.

Building Custom Skills: Three Complete Examples¶

Theory is useful. Working code is better. Here are three production-grade skills, built step by step.

Skill 1: /commit – Consistent Git Commits¶

The problem: you want every commit to follow the same format – conventional commit messages, co-authorship attribution, pre-commit verification. Typing these instructions every time is tedious and error-prone.

Create the directory:

bash
mkdir -p .claude/skills/commit

Create the skill:

markdown
---
name: commit
description: Create a well-formatted git commit. Use when user says "commit",
  "save changes", "commit this", or "check in".
allowed-tools: Bash, Read, Grep
model: sonnet
effort: medium
disable-model-invocation: true
---

# Commit Workflow

## Pre-Commit Checks
1. Run `git status` to see all changes (staged and unstaged)
2. Run `git diff` to review what will be committed
3. Run `git log --oneline -5` to see recent commit message style

## Commit Message Format
Use conventional commits:
- `feat:` new feature
- `fix:` bug fix
- `docs:` documentation only
- `refactor:` code restructuring, no behavior change
- `test:` adding or updating tests
- `chore:` maintenance tasks

Write a concise message (under 72 characters) that explains WHY,
not WHAT. The diff shows what changed. The message explains the intent.

## Execution
1. Stage relevant files (prefer specific files over `git add -A`)
2. Do NOT stage files that appear to contain secrets (.env, *.pem,
   credentials.*, token.*)
3. Create the commit with:
   ```
   git commit -m "$(cat <<'EOF'
   <type>: <description>

   Co-Authored-By: Claude <noreply@anthropic.com>
   EOF
   )"
   ```
4. Run `git status` to verify the commit succeeded
5. Show the commit hash and summary

## Rules
- NEVER amend a previous commit unless explicitly asked
- NEVER force push
- If pre-commit hooks fail, fix the issue and create a NEW commit
- If there are no changes to commit, say so and stop

Note the disable-model-invocation: true setting. You do not want Claude Code auto-triggering a commit just because you mentioned the word “commit” in conversation. Commits are explicit actions.

Skill 2: /code-review – Severity-Ranked Findings¶

The problem: code reviews are inconsistent. Sometimes you get surface-level feedback, sometimes deep analysis. You want every review to follow the same priority order and output format.

markdown
---
name: code-review
description: Reviews code for bugs, security issues, performance,
  and readability. Use when user says "review", "check my code",
  "code review", "find bugs", "audit this file", "look at this",
  "is this good", "what do you think of this code".
allowed-tools: Read, Grep, Glob
model: sonnet
effort: high
---

# Code Review

## Review Order (this order is mandatory)
1. **Security** -- injection, auth bypass, data exposure, secrets
2. **Correctness** -- logic errors, race conditions, null handling, edge cases
3. **Performance** -- N+1 queries, unnecessary allocations, missing indexes
4. **Readability** -- naming, structure, complexity, missing types

## Output Format
For each finding, use exactly this format:

**[SEVERITY] file_path:line_number -- Description**
Fix: concrete code change

Severity levels:
- **CRITICAL** -- Will cause data loss, security breach, or crash
  in production. Must fix before merge.
- **IMPORTANT** -- Bug or significant issue. Should fix before merge.
- **SUGGESTION** -- Improvement. Nice to have, not blocking.

## Rules
- Read the ENTIRE file before making findings. Do not review
  line-by-line as you read.
- Group findings by severity, CRITICAL first.
- Skip nitpicks. If it works and is readable, do not comment.
- Include the fix, not just the problem.
- If the code is clean, say "No significant issues found" and stop.
  Do not invent findings.
- Reference specific line numbers for every finding.
- If you find zero CRITICAL or IMPORTANT issues, say so explicitly
  so the author knows it was reviewed, not skipped.

This skill uses model: sonnet rather than opus. Code review is well within Sonnet’s capability for most codebases, and the speed difference matters when you are reviewing multiple files in a session.

Skill 3: /scaffold – Project Bootstrapping¶

The problem: every new project starts with the same boilerplate – directory structure, config files, linting setup, test framework, gitignore. You want a single command that generates a production-ready project skeleton.

markdown
---
name: scaffold
description: Bootstrap a new project with proper structure and tooling.
  Use when user says "new project", "scaffold", "bootstrap", "init",
  "create project", "set up a new", "start a new".
argument-hint: [type] [name]
allowed-tools: Bash, Write, Read
model: sonnet
effort: medium
---

# Project Scaffold

## Arguments
$ARGUMENTS should be: [type] [name]
- type: python, typescript, node, rust
- name: project name (kebab-case)

Example: `/scaffold python my-api`

## Python Project Structure

$NAME/ ├── src/ │ └── $NAME/ │ ├── init.py │ └── main.py ├── tests/ │ ├── init.py │ ├── conftest.py │ └── test_main.py ├── pyproject.toml ├── .gitignore ├── .python-version └── README.md


## pyproject.toml Template
Include:
- Project metadata (name, version 0.1.0, requires-python >= 3.12)
- Dependencies section (empty)
- Dev dependencies: pytest, pytest-asyncio, ruff, pyright
- Ruff config: line-length 88, target Python 3.12

## TypeScript Project Structure

$NAME/ ├── src/ │ └── index.ts ├── tests/ │ └── index.test.ts ├── package.json ├── tsconfig.json ├── .gitignore └── README.md


## After Scaffolding
1. Initialize git: `git init`
2. Create initial commit: `git add -A && git commit -m "chore: initial scaffold"`
3. Run linter to verify clean start
4. Run test suite to verify it passes
5. Show the project tree and confirm everything is working

## Rules
- Always use the latest stable language version
- Always include a test that passes out of the box
- Always include linting and type checking config
- README should contain: project name, one-line description,
  setup instructions, and how to run tests
- .gitignore should cover: OS files, IDE files, build artifacts,
  virtual environments, dependency directories

Notice how the skill provides exact directory structures and specific configuration requirements. Vague instructions like “set up testing” produce inconsistent results. Explicit instructions like “dev dependencies: pytest, pytest-asyncio, ruff, pyright” produce the same result every time.

Auto-Activating Skills¶

Some skills should load without any explicit invocation. They enforce conventions by being present whenever you work in certain areas of the codebase.

Example: API Conventions¶

markdown
---
name: api-conventions
description: Enforces API coding conventions. Auto-activates for API files.
user-invocable: false
paths:
  - "src/api/**"
  - "**/routes/**"
  - "**/endpoints/**"
---

# API Conventions

When working in API route handlers, follow these conventions:

## Request Validation
- Validate all request parameters at the handler level
- Use Pydantic models for request/response schemas
- Return 422 for validation errors with field-level details

## Response Format
All responses follow this structure:
```json
{
  "data": {},
  "error": null,
  "meta": { "request_id": "...", "timestamp": "..." }
}

Error Handling¶

  • 400: Bad request (client error in request format)
  • 401: Unauthorized (missing or invalid authentication)
  • 403: Forbidden (authenticated but insufficient permissions)
  • 404: Not found
  • 422: Validation error (request parsed but semantically invalid)
  • 500: Internal server error (never expose internal details)

Security¶

  • Never include stack traces in production error responses
  • Log the full error server-side with the request_id
  • Rate limit all public endpoints
  • Validate Content-Type header matches expected format

This skill has `user-invocable: false` because `/api-conventions` is not a meaningful standalone command. It exists to inject conventions into context whenever you touch API files. You never think about it. It is always there when you need it.

### Combining Path Activation with Explicit Invocation

A skill can be both path-activated and manually invocable:

```yaml
name: test-conventions
description: Testing conventions and patterns
paths:
  - "tests/**"
  - "**/test_*.py"

This loads automatically when you edit test files, and you can also explicitly invoke /test-conventions when you want to review the conventions without opening a test file.

Skill Libraries: Organizing Skills for a Team¶

When skills are in .claude/skills/ and committed to the repository, they are shared with everyone who clones the repo. This is powerful – and requires discipline.

Directory Structure for Team Skills¶

.claude/
└── skills/
    ├── commit/
    │   └── SKILL.md
    ├── code-review/
    │   └── SKILL.md
    ├── security-audit/
    │   ├── SKILL.md
    │   └── references/
    │       ├── owasp-top-10.md
    │       └── cwe-checklist.md
    ├── fix-issue/
    │   └── SKILL.md
    ├── pr-summary/

    └── api-conventions/

Note the references/ subdirectory in security-audit. Skills can include additional files alongside SKILL.md. The skill body can reference these files with instructions like “Read the OWASP checklist in this skill’s references/ directory.” This keeps specialized knowledge co-located with the skill that uses it.

Personal vs. Team Skills¶

Location Visibility Use Case
.claude/skills/ (project) Everyone on the team Team standards, project workflows
~/.config/claude/skills/ (global) Only you Personal preferences, cross-project tools

Keep team skills focused on team needs: commit conventions, review standards, deployment checklists. Keep personal skills for your own workflow: your preferred research format, your note-taking style, your debugging approach.

Version Control Considerations¶

Skills in .claude/skills/ are code. Treat them like code:

  • Review skill changes in PRs. A skill that changes the deployment checklist affects every engineer on the team.
  • Test skills after modification. Run the skill against a known input and verify the output matches expectations.
  • Document skill purpose. The description field is not just for Claude Code – it is for your teammates who see the skill in the autocomplete menu.
  • Keep skills small. A 500-line skill is hard to review, hard to maintain, and consumes context budget. If a skill is growing beyond 200 lines, consider whether it should be two skills.

Skill Naming Conventions¶

Establish a convention for your team. Consistent naming makes the skill library scannable.

Convention Example When
Action-noun /review-pr, /fix-issue, /audit-deps Most skills
Noun /changelog, /scaffold When the action is obvious
Domain-prefix /api-review, /db-migrate When you have domain-specific skills

Avoid: - Names that overlap with built-in commands (/help, /clear, /compact) - Names longer than 20 characters - Names with underscores (use hyphens)

What Most People Get Wrong¶

After building and reviewing hundreds of skill configurations, these are the recurring mistakes.

1. The Description Is Too Vague¶

The most common problem. Your skill does not auto-trigger because the description says “Reviews code” instead of listing the twelve phrases that should trigger it. Write the description as if you are teaching a junior engineer which Slack messages should route to this skill.

2. The Instructions Are Too Abstract¶

“Review the code for issues” produces wildly different results depending on the model’s mood. “Check in this order: (1) SQL injection, (2) authentication bypass, (3) data exposure, (4) hardcoded secrets. For each finding, output severity, CWE number, file path, line number, and a concrete fix” produces the same result every time.

Specificity is not rigidity. It is reliability.

3. Overusing context: fork¶

Forking creates a clean context, but it also loses all conversational history. If you have been discussing a complex architecture for twenty minutes and then invoke a skill that forks, the skill has no idea what you were talking about. Use fork for standalone tasks. Keep the main context for tasks that build on the conversation.

4. Allowing Too Many Tools¶

allowed-tools: Read, Grep, Glob, Bash, Edit, Write, WebFetch, WebSearch, Agent is not a skill configuration. It is an abdication of configuration. Every tool in that list is a tool the skill can use without permission checks. A code-review skill that can write files and run shell commands is a code-review skill that might accidentally modify your codebase during review. Be deliberate: grant the minimum tools necessary.

5. Building Monolithic Skills¶

A single skill that handles “review code, fix issues, write tests, and deploy” will do all four things poorly. Skills work best when they are focused on one workflow. Build four skills, not one mega-skill. Each skill is better because it is smaller, more specific, and easier to iterate on.

6. Not Testing the Skill¶

You write a SKILL.md, use it once, and ship it to the team. The first time a teammate uses it, the output is different from what you expected because the model interpreted an ambiguous instruction differently.

Test your skills. Run them against known inputs. Check that the output format matches. Check that the priority order is correct. Check that edge cases are handled (what happens when there are no findings? what happens when the argument is empty?).

7. Ignoring Context Budget¶

Every skill instruction, every dynamic injection, every reference file consumes context tokens. A skill that injects a 3,000-line diff plus a 500-line OWASP checklist plus a 200-line instruction set has consumed a significant portion of the context window before the model starts working. Monitor your context usage with /context after loading heavy skills. If you are consistently hitting context limits, your skills are too large.

8. Duplicating Logic Between Skills and CLAUDE.md¶

If your CLAUDE.md says “always use conventional commits” and your /commit skill also says “use conventional commits,” you have duplication. When you update one, you will forget the other. Put universal rules in CLAUDE.md or rules files. Put workflow-specific instructions in skills. Never put the same instruction in both places.

Building a Skill Development Workflow¶

The best approach to building skills is iterative:

  1. Manual first. Do the workflow manually three times. Note exactly what steps you follow, what tools you use, and what format you want the output in.

  2. Draft the skill. Write the SKILL.md based on your manual workflow. Be more specific than you think necessary.

  3. Test against known input. Run the skill against a file or scenario where you already know what the correct output looks like. Compare.

  4. Adjust specificity. If the output varies between runs, your instructions are too vague somewhere. Find where and add specificity.

  5. Test the auto-trigger. Say the phrases you think should trigger the skill without using the slash command. If it does not trigger, improve the description.

  6. Ship and monitor. Commit the skill. Ask a teammate to use it. Get feedback on the first run.

This workflow takes 30 minutes per skill. The return is every future invocation producing consistent, high-quality results without any prompt engineering from the user.

Chapter Summary¶

Five things to remember about skills:

  1. Skills are instructions, not code. They reshape Claude Code’s behavior by injecting instructions into the system prompt. The quality of the skill is the quality of the instructions.

  2. The description field controls auto-triggering. If your skill does not activate when it should, the description is the problem. List every phrase that should trigger it.

  3. Dynamic context injection (!command``) eliminates round-trips. Pre-load data the skill always needs. Let the model fetch data it sometimes needs via tool calls.

  4. Specificity produces consistency. “Check security, then logic, then performance” with exact output formats produces the same result every time. “Review the code” does not.

  5. Skills are team infrastructure. Project skills in .claude/skills/ are shared through version control. Treat them with the same rigor as production code – review changes in PRs, test after modification, keep them focused and small.

The next chapter covers agents – specialized subagent configurations that give you model and tool isolation for complex tasks. Skills tell Claude Code what to do. Agents tell it who to be while doing it. As discussed in Chapter 1, the configuration stack layers these primitives: CLAUDE.md sets the foundation, rules scope the standards, skills define the workflows, and agents provide the expertise. We are building up, one layer at a time.
