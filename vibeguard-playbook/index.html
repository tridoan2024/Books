Part I — Pre-Development: Think Before You Code

Chapter 1: Proposal Review & Enhancement

A principal engineer never starts coding from a raw idea. You validate, research, challenge, and refine the proposal BEFORE writing a single line of code.

1.1 Set Up Your rcode Environment

Open terminal. Create the project. Launch rcode.

mkdir -p vibeguard && cd vibeguard
git init
rcode

First thing — teach rcode your standards. This is what separates a principal from a junior. A junior types "build me a security scanner." A principal configures the AI's brain first:

> /init

This creates .rcode/ with default settings. Now configure it:

> /config set model claude-opus-4
> /config set effort high
> /config set permission_mode ask

Why Opus + high effort for this phase? Architecture decisions have the highest blast radius. A wrong choice here costs weeks. Pay for the best reasoning now.

1.2 Deep Proposal Review Using rcode

Feed rcode the proposal. Don't say "build this." Say "tear this apart":

> I have a product proposal for VibeGuard — a security scanner for AI-generated code.
  Before we write any code, I need you to act as a principal engineer doing a
  proposal review. Read this specification and give me:

  1. STRENGTHS — What's well-defined and viable
  2. GAPS — What's missing from the spec that we'll need to figure out
  3. RISKS — Technical risks, market risks, execution risks
  4. UNKNOWNS — Things we need to research before committing
  5. SCOPE CONCERNS — Is this too big for v1? What should we cut?
  6. COMPETITIVE ANALYSIS — Who else does this? How are we different?

  Here's the spec:
  [paste the VibeGuard specification]

rcode advantage over dcode: rcode's /think command activates extended thinking, giving the model more reasoning tokens for complex analysis:

> /think on

Now rcode uses extended thinking for every response — essential for architecture work.

1.3 Research Phase Using Multi-Agent Swarm

This is where rcode's swarm system crushes dcode. Instead of researching one thing at a time, you deploy a research team:

> /tasks swarm research "Research the competitive landscape for AI code security scanners.

  Researcher 1: Analyze existing SAST tools (Snyk, SonarQube, Semgrep, Checkmarx).
  What do they do well? What do they miss for AI-generated code?

  Researcher 2: Research the hallucinated package problem. Find real CVEs, published
  research papers, and documented incidents of AI suggesting fake packages.

  Researcher 3: Research MCP security. What vulnerabilities exist in Model Context
  Protocol configurations? What are the OWASP guidelines for AI agents?

  Writer: Synthesize all findings into a competitive analysis document."

The swarm system (src/swarm/) deploys 4 agents in parallel: - 3 Researchers (each with ["research", "analysis"] capabilities) - 1 Writer (with ["writing", "summarisation"] capabilities) - Communication mode: Broadcast (all share findings) - Each agent gets its own git worktree for isolation

dcode cannot do this. It has no swarm system. You'd research one topic at a time.

1.4 Save Research to Memory

After the swarm completes, save the findings permanently:

> /memory add vibeguard-competitive-landscape "SAST landscape 2026: Snyk (dep scanning,
  no AI rules), SonarQube (code quality, no hallucination detection), Semgrep (pattern
  matching, no MCP auditing). Gap: NONE scan for hallucinated packages or MCP configs.
  VibeGuard unique value: AI-specific rules + multi-model consensus." project

> /memory add vibeguard-hallucinated-packages "Hallucinated package attacks: documented
  since 2024. AI suggests plausible but nonexistent package names. Attackers register
  them on npm/PyPI with malicious code. 38% of Copilot suggestions reference packages
  that don't exist in top-10K npm. Critical gap in all existing SAST tools." project

Why memory matters: Next week when you return to this project, run /memory search vibeguard and all your research is instantly available. dcode has no persistent memory between sessions.

Chapter 2: Product Requirements Document (PRD)

2.1 Create a Skill for PRD Generation

A principal doesn't manually write PRDs from scratch every time. Create a reusable skill:

mkdir -p .rcode/skills/prd

Create .rcode/skills/prd/SKILL.md:

---
name: prd
description: "Generate a Product Requirements Document from a product spec"
model: claude-opus-4
effort: high
user_invocable: true
argument_hint: "product name"
---

# PRD Generator

You are a senior product manager at a top-tier tech company. Generate a comprehensive
Product Requirements Document for $ARGUMENTS.

## Required Sections

1. **Executive Summary** — 2-3 sentences on what this is and why it matters
2. **Problem Statement** — What pain exists, who feels it, how big is it
3. **Target Users** — Personas with names, roles, pain points, goals
4. **User Stories** — Epics broken into stories with acceptance criteria
5. **Functional Requirements** — Numbered, testable, with priority (P0/P1/P2)
6. **Non-Functional Requirements** — Performance, security, scalability, reliability
7. **Success Metrics** — KPIs with targets and measurement methods
8. **Scope** — What's IN v1, what's OUT (and why), what's FUTURE
9. **Dependencies & Risks** — Technical, market, resource risks with mitigations
10. **Timeline** — Phase breakdown with milestones

## Context

Current project state:
!`cat CLAUDE.md 2>/dev/null || echo "No CLAUDE.md yet"`

Git status:
!`git log --oneline -5 2>/dev/null || echo "No commits yet"`

Research findings:
!`cat docs/research/*.md 2>/dev/null || echo "No research docs yet"`

## Rules
- Every requirement must be testable
- Every user story must have acceptance criteria
- Scope must be ruthlessly prioritized for v1
- Write for engineers, not executives

Now invoke it:

> /prd VibeGuard

rcode loads the skill, runs the !command` injections to gather context, substitutes$ARGUMENTS` with "VibeGuard", and generates a complete PRD.

Save it:

> Save the PRD to docs/PRD.md

2.2 Review the PRD Using /review

> /review docs/PRD.md --strict

rcode's review system (src/commands/review.rs) analyzes the document with severity-rated findings: are requirements testable? Are acceptance criteria specific? Are success metrics measurable?

2.3 Iterate with /passes

Use multi-pass refinement to make the PRD production-quality:

> /passes start --pipeline thorough

This runs the thorough pipeline: Draft → Review → Refine → Verify → Polish. Each pass builds on the previous one. The Draft pass generates, Review critiques, Refine addresses feedback, Verify checks consistency, Polish fixes formatting.

dcode has no multi-pass system. You'd manually ask "now review it" then "now fix the issues" — losing context each time.

Chapter 3: Software Design Document (SDD)

3.1 Architecture Design with Swarm

Deploy the full_stack team template for architecture work:

> /tasks swarm full_stack "Design the complete software architecture for VibeGuard.

  Architect: Design the system architecture:
  - Rust workspace structure (crates/engine, crates/rules, crates/wasm, crates/cli)
  - TypeScript packages (packages/cli, packages/vscode, packages/shared)
  - Data flow: vg scan → file discovery → rule matching → finding aggregation → output
  - The ScanRule trait: how rules register, execute, and report findings
  - SARIF output format integration
  - Plugin architecture for custom rules
  - Performance: <5 seconds for 500 files
  - Show component diagram, sequence diagrams, and dependency graph

  Coder 1: Define all core types in Rust:
  - ScanTarget, Finding, ScanResult, RuleConfig, VibeguardConfig
  - ScanRule trait with scan() method
  - RuleRegistry for rule management
  - OutputFormatter trait for SARIF/JSON/text
  - All types must have serde derives and doc comments

  Coder 2: Define the AI reasoning types:
  - AiProvider trait with analyze_finding() method
  - MultiModelConsensus with voting logic
  - CostTracker for per-scan cost limits
  - ContextBuilder for code extraction around findings
  - Prompt templates for each rule category

  Tester: Write the test strategy document:
  - Unit test approach for each rule (true positive + false positive cases)
  - Integration test approach for end-to-end scanning
  - Performance benchmarks (target: <100ms per file)
  - Security testing (scanner must not execute scanned code)
  - CI/CD test pipeline design"

The full_stack template deploys: - 1 Architect (["design", "architecture"]) - 2 Coders (["coding", "implementation"]) - 1 Tester (["testing", "qa"]) - Communication: Parallel (all work simultaneously) - Timeout: 600 seconds

3.2 Architecture Decision Records

> Create ADRs for every major technical decision. Save to docs/adr/:

  ADR-001: Why Rust + WASM (not pure TypeScript)
  ADR-002: Why tree-sitter (not language-specific parsers)
  ADR-003: Why SARIF output (not custom JSON)
  ADR-004: Why multi-model consensus (not single model)
  ADR-005: Why open-core business model (not SaaS-only)
  ADR-006: Why MIT license for core (not Apache 2.0 or AGPL)

  Each ADR must follow this format:
  - Status: Accepted
  - Context: What problem are we solving
  - Decision: What we chose
  - Consequences: Good and bad
  - Alternatives Considered: What we rejected and why

3.3 Create CLAUDE.md — The Project Brain

This is the most important file in any rcode project. It encodes your engineering standards so rcode (and any future developer) produces consistent, high-quality code:

> Create CLAUDE.md for the VibeGuard project. This is a Rust + TypeScript hybrid
  security scanner. Include:

  ## Architecture Principles
  - Hexagonal architecture: scanning engine has zero external dependencies
  - Every rule is a pure function: input file → output findings (no side effects)
  - Plugin system via dynamic loading (not compiled-in)
  - Cost-capped AI: hard limit per scan, graceful degradation

  ## Coding Standards — Rust
  - Edition 2024, no unsafe except where required by FFI
  - No unwrap() except in tests — use anyhow::Result everywhere
  - Every pub fn gets a doc comment with /// and # Examples
  - Error types per crate, aggregated at binary boundary
  - Use thiserror for library errors, anyhow for binary errors
  - Clippy pedantic, deny all warnings in CI
  - Format with rustfmt, enforce in CI

  ## Coding Standards — TypeScript
  - Strict mode, no any types
  - ESLint + Prettier, enforce in CI
  - All functions typed, no implicit returns

  ## Testing Standards
  - Every rule: minimum 10 test cases (5 true positive, 5 true negative)
  - Integration tests use real files in test-fixtures/
  - Performance benchmarks in benches/
  - No mocks for file I/O — use tempdir

  ## Security Standards (we're a security tool — we MUST be secure ourselves)
  - Never execute scanned code
  - Sandbox file access to scan target directory
  - All network calls (registry lookup, AI API) are opt-in
  - No telemetry without explicit consent
  - Credential handling: memory-only, zero-to-disk

  ## Git Workflow
  - Branch naming: feat/, fix/, docs/, refactor/, test/, chore/
  - Commit messages: imperative mood, reference issue numbers
  - PR required for all changes, minimum 1 approval
  - CI must pass before merge
  - Squash merge to main

Chapter 8: Code Review Process

8.1 Self-Review Before PR

Before pushing, review your own code with rcode's review command:

> /model opus
> /effort high
> /review src/ --strict

The review system (src/commands/review.rs) produces severity-rated findings: - 🔴 Critical (blocks merge) - 🟠 High (blocks merge) - 🟡 Medium - 🟢 Low - ℹ️ Info - 💅 Nitpick

Each finding has: file:line, category (Bug/Security/Performance/...), description, and suggested fix.

8.2 Multi-Pass Review

For critical modules (the scanning engine, permission system), use multi-pass:

> /passes start --pipeline thorough

Pass 1 (Draft): "Here's the code" Pass 2 (Review): "Find every issue — bugs, security, performance, style" Pass 3 (Refine): "Fix every issue found in Pass 2" Pass 4 (Verify): "Verify all fixes are correct, write tests for each fix" Pass 5 (Polish): "Final cleanup — comments, naming, formatting"

8.3 Security Review Using Swarm

For security-critical code, deploy the code_review team:

> /tasks swarm code_review "Security review of crates/engine/src/orchestrator.rs.

  Coder: Read the orchestrator code and explain every security decision.

  Reviewer 1: Review for input validation vulnerabilities:
  - Can a malicious file path escape the scan directory?
  - Can a crafted file cause the scanner to execute code?
  - Can a crafted .vibeguard.yml inject malicious rules?

  Reviewer 2: Review for resource exhaustion:
  - Can a huge file cause OOM?
  - Can deeply nested directories cause stack overflow?
  - Can a symlink loop cause infinite scanning?

10.2 STRIDE Threat Model

> /model opus

> Create a STRIDE threat model for VibeGuard. Cover every component:
  - CLI binary (user input, file system access)
  - Scanning engine (processes untrusted code)
  - AI reasoning engine (sends code to external APIs)
  - Configuration system (loads YAML from disk)
  - Plugin system (loads third-party code)
  - CI/CD integration (runs in automated pipelines)
  - VS Code extension (runs in user's IDE)

  S - Spoofing: Can identity be faked?
  T - Tampering: Can data be modified?
  R - Repudiation: Can actions be denied?
  I - Information Disclosure: Can data leak?
  D - Denial of Service: Can service be disrupted?
  E - Elevation of Privilege: Can permissions be escalated?

  Rate each: Low/Medium/High risk
  Propose mitigations for all Medium and High risks
  Save to docs/security/THREAT_MODEL.md
