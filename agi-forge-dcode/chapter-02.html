Chapter 2: Claude Code Architecture — The Keystone

The 9-Layer Stack, 10 Extension Points, and the Streaming Agent Loop That Powers Everything

You are about to write a hook that fires on PreToolUse and blocks dangerous bash commands. But you need to know: does the hook receive the command before or after the permission check? Does it get a raw string or a JSON object? What exit code blocks execution versus logging a warning?

These are not trivia questions. They are the difference between a hook that protects your system and one that silently fails while rm -rf . executes underneath it.

You cannot answer them without understanding the architecture. This chapter is that understanding — the complete technical map of Claude Code from the CLI entry point to the terminal pixel. Every configuration decision in this book — how you structure CLAUDE.md, what permissions to set, which lifecycle events to hook, how to decompose the monolith into specialized CLIs — flows from what you learn here.

2.1 The 9-Layer Architecture

Claude Code is organized as a 9-layer stack. Each layer has a specific responsibility, clear boundaries with adjacent layers, and defined extension points for customization. Understanding which layer to modify for a given goal is the difference between a clean extension and a fragile hack.

Layer 1: Entry Points

Claude Code has four distinct entry modes, each serving a different deployment pattern:

Entry Mode File Purpose
Interactive CLI cli.tsx → main.tsx The default. Terminal REPL with React/Ink TUI
MCP Server mcp.ts Expose Claude Code as an MCP server. Other tools connect to it
HTTP/WebSocket Server server/ Remote sessions via cc:// URLs. SDK bridges for VS Code, Cursor
SDK Stub agentSdkTypes.ts npm package @anthropic-ai/sdk-claude-code. Programmatic API

The CLI entry point (cli.tsx) is a fast-path dispatcher. It checks for simple flags (--version, --mcp, --daemon) before loading the full React application. The -p (non-interactive) mode skips 52 subcommand registrations, saving approximately 65ms of startup time. The --bare mode gates off roughly 90% of the startup sequence.

Boot sequence performance optimizations:

cli.tsx (fast-path dispatch, ~5ms)
  ↓
main.tsx with import-time parallelism:
  - profileCheckpoint('main_tsx_entry')
  - startMdmRawRead() + startKeychainPrefetch()   ← parallel with 135ms of requires
  ↓
setup():
  - Node.js ≥18 check
  - UDS messaging server initialization
  - setCwd() (critical: must be early, other systems depend on it)
  - FileChanged watcher registration
  - Background jobs: memory loading, plugin discovery, hook wiring
  ↓
launchRepl() via dynamic import (keeps React off the critical path)
  ↓
startDeferredPrefetches(): user context warming, model capability probing

Multi-CLI implication: Each specialized CLI needs a different entry mode. The Orchestrator uses the SDK entry for programmatic spawning. The Research CLI might run as an MCP server, exposing its findings as resources. The Coder CLI uses the interactive entry for full tool access. Understanding entry modes lets you choose the right surface for each specialist.

Layer 2: The Query Engine (The Agentic Loop)

This is the single most important component in Claude Code. Everything else is configuration and tooling around this loop.

The QueryEngine (query.ts, QueryEngine.ts) implements a while(true) agentic loop that alternates between LLM inference and tool execution:

┌────────────────────────────────────────────────────────────┐
│                    THE QUERY ENGINE                         │
│                                                            │
│  1. Assemble context                                       │
│     ├── System prompt (CLAUDE.md + rules + skills + memory)│
│     ├── Conversation history                               │
│     ├── Tool schemas (52 tools = ~10K tokens)             │
│     └── Git context + directory listing                    │
│                                                            │
│  2. Call Claude API (streaming SSE)                        │
│     └── Response contains: thinking + text + tool_use      │
│                                                            │
│  3. Process response                                       │
│     ├── stop_reason == "end_turn" → EXIT LOOP              │
│     └── stop_reason == "tool_use" → CONTINUE               │
│                                                            │
│  4. For each tool_use in response:                         │
│     a. Permission check (deny → ask → allow cascade)       │
│     b. PreToolUse hooks (exit 2 = BLOCK)                  │
│     c. Execute tool                                        │
│     d. PostToolUse hooks (auto-lint, notify, log)         │
│     e. Truncate result if > threshold                      │
│     f. Append tool_result to conversation                  │
│                                                            │
│  5. Check turn count (max ~100, configurable)              │
│     └── Exceeded → EXIT LOOP with warning                  │
│                                                            │
│  6. Check context budget                                   │
│     ├── Near limit → trigger auto-compaction               │
│     └── At limit → emergency context collapse              │
│                                                            │
│  7. LOOP BACK TO STEP 2                                    │
│                                                            │
│  Post-loop side effects:                                   │
│  ├── Stop hooks                                            │
│  ├── TaskCompleted hooks (if in teammate mode)             │
│  ├── TeammateIdle hooks                                    │
│  ├── Memory extraction (background subagent)               │
│  ├── Auto-dream check (24h cooldown, 5-session minimum)   │
│  └── Prompt suggestion generation                          │
└────────────────────────────────────────────────────────────┘

17-step per-iteration pipeline (the full internal sequence):

  1. Read pending messages from UDS inbox (if UDS_INBOX enabled)
  2. Assemble system prompt (layers 1–8 of context system)
  3. Apply prompt caching (two-phase: static global + dynamic per-org)
  4. Build API request with beta headers
  5. Stream API response via SSE
  6. Parse content blocks as they arrive (thinking, text, tool_use)
  7. For each tool_use: extract tool name and input from JSON stream
  8. Permission check (4-layer cascade: managed → user → project → local)
  9. If permission mode is auto: run 2-stage TRANSCRIPT_CLASSIFIER
  10. Fire PreToolUse hooks (can block with exit code 2)
  11. Execute tool via StreamingToolExecutor
  12. Fire PostToolUse hooks
  13. Truncate oversized results
  14. Append tool_result to conversation history
  15. Update token budget tracking
  16. Check compaction triggers
  17. Loop or exit based on stop_reason

Multi-CLI implication: The query engine is the component every specialized CLI shares. The difference between CLIs is what tools are registered (step 11), what permissions are configured (step 8), and what system prompt is assembled (step 2). The loop itself is identical. This means a multi-CLI system is not 8 separate codebases — it is 8 configurations of the same engine.

Layer 3: The Context System

Every turn, Claude Code assembles a context window from nine sources. The order matters — later sources take priority, and the total must fit within the model's context limit.

Source Typical Tokens Content
1. Base system prompt 3,000–5,000 Core instructions, tool usage rules, behavior guidelines
2. CLAUDE.md hierarchy 500–2,000 ~/.claude/CLAUDE.md → project → local (merged, project wins)
3. Matching rules 200–600 Rules whose paths: frontmatter matches files in scope
4. Active skill instructions 200–800 Loaded by /command or auto-detection
5. Git context 200–500 Branch, recent commits, status, dirty files
6. Directory listing 200–1,000 Project file structure snapshot
7. Memory context 100–500 MEMORY.md index + relevant memory files
8. Tool definitions 3,000–10,000 JSON schemas for all registered tools
9. Conversation history 50,000–150,000 Messages, tool calls, tool results

Total system prompt: 7,000–15,000 tokens. This is the overhead paid on every API call.

The CLAUDE.md trick: Claude Code places CLAUDE.md content inside a synthetic user message rather than the system message. This is a prompt-caching optimization — the system prompt remains static (maximizing cache hits), while CLAUDE.md content rides in the user turn where it is still visible to the model but does not invalidate the system prompt cache.

Two-phase prompt caching:

Phase 1: Static Global Cache
  ├── Base system prompt (rarely changes)
  ├── Tool schemas (change only when tools are added/removed)
  └── Global settings
  → Cache TTL: Long (content-hash based, survives across sessions)

Phase 2: Dynamic Per-Org Cache
  ├── CLAUDE.md content
  ├── Git context (changes per turn)
  ├── Memory context
  └── Conversation history
  → Cache TTL: Short (invalidated by content changes)

Content-hash temp files are used to preserve cache validity across sessions. The system writes deterministic temporary files whose paths are derived from content hashes, so the API's prompt cache sees the same "file" path and reuses cached prefill.

Multi-CLI implication: Each specialized CLI assembles a different context. The Security CLI loads security-focused rules and a CLAUDE.md with threat-modeling instructions. The Coder CLI loads coding-standard rules and a CLAUDE.md with implementation guidelines. The tool schemas section varies because each CLI registers different tools. This is the primary mechanism for CLI specialization — not code changes, but context changes.

The 40-member tool interface:

Every tool implements a protocol with approximately 40 methods and properties. The critical ones for multi-CLI design:

interface Tool<Input, Output, Progress> {
  // === Identity ===
  name: string;
  category: string;
  description(): string;            // Shown to the model
  prompt(): string;                 // Usage instructions for the model

  // === Input/Output ===
  inputSchema: ZodSchema;           // Zod schema (validated before execution)
  call(input, context): Promise<ToolResult>;

  // === Behavior Flags (fail-closed: all default false) ===
  isReadOnly(): boolean;            // Can this tool modify state?
  isConcurrencySafe(): boolean;     // Can run parallel with other tools?
  isDestructive(): boolean;         // Could cause irreversible damage?
  isEnabled(): boolean;             // Should this tool be available?
  shouldDefer: boolean;             // Lazy-loaded (fetched on demand)?
  alwaysLoad: boolean;              // Always in tool pool regardless of filtering?

  // === Permission Integration ===
  checkPermissions(input): Promise<PermissionDecision>;
  validateInput(input): Promise<void>;
  toAutoClassifierInput(): ClassifierInput;  // Feeds the AI permission classifier

  // === Rendering (React/Ink) — 7 methods ===
  renderToolUseMessage(input): ReactNode;
  renderToolResultMessage(result): ReactNode;
  renderToolUseProgressMessage(progress): ReactNode;
  // ... 4 more rendering methods

  // === MCP/LSP Markers ===
  isMcp: boolean;
  isLsp: boolean;
}

The tool assembly pipeline:

getAllBaseTools()                    → All tools for the current environment
  ↓
getTools(permissionContext)         → Filter by deny rules + isEnabled()
  ↓
assembleToolPool(permCtx, mcp)     → Merge built-in + MCP tools, sort for cache stability
  ↓
filterToolsByDenyRules()           → Strip denied tools before model sees them

The sort step is critical for prompt caching: tools are sorted deterministically so the tool schema section of the system prompt has a stable hash across turns.

Multi-CLI implication: The primary differentiation mechanism between CLIs. The Orchestrator registers only AgentTool, TaskCreate/Update/List, SendMessage, SyntheticOutput — it literally cannot edit files. The Security CLI registers Read, Grep, Glob, and custom SAST tools — it can observe but not modify. The Coder CLI gets the full file/shell tool set. Tool filtering is how you enforce the principle that each specialist does one thing well and cannot do the rest.

Layer 5: The Permission System

Four permission modes, evaluated in a specific cascade:

Managed (enterprise, cannot override)
  → User (~/.claude/settings.json)
    → Project (.claude/settings.json, committed to git)
      → Local (.claude/settings.local.json, gitignored)

The 2-stage TRANSCRIPT_CLASSIFIER:

When permission mode is auto (the AI-powered mode), Claude Code runs a two-stage security classifier:

Tool request arrives
  ↓
Stage 1: Fast Heuristics (~1ms)
  ├── Is it "rm -rf /"? → DENY
  ├── Is it a known-safe read-only tool? → ALLOW
  ├── Does it match an explicit deny pattern? → DENY
  ├── Does it match an explicit allow pattern? → ALLOW
  └── Uncertain? → proceed to Stage 2
  ↓
Stage 2: Model Inference (~200ms)
  ├── Feed tool name, input, context to classifier model
  ├── Model evaluates: is this tool call safe given the conversation context?
  └── Returns: ALLOW / DENY / ASK_USER

The BASH_CLASSIFIER (bashPermissions.ts, 7,200 lines) is the most sophisticated component of the permission system. It uses tree-sitter AST analysis to parse bash commands and evaluate:

  • Command intent classification (read, write, delete, network, process management)
  • Path traversal detection (is the command operating outside the project directory?)
  • Destructiveness scoring (reversible vs irreversible operations)
  • Sed pattern validation (is the sed command a safe substitution or a destructive rewrite?)
  • Denial learning (tracks previously denied patterns to auto-deny similar future requests)

Multi-CLI implication: Each CLI needs a different permission model. The Security CLI should be read-only — it can read any file but cannot modify anything. The Coder CLI can modify files but should not be able to deploy. The Orchestrator can spawn agents but should not be able to read source files directly. The permission system is already designed for this — the deny rules simply differ per CLI configuration.

Four hook types (not just shell scripts):

  1. Command hooks — Shell scripts receiving JSON on stdin. Exit code 0 = allow, 2 = block.
  2. HTTP hooks — POST requests with JSON body to external endpoints. SSRF-guarded, 5s timeout.
  3. Prompt hooks — Single-turn LLM evaluation. The hook is a yes/no decision prompt.
  4. Agent hooks — Spawns a subagent with tools for verification. The subagent decides.

Hook response protocol (JSON on stdout):

{
  "continue": true,
  "suppressOutput": false,
  "decision": "approve",
  "reason": "Command is safe",
  "systemMessage": "Optional message injected into conversation",
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "allow"
  },
  "async": false,
  "asyncTimeout": 5000
}

Post-turn hook execution order: 1. Stop hooks → can block continuation 2. TaskCompleted hooks (teammate mode) 3. TeammateIdle hooks 4. Side effects: prompt suggestion, memory extraction, auto-dream trigger, computer-use cleanup

Multi-CLI implication: Hooks are the orchestration nervous system. The Orchestrator uses TaskCompleted hooks to know when a specialist finishes. TeammateIdle hooks trigger task reassignment. PreToolUse hooks in the Security CLI block any write operations. PostToolUse hooks in the Validator CLI auto-run tests after every code change. Hooks transform the query engine from a generic loop into a domain-specific workflow engine.

Layer 7: The Memory System

Three memory modes, each with different persistence and scope:

Kairos (Assistant) Append-only daily log, nightly /dream consolidation Kairos-specific

Auto-memory extraction: - Spawns a forked subagent in background - Subagent gets: Read, Grep, Glob, WebFetch, Write (to memory dir only) - Triggers periodically (token-based: minimum 10K tokens, 5K between updates, 3 tool calls between) - Shares parent's prompt cache for efficiency

Multi-CLI implication: Each CLI has private memory (its own MEMORY.md). The Security CLI remembers vulnerability patterns across sessions. The Coder CLI remembers refactoring decisions. The shared memory bus (Chapter 14) is separate from private memory — it is typed, schema-validated shared state for cross-CLI coordination.

2.2 The 10 Extension Points

Claude Code's architecture exposes 10 distinct extension points, ordered from lowest to highest modification effort:

# Extension Point Mechanism Effort Scope
1 CLAUDE.md Natural language instructions Zero code Behavior guidance
2 Rules Path-scoped markdown with YAML frontmatter Zero code Coding standards
3 Settings JSON configuration (permissions, env, hooks) Zero code Guardrails, env
4 Hooks Shell scripts/HTTP/prompts at lifecycle events Shell scripts Automation
5 Skills Packaged workflows as /commands Markdown + config Workflows
6 Agents Specialized subagent definitions Markdown + config Delegation
7 MCP Servers External tool providers via JSON-RPC Any language New capabilities
8 Tool Registry Add/remove/modify tools in tools.ts TypeScript Tool behavior
9 Permission Engine Custom permission logic in permissions/ TypeScript Security model
10 Query Engine Modify the agentic loop in query.ts TypeScript Core behavior

For multi-CLI systems: Extension points 1–7 require zero source code changes. You create 8 different configurations of the same Claude Code binary. Extension points 8–10 require forking — used when a CLI needs tools or behaviors that cannot be achieved through configuration alone.

Practical guideline: Start with configuration (points 1–7). Only fork when you hit a wall that configuration cannot solve. The most common reasons to fork:

  • A tool needs to influence the loop's next decision (not just return a result)
  • You need permission semantics beyond allow/deny (budget-based, time-based, role-based)
  • You need to strip capabilities for compliance (no web access, no bash outside sandbox)
  • You need deep telemetry or billing gates in the execution path

2.3 Streaming Tool Execution

This is Claude Code's most distinctive architectural innovation and the one most often misunderstood.

Traditional agentic systems (LangChain, AutoGPT, most frameworks): 1. Send prompt to model 2. Wait for complete response 3. Parse tool calls from response 4. Execute tools 5. Send results back

Claude Code's StreamingToolExecutor: 1. Send prompt to model 2. As response streams in, parse tool calls during streaming 3. Start executing tools before the response is complete 4. Buffer results in tool order for deterministic output

class StreamingToolExecutor {
  addTool(block, assistantMessage)   // Called DURING streaming
  getRemainingResults()              // Yields results in tool-call order

  // Concurrency control:
  // - Tools with isConcurrencySafe=true → run in parallel
  // - Tools with isConcurrencySafe=false → exclusive access
}

Why this matters: When Claude returns three tool calls in one response — say, Read file A, Read file B, and Grep for pattern C — the streaming executor starts reading file A as soon as that tool_use block is complete in the stream, even while file B and pattern C are still being generated by the model. Read and Grep are both isConcurrencySafe=true, so all three execute in parallel.

Concurrency rules:

Tool isConcurrencySafe isReadOnly Why
Read true true Pure observation, no side effects
Grep true true Pure observation
Glob true true Pure observation
WebFetch true true External read, no local side effects
Edit false false Modifies files, could conflict
Write false false Modifies files, could conflict
Bash dynamic dynamic Depends on command analysis
Agent true false Separate process, isolated context

Max concurrency: CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY (default 10). Adjustable for machines with more or fewer cores.

Multi-CLI implication: The Coder CLI benefits most from streaming execution — it often generates multiple file reads followed by edits. RCode's tokio::task::JoinSet provides true OS-level parallelism for this pattern, executing tools across CPU cores rather than on JavaScript's single-threaded event loop. This is one of the primary performance advantages of the Rust rewrite.

2.4 The Complete Request Trace

Let's trace a single request through all 9 layers. You type:

Fix the off-by-one error in src/parser.py line 42

Layer 1 — Entry: The Ink TUI captures your input, wraps it as a user message.

Layer 3 — Context Assembly (~50ms): - CLAUDE.md loaded (42 lines → ~150 tokens) - Python rule matched (src/parser.py → **/*.py) → python.md loaded (~200 tokens) - Git context: branch fix/parser-bug, 1 modified file (~100 tokens) - Tool schemas: 52 tools → ~10,000 tokens - Total system prompt: ~12,000 tokens

Layer 2 — Query Engine, Iteration 1:

API call (~700ms first token). Claude returns: - Thinking block: "Let me read the file to understand the bug around line 42" - Tool_use block: Read(file_path="/src/parser.py")

Layer 5 — Permission: Read(*) is in the allow list → allowed (~1ms)

Layer 6 — Hooks: No PreToolUse hooks configured for Read → skip

Layer 4 — Tool Execution: File read (~10ms). 200 lines → ~800 tokens.

Layer 6 — Hooks: No PostToolUse hooks for Read → skip

Layer 8 — State: Update token counter (+800 tokens for tool result)

Layer 2 — Query Engine, Iteration 2:

API call (~700ms). Claude returns: - Thinking: "Found the bug — the loop iterates one element too far" - Text: "I found the off-by-one error..." - Tool_use: Edit(file_path="/src/parser.py", old_string="for i in range(len(items))", new_string="for i in range(len(items) - 1)")

Layer 5 — Permission: Edit uses default mode → auto-allowed in this config

Layer 6 — PreToolUse hooks: None for Edit → skip

Layer 4 — Tool Execution: String replacement (~5ms). Success.

Layer 6 — PostToolUse hooks: auto-lint-python.sh fires (~200ms): - Reads JSON input from stdin - Detects .py extension - Runs ruff check --fix src/parser.py - Runs ruff format src/parser.py - Exit code 0

Layer 2 — Query Engine, Iteration 3:

API call. Claude returns: - Tool_use: Bash(command="python -m pytest tests/test_parser.py -x")

Layer 5 — Permission: Bash(python -m pytest *) is in the allow list → allowed

Layer 6 — PreToolUse hooks: block-destructive.sh fires (~50ms): - Reads JSON input - Checks python -m pytest tests/test_parser.py -x against destructive patterns - No match → exit code 0

Layer 4 — Tool Execution: Bash (~3s). Output: "5 passed, 0 failed"

Layer 2 — Query Engine, Iteration 4:

API call. Claude returns: - Text only: "Fixed the off-by-one error in src/parser.py:42. The loop was iterating one element too far. All 5 tests pass." - stop_reason: end_turn

Layer 2 — Post-loop: Stop hooks → memory extraction check → no auto-dream (cooldown not elapsed)

Layer 9 — UI: Final text rendered to terminal.

Total: 4 iterations, 3 tool calls, ~8 seconds, ~$0.05 (Sonnet)

The Rust advantage for multi-CLI: True OS-level parallelism means the Orchestrator can spawn and monitor 8 specialist CLIs without the event-loop bottleneck. Each specialist runs in its own process with its own Tokio runtime. Communication happens through typed channels or Unix Domain Sockets. The 6MB binary means deployment is trivial — no Node.js installation, no npm dependencies, just a single executable per specialist.

The critical insight from this chapter: Claude Code is not a monolith — it is a framework with pluggable layers. Building a multi-CLI system does not require building 8 separate applications. It requires configuring the same engine 8 different ways.

2.8 The API Client and Retry Logic

The API client (services/api/claude.ts, approximately 800 lines) is the bridge between the query engine and the Claude API. Its design decisions directly affect multi-CLI reliability.

Connection Management

Claude Code maintains a persistent HTTP/2 connection to the API endpoint. The base URL defaults to https://api.anthropic.com but is overridable via:

Variable Purpose
ANTHROPIC_BASE_URL Direct API endpoint override
CLAUDE_CODE_USE_BEDROCK Route through AWS Bedrock
CLAUDE_CODE_USE_VERTEX Route through Google Vertex AI
CLAUDE_CODE_USE_FOUNDRY Route through Anthropic Foundry

Each provider has different beta header support. Bedrock uses extraBodyParams instead of headers for some capabilities. Vertex's countTokens endpoint only accepts claude-code-20250219 and interleaved-thinking headers. These provider differences matter when deploying multi-CLI systems across cloud environments.

Retry Strategy

The API client implements exponential backoff with jitter for transient failures:

Attempt 1: immediate
Attempt 2: wait 1s + random(0, 0.5s)
Attempt 3: wait 2s + random(0, 1.0s)
Attempt 4: wait 4s + random(0, 2.0s)
(max 4 retries)

Retryable status codes: 429 (rate limited), 500, 502, 503, 529 (overloaded). The Retry-After header is respected when present — if the API says "wait 30 seconds," the client waits 30 seconds regardless of the exponential backoff schedule.

Non-retryable errors (fail immediately): 400 (bad request — schema violation), 401 (invalid API key), 403 (permission denied), 404.

Multi-CLI implication: Each specialist CLI has its own API connection. Rate limits are per-API-key, not per-connection. A multi-CLI system with 8 specialists hitting the same API key can exhaust rate limits 8x faster. Solutions: - Use separate API keys per CLI (if your plan supports it) - Use task-budgets to limit token consumption per specialist - Route through different providers: Coder through Bedrock, Research through direct API, Planner through Vertex - Implement a token budget manager in the Orchestrator that gates specialist requests

Session Headers

Every API request includes: - X-Claude-Code-Session-Id — Unique session identifier for proxy aggregation and debugging - Anthropic-Organization-Id — Organization routing (if applicable) - Anthropic-Beta — Comma-separated list of enabled beta features

The session ID is critical for multi-CLI debugging: when something goes wrong, you can correlate API logs across all specialist CLIs by their session IDs. The Orchestrator should log all specialist session IDs for traceability.

Token Counting

Claude Code tracks token usage with per-turn granularity:

interface TurnCost {
  inputTokens: number;
  outputTokens: number;
  cacheReadTokens: number;
  cacheCreationTokens: number;
  thinkingTokens: number;
  model: string;
  costUSD: number;
}

The costUSD field is computed using the model's known pricing. The running total is displayed by /stats and persisted to the session store. For multi-CLI systems, the Orchestrator should aggregate costs across all specialists to provide a single cost dashboard for each user request.

2.9 The SSE Streaming Protocol

Understanding the wire protocol between Claude Code and the API is necessary for diagnosing performance issues and building custom CLIs that communicate with the same API.

The Claude API streams responses as Server-Sent Events (SSE). Each event contains a JSON payload with a specific type:

event: message_start
data: {"type":"message_start","message":{"id":"msg_01...","model":"claude-opus-4-6",...}}

event: content_block_start
data: {"type":"content_block_start","index":0,"content_block":{"type":"thinking","thinking":""}}

event: content_block_delta
data: {"type":"content_block_delta","index":0,"delta":{"type":"thinking_delta","thinking":"Let me analyze..."}}

event: content_block_stop
data: {"type":"content_block_stop","index":0}

event: content_block_start
data: {"type":"content_block_start","index":1,"content_block":{"type":"text","text":""}}

event: content_block_delta
data: {"type":"content_block_delta","index":1,"delta":{"type":"text_delta","text":"I'll fix the bug..."}}

event: content_block_start
data: {"type":"content_block_start","index":2,"content_block":{"type":"tool_use","id":"toolu_01...","name":"Read","input":{}}}

event: content_block_delta
data: {"type":"content_block_delta","index":2,"delta":{"type":"input_json_delta","partial_json":"{\"file_path\":\"/src/main.py\"}"}}

event: content_block_stop
data: {"type":"content_block_stop","index":2}

event: message_delta
data: {"type":"message_delta","delta":{"stop_reason":"tool_use"},"usage":{"output_tokens":127}}

event: message_stop
data: {"type":"message_stop"}

A single response can contain multiple content blocks of different types:

Block Type Purpose Processing
thinking Extended reasoning (chain of thought) Displayed if /thinking enabled, otherwise buffered silently
text Human-readable response Displayed character-by-character as deltas arrive
tool_use Tool call request Accumulated until content_block_stop, then executed

The interleaved-thinking-2025-05-14 beta header enables thinking blocks to appear between other content blocks, not just at the start. This allows the model to reason between tool calls:

thinking → "Let me read the file..."
tool_use → Read(src/auth.py)
thinking → "I see the bug on line 47, the comparison uses == instead of is..."
tool_use → Edit(src/auth.py, ...)
text → "Fixed the identity comparison bug."

Stream watchdog: The CLAUDE_ENABLE_STREAM_WATCHDOG environment variable enables a watchdog that monitors for stalled streams. If no SSE event arrives within a timeout window, the watchdog terminates the connection and triggers a retry. This is critical for production multi-CLI systems where a stalled API connection should not block the entire pipeline.

Beta headers sent with every request:

Claude Code sends up to 17 beta headers with each API request, enabling capabilities that are not yet part of the stable API:

Header What It Enables
claude-code-20250219 Base Claude Code capability
interleaved-thinking-2025-05-14 Thinking blocks between content blocks
context-1m-2025-08-07 1M token context window (opt-in)
structured-outputs-2025-12-15 JSON schema enforcement on output
web-search-2025-03-05 Built-in web search capability
effort-2025-11-24 Effort levels (low/medium/high/max)
task-budgets-2026-03-13 Per-task token budgets
fast-mode-2026-02-01 Reduced-latency generation
token-efficient-tools-2026-03-28 Compressed tool schemas
advisor-tool-2026-03-01 Separate advisor model for review

Multi-CLI implication: Different CLIs can use different beta headers. The Research CLI benefits from context-1m (massive context for RAG). The Validator CLI uses fast-mode (quick test result processing). The Planner CLI uses effort set to max for deep architectural reasoning. The Orchestrator uses task-budgets to control per-specialist token spend.

Multi-CLI Cost Optimization Strategy

The cost advantage of multi-CLI is counterintuitive. Running 8 CLIs sounds 8x more expensive. In practice, it is often cheaper because:

  1. Smaller context per CLI. A specialist CLI carries 20K tokens of domain context, not 150K tokens of everything. Smaller context = cheaper per turn.

  2. Model mixing. The Orchestrator runs on Opus ($15/$75) because decision quality matters. The Validator runs on Haiku ($0.80/$4) because test results are deterministic. The Coder runs on Sonnet ($3/$15) as a balanced choice. A typical task might use:

CLI Model Turns Cost
Orchestrator Opus 3 $0.45
Research Sonnet 4 $0.20
Planner Opus 2 $0.30
Coder Sonnet 8 $0.48
Security Sonnet 3 $0.15
Validator Haiku 5 $0.05
QA Sonnet 3 $0.12
Doc Haiku 4 $0.04
Total 32 $1.79

The same task on a single Opus CLI with a saturated context window: ~15 turns at ~$1.50/turn = $22.50. The multi-CLI approach is 12x cheaper because it uses the right model for each subtask and keeps contexts lean.

  1. Prompt caching. Each specialist CLI's system prompt is stable across turns (same tools, same CLAUDE.md, same rules). High cache hit rates mean the 7,000–15,000 token system prompt is paid at the cache read rate (1/10th the input rate for Opus), not the full input rate.

  2. task-budgets beta header. The task-budgets-2026-03-13 header enables per-spawned-task token budgets. The Orchestrator can allocate 50K tokens to the Validator and 200K tokens to the Coder, preventing runaway costs from any single specialist.

2.10 Architecture Principles for What Follows

Seven principles extracted from this architecture govern every design decision in the remaining 25 chapters:

Principle 1: The loop is universal; the configuration is specific. The query engine is the same for every CLI. Specialization happens through tools, permissions, hooks, memory, and context — never by modifying the loop itself.

Principle 2: Tools define capability boundaries. A CLI that does not have the Edit tool cannot modify files. A CLI that does not have the Bash tool cannot execute commands. Tool registration is the primary mechanism for enforcing what a specialist can and cannot do.

Principle 3: Permissions enforce trust boundaries. Even if a CLI has the Bash tool, deny rules can restrict it to specific command patterns. Permission layers (managed → user → project → local) allow team-level and individual-level control.

Principle 4: Hooks are the nervous system. PreToolUse hooks are guardrails. PostToolUse hooks are automation. TaskCompleted hooks are orchestration signals. FileChanged hooks are reactive triggers. The hook system transforms a generic loop into a domain-specific workflow engine.

Principle 5: Context is expensive and finite. Every token in the system prompt is paid on every API call. Every tool result grows the conversation history. Lean CLAUDE.md files, targeted searches, and prompt caching are not optimizations — they are requirements.

Principle 6: Streaming is not optional. The StreamingToolExecutor's ability to begin tool execution during response generation is a fundamental performance advantage. Any custom CLI implementation must preserve this capability.

Principle 7: The number of CLIs is unlimited. Eight is the teaching example. The architecture supports 2 CLIs or 200. A Doctor CLI, a Lawyer CLI, a Therapy CLI, a Finance CLI — any domain that benefits from a specialized system prompt, curated tool set, and dedicated context window is a candidate for its own CLI. The patterns in this book apply equally to all of them.

2.12 The Skill and Plugin Loading Architecture

Skills and plugins are the highest-level extension points that do not require source code modification. Understanding how they load is essential for building CLI-specific workflows.

Skill Discovery and Loading

Skills are discovered from multiple locations, merged in priority order:

1. Managed settings skills (enterprise-deployed, highest priority)
2. CLI flag skills (--skills argument, session-only)
3. Project skills (.claude/skills/*/SKILL.md)
4. User skills (~/.claude/skills/*/SKILL.md)
5. Plugin-provided skills (.claude-plugin/marketplace.json)

Each skill's SKILL.md file has YAML frontmatter controlling its behavior:

---
name: security-audit
description: "OWASP/CWE security analysis. Use when user says 'audit', 'security', 'vulnerabilities'."
argument-hint: "[target-path]"
user-invocable: true
allowed-tools: Read, Grep, Glob, Bash, Agent
model: opus
effort: high
context: fork
agent: general-purpose
paths:
  - "src/auth/**"
  - "src/middleware/**"
---

Key frontmatter fields for multi-CLI:

Field Effect on Skill Behavior
allowed-tools Restricts which tools the skill can invoke — essential for specialist CLIs
model Override the CLI's default model for this skill (e.g., Opus for security, Haiku for linting)
effort Set reasoning depth (low for quick tasks, max for architecture)
context: fork Run in a subagent with isolated context — prevents skill from polluting main conversation
paths Auto-activate when files matching these globs are in scope
disable-model-invocation: true Only the user can trigger this skill (prevents model from auto-loading it)

Dynamic context injection: Skills support !`command` syntax that executes shell commands before the skill content reaches the model. The /pr-summary skill, for example, runs gh pr diff $ARGUMENTS during loading, so the model starts with the full diff already in context — no additional tool calls needed.

Plugin Marketplace

Claude Code supports a plugin ecosystem via .claude-plugin/marketplace.json. Plugins can provide:

  • Additional tools (via MCP server registration)
  • Additional skills (SKILL.md files in the plugin directory)
  • Additional hooks (wired through the plugin's configuration)
  • Additional agent definitions

Plugins are hot-reloadable — you can install or update a plugin without restarting the session. The plugin discovery service runs in the background during session setup.

Multi-CLI implication: Each specialist CLI can load different skills and plugins. The Security CLI loads security-audit and threat-model skills. The Coder CLI loads refactor, fix-issue, and scaffold skills. The Documentation CLI loads doc-writer and changelog skills. Skills are the mechanism for encoding domain-specific workflows without modifying source code.

The Complete Loading Sequence

At session start, Claude Code executes this loading sequence (simplified):

1. Parse CLI arguments (--model, --skills, --agents, etc.)
2. Load settings cascade (managed → user → project → local)
3. Discover and load plugins (background)
4. Discover and load skills (from all 5 sources)
5. Discover and load agent definitions (from .claude/agents/)
6. Discover and load rules (from .claude/rules/)
7. Register hooks (from settings.json + plugins + agent frontmatter)
8. Initialize MCP connections (from settings + plugins)
9. Build tool pool (built-in + MCP + deferred)
10. Assemble initial system prompt
11. Fire SessionStart hooks
12. Enter the query engine loop

Steps 3–8 run in parallel where possible. The total boot time on a well-configured project is approximately 300ms for TypeScript, 10ms for RCode.

For multi-CLI systems, the Orchestrator typically completes this sequence first, then spawns specialists. Each specialist runs its own loading sequence independently, which is why startup parallelism matters — 8 specialists loading sequentially would add 2.4 seconds. Loading in parallel brings it to ~300ms total.

This is the keystone. Every chapter that follows builds on this decomposition.
