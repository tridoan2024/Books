Chapter 8: The Planner CLI — The Architect

ULTRAPLAN, Task Decomposition, Dependency Graphs, ADR Generation, and Thinking Before Building

The most expensive mistake in software engineering is not a bug — it is building the wrong thing. A bug costs hours to fix. A misarchitected system costs weeks to rearchitect, months of technical debt, and the morale cost of throwing away work that someone believed was finished. The Planner CLI exists to prevent this class of mistake. It thinks before the Coder builds. It decomposes before the Orchestrator dispatches. It documents decisions before they become implicit assumptions buried in code.

In a multi-CLI system without a Planner, the Orchestrator performs planning inline — decomposing prompts into tasks as part of its intent analysis engine (Chapter 6). This works for simple tasks. For complex tasks — "Build a multi-tenant SaaS with RBAC, webhooks, and audit logging" — inline planning produces shallow decompositions that miss critical dependencies and architectural decisions. The Orchestrator is optimized for speed: fast intent classification, fast dispatch. The Planner is optimized for depth: thorough analysis, dependency mapping, risk identification, and design documentation.

This chapter builds the Planner CLI from four components: the ULTRAPLAN integration (leveraging Claude Code's most powerful reasoning mode), the task decomposition engine (producing the dependency graphs that the Orchestrator executes), the ADR generation system (documenting architectural decisions as they are made), and the diagram generation pipeline (visualizing architecture for human review). By the end, you will have a Planner CLI that turns vague requirements into precise, documented, executable plans.

8.1 ULTRAPLAN: Remote Planning with Maximum Reasoning

Claude Code's ULTRAPLAN feature flag (from constants/features.ts) enables a powerful pattern: offloading complex planning to a remote Opus instance with extended thinking enabled. When a planning task is too complex for inline reasoning — requiring analysis of multiple subsystems, evaluation of architectural tradeoffs, or decomposition of a 50-task project — ULTRAPLAN delegates the planning itself to a high-reasoning model.

How ULTRAPLAN Works

User prompt: "Build a multi-tenant SaaS with RBAC, webhooks, and audit logging"

┌─────────────────────────────────────────┐
│ Orchestrator: Intent = Build (complex)   │

└────────────────┬────────────────────────┘

┌─────────────────────────────────────────┐
│ Planner CLI: Activates ULTRAPLAN         │
│ Remote Opus instance with extended       │
│ thinking (budget_tokens: 16384)          │
│                                          │
│ Thinks deeply about:                     │
│ - Architecture patterns for multi-tenant │
│ - RBAC model (roles, permissions, scopes)│

│ - What must be sequential vs parallel    │
│ - What can go wrong                      │
└────────────────┬────────────────────────┘

┌─────────────────────────────────────────┐
│ Output: Detailed task DAG + ADR + risks  │
│ 25-40 tasks with dependency chains       │
│ 3-5 architectural decisions documented   │
│ Risk register with mitigations           │
└─────────────────────────────────────────┘

The key insight is that planning and execution have different optimal reasoning depths. Execution benefits from speed — the Coder should write code quickly, not deliberate over every line. Planning benefits from depth — the Planner should consider every tradeoff, because a missed dependency discovered during execution costs 10x more to fix than one caught during planning.

This asymmetry is why the Planner CLI is a separate specialist rather than a phase of the Orchestrator. As a separate CLI, it gets its own context window dedicated entirely to planning. It can read architecture docs, review existing patterns, consider alternatives, and produce a thorough decomposition without competing for context space with dispatch logistics and specialist status tracking. The Orchestrator's context stays clean for coordination; the Planner's context fills with analysis that produces better plans.

When to Use ULTRAPLAN vs. Inline Planning

Not every task needs ULTRAPLAN. The Orchestrator's inline intent analysis (Chapter 6) handles simple prompts efficiently. The decision tree:

  • 1-3 tasks, clear dependencies: Inline planning in the Orchestrator. No Planner CLI needed. Example: "Add a health check endpoint."
  • 4-10 tasks, straightforward decomposition: Standard Planner analysis without ULTRAPLAN. Example: "Add user registration with email verification."
  • 10+ tasks, architectural decisions required: ULTRAPLAN with extended thinking. Example: "Build a multi-tenant SaaS with RBAC, webhooks, and audit logging."
  • Greenfield project or major refactor: ULTRAPLAN with maximum thinking budget (16K tokens) and multiple planning iterations. Example: "Design the architecture for a new microservices platform."

The PlanningContext is crucial. The Planner CLI does not plan in a vacuum — it receives context from the Orchestrator about the existing system: what technologies are already in use, what architectural patterns are established, what constraints exist (budget, timeline, team size). Without this context, the Planner might recommend a microservices architecture for a project that is already a monolith, or suggest PostgreSQL for a project already committed to MongoDB.

pub struct PlanningContext {
    pub architecture_summary: String,     // Existing system overview
    pub tech_stack_description: String,   // Languages, frameworks, databases in use
    pub constraints_description: String,  // Budget, timeline, compliance requirements
    pub existing_patterns: Vec<Pattern>,  // Code patterns from the existing codebase
    pub related_adrs: Vec<ADR>,           // Previous architectural decisions
    pub team_capabilities: Vec<String>,   // What specialist CLIs are available
}

pub struct Pattern {
    pub name: String,
    pub description: String,
    pub example_file: String,
    pub frequency: u32,  // How often this pattern appears in the codebase
}

The Planner CLI populates this context by reading key files at the start of every planning session: package.json or Cargo.toml for dependencies, existing CLAUDE.md files for conventions, previous ADRs for established decisions, and a sample of source files for coding patterns. This codebase scan takes 10-20 seconds but prevents the most common planning mistake: recommending something incompatible with the existing system.

EnterPlanMode and ExitPlanMode

Claude Code provides built-in plan mode through the EnterPlanMode and ExitPlanMode tools. These tools create a structured planning workflow:

  1. EnterPlanMode — Transitions the agent into planning mode. In this mode, the agent explores the codebase (Read, Grep, Glob) but does not modify anything (no Edit, Write, or Bash that changes files). This restriction ensures that planning is pure analysis, not premature implementation.

  2. Plan exploration — The agent reads relevant files, understands existing patterns, identifies extension points, and considers tradeoffs.

  3. Plan writing — The agent writes its plan to a designated plan file (.claude/plans/), documenting the approach, the files to be modified, and the expected outcome.

  4. ExitPlanMode — Presents the plan to the human for approval. The human can approve, modify, or reject the plan before any code changes occur.

For the Planner CLI, plan mode is the default operating mode — it never exits plan mode to write code. Its output is always a plan, never an implementation. The Planner writes the plan, the human approves it, and the Orchestrator dispatches the approved plan's tasks to specialist CLIs for execution.

This human-in-the-loop approval step is not a formality. It is where the human catches assumptions the Planner made that do not match reality: "You assumed we want REST, but this project uses GraphQL." "You estimated 5 tasks for the auth module, but we already have half of it built." "You missed that we need HIPAA compliance for the data storage." No amount of codebase analysis catches requirements that exist only in the human's head. The approval step surfaces them before any specialist starts working.

The approval can also modify the plan. The human might accept the task graph but reject an ADR: "I disagree with using Redis — we want to keep the infrastructure simple, use in-memory caching for now." The Planner regenerates the affected tasks, and the Orchestrator dispatches the modified plan.

8.2 Task Decomposition with Dependency Graphs

The Planner CLI's primary output is a task graph — the same TaskDAG structure introduced in Chapter 6, but produced through careful analysis rather than inline heuristics.

The Decomposition Process

Decomposition follows a top-down, recursive strategy:

  1. Identify top-level components — What major subsystems does this requirement need? (Auth, API routes, database schema, deployment config)
  2. For each component, identify tasks — What concrete work items produce this component? (Schema migration, model definition, route handler, middleware, tests)
  3. Map dependencies — Which tasks must complete before others can start? (Schema before models, models before routes, routes before tests)
  4. Identify parallelism — Which tasks are independent and can run in parallel? (Frontend and backend work, different API endpoints, independent microservices)
  5. Assign specialists — Which CLI is best suited for each task? (Coder for implementation, Security for audit, Validator for testing)
  6. Estimate effort — How complex is each task? (Simple: <2 min, Moderate: 2-5 min, Complex: 5-10 min)

A Worked Example: Multi-Tenant SaaS Planning

Consider the requirement: "Build a multi-tenant SaaS application with RBAC, webhooks, and audit logging."

A naive decomposition produces 5 tasks: "build API," "add auth," "add webhooks," "add audit logging," "write tests." This is useless — each task is vague, no dependencies are specified, and the specialist CLIs will spend most of their time figuring out what to do rather than doing it.

The Planner CLI's ULTRAPLAN decomposition produces something like this:

Component: Tenant Isolation
  T1: Schema migration — tenants table, tenant_id FK on all tables [coder, depends: none]
  T2: Tenant middleware — extract tenant from JWT, scope all queries [coder, depends: T1]
  T3: Tenant isolation test — verify cross-tenant data leak is impossible [qa, depends: T2]

Component: RBAC
  T4: Permission model — roles table, permissions table, role_permissions join [coder, depends: T1]
  T5: RBAC middleware — check permission before route handler [coder, depends: T4]
  T6: RBAC admin API — CRUD for roles and permission assignments [coder, depends: T4, T5]

Component: Webhooks
  T7: Webhook registration API — CRUD for webhook URLs per tenant [coder, depends: T1]
  T8: Event emitter — async webhook delivery with retry [coder, depends: T7]
  T9: Webhook signature — HMAC-SHA256 signing for payload verification [coder, depends: T7]

Component: Audit Logging
  T10: Audit schema — event type, actor, target, before/after JSON [coder, depends: T1]
  T11: Audit middleware — auto-log all write operations [coder, depends: T10]
  T12: Audit query API — search and filter audit log by tenant [coder, depends: T10]

Component: Security
  T13: Security audit — tenant isolation, RBAC bypass, webhook SSRF [security, depends: T2, T5, T8]
  T14: Fix security findings [coder, depends: T13]

Component: Validation
  T15: Integration test suite [coder, depends: T2, T5, T8, T11]
  T16: Run full validation pipeline [validator, depends: T15]

ADRs:
  ADR-0005: Tenant isolation strategy (row-level security vs application-level filtering)
  ADR-0006: Webhook delivery guarantee (at-least-once with idempotency keys)
  ADR-0007: Audit log storage (same database vs separate time-series DB)

Critical Path: T1 → T4 → T5 → T15 → T16 (5 sequential steps)
Parallel Waves: Wave 1 [T1], Wave 2 [T2,T4,T7,T10], Wave 3 [T3,T5,T6,T8,T9,T11,T12],
                Wave 4 [T13,T15], Wave 5 [T14,T16]

Notice how the decomposition identifies that T2, T4, T7, and T10 all depend only on T1 (the tenant schema) — they can run in parallel in Wave 2. This parallelism reduces execution time from 16 sequential steps to 5 wave cycles.

// clis/planner/src/decomposition.rs
pub struct DecompositionEngine;

impl DecompositionEngine {
    pub fn decompose(&self, requirement: &str, context: &PlanningContext) -> TaskPlan {
        // Phase 1: Component identification
        let components = self.identify_components(requirement);

        // Phase 2: Task generation per component
        let mut tasks = Vec::new();
        for component in &components {
            let component_tasks = self.generate_tasks(component, context);
            tasks.extend(component_tasks);
        }

        // Phase 3: Dependency mapping
        let dependencies = self.infer_dependencies(&tasks);

        // Phase 4: Specialist assignment
        for task in &mut tasks {
            task.assigned_cli = self.assign_specialist(task, context);
        }

        // Phase 5: Effort estimation
        for task in &mut tasks {
            task.estimated_effort = self.estimate_effort(task);
        }

        // Phase 6: Critical path analysis
        let critical_path = self.find_critical_path(&tasks, &dependencies);

        TaskPlan {
            tasks,
            dependencies,
            critical_path,
            estimated_total_time: self.estimate_total_time(&tasks, &dependencies),
            parallel_waves: self.compute_waves(&tasks, &dependencies),
        }
    }

    fn find_critical_path(&self, tasks: &[PlanTask], deps: &[Dependency]) -> Vec<String> {
        // Longest path through the DAG (determines minimum total execution time)
        // Uses topological sort + dynamic programming
        let sorted = topological_sort(tasks, deps);
        let mut distances: HashMap<String, (u64, Vec<String>)> = HashMap::new();

        for task_id in sorted {
            let task = tasks.iter().find(|t| t.id == task_id).unwrap();
            let task_duration = task.estimated_effort.as_millis() as u64;

            let (max_predecessor_dist, max_path) = deps.iter()
                .filter(|d| d.dependent == task_id)
                .filter_map(|d| distances.get(&d.dependency))
                .max_by_key(|(dist, _)| *dist)
                .map(|(dist, path)| (*dist, path.clone()))
                .unwrap_or((0, Vec::new()));

            let mut path = max_path;
            path.push(task_id.clone());
            distances.insert(task_id, (max_predecessor_dist + task_duration, path));
        }

        distances.into_values()
            .max_by_key(|(dist, _)| *dist)
            .map(|(_, path)| path)
            .unwrap_or_default()
    }
}

Critical Path Analysis

The critical path is the longest chain of sequential dependencies in the task graph. It determines the minimum possible execution time — even with unlimited parallelism, you cannot finish faster than the critical path. Knowing the critical path helps the Planner make two important decisions:

  1. Where to invest optimization effort — Reducing a task on the critical path directly reduces total execution time. Reducing an off-critical-path task has no effect (it just finishes earlier, but was not the bottleneck).

  2. Where to split tasks — A complex task on the critical path might be decomposable into two phases: an essential first phase that unblocks downstream work, and a refinement phase that can run in parallel. For example, "Implement authentication" might split into "Implement basic JWT middleware" (unblocks API routes) and "Add token refresh and revocation" (runs in parallel with route development).

Risk Identification

During decomposition, the Planner CLI identifies risks — things that could go wrong during execution:

pub struct Risk {
    pub description: String,
    pub likelihood: RiskLevel,  // Low, Medium, High
    pub impact: RiskLevel,
    pub mitigation: String,
    pub affected_tasks: Vec<String>,
}

Common risks the Planner flags: - External dependency risk: "The webhook delivery system depends on a queue service. If the queue is unavailable, webhooks fail silently." Mitigation: add dead-letter queue and alerting. - Performance risk: "Audit logging on every write adds ~5ms latency per request. For high-throughput endpoints, this may be unacceptable." Mitigation: async audit logging with batch writes. - Security risk: "Multi-tenant row-level security must be verified with adversarial testing — a single missed query could leak data between tenants." Mitigation: dedicated QA test suite for tenant isolation.

Risks feed into the Orchestrator's dispatch decisions. A high-impact risk on the critical path may cause the Orchestrator to schedule a dedicated review task immediately after the risky implementation, rather than waiting for the standard security audit at the end.

8.3 ADR Generation

Architectural Decision Records (ADRs) document the decisions made during planning — what was decided, why, what alternatives were considered, and what tradeoffs were accepted. Without ADRs, architectural decisions become implicit knowledge locked in one person's head (or one session's context). When someone asks "why did we choose PostgreSQL over MongoDB?" three months later, the ADR provides the answer.

ADR Template

The Planner CLI generates ADRs following the standard format established by Michael Nygard, adapted for multi-CLI systems:

// clis/planner/src/adr.rs
pub struct ADR {
    pub number: u32,
    pub title: String,
    pub status: ADRStatus,
    pub context: String,        // What is the issue that we're facing?
    pub decision: String,       // What is the change we're proposing/have agreed to?
    pub alternatives: Vec<Alternative>,
    pub consequences: String,   // What are the positive and negative consequences?
    pub date: NaiveDate,
}

pub struct Alternative {
    pub name: String,
    pub pros: Vec<String>,
    pub cons: Vec<String>,
    pub rejected_reason: String,
}

pub enum ADRStatus {
    Proposed,
    Accepted,
    Deprecated,
    Superseded { by: u32 },
}

impl ADR {
    pub fn to_markdown(&self) -> String {
        format!(r#"# ADR-{:04}: {}

## Status
{}

## Context
{}

## Decision
{}

## Alternatives Considered
{}

## Consequences
{}

---
Date: {}
"#,
            self.number,
            self.title,
            self.status.display(),
            self.context,
            self.decision,
            self.alternatives.iter()
                .map(|a| format!("### {}\n**Pros:** {}\n**Cons:** {}\n**Rejected because:** {}",
                    a.name, a.pros.join(", "), a.cons.join(", "), a.rejected_reason))
                .collect::<Vec<_>>()
                .join("\n\n"),
            self.consequences,
            self.date,
        )
    }
}

When ADRs Get Generated

The Planner CLI generates an ADR for every decision that: - Affects more than one component (e.g., database choice affects schema, queries, migrations, deployment) - Is difficult to reverse later (e.g., choice of ORM, authentication strategy, API versioning scheme) - Has multiple viable alternatives (e.g., REST vs GraphQL, PostgreSQL vs MongoDB) - Introduces a new dependency (e.g., adding Redis for caching, adding RabbitMQ for queuing)

The Planner does not generate ADRs for implementation details that are easily changed (variable names, code organization within a single file, choice of logging format). The ADR represents a commitment — a decision that other work will build upon and that becomes expensive to reverse.

Linking ADRs to the Knowledge Graph

When the Research CLI (Chapter 7) builds a knowledge graph about the project's technology choices, ADRs feed directly into that graph. Each ADR creates a Decision node with edges to the technologies it references. When a future planning session considers adding a new technology, the Planner queries the knowledge graph: "Are there any existing ADRs that constrain this choice?" If ADR-0001 chose PostgreSQL and ADR-0003 chose Prisma ORM, a new planning task that requires database access should use Prisma with PostgreSQL — not introduce a competing ORM or database.

This cross-referencing prevents a common multi-CLI failure: decisions made in session 1 being contradicted by decisions in session 5 because no one remembered the original rationale.

ADR Storage and Referencing

ADRs are stored in docs/decisions/ and numbered sequentially:

docs/decisions/
├── 0001-use-postgresql-over-mongodb.md
├── 0002-jwt-rs256-for-authentication.md
├── 0003-prisma-orm-for-database-access.md
└── 0004-redis-for-session-and-rate-limiting.md

When the Planner CLI generates a task plan, it references relevant ADRs in the task descriptions. When the Coder CLI receives a task that says "Implement the user model using Prisma (see ADR-0003)," it can read the ADR to understand why Prisma was chosen and what constraints apply.

ADR Lifecycle in Multi-CLI Systems

ADRs in a multi-CLI system have a richer lifecycle than traditional ADRs. A traditional ADR is written once by a human architect and occasionally superseded. In a multi-CLI system, ADRs are generated by the Planner, approved by the human, referenced by specialists during execution, and potentially challenged by the Security CLI or QA CLI during their audits.

Consider this scenario: the Planner creates ADR-0002 recommending JWT with RS256 for authentication. The Coder implements it. The Security CLI audits the implementation and discovers that the RS256 key rotation strategy is insufficient — it recommends adding key rotation via JWKS endpoint. This finding does not invalidate ADR-0002 (the choice of RS256 was correct), but it adds a constraint. The Planner appends a "Subsequent Constraints" section to the ADR:

## Subsequent Constraints
- 2026-04-02: Security CLI audit (session 47) identified that RS256 requires 
  JWKS endpoint for key rotation. Implementation must include /.well-known/jwks.json
  endpoint and key rotation scheduler. See T14-security-findings.json.

This creates a living record of how decisions evolve during execution — valuable for future planning sessions and for understanding why the system looks the way it does.

8.4 Diagram Generation

Architecture diagrams serve two audiences: the human who reviews the plan, and the specialist CLIs that execute it. The Planner CLI generates Mermaid diagrams embedded in the plan output.

Component Diagrams

// clis/planner/src/diagrams.rs
pub struct DiagramGenerator;

impl DiagramGenerator {
    pub fn component_diagram(&self, plan: &TaskPlan) -> String {
        let mut mermaid = String::from("graph TB\n");

        // Group tasks by component
        let components = plan.group_by_component();

        for (name, tasks) in &components {
            mermaid.push_str(&format!("    subgraph {}\n", name));
            for task in tasks {
                let shape = match task.assigned_cli.as_str() {
                    "coder" => format!("[{}]", task.short_name()),
                    "security" => format!("{{{{{}}}}}", task.short_name()),
                    "validator" => format!("({})", task.short_name()),
                    _ => format!("[{}]", task.short_name()),
                };
                mermaid.push_str(&format!("        {}{}\n", task.id, shape));
            }
            mermaid.push_str("    end\n");
        }

        // Add dependency arrows
        for dep in &plan.dependencies {
            mermaid.push_str(&format!("    {} --> {}\n", dep.dependency, dep.dependent));
        }

        mermaid
    }

    pub fn sequence_diagram(&self, plan: &TaskPlan) -> String {
        let mut mermaid = String::from("sequenceDiagram\n");
        mermaid.push_str("    participant H as Human\n");
        mermaid.push_str("    participant O as Orchestrator\n");

        // Add specialist participants
        let specialists: HashSet<&str> = plan.tasks.iter()
            .map(|t| t.assigned_cli.as_str())
            .collect();
        for spec in &specialists {
            mermaid.push_str(&format!("    participant {} as {} CLI\n", 
                spec.chars().next().unwrap().to_uppercase().collect::<String>(), 
                spec));
        }

        // Add interactions per wave
        for (wave_num, wave) in plan.parallel_waves.iter().enumerate() {
            mermaid.push_str(&format!("    Note over O: Wave {}\n", wave_num + 1));
            for task in wave {
                let spec_initial = task.assigned_cli.chars().next().unwrap()
                    .to_uppercase().collect::<String>();
                mermaid.push_str(&format!("    O->>{}:{}\n", spec_initial, task.short_name()));
            }
            for task in wave {
                let spec_initial = task.assigned_cli.chars().next().unwrap()
                    .to_uppercase().collect::<String>();
                mermaid.push_str(&format!("    {}-->>O: Complete\n", spec_initial));
            }
        }

        mermaid
    }
}

For complex systems, the Planner generates data flow diagrams showing how information moves between components:

These diagrams are not decorative. They serve as specification documents for the specialist CLIs. When the Coder CLI receives a task to implement the API Gateway, the data flow diagram shows it exactly which services it connects to, what data formats flow between them, and where the trust boundaries lie. The Security CLI uses the same diagram to identify trust boundary crossings that require input validation.

The Planner CLI generates three types of diagrams, each serving a different purpose:

Diagram Type Audience Shows When Generated
Component Human reviewer System structure, subsystem boundaries Every plan
Sequence Orchestrator Execution order, specialist interactions Complex plans (>10 tasks)
Data flow Security CLI + Coder Trust boundaries, data movement Plans involving external integrations

The diagrams are embedded in the plan output as Mermaid code blocks. The human reviewer sees rendered diagrams. The specialist CLIs receive the raw Mermaid source, which they can parse for structural information (e.g., the Security CLI extracts trust boundary crossings from the data flow diagram to prioritize audit scope).

8.5 The Complete Planner CLI CLAUDE.md

# PlannerCLI — The Architect

## Identity
You are the Planner CLI — the architect of a multi-CLI agent system.
You THINK before anyone builds. You decompose requirements into precise,
executable task graphs. You document architectural decisions. You identify
risks before they become bugs.

## The Planning Principle
You have NO implementation tools: no file_write, no bash (except read-only), no Edit.
You have: Read, Grep, Glob, WebSearch, WebFetch, EnterPlanMode, ExitPlanMode.
If you find yourself wanting to write code, STOP. That is the Coder's job.
Your job is to produce the PLAN that the Coder follows.

## Planning Methodology

### Phase 1: UNDERSTAND
- Read the requirement carefully. Identify ambiguities.
- Search the existing codebase for relevant patterns, conventions, dependencies.
- List implicit requirements the human did not state but the goal requires.
- If critical information is missing, list what you need — do NOT guess.

### Phase 2: DESIGN
- Identify major components/subsystems needed.
- For each component: what tasks produce it?
- Map dependencies: what must happen before what?
- Identify parallel opportunities: what is independent?
- Assign each task to the appropriate specialist CLI.
- Estimate effort per task (simple/moderate/complex).

### Phase 3: DOCUMENT
- Write an ADR for every architectural decision with alternatives and rationale.
- Generate component diagram (Mermaid) showing system structure.
- Generate sequence diagram showing execution flow.
- Compute the critical path and total estimated time.

### Phase 4: DELIVER
- Output the complete task plan as structured JSON (TaskDAG format).
- Include ADRs as separate documents.
- Include Mermaid diagrams inline.
- Present to the Orchestrator for dispatch.

## Output Format
Your output MUST include:
1. **Task Graph** — JSON with id, description, assigned_cli, depends_on, priority
2. **Critical Path** — The longest sequential chain, with estimated total time
3. **ADRs** — One per architectural decision, in docs/decisions/ format
4. **Diagrams** — Component diagram + sequence diagram in Mermaid
5. **Risk Register** — Known risks with likelihood, impact, and mitigation

## Quality Gates
- Every task must have: clear description, assigned specialist, dependencies listed
- No circular dependencies in the task graph
- No task estimated at more than 10 minutes (split if larger)
- Every architectural decision must have at least 2 alternatives considered
- The critical path must be identified and documented

## What You Do NOT Do
- Write production code (Coder)
- Audit security (Security CLI)
- Run tests (Validator CLI)
- Research topics in depth (Research CLI — you may do light research for planning context)
- You DESIGN the work. Others EXECUTE the work.

## Planning Anti-Patterns to Avoid
1. **Over-decomposition**: Breaking a 2-line config change into 5 tasks with dependencies. If the whole thing takes <2 minutes, it is ONE task.
2. **Under-decomposition**: A single task that says "Build the entire backend." If a specialist needs >10 minutes, split the task.
3. **Phantom dependencies**: Making task B depend on task A when they are actually independent. This serializes work that could be parallel.
4. **Missing implicit tasks**: Forgetting database migrations, error handling, input validation, environment config. These are always needed.
5. **Assumption blindness**: Planning based on assumptions about the codebase without reading it first. Always scan the existing code before decomposing.

8.6 Planning Patterns for Different Project Scales

Small Projects (5-10 tasks)

For small projects, the Planner's value is in preventing missed dependencies. A 5-task project does not need wave scheduling, but it does need the Planner to identify that "write integration tests" depends on both "implement API routes" and "set up test database" — a dependency that inline planning often misses.

Medium Projects (15-30 tasks)

For medium projects, the Planner adds wave scheduling and parallel optimization. Identifying which tasks can run in parallel cuts total execution time by 30-50%. ADRs capture the 3-5 architectural decisions that shape the project.

Large Projects (50+ tasks)

For large projects, the Planner becomes essential. Without it, the Orchestrator's inline decomposition produces a flat task list with missing dependencies and no architectural coherence. The Planner breaks the project into phases (MVP → Core Features → Polish → Launch), each with its own task graph. It identifies shared infrastructure that multiple phases depend on and schedules it first. It generates a risk register that flags the hardest technical challenges and suggests mitigation strategies.

ULTRAPLAN is most valuable at this scale. A 50-task decomposition with correct dependencies, specialist assignments, and effort estimates requires deep reasoning that benefits from extended thinking budgets. At $15 per million input tokens for Opus with 16K thinking tokens, a single ULTRAPLAN call costs approximately $0.30-0.50. That is a negligible cost compared to the time saved by not discovering a missing dependency 40 tasks into execution.

Non-Engineering Planning

The Planner CLI pattern extends to any domain where work must be decomposed before execution. A legal case Planner decomposes "Prepare for the Johnson deposition" into research tasks (review prior testimony, identify key documents), preparation tasks (draft question outlines, prepare exhibits), and logistics tasks (schedule court reporter, reserve conference room), with dependencies and assignments to specialist CLIs.

A medical treatment Planner decomposes a complex care plan into diagnostic tasks (order labs, schedule imaging), treatment tasks (prescribe medications, schedule procedures), monitoring tasks (follow-up appointments, lab rechecks), with dependencies that ensure diagnostics complete before treatment begins. The critical path analysis identifies which diagnostic results are blocking treatment decisions.

The architecture remains identical. The decomposition methodology changes. The specialist roster changes. But the Planner's core behavior — understand, design, document, deliver — remains constant across domains.

8.7 Testing the Planner CLI

Test 1: Decomposition Completeness

Requirement: "Build user registration with email verification"
Expected: Plan includes database schema, API route, email service integration,
verification token generation, verification endpoint, error handling, tests.
Missing any of these = incomplete decomposition.

Test 2: Dependency Correctness

Requirement: "Build a REST API with database access"
Expected: Schema migration task comes BEFORE model task.
Model task comes BEFORE route handler task.
Route handler task comes BEFORE integration test task.
Violations = incorrect dependency ordering.

Test 3: No Tool Restriction Violations

Scenario: Planner receives "implement the login endpoint"
Expected: Planner produces a PLAN for the login endpoint.
Does NOT attempt to use file_write, Edit, or Bash to create files.

Test 4: ADR Generation

Requirement: "Add caching to the API"
Expected: ADR generated for cache technology choice (Redis vs Memcached vs in-memory),
with alternatives, pros/cons, and rationale.

Test 5: Critical Path Identification

Input: 20-task graph with known critical path
Expected: Planner correctly identifies the critical path
and reports the estimated minimum execution time.

Test 6: Risk Identification

Requirement: "Build a multi-tenant system with shared database"
Expected: Planner identifies tenant isolation as a high-impact risk
and includes a dedicated tenant isolation test task in the plan.

Test 7: Iterative Planning

Scenario: Human rejects ADR for Redis caching, requests in-memory alternative.
Expected: Planner regenerates affected tasks (no Redis setup task,
modified caching implementation task), preserves unaffected tasks.

If all seven tests pass, your Planner CLI decomposes completely, orders dependencies correctly, respects its tool restrictions, documents decisions, identifies the critical path, flags risks, and handles plan revisions gracefully.
