Chapter 22: Scaling — From Laptop to Enterprise

There is a moment in every multi-CLI builder's journey when the laptop fan starts screaming. You have eight Claude Code instances running — a Planner decomposing a migration, a Security auditor scanning dependencies, three Coders implementing features in parallel, a Reviewer watching their output, a Documentation writer updating the README, and a QA validator running tests. Your MacBook's 32GB of RAM is at 94%. Swap is engaged. One of the Coders silently OOM-killed in the background, and you do not notice until its PR comes back empty.

You have built a system that works. Now you need a system that scales.

This chapter maps the path from a single developer running everything on one machine to an enterprise deployment serving fifty engineers across three continents. The territory is technical — resource allocation, token budgets, container orchestration, cost modeling — but the underlying question is architectural: how do you grow a multi-CLI system without losing the properties that made it powerful?

The answer comes in three tiers. Each has its own resource profile, failure modes, and economics. You do not climb because the higher tier is "better." You climb because you have hit the ceiling of the tier you are on.

22.1 Tier 1: The Solo Developer — Eight CLIs on a MacBook

The first tier is where everyone starts. One machine. One developer. Multiple CLI sessions sharing a single set of hardware resources.

The Resource Calculator

Each Claude Code session maintains a streaming connection to the API, a local tool execution environment, and an in-memory conversation history. The resource profile varies dramatically based on activity:

Per-session resource profile:

Memory:
  Idle/light usage:              ~20 MB RSS
  Active with tools:             ~60-80 MB RSS
  Heavy context (500+ turns):    ~125 MB RSS
  With 1M context window:        ~200 MB RSS

CPU:
  Idle (waiting for API):        ~0% (event loop sleeping)
  Tool execution (bash):         Spikes to 100% of one core
  Streaming response:            ~5-10% of one core
  Context compaction:            ~15% of one core for 2-5 seconds

Model Assignment Per CLI

Not every CLI needs the most expensive model. This is the single biggest cost optimization available, and most builders miss it entirely.

# .forge/model_assignment.toml

[models]
planner     = "opus"      # Complex decomposition — worth the cost
security    = "opus"      # Cannot afford false negatives
coder       = "sonnet"    # Balanced: good code quality at 5x lower cost
reviewer    = "sonnet"    # Pattern matching, not deep reasoning
qa          = "sonnet"    # Test generation is well-structured
docs        = "haiku"     # Documentation is formulaic — fast and cheap
validator   = "haiku"     # Lint-level checks, binary pass/fail

The cost difference is dramatic:

Model Input (1M tokens) Output (1M tokens) Typical Session Cost
Opus 4 $15.00 $75.00 $2.50-5.00
Sonnet 4 $3.00 $15.00 $0.50-1.00
Haiku 4.5 $0.80 $4.00 $0.08-0.15

An 8-CLI forge with all-Opus assignment costs ~$25 per complex task. With model mixing, the same forge costs ~$8 — a 3x reduction with minimal quality impact, because the Haiku CLIs (docs, validator) were never doing work that required Opus-level reasoning.

Model Escalation

The forge implements a model escalation strategy: start with the assigned model, and escalate when the task demands it:

pub struct ModelEscalation {
    pub base_model: ModelId,
    pub escalation_model: ModelId,
    pub escalation_triggers: Vec<EscalationTrigger>,
}

pub enum EscalationTrigger {
    RepeatedFailure { threshold: u32 },     // 3 consecutive failures
    ComplexityScore { threshold: f64 },      // Task complexity above 0.8
    SecurityFinding { severity: Severity },  // Critical security finding
    HumanOverride,                           // Operator requests escalation
}

impl ModelEscalation {
    pub fn should_escalate(&self, context: &TaskContext) -> bool {
        self.escalation_triggers.iter().any(|trigger| {
            match trigger {
                EscalationTrigger::RepeatedFailure { threshold } => {
                    context.consecutive_failures >= *threshold
                }
                EscalationTrigger::ComplexityScore { threshold } => {
                    context.estimated_complexity >= *threshold
                }
                EscalationTrigger::SecurityFinding { severity } => {
                    context.has_finding_above(*severity)
                }
                EscalationTrigger::HumanOverride => context.human_escalated,
            }
        })
    }
}

The Security CLI starts on Opus by default, but the Coder CLI starts on Sonnet and escalates to Opus only when it has failed three consecutive times or the task complexity score exceeds 0.8. This adaptive approach optimizes for cost while maintaining quality where it matters.

Token Budget Management

The task-budgets beta header enables per-task token limits, preventing runaway sessions from consuming unbounded API credits:

pub struct TokenBudget {
    pub session_limit: u64,       // Max tokens per session
    pub daily_limit: u64,         // Max tokens per day
    pub task_limit: Option<u64>,  // Max tokens per individual task
    pub spent: AtomicU64,
}

impl TokenBudget {
    pub fn can_spend(&self, estimated_tokens: u64) -> BudgetDecision {
        let current = self.spent.load(Ordering::Relaxed);

        if current + estimated_tokens > self.session_limit {
            return BudgetDecision::Blocked("Session budget exceeded");
        }

        if self.daily_spent() + estimated_tokens > self.daily_limit {
            return BudgetDecision::Blocked("Daily budget exceeded");
        }

        BudgetDecision::Allowed
    }

    pub fn record_usage(&self, tokens: u64) {
        self.spent.fetch_add(tokens, Ordering::Relaxed);
    }
}

Budget configuration per CLI:

# .forge/budgets.toml

[budgets.planner]
session_limit = 500_000    # Opus sessions are expensive
daily_limit   = 2_000_000

[budgets.security]
session_limit = 300_000
daily_limit   = 1_500_000

[budgets.coder]
session_limit = 200_000    # Sonnet is cheaper per token
daily_limit   = 1_000_000

[budgets.docs]
session_limit = 100_000    # Haiku is very cheap
daily_limit   = 500_000

When a CLI exhausts its budget, the orchestrator receives a BudgetExhausted event and can either allocate more tokens, switch to a cheaper model, or defer the task to the next budget period.

BG_SESSIONS — Background Work

The BG_SESSIONS feature gate enables background sessions — Claude Code instances that run without a terminal. They execute tasks, report results via SendUserMessage, and terminate when complete.

For the solo developer, this is the bridge to KAIROS (Chapter 20) without running a full daemon. You type a prompt, the orchestrator dispatches specialists as background sessions, and you continue working while they execute:

# Foreground: you're coding in the main session
# Background: security scan running in parallel
claude --background --profile security-auditor \
  --prompt "Scan all files changed in last commit for OWASP Top 10"

The background session writes results to .claude/bg_sessions/<session_id>/result.json. The orchestrator polls for completion and synthesizes results into the main session.

Rate Limit Coordination

When multiple developers share the same API key (common in team deployments), rate limit coordination becomes critical. Without it, two developers running full 8-CLI forges simultaneously send 16 concurrent requests to the API — easily hitting rate limits.

pub struct RateLimitCoordinator {
    shared_semaphore: Arc<Semaphore>,
    per_user_limits: HashMap<String, Semaphore>,
    global_rpm: u64,
}

impl RateLimitCoordinator {
    pub async fn acquire(&self, user: &str) -> RateLimitGuard {
        // Global limit: shared across all users
        let global = self.shared_semaphore.acquire().await.unwrap();

        // Per-user limit: fair share allocation
        let user_limit = self.per_user_limits
            .get(user)
            .expect("User not registered");
        let user_guard = user_limit.acquire().await.unwrap();

        RateLimitGuard { global, user_guard }
    }
}

Configuration:

# .forge/rate_limits.toml

[limits]
global_rpm = 1000           # API rate limit for the team key
per_user_rpm = 200          # Fair share per developer
per_cli_rpm = 50            # Per-CLI limit
burst_allowance = 20        # Short burst above per_cli_rpm

Conflict Resolution

When multiple developers' CLIs modify the same file, the shared infrastructure needs conflict resolution. The forge uses git worktrees (Chapter 17) to isolate each developer's work, but conflicts arise at the knowledge layer:

pub struct ConflictResolver {
    strategy: ConflictStrategy,
}

pub enum ConflictStrategy {
    LastWriteWins,              // Simple but lossy
    MergeFindings,              // Combine non-contradicting findings
    HumanReview,                // Flag for human decision
    PriorityBased { cli_priority: HashMap<CliRole, u32> },
}

impl ConflictResolver {
    pub fn resolve(
        &self,
        existing: &MemoryEntry,
        incoming: &MemoryEntry,
    ) -> Resolution {
        match &self.strategy {
            ConflictStrategy::MergeFindings => {
                if existing.contradicts(incoming) {
                    Resolution::HumanReview {
                        existing: existing.clone(),
                        incoming: incoming.clone(),
                        reason: "Contradicting findings from different CLIs",
                    }
                } else {
                    Resolution::Merge(existing.merge_with(incoming))
                }
            }
            ConflictStrategy::PriorityBased { cli_priority } => {
                let existing_priority = cli_priority.get(&existing.source_cli).unwrap_or(&0);
                let incoming_priority = cli_priority.get(&incoming.source_cli).unwrap_or(&0);
                if incoming_priority >= existing_priority {
                    Resolution::Accept(incoming.clone())
                } else {
                    Resolution::Reject("Lower priority CLI cannot override higher priority finding")
                }
            }
            _ => Resolution::Accept(incoming.clone()),
        }
    }
}

The default strategy for most teams: MergeFindings with HumanReview fallback. Security findings from the Security CLI take priority over Coder CLI findings on the same file — the priority-based strategy handles this cleanly.

PostgreSQL for Shared State

At enterprise scale, SQLite is not sufficient for shared state. The memory system moves to PostgreSQL with row-level locking:

pub struct EnterpriseMemoryStore {
    pool: sqlx::PgPool,
}

impl EnterpriseMemoryStore {
    pub async fn write_finding(
        &self,
        finding: &SecurityFinding,
        cli_id: &str,
    ) -> Result<(), MemoryError> {
        sqlx::query!(
            r#"
            INSERT INTO findings (id, cli_id, severity, file_path, description, created_at)
            VALUES ($1, $2, $3, $4, $5, NOW())
            ON CONFLICT (id) DO UPDATE SET
                severity = EXCLUDED.severity,
                description = EXCLUDED.description,
                updated_at = NOW()
            "#,
            finding.id,
            cli_id,
            finding.severity.to_string(),
            finding.file_path,
            finding.description,
        )
        .execute(&self.pool)
        .await?;

        Ok(())
    }

    pub async fn get_findings_since(
        &self,
        since: DateTime<Utc>,
        min_severity: Severity,
    ) -> Result<Vec<SecurityFinding>, MemoryError> {
        sqlx::query_as!(
            SecurityFinding,
            r#"
            SELECT id, cli_id, severity, file_path, description, created_at
            FROM findings
            WHERE created_at >= $1 AND severity >= $2
            ORDER BY severity DESC, created_at DESC
            "#,
            since,
            min_severity.to_string(),
        )
        .fetch_all(&self.pool)
        .await
        .map_err(Into::into)
    }
}

Redis for Coordination

Redis handles the transient coordination state: rate limits, work queues, pub/sub for inter-CLI events, and distributed locking:

pub struct ForgeRedis {
    client: redis::Client,
}

impl ForgeRedis {
    pub async fn enqueue_task(&self, task: &ForgeTask) -> Result<(), RedisError> {
        let mut conn = self.client.get_multiplexed_async_connection().await?;
        let payload = serde_json::to_string(task)?;
        redis::cmd("LPUSH")
            .arg("forge:tasks:pending")
            .arg(payload)
            .query_async(&mut conn)
            .await?;
        Ok(())
    }

    pub async fn dequeue_task(&self, timeout_secs: u64) -> Result<Option<ForgeTask>, RedisError> {
        let mut conn = self.client.get_multiplexed_async_connection().await?;
        let result: Option<String> = redis::cmd("BRPOP")
            .arg("forge:tasks:pending")
            .arg(timeout_secs)
            .query_async(&mut conn)
            .await?;

        match result {
            Some(payload) => Ok(Some(serde_json::from_str(&payload)?)),
            None => Ok(None),
        }
    }
}

22.4 BRIDGE_MODE — Remote and Federated Architecture

The BRIDGE_MODE feature gate enables remote control of Claude Code sessions. A bridge client connects to a headless Claude Code process and controls it programmatically — sending prompts, receiving streaming responses, and managing the conversation state.

For enterprise scaling, BRIDGE_MODE enables a federated architecture where the coordinator and specialists can run on different machines, different networks, or different continents:

┌──────────────────────┐     Bridge      ┌──────────────────────┐
│  COORDINATOR          │◄──────────────►│  SECURITY AUDITOR     │
│  (New York)           │    WebSocket    │  (London)             │
│                       │                 │  Opus, full OWASP     │
│  Dispatch + Synthesis │                 │  Local to EU repos    │
└──────────────────────┘                  └──────────────────────┘
         │
         │ Bridge
         ▼
┌──────────────────────┐     Bridge      ┌──────────────────────┐
│  CODER                │◄──────────────►│  QA VALIDATOR         │
│  (San Francisco)      │    WebSocket    │  (Tokyo)              │
│  Sonnet, backend      │                 │  Haiku, fast          │
│  Local to US repos    │                 │  Local to APAC repos  │
└──────────────────────┘                  └──────────────────────┘

The coordinator in New York dispatches a security audit to London (data residency requirement: EU code stays in EU), dispatches code review to San Francisco (close to the developers), and sends QA validation to Tokyo (close to the APAC test infrastructure). Each specialist runs in its own bridge-connected session, and results flow back to the coordinator for synthesis.

Bridge mode supports up to 32 parallel CCR (Claude Code Remote) sessions per coordinator — the DAEMON feature gate's limit from the source code. For most enterprises, this is sufficient for the busiest coordinator.

Data Residency

Bridge mode solves a compliance problem that monolithic deployment cannot. When EU regulations require that EU source code is processed within EU borders, a single-machine deployment either runs in the EU (adding latency for US developers) or violates compliance. Bridge mode allows the coordinator to route tasks to the correct geographic region based on the repository's regulatory classification:

pub struct GeoRouter {
    regions: HashMap<String, Vec<BridgeEndpoint>>,
    repo_classification: HashMap<String, Region>,
}

impl GeoRouter {
    pub fn route_task(&self, task: &ForgeTask) -> &BridgeEndpoint {
        let region = self.repo_classification
            .get(&task.repo)
            .unwrap_or(&Region::Default);

        let endpoints = self.regions.get(&region.to_string()).unwrap();
        // Round-robin within the region
        &endpoints[task.id.as_bytes()[0] as usize % endpoints.len()]
    }
}

22.5 Multi-LLM Mixing: The Right Model for Every Token

Beyond assigning different models to different CLIs, advanced deployments mix models within a single CLI session:

Phase Model Reasoning
Intent analysis Haiku Fast, cheap keyword matching
Task planning Opus Complex decomposition needs deep reasoning
Code generation Sonnet Best cost/quality ratio for implementation
Code review Sonnet Pattern matching, structured output
Security audit Opus Cannot afford false negatives on security
Documentation Haiku Formulaic, high-structure output
Synthesis Opus Combining multiple specialist outputs

The orchestrator implements model routing per turn:

pub fn select_model(phase: &ForgePhase, context: &TaskContext) -> ModelId {
    match phase {
        ForgePhase::Intent => ModelId::Haiku,
        ForgePhase::Plan => ModelId::Opus,
        ForgePhase::Execute => match context.cli_role {
            CliRole::Security => ModelId::Opus,
            CliRole::Coder | CliRole::Reviewer | CliRole::QA => ModelId::Sonnet,
            CliRole::Docs | CliRole::Validator => ModelId::Haiku,
            _ => ModelId::Sonnet,
        },
        ForgePhase::Debate => ModelId::Sonnet,
        ForgePhase::Synthesize => ModelId::Opus,
    }
}

The cost impact is substantial. A complex task that runs through all six phases on Opus costs ~$5. With model mixing, the same task costs ~$2 — the intent phase runs on Haiku (pennies), the execution phase runs three Sonnet CLIs and one Opus CLI, and only the planning and synthesis phases use Opus at full price.

Effort Levels

Claude Code's effort system (low, medium, high) controls how much reasoning the model applies. For the multi-CLI forge, effort maps to task criticality:

# .forge/effort.toml

22.8 Operational Playbook

Running a multi-CLI forge in production requires documented procedures for common scenarios.

Scenario: Budget Exhaustion Mid-Task

A complex task hits the daily token budget at 3 PM. Three Coder CLIs are mid-implementation. The orchestrator: 1. Receives BudgetExhausted from the budget service 2. Signals in-progress CLIs to checkpoint their work (save partial results, commit partial changes) 3. Queues remaining subtasks for the next budget period 4. Sends a Brief notification: "Budget exhausted at 83% task completion. Checkpointed work for 3 CLIs. Will resume tomorrow at 00:00 UTC."

The developer can either wait for the budget reset or allocate emergency tokens from the reserve pool.

Scenario: API Provider Outage

Anthropic's API returns 503 for 15 minutes during a KAIROS overnight run: 1. Retry logic (Chapter 21) handles the first 3 minutes with exponential backoff 2. After 3 minutes of continuous failure, the circuit breaker trips 3. The orchestrator enters degraded mode: no new task dispatches, existing tasks paused 4. Monitoring sends a PagerDuty alert 5. When the API recovers, the circuit breaker resets after 3 successful health checks 6. Paused tasks resume from their last checkpoint

pub struct CircuitBreaker {
    state: CircuitState,
    failure_count: u32,
    failure_threshold: u32,
    recovery_timeout: Duration,
    last_failure: Option<Instant>,
}

pub enum CircuitState {
    Closed,     // Normal operation
    Open,       // Failing — reject all requests
    HalfOpen,   // Testing recovery — allow one request
}

impl CircuitBreaker {
    pub fn can_proceed(&mut self) -> bool {
        match self.state {
            CircuitState::Closed => true,
            CircuitState::Open => {
                if self.last_failure.map_or(false, |t| t.elapsed() > self.recovery_timeout) {
                    self.state = CircuitState::HalfOpen;
                    true  // Allow one test request
                } else {
                    false
                }
            }
            CircuitState::HalfOpen => false,  // Wait for test result
        }
    }

    pub fn record_success(&mut self) {
        self.failure_count = 0;
        self.state = CircuitState::Closed;
    }

    pub fn record_failure(&mut self) {
        self.failure_count += 1;
        if self.failure_count >= self.failure_threshold {
            self.state = CircuitState::Open;
            self.last_failure = Some(Instant::now());
        }
    }
}

Scenario: Worker Pod OOM-Kill in Kubernetes

A Coder CLI pod exceeds its 256Mi memory limit while running cargo build inside a tool call: 1. Kubernetes kills the pod 2. The Deployment controller restarts it (standard k8s behavior) 3. The orchestrator detects the worker disconnect via the Bridge heartbeat 4. The orchestrator marks the task as Stalled 5. After the pod restarts, the orchestrator re-dispatches the task with a memory annotation: --tool-memory-limit 512Mi 6. If the task fails again, it escalates to human review with the recommendation: "This task requires more memory than the pod limit. Consider increasing limits or running on a dedicated node."

Scenario: Scaling During Sprint

A team of 10 developers starts a two-week sprint. PR volume triples. The KAIROS PR review pipeline cannot keep up: 1. The forge_pending_tasks metric exceeds the HPA threshold (average > 3 pending per pod) 2. HPA scales Coder pods from 2 to 6 3. HPA scales SecurityAuditor pods from 1 to 3 4. The rate limit coordinator redistributes the per-CLI allowance across more pods 5. Two weeks later, sprint ends. PR volume drops. HPA scales back to baseline. 6. Budget report shows the sprint cost $800 more than a normal two weeks — a trivial cost compared to the developer productivity gained from sub-5-minute automated reviews.

22.10 The Decision Framework: When to Upgrade

Do not upgrade tiers preemptively. Each tier adds complexity. Only move when you hit concrete pain points:

Pain Point Current Tier Upgrade To
Laptop fan screaming, OOM-kills Tier 1 Tier 1 with Rust CLIs first
Multiple developers stepping on each other's memory Tier 1 Tier 2
API rate limit contention across the team Tier 1/2 Tier 2 with rate coordination
Data residency requirements Tier 2 Tier 3 with BRIDGE_MODE
Need 24/7 availability (KAIROS daemon) Tier 2 Tier 3 with Kubernetes
>20 concurrent CLI sessions Tier 2 Tier 3 with HPA
Multi-region team with latency concerns Tier 2 Tier 3 with federated bridges

The most common mistake is jumping to Tier 3 for the technology prestige. A solo developer does not need Kubernetes. A team of three does not need Redis-backed rate limiting. Start with the simplest architecture that works and add complexity only when the pain justifies it.

22.11 Non-Engineering Domains

Legal Discovery. A law firm processes document review for a class-action lawsuit: 50,000 documents across three jurisdictions. The Tier 3 deployment runs 20 specialist CLIs — one per document category (contracts, emails, financial records, regulatory filings) — with geo-routing that ensures EU documents are processed in Frankfurt, US documents in Virginia. The coordinator synthesizes findings into a privilege log and a relevance scoring matrix. Model mixing keeps costs manageable: Haiku for initial categorization ($0.002/document), Sonnet for relevance assessment ($0.02/document), Opus for privilege determination ($0.15/document). Total cost for 50,000 documents: approximately $8,500 — less than a single paralegal's monthly salary.

Climate Research. A climate modeling consortium runs KAIROS on a shared compute cluster. Twelve specialist CLIs analyze satellite data, ocean temperature readings, atmospheric composition samples, and ice core data. Each runs in its own Kubernetes pod with access to a shared PostgreSQL database of historical findings. Budget management is critical — the consortium allocates $5,000/month across twelve research teams, and the token budget coordinator ensures fair distribution. Dream consolidation runs weekly, cross-referencing findings across research domains: when the ocean temperature CLI detects an anomaly that correlates with the atmospheric CLI's CO2 readings, the consolidated finding gets elevated to the cross-domain knowledge base.

Supply Chain Logistics. A global shipping company runs the forge across three continents. The route optimization CLI runs in Singapore (closest to major shipping lanes), the customs compliance CLI runs in regional instances (US, EU, APAC — each with jurisdiction-specific rules), and the risk assessment CLI runs centrally in Amsterdam. BRIDGE_MODE connects them all. When a container ship diverts due to a Red Sea closure, the event triggers all three CLIs simultaneously: route optimization calculates alternatives, customs compliance checks documentation requirements for the new route, and risk assessment evaluates cargo insurance implications. The federated architecture ensures sub-second response because each CLI is geographically close to its data source.
