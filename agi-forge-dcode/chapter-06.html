Chapter 6: The Orchestrator CLI — The Brain

COORDINATOR_MODE, Intent Analysis, Task DAGs, and the Art of Not Doing the Work Yourself

Eight musicians warming up independently are not an orchestra. Eight specialist CLIs running in isolation are not a system. The difference is a conductor — a component whose entire job is to decide what happens next, who does it, and how the results fit together.

The Orchestrator CLI is that conductor. It receives human prompts, decomposes them into concrete tasks, dispatches those tasks to the right specialists, manages dependencies and parallel execution, handles failures, and synthesizes the results into a coherent response. It never writes a line of code. It never runs a test. It never reads a source file. This restriction is not a limitation — it is the most important architectural decision in the entire system.

Think about what happens when a CTO tries to also be the lead developer. They read every pull request, make implementation decisions at the code level, and become the bottleneck for every feature. The organization slows to the speed of one person's attention. Now imagine a CTO who sets direction, allocates resources, resolves conflicts between teams, and lets teams execute independently. The organization moves at the speed of its teams — parallel, specialized, and fast. The Orchestrator CLI is the second CTO.

This chapter builds the Orchestrator from three components: the Intent Analysis Engine (understanding what the human wants), the Task DAG Engine (organizing the work), and the Dispatch Protocol (getting the work done). We will also cover the subtleties that separate a working Orchestrator from a production Orchestrator: conflict resolution between specialists, failure cascade management, concurrency control, and the debate pattern that produces higher-confidence results than any single specialist. By the end, you will have a working Orchestrator that turns a single human prompt into coordinated multi-specialist execution.

6.1 Why the Orchestrator Must Be Restricted

Consider what happens without restriction. A coordinator with file-read access will read files to "understand the codebase" before delegating. Its context window fills with source code. It starts making implementation decisions that belong to specialists. It becomes a bottleneck — reading, analyzing, then delegating work it already half-decided. The specialist receives a task colored by the coordinator's incomplete analysis and produces work that satisfies the coordinator's biased framing rather than the actual requirement.

This is the god-orchestrator anti-pattern — the most common failure mode in multi-agent systems. The coordinator tries to be omniscient. It reads everything, understands everything, micromanages everything. In doing so, it reproduces the single-context bottleneck that the multi-agent architecture was designed to eliminate.

The anti-pattern is seductive because it feels productive. The coordinator reads server.js and thinks "Now I understand the codebase, I can delegate more precisely." But that understanding cost 3,000 tokens of context, and the Coder specialist is going to read the same file anyway. Worse, the coordinator's "understanding" is shallow — it skimmed the file, not traced the data flow — and it will now write task descriptions that reflect its shallow analysis. The specialist, constrained by the coordinator's framing, produces work that is technically correct but architecturally misguided. The coordinator has not helped. It has introduced a bias.

The pattern gets catastrophic at scale. A coordinator managing 15 tasks across 6 specialists that also reads every relevant file will consume 80,000-120,000 tokens of context before issuing a single delegation. Each subsequent LLM call now processes all that context. At $15 per million input tokens for Opus, a bloated coordinator costs 10-20x more than a lean one. And the quality is worse — LLMs attend less precisely to earlier content as context grows. The instructions from turn 1 fade as the context fills with source code the coordinator should never have read.

Claude Code's COORDINATOR_MODE solves this by enforcing restriction at the tool level, making it structurally impossible for the coordinator to fall into this trap:

// coordinator/coordinatorMode.ts (from td_claudecode source)
export function isCoordinatorMode(): boolean {
  if (feature('COORDINATOR_MODE')) {
    return isEnvTruthy(process.env.CLAUDE_CODE_COORDINATOR_MODE)
  }
  return false
}

When COORDINATOR_MODE is active, the main agent's tool set is restricted to exactly these:

Tool Purpose
AgentTool Spawn and dispatch specialist workers
SendMessage Continue communication with running workers
TaskStopTool Kill a running worker
SyntheticOutput Generate structured output without tool execution
TeamCreate Create a team of workers
TeamDelete Disband a team

That is the complete list. No file_read. No file_write. No bash. No grep. No glob. No filesystem access whatsoever. Workers get the full 40+ tool set. Workers cannot spawn their own agents — no AgentTool access, no recursive delegation. The hierarchy is structurally enforced: coordinator delegates down, workers execute and report up.

This is not just a convention — it is an architectural invariant enforced by the runtime. Even if the LLM behind the Orchestrator hallucinates a file_read tool call, the tool registry rejects it. The coordinator cannot cheat its way into reading files, just as a process running in a Linux namespace cannot access files outside its mount. The restriction is at the boundary, not in the prompt.

The separation in practice:

  • Coordinator: Operates on abstractions — task descriptions, dependency graphs, completion status, error summaries. A coordinator managing 12 tasks consumes ~5,000 tokens of metadata.
  • Workers: Operate on concrete artifacts — files, processes, test outputs. A worker executing one task uses its full 200K token context for that single concern.

This separation keeps the coordinator's context clean. Without it, a coordinator that also reads source files consumes 50,000 tokens before delegating anything — and every subsequent turn pays the cost of that bloated context.

6.2 The Intent Analysis Engine

When a human says "Build me a REST API with JWT auth and PostgreSQL," the Orchestrator must decompose this into concrete, assignable tasks. This decomposition is the hardest part of orchestration — it requires understanding the problem domain well enough to enumerate implicit requirements that the user did not specify: error handling, input validation, logging, migration scripts, environment configuration.

The challenge is that human prompts are deliberately ambiguous. "Build me a REST API" does not specify whether you want Express or Fastify, PostgreSQL with raw SQL or Prisma ORM, JWT with HS256 or RS256, or whether you need rate limiting, CORS configuration, health endpoints, or Swagger documentation. A single-agent system resolves these ambiguities by reading existing code and making educated guesses. A multi-agent system must resolve them upfront — because once the Orchestrator dispatches tasks to six specialists in parallel, changing a fundamental decision (like ORM choice) means throwing away work.

This is why intent analysis is a separate engine, not just a prompt instruction. It must produce a complete task specification before any specialist starts working.

Heuristic-First, LLM-Second

The intent analysis engine uses a two-stage approach: fast heuristic matching for common patterns, falling back to LLM analysis for novel requests. This dual-stage design matters for responsiveness. A user who types "fix the login bug" expects the Orchestrator to start working immediately, not spend 3 seconds analyzing the prompt with an LLM call. The heuristic stage gives that instant responsiveness for the 60-70% of prompts that match well-understood patterns.

Stage 1: Keyword and Pattern Matching (zero latency)

// clis/orchestrator/src/intent/mod.rs
pub enum Intent {
    Build { components: Vec<Component> },
    Fix { target: String, error: Option<String> },
    Review { scope: ReviewScope },
    Audit { focus: AuditFocus },
    Research { topic: String },
    Deploy { environment: String },
    Refactor { target: String, goal: String },
    Custom(String),
}

pub fn classify_intent(prompt: &str) -> Intent {
    let lower = prompt.to_lowercase();

    // Build/create patterns
    if contains_any(&lower, &["build", "create", "implement", "add", "new"]) {
        let components = extract_components(&lower);
        return Intent::Build { components };
    }

    // Fix/debug patterns
    if contains_any(&lower, &["fix", "bug", "error", "broken", "crash", "fail"]) {
        return Intent::Fix {
            target: extract_target(&lower),
            error: extract_error_context(prompt),
        };
    }

    // Security audit patterns
    if contains_any(&lower, &["audit", "security", "vulnerability", "owasp", "cve"]) {
        return Intent::Audit {
            focus: classify_audit_focus(&lower),
        };
    }

    // ... more patterns

    Intent::Custom(prompt.to_string())
}

This matches the pattern from DApp's intent-analyzer.js, which performs prompt analysis for worker allocation. The heuristic stage resolves 60-70% of prompts with zero LLM calls and zero latency. Notice the pattern: each match extracts structured information from the free-text prompt. extract_components might identify "REST API," "JWT auth," and "PostgreSQL" as three separate components. extract_error_context might pull a stack trace from the prompt. The heuristic is not just a router — it is a parser that converts natural language into structured intent.

The heuristic approach also handles compound prompts — "fix the auth bug and then add rate limiting" — by detecting conjunctions and splitting into multiple intents:

pub fn classify_intents(prompt: &str) -> Vec<Intent> {
    let segments = split_compound_prompt(prompt);
    segments.iter().map(|s| classify_intent(s)).collect()
}

fn split_compound_prompt(prompt: &str) -> Vec<&str> {
    // Split on: "and then", "also", "after that", ";", numbered lists
    let connectors = ["and then", "then also", "after that", "also", "; "];
    let mut segments = vec![prompt];
    for conn in &connectors {
        segments = segments.iter()
            .flat_map(|s| s.splitn(2, conn))
            .collect();
    }
    segments
}

Stage 2: LLM-Assisted Decomposition (for complex/ambiguous prompts)

When the heuristic stage produces Intent::Custom, the Orchestrator uses its LLM to analyze the prompt. But critically, it does not ask the LLM to produce code or read files — it asks for a structured task decomposition:

## Task Decomposition Prompt (injected into Orchestrator's CLAUDE.md)

When you receive a prompt that does not match a standard pattern, decompose it:

1. Identify the TOP-LEVEL GOAL (one sentence)
2. List IMPLICIT REQUIREMENTS the user did not state but the goal requires
3. Decompose into CONCRETE TASKS, each with:
   - task_id: unique identifier
   - description: what to do (specific enough for a specialist)
   - assigned_cli: which specialist handles this
   - depends_on: list of task_ids that must complete first
   - estimated_effort: low/medium/high
4. Output as JSON matching the TaskGraph schema

6.3 The Task DAG Engine

A task list is sequential. A task DAG (Directed Acyclic Graph) is parallel. The difference is why multi-CLI systems outperform single-agent systems on complex tasks.

When a single Claude Code agent handles "build a REST API," it works sequentially: schema, then config, then auth, then routes, then tests, then audit. Nine steps, one after another. Even if steps 1 and 2 are independent, the agent does them sequentially because it is a single process with a single context. Total wall time: the sum of all steps.

With a Task DAG, the Orchestrator identifies that steps 1 and 2 are independent and dispatches them in parallel. Steps 3, 4, and 5 become eligible as soon as their dependencies complete — and since they are independent of each other, they also run in parallel. The critical path through the DAG (the longest chain of sequential dependencies) determines total wall time, not the sum of all steps. For the REST API example, the critical path is T1 → T3 → T6 → T8 — four sequential stages instead of nine. The other tasks execute in parallel alongside the critical path, consuming no additional wall time.

This is not a theoretical speedup. In practice, a 9-task REST API build that takes 8-12 minutes with a single agent completes in 4-6 minutes with a well-structured DAG. The speedup is bounded by the critical path length, not the total number of tasks.

The Data Structure

// clis/orchestrator/src/tasks/dag.rs
use std::collections::{HashMap, HashSet, VecDeque};

#[derive(Debug, Clone)]
pub struct Task {
    pub id: String,
    pub description: String,
    pub assigned_cli: String,
    pub depends_on: Vec<String>,
    pub status: TaskStatus,
    pub result: Option<TaskResult>,
    pub priority: Priority,
}

#[derive(Debug, Clone, PartialEq)]
pub enum TaskStatus {
    Pending,
    Ready,       // All dependencies satisfied
    Running,
    Completed,
    Failed { error: String, retries: u32 },
    Blocked,     // Dependency failed
}

pub struct TaskDAG {
    tasks: HashMap<String, Task>,
    dependents: HashMap<String, Vec<String>>, // task_id → tasks that depend on it
}

impl TaskDAG {
    /// Returns all tasks whose dependencies are satisfied and status is Ready
    pub fn ready_tasks(&self) -> Vec<&Task> {
        self.tasks.values()
            .filter(|t| t.status == TaskStatus::Ready)
            .collect()
    }

    /// Mark a task as completed and promote its dependents to Ready
    pub fn complete_task(&mut self, task_id: &str, result: TaskResult) {
        if let Some(task) = self.tasks.get_mut(task_id) {
            task.status = TaskStatus::Completed;
            task.result = Some(result);
        }

        // Check if any dependents are now unblocked
        if let Some(deps) = self.dependents.get(task_id) {
            for dep_id in deps {
                if self.all_dependencies_met(dep_id) {
                    if let Some(dep) = self.tasks.get_mut(dep_id) {
                        dep.status = TaskStatus::Ready;
                    }
                }
            }
        }
    }

    /// Mark a task as failed and cascade failure to dependents
    pub fn fail_task(&mut self, task_id: &str, error: String) {
        if let Some(task) = self.tasks.get_mut(task_id) {
            task.status = TaskStatus::Failed { error, retries: 0 };
        }

        // Cascade: block all downstream tasks
        self.cascade_block(task_id);
    }

    fn all_dependencies_met(&self, task_id: &str) -> bool {
        self.tasks.get(task_id)
            .map(|t| t.depends_on.iter()
                .all(|dep| self.tasks.get(dep)
                    .map(|d| d.status == TaskStatus::Completed)
                    .unwrap_or(false)))
            .unwrap_or(false)
    }
}

Wave Scheduling

The DAG enables wave scheduling — executing independent tasks in parallel waves. A "wave" is a set of tasks that can all execute simultaneously because none of them depend on each other. When all tasks in a wave complete, the next wave becomes eligible.

Wave 1 (parallel):  [Schema Design]  [Environment Config]  [CI Setup]
                          ↓                    ↓
Wave 2 (parallel):  [API Routes]      [Docker Config]
                          ↓
Wave 3 (parallel):  [Auth Middleware]  [RBAC Logic]
                          ↓                ↓
Wave 4 (sequential): [Integration Tests]
                          ↓
Wave 5 (sequential): [Security Audit]

The Orchestrator dispatches all Wave 1 tasks simultaneously. As each completes, it promotes Wave 2 tasks to Ready status. Independent tasks within the same wave run in parallel — the Coder builds API routes while DevOps configures Docker.

The wave scheduler does not compute waves upfront. Instead, it dynamically discovers the next batch by checking which tasks have all dependencies satisfied. This handles the common case where tasks within a wave complete at different speeds — a fast task in Wave 2 that has all its Wave 1 dependencies met starts immediately, without waiting for slower Wave 1 tasks that do not affect it.

// clis/orchestrator/src/tasks/scheduler.rs
pub struct WaveScheduler {
    dag: TaskDAG,
    max_concurrent: usize,
    running: HashSet<String>,
}

impl WaveScheduler {
    pub fn next_dispatch_batch(&mut self) -> Vec<&Task> {
        let ready = self.dag.ready_tasks();
        let capacity = self.max_concurrent - self.running.len();

        // Dispatch up to capacity, prioritizing higher-priority tasks
        let mut batch: Vec<&Task> = ready.into_iter()
            .take(capacity)
            .collect();
        batch.sort_by_key(|t| std::cmp::Reverse(t.priority.clone()));

        for task in &batch {
            self.running.insert(task.id.clone());
        }

        batch
    }
}

DAG Validation and Cycle Detection

A critical property of the task DAG is that it must be acyclic. If Task A depends on Task B, and Task B depends on Task A, neither can ever start. This is a deadlock — and it is particularly insidious in LLM-generated task graphs because the LLM can introduce circular dependencies without realizing it.

The Orchestrator must validate the DAG before executing it:

impl TaskDAG {
    /// Validate the DAG: check for cycles, orphan dependencies, and unreachable tasks
    pub fn validate(&self) -> Result<(), DagError> {
        // Topological sort detects cycles
        let mut in_degree: HashMap<&str, usize> = HashMap::new();
        for (id, task) in &self.tasks {
            in_degree.entry(id.as_str()).or_insert(0);
            for dep in &task.depends_on {
                *in_degree.entry(dep.as_str()).or_insert(0) += 0;
                // Validate dependency exists
                if !self.tasks.contains_key(dep) {
                    return Err(DagError::OrphanDependency {
                        task: id.clone(),
                        missing_dep: dep.clone(),
                    });
                }
            }
        }

        // Kahn's algorithm for topological sort
        let mut queue: VecDeque<&str> = in_degree.iter()
            .filter(|(_, &deg)| deg == 0)
            .map(|(&id, _)| id)
            .collect();

        let mut sorted_count = 0;
        while let Some(node) = queue.pop_front() {
            sorted_count += 1;
            if let Some(deps) = self.dependents.get(node) {
                for dep in deps {
                    if let Some(deg) = in_degree.get_mut(dep.as_str()) {
                        *deg -= 1;
                        if *deg == 0 {
                            queue.push_back(dep.as_str());
                        }
                    }
                }
            }
        }

        if sorted_count != self.tasks.len() {
            return Err(DagError::CycleDetected);
        }

        Ok(())
    }
}

This validation catches three classes of errors: circular dependencies (cycles), references to non-existent tasks (orphan dependencies), and tasks that can never execute because their dependency chain contains a cycle. Run this validation immediately after constructing the DAG from LLM output — before dispatching any tasks.

## 6.4 The Dispatch Protocol

With the intent analyzed and the task DAG built and validated, the Orchestrator dispatches work to specialists. The dispatch protocol defines how tasks are sent, how results are collected, and how failures are handled.

### The Art of Writing Complete Dispatch Prompts

The most common Orchestrator bug is not a code error — it is an incomplete dispatch prompt. When the Orchestrator sends a specialist a task description that says "implement the auth module," the specialist has to make dozens of decisions that the Orchestrator should have made: which auth strategy? which library? what token format? where do the files go? what patterns do the existing routes follow? The specialist answers these questions by reading the codebase — spending time and tokens on discovery that the Orchestrator could have resolved upfront.

A well-written dispatch prompt is a complete specification. It tells the specialist exactly what to produce, where to put it, what format to follow, and what constraints to respect. The specialist's job is execution, not design. Design is the Orchestrator's job.

Compare these two dispatch prompts:

**Bad (underspecified):**

"Implement JWT authentication for the API."

**Good (self-contained):**

"Create JWT authentication at src/auth/ with these files: - src/auth/middleware.ts: Express middleware, verify RS256 JWT from Authorization: Bearer header - src/auth/service.ts: register(email, password) → user, login(email, password) → {accessToken, refreshToken} - src/auth/routes.ts: POST /auth/register, POST /auth/login, POST /auth/refresh - Use bcrypt (cost=12) for passwords. Access tokens expire in 15m, refresh tokens in 7d. - Tokens signed with RS256 using key from process.env.JWT_PRIVATE_KEY - All errors return {error: string, code: string} JSON, never stack traces. - Follow the existing route pattern in src/routes/health.ts for structure."

The second prompt eliminates ambiguity. The specialist produces exactly what the Orchestrator expects, on the first try, without needing to read existing code or ask clarifying questions. This is the synthesis responsibility: the Orchestrator thinks hard enough upfront to write complete specifications.

### Dispatch via MCP

Each specialist CLI runs as an MCP server (as established in Chapter 4). The Orchestrator dispatches tasks by calling tools on the appropriate specialist:

Orchestrator → Security CLI (MCP tools/call): sast_scan({"target": "src/auth/"}) Security CLI → Orchestrator (MCP result): {"findings": [...], "status": "complete"}

For tasks that require LLM reasoning (not just tool execution), the Orchestrator uses the AgentTool to spawn a specialist with a self-contained prompt:

```json
{
  "tool": "AgentTool",
  "input": {
    "agent": "coder",
    "prompt": "Create a PostgreSQL migration file at migrations/001_users.sql with columns: id (uuid, primary key), email (varchar 255, unique, not null), password_hash (varchar 255, not null), role (varchar 50, default 'user'), created_at (timestamptz, default now()), updated_at (timestamptz, default now()). Include an index on email."
  }
}

Notice: the prompt is entirely self-contained. It includes exact file paths, column types, constraints, and default values. The specialist does not need to read files or ask questions — it has everything needed to execute. This is the synthesis responsibility from Section 6.1: the Orchestrator thinks hard enough to write complete specifications.

Dispatch with Context from Previous Specialists

When a task depends on the output of a previous task, the Orchestrator includes that output in the dispatch. This is the inter-wave handoff — the critical moment where results from one specialist become inputs for another.

The Orchestrator does not simply forward the raw output from one specialist to the next. It curates the handoff: extracting the relevant parts, translating them into the terms the receiving specialist needs, and adding any context that the receiving specialist requires but the sending specialist did not produce.

{
  "tool": "AgentTool",
  "input": {
    "agent": "security",
    "prompt": "Audit the authentication module. The Coder CLI created these files:\n\n- src/auth/middleware.ts (JWT verification, RS256)\n- src/auth/rbac.ts (role-based access control)\n- src/models/user.ts (Prisma model with password_hash)\n\nFocus on: JWT algorithm confusion, token expiry, RBAC bypass, password storage.\nReport findings as JSON with severity, CWE, file, line, and remediation."
  }
}

The Orchestrator synthesizes the Coder's output into the Security CLI's input. Each specialist receives a complete, standalone brief.

Result Collection and Status Tracking

When specialists complete tasks, the Orchestrator collects results through the MCP response channel. But collection is not passive — the Orchestrator actively monitors specialist progress and handles several states:

pub enum SpecialistStatus {
    Running { progress: Option<String>, elapsed: Duration },
    Completed { result: TaskResult, tokens_used: u64 },
    Failed { error: String, partial_output: Option<String> },
    TimedOut { last_activity: Duration },
    Stalled { no_tool_calls_for: Duration },
}

Stall detection is particularly important. A specialist that has not made a tool call in 60 seconds is likely stuck — the LLM is generating text without taking action, or it has entered a reasoning loop. The Orchestrator can either nudge the specialist with a SendMessage ("Please proceed with the implementation — avoid over-analyzing") or kill it and retry.

Timeout management prevents runaway specialists from blocking the entire DAG. Each task gets a timeout proportional to its estimated complexity: 2 minutes for simple tasks, 5 minutes for moderate, 10 minutes for complex. If a specialist exceeds its timeout, the Orchestrator kills it and applies the failure strategy.

Failure Handling

When a specialist fails, the Orchestrator has three strategies, chosen based on the failure type:

pub enum FailureStrategy {
    /// Retry the same specialist with the same prompt (transient errors)
    Retry { max_attempts: u32, backoff_ms: u64 },

    /// Retry with a modified prompt (add error context)
    RetryWithContext { error_message: String },

    /// Cascade failure (block downstream tasks, report to human)
    CascadeBlock,
}

impl Orchestrator {
    fn handle_failure(&mut self, task: &Task, error: &str) -> FailureStrategy {
        // Transient errors: retry
        if error.contains("rate_limit") || error.contains("overloaded") {
            return FailureStrategy::Retry { max_attempts: 3, backoff_ms: 5000 };
        }

        // Tool errors: retry with context
        if error.contains("tool not found") || error.contains("permission denied") {
            return FailureStrategy::RetryWithContext {
                error_message: format!("Previous attempt failed: {}. Adjust approach.", error),
            };
        }

        // Fundamental failures: cascade
        FailureStrategy::CascadeBlock
    }
}

The cascade block strategy deserves special attention. When a task fails fatally (compilation error, permission denied, fundamental misunderstanding of the requirement), all tasks that depend on it are also blocked. The Orchestrator marks the entire downstream subgraph as Blocked and reports to the human with a clear explanation: which task failed, why, what downstream work is affected, and what options exist (retry with different approach, modify the requirement, or abandon the task).

The worst thing an Orchestrator can do after a failure is continue dispatching tasks that depend on the failed output. This produces cascade waste — specialists doing work based on inputs that no longer exist, generating artifacts that will need to be thrown away.

Dispatch Concurrency Limits

Not all tasks should run in parallel without limit. Each specialist CLIs consumes API tokens, and spawning 15 parallel specialists generates $50+ of API costs per minute. The Orchestrator uses a concurrency semaphore to limit parallel execution:

pub struct DispatchConfig {
    pub max_concurrent_specialists: usize,  // Default: 4
    pub max_concurrent_per_cli: usize,      // Default: 2
    pub timeout_simple: Duration,           // Default: 2 minutes
    pub timeout_moderate: Duration,         // Default: 5 minutes
    pub timeout_complex: Duration,          // Default: 10 minutes
    pub stall_threshold: Duration,          // Default: 60 seconds
}

The max_concurrent_per_cli limit prevents a single specialist type from monopolizing the dispatch slots. Without it, a DAG with 8 independent Coder tasks and 2 Security tasks might dispatch all 8 Coder tasks first, delaying Security work that could run in parallel.

6.5 Conflict Resolution

When specialists disagree, the Orchestrator must resolve the conflict. This is not an edge case — it is a routine occurrence in any multi-specialist system. The Security CLI will flag code that the Coder CLI just wrote. The Validator CLI will reject code that passes the Coder's own tests but fails the linter. The Planner CLI will recommend a redesign that invalidates work already completed by other specialists. These conflicts are features, not bugs — they are the mechanism by which multiple perspectives produce better results than a single perspective.

The challenge is resolving conflicts efficiently. An Orchestrator that sends every Security finding back to the Coder generates unnecessary round trips. An Orchestrator that ignores MEDIUM findings ships insecure code. The resolution protocol must be calibrated: strict enough to prevent real problems, lenient enough to avoid grinding the pipeline to a halt.

The Protocol

1. Security CLI reports: CRITICAL finding in src/auth/middleware.ts:23 (CWE-347: JWT algorithm confusion)
2. Orchestrator evaluates: CRITICAL severity = blocking. Cannot ship.
3. Orchestrator dispatches to Coder: "Fix the JWT algorithm confusion at src/auth/middleware.ts:23.
   The Security CLI reports: 'Algorithm is not validated on verification. An attacker can
   re-sign the token with HS256 using the public key as the HMAC secret.'
   Fix: Pin algorithm to RS256 in verify() call options."
4. Coder fixes and reports completion.
5. Orchestrator dispatches to Security: "Re-audit src/auth/middleware.ts. Specifically verify
   that the JWT algorithm confusion finding (CWE-347) at line 23 has been remediated."
6. Security CLI re-audits and reports: finding resolved.
7. Orchestrator proceeds.

Priority Rules

The resolution hierarchy codifies organizational judgment. In real engineering organizations, these rules exist as cultural norms ("security always wins"). In a multi-CLI system, they must be explicit because the Orchestrator cannot intuit cultural norms — it follows rules.

Scenario Resolution Rationale
Security says CRITICAL, Coder says "done" Security wins. Coder must address. CRITICAL findings represent exploitable vulnerabilities. Shipping them is unacceptable regardless of schedule.
Security says MEDIUM, Coder says "too complex to fix now" Orchestrator decides. May defer to backlog. MEDIUM findings are real but not immediately exploitable. The tradeoff between security and velocity is a judgment call.
Validator says tests fail, Coder says tests are wrong QA CLI breaks the tie — reviews both the code and the test. Neither the code author nor the test suite can be trusted when they disagree. An independent arbiter (QA) examines both.
Planner says "needs redesign", Coder already implemented Orchestrator evaluates scope. If redesign is >50% rework, escalate to human. Redesigning completed work is expensive. The Orchestrator should not make this call unilaterally — it affects the human's time and budget.
Two specialists produce conflicting code Orchestrator asks both to explain. Chooses based on the original requirement, not the specialist's preference. Specialists optimize for their domain. The original requirement is the tiebreaker.

Implementing Conflict Detection

The Orchestrator detects conflicts by comparing specialist outputs against expectations:

pub enum ConflictType {
    SecurityBlock {
        finding_severity: Severity,
        cwe: String,
        affected_file: String,
        remediation: String,
    },
    TestFailure {
        test_name: String,
        expected: String,
        actual: String,
        error_output: String,
    },
    DesignConflict {
        planner_recommendation: String,
        existing_implementation: String,
        rework_estimate: ReworkScope,
    },
    MergeConflict {
        file: String,
        specialist_a: String,
        specialist_b: String,
    },
}

pub enum ReworkScope {
    Minor,    // < 20% of completed work
    Moderate, // 20-50%
    Major,    // > 50% — escalate to human
}

impl Orchestrator {
    fn resolve_conflict(&mut self, conflict: ConflictType) -> Resolution {
        match conflict {
            ConflictType::SecurityBlock { finding_severity, .. } => {
                match finding_severity {
                    Severity::Critical | Severity::High => Resolution::MustFix,
                    Severity::Medium => Resolution::DeferToBacklog,
                    Severity::Low | Severity::Info => Resolution::Accept,
                }
            }
            ConflictType::TestFailure { .. } => Resolution::MustFix,
            ConflictType::DesignConflict { rework_estimate, .. } => {
                match rework_estimate {
                    ReworkScope::Minor | ReworkScope::Moderate => Resolution::MustFix,
                    ReworkScope::Major => Resolution::EscalateToHuman,
                }
            }
            ConflictType::MergeConflict { .. } => {
                // Two specialists modified the same file
                // This should not happen with proper worktree isolation
                Resolution::EscalateToHuman
            }
        }
    }
}

Merge conflicts are a special class. They should never occur if the Orchestrator properly isolates specialists using git worktrees (covered in Section 6.7). If they do occur, something is wrong with the isolation — the Orchestrator should not try to auto-resolve, but escalate to the human.

6.6 The Complete Orchestrator CLAUDE.md

Here is the full CLAUDE.md template for the Orchestrator CLI, incorporating every pattern from this chapter:

# OrchestratorCLI — Multi-Agent Project Manager

## Identity
You are the Orchestrator — the brain of a multi-CLI agent system.
You NEVER execute tasks directly. You decompose, delegate, monitor, and synthesize.
You operate on ABSTRACTIONS (task descriptions, dependencies, status) — never on
concrete artifacts (files, processes, test output).

## The Rule of Restriction
You have ONLY coordination tools: AgentTool, SendMessage, TaskStop, TaskCreate,
TaskList, TaskUpdate, SyntheticOutput.
You do NOT have: file_read, file_write, bash, grep, glob, or any filesystem tool.
If you find yourself wanting to read a file, you are doing the wrong job.
Delegate to the appropriate specialist instead.

## Four-Phase Workflow
For every task:

### Phase 1: ANALYZE
- Parse the human's prompt for intent, scope, and implicit requirements
- Identify which specialists are needed
- Estimate complexity (simple=1 specialist, moderate=2-4, complex=5+)

### Phase 2: DECOMPOSE
- Break into concrete subtasks with dependencies
- Each subtask must specify: what, which CLI, depends_on, priority
- Independent tasks should run in parallel (wave scheduling)
- No subtask should require >30 minutes of specialist work

### Phase 3: DISPATCH
- Send each subtask to the assigned specialist as a SELF-CONTAINED prompt
- Include: exact file paths, specific requirements, expected output format
- Never say "based on earlier work" — each dispatch is standalone
- Monitor progress. Handle failures (retry transient, cascade fatal).

### Phase 4: SYNTHESIZE
- Collect all specialist results
- Verify consistency (do the pieces fit together?)
- Resolve conflicts (Security vs Coder: Security wins for CRITICAL/HIGH)
- Present the final result to the human

## Available Specialists
- **coder**: Writes production code. Test-first. Match existing patterns.
- **security**: Read-only security audit. Structured JSON findings with CWE.
- **validator**: Runs lint/typecheck/build/test. Reports SHIP or BLOCK.
- **qa**: Adversarial testing. Finds edge cases and failure modes.
- **researcher**: Multi-source information gathering. Cited facts only.
- **planner**: Architecture and task decomposition. Outputs task DAGs.
- **documenter**: Documentation generation. Reads code, writes docs.

## Conflict Resolution
1. CRITICAL/HIGH security findings ALWAYS block. Coder must fix before proceeding.
2. Test failures ALWAYS block. Coder must fix before proceeding.
3. MEDIUM security findings: defer to backlog if Coder explains the tradeoff.
4. When specialists disagree on approach: return to the original human requirement.
5. When rework exceeds 50% of completed work: escalate to human for decision.

## Communication
- Report progress at natural milestones (Phase transitions, specialist completions)
- Show the task graph to the human so they can see what is happening
- When blocked (all specialists failed, or conflict cannot be resolved): explain
  what happened, what was tried, and ask the human for direction

6.7 Orchestration Patterns from Existing Projects

The reader's existing projects provide proven orchestration patterns that should be incorporated. These are not theoretical designs — they are battle-tested implementations that have coordinated real multi-agent work.

From DApp/SwarmForge: The 6-Phase Lifecycle

DApp's orchestrator.js implements a 6-phase lifecycle manager that has processed hundreds of multi-agent tasks:

  1. Receive — Parse prompt, validate input
  2. Analyze — Intent classification (from intent-analyzer.js)
  3. Plan — Task DAG generation (from task-dag.js)
  4. Execute — Parallel dispatch with semaphore-controlled concurrency (from semaphore.js)
  5. Synthesize — Merge results, resolve conflicts
  6. Deliver — Present final output, persist artifacts

The key innovation in DApp is the semaphore: it limits parallel execution to N concurrent workers (configurable), preventing resource exhaustion when the DAG has many independent tasks. Without the semaphore, a DAG with 20 independent tasks would spawn 20 concurrent API calls, burning $3+ per second in API costs and likely hitting rate limits. The semaphore throttles to 4-6 concurrent workers — enough for meaningful parallelism, sustainable for API budgets.

// Adapted from DApp's semaphore.js pattern
pub struct ConcurrencySemaphore {
    max_concurrent: usize,
    active: Arc<AtomicUsize>,
}

impl ConcurrencySemaphore {
    pub async fn acquire(&self) {
        loop {
            let current = self.active.load(Ordering::SeqCst);
            if current < self.max_concurrent {
                if self.active.compare_exchange(
                    current, current + 1,
                    Ordering::SeqCst, Ordering::SeqCst
                ).is_ok() {
                    return;
                }
            }
            tokio::time::sleep(Duration::from_millis(100)).await;
        }
    }

    pub fn release(&self) {
        self.active.fetch_sub(1, Ordering::SeqCst);
    }
}

From claudecode_team: The Debate Pattern

The claudecode_team project implements a debate-and-synthesis pattern where multiple agents analyze the same problem independently, then debate their findings before a final synthesis. The supervisor's 4-phase workflow:

  1. Plan — Decompose the question
  2. Spawn — Multiple agents investigate independently
  3. Debate — Agents see each other's findings and argue (2 rounds)
  4. Synthesize — A synthesis agent merges the strongest arguments

This pattern is valuable for security audits: have multiple security agents audit the same code independently, then debate their findings. Vulnerabilities found by multiple agents are higher confidence. Findings disputed in debate are re-examined. The synthesis produces a consensus report that is more thorough than any single agent's analysis.

The debate pattern applies beyond security. For architectural decisions, have a Planner agent and a Coder agent independently assess the approach — the Planner optimizes for maintainability while the Coder optimizes for implementability. Their debate surfaces tradeoffs that neither would identify alone. For documentation, have a Documentation agent write docs and a QA agent review them — the QA agent finds gaps by trying to follow the docs as a novice would.

The cost of debate is 2-3x the cost of a single-agent analysis (multiple agents + debate rounds + synthesis). Use it for high-stakes decisions where correctness matters more than speed: security audits, architecture reviews, and production deployment decisions. Do not use it for routine tasks like writing CRUD endpoints.

From DApp: Git Worktree Isolation

DApp's worktree.js provides per-worker git worktree isolation. Each worker operates in its own copy of the repository, preventing file conflicts when multiple specialists modify code simultaneously. When all workers complete, the Orchestrator merges the worktrees:

# Per-worker isolation
git worktree add .workers/coder-001 -b worker/coder-001
git worktree add .workers/security-001 -b worker/security-001

# After completion: merge
git checkout main
git merge worker/coder-001 --no-ff -m "Merge coder results"
git merge worker/security-001 --no-ff -m "Merge security results"

This pattern eliminates the most dangerous failure mode in parallel multi-CLI execution: two specialists editing the same file simultaneously.

The worktree approach has overhead — creating a worktree copies the working tree (though it shares the .git directory), and merging at the end can produce conflicts. For small projects this overhead is negligible. For large monorepos with thousands of files, worktree creation can take seconds. The Orchestrator should create worktrees lazily — only when dispatching a task that modifies files, not for read-only tasks like security audits.

From MultiCLIs: The Shared Blackboard Pattern

The MultiCLIs project introduces a shared blackboard — a structured file that all specialists can read and the Orchestrator can write. The blackboard contains the current system state: which tasks are running, what has been completed, what decisions have been made, and what conflicts exist.

# Blackboard — Session 2024-12-15T10:30:00Z

## Completed Tasks
- T1: Schema migration created (migrations/001_users.sql) ✓
- T2: Environment config (.env.template, config.ts) ✓

## Running Tasks
- T3: JWT auth implementation (coder-001) — 45s elapsed
- T4: API routes (coder-002) — 30s elapsed

## Decisions Made
- ORM: Prisma (chosen by Planner based on existing package.json)
- Auth: RS256 JWT (chosen for key rotation support)
- Database: PostgreSQL 15 (existing project dependency)

## Pending Conflicts
- None

The blackboard serves two purposes: it gives the Orchestrator a persistent view of system state (surviving context window compaction), and it gives specialists read-only access to decisions that affect their work. A specialist writing API routes can check the blackboard to confirm that Prisma was chosen as the ORM, without needing the Orchestrator to include that in every dispatch prompt.

6.8 Testing the Orchestrator

Testing the Orchestrator is fundamentally different from testing a specialist CLI. A specialist's correctness depends on whether it produced the right output for a given input — write a test, check the result. The Orchestrator's correctness depends on whether it made the right decisions about decomposition, delegation, and conflict resolution — which requires testing behavior, not output.

Unit Tests: Intent Classification

Test that the intent analysis engine correctly classifies prompts:

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_build_intent() {
        let intent = classify_intent("Build a REST API with JWT auth and PostgreSQL");
        assert!(matches!(intent, Intent::Build { .. }));
        if let Intent::Build { components } = intent {
            assert!(components.iter().any(|c| c.name.contains("REST")));
            assert!(components.iter().any(|c| c.name.contains("JWT")));
            assert!(components.iter().any(|c| c.name.contains("PostgreSQL")));
        }
    }

    #[test]
    fn test_fix_intent_with_error() {
        let intent = classify_intent("fix the login bug: TypeError: Cannot read properties of undefined");
        assert!(matches!(intent, Intent::Fix { .. }));
        if let Intent::Fix { error, .. } = intent {
            assert!(error.unwrap().contains("TypeError"));
        }
    }

    #[test]
    fn test_compound_prompt() {
        let intents = classify_intents("fix the auth bug and then add rate limiting");
        assert_eq!(intents.len(), 2);
        assert!(matches!(intents[0], Intent::Fix { .. }));
        assert!(matches!(intents[1], Intent::Build { .. }));
    }
}

Integration Tests: Dispatch Scenarios

Before deploying the Orchestrator to production, verify it handles the core scenarios. These tests use mock specialists that simulate real behavior without making API calls:

Test 1: Simple Delegation

Prompt: "Run a security audit on src/auth/"
Expected: Orchestrator delegates to Security CLI. Does NOT attempt to read files itself.

Test 2: Multi-Specialist Decomposition

Prompt: "Build a REST API with JWT auth, PostgreSQL, and test coverage"
Expected: 8+ subtasks across Coder, Security, Validator, and possibly Planner.
Independent tasks dispatched in parallel waves.

Test 3: Conflict Resolution

Prompt: "Build a login endpoint"
After Coder completes → Security finds SQL injection → Coder must fix → Security re-audits.
Expected: The cycle completes. Final result has no CRITICAL findings.

Test 4: Failure Cascade

Scenario: Coder CLI crashes mid-task.
Expected: Orchestrator detects failure. Downstream tasks blocked. Human notified.

Test 5: Restriction Enforcement

Prompt: "Read server.js and tell me what it does"
Expected: Orchestrator delegates to a specialist (Coder or Researcher).
Does NOT attempt to use file_read (which it does not have).

Test 6: DAG Cycle Detection

Input: A task graph where T3 depends on T5, and T5 depends on T3.
Expected: validate() returns Err(DagError::CycleDetected).
No tasks are dispatched.

Test 7: Stall Detection

Scenario: Coder CLI generates 2000 tokens of text without a single tool call.
Expected: Orchestrator detects stall after 60 seconds.
Sends nudge via SendMessage. If still stalled after 30 more seconds, kills and retries.

If all seven tests pass, your Orchestrator correctly decomposes, delegates, resolves conflicts, handles failures, detects edge cases, and respects its tool restrictions.

Performance Benchmarks

Establish baseline metrics for your Orchestrator to detect regressions:

Metric Target How to Measure
Intent classification latency <5ms (heuristic), <3s (LLM) Instrument classify_intent()
DAG construction time <100ms for 20 tasks Instrument TaskDAG::new()
Dispatch overhead per task <50ms Time between task Ready and specialist spawn
Conflict resolution round-trip <30s total Time from Security finding to Coder fix to re-audit
Total orchestration overhead <15% of total execution time Orchestrator time / (Orchestrator + all specialist time)

The last metric is the most important. If the Orchestrator spends 40% of wall time on coordination overhead, the multi-CLI system is slower than a single agent. Target 10-15% overhead — the Orchestrator should be the lightest process in the system.

6.9 The Orchestrator as a Non-Engineering Coordinator

Everything in this chapter has used engineering examples — REST APIs, security audits, test suites. But the Orchestrator pattern applies far beyond software engineering.

Consider a legal research Orchestrator that receives "Analyze this contract for liability risks." It dispatches to a Contract Analyst CLI (reads clauses, identifies obligations), a Case Law Researcher CLI (finds relevant precedents), a Jurisdiction CLI (checks state-specific regulations), and a Plain Language CLI (translates findings for non-lawyers). The Orchestrator manages the same concerns: task decomposition, dependency management, conflict resolution (the Case Law CLI finds a precedent that contradicts the Jurisdiction CLI's analysis), and synthesis.

Consider a medical research Orchestrator that receives "What treatment options exist for this patient profile?" It dispatches to a Literature Review CLI, a Drug Interaction CLI, a Clinical Guidelines CLI, and a Patient History CLI. Conflict resolution becomes life-critical: when the Drug Interaction CLI flags a contraindication that the Clinical Guidelines CLI recommends, the Orchestrator must always defer to the safety-focused specialist.

The architecture is the same. The CLAUDE.md templates change. The resolution priorities change. The domain knowledge changes. But the Orchestrator's core behavior — decompose, delegate, monitor, resolve, synthesize — remains constant. This is the power of the pattern: once you build one Orchestrator, you can build any Orchestrator by changing its specialist roster and its resolution rules.
