Chapter 17: One Prompt to Rule Them All

There is a moment every engineer remembers. It is the moment they realize that the project they just described in a single sentence — "build a secure API with auth, tests, and docs" — would take their team two days of standup meetings, Jira grooming, Slack threads, and pull request reviews before anything ships. Twenty words in a Slack message. Forty hours of calendar time.

In a single Claude Code session, you could type that prompt and have something working in forty minutes. But by minute twenty, the context window is groaning under the weight of 180,000 accumulated tokens. The model forgot it used plaintext password comparison fifteen turns ago. The tests cover happy paths but miss the authorization edge cases that were compacted away at turn thirty. The documentation references endpoints that were renamed during a mid-session refactor. You get a B-minus deliverable and spend another hour cleaning up the contradictions.

This chapter is about the system that turns that prompt into an A-plus deliverable in eleven minutes. Not by working harder. By working as a team.

Everything you have built across the preceding sixteen chapters converges here. The specialized CLIs from Part II. The communication protocols from Chapter 15. The hooks and skills from Chapter 16. The shared memory from Chapter 14. This is the chapter where you type one prompt and watch a team of AI agents plan, debate, build, test, audit, document, and deliver a production-ready system while you refill your coffee.

17.1 The Prompt That Starts Everything

A terminal. A cursor. A single line:

$ forge "Build me a secure REST API for a bookstore with auth, CRUD, tests, and docs"

That is the entire user interface. One prompt. No configuration flags, no YAML files, no agent selection menus. Twenty-six words into a command line.

Behind that command line, six systems activate simultaneously. To understand what happens next, we need to look at the orchestration engine that receives this prompt — and what it does in the first 400 milliseconds before any LLM call fires.

17.2 Phase 0: Intent Analysis (0–400ms)

The first thing that happens is not an API call. It is pattern matching. The intent analyzer — a lightweight keyword scanner that uses no LLM tokens — parses the prompt against a set of role-detection patterns:

pub struct IntentAnalyzer {
    role_patterns: Vec<(CliRole, Vec<Regex>)>,
    project_patterns: Vec<(ProjectType, Vec<Regex>)>,
}

#[derive(Debug, Clone, PartialEq)]
pub enum CliRole {
    Orchestrator,
    Architect,
    Coder,
    Security,
    Validator,
    QA,
    Documentation,
    Research,
    Custom(String),
}

#[derive(Debug, Clone)]
pub enum ProjectType {
    Api,
    Frontend,
    FullStack,
    Cli,
    Library,
    Infrastructure,
    Unknown,
}

#[derive(Debug)]
pub struct IntentResult {
    pub roles: Vec<CliRole>,
    pub project_type: ProjectType,
    pub suggested_workers: usize,
    pub priority_hints: Vec<String>,  // "security-first", "speed-over-quality", etc.
}

impl IntentAnalyzer {
    pub fn analyze(&self, prompt: &str) -> IntentResult {
        let lower = prompt.to_lowercase();

        let mut roles = vec![CliRole::Orchestrator]; // Always present

        for (role, patterns) in &self.role_patterns {
            if patterns.iter().any(|p| p.is_match(&lower)) {
                roles.push(role.clone());
            }
        }

        // Rule: 2+ non-orchestrator roles → always include QA
        let non_orch = roles.iter().filter(|r| **r != CliRole::Orchestrator).count();
        if non_orch >= 2 && !roles.contains(&CliRole::QA) {
            roles.push(CliRole::QA);
        }

        // Detect project type
        let project_type = self.detect_project_type(&lower);

        // Extract priority hints from qualifiers
        let priority_hints = self.extract_priority_hints(&lower);

        let suggested_workers = (roles.len() - 1).min(8); // Exclude orchestrator

        IntentResult {
            roles,
            project_type,
            suggested_workers,
            priority_hints,
        }
    }

    fn extract_priority_hints(&self, prompt: &str) -> Vec<String> {
        let mut hints = Vec::new();
        if prompt.contains("secure") || prompt.contains("security") {
            hints.push("security-first".into());
        }
        if prompt.contains("fast") || prompt.contains("quick") || prompt.contains("mvp") {
            hints.push("speed-over-quality".into());
        }
        if prompt.contains("production") || prompt.contains("enterprise") {
            hints.push("production-grade".into());
        }
        hints
    }
}

The prompt "Build me a secure REST API for a bookstore with auth, CRUD, tests, and docs" triggers four patterns:

  • Coder: "API", "REST", "auth", "CRUD"
  • Security: "secure"
  • QA: "tests"
  • Documentation: "docs"

The intent analyzer also extracts the priority hint "security-first" from the word "secure." This hint propagates through the entire pipeline — during debate, security findings carry higher weight than convenience arguments.

IntentResult {
    roles: [Orchestrator, Coder, Security, QA, Documentation],
    project_type: Api,
    suggested_workers: 4,
    priority_hints: ["security-first"],
}

Notice what is missing: no Research CLI (the user did not ask for background investigation), no Planner CLI (the Orchestrator handles planning for small projects). The system does not waste resources on roles the prompt does not need.

This entire analysis takes under 50 milliseconds. No API call. No token consumption. Pure pattern matching in Rust — zero allocation for the common case because the regex set is compiled once at startup.

17.3 The Six-Phase Orchestration Model

Every forge project follows the same lifecycle, regardless of whether the user types five words or five paragraphs:

INTENT → PLAN → SPAWN → EXECUTE → DEBATE → SYNTHESIZE

Each phase has a clear entry condition, a defined output artifact, and a failure mode with recovery:

Phase Time Model Output Failure Recovery
1. INTENT 0–0.4s None roles[], projectType N/A (pure regex)
2. PLAN 0.4s–45s Opus task_dag.toml Retry 3x with error feedback
3. SPAWN 45s–60s None worker processes, git worktrees Retry process spawn
4. EXECUTE 60s–480s Sonnet/Haiku code, tests, configs, docs Stall detection + retry
5. DEBATE 480s–570s Sonnet conflict resolutions, refinements Max 2 rounds, then escalate
6. SYNTHESIZE 570s–660s Opus final delivery, quality gate Refinement loop (max 2)

Total wall-clock time: approximately eleven minutes. Let us walk through each phase using our bookstore API prompt.

Phase 2: Plan (0.4s–45s)

The Orchestrator CLI activates first. Running on Claude Opus — the most capable model, reserved for planning and synthesis — it receives the user's prompt along with any cross-session memory from previous runs. Its job: decompose the prompt into a task DAG.

The Orchestrator's CLAUDE.md restricts it to exactly six tools, as we saw in Chapter 6:

AgentTool, TaskCreate, TaskUpdate, SendMessage, SyntheticOutput, TeamCreate

It cannot read files, write code, or execute bash commands. Its only job is to orchestrate. This is the separation of concerns that makes the forge reliable — the brain does not touch the keyboard.

The Orchestrator produces a task DAG in TOML format:

The Task DAG Engine

The DAG is not just a list of tasks — it is a dependency-aware scheduler that validates structural integrity:

pub struct TaskDag {
    tasks: HashMap<String, DagTask>,
    execution_order: Vec<Vec<String>>,  // Waves
}

#[derive(Debug, Clone)]
pub struct DagTask {
    pub id: String,
    pub role: CliRole,
    pub model: ModelTier,
    pub description: String,
    pub depends_on: Vec<String>,
    pub status: TaskStatus,
    pub assigned_to: Option<String>,   // CLI instance ID
}

#[derive(Debug, Clone, PartialEq)]
pub enum TaskStatus {
    Pending,
    Ready,
    Active,
    Complete,
    Failed,
    Stalled,
}

impl TaskDag {
    pub fn from_toml(content: &str) -> Result<Self, DagError> {
        let raw: RawDag = toml::from_str(content)?;
        let mut tasks = HashMap::new();

        for task in raw.tasks {
            // Validate all dependencies reference existing tasks
            for dep in &task.depends_on {
                if !raw.tasks.iter().any(|t| &t.id == dep) {
                    return Err(DagError::MissingDependency {
                        task: task.id.clone(),
                        missing: dep.clone(),
                    });
                }
            }
            tasks.insert(task.id.clone(), task.into());
        }

        let mut dag = Self { tasks, execution_order: Vec::new() };
        dag.detect_cycles()?;
        dag.compute_waves();
        Ok(dag)
    }

    fn detect_cycles(&self) -> Result<(), DagError> {
        let mut visited = HashSet::new();
        let mut stack = HashSet::new();

        for id in self.tasks.keys() {
            self.dfs_cycle(id, &mut visited, &mut stack)?;
        }
        Ok(())
    }

    fn dfs_cycle(
        &self,
        id: &str,
        visited: &mut HashSet<String>,
        stack: &mut HashSet<String>,
    ) -> Result<(), DagError> {
        if stack.contains(id) {
            return Err(DagError::CycleDetected(id.to_string()));
        }
        if visited.contains(id) {
            return Ok(());
        }

        stack.insert(id.to_string());

        if let Some(task) = self.tasks.get(id) {
            for dep in &task.depends_on {
                self.dfs_cycle(dep, visited, stack)?;
            }
        }

        stack.remove(id);
        visited.insert(id.to_string());
        Ok(())
    }

    fn compute_waves(&mut self) {
        let mut remaining: HashSet<String> = self.tasks.keys().cloned().collect();
        let mut completed: HashSet<String> = HashSet::new();

        while !remaining.is_empty() {
            let wave: Vec<String> = remaining.iter()
                .filter(|id| {
                    self.tasks[*id].depends_on.iter()
                        .all(|dep| completed.contains(dep))
                })
                .cloned()
                .collect();

            if wave.is_empty() {
                // Should not happen if cycle detection passed
                break;
            }

            for id in &wave {
                remaining.remove(id);
                completed.insert(id.clone());
            }

            self.execution_order.push(wave);
        }
    }

    pub fn get_ready_tasks(&self) -> Vec<&DagTask> {
        self.tasks.values()
            .filter(|t| t.status == TaskStatus::Ready)
            .collect()
    }

    pub fn mark_complete(&mut self, task_id: &str) {
        if let Some(task) = self.tasks.get_mut(task_id) {
            task.status = TaskStatus::Complete;
        }

        // Promote newly unblocked tasks to Ready
        let newly_ready: Vec<String> = self.tasks.iter()
            .filter(|(_, t)| t.status == TaskStatus::Pending)
            .filter(|(_, t)| {
                t.depends_on.iter().all(|dep| {
                    self.tasks.get(dep)
                        .map(|d| d.status == TaskStatus::Complete)
                        .unwrap_or(false)
                })
            })
            .map(|(id, _)| id.clone())
            .collect();

        for id in newly_ready {
            self.tasks.get_mut(&id).unwrap().status = TaskStatus::Ready;
        }
    }
}

If the Orchestrator's plan contains a circular dependency — say, "QA depends on coder, but coder depends on QA" — the DAG constructor returns Err(DagError::CycleDetected) immediately. The Orchestrator catches this, logs the error, and re-prompts with explicit feedback: "Your plan contains a cycle: coder-1 → qa-1 → coder-1. Remove the cycle and try again." Up to three retries before the project enters error state.

Phase 3: Spawn (45s–60s)

With the DAG validated, the Orchestrator spawns worker processes. Each worker gets its own git worktree — an isolated branch of the repository where it can make changes without conflicting with other workers:

pub struct WorktreeManager {
    forge_dir: PathBuf,
}

impl WorktreeManager {
    pub fn create_worktree(
        &self,
        worker_id: &str,
        base_branch: &str,
    ) -> Result<WorktreeInfo, ForgeError> {
        let branch = format!("worker/{}", worker_id);
        let worktree_path = self.forge_dir.join("worktrees").join(worker_id);

        std::process::Command::new("git")
            .args(["worktree", "add"])
            .arg(&worktree_path)
            .args(["-b", &branch])
            .arg(base_branch)
            .output()?;

        Ok(WorktreeInfo {
            path: worktree_path,
            branch,
            worker_id: worker_id.to_string(),
        })
    }

    pub fn merge_worktree(
        &self,
        worker_id: &str,
        target_branch: &str,
    ) -> Result<(), ForgeError> {
        let branch = format!("worker/{}", worker_id);
        std::process::Command::new("git")
            .args(["checkout", target_branch])
            .output()?;

        std::process::Command::new("git")
            .args(["merge", "--no-ff", &branch, "-m"])
            .arg(format!("Merge {} into {}", worker_id, target_branch))
            .output()?;

        Ok(())
    }

    pub fn cleanup(&self, worker_id: &str) -> Result<(), ForgeError> {
        let worktree_path = self.forge_dir.join("worktrees").join(worker_id);
        std::process::Command::new("git")
            .args(["worktree", "remove", "--force"])
            .arg(&worktree_path)
            .output()?;
        Ok(())
    }
}

Each CLI process runs with its own CLAUDE.md (defining its role), its own context window, and its own worktree. They share only the .forge/artifacts/ directory for inter-agent communication.

Phase 4: Execute (60s–480s)

This is where the actual work happens. The DAG schedules tasks in waves:

Wave 2: [coder-1, security-1]       → src/*, security-report.md

         Duration: ~240s (coder) / ~120s (security).

Wave 3: [qa-1, docs-1]              → tests/*, docs/*
         Both depend on coder-1.
         Run in parallel.

The Semaphore: Controlled Parallelism

The system does not blindly spawn every ready task. A semaphore limits simultaneous active workers:

pub struct ForgeSemaphore {
    permits: Arc<Mutex<usize>>,
    notify: Arc<tokio::sync::Notify>,
    limit: usize,
}

impl ForgeSemaphore {
    pub fn new(limit: usize) -> Self {
        Self {
            permits: Arc::new(Mutex::new(limit)),
            notify: Arc::new(tokio::sync::Notify::new()),
            limit,
        }
    }

    pub async fn acquire(&self) {
        loop {
            {
                let mut permits = self.permits.lock().unwrap();
                if *permits > 0 {
                    *permits -= 1;
                    return;
                }
            }
            self.notify.notified().await;
        }
    }

    pub fn release(&self) {
        let mut permits = self.permits.lock().unwrap();
        *permits = (*permits + 1).min(self.limit);
        self.notify.notify_one();
    }
}

For our bookstore project, the semaphore limit is 4 (matching suggested_workers from intent analysis). If Wave 2 tried to launch five workers simultaneously, the fifth would queue until one completes.

Shared Filesystem Communication

Workers do not communicate via message passing during the execution phase. They communicate through the shared filesystem:

.forge/
  artifacts/
    user-prompt.md          <- original prompt
    architecture.md         <- architect's output
    task_dag.toml           <- the validated DAG
    security-report.md      <- security findings
  tasks/
    architect-1.json        <- task status + timing
    coder-1.json
    security-1.json
    qa-1.json
    docs-1.json
  messages/                 <- inter-agent messages (debate phase)
  memory.toml              <- shared memory across workers
  worktrees/
    architect-1/            <- git worktree (isolated branch)
    coder-1/
    security-1/
    qa-1/
    docs-1/

This is deliberate. Filesystem-based communication has three advantages over message passing: (1) it is inherently persistent — if a worker crashes, its partial output survives on disk; (2) it is debuggable — you can cat .forge/artifacts/architecture.md at any time to see what the architect decided; and (3) it requires no coordination protocol — workers write when ready, readers poll when needed.

The artifacts directory is the shared knowledge base. When the architect writes architecture.md, the Coder CLI reads it to understand framework choices. When the Coder writes source files, the Security CLI reads them to audit. The QA CLI reads both the architecture and the implementation to write tests that match the actual code paths.

Each task status file tracks timing and metadata:

# .forge/tasks/coder-1.toml

status = "active"

Worker Health Monitor

Each worker has a stall timeout. If a CLI produces no output for 120 seconds, it is considered stalled:

pub struct HealthMonitor {
    workers: Arc<DashMap<String, WorkerHealth>>,
    stall_timeout: Duration,
    check_interval: Duration,
}

struct WorkerHealth {
    last_output: Instant,
    status: TaskStatus,
    retries: u32,
    max_retries: u32,
}

impl HealthMonitor {
    pub async fn run(&self, orchestrator: Arc<Orchestrator>) {
        let mut interval = tokio::time::interval(self.check_interval);

        loop {
            interval.tick().await;

            for entry in self.workers.iter() {
                let (id, health) = entry.pair();
                if health.status != TaskStatus::Active {
                    continue;
                }

                let silent_duration = Instant::now() - health.last_output;
                if silent_duration > self.stall_timeout {
                    if health.retries < health.max_retries {
                        tracing::warn!("Worker {} stalled — retrying", id);
                        orchestrator.retry_worker(id).await;
                    } else {
                        tracing::error!("Worker {} failed after max retries", id);
                        orchestrator.fail_worker(id).await;
                    }
                }
            }
        }
    }
}

A stalled worker gets one retry. If the retry also stalls, the worker is marked Failed, and the Orchestrator evaluates whether the task is critical. A failed Documentation CLI is non-critical — the system can deliver without docs. A failed Coder CLI is critical — without endpoints, QA has nothing to test. Critical failures trigger cascading failure marking through the DAG.

17.4 The Debate Phase: When CLIs Disagree

This is the phase that separates a multi-agent system from a multi-agent team. Without debate, you have parallel execution — five agents running independently, producing five artifacts that may contradict each other. With debate, you have adversarial collaboration — agents reading each other's work, identifying conflicts, and resolving them.

The debate is where the system earns its keep.

The Bookstore Conflict

During Wave 2, the Coder CLI implements user registration. Under time pressure and optimizing for speed, it produces this:

// Coder CLI's implementation (first pass)
async fn register(
    State(pool): State<PgPool>,
    Json(body): Json<RegisterRequest>,
) -> Result<Json<AuthResponse>, ApiError> {
    let user = sqlx::query_as!(User,
        "INSERT INTO users (email, password, name) VALUES ($1, $2, $3) RETURNING *",
        body.email,
        body.password,  // Stored as plaintext
        body.name,
    )
    .fetch_one(&pool)
    .await?;

    let token = create_jwt(&user)?;
    Ok(Json(AuthResponse { user: user.into(), token }))
}

The Coder CLI was focused on getting the CRUD flow working. It stored the password as plaintext. It is the kind of mistake an LLM optimizing for task completion absolutely will make — because "store the password" is a completed subtask and the agent is moving to the next one.

Meanwhile, the Security CLI reads the Coder's output from the shared artifacts directory and its findings light up:

# Security Audit — CRITICAL Findings

## CRITICAL: Plaintext Password Storage (CWE-256)
- **File:** src/routes/auth.rs:14
- **Issue:** User passwords stored without hashing
- **Impact:** Complete credential exposure on any database breach
- **Remediation:** Use argon2 with default params (memory 19456, iterations 2, parallelism 1)
- **OWASP:** A02:2021 — Cryptographic Failures

## HIGH: Missing Rate Limiting on Auth Endpoints
- **File:** src/routes/auth.rs (all routes)
- **Issue:** No rate limiting on /auth/login or /auth/register
- **Impact:** Brute force and credential stuffing attacks
- **Remediation:** tower-governor middleware, 5 attempts/15min on auth routes

## MEDIUM: JWT Secret from Environment Without Validation
- **File:** src/config.rs:8
- **Issue:** JWT_SECRET read from env without checking existence or strength
- **Remediation:** Validate length >= 32 chars at startup, fail fast

Both CLIs write their findings to .forge/artifacts/. The stage is set for debate.

How Debate Works

The Orchestrator manages debate by sending messages (via the SendMessage tool from Chapter 15) that include cross-references to other CLIs' findings:

pub struct DebateRound {
    pub round: u32,
    pub participants: Vec<String>,
    pub topic_injections: HashMap<String, Vec<Finding>>,
}

impl Orchestrator {
    pub async fn run_debate(&self, rounds: u32) -> Vec<DebateOutcome> {
        let mut outcomes = Vec::new();

        for round in 1..=rounds {
            // Collect all findings from all CLIs
            let all_findings = self.collect_findings().await;

            // For each CLI, inject OTHER CLIs' findings as context
            let mut debate_futures = Vec::new();

            for cli in &self.active_clis {
                let other_findings: Vec<Finding> = all_findings.iter()
                    .filter(|f| f.author != cli.id)
                    .cloned()
                    .collect();

                let prompt = format!(
                    "DEBATE ROUND {round}/{rounds}: Review these findings from \
                     other team members and update your analysis.\n\n{}",
                    format_findings(&other_findings)
                );

                debate_futures.push(cli.send_message(prompt));
            }

            let round_results = futures::future::join_all(debate_futures).await;
            outcomes.extend(round_results.into_iter().flatten());
        }

        outcomes
    }
}

In Round 1, the Coder CLI reads the Security CLI's findings and responds:

# Coder Response to Security Findings (Round 1)

## RE: Plaintext Password Storage — ACCEPTED
Security is correct. Revised implementation:

```rust
use argon2::{self, Config, Variant};

async fn register(
    State(pool): State<PgPool>,
    Json(body): Json<RegisterRequest>,
) -> Result<Json<AuthResponse>, ApiError> {
    // Validate password strength
    if body.password.len() < 8 {
        return Err(ApiError::Validation("Password must be at least 8 chars".into()));
    }

    let salt = rand::random::<[u8; 32]>();
    let config = Config {
        variant: Variant::Argon2id,
        ..Config::default()
    };
    let hashed = argon2::hash_encoded(body.password.as_bytes(), &salt, &config)?;

    let user = sqlx::query_as!(User,
        "INSERT INTO users (email, password_hash, name) VALUES ($1, $2, $3) RETURNING *",
        body.email,
        hashed,
        body.name,
    )
    .fetch_one(&pool)
    .await?;

    let token = create_jwt(&user)?;
    Ok(Json(AuthResponse { user: user.into(), token }))
}

RE: Rate Limiting — ACCEPTED

Adding tower-governor to auth routes.

RE: JWT Secret Validation — ACCEPTED

Added startup validation in main.rs.

The Coder accepted all three findings. This is the common case — security findings are usually correct, and the revised implementation improves the code.

### When CLIs Push Back

In Round 2, the Security CLI reviews the revised implementation and flags a new concern:

```markdown
## NEW FINDING: JWT Expiry Too Long
- 24h JWT expiry is excessive for a bookstore API
- Recommendation: 15-minute access token + 7-day refresh token

The Coder CLI disagrees:

## RE: JWT Expiry — PARTIALLY REJECTED
- 15-minute access tokens create poor UX for API consumers
  who hold tokens client-side
- Compromise: 1-hour access token + 7-day refresh token
- Adding /auth/refresh endpoint for token rotation

This is adversarial collaboration. Neither agent is wrong in isolation — Security optimizes for minimal exposure window, Coder optimizes for usability. The compromise of one-hour access tokens with refresh rotation is the kind of nuanced decision that emerges from debate. It is the same decision a senior engineer and a security engineer would reach in a conference room, except it happened in thirty seconds at zero salary cost.

The Orchestrator does not intervene in debate content. It manages the process — how many rounds, when to stop — but the CLIs resolve technical conflicts themselves. The priority_hints from intent analysis matter here: because the prompt included "secure," the Orchestrator set a "security-first" priority. If the Coder and Security CLI had an unresolvable disagreement after two rounds, the Orchestrator would weight the Security CLI's position higher.

The Resolution Protocol

When debate rounds end, the Orchestrator aggregates outcomes using a structured resolution:

#[derive(Debug)]
pub enum Resolution {
    Accepted,                    // Finding accepted, code updated
    PartiallyAccepted(String),   // Compromise reached
    Rejected(String),            // Finding rejected with rationale
    Escalated,                   // Unresolved — Orchestrator decides
}

pub struct DebateAggregator {
    priority_hints: Vec<String>,
}

impl DebateAggregator {
    pub fn aggregate(&self, outcomes: &[DebateOutcome]) -> Vec<ResolvedFinding> {
        let mut resolved = Vec::new();

        for outcome in outcomes {
            let resolution = match (&outcome.response_type, &outcome.round) {
                // Accepted in round 1 — no conflict
                (ResponseType::Accepted, 1) => Resolution::Accepted,
                // Partial acceptance — compromise reached
                (ResponseType::Partial(compromise), _) => {
                    Resolution::PartiallyAccepted(compromise.clone())
                }
                // Still rejected after round 2 — check priority hints
                (ResponseType::Rejected(reason), 2) => {
                    if self.should_escalate(&outcome.author_role, &outcome.target_role) {
                        Resolution::Escalated
                    } else {
                        Resolution::Rejected(reason.clone())
                    }
                }
                _ => Resolution::Accepted,
            };

            resolved.push(ResolvedFinding {
                finding: outcome.finding.clone(),
                resolution,
                author: outcome.author_role.clone(),
                target: outcome.target_role.clone(),
            });
        }

        resolved
    }

    fn should_escalate(&self, author: &CliRole, _target: &CliRole) -> bool {
        // Security-first: security findings always escalate
        if self.priority_hints.contains(&"security-first".to_string())
            && *author == CliRole::Security
        {
            return true;
        }

        // Default: unresolved after 2 rounds escalates to Orchestrator
        true
    }
}

Escalated findings go to the Orchestrator, which runs on Opus — the most capable model. The Orchestrator sees both positions, the priority hints, and the full project context, then makes a final decision. In practice, escalation is rare — most conflicts resolve in Round 1 (acceptance) or Round 2 (compromise).

Model Escalation

Sometimes a task assigned to Sonnet needs Opus. The system detects this through failure patterns:

pub struct ModelEscalation {
    failure_threshold: u32,    // Consecutive failures before escalation
    escalation_model: ModelTier,
}

impl ModelEscalation {
    pub fn should_escalate(&self, worker: &WorkerHealth) -> bool {
        // Repeated test failures
        if worker.consecutive_failures >= self.failure_threshold {
            return true;
        }

        // Debate deadlock — same position after both rounds
        if worker.debate_positions_unchanged {
            return true;
        }

        // DAG structural failure on retry
        if worker.structural_errors > 0 && worker.retries > 0 {
            return true;
        }

        false
    }
}

Model escalation is a cost-control mechanism. You start cheap and escalate only when evidence demands it.

17.6 Failure Recovery

Production systems fail. Agents timeout. API calls return 529s. Git worktrees conflict. The quality of an orchestration system is measured by how gracefully it recovers.

Four Failure Categories

Category 1: Agent Stall. A CLI stops producing output for more than 120 seconds. The health monitor detects this and marks the agent as stalled. Recovery: kill the process, retry once with a fresh context window. If the retry stalls, mark the task as Failed.

Category 2: Task Failure. A CLI exits with a non-zero exit code — the model refused a tool call, the context window overflowed, or the agent encountered an unresolvable conflict. Recovery: one retry. If the failed task has no dependents, it is logged and skipped. If it has dependents, those dependents are also marked Failed in a cascading fashion through the DAG.

Category 3: DAG Structural Failure. The Orchestrator generates a plan with circular dependencies or references non-existent tasks. Recovery: the DAG constructor returns an error immediately, and the Orchestrator re-prompts with explicit error feedback. Up to three retries.

Category 4: Synthesis Quality Failure. The synthesis scores below 7 on the quality gate. Recovery: re-debate with quality feedback injected, then re-synthesize. Maximum two refinement loops. If the score remains below 7, deliver the best synthesis with a warning flag.

The Rollback Protocol

Each worker operates in its own git worktree. The merge step happens only after all workers complete and the quality gate passes. If a worker's branch contains broken code, the Orchestrator can:

  1. Skip the branch — exclude it from the final merge
  2. Cherry-pick — take only specific commits
  3. Rebuild — spawn a new worker with error context

The worktree isolation pattern means no worker can corrupt the main branch. Every change is reversible until the final merge.

17.7 The Quality Gate

After debate and before delivery, the Orchestrator runs a quality assessment:

pub struct QualityGate {
    threshold: f64,     // Minimum score (0-10)
    max_refinements: u32,
    assessor_model: ModelTier,
}

#[derive(Debug, Deserialize)]
pub struct QualityAssessment {
    pub score: f64,
    pub feedback: String,
    pub issues: Vec<String>,
    pub verdict: QualityVerdict,
}

#[derive(Debug, Deserialize, PartialEq)]
#[serde(rename_all = "lowercase")]
pub enum QualityVerdict {
    Pass,
    Refine,
}

impl QualityGate {
    pub async fn assess(
        &self,
        synthesis: &str,
        original_prompt: &str,
    ) -> Result<QualityAssessment, ForgeError> {
        let assessment_prompt = format!(
            "Rate this synthesis against the original request.\n\n\
             Original: {original_prompt}\n\n\
             Synthesis:\n{synthesis}\n\n\
             Score 1-10. 7+ = pass. Output JSON only:\n\
             {{\"score\": N, \"feedback\": \"...\", \"issues\": [...], \"verdict\": \"pass|refine\"}}"
        );

        let response = self.llm.call(
            &assessment_prompt,
            self.assessor_model,
        ).await?;

        serde_json::from_str(&response).map_err(ForgeError::from)
    }

    pub async fn gate_with_refinement(
        &self,
        orchestrator: &Orchestrator,
    ) -> QualityAssessment {
        for attempt in 0..=self.max_refinements {
            let synthesis = orchestrator.synthesize().await;
            let assessment = self.assess(&synthesis, &orchestrator.prompt).await.unwrap();

            if assessment.verdict == QualityVerdict::Pass {
                return assessment;
            }

            if attempt < self.max_refinements {
                // Re-debate with quality feedback
                orchestrator.inject_quality_feedback(&assessment).await;
                orchestrator.run_debate(1).await;
            }
        }

        // Deliver best effort with warning
        tracing::warn!("Quality gate did not pass after refinements");
        self.assess(&orchestrator.synthesize().await, &orchestrator.prompt).await.unwrap()
    }
}

For our bookstore project, the first synthesis scores 8/10. Verdict: pass. The system moves to delivery.

17.8 Prompting Patterns for the Forge

The orchestration protocol is only as good as the prompt that activates it. There is a sweet spot — prompts that are specific enough to guide the system without being so detailed that they constrain it.

The Anatomy of an Effective One-Liner

Every effective forge prompt contains four elements:

  1. The What: what you are building (REST API, CLI tool, dashboard)
  2. The Domain: the business context (bookstore, healthcare, fintech)
  3. The Qualities: non-functional requirements (secure, fast, tested)
  4. The Deliverables: what you expect to receive (tests, docs, deployment)
"Build me a secure REST API for a bookstore with auth, CRUD, tests, and docs"
  ───┬───   ──┬──  ───┬────     ────┬────  ──┬─  ──┬──  ──┬──      ──┬─
     |        |       |             |        |     |      |          |
   What    Quality  What         Domain    Quality What  Deliverable Deliverable

Ten Prompt Patterns

Pattern 1: The Full Stack

"Build a real-time chat app with React, WebSocket, user auth, message persistence, and E2E tests"

Triggers: Coder (frontend + backend) + Security + QA. Five to six CLIs.

Pattern 2: The Security-First

"Create a healthcare API with HIPAA-compliant data handling, encryption at rest, audit logging, and penetration test report"

Triggers: Coder + Security + QA + Documentation. The word "HIPAA" triggers healthcare-specific compliance checks.

Pattern 3: The Migration

"Migrate the Express API to Axum, maintain backward compatibility, update all tests, and verify zero performance regression"

Triggers: Coder + Validator + QA. Migration pattern activates a specialized workflow where the Coder reads existing code before writing new code.

Pattern 4: The Infrastructure

"Set up CI/CD with GitHub Actions, Docker multi-stage builds, staging/production environments, and security scanning"

Triggers: Coder (DevOps mode) + Security. Minimal code writing, heavy configuration.

Pattern 5: The Refactor

"Refactor the authentication module to use the repository pattern, add comprehensive error types, preserve all existing tests"

Triggers: Coder + Validator. The word "preserve" signals behavior-preserving transformation.

Pattern 6: The Audit

"Perform a complete security audit of src/api/, check OWASP Top 10, review all auth flows, and produce a remediation plan with priorities"

Triggers: Security + Documentation. Read-only analysis with report output.

Pattern 7: The Greenfield Library

"Create a Rust crate for rate limiting with token bucket and sliding window algorithms, benchmarks, and docs.rs-ready documentation"

Triggers: Coder + QA + Documentation. The "Rust crate" keyword activates Rust-specific tooling.

Pattern 8: The Bug Fix

"Fix the race condition in the order processing pipeline where concurrent requests create duplicate line items"

Triggers: Coder + QA. The "race condition" keyword activates concurrency-focused analysis.

Pattern 9: The Research-First

"Evaluate three state management approaches for our React app — compare Redux Toolkit, Zustand, and Jotai with a recommendation"

Triggers: Research + Documentation. The "evaluate" and "compare" keywords activate the Research CLI.

Pattern 10: The Production Hardening

"Add structured logging, health checks, graceful shutdown, circuit breakers, and Prometheus metrics to the API service"

Triggers: Coder + Validator + Documentation. The compound list of operational concerns spawns parallel implementation tasks.

Prompts That Fail

Too vague: "Build me something cool with AI." — No specific roles triggered. Falls back to generic configuration.

Too prescriptive: "Build a REST API using Axum 0.7.5 with SQLx 0.8 using PostgreSQL, JWT auth using jsonwebtoken 9.0.0 with RS256, argon2 cost factor exactly 3, exactly 7 endpoints..." — This over-constrains the agents. The architect has nothing to decide. The security CLI cannot recommend alternatives. You have reduced the forge to a transcription engine.

17.11 The COORDINATOR_MODE Bridge

Everything described in this chapter maps directly to patterns that already exist inside Claude Code. The COORDINATOR_MODE feature gate — feature('COORDINATOR_MODE') — implements the same orchestration at the CLI level:

COORDINATOR_MODE Allowed Tools:
  AgentTool        (spawn sub-agents)
  TaskStop         (halt execution)
  SendMessage      (inter-agent communication)
  SyntheticOutput  (compose results)
  TeamCreate       (create agent teams)
  TeamDelete       (disband agent teams)

The coordinator runs with a restricted tool set — it cannot read files, write code, or execute bash commands. Its only job is to orchestrate other agents. This is the same separation of concerns that the forge implements.

The BRIDGE_MODE feature gate extends this with remote session support — up to 32 parallel sessions with JWT-based authentication and cross-machine coordination. The DAEMON gate adds process supervision with worker spawning and crash recovery.

These are not theoretical features. They are implemented in Claude Code's source, behind feature gates, waiting for general availability. The forge is an external implementation of the same patterns — proof that multi-CLI orchestration works in production today, without waiting for feature gates to flip.

The architecture is converging. Whether you build your orchestrator externally, use Claude Code's internal coordinator, or wait for the official multi-agent API, the patterns are identical: intent analysis → task decomposition → parallel execution → adversarial debate → quality-gated synthesis. This chapter has given you all of them.

What makes this bridge important is what it implies about the future. Today, the forge is an external orchestrator that spawns Claude Code CLI instances. Tomorrow, COORDINATOR_MODE matures, and the Orchestrator CLI becomes a first-class citizen inside Claude Code itself — no external process management, no worktree gymnastics, no filesystem-based coordination. The same CLAUDE.md that defines the Orchestrator today will work inside COORDINATOR_MODE tomorrow. The same task DAG format. The same debate protocol. The same quality gate. Your investment in understanding these patterns is future-proof because the patterns are the same at every level of abstraction.

The DAEMON gate deserves special attention. It adds automatic crash recovery — if a worker CLI dies, the daemon restarts it with the last known context and re-attaches it to its assigned task. This eliminates Category 1 failures (agent stalls) entirely, because the daemon detects process death faster than the 120-second stall timeout and restarts immediately. Combined with worktree isolation, this means a forge running under DAEMON supervision is resilient to individual CLI crashes without any data loss or merge corruption.

17.12 Non-Engineering Domains

The orchestration protocol is domain-agnostic. The six-phase model — intent, plan, spawn, execute, debate, synthesize — maps to any domain where specialized work can be decomposed and parallelized.

Pharmaceutical Research. A drug discovery pipeline has computational chemists, toxicologists, regulatory specialists, and clinical trial designers. The Orchestrator CLI decomposes "evaluate compound X for cardiovascular indication" into parallel work streams: computational chemistry screens for binding affinity, toxicology flags hepatotoxicity risks, regulatory reviews patent landscape, clinical trial design estimates Phase II requirements. The debate phase catches the case where the chemist's enthusiasm for binding affinity conflicts with the toxicologist's liver enzyme concerns. The synthesis produces a go/no-go recommendation with quantified uncertainty. The priority hint "safety-first" (from the pharmaceutical domain detection) ensures toxicology findings carry higher weight than efficacy excitement during debate resolution.

Legal Mergers and Acquisitions. An M&A due diligence team has corporate lawyers, financial analysts, IP specialists, and regulatory compliance officers. "Evaluate acquisition target Y" becomes parallel work streams: financial analysis, IP portfolio review, regulatory risk assessment, and employment law review. The debate surfaces conflicts — the financial analyst sees value in the IP portfolio, but the IP specialist identifies three patents approaching expiration and two pending litigation claims. The synthesis weights both perspectives using the priority hints from the original prompt. The quality gate checks completeness: did every jurisdiction get covered? Did the employment law review account for all subsidiaries? A score below 7 triggers a refinement loop that addresses the specific gaps.

Film Post-Production. A visual effects pipeline has compositors, colorists, sound designers, and quality control specialists. "Final grade reel 3" spawns parallel work: color grading, sound mixing, visual effects compositing, and continuity checking. The debate phase catches the colorist's warm grade conflicting with the sound designer's cold-weather ambient audio cues — visual warmth paired with wind-and-snow sound design creates cognitive dissonance for the viewer. The quality gate ensures technical specifications (color space, audio levels, frame rate) meet delivery requirements for the target distribution platform.

Academic Peer Review. A manuscript review pipeline has methodology reviewers, statistical analysts, domain experts, and ethics board members. "Review submission XYZ" spawns parallel evaluation streams. The debate phase is where this shines — the statistician questions the sample size, the domain expert defends it as standard for the subfield, and the methodology reviewer identifies a confounding variable neither noticed. The adversarial collaboration produces a more thorough review than any single reviewer would generate, and the synthesis provides the editor with a structured accept/revise/reject recommendation backed by evidence from every perspective.

The pattern is universal: decompose into specialists, execute in parallel, debate to catch conflicts, synthesize with quality gates. The CLIs change. The domain knowledge changes. The protocol remains the same.
