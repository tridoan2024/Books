Chapter 19: Learning and Evolution — The Self-Improving System

There is a number that haunts every team that builds with AI agents: time-to-first-useful-output. On day one of your multi-CLI deployment, the Coder CLI takes forty-five seconds to produce a function because it does not know your project uses dependency injection. The Security CLI flags seventeen false positives because it has not learned that your eval() calls are sandboxed behind a WASM boundary. The QA CLI generates tests that import from paths that do not exist because it has not mapped your module aliases.

Fast-forward one hundred days. The Coder CLI produces that function in eleven seconds — it knows the injection pattern, has seen four hundred successful implementations, and pre-loads the right factory templates. The Security CLI fires three findings, all genuine, because its false-positive model has been trained on fourteen weeks of human feedback. The QA CLI generates tests that compile on the first run ninety-two percent of the time.

The difference is not hardware or a model upgrade. It is learning. The system got smarter by doing work, observing outcomes, and updating itself.

This chapter covers the specific mechanisms that enable genuine self-improvement: feedback loops per CLI, memory consolidation, skill auto-tuning, cross-CLI knowledge transfer, and knowledge version control.

19.1 Feedback Loops Per CLI

Every CLI executes a cycle thousands of times: receive task, produce output, observe outcome. What separates a static tool from a learning system is what happens between "observe outcome" and "receive next task."

The Coder CLI Feedback Loop

Code either compiles or it does not. Tests either pass or they do not. These binary signals create a clean gradient for learning.

#[derive(Debug, Serialize, Deserialize)]
pub struct CoderOutcome {
    pub task_type: String,
    pub language: String,
    pub patterns_used: Vec<String>,
    pub build_success: bool,
    pub test_pass_rate: f64,
    pub lint_violations: u32,
    pub human_edit_distance: u32,   // Lines changed after delivery
    pub time_to_completion_ms: u64,
    pub model_turns: u32,
}

pub struct CoderFeedbackLoop {
    outcomes: Vec<CoderOutcome>,
    rule_extractor: RuleExtractor,
    confidence_threshold: f64,      // Default: 0.80
}

impl CoderFeedbackLoop {
    pub fn record(&mut self, outcome: CoderOutcome) {
        self.outcomes.push(outcome);

        // Check for extractable patterns every 50 outcomes
        if self.outcomes.len() % 50 == 0 {
            self.extract_rules();
        }
    }

    fn extract_rules(&mut self) {
        let clusters = self.rule_extractor.cluster(&self.outcomes);

        for cluster in clusters {
            if cluster.confidence >= self.confidence_threshold {
                self.promote_rule(&cluster);
            }
        }
    }

    fn promote_rule(&self, cluster: &PatternCluster) {
        // Write rule to CLAUDE.md learned patterns section
        let rule = format!(
            "- {} (confidence: {:.2}, n={})\n",
            cluster.rule_text,
            cluster.confidence,
            cluster.observation_count
        );

        // Append to learned patterns section
        append_to_claude_md("## Learned Patterns", &rule);
    }
}

The human edit distance is the most powerful signal. When the Coder delivers code and a human modifies twelve lines, those twelve lines are pure learning signal. The diff between what the Coder wrote and what the human kept tells you exactly where the Coder's mental model diverges from the team's preferences.

Over hundreds of tasks, patterns emerge:

The Architect CLI Feedback Loop

The Architect's loop operates on the longest timescale. Architectural advice cannot be validated in seconds — it takes weeks or months to know whether a decomposition strategy was sound. The Architect's feedback loop relies on retrospective signals: did the proposed architecture lead to fewer cross-team merge conflicts? Did the suggested API boundary reduce the ripple effect of changes?

These signals are harvested from git history, CI metrics, and incident reports. The Architect CLI maintains a decision journal:

#[derive(Debug, Serialize, Deserialize)]
pub struct ArchitectDecision {
    pub date: NaiveDate,
    pub decision: String,
    pub rationale: String,
    pub outcome_30d: Option<String>,
    pub score: Option<f64>,         // -1.0 to +1.0
    pub measured_metrics: Vec<MetricDelta>,
}

#[derive(Debug, Serialize, Deserialize)]
pub struct MetricDelta {
    pub name: String,              // "merge_conflicts", "p95_latency_ms", "incident_count"
    pub before: f64,
    pub after: f64,
    pub improvement: bool,
}

pub struct ArchitectFeedbackLoop {
    decisions: Vec<ArchitectDecision>,
    review_interval_days: u32,     // Default: 30
}

impl ArchitectFeedbackLoop {
    pub fn pending_reviews(&self) -> Vec<&ArchitectDecision> {
        let cutoff = Utc::now().naive_utc().date() - Duration::days(self.review_interval_days as i64);
        self.decisions.iter()
            .filter(|d| d.date <= cutoff && d.outcome_30d.is_none())
            .collect()
    }

    pub fn record_outcome(
        &mut self,
        decision_date: NaiveDate,
        outcome: String,
        metrics: Vec<MetricDelta>,
    ) {
        if let Some(d) = self.decisions.iter_mut().find(|d| d.date == decision_date) {
            let score = metrics.iter()
                .map(|m| if m.improvement { 0.3 } else { -0.3 })
                .sum::<f64>()
                .clamp(-1.0, 1.0);

            d.outcome_30d = Some(outcome);
            d.score = Some(score);
            d.measured_metrics = metrics;
        }
    }
}

The decision journal creates institutional memory that outlives any individual session. When a new architectural question arises — "should we split this service?" — the Architect CLI can consult its journal: "We split the auth service on 2026-03-15. Outcome after 30 days: incidents dropped from 5 to 2, but deployment complexity increased. Score: +0.7." This evidence-based reasoning is what separates architectural wisdom from architectural opinion.

The Security CLI Feedback Loop

Security's primary failure mode is not "wrong answer" but "wrong alarm." A false positive erodes trust and trains the team to ignore findings. The metric that matters: false positive rate.

#[derive(Debug, Serialize, Deserialize)]
pub struct SecurityOutcome {
    pub finding_type: String,
    pub file: String,
    pub line: u32,
    pub severity: Severity,
    pub human_verdict: HumanVerdict,
    pub dismissal_reason: Option<String>,
    pub code_pattern: String,  // AST pattern that triggered the finding
}

#[derive(Debug, Serialize, Deserialize, PartialEq)]
pub enum HumanVerdict {
    TruePositive,
    FalsePositive,
    Deferred,
}

pub struct SecurityFeedbackLoop {
    outcomes: Vec<SecurityOutcome>,
    suppression_model: SuppressionModel,
    min_observations: usize,   // Default: 5
}

pub struct SuppressionModel {
    suppressions: Vec<Suppression>,
}

pub struct Suppression {
    pub finding_type: String,
    pub code_pattern: String,
    pub fp_rate: f64,
    pub observation_count: usize,
    pub action: SuppressionAction,
}

#[derive(Debug)]
pub enum SuppressionAction {
    Downgrade(Severity),  // From HIGH to INFO
    Suppress,             // Remove from report
}

impl SecurityFeedbackLoop {
    pub fn record(&mut self, outcome: SecurityOutcome) {
        self.outcomes.push(outcome);
        self.update_suppression_model();
    }

    fn update_suppression_model(&mut self) {
        // Group by (finding_type, code_pattern)
        let mut groups: HashMap<(String, String), Vec<&SecurityOutcome>> = HashMap::new();

        for outcome in &self.outcomes {
            groups.entry((outcome.finding_type.clone(), outcome.code_pattern.clone()))
                .or_default()
                .push(outcome);
        }

        for ((finding_type, pattern), outcomes) in &groups {
            if outcomes.len() < self.min_observations {
                continue;
            }

            let fp_count = outcomes.iter()
                .filter(|o| o.human_verdict == HumanVerdict::FalsePositive)
                .count();
            let fp_rate = fp_count as f64 / outcomes.len() as f64;

            if fp_rate > 0.80 {
                self.suppression_model.suppressions.push(Suppression {
                    finding_type: finding_type.clone(),
                    code_pattern: pattern.clone(),
                    fp_rate,
                    observation_count: outcomes.len(),
                    action: SuppressionAction::Downgrade(Severity::Info),
                });
            }
        }
    }
}

The suppression model is conservative by design. A pattern needs at least five false-positive observations before consideration, and it is never fully suppressed — it is downgraded from HIGH to INFO and moved to an appendix. If the pattern later appears in an exploitable context, the lower severity means it is still visible.

The QA CLI Feedback Loop

QA measures the most frustrating metric in automated testing: first-run compile rate. Nothing kills confidence faster than tests that do not compile.

#[derive(Debug, Serialize, Deserialize)]
pub struct QAOutcome {
    pub test_type: String,        // "unit", "integration", "e2e"
    pub target_function: String,
    pub compiled_first_run: bool,
    pub passed_first_run: bool,
    pub coverage_delta: f64,      // Percentage points added
    pub assertion_count: u32,
    pub human_modified: bool,
}

A test that fails on the first run and reveals a genuine bug is the highest-value outcome. A test that passes immediately but adds zero coverage is the lowest. The QA CLI learns to prioritize edge cases and boundary conditions over happy-path tests that inflate coverage numbers.

19.2 The Pattern Extraction Engine

Individual outcome records are useful. Patterns across thousands of them are transformative. The extraction engine runs as a background process and follows three stages.

Stage 1: Clustering

Raw outcome records are clustered by semantic similarity. The system feeds batches of fifty records to a fast model and asks: "Which of these outcomes share a common root cause?"

pub struct PatternCluster {
    pub rule_text: String,
    pub confidence: f64,
    pub observation_count: usize,
    pub supporting_outcomes: Vec<usize>,  // Indices into outcome buffer
    pub contradictions: usize,
}

pub struct RuleExtractor {
    clusters: Vec<PatternCluster>,
}

impl RuleExtractor {
    pub fn cluster(&self, outcomes: &[CoderOutcome]) -> Vec<PatternCluster> {
        // Group outcomes by language + task_type
        let mut groups: HashMap<(String, String), Vec<&CoderOutcome>> = HashMap::new();

        for outcome in outcomes {
            groups.entry((outcome.language.clone(), outcome.task_type.clone()))
                .or_default()
                .push(outcome);
        }

        let mut clusters = Vec::new();

        for ((lang, task), group) in &groups {
            // Find patterns where human_edit_distance > 0 consistently
            let edited: Vec<&&CoderOutcome> = group.iter()
                .filter(|o| o.human_edit_distance > 0)
                .collect();

            if edited.len() >= 5 {
                // Send to LLM for semantic clustering
                let cluster = self.llm_cluster(lang, task, &edited);
                clusters.extend(cluster);
            }
        }

        clusters
    }
}

Stage 2: Rule Extraction

Each cluster is distilled into a candidate rule with a specific format designed to slot into CLAUDE.md:

# Candidate rule
[rule]
text = "When generating TypeScript, prefer const assertions and literal types over broad primitives"
confidence = 0.88
observations = 26
contradictions = 3
source = "coder_outcome_clustering"
trigger = "typescript_type_declarations"
exceptions = ["External API types where broad types are required by the schema"]

Stage 3: Validation

Candidate rules enter a validation period. For the next fifty tasks matching the trigger condition, the system tracks whether the rule improves outcomes. If human edit distance decreases, the rule is promoted to active. If outcomes are unchanged or worse, the rule is archived.

This three-stage pipeline ensures learning is evidence-based, not hallucinatory. The system does not learn from a single anecdote — it learns from statistical patterns validated by real-world outcomes.

19.3 Memory Consolidation — Daily and Weekly Compression

Raw memories are expensive. Every observation consumes context window space. The solution is memory consolidation: scheduled compression of raw memories into denser representations.

Daily Compression: EXTRACT_MEMORIES

The EXTRACT_MEMORIES feature gate runs a lightweight extraction agent after each conversation. This agent has restricted tool access — read-only shell, write-only to memory directory — and a hard five-turn limit.

pub struct MemoryExtractor {
    memory_dir: PathBuf,
    max_turns: usize,
}

impl MemoryExtractor {
    pub async fn extract(&self, conversation: &[Message]) -> Vec<ExtractedMemory> {
        let recent_messages: Vec<&Message> = conversation.iter()
            .rev()
            .take(10)  // Only look at recent turns
            .collect();

        // Skip if user already wrote to memory this turn
        if self.user_wrote_memory_this_turn(&recent_messages) {
            return vec![];
        }

        let extraction_prompt = format!(
            "Review these messages and extract key facts worth remembering.\n\
             Priority: decisions > bug_recipes > api_patterns > architecture > preferences\n\
             Write each memory to a separate file in {}.\n\
             Maximum 3 memories per extraction.\n\n\
             {}",
            self.memory_dir.display(),
            format_messages(&recent_messages)
        );

        // Run as forked subagent with restricted tools
        self.run_extraction_agent(&extraction_prompt).await
    }
}

What gets extracted follows a priority hierarchy: 1. Decisions — "We chose PostgreSQL because of ACID requirements" 2. Bug recipes — "The race condition was caused by missing SELECT FOR UPDATE" 3. API patterns — "Auth uses Bearer JWT; refresh via POST /auth/refresh" 4. Architecture facts — "Payment service uses event bus, not direct calls" 5. User preferences — "Always use ruff for Python formatting"

Weekly Compression: KAIROS_DREAM

Daily extraction accumulates memories. Weekly consolidation organizes them. The KAIROS_DREAM agent runs during idle periods — the system reviews and reorganizes its memories while idle, the way human memory consolidation occurs during sleep.

The four-phase dream cycle (as detailed in Chapter 14):

Phase 1 — Orient. Read the memory index and existing topic files. Understand the current state of knowledge.

Phase 2 — Gather. Search for new signal in daily logs and session transcripts. Only look for things that the Orient phase suggested might matter.

Phase 3 — Consolidate. Merge new signal into existing files rather than creating duplicates. Convert relative dates to absolute. Delete contradicted facts.

Phase 4 — Prune and Index. Update MEMORY.md, enforce the 200-line / 25KB limit. The index is an index, not a dump.

The result: a memory directory that is smaller and more useful than before. Effective learning means accumulating more understanding, which requires compressing raw data into denser representations.

Dream Scheduling in the Multi-CLI Forge

In a single-CLI setup, dream consolidation runs during idle periods between sessions. In a multi-CLI forge, the scheduling is more complex because multiple CLIs share the memory directory and each has its own dream schedule.

The forge-level dream coordinator (introduced in Chapter 14) manages this:

Individual CLI learning is powerful. Cross-CLI learning is transformative.

Route 1 → Coder CLI's learned rules:

The routing is pull-based, not push-based. Each CLI reads what it needs at session start rather than receiving real-time notifications. The pull model is simpler, more reliable, and does not require a message broker. When the Coder CLI starts a new session, it checks the shared memory directory for entries targeting "coder" and loads them into its context window alongside its CLAUDE.md rules.

The KAIROS_DREAM consolidation agent also processes shared memory during its overnight run. It merges duplicate entries across CLIs, removes findings that have been fixed (by checking whether the flagged code still exists), and escalates patterns that appear across multiple finding types into higher-level rules. For example, if the knowledge bus has routed three different SQL injection findings over two months, the dream agent consolidates them into a single "parameterized query" rule with combined evidence.

Implementation: The Knowledge Bus

pub struct KnowledgeBus {
    shared_dir: PathBuf,
    routing_rules: Vec<RoutingRule>,
}

pub struct RoutingRule {
    pub source_cli: CliRole,
    pub finding_type: String,
    pub targets: Vec<CliRole>,
    pub transform: TransformType,
}

pub enum TransformType {
    DirectCopy,                    // Copy finding as-is
    ExtractRule,                   // Extract a CLAUDE.md rule
    GenerateTestTemplate,          // Create test template
    DesignPrinciple,               // Extract architectural principle
}

impl KnowledgeBus {
    pub fn route(&self, finding: &Finding) -> Vec<RoutedKnowledge> {
        let mut routed = Vec::new();

        for rule in &self.routing_rules {
            if rule.source_cli == finding.source_cli
                && rule.finding_type == finding.finding_type
            {
                for target in &rule.targets {
                    let knowledge = match &rule.transform {
                        TransformType::DirectCopy => {
                            RoutedKnowledge::direct(finding, target)
                        }
                        TransformType::ExtractRule => {
                            RoutedKnowledge::as_rule(finding, target)
                        }
                        TransformType::GenerateTestTemplate => {
                            RoutedKnowledge::as_test_template(finding, target)
                        }
                        TransformType::DesignPrinciple => {
                            RoutedKnowledge::as_principle(finding, target)
                        }
                    };
                    routed.push(knowledge);
                }
            }
        }

        routed
    }

    pub fn write_to_shared(&self, knowledge: &[RoutedKnowledge]) {
        for k in knowledge {
            let target_file = self.shared_dir.join(k.target_file());
            let mut file = OpenOptions::new()
                .create(true)
                .append(true)
                .open(&target_file)
                .expect("Failed to open shared memory file");

            writeln!(file, "\n### [{}] {}", k.timestamp, k.title)
                .expect("Failed to write");
            writeln!(file, "- **Source:** {} CLI, finding #{}", k.source, k.finding_id)
                .expect("Failed to write");
            writeln!(file, "- **Confidence:** {:.2}", k.confidence)
                .expect("Failed to write");
            writeln!(file, "- **Rule:** {}", k.rule_text)
                .expect("Failed to write");
        }
    }
}

19.5 Knowledge Version Control

Learned knowledge changes over time. A rule that was correct six months ago might be wrong today because the codebase migrated from one framework to another. Knowledge version control lets you diff your system's intelligence like you diff source code.

pub struct KnowledgeVersion {
    pub version: u64,
    pub timestamp: DateTime<Utc>,
    pub changes: Vec<KnowledgeChange>,
    pub trigger: VersionTrigger,
}

pub enum KnowledgeChange {
    RuleAdded(LearnedRule),
    RuleUpdated { old: LearnedRule, new: LearnedRule },
    RuleRetired { rule: LearnedRule, reason: String },
    SuppressionAdded(Suppression),
    PatternPromoted(PatternCluster),
}

pub enum VersionTrigger {
    DreamConsolidation,
    ManualEdit,
    PatternExtraction,
    CrossCliTransfer,
}

Every change to the knowledge base — whether from dream consolidation, manual editing, pattern extraction, or cross-CLI transfer — is recorded as a versioned change. This creates a timeline:

v47  [2026-03-15] Dream: Merged 3 auth memories, retired old JWT pattern
v48  [2026-03-16] Pattern: Extracted "prefer-literal-types" rule (n=26)
v49  [2026-03-16] CrossCLI: Security finding #001 propagated to Coder, QA
v50  [2026-03-22] Dream: Pruned 12 stale entries, updated API conventions
v51  [2026-03-23] Manual: User corrected database-schema.md

You can diff any two versions to see what the system learned — or unlearned — between them. This is particularly valuable when debugging: if the Coder CLI suddenly starts producing code that does not match your team's conventions, you can check the knowledge version history to see which rule changed and revert it.

The Diff Interface

impl KnowledgeVersion {
    pub fn diff(from: &Self, to: &Self) -> KnowledgeDiff {
        let mut added = Vec::new();
        let mut removed = Vec::new();
        let mut modified = Vec::new();

        for change in &to.changes {
            match change {
                KnowledgeChange::RuleAdded(rule) => {
                    added.push(format!("+ {}", rule.text));
                }
                KnowledgeChange::RuleRetired { rule, reason } => {
                    removed.push(format!("- {} (reason: {})", rule.text, reason));
                }
                KnowledgeChange::RuleUpdated { old, new } => {
                    modified.push(format!(
                        "~ {} → {} (confidence: {:.2} → {:.2})",
                        old.text, new.text, old.confidence, new.confidence
                    ));
                }
                _ => {}
            }
        }

        KnowledgeDiff { added, removed, modified }
    }
}

19.6 SKILL_IMPROVEMENT: Self-Evolving Workflows

The feedback loop is not limited to CLAUDE.md rules. Skills — the packaged workflow instructions from Chapter 16 — also evolve based on outcomes. The SKILL_IMPROVEMENT mechanism tracks each skill's effectiveness and updates its instructions when performance degrades.

As covered in Chapter 16, the mechanism works through three stages:

  1. Collect outcomes. Each skill invocation records whether the output was accepted, rejected, or modified by the user.

  2. Scoring pass. When the outcome buffer reaches the threshold (default: 5), compute the acceptance rate and identify failure patterns.

  3. Improvement prompt. If the acceptance rate falls below 90%, generate an improvement prompt that includes the current skill instructions, the failure patterns, and examples of user modifications. The LLM produces revised instructions.

What makes this powerful in the multi-CLI context is that skills shared via MCP_SKILLS (Chapter 16) propagate improvements across the fleet. When the Security CLI's security-audit skill improves based on feedback from the Orchestrator's pipeline usage, that improvement is available to every CLI that invokes the skill remotely.

pub struct SkillEvolutionTracker {
    skill_name: String,
    versions: Vec<SkillVersion>,
}

#[derive(Debug)]
pub struct SkillVersion {
    pub version: u32,
    pub timestamp: DateTime<Utc>,
    pub body: String,
    pub acceptance_rate: f64,
    pub observations: usize,
    pub trigger: String,  // "manual", "improvement_pass", "rollback"
}

impl SkillEvolutionTracker {
    pub fn lineage(&self) -> String {
        let mut output = format!("Skill: {}\n\n", self.skill_name);

        for v in &self.versions {
            output.push_str(&format!(
                "v{} [{}] acceptance: {:.0}% (n={}) trigger: {}\n",
                v.version,
                v.timestamp.format("%Y-%m-%d"),
                v.acceptance_rate * 100.0,
                v.observations,
                v.trigger,
            ));
        }

        output
    }

    pub fn rollback_to(&mut self, version: u32) -> Option<&SkillVersion> {
        self.versions.iter().find(|v| v.version == version)
    }
}

The lineage view shows the complete evolution of a skill:

Skill: commit

v2 [2026-02-01] acceptance: 88% (n=12) trigger: improvement_pass

Version 4 was an improvement attempt that degraded performance — the LLM's revision removed a step that users relied on. The rollback to v3 restored the previous high-performing version. This rollback capability is the safety net that makes automated skill improvement safe to deploy.

19.6 Non-Engineering Domains

Medical Diagnostics. A diagnostic system has specialist CLIs for radiology, pathology, cardiology, and internal medicine. The radiology CLI learns suppression rules for common false positives — motion artifacts that look like lesions, normal anatomical variants flagged as abnormalities. These suppressions propagate to the pathology CLI so it does not waste time on referrals triggered by known false positives. The knowledge version control creates an audit trail required by regulatory bodies.

Financial Trading. Trading desk CLIs learn from market outcomes. The strategy CLI records which entry signals led to profitable trades versus losses. The risk CLI learns which portfolio configurations survived drawdowns. Cross-CLI transfer ensures that the strategy CLI's position-sizing rules reflect the risk CLI's latest volatility estimates. The weekly dream consolidation prunes stale market patterns — a pattern from a low-volatility regime is irrelevant during a crisis.

Legal Research. A legal research system has CLIs for case law, statutes, regulatory filings, and contract analysis. The case law CLI learns which jurisdictional patterns lead to favorable precedent citations. When it discovers that a particular appellate court consistently rules differently from the trial courts on data privacy, that knowledge propagates to the contract analysis CLI, which adjusts its risk assessments for contracts governed by that jurisdiction.

The key insight is that all metrics improve together because the learning loops are connected. The Security CLI's improving false-positive rate feeds better rules to the Coder CLI, which produces more secure code on the first pass, which reduces the Security CLI's workload, which lets it focus on more nuanced findings. The virtuous cycle compounds over weeks and months.

When metrics plateau — and they will — it indicates that the system has absorbed the easy patterns and the remaining improvement requires either more data (more sessions) or more sophisticated analysis (model upgrades, new pattern extractors). This is the signal to evaluate whether your current model tier is sufficient or whether escalating specific CLIs to a more capable model would unlock the next improvement plateau.
