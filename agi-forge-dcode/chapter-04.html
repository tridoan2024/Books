PART II: BUILDING THE FORGE

Chapter 4: Setting Up the Forge — Your Multi-CLI Development Environment

From Architecture to Running Code: Two Paths, One Workspace, First Communication

A blacksmith does not forge steel on a kitchen table. Before the first tool takes shape, the forge itself must be built — the anvil positioned, the bellows connected, the coal arranged, the water trough filled. Skip this step and every tool you produce will be flawed in ways you cannot diagnose.

You have studied the architecture (Chapter 2). You understand the 88 levers available to you (Chapter 3). Now you build the physical workspace — directory structures, build systems, configuration hierarchies, and version control strategies — that will support a fleet of specialized CLI agents working in concert.

This chapter walks both paths: forking Claude Code's TypeScript source for rapid iteration, and building from scratch in Rust for production-grade performance. By the end, you will have two CLI agents running simultaneously, with one dispatching work to the other and receiving results back over MCP.

This is not a thought experiment. You will type commands, compile binaries, and watch agents communicate.

4.1 Two Paths: TypeScript Fork vs. Rust Rewrite

Every builder of multi-CLI systems faces a decision early. Claude Code is open-source TypeScript — approximately 1,884 source files and 512,000 lines of code running on Bun with a React/Ink terminal interface. You can fork it, modify it, and have a custom CLI within hours.

Or you can start from scratch in Rust, building exactly what you need with full control over memory layout, concurrency model, and binary size.

Both paths are legitimate. Both have tradeoffs that matter differently depending on your context. And nothing stops you from walking both simultaneously — TypeScript for rapid prototyping, Rust for production deployment. This is, in fact, the approach documented in the reader's existing projects: td_claudecode (the TypeScript source, 512K LOC) alongside RCode (the Rust rewrite, now approaching 16K LOC across 32 modules).

The TypeScript Fork Path

The fastest route to a working custom CLI is forking Claude Code directly. You inherit the entire feature set — the conversation engine with its 17-step query pipeline, the tool framework with 52 built-in tools, the permission system including the TRANSCRIPT_CLASSIFIER and BASH_CLASSIFIER, the MCP client and server, the React/Ink TUI with 89 React hooks — and you modify only the pieces you need.

Step 1: Clone and build the source.

git clone https://github.com/anthropics/claude-code.git upstream-claude-code
cd upstream-claude-code
bun install
bun run build
./dist/cli.js --version

Claude Code uses Bun as both runtime and package manager. The build chain compiles TypeScript into a bundled CLI at dist/. The React/Ink components render to the terminal — the same declarative component model as web React, but targeting a TTY instead of a browser DOM.

If --version prints a version string, your build works.

Step 2: Understand the build system internals.

From the source analysis in Chapter 2, we know the entry point dispatches through cli.tsx → main.tsx. The build system uses Bun's bundler, which evaluates feature('FLAG_NAME') calls at compile time — this is how feature flags work. Disabled features are dead-code eliminated from the bundle, which is why the npm package is smaller than building from source with all flags enabled.

The critical directories for customization:

upstream-claude-code/
├── src/
│   ├── entrypoints/          # cli.tsx, main.tsx, mcp.ts
│   ├── tools/                # 40+ tool implementations
│   ├── commands/             # 100+ slash commands
│   ├── hooks/                # 88 React hooks (the UI layer)
│   ├── context/              # System prompt assembly
│   ├── constants/            # betas.ts, features.ts, tools list
│   ├── coordinator/          # coordinatorMode.ts (19KB)
│   ├── bridge/               # 33 modules for remote sessions
│   ├── services/             # 108+ service files
│   └── utils/                # Shared utilities including permissions
├── CLAUDE.md                 # The system prompt (THIS you modify)
└── package.json              # Build configuration

Step 3: Establish a forking strategy that survives upstream updates.

The naive approach — editing files directly — makes pulling upstream updates impossible without merge conflicts on every changed file. A professional forking strategy separates customizations from upstream source.

Create a custom/ overlay directory:

upstream-claude-code/
├── src/                    # Upstream source (DO NOT modify)
├── custom/                 # Your additions
│   ├── tools/              # Custom tool implementations
│   ├── commands/           # Custom slash commands
│   ├── config/             # Override configuration
│   └── patches/            # Surgical patches to upstream files
├── build-custom.sh         # Build script that applies overlays
└── CLAUDE.md               # Your custom system prompt (this IS modified)

The build-custom.sh script copies your custom/tools/ into the source tree before building and applies patches from custom/patches/ using git apply. This lets you rebase onto new upstream releases with minimal friction. The only file you modify directly in the upstream tree is CLAUDE.md, which is designed to be user-customized.

#!/usr/bin/env bash
set -euo pipefail

UPSTREAM="$(cd "$(dirname "$0")" && pwd)"

# Apply overlay: copy custom tools into source
cp -r custom/tools/* src/tools/ 2>/dev/null || true
cp -r custom/commands/* src/commands/ 2>/dev/null || true

# Apply patches: surgical modifications to upstream files
for patch in custom/patches/*.patch; do
    [[ -f "${patch}" ]] && git apply --check "${patch}" && git apply "${patch}"
done

# Build with custom feature flags
COORDINATOR_MODE=true bun run build

echo "Custom build complete: ./dist/cli.js"

Step 4: Modify the tool registry.

To create a specialized CLI, you add or remove tools from the tool registry. In Claude Code's architecture, the registry is populated in src/tools/ where each tool exports a definition object. For an Orchestrator CLI, you would create a patch that restricts the registry to coordination-only tools:

// custom/patches/orchestrator-tools.patch
// Restricts tool set to: AgentTool, TaskStop, SendMessage, SyntheticOutput

Alternatively — and this is the approach we recommend — use the environment variable CLAUDE_CODE_COORDINATOR_MODE=true documented in Chapter 3. This achieves the same restriction without any source modification, and it survives upstream updates.

Step 5: Test the fork.

Before any customization, verify the unmodified build works end-to-end:

# Run in non-interactive mode to test the engine
echo "What is 2 + 2?" | ./dist/cli.js -p

# Run in MCP server mode to test the protocol
echo '{"jsonrpc":"2.0","method":"initialize","id":1,"params":{"protocolVersion":"1.0"}}' | \
  ./dist/cli.js --mcp 2>/dev/null | head -1

If both commands produce output, your fork has a working engine and a working MCP server. From here, every customization is additive — you are layering specialist behavior on top of a proven foundation.

Step 6: Create your first custom tool.

Tools in Claude Code follow a consistent interface. Each tool exports a definition with name, category, input schema (Zod validation), and an async execution function. Here is a minimal custom tool:

// custom/tools/loc-counter.ts
import { z } from 'zod';
import { execSync } from 'child_process';
import type { Tool } from '../src/tools/tool';

export const LocCounter: Tool = {
  name: 'loc_count',
  category: 'analysis',
  description: 'Count lines of code by language in a directory',
  inputSchema: z.object({
    directory: z.string().describe('Directory to analyze'),
  }),
  isConcurrencySafe: true,
  async exec(input) {
    const cmd = `find ${input.directory} -type f -name '*.ts' -o -name '*.py' -o -name '*.rs' | xargs wc -l | tail -1`;
    const result = execSync(cmd, { encoding: 'utf-8', timeout: 30000 });
    return { type: 'text', text: result.trim() };
  },
};

Register it in the tool registry and rebuild. Your custom CLI now has a capability that stock Claude Code does not.

The Rust Rewrite Path

The TypeScript fork carries baggage: a large dependency tree, garbage collection pauses, startup time measured in seconds, and a binary size that dwarfs what is strictly necessary. If you are building a fleet of 8+ CLIs that may run concurrently on developer machines or in CI pipelines, these costs compound.

The Rust rewrite trades development speed for runtime performance. You build exactly what you need, with zero overhead you did not choose.

Here is the actual Cargo.toml from the RCode project — a working Rust rewrite with 32 modules. Each dependency was selected for a specific architectural reason:

[package]
name = "rcode"
version = "0.2.0"
edition = "2024"
description = "AI coding agent for the terminal"

[dependencies]
# === Async Runtime ===

futures = "0.3"
async-trait = "0.1"

# === HTTP + Streaming ===
reqwest = { version = "0.12", features = ["stream", "json", "rustls-tls"],
            default-features = false }
eventsource-stream = "0.2"
reqwest-eventsource = "0.6"

# === Terminal UI ===
ratatui = "0.29"
crossterm = "0.28"
syntect = "5"

# === Code Intelligence ===
tree-sitter = "0.24"
tree-sitter-rust = "0.23"
tree-sitter-python = "0.23"
tree-sitter-javascript = "0.23"
tree-sitter-typescript = "0.23"
tree-sitter-go = "0.23"

# === Plugin System ===
mlua = { version = "0.10", features = ["luau", "async", "serialize"] }

# === Configuration ===
figment = { version = "0.10", features = ["toml", "json", "env"] }

# === Persistence ===
rusqlite = { version = "0.32", features = ["bundled"] }

# === File System ===
ignore = "0.4"
notify = { version = "8", features = ["macos_kqueue"] }

# === Serialization ===
serde = { version = "1", features = ["derive"] }
serde_json = "1"
toml = "0.8"

Why Tokio: Every tool execution, API call, and file operation runs as an async task on Tokio's work-stealing thread pool. Unlike Node.js's single-threaded event loop, Tokio gives you true OS-level parallelism. When your security scanner parses ASTs in one task and your LLM client streams tokens in another, they run on separate CPU cores simultaneously. For a fleet of 8 CLIs, this means each CLI fully utilizes multi-core hardware.

Why reqwest + eventsource: The SSE (Server-Sent Events) streaming protocol is essential for token-by-token output from Claude's API. The rustls-tls feature eliminates OpenSSL dependency and cross-compilation headaches.

Why tree-sitter: Parses source code into concrete syntax trees in real time. The Security CLI's SAST scanner walks ASTs to find SQL injection — not by grepping for string concatenation, but by identifying interpolation nodes inside database query function calls. Five language grammars ship with the binary; adding more is a one-line Cargo.toml change.

Why mlua: Embeds the Luau runtime (Roblox's typed Lua) for user extensibility. Custom tools are Lua scripts loaded at startup — no recompilation required. The async feature lets Lua scripts await Rust async functions.

Why figment: Layered configuration merging. A single Config struct is populated from compiled defaults → TOML files → environment variables → CLI flags. Critical for multi-CLI systems where you have team config, per-CLI overrides, per-project overrides, and per-invocation flags.

Why rusqlite with bundled: SQLite compiled directly into the binary. Every session, memory entry, and conversation turn is persisted. The bundled feature means zero system library dependency — your single binary carries its own database engine.

Building the Rust CLI:

cargo build --release
ls -lh target/release/rcode
# ~7.5MB, statically linked, starts in <50ms

The Shared Core Library (forge-core)

Whether you choose TypeScript or Rust, the first thing to build is the shared library that all specialist CLIs depend on. This library defines the contracts — the trait interfaces, message types, and configuration structures — that ensure every CLI speaks the same language.

For the Rust path, the forge-core crate contains:

// shared/forge-core/src/lib.rs
pub mod tools;      // The Tool trait and ToolResult types
pub mod llm;        // LLM client abstraction (provider-agnostic)
pub mod mcp;        // MCP server and client implementations  
pub mod config;     // Configuration structs (deserialized from TOML)
pub mod messages;   // Inter-CLI message types
pub mod memory;     // Shared memory read/write interface
pub mod session;    // Session persistence (SQLite)
pub mod context;    // System prompt assembly

The most important definition is the Tool trait:

// shared/forge-core/src/tools/mod.rs
use async_trait::async_trait;
use serde_json::Value;

#[async_trait]
pub trait Tool: Send + Sync {
    fn name(&self) -> &str;
    fn description(&self) -> &str;
    fn input_schema(&self) -> Value;
    fn is_concurrency_safe(&self) -> bool { true }
    async fn execute(&self, input: Value, ctx: &ToolContext) -> anyhow::Result<ToolResult>;
}

pub struct ToolContext {
    pub project_root: std::path::PathBuf,
    pub shared_memory_path: std::path::PathBuf,
    pub session_id: String,
    pub cli_identity: String,  // "security-cli", "coder-cli", etc.
}

pub enum ToolResult {
    Text(String),
    Json(Value),
    Error { message: String, recoverable: bool },
}

Every tool in every CLI implements this trait. The Orchestrator does not need to know how the Security CLI's SAST scanner works — it calls execute() with a JSON input and receives a ToolResult. This contract is what makes MCP interoperability possible: the trait maps directly to the MCP tools/call protocol.

For TypeScript, the equivalent is a shared npm package with TypeScript interfaces:

// packages/forge-core/src/tool.ts
export interface Tool {
  name: string;
  description: string;
  inputSchema: ZodSchema;
  isConcurrencySafe: boolean;
  exec(input: any, context: ToolContext): Promise<ToolResult>;
}

The shared library is the single most important architectural decision in the workspace. Get this right and adding new CLIs is trivial — implement the Tool trait, write a CLAUDE.md, register as MCP server. Get this wrong and every CLI reinvents the wheel.

The forge-core library should be treated as your team's API surface. Changes to it affect every CLI, so treat it with the same rigor as a public API: semantic versioning, backward compatibility where possible, and deprecation cycles when breaking changes are unavoidable. In practice, the core library stabilizes quickly — the Tool trait, ToolContext, and ToolResult types rarely change once the first two CLIs are working. What changes is each CLI's implementation of those traits.

Design principles behind this layout:

The upstream/ directory contains Claude Code as a git submodule — your reference implementation. You read it, study it, but never modify it. When Anthropic releases updates, you pull them here and selectively port improvements:

git submodule add https://github.com/anthropics/claude-code.git upstream/claude-code
git submodule update --init --recursive

The clis/ directory holds specialized CLIs. Each subdirectory is a self-contained project with its own source, configuration, and CLAUDE.md — its own identity. The Orchestrator's CLAUDE.md says "You are a project manager who decomposes tasks and delegates to specialists." The Security CLI's CLAUDE.md says "You are a security auditor. You never modify code. You analyze and report." These are not different prompts sent to the same binary; they are different binaries with different tool sets, different permission boundaries, and different behavioral contracts.

The shared/ directory is the nervous system. The forge-core crate provides the shared library — the tool trait, message types, configuration structs, and LLM client — that all CLIs depend on. The memory/ subdirectory holds the shared state: project context, architectural decisions, security findings. The protocols/ subdirectory defines the JSON schema for inter-CLI messages. The config/ subdirectory holds team-wide configuration.

The orchestration/ directory contains infrastructure that connects CLIs: the task DAG engine (covered in Chapter 6), the message bus (covered in Chapter 15), and the monitoring dashboard.

Note on specialist CLIs beyond the core 8: The workspace structure supports unlimited specialists. Need a Doctor CLI for analyzing medical codebases? Add clis/doctor/. A Legal CLI for contract analysis? Add clis/legal/. A Financial CLI for audit compliance? Add clis/financial/. The 8-CLI architecture in this book is a teaching scaffold — production systems have as many specialists as the domain requires.

4.4 The Build Pipeline

With 8+ CLIs in the workspace, sequential building is unacceptable. A parallel build pipeline compiles the entire fleet in minutes:

#!/usr/bin/env bash
# shared/scripts/build-all.sh
set -euo pipefail

FORGE_ROOT="$(cd "$(dirname "$0")/../.." && pwd)"
CLIS_DIR="${FORGE_ROOT}/clis"
FAILED=()
PIDS=()

# Discover all CLIs dynamically — not hardcoded
mapfile -t CLIS < <(find "${CLIS_DIR}" -maxdepth 1 -mindepth 1 -type d -exec basename {} \; | sort)

echo "=== Multi-CLI Forge — Parallel Build ==="
echo "Building ${#CLIS[@]} CLIs from ${CLIS_DIR}/"
echo ""

for cli in "${CLIS[@]}"; do
    cli_dir="${CLIS_DIR}/${cli}"

    echo "  START ${cli}"
    (
        cd "${cli_dir}"
        if [[ -f "Cargo.toml" ]]; then
            cargo build --release 2>&1 | tail -3
        elif [[ -f "package.json" ]]; then
            bun run build 2>&1 | tail -3
        fi
    ) &
    PIDS+=($!)
done

# Wait for all builds; collect failures
for i in "${!PIDS[@]}"; do
    if ! wait "${PIDS[$i]}"; then
        FAILED+=("${CLIS[$i]}")
    fi
done

echo ""
if [[ ${#FAILED[@]} -gt 0 ]]; then
    echo "=== BUILD FAILED ==="
    printf '  FAIL  %s\n' "${FAILED[@]}"
    exit 1
fi

echo "=== All ${#CLIS[@]} CLIs built successfully ==="

# Report binary sizes
for cli in "${CLIS[@]}"; do
    binary="${CLIS_DIR}/${cli}/target/release/${cli}-cli"
    if [[ -f "${binary}" ]]; then
        size=$(du -h "${binary}" | cut -f1)
        echo "  OK    ${cli}  ${size}"
    fi
done

Key difference from a naive approach: this script discovers CLIs dynamically by scanning the clis/ directory rather than hardcoding a list. When you add a new specialist (e.g., clis/financial/), it is automatically included in the next build without modifying the script.

On a machine with 8+ cores, the Rust workspace's build cache means incremental builds take seconds. The Cargo workspace shares compiled dependencies across all CLIs — tokio, serde, reqwest are compiled once and linked into every binary.

CI/CD Integration

The build pipeline wraps cleanly into GitHub Actions:

# .github/workflows/build-all.yaml
name: Build All CLIs
on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          submodules: recursive
      - uses: dtolnay/rust-toolchain@stable
      - uses: Swatinem/rust-cache@v2
        with:
          workspaces: ". -> target"
      - name: Build all CLIs
        run: cargo build --release --workspace
      - name: Run integration tests
        run: bash shared/scripts/test-integration.sh
      - name: Upload binaries
        uses: actions/upload-artifact@v4
        with:
          name: cli-binaries
          path: target/release/*-cli

The rust-cache action caches compiled dependencies across CI runs. After the first build, subsequent runs only compile your code — not the hundreds of dependency crates. The submodules: recursive flag ensures the Claude Code upstream reference is available for any tests that reference it.

For production releases, add a matrix strategy that builds for multiple platforms:

strategy:
  matrix:
    include:
      - os: ubuntu-latest
        target: x86_64-unknown-linux-gnu
      - os: macos-latest
        target: aarch64-apple-darwin
      - os: windows-latest
        target: x86_64-pc-windows-msvc

Each platform produces a set of statically-linked binaries — no runtime dependencies to install on the target machine. The entire 8-CLI fleet deploys as 8 files totaling approximately 60MB.

4.5 Configuration Hierarchy

A multi-CLI system needs configuration that is both unified (shared settings apply everywhere) and specialized (each CLI has its own overrides). A four-layer hierarchy solves this:

Layer 4: Runtime overrides (env vars, CLI flags)     ← highest priority
Layer 3: CLI-specific config (clis/<name>/config.toml)
Layer 2: Team config (shared/config/team.toml)
Layer 1: Compiled defaults (hardcoded in binary)     ← lowest priority

Layer 2: Team configuration — applies to every CLI:

# shared/config/team.toml
[shared]
memory_path = "shared/memory"
log_level = "info"

[models]
default = "claude-sonnet-4-6"
deep = "claude-opus-4-6"
fast = "claude-haiku-4-5"

[cost]
max_per_session_usd = 5.00
max_per_task_usd = 2.00
warn_threshold = 0.80

[orchestration]
max_parallel_agents = 4
task_timeout_seconds = 300
retry_on_failure = true
max_retries = 2

Layer 3: CLI-specific configuration — overrides team settings for one specialist:

# clis/security/config.toml
[identity]
name = "SecurityCLI"
role = "security-guardian"
color = "red"

[model]
default = "claude-opus-4-6"  # Override: security needs depth

[tools]
enabled = ["sast_scan", "dep_audit", "secret_scan", "threat_model",
           "file_read", "directory_list", "grep_search"]
disabled = ["file_write", "file_edit", "file_delete"]

[permissions]
mode = "strict"

[permissions.bash]
allow = ["grep", "find", "wc", "semgrep", "trivy", "gitleaks"]
deny = ["rm", "mv", "cp", "dd", "chmod", "chown", "curl -X POST"]

[memory]
shared_topics = ["project-context", "architecture-decisions", "security-findings"]
private_topics = ["vulnerability-patterns", "false-positive-cache"]

In Rust, Figment merges all four layers in three lines:

use figment::{Figment, providers::{Toml, Env, Serialized}};

let config: CliConfig = Figment::new()
    .merge(Serialized::defaults(CliConfig::default()))   // Layer 1
    .merge(Toml::file("../../shared/config/team.toml"))  // Layer 2
    .merge(Toml::file("config.toml"))                    // Layer 3
    .merge(Env::prefixed("SECURITY_CLI_"))                // Layer 4
    .extract()?;

A team member who wants to override the model for a single run:

SECURITY_CLI_MODEL_DEFAULT=claude-haiku-4-5 ./security-cli --prompt "Quick scan"

No config file modified. No restart required. Environment variable naming follows: prefix + section + key in UPPER_SNAKE_CASE. This convention is borrowed directly from 12-factor app methodology — every configuration value can be overridden by the deployment environment, making your CLIs as comfortable in a Docker container as on a developer laptop. In Chapter 17, we will use this hierarchy to configure per-task overrides when the Orchestrator dispatches work to specialists with task-specific model and effort requirements.

4.6 Version Control Strategy

A multi-CLI workspace has unique version control challenges: an upstream dependency (Claude Code), 8+ independent CLI projects that share code, and orchestration infrastructure that ties them together.

Upstream Tracking

Claude Code is pinned as a git submodule at a specific commit:

# Add the submodule (one-time)
git submodule add https://github.com/anthropics/claude-code.git upstream/claude-code

# When Anthropic releases updates:
cd upstream/claude-code
git fetch origin
git log --oneline HEAD..origin/main | head -20  # Review changes FIRST

# If safe, update:
git checkout origin/main
cd ../..
git add upstream/claude-code
git commit -m "chore: update upstream claude-code to $(cd upstream/claude-code && git rev-parse --short HEAD)"

Never blindly update the submodule. Read the changelog. If an upstream change alters the tool interface or permission protocol, handle it in a dedicated branch first.

The .gitignore

A multi-CLI workspace generates artifacts that should never be committed:

# Build artifacts
target/
dist/
node_modules/

# CLI-specific secrets and local config
clis/*/.claude/settings.local.json
clis/*/.env
clis/*/.env.local

# Shared memory (runtime state, not source)
shared/memory/findings.db
shared/memory/*.db-journal

# IDE and OS
.idea/
.vscode/
.DS_Store

# Session data
*.session.json
*.pid

The critical entries are the local settings files (settings.local.json) and the findings database. Shared memory markdown files (like project-context.md and decisions.md) are committed — they are part of the project's knowledge base. But the SQLite databases and runtime state files are regenerated each session.

Branch Strategy

main                        # Stable, tested, deployable
├── develop                 # Integration branch
│   ├── cli/security        # Security CLI feature work
│   ├── cli/orchestrator    # Orchestrator feature work
│   ├── feat/message-bus    # Cross-cutting features
│   └── fix/sast-false-pos  # Bug fixes

Unified Versioning

All CLIs share a single version number, released together. Orchestrator v1.3.0 is always tested against Security CLI v1.3.0. No version mismatch debugging.

#!/usr/bin/env bash
# shared/scripts/release.sh
set -euo pipefail

readonly VERSION="${1:?Usage: release.sh <version>}"

# Update all Cargo.toml versions atomically
for toml in clis/*/Cargo.toml shared/*/Cargo.toml; do
    sed -i '' "s/^version = \".*\"/version = \"${VERSION}\"/" "${toml}"
done

# Build, test, tag
bash shared/scripts/build-all.sh
bash shared/scripts/test-integration.sh

git add -A
git commit -m "release: v${VERSION}"
git tag "v${VERSION}"
echo "Tagged v${VERSION}. Push with: git push origin main --tags"

4.7 The "Hello Multi-CLI" Proof of Concept

Every system needs a smoke test — the simplest possible proof that the pieces connect. For a multi-CLI system: start two CLIs, send a task from one to the other, verify the result flows back.

We use the Orchestrator and Coder CLIs. The Orchestrator receives a prompt, decomposes it, dispatches to the Coder over MCP, and presents the synthesized result.

Step 1: Create the Coder CLI's MCP Server Mode

The Coder CLI needs to run as an MCP server — a process that listens on stdio for JSON-RPC messages:

// clis/coder/src/main.rs (simplified)
use clap::Parser;
use forge_core::mcp::McpServer;

#[derive(Parser)]
struct Args {
    #[arg(long)]
    mcp_server: bool,

    #[arg(long)]
    prompt: Option<String>,
}

#[tokio::main]
async fn main() -> anyhow::Result<()> {
    let args = Args::parse();

    if args.mcp_server {
        // MCP mode: read JSON-RPC from stdin, write to stdout
        let server = McpServer::new("coder-cli")
            .with_tool(FileWriteTool::new())
            .with_tool(FileEditTool::new())
            .with_tool(BashTool::new())
            .with_tool(GrepTool::new());
        server.run_stdio().await?;
    } else {
        // Interactive mode: launch TUI
        run_interactive(args.prompt).await?;
    }

    Ok(())
}

Step 2: Configure the Orchestrator's MCP Connection

The Orchestrator's .claude/settings.json (or equivalent MCP config) tells it how to spawn and connect to the Coder:

{
  "mcpServers": {
    "coder": {
      "command": "./clis/coder/target/release/coder-cli",
      "args": ["--mcp-server"],
      "env": {
        "CODER_CLI_SHARED_MEMORY": "./shared/memory"
      }
    }
  }
}

When the Orchestrator starts, it spawns the Coder as a child process, performs the MCP handshake (initialize → initialized), discovers its tools (tools/list), and makes them available alongside its own.

Step 3: Write the Orchestrator's CLAUDE.md

# Orchestrator CLI

You are the Orchestrator — a project manager for a team of specialized CLI agents.

## Rules
1. NEVER write code yourself. Always delegate to a specialist.
2. Decompose every task into subtasks appropriate for each specialist.
3. Dispatch subtasks via MCP tool calls.
4. Collect results and synthesize a coherent response.

## Available Specialists
- **coder** (MCP): file_write, file_edit, bash, grep_search

## Response Format
After delegation completes, summarize:
- What was done (by which specialist)
- Files created or modified
- How to verify the result

Step 4: Run the Test

# Terminal 1: Start the Orchestrator (it auto-spawns the Coder via MCP)
cd clis/orchestrator
CLAUDE_CODE_COORDINATOR_MODE=true ./target/release/orchestrator-cli

Note the CLAUDE_CODE_COORDINATOR_MODE=true environment variable from Chapter 3. This restricts the Orchestrator to coordination-only tools (AgentTool, SendMessage, TaskStop, SyntheticOutput, TeamCreate, TeamDelete) — it physically cannot write files or run arbitrary bash commands. The Orchestrator delegates; the Coder executes. This separation of concerns is enforced at the binary level, not just the prompt level.

Type the prompt:

Create a hello world Express.js API with a /health endpoint returning {"status":"ok"}

Step 5: Watch the Delegation

The communication flow:

You → Orchestrator: "Create Express API with /health"
    Orchestrator → LLM: [decides to delegate to coder]
    Orchestrator → Coder (MCP tools/call): file_write("package.json", {...})
    Coder → Orchestrator (MCP result): {"success": true}
    Orchestrator → Coder (MCP tools/call): file_write("server.js", {...})
    Coder → Orchestrator (MCP result): {"success": true}
    Orchestrator → Coder (MCP tools/call): bash("npm install")
    Coder → Orchestrator (MCP result): {"stdout": "added 64 packages"}
Orchestrator → You: "Created Express API. Files: package.json, server.js. Run: node server.js"

Each arrow represents a JSON-RPC message over the stdio pipe. The Orchestrator makes decisions; the Coder executes them.

Step 6: Verify

cd test-project
node server.js &
sleep 2
curl -s http://localhost:3000/health | python3 -m json.tool
# → {"status": "ok"}
kill %1

What This Proves

This simple test validated five critical properties:

  1. Process isolation — The Coder runs as a separate process. If it crashes, the Orchestrator survives.
  2. MCP communication — Tool calls flow over JSON-RPC stdio pipes, the standard protocol.
  3. Delegation — The Orchestrator's LLM correctly identified the task type and routed it.
  4. Tool bridging — The Coder's tools were accessible to the Orchestrator as if they were its own.
  5. Result synthesis — Multiple tool results were collected and presented coherently.

This is the nucleus. Every feature built from here — task DAG, shared memory, security pipeline, parallel execution — is an extension of this pattern: one CLI dispatches, another executes, results flow back.

4.8 Consolidating Existing Projects

If you have been following along with the reader's journey through this book, you may already have several implementations that partially solve the multi-CLI problem:

  • A Claude Code source extraction (td_claudecode) — the reference implementation
  • A Rust rewrite (RCode) — the performance path
  • A Node.js orchestrator (DApp/SwarmForge) — the orchestration patterns
  • An Express debate system (claudecode_team) — the multi-agent communication
  • A Python terminal hub (MultiCLIs) — the process management

The workspace structure in this chapter is designed to consolidate these into a single coherent system. The mapping:

Existing Project Becomes What You Keep
td_claudecode upstream/claude-code The reference implementation (submodule)
RCode engine modules shared/forge-core The tool trait, LLM client, session management
RCode tools clis/*/src/tools/ Per-CLI tool implementations
DApp orchestrator.js clis/orchestrator/ Task decomposition, intent analysis patterns
DApp task-dag.js orchestration/task-dag/ Task dependency graph engine
claudecode_team orchestration/message-bus/ Debate/synthesis communication patterns
MultiCLIs shared/scripts/ Process launch and management scripts

The consolidation is not a rewrite — it is a re-homing. The orchestration logic from DApp's task-dag.js does not change; it moves to a shared location where all CLIs can use it. The RCode engine modules become the shared library. The Claude Code source remains the reference you study but never modify.

Practical consolidation steps:

# 1. Initialize the workspace
mkdir -p ~/multi-cli-forge
cd ~/multi-cli-forge
git init

# 2. Add Claude Code as upstream reference
git submodule add https://github.com/anthropics/claude-code.git upstream/claude-code

# 3. Create the shared library from RCode's engine
mkdir -p shared/forge-core/src
# Port: engine/mod.rs, tools/mod.rs, llm/client.rs, session/mod.rs
# These become the foundation every CLI builds on

# 4. Create the first two CLIs
mkdir -p clis/orchestrator/src clis/coder/src

# 5. Port DApp's orchestration logic
mkdir -p orchestration/task-dag
# Translate task-dag.js patterns to Rust or keep as a Node.js service

# 6. Set up shared memory
mkdir -p shared/memory
echo "# Project Context" > shared/memory/project-context.md
echo "# Architecture Decisions" > shared/memory/decisions.md

# 7. Create workspace Cargo.toml
cat > Cargo.toml << 'EOF'
[workspace]
resolver = "2"
members = ["shared/forge-core", "clis/orchestrator", "clis/coder"]
EOF

# 8. First build
cargo build --release

The workspace grows organically. You do not need all 8 CLIs on day one. Start with Orchestrator and Coder (as demonstrated in the "Hello Multi-CLI" test above), then add specialists as the need arises. Each new specialist is a cargo init in clis/, a Cargo.toml referencing forge-core, and a CLAUDE.md defining the specialist's identity.
