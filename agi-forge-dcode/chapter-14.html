PART IV: CONNECTING THE SYSTEM

Chapter 14: Shared Memory Architecture

The Memory Bus, Knowledge Extraction, Background Consolidation, and Cross-CLI State Transfer

There is a moment in every multi-CLI system's evolution when you realize the agents are not learning. They execute tasks. They produce correct output. They follow their CLAUDE.md instructions with mechanical precision. But every session starts from zero. The Security CLI catches the same SQL injection pattern it caught last week — and does not recognize it. The Coder CLI reimplements a utility function it wrote three days ago in a different module. The QA CLI generates edge case tests for a boundary condition that was already tested and fixed in a previous session.

Your agents have amnesia. Expensive, token-consuming, time-wasting amnesia.

This chapter builds the memory infrastructure that transforms a collection of independent CLIs into a system that learns. We start with the three-dimensional memory problem that multi-agent architectures face. We dissect Claude Code's eight native memory layers from the source — from the permanent constitutional layer of CLAUDE.md down to the experimental KAIROS_DREAM consolidation agent that literally dreams, reviewing and organizing memories while the developer sleeps. Then we build the shared memory bus: a typed, scored, decaying knowledge substrate that connects every CLI to a common knowledge pool while preserving each agent's private memory space.

By the end, the Coder's implementation decisions will inform the Security CLI's analysis. The Reviewer's feedback will shape the Coder's future output. The Architect's design constraints will propagate to every agent that touches the codebase. Your multi-CLI system will not just execute — it will remember.

14.1 The Memory Problem in Multi-CLI Systems

Single-agent systems have a straightforward memory model: one context window, one conversation history, one set of accumulated knowledge. When Claude Code operates as a single CLI, its memory architecture serves one master. Every memory is written by the same agent that reads it. No conflicts, no synchronization issues, no questions about who knows what.

Multi-agent systems shatter this simplicity along three dimensions.

Dimension 1: Temporal Isolation. Each CLI session is a temporal island. Knowledge generated in session A is invisible to session B, even when both involve the same CLI operating on the same codebase. Yesterday's Coder does not talk to today's Coder. The Security CLI that spent 50,000 tokens analyzing an authentication module on Monday has no way to tell Tuesday's Security CLI what it found. Every session reinvests the same tokens discovering the same patterns.

Dimension 2: Agent Isolation. Even within the same time window, different CLIs do not share context. The Security CLI does not know what the Coder just implemented. The Reviewer does not know what the Architect decided. Each agent operates in a private universe of information. When the Coder discovers a timing vulnerability in a token validation library and fixes it, that discovery — the vulnerability, the fix, the rationale — evaporates when the session ends. The Security CLI, scanning later, must either re-discover the vulnerability (wasting tokens on analysis the Coder already completed) or miss it entirely because the fix is already in place and there is no record of why the code looks the way it does.

Dimension 3: Granularity Mismatch. Different agents need different types of knowledge at different levels of detail. The Coder needs implementation patterns: "use bcrypt with cost factor 12 for password hashing." The Security CLI needs threat context: "this endpoint processes untrusted XML input — watch for XXE." The Architect needs system-level constraints: "the auth service must handle 10K concurrent sessions with sub-100ms latency." A flat shared store where every agent dumps everything creates noise. Each agent wastes tokens parsing information it does not need.

Solving all three dimensions simultaneously — bridging time, bridging agents, and filtering by relevance — requires a structured architecture. That architecture is the memory bus.

14.2 The Memory Bus Architecture

The memory bus is the central nervous system of the multi-CLI forge. It is not a database. It is not a message queue. It is a structured knowledge substrate with well-defined read/write semantics, conflict resolution policies, and access control boundaries.

Each CLI connects to the shared bus through a standardized interface. The bus itself is a SQLite database — the same technology Claude Code uses internally for session storage — with a schema designed for typed, scored, decaying knowledge.

But the bus is not the only memory each CLI has. Every agent maintains a private memory store for agent-specific knowledge that should not pollute the shared space. The Coder's private memory might contain preferred library versions and implementation shortcuts. The Security CLI's private memory contains its threat model heuristics, CVE watchlists, and scanning strategies. This two-tier model — shared bus plus private stores — mirrors how human teams work. You share decisions with the team. You keep your personal notes to yourself.

The Shared Memory Schema

CREATE TABLE memories (
    id          TEXT PRIMARY KEY,
    type        TEXT NOT NULL CHECK(type IN (
                    'user', 'feedback', 'project', 'reference',
                    'decision', 'pattern', 'threat', 'test_intel',
                    'perf_baseline', 'doc_gap'
                )),
    key         TEXT NOT NULL,
    value       TEXT NOT NULL,
    importance  INTEGER NOT NULL DEFAULT 3 CHECK(importance BETWEEN 1 AND 5),
    written_by  TEXT NOT NULL,
    created_at  TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at  TEXT NOT NULL DEFAULT (datetime('now')),
    accessed_at TEXT,
    access_count INTEGER DEFAULT 0,
    decay_rate  REAL DEFAULT 1.0,
    is_archived INTEGER DEFAULT 0,

    UNIQUE(type, key)
);

CREATE TABLE memory_links (
    from_id     TEXT NOT NULL REFERENCES memories(id),
    to_id       TEXT NOT NULL REFERENCES memories(id),
    relation    TEXT NOT NULL CHECK(relation IN (
                    'related_to', 'evolved_from', 'caused_by',
                    'part_of', 'contradicts', 'supersedes'
                )),
    strength    REAL DEFAULT 1.0 CHECK(strength BETWEEN 0.0 AND 1.0),
    created_at  TEXT NOT NULL DEFAULT (datetime('now')),
    PRIMARY KEY (from_id, to_id, relation)
);

CREATE TABLE memory_access_log (
    memory_id   TEXT NOT NULL REFERENCES memories(id),
    accessed_by TEXT NOT NULL,
    accessed_at TEXT NOT NULL DEFAULT (datetime('now')),
    context     TEXT
);

Four critical dimensions of shared knowledge are captured here:

  1. Typed memories with a constrained taxonomy. Types enable agents to query only what is relevant — the Security CLI reads threat and decision memories; it rarely needs test_intel.

  2. Provenance tracking via written_by. A threat memory written by the Security CLI carries more authority than one written by the Coder CLI. Provenance enables trust weighting.

  3. Temporal metadata with created_at, updated_at, and accessed_at. These drive the decay algorithm that keeps memory fresh. A decision from six months ago that nobody has accessed should fade. A pattern accessed daily should persist.

  4. Relational links via memory_links. A decision about using JWT tokens is related_to a threat about token replay attacks. A pattern for input validation evolved_from an earlier pattern. A new threat finding contradicts an old assessment. These links create a knowledge graph that agents can traverse.

Read/Write Semantics

The bus enforces clear ownership:

  • Any CLI can read any shared memory. Access is logged in memory_access_log.
  • Any CLI can write new memories. The written_by field records provenance.
  • Only the writing CLI can update its own memories. If the Security CLI wrote a threat assessment, the Coder CLI cannot modify it — it writes a new memory that contradicts or supersedes the original, linked via memory_links.
  • Archival is decay-driven. A memory is archived when its relevance score drops below threshold, or when two or more CLIs agree it is obsolete.

These semantics prevent the most dangerous failure mode in shared memory: one agent silently overwriting another agent's knowledge.

The Rust Memory Bus Implementation

use rusqlite::{Connection, params};
use chrono::Utc;
use uuid::Uuid;

pub struct MemoryBus {
    conn: Connection,
}

pub struct Memory {
    pub id: String,
    pub memory_type: MemoryType,
    pub key: String,
    pub value: String,
    pub importance: u8,
    pub written_by: String,
    pub created_at: String,
    pub updated_at: String,
    pub access_count: u32,
    pub decay_rate: f64,
    pub is_archived: bool,
}

#[derive(Debug, Clone, PartialEq)]
pub enum MemoryType {
    User,
    Feedback,
    Project,
    Reference,
    Decision,
    Pattern,
    Threat,
    TestIntel,
    PerfBaseline,
    DocGap,
}

impl MemoryType {
    pub fn as_str(&self) -> &str {
        match self {
            Self::User => "user",
            Self::Feedback => "feedback",
            Self::Project => "project",
            Self::Reference => "reference",
            Self::Decision => "decision",
            Self::Pattern => "pattern",
            Self::Threat => "threat",
            Self::TestIntel => "test_intel",
            Self::PerfBaseline => "perf_baseline",
            Self::DocGap => "doc_gap",
        }
    }

    /// Default decay half-life in days for each memory type
    pub fn decay_half_life_days(&self) -> f64 {
        match self {
            Self::Decision => 120.0,   // Architectural decisions persist
            Self::Pattern => 90.0,     // Implementation patterns last
            Self::User => 90.0,        // User context changes slowly
            Self::Feedback => 60.0,    // Behavioral guidance
            Self::Threat => 60.0,      // Security findings
            Self::Reference => 45.0,   // External pointers
            Self::Project => 30.0,     // Project context changes fast
            Self::TestIntel => 30.0,   // Test insights evolve
            Self::DocGap => 30.0,      // Doc gaps get filled
            Self::PerfBaseline => 14.0, // Performance data goes stale fast
        }
    }
}

impl MemoryBus {
    pub fn open(db_path: &str) -> Result<Self, rusqlite::Error> {
        let conn = Connection::open(db_path)?;
        conn.execute_batch(include_str!("schema.sql"))?;
        Ok(Self { conn })
    }

    pub fn write(
        &self,
        memory_type: MemoryType,
        key: &str,
        value: &str,
        written_by: &str,
        importance: u8,
    ) -> Result<String, MemoryError> {
        // Validate importance range
        if !(1..=5).contains(&importance) {
            return Err(MemoryError::InvalidImportance(importance));
        }

        // Check for existing memory with same type+key
        let existing: Option<Memory> = self.find_by_type_key(&memory_type, key)?;

        if let Some(existing) = existing {
            if existing.written_by == written_by {
                // Same author: update in place
                self.conn.execute(
                    "UPDATE memories SET value = ?1, importance = ?2,
                     updated_at = datetime('now') WHERE id = ?3",
                    params![value, importance, existing.id],
                )?;
                return Ok(existing.id);
            } else {
                // Different author: apply conflict resolution strategy
                return self.resolve_conflict(
                    &existing, &memory_type, key, value, written_by, importance,
                );
            }
        }

        // No conflict: insert new memory
        let id = Uuid::new_v4().to_string();
        self.conn.execute(
            "INSERT INTO memories (id, type, key, value, importance, written_by,
             decay_rate, created_at, updated_at)
             VALUES (?1, ?2, ?3, ?4, ?5, ?6, ?7, datetime('now'), datetime('now'))",
            params![
                id,
                memory_type.as_str(),
                key,
                value,
                importance,
                written_by,
                memory_type.decay_half_life_days() / 30.0, // normalize to base 30
            ],
        )?;
        Ok(id)
    }

    pub fn read(
        &self,
        memory_type: &MemoryType,
        key: &str,
        reader: &str,
    ) -> Result<Option<Memory>, MemoryError> {
        let memory = self.find_by_type_key(memory_type, key)?;

        if let Some(ref m) = memory {
            // Log access
            self.conn.execute(
                "INSERT INTO memory_access_log (memory_id, accessed_by)
                 VALUES (?1, ?2)",
                params![m.id, reader],
            )?;
            // Update access count and timestamp
            self.conn.execute(
                "UPDATE memories SET access_count = access_count + 1,
                 accessed_at = datetime('now') WHERE id = ?1",
                params![m.id],
            )?;
        }

        Ok(memory)
    }

    /// Query memories relevant to a specific CLI, ordered by relevance
    pub fn query_for_cli(
        &self,
        cli_name: &str,
        types: &[MemoryType],
        limit: usize,
    ) -> Result<Vec<Memory>, MemoryError> {
        let type_list: Vec<&str> = types.iter().map(|t| t.as_str()).collect();
        let placeholders = type_list.iter()
            .map(|_| "?")
            .collect::<Vec<_>>()
            .join(",");

        let sql = format!(
            "SELECT * FROM memories
             WHERE is_archived = 0 AND type IN ({})
             ORDER BY importance DESC, updated_at DESC
             LIMIT ?",
            placeholders
        );

        // Execute and collect results
        let memories = self.execute_query(&sql, &type_list, limit)?;

        // Log all accesses
        for m in &memories {
            self.conn.execute(
                "INSERT INTO memory_access_log (memory_id, accessed_by)
                 VALUES (?1, ?2)",
                params![m.id, cli_name],
            )?;
        }

        Ok(memories)
    }

    pub fn link(
        &self,
        from_id: &str,
        to_id: &str,
        relation: &str,
        strength: f64,
    ) -> Result<(), MemoryError> {
        self.conn.execute(
            "INSERT OR REPLACE INTO memory_links
             (from_id, to_id, relation, strength, created_at)
             VALUES (?1, ?2, ?3, ?4, datetime('now'))",
            params![from_id, to_id, relation, strength],
        )?;
        Ok(())
    }
}

This implementation enforces the ownership semantics at the API level. The write method checks for existing memories with the same type + key pair and routes to the conflict resolution engine when different authors collide. The read method logs every access, feeding data into the relevance scoring algorithm. The query_for_cli method returns memories filtered by type and ordered by importance — each CLI requests only the types it cares about.

14.3 The Eight Memory Layers of Claude Code

Before building the multi-CLI memory system, we need to understand what Claude Code already provides. The source code reveals eight distinct memory layers, each serving a different purpose at a different timescale. Our shared memory bus must integrate with — not replace — this native infrastructure.

Layer 1: CLAUDE.md — The Constitutional Layer

Location: ~/.claude/CLAUDE.md (user), .claude/CLAUDE.md (project)
Lifetime: Permanent (version-controlled)
Scope: All sessions in the project

CLAUDE.md is the oldest and most stable memory layer. It is loaded into the system prompt at every session start. For multi-CLI systems, CLAUDE.md serves as the constitutional layer — shared rules that all agents must follow. Each CLI loads the same project CLAUDE.md, ensuring behavioral consistency, but each can have its own overlay with agent-specific instructions that extend (never override) the project constitution. As discussed in Chapter 5, the CLAUDE.md hierarchy per CLI is: managed, user, project, local — with each layer able to add constraints but never remove them.

Layer 2: Session Persistence — The Transcript Archive

Location: ~/.claude/projects/<project>/sessions/<session-id>.jsonl
Lifetime: Indefinite (append-only JSONL)
Scope: Single session

Every conversation turn is persisted as a JSONL entry — user messages, assistant responses, tool calls, tool results. A typical hour-long coding session produces 50-200KB of JSONL data. These transcripts serve two purposes: session restoration (resume where you left off) and cross-session knowledge mining (the KAIROS_DREAM system reads these during consolidation).

Layer 3: memdir/ (MEMORY.md) — The Typed Memory Store

Location: ~/.claude/projects/<project>/memory/MEMORY.md (index)
         ~/.claude/projects/<project>/memory/*.md (topic files)
Lifetime: Persistent, pruned by consolidation
Size cap: 200 lines / 25KB for MEMORY.md

Claude Code's primary persistent memory system. The memdir/ module implements a file-based store with a typed taxonomy — four types: user, feedback, project, reference. Each memory file uses YAML frontmatter for structured metadata. The MEMORY.md entrypoint serves as an index — one line per memory, under 150 characters, linking to topic files. The hard caps (200 lines, 25KB) are enforced by truncateEntrypointContent() in memdir.ts. This is critical: MEMORY.md is injected into the system prompt every turn, so its size directly impacts available context for actual work.

Layer 4: Git as Memory

Location: .git/
Lifetime: Permanent
Scope: All sessions, all time

Claude Code injects git context into every turn: current branch, default branch, git status --short (capped at 2,000 characters), and the last five commits. For multi-CLI systems, git is the ultimate source of truth. When the memory bus records that the Coder fixed a vulnerability, and git shows the actual diff, the git evidence takes precedence. Memory complements git — it never replaces it.

Layer 5: Conversation Context — The Working Set

Location: In-memory (API message array)
Lifetime: Current session only
Size: Up to 200K tokens

The most ephemeral layer. Managed by seven compaction strategies (auto-compact, reactive compact, context collapse, micro-compact, snip compaction, manual /compact, and the 1M context opt-in). For multi-CLI systems, conversation context is strictly private — no agent should access another agent's in-flight conversation.

Layer 6: Key-Value Storage — Session State

Location: ~/.claude/projects/<project>/settings.json
Lifetime: Persistent
Scope: Project

Simple key-value pairs for configuration: model preferences, permission settings, feature flags. Stores per-agent configuration that persists across sessions but does not belong in the shared memory bus.

Layer 7: EXTRACT_MEMORIES — Automatic Knowledge Extraction

Feature flag: feature('EXTRACT_MEMORIES')
Trigger: Post-sampling hook (after each agent turn)
Source: services/extractMemories/extractMemories.ts

This is where Claude Code's memory transcends simple note-taking. A background subagent runs after every conversation turn, scanning the dialogue for knowledge worth preserving. We dissect this system in Section 14.4.

Layer 8: AGENT_MEMORY_SNAPSHOT — Cross-Agent State Transfer

Feature flag: feature('AGENT_MEMORY_SNAPSHOT')
Trigger: Agent session end
Source: tools/AgentTool/agentMemory.ts
Scopes: user (~/.claude/agent-memory/), project (.claude/agent-memory/), local (.claude/agent-memory-local/)

The newest memory layer. Captures an agent's accumulated knowledge at session end and makes it available to other agents — or to the same agent in future sessions. This is the closest thing Claude Code has to inter-agent memory sharing. Section 14.5 covers it in depth.

14.4 EXTRACT_MEMORIES: The Automatic Knowledge Miner

Most memory systems are passive — they store what you explicitly tell them. Claude Code's EXTRACT_MEMORIES is active. It mines knowledge from conversations without being asked.

The Extraction Pipeline

User message --> Agent response --> Post-sampling hook fires
                                            |
                                            v
                                 +------------------------+
                                 | EXTRACT_MEMORIES       |
                                 | gate check:            |
                                 |  feature flag enabled? |
                                 |  autoMemoryEnabled?    |
                                 +----------+-------------+
                                            | yes
                                            v
                                 +------------------------+
                                 | Count new messages     |
                                 | since last extraction. |
                                 | Skip if agent already  |
                                 | wrote to memory files. |
                                 +----------+-------------+
                                            |
                                            v
                                 +------------------------+
                                 | Fork subagent:         |
                                 |  - conversation tail   |
                                 |  - memory directory    |
                                 |  - typed memory rules  |
                                 |  - 5-turn max budget   |
                                 +----------+-------------+
                                            |
                                            v
                                 +------------------------+
                                 | Subagent writes to     |
                                 | memdir/ using          |
                                 | FileEdit/FileWrite     |
                                 | (restricted toolset)   |
                                 +------------------------+

The extraction subagent operates under heavy constraints. From createAutoMemCanUseTool in the source:

  • Read tools: GrepTool, GlobTool, ReadTool — it can examine the codebase for context
  • Write tools: FileEditTool, FileWriteTool — but ONLY to paths within the memory directory
  • Shell: Read-only bash commands only (ls, find, grep, cat, stat, wc, head, tail)
  • Budget: Maximum 5 agentic turns before forced stop

Everything else — write-access Bash, WebFetchTool, AgentTool, network operations — is denied. The subagent cannot modify code, cannot spawn other agents, cannot make external requests. It can only read the world and write to memory.

This constraint model is critical for multi-CLI systems. When eight CLIs each run extraction subagents concurrently, you need absolute certainty that an extraction agent cannot accidentally modify code or interfere with another CLI's work. The restricted toolset is a security boundary.

The Skip Heuristic and Coalescing Pattern

The pipeline includes an important optimization: if the main agent already wrote to memory files during the conversation (the user said "remember this"), the extraction subagent skips entirely. No double-writing.

When extraction is already running and another turn completes, the new context is stashed rather than spawning a second extraction agent. When the current extraction finishes, it checks for stashed context and runs a trailing pass. This coalescing pattern prevents resource contention — at most one extraction subagent runs per CLI at any time.

Multi-CLI Extraction Strategy

In the forge architecture, each CLI runs its own EXTRACT_MEMORIES pipeline writing to its private memory store. A separate memory aggregator process periodically scans private stores and promotes high-importance memories to the shared bus:

pub struct MemoryAggregator {
    bus: MemoryBus,
    private_stores: Vec<PrivateStore>,
    promote_threshold: u8, // importance >= this -> shared bus
}

pub struct PrivateStore {
    cli_name: String,
    memory_dir: PathBuf,
}

impl MemoryAggregator {
    /// Scan all private stores and promote high-importance memories
    pub fn aggregate(&self) -> Result<AggregationReport, MemoryError> {
        let mut promoted = 0;
        let mut skipped = 0;

        for store in &self.private_stores {
            let memory_files = glob_files(&store.memory_dir, "*.md");

            for file_path in memory_files {
                let content = std::fs::read_to_string(&file_path)?;
                let metadata = parse_frontmatter(&content)?;

                let importance: u8 = metadata
                    .get("importance")
                    .and_then(|v| v.parse().ok())
                    .unwrap_or(3);

                if importance >= self.promote_threshold {
                    let memory_type = metadata
                        .get("type")
                        .and_then(|t| MemoryType::from_str(t).ok())
                        .unwrap_or(MemoryType::Project);

                    let key = metadata
                        .get("title")
                        .cloned()
                        .unwrap_or_else(|| file_stem(&file_path));

                    // Check if this memory already exists on the bus
                    if self.bus.find_by_type_key(&memory_type, &key)?.is_none() {
                        self.bus.write(
                            memory_type,
                            &key,
                            &strip_frontmatter(&content),
                            &store.cli_name,
                            importance,
                        )?;
                        promoted += 1;
                    } else {
                        skipped += 1;
                    }
                }
            }
        }

        Ok(AggregationReport { promoted, skipped })
    }
}

This two-phase model — extract privately, promote selectively — keeps the shared bus clean. Low-importance observations stay in private stores. Only knowledge scoring importance 4 or higher reaches the bus where all CLIs can access it.

14.5 AGENT_MEMORY_SNAPSHOT: Cross-Agent State Transfer

While EXTRACT_MEMORIES mines knowledge from conversations, AGENT_MEMORY_SNAPSHOT captures the state of an agent — its accumulated understanding, active context, working hypotheses — and persists it for future use.

The feature is gated behind feature('AGENT_MEMORY_SNAPSHOT') and operates at session boundaries. The key insight is the three-scope memory model from agentMemory.ts:

Scope Location Purpose
user ~/.claude/agent-memory/<agentType>/MEMORY.md Cross-project expertise, global preferences
project .claude/agent-memory/<agentType>/MEMORY.md Project-specific decisions, conventions
local .claude/agent-memory-local/<agentType>/MEMORY.md Machine-specific configs (git-ignored)

For multi-CLI systems, the project scope is the bridge between agents. When the Coder CLI writes a snapshot to .claude/agent-memory/coder/MEMORY.md, the Security CLI can read it at the same path. Every agent's project-scope memory is readable by every other agent.

Snapshot Merge Strategy

When an agent session ends with a pending snapshot update, Claude Code runs a reasoning-based merge:

  1. Read current MEMORY.md
  2. Compare with knowledge gained in the just-completed session
  3. Update MEMORY.md with new learnings, resolving conflicts
  4. Prune outdated entries that the session's work superseded

This is not a blind append. If yesterday's snapshot says "the billing module uses Stripe v2 API" and today's session revealed a migration to Stripe v3, the agent updates — not duplicates — the memory.

The Rust Snapshot Manager

pub struct SnapshotManager {
    agent_type: String,
    scopes: Vec<SnapshotScope>,
}

pub struct SnapshotScope {
    pub scope: ScopeType,
    pub memory_dir: PathBuf,
}

#[derive(Debug, Clone)]
pub enum ScopeType {
    User,    // ~/.claude/agent-memory/<agent>/
    Project, // .claude/agent-memory/<agent>/
    Local,   // .claude/agent-memory-local/<agent>/
}

impl SnapshotManager {
    pub fn load_snapshot_prompt(&self) -> String {
        let mut prompt = String::new();

        for scope in &self.scopes {
            let memory_md = scope.memory_dir.join("MEMORY.md");
            if memory_md.exists() {
                let content = std::fs::read_to_string(&memory_md)
                    .unwrap_or_default();
                let scope_note = match scope.scope {
                    ScopeType::User => "General learnings (all projects):",
                    ScopeType::Project => "Project-specific learnings:",
                    ScopeType::Local => "Local machine context:",
                };
                prompt.push_str(&format!("\n## {}\n{}\n", scope_note, content));
            }
        }

        prompt
    }

    /// Merge new session knowledge into the snapshot
    pub fn merge_snapshot(
        &self,
        scope: ScopeType,
        session_learnings: &str,
    ) -> Result<(), MemoryError> {
        let target = self.scopes.iter()
            .find(|s| std::mem::discriminant(&s.scope) == std::mem::discriminant(&scope))
            .ok_or(MemoryError::ScopeNotFound)?;

        let memory_md = target.memory_dir.join("MEMORY.md");
        std::fs::create_dir_all(&target.memory_dir)?;

        let existing = if memory_md.exists() {
            std::fs::read_to_string(&memory_md)?
        } else {
            String::new()
        };

        // The merge is LLM-assisted: we build a prompt that asks the agent
        // to reconcile old and new knowledge, resolve contradictions,
        // and prune superseded entries.
        let merge_prompt = format!(
            "## Current memory:\n{}\n\n## New session learnings:\n{}\n\n\
             Merge these. Update existing entries when new information supersedes \
             old. Remove contradicted facts. Keep total under 200 lines.",
            existing, session_learnings
        );

        // This would be sent to the LLM for reasoning-based merge
        // For now, we append with a timestamp separator
        let merged = format!(
            "{}\n\n---\n*Updated: {}*\n\n{}",
            existing,
            Utc::now().format("%Y-%m-%d %H:%M"),
            session_learnings
        );

        std::fs::write(&memory_md, merged)?;
        Ok(())
    }
}

14.6 KAIROS_DREAM: Background Memory Consolidation

KAIROS_DREAM is the most ambitious feature in Claude Code's memory stack. It is a background consolidation agent that reviews accumulated session transcripts, extracts cross-session patterns, prunes stale knowledge, and reorganizes memories for optimal retrieval. It literally dreams — running as a forked subagent while the main session continues.

The Four-Stage Pipeline

From the consolidation prompt in services/autoDream/consolidationPrompt.ts:

Phase 1 — ORIENT. List the memory directory, read MEMORY.md, skim existing topic files, assess current state of knowledge.

Phase 2 — GATHER. Scan daily logs (logs/YYYY/MM/YYYY-MM-DD.md), check for drifted facts (memories contradicted by current code), grep session transcripts for narrow terms, identify new information worth persisting.

Phase 3 — CONSOLIDATE. Merge new signal into existing topic files, convert relative dates to absolute dates, delete contradicted facts, write new memories using typed format.

Phase 4 — PRUNE AND INDEX. Update MEMORY.md to stay under 200 lines / 25KB, remove pointers to stale memories, demote verbose entries (>200 chars) to topic files, add pointers to newly important memories, resolve contradictions between files.

Trigger Conditions

The dream agent does not run on every session. Three gates checked in order of computational cost:

pub struct DreamScheduler {
    min_hours_between: f64,     // default: 24
    min_sessions_since: usize,  // default: 5
    lock_path: PathBuf,
}

impl DreamScheduler {
    pub fn should_dream(&self) -> DreamDecision {
        // Gate 1: Time — enough hours since last consolidation?
        let hours_since = self.hours_since_last_consolidation();
        if hours_since < self.min_hours_between {
            return DreamDecision::Skip("Too soon since last consolidation");
        }

        // Gate 2: Volume — enough new sessions to justify consolidation?
        let new_sessions = self.sessions_since_last_consolidation();
        if new_sessions < self.min_sessions_since {
            return DreamDecision::Skip("Not enough new sessions");
        }

        // Gate 3: Lock — no other process mid-consolidation?
        match self.try_acquire_lock() {
            Ok(lock) => DreamDecision::Proceed(lock),
            Err(_) => DreamDecision::Skip("Another consolidation in progress"),
        }
    }

    fn try_acquire_lock(&self) -> Result<DreamLock, LockError> {
        // File-based lock checked by PID to detect stale locks
        if self.lock_path.exists() {
            let lock_content = std::fs::read_to_string(&self.lock_path)?;
            let lock_pid: u32 = lock_content.parse()?;
            if process_alive(lock_pid) {
                return Err(LockError::Held(lock_pid));
            }
            // Stale lock from crashed process — remove it
            std::fs::remove_file(&self.lock_path)?;
        }

        std::fs::write(&self.lock_path, std::process::id().to_string())?;
        Ok(DreamLock { path: self.lock_path.clone() })
    }
}

pub struct DreamLock {
    path: PathBuf,
}

impl Drop for DreamLock {
    fn drop(&mut self) {
        let _ = std::fs::remove_file(&self.path);
    }
}

Like the extraction subagent, the dream agent operates under restricted permissions: read-only bash, read tools, and write access only to the memory directory. The prompt explicitly instructs it to grep narrowly: "Don't exhaustively read transcripts. Look only for things you already suspect matter." A week's worth of transcripts could be megabytes — the dream agent has a finite context window.

Multi-CLI Dream Coordination

In the forge architecture, each CLI could trigger its own dream cycle. But eight simultaneous dream agents thrashing the same memory store is a recipe for conflicts. The solution: a single forge-level dream coordinator.

pub struct ForgeDreamCoordinator {
    coordinator_cli: String,      // "architect-cli" — broadest view
    scheduler: DreamScheduler,
    private_stores: Vec<PathBuf>, // All CLI private stores
    shared_bus: MemoryBus,
}

impl ForgeDreamCoordinator {
    /// Run by the Architect CLI only
    pub fn consolidate(&self) -> Result<ConsolidationReport, MemoryError> {
        let lock = match self.scheduler.should_dream() {
            DreamDecision::Proceed(lock) => lock,
            DreamDecision::Skip(reason) => return Ok(ConsolidationReport::skipped(reason)),
        };

        let mut report = ConsolidationReport::new();

        // Phase 1: Orient — read all private stores
        let private_memories = self.gather_private_memories()?;

        // Phase 2: Gather — identify cross-CLI patterns
        let cross_cutting = self.find_cross_cutting_patterns(&private_memories)?;

        // Phase 3: Consolidate — merge into shared bus
        for pattern in cross_cutting {
            self.shared_bus.write(
                pattern.memory_type,
                &pattern.key,
                &pattern.synthesized_value,
                &self.coordinator_cli,
                pattern.importance,
            )?;
            report.consolidated += 1;
        }

        // Phase 4: Prune — archive low-relevance shared memories
        let pruned = self.prune_shared_bus()?;
        report.pruned = pruned;

        // Resolve contradictions between CLI findings
        let resolved = self.resolve_contradictions()?;
        report.contradictions_resolved = resolved;

        // lock is automatically released when dropped
        Ok(report)
    }
}

The Architect CLI owns the dream cycle because it has the broadest view of the system. It reads all eight private stores plus the shared bus, consolidates cross-cutting patterns, resolves inter-agent contradictions, and updates the shared bus with synthesized knowledge.

14.7 Conflict Resolution

Two CLIs write to the shared memory bus simultaneously. Both target the same type + key pair. What happens?

The forge implements a three-tier resolution hierarchy.

Tier 1: Last-Write-Wins (Low-Stakes Memories)

For memory types where recency matters more than completeness — project status updates, reference pointers, perf_baseline measurements — the most recent write wins. The previous value is preserved in an audit log.

fn resolve_last_write_wins(
    &self,
    existing: &Memory,
    new_value: &str,
    new_writer: &str,
    importance: u8,
) -> Result<String, MemoryError> {
    // Audit trail: save the old value
    self.conn.execute(
        "INSERT INTO memory_audit (memory_id, previous_value, previous_written_by,
         replaced_at, replaced_by)
         VALUES (?1, ?2, ?3, datetime('now'), ?4)",
        params![existing.id, existing.value, existing.written_by, new_writer],
    )?;

    // Overwrite
    self.conn.execute(
        "UPDATE memories SET value = ?1, written_by = ?2, importance = ?3,
         updated_at = datetime('now') WHERE id = ?4",
        params![new_value, new_writer, importance, existing.id],
    )?;

    Ok(existing.id.clone())
}

Tier 2: Merge (Accumulative Memories)

For types where both contributions have value — pattern libraries, test_intel collections, doc_gap inventories — conflicts are merged by appending with authorship markers:

fn resolve_merge(
    &self,
    existing: &Memory,
    new_value: &str,
    new_writer: &str,
    importance: u8,
) -> Result<String, MemoryError> {
    let merged = format!(
        "{}\n\n---\n*Added by {} on {}:*\n\n{}",
        existing.value,
        new_writer,
        Utc::now().format("%Y-%m-%d"),
        new_value
    );

    self.conn.execute(
        "UPDATE memories SET value = ?1, importance = MAX(importance, ?2),
         updated_at = datetime('now') WHERE id = ?3",
        params![merged, importance, existing.id],
    )?;

    Ok(existing.id.clone())
}

Tier 3: Explicit Contradiction (High-Stakes Memories)

For types where disagreement is meaningful — decision rationales, threat assessments — conflicts create a contradicts link rather than overwriting:

fn resolve_contradiction(
    &self,
    existing: &Memory,
    memory_type: &MemoryType,
    key: &str,
    new_value: &str,
    new_writer: &str,
    importance: u8,
) -> Result<String, MemoryError> {
    // Create the new memory with a modified key
    let new_key = format!("{}__alt_{}", key, new_writer);
    let new_id = Uuid::new_v4().to_string();

    self.conn.execute(
        "INSERT INTO memories (id, type, key, value, importance, written_by,
         created_at, updated_at)
         VALUES (?1, ?2, ?3, ?4, ?5, ?6, datetime('now'), datetime('now'))",
        params![new_id, memory_type.as_str(), new_key, new_value, importance, new_writer],
    )?;

    // Link as contradiction
    self.link(&new_id, &existing.id, "contradicts", 1.0)?;

    Ok(new_id)
}

When the Security CLI rates an endpoint as "high risk" and the Coder CLI rates it as "mitigated," both assessments persist with a contradicts link. The next agent to query sees both and can make an informed judgment.

The conflict resolution strategy is selected by the resolve_conflict method:

impl MemoryBus {
    fn resolve_conflict(
        &self,
        existing: &Memory,
        memory_type: &MemoryType,
        key: &str,
        value: &str,
        writer: &str,
        importance: u8,
    ) -> Result<String, MemoryError> {
        match memory_type {
            // Tier 1: Last-write-wins
            MemoryType::Project
            | MemoryType::Reference
            | MemoryType::PerfBaseline => {
                self.resolve_last_write_wins(existing, value, writer, importance)
            }

            // Tier 2: Merge
            MemoryType::Feedback
            | MemoryType::Pattern
            | MemoryType::TestIntel
            | MemoryType::DocGap => {
                self.resolve_merge(existing, value, writer, importance)
            }

            // Tier 3: Contradiction
            MemoryType::Decision
            | MemoryType::Threat => {
                self.resolve_contradiction(
                    existing, memory_type, key, value, writer, importance,
                )
            }

            // Private types — should not reach shared bus
            MemoryType::User => Err(MemoryError::PrivateType),
        }
    }
}

14.8 Memory Relevance Scoring and Decay

Memory without decay is a hoarder's attic. Every fact ever recorded competing for tokens in the system prompt. Claude Code's 200-line / 25KB cap on MEMORY.md is a hard boundary against this. In a multi-CLI system where eight agents each generate memories at every turn, the problem is amplified eightfold.

The Relevance Score

Every memory carries a dynamic relevance score computed from five factors:

pub fn compute_relevance(memory: &Memory, now: &DateTime<Utc>) -> f64 {
    // Factor 1: Importance (user-assigned, 1-5)
    // Importance 5 gets 2x multiplier
    let importance_score = memory.importance as f64
        * if memory.importance == 5 { 2.0 } else { 1.0 };

    // Factor 2: Recency — exponential decay from last update
    let days_since = (*now - memory.updated_at)
        .num_seconds() as f64 / 86400.0;
    let half_life = 30.0 * memory.decay_rate;
    let recency_score = 2.0_f64.powf(-days_since / half_life);

    // Factor 3: Access frequency
    let access_score = (memory.access_count as f64 / 10.0).min(1.0);

    // Factor 4: Link density — well-connected memories are more valuable
    let link_count = memory.outgoing_links + memory.incoming_links;
    let link_score = (link_count as f64 / 5.0).min(1.0);

    // Factor 5: Cross-agent access — memories read by multiple CLIs
    let cross_agent_score = (memory.unique_readers as f64 / 4.0).min(1.0);

    // Weighted combination (range: 0.0 to 10.0)
    (importance_score * 0.30
        + recency_score * 0.25
        + access_score * 0.20
        + link_score * 0.10
        + cross_agent_score * 0.15)
        * 10.0
}

The cross_agent_score factor is unique to multi-CLI systems. It captures a signal that does not exist in single-agent architectures: if the Coder, Security CLI, and Reviewer all read the same decision memory, that decision is load-bearing knowledge that should resist decay.

The Pruning Cycle

Pruning runs as part of the KAIROS_DREAM consolidation:

pub fn prune_shared_bus(
    bus: &MemoryBus,
    threshold: f64,
) -> Result<PruneReport, MemoryError> {
    let now = Utc::now();
    let candidates = bus.query_all_active()?;
    let mut archived = 0;
    let mut demoted = 0;

    for memory in &candidates {
        let score = compute_relevance(memory, &now);

        if score < threshold {
            // Check: does any other memory depend on this one?
            let dependents = bus.count_dependents(&memory.id)?;

            if dependents > 0 {
                // Don't archive — reduce importance to slow future decay
                bus.reduce_importance(&memory.id)?;
                demoted += 1;
            } else {
                // Safe to archive (removed from system prompt, kept in DB)
                bus.archive(&memory.id)?;
                archived += 1;
            }
        }
    }

    Ok(PruneReport { archived, demoted })
}

The dependency check is critical. If Memory A ("SQL injection in user search") is referenced by Memory B ("all search endpoints must use parameterized queries"), archiving A would leave B without justification. The algorithm reduces A's importance instead, slowing its decay while preserving the reference chain.

Archived memories are not deleted. They are removed from system prompt injection (saving tokens) but remain in the database, queryable through explicit search. If a future conversation matches an archived memory, it can be resurrected — its is_archived flag cleared and timestamps refreshed, restarting the decay clock.

14.9 System Prompt Injection: Top-N Memory Loading

The shared bus automatically injects the highest-relevance memories into each CLI's system prompt at session start. This is the auto-sync pattern: each CLI begins every session with the most important shared knowledge already in context — no explicit memory queries needed.

pub struct MemoryInjector {
    bus: MemoryBus,
    injection_limit: usize, // default: 30
}

impl MemoryInjector {
    /// Build the memory block for a CLI's system prompt
    pub fn build_injection(
        &self,
        cli_name: &str,
        readable_types: &[MemoryType],
    ) -> String {
        let now = Utc::now();

        // Query all non-archived memories of readable types
        let memories = self.bus
            .query_for_cli(cli_name, readable_types, 500)
            .unwrap_or_default();

        // Score and sort by relevance
        let mut scored: Vec<(f64, &Memory)> = memories.iter()
            .map(|m| (compute_relevance(m, &now), m))
            .collect();
        scored.sort_by(|a, b| b.0.partial_cmp(&a.0).unwrap());

        // Take top N
        let top_n: Vec<_> = scored.into_iter()
            .take(self.injection_limit)
            .collect();

        if top_n.is_empty() {
            return String::new();
        }

        let mut output = String::from("<shared-memory>\n");

        for (score, memory) in &top_n {
            output.push_str(&format!(
                "## [{}] {} (by {}, relevance: {:.1})\n{}\n\n",
                memory.memory_type.as_str(),
                memory.key,
                memory.written_by,
                score,
                memory.value,
            ));
        }

        output.push_str("</shared-memory>\n");
        output
    }
}

Each CLI's system prompt is seeded with the highest-relevance memories from the shared bus, filtered to only the types that CLI is configured to read. The Coder starts every session knowing the most important decisions, patterns, and threats. The Security CLI starts every session knowing the most critical vulnerabilities and their current status.

14.12 Non-Engineering Domains

The shared memory bus pattern is not specific to software engineering. Any domain where multiple specialists collaborate over time on the same body of work faces the same three-dimensional memory problem.

Medical Research Teams. A clinical research team has specialists — biostatisticians, principal investigators, data safety monitoring boards, regulatory affairs. The biostatistician's analysis of adverse event data should inform the PI's dosing decisions. The DSMB's safety findings should propagate to every team member. The memory bus maps directly: threat becomes safety_signal, decision becomes protocol_amendment, pattern becomes statistical_method. The conflict resolution hierarchy ensures the DSMB's safety assessment overrides the PI's efficacy optimism — authority based on role, not recency.

Legal Case Teams. Paralegals research precedent, associates draft motions, partners review strategy. Each specialist discovers facts that affect the others' work. The paralegal's discovery of a key deposition excerpt should inform the associate's brief. The partner's strategic decision to pursue a particular theory of liability should constrain the paralegal's research focus. The KAIROS_DREAM pattern maps to weekly case review meetings — consolidating findings, resolving contradictions between witnesses, pruning superseded theories.

Intelligence Analysis. Multiple analysts examine the same situation from different specializations — signals intelligence, human intelligence, open-source intelligence. Each analyst's findings inform the others' interpretations. The explicit contradiction pattern (Tier 3 conflict resolution) is critical: when SIGINT suggests one conclusion and HUMINT suggests another, the disagreement itself is the most important intelligence product. Silently resolving conflicts in intelligence analysis gets people killed.

14.14 The Memory Manifesto

We close with the principles that should guide every memory architecture decision in the multi-CLI forge.

Principle 1: Memory is a tax. Every memory injected into the system prompt consumes tokens that could be used for reasoning. The best memory system is the smallest one that preserves essential knowledge. If a memory can be derived from the codebase, do not store it — git log and grep are faster and more accurate than recalled facts.

Principle 2: Conflicts are signal, not noise. When two agents disagree, the disagreement is information. Never silently resolve conflicts. Surface them. Link them. Let the next agent — or the human — decide.

Principle 3: Decay is a feature. Knowledge that is not accessed is knowledge that is not needed. Aggressive decay with easy resurrection beats permanent storage with token-wasting clutter.

Principle 4: Provenance matters. A threat finding from the Security CLI carries more weight than one from the Docs CLI. Track who wrote what. Use provenance to weight trust.

Principle 5: Private before shared. Not every observation deserves a place on the shared bus. Extract privately, promote selectively. The shared bus should contain knowledge that changes behavior across multiple agents.

Principle 6: The dream is essential. Without consolidation, memory grows without bound. The KAIROS_DREAM pattern — periodic review, synthesis, pruning — is infrastructure, not overhead.

Principle 7: Memory serves the mission. Every memory should pass one question: "Does this change what an agent would do?" If the answer is no, do not store it.
