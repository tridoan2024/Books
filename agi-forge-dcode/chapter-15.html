Chapter 15: Inter-CLI Communication Protocol

Mailboxes, Unix Domain Sockets, MCP Transport, and the Message Envelope

Eight processes. Eight independent Claude Code instances, each with its own context window, tool registry, and streaming agent loop. They share a filesystem. They share a goal. They share nothing else.

Chapter 14 gave us shared memory — a passive layer where CLIs read and write state to common locations on disk. Memory answers the question what does everyone know? But it cannot answer the question that matters more: what should everyone do next?

For that, you need communication. Active, directed, reliable communication. Not "write a file and hope someone reads it" — real messaging. The Orchestrator tells the Coder to start implementing. The Security CLI tells the Orchestrator it found a critical vulnerability. The QA CLI broadcasts that the test database is corrupted. The Architect asks the Coder a question and blocks until it gets an answer.

This is inter-process communication, one of the oldest problems in computer science. But multi-CLI AI agent systems add a twist that Dijkstra never contemplated in 1965: the processes doing the communicating are stochastic. They hallucinate. They misinterpret instructions. They sometimes decide, mid-task, that the task has changed. Building a communication protocol for agents that might go off-script at any moment is a different engineering problem than building one for deterministic programs.

Claude Code's source reveals three distinct communication layers. The mailbox system — file-based JSON inboxes with advisory locking — handles teammate-to-teammate messaging within a swarm. The SendMessage tool provides a high-level interface that routes messages by name, handles structured protocol messages like shutdown requests, and bridges between in-process and tmux-based teammates. And the UDS_INBOX feature — Unix Domain Sockets gated behind a feature flag — enables zero-overhead peer-to-peer messaging between any two Claude Code sessions on the same machine.

This chapter dissects all three. Then we build the Rust implementation of a complete multi-CLI communication stack.

15.1 The Communication Problem

Multi-CLI systems face communication challenges that single-agent systems never encounter. Consider the reference architecture from Chapter 1. The Orchestrator receives: "Build a REST API for user authentication with OAuth2, rate limiting, and comprehensive test coverage." It decomposes into subtasks and dispatches:

Orchestrator CLI
  +-- Research CLI  -> "Survey OAuth2 libraries, compare passport vs oauth2-server"
  +-- Planner CLI   -> "Create implementation DAG with dependencies"
  +-- Coder CLI     -> "Implement API routes and middleware"
  +-- Security CLI  -> "Audit implementation for OWASP Top 10"
  +-- QA CLI        -> "Write integration tests for all endpoints"
  +-- Docs CLI      -> "Generate OpenAPI spec and developer guide"

This decomposition creates six simultaneous communication needs:

  1. Task dispatch. The Orchestrator sends detailed instructions to each specialist. One-to-one, directed, potentially hundreds of tokens.

  2. Progress reporting. Each specialist periodically informs the Orchestrator of status. Fire-and-forget — no acknowledgment needed.

  3. Dependency notification. The Planner tells the Coder when the DAG is ready. The Coder tells Security when code is committed. Event-driven, triggered by state transitions.

  4. Queries. Security finds a suspicious pattern in auth middleware. It asks the Coder: "Why eval() instead of JSON.parse() on line 47?" Request/response — Security pauses until the Coder answers.

  5. Broadcasts. QA discovers the test database is corrupted. Every CLI needs to know immediately. One-to-many, urgent.

  6. Coordination. Coder and Docs both want to modify README.md. They need a primitive — a lock or lease — to avoid clobbering each other's changes.

No single communication pattern handles all six. This is why Claude Code implements multiple layers.

Why Shared Files Are Not Enough

The naive approach — CLI A writes a JSON file, CLI B polls for it — fails in production for four reasons:

Race conditions. Two CLIs write to the same file simultaneously. One write is silently lost. Claude Code's tasks.ts uses proper-lockfile with exponential backoff (30 retries, 5ms minimum, 100ms maximum) specifically because a 10-agent swarm creates enough contention to blow past naive locking.

Polling overhead. There is no inotify for AI agents. Each CLI must poll at some interval. Claude Code's mailbox polls at tool-round boundaries — the pause between tool executions. Messages can be delayed by the duration of a tool execution (30+ seconds for complex bash commands).

No delivery guarantees. Files can be read before fully written. Files can be deleted before the sender confirms delivery. Files can be orphaned when a CLI crashes mid-write.

No addressing. Files do not have a "to" field. You need a naming convention (inbox-coder.json) and every CLI must know it. Fragile and does not scale.

Claude Code builds an entire messaging layer on top of the filesystem. The distinction between "using the filesystem as a messaging layer" and "building a messaging layer on the filesystem" matters enormously.

15.2 Three Communication Patterns

Every message between CLIs falls into one of three patterns, each demanding different infrastructure.

Pattern 1: Request/Response

The caller sends a message and waits for a reply. The most intuitive pattern and the most dangerous for agent systems.

Security CLI                          Coder CLI
    |                                     |
    +---- "Why eval() on line 47?" ------>|
    |                                     +-- [reads code, formulates answer]
    |<---- "Legacy compat, will fix" -----+
    |                                     |

Claude Code's SendMessage tool supports structured messages with request_id fields for correlation. From SendMessageTool.ts:

const StructuredMessage = z.discriminatedUnion('type', [
  z.object({
    type: z.literal('shutdown_request'),
    reason: z.string().optional(),
  }),
  z.object({
    type: z.literal('shutdown_response'),
    request_id: z.string(),
    approve: semanticBoolean(),
    reason: z.string().optional(),
  }),
  z.object({
    type: z.literal('plan_approval_response'),
    request_id: z.string(),
    approve: semanticBoolean(),
    feedback: z.string().optional(),
  }),
])

Deadlock danger. If CLI A sends a request to CLI B, and CLI B simultaneously sends a request to CLI A, both block forever. Agent systems are especially prone because the LLM does not understand blocking semantics. Mitigation: every request/response exchange must have a timeout.

Pattern 2: Fire-and-Forget

The sender emits a message and continues immediately. No reply expected.

When to use: Status updates, progress reports, task completion notifications, idle notifications. Agent teams' idle notifications are the purest example — after every turn, a teammate sends an idle notification to the lead. No acknowledgment needed. This creates a heartbeat-like pattern without explicit heartbeat infrastructure.

Pattern 3: Pub/Sub (Broadcast)

One sender, multiple receivers. Claude Code supports "*" as the recipient in SendMessage, which broadcasts to all teammates. The prompt warns that broadcast is expensive — linear in team size. In a 10-agent swarm, a broadcast generates 10 inbox writes, each requiring lock acquisition.

15.3 Transport Layer: Three Mechanisms

Communication patterns describe what agents need to say. Transport mechanisms describe how the bytes move.

Transport 1: File-Based Mailbox

Claude Code's primary communication transport for agent teams. Every teammate gets an inbox file:

~/.claude/teams/{team-name}/inboxes/
  +-- team-lead.json
  +-- researcher.json
  +-- implementer.json
  +-- security-auditor.json

Each inbox is a JSON array of TeammateMessage objects:

#[derive(Debug, Serialize, Deserialize)]
pub struct TeammateMessage {
    pub from: String,        // Sender agent name
    pub text: String,        // Message content
    pub timestamp: String,   // ISO 8601
    pub read: bool,          // Consumed?
    pub color: Option<String>,
    pub summary: Option<String>, // 5-10 word UI preview
}

The critical engineering is in the locking. The write protocol:

  1. Acquire advisory lock on {agent-name}.json
  2. Read existing messages
  3. Append new message
  4. Write entire array back
  5. Release lock

This read-modify-write cycle is not atomic at the filesystem level. The lock makes it safe. Without it, concurrent writes would each read the same array, each append, and one would clobber the other's append.

use std::fs;
use std::path::{Path, PathBuf};
use fs2::FileExt; // Cross-platform file locking

pub struct Mailbox {
    inbox_dir: PathBuf,
}

impl Mailbox {
    pub fn new(team_dir: &Path) -> Self {
        let inbox_dir = team_dir.join("inboxes");
        fs::create_dir_all(&inbox_dir).ok();
        Self { inbox_dir }
    }

    pub fn send(
        &self,
        to: &str,
        message: TeammateMessage,
    ) -> Result<(), MailboxError> {
        let inbox_path = self.inbox_dir.join(format!("{}.json", to));
        let lock_path = self.inbox_dir.join(format!("{}.lock", to));

        // Acquire lock with retry
        let lock_file = fs::OpenOptions::new()
            .create(true)
            .write(true)
            .open(&lock_path)?;

        let mut retries = 0;
        let max_retries = 30;
        loop {
            match lock_file.try_lock_exclusive() {
                Ok(()) => break,
                Err(_) if retries < max_retries => {
                    retries += 1;
                    let backoff = std::cmp::min(5 * (1 << retries), 100);
                    std::thread::sleep(std::time::Duration::from_millis(backoff));
                }
                Err(e) => return Err(MailboxError::LockFailed(e.to_string())),
            }
        }

        // Read-modify-write under lock
        let mut messages: Vec<TeammateMessage> = if inbox_path.exists() {
            let data = fs::read_to_string(&inbox_path)?;
            serde_json::from_str(&data).unwrap_or_default()
        } else {
            Vec::new()
        };

        messages.push(message);
        fs::write(&inbox_path, serde_json::to_string_pretty(&messages)?)?;

        // Release lock
        lock_file.unlock()?;
        Ok(())
    }

    pub fn receive(&self, agent_name: &str) -> Result<Vec<TeammateMessage>, MailboxError> {
        let inbox_path = self.inbox_dir.join(format!("{}.json", agent_name));
        let lock_path = self.inbox_dir.join(format!("{}.lock", agent_name));

        if !inbox_path.exists() {
            return Ok(Vec::new());
        }

        let lock_file = fs::OpenOptions::new()
            .create(true)
            .write(true)
            .open(&lock_path)?;
        lock_file.lock_exclusive()?;

        let data = fs::read_to_string(&inbox_path)?;
        let mut messages: Vec<TeammateMessage> = serde_json::from_str(&data)?;

        // Mark unread messages as read, collect them
        let unread: Vec<TeammateMessage> = messages.iter()
            .filter(|m| !m.read)
            .cloned()
            .collect();

        for msg in &mut messages {
            msg.read = true;
        }

        fs::write(&inbox_path, serde_json::to_string_pretty(&messages)?)?;
        lock_file.unlock()?;

        Ok(unread)
    }

    /// Broadcast to all inboxes in the team
    pub fn broadcast(
        &self,
        from: &str,
        text: &str,
        summary: Option<&str>,
    ) -> Result<usize, MailboxError> {
        let mut sent = 0;
        for entry in fs::read_dir(&self.inbox_dir)? {
            let entry = entry?;
            let path = entry.path();
            if path.extension().map(|e| e == "json").unwrap_or(false) {
                let agent = path.file_stem()
                    .and_then(|s| s.to_str())
                    .unwrap_or("");
                if agent != from {
                    let msg = TeammateMessage {
                        from: from.to_string(),
                        text: text.to_string(),
                        timestamp: chrono::Utc::now().to_rfc3339(),
                        read: false,
                        color: None,
                        summary: summary.map(String::from),
                    };
                    self.send(agent, msg)?;
                    sent += 1;
                }
            }
        }
        Ok(sent)
    }
}

Message delivery timing: Tmux-based teammates poll their inbox at startup and between turns. In-process teammates receive messages through queuePendingMessage — direct injection into the agent's message queue, bypassing the filesystem entirely. Sub-millisecond delivery for co-located agents, turn-bounded latency for tmux agents.

Transport 2: Unix Domain Sockets (UDS_INBOX)

Behind the UDS_INBOX feature flag, Claude Code implements peer-to-peer messaging over Unix Domain Sockets. Each instance creates a listener at /tmp/cc-socks/{pid}.sock.

Three advantages over file-based mailboxes:

  1. Zero polling. Event-driven — the kernel notifies immediately when a message arrives.
  2. Bidirectional. A single connection supports send and receive. Natural for request/response.
  3. No serialization overhead. No read-modify-write cycle. Raw bytes stream directly.
use std::os::unix::net::{UnixListener, UnixStream};
use std::io::{Read, Write, BufReader, BufRead};

pub struct UdsTransport {
    socket_path: PathBuf,
    listener: Option<UnixListener>,
}

impl UdsTransport {
    pub fn bind(socket_dir: &Path, pid: u32) -> Result<Self, TransportError> {
        let socket_path = socket_dir.join(format!("{}.sock", pid));
        // Clean up stale socket from crashed process
        if socket_path.exists() {
            fs::remove_file(&socket_path)?;
        }
        let listener = UnixListener::bind(&socket_path)?;
        listener.set_nonblocking(true)?;
        Ok(Self { socket_path, listener: Some(listener) })
    }

    pub fn send_to(target: &Path, envelope: &MessageEnvelope) -> Result<(), TransportError> {
        let mut stream = UnixStream::connect(target)?;
        let payload = serde_json::to_vec(envelope)?;
        // Length-prefixed framing: 4 bytes big-endian length + payload
        let len = (payload.len() as u32).to_be_bytes();
        stream.write_all(&len)?;
        stream.write_all(&payload)?;
        stream.flush()?;
        Ok(())
    }

    pub fn try_receive(&self) -> Result<Option<MessageEnvelope>, TransportError> {
        let listener = self.listener.as_ref()
            .ok_or(TransportError::NotBound)?;

        match listener.accept() {
            Ok((stream, _addr)) => {
                let mut reader = BufReader::new(stream);
                // Read length prefix
                let mut len_buf = [0u8; 4];
                reader.read_exact(&mut len_buf)?;
                let len = u32::from_be_bytes(len_buf) as usize;
                // Read payload
                let mut payload = vec![0u8; len];
                reader.read_exact(&mut payload)?;
                let envelope: MessageEnvelope = serde_json::from_slice(&payload)?;
                Ok(Some(envelope))
            }
            Err(ref e) if e.kind() == std::io::ErrorKind::WouldBlock => Ok(None),
            Err(e) => Err(TransportError::Io(e)),
        }
    }
}

impl Drop for UdsTransport {
    fn drop(&mut self) {
        let _ = fs::remove_file(&self.socket_path);
    }
}

The addressing scheme supports three formats: "researcher" (teammate name, routes through mailbox), "uds:/path/to/socket" (direct UDS connection), and "bridge:session_id" (remote relay). When a message arrives over UDS, it is wrapped in an XML envelope for the receiving agent:

<cross-session-message from="uds:/tmp/cc-socks/1234.sock">
  Check if tests pass over there
</cross-session-message>

The from attribute enables reply routing — the recipient copies from into to for the response.

Delivery guarantees: UDS provides reliable, in-order delivery within a connection. If the recipient crashes, the sender gets an immediate error — strictly stronger than file-based mailboxes where a crash leaves orphaned inbox files.

Transport 3: MCP as Communication Backbone

The Model Context Protocol makes tool invocation the messaging primitive. Instead of sending a text message "run the security audit," CLI A calls the security_audit tool exposed by CLI B's MCP server. Parameters are the payload. The result is the response.

This collapses the distinction between "sending a message" and "invoking a capability" — the most natural communication model for AI agents, which already think in terms of tool calls.

MCP supports three wire protocols:

Protocol Latency Deployment
stdio ~1ms Parent spawns child process
HTTP/SSE ~50-200ms HTTP server + Server-Sent Events
WebSocket ~5-20ms Bridge/remote sessions

Wire CLIs together via MCP:

{
  "mcpServers": {
    "security-cli": {
      "command": "rcode",
      "args": ["--mcp", "--agent", "security"],
      "env": { "RCODE_SECURITY_MODE": "true" }
    },
    "qa-cli": {
      "command": "rcode",
      "args": ["--mcp", "--agent", "qa"],
      "env": { "RCODE_QA_MODE": "true" }
    }
  }
}

Each spawned MCP server is a fully functional CLI instance, running headless, exposing its tools via stdio. The Orchestrator calls their tools as if they were local — each call executes in a separate process with a separate context window.

The MCP approach has a profound advantage over mailbox and UDS: tool schemas are the API contract. When the Security CLI exposes a run_security_audit tool via MCP, the tool's JSON schema defines exactly what parameters it accepts and what it returns. The Orchestrator does not need to construct a free-form text message hoping the Security CLI interprets it correctly — it calls a typed function. This eliminates an entire class of communication failures: misunderstood messages, missing parameters, ambiguous instructions.

The Rust implementation wraps MCP tool calls in a communication abstraction:

pub struct McpTransport {
    servers: HashMap<String, McpServerHandle>,
}

pub struct McpServerHandle {
    pub cli_name: String,
    pub process: std::process::Child,
    pub stdin: std::process::ChildStdin,
    pub stdout: BufReader<std::process::ChildStdout>,
}

impl McpTransport {
    pub fn spawn_cli(
        &mut self,
        cli_name: &str,
        command: &str,
        args: &[&str],
        env: &HashMap<String, String>,
    ) -> Result<(), TransportError> {
        let mut cmd = std::process::Command::new(command);
        cmd.args(args)
            .envs(env)
            .stdin(std::process::Stdio::piped())
            .stdout(std::process::Stdio::piped())
            .stderr(std::process::Stdio::null());

        let mut child = cmd.spawn()?;
        let stdin = child.stdin.take().unwrap();
        let stdout = BufReader::new(child.stdout.take().unwrap());

        self.servers.insert(cli_name.to_string(), McpServerHandle {
            cli_name: cli_name.to_string(),
            process: child,
            stdin,
            stdout,
        });
        Ok(())
    }

    pub fn call_tool(
        &mut self,
        cli_name: &str,
        tool_name: &str,
        params: serde_json::Value,
    ) -> Result<serde_json::Value, TransportError> {
        let server = self.servers.get_mut(cli_name)
            .ok_or(TransportError::ServerNotFound(cli_name.to_string()))?;

        // Send JSON-RPC request over stdio
        let request = serde_json::json!({
            "jsonrpc": "2.0",
            "id": uuid::Uuid::new_v4().to_string(),
            "method": "tools/call",
            "params": {
                "name": tool_name,
                "arguments": params
            }
        });

        let request_bytes = serde_json::to_vec(&request)?;
        server.stdin.write_all(&request_bytes)?;
        server.stdin.write_all(b"\n")?;
        server.stdin.flush()?;

        // Read JSON-RPC response
        let mut line = String::new();
        server.stdout.read_line(&mut line)?;
        let response: serde_json::Value = serde_json::from_str(&line)?;

        Ok(response)
    }
}

Transport Comparison and Selection

Criterion Mailbox UDS MCP (stdio)
Setup cost None (filesystem) Socket bind Process spawn
Latency 50-500ms (polling) <1ms ~1ms
Bidirectional No (separate inbox) Yes Yes
Typed interface No (raw text) No (raw bytes) Yes (JSON Schema)
Cross-machine No No Yes (HTTP/SSE)
Crash detection No signal Connection error Process exit
Concurrent safety Advisory locks Kernel-managed Process isolation

The right transport depends on the deployment: - Same process, same machine: In-memory queue (fastest, zero overhead) - Different processes, same machine: UDS (fast, bidirectional, crash detection) - Different processes, shared filesystem: Mailbox (universal fallback) - Different machines: MCP over HTTP/SSE or WebSocket

15.4 The Typed Message Envelope

Raw strings are not enough for production systems. You need structured messages with metadata for routing, correlation, error handling, and debugging.

The Problem

Without correlation IDs, messages are ambiguous. If the Coder is working on multiple tasks, "fix these 3 issues" could refer to auth issues or rate limiting issues. Context is implicit in conversation history, not explicit in the message.

The Envelope

use serde::{Serialize, Deserialize};
use uuid::Uuid;
use chrono::{DateTime, Utc};

#[derive(Debug, Serialize, Deserialize, Clone)]
pub struct MessageEnvelope {
    // Identity
    pub id: String,
    pub correlation_id: Option<String>,
    pub causation_id: Option<String>,

    // Routing
    pub sender: AgentAddress,
    pub recipient: RecipientAddress,

    // Payload
    pub message_type: MessageType,
    pub payload: serde_json::Value,
    pub summary: Option<String>,

    // Metadata
    pub timestamp: DateTime<Utc>,
    pub ttl_seconds: Option<u64>,
    pub priority: Priority,
    pub requires_response: bool,

    // Delivery
    pub delivery_attempts: u32,
    pub max_retries: u32,
    pub acknowledged_at: Option<DateTime<Utc>>,
}

#[derive(Debug, Serialize, Deserialize, Clone)]
pub struct AgentAddress {
    pub agent_id: String,
    pub agent_name: String,
    pub team_name: String,
    pub session_id: Option<String>,
}

#[derive(Debug, Serialize, Deserialize, Clone)]
pub struct RecipientAddress {
    pub agent_name: String,       // or "*" for broadcast
    pub team_name: Option<String>,
    pub socket_path: Option<String>,
    pub bridge_id: Option<String>,
}

#[derive(Debug, Serialize, Deserialize, Clone)]
pub enum MessageType {
    TaskAssignment,
    TaskUpdate,
    TaskCompleted,
    Query,
    QueryResponse,
    Broadcast,
    ShutdownRequest,
    ShutdownResponse,
    PlanApprovalRequest,
    PlanApprovalResponse,
    Error,
    Heartbeat,
    HeartbeatAck,
}

#[derive(Debug, Serialize, Deserialize, Clone, PartialEq, Eq, PartialOrd, Ord)]
pub enum Priority {
    Low,
    Normal,
    High,
    Critical,
}

impl MessageEnvelope {
    pub fn new(
        sender: AgentAddress,
        recipient: RecipientAddress,
        message_type: MessageType,
        payload: serde_json::Value,
    ) -> Self {
        Self {
            id: Uuid::new_v4().to_string(),
            correlation_id: None,
            causation_id: None,
            sender,
            recipient,
            message_type,
            payload,
            summary: None,
            timestamp: Utc::now(),
            ttl_seconds: None,
            priority: Priority::Normal,
            requires_response: false,
            delivery_attempts: 0,
            max_retries: 3,
            acknowledged_at: None,
        }
    }

    pub fn with_correlation(mut self, id: &str) -> Self {
        self.correlation_id = Some(id.to_string());
        self
    }

    pub fn with_causation(mut self, id: &str) -> Self {
        self.causation_id = Some(id.to_string());
        self
    }

    pub fn is_expired(&self) -> bool {
        if let Some(ttl) = self.ttl_seconds {
            let elapsed = (Utc::now() - self.timestamp).num_seconds() as u64;
            elapsed > ttl
        } else {
            false
        }
    }

    pub fn is_broadcast(&self) -> bool {
        self.recipient.agent_name == "*"
    }
}

The correlation_id traces message flows across agents. When the Orchestrator dispatches a task, it generates a correlation ID. Every subsequent message — progress, queries, errors, completion — carries it. This enables conversation threading, latency measurement, failure analysis, and audit logging.

The causation_id creates a causal chain: Message B was caused by Message A. Message C was caused by Message B. You can reconstruct the exact sequence that led to any outcome — invaluable for debugging multi-agent failures.

15.5 The Message Router

The router resolves recipient addresses and selects transport:

pub struct MessageRouter {
    mailbox: Mailbox,
    uds: Option<UdsTransport>,
    team_config: TeamConfig,
}

pub struct TeamConfig {
    pub name: String,
    pub members: Vec<TeamMember>,
}

pub struct TeamMember {
    pub agent_id: String,
    pub agent_name: String,
    pub backend: BackendType,
    pub socket_path: Option<String>,
    pub is_active: bool,
}

#[derive(Debug, Clone, PartialEq)]
pub enum BackendType {
    InProcess,
    Tmux,
    ITerm2,
}

impl MessageRouter {
    pub fn route(&self, envelope: &MessageEnvelope) -> Result<DeliveryReport, RoutingError> {
        // Check TTL
        if envelope.is_expired() {
            return Err(RoutingError::MessageExpired);
        }

        // Broadcast expansion
        if envelope.is_broadcast() {
            return self.route_broadcast(envelope);
        }

        // Direct UDS address
        if let Some(ref socket_path) = envelope.recipient.socket_path {
            return self.route_uds(envelope, socket_path);
        }

        // Name resolution
        let recipient_name = &envelope.recipient.agent_name;
        let member = self.team_config.members.iter()
            .find(|m| m.agent_name == *recipient_name)
            .ok_or(RoutingError::RecipientNotFound(recipient_name.clone()))?;

        if !member.is_active {
            return Err(RoutingError::RecipientInactive(recipient_name.clone()));
        }

        // Transport selection based on backend type
        match member.backend {
            BackendType::InProcess => {
                // Direct memory injection — handled by engine, not router
                Ok(DeliveryReport::queued_in_process())
            }
            BackendType::Tmux | BackendType::ITerm2 => {
                // File-based mailbox
                if let Some(ref socket) = member.socket_path {
                    if self.uds.is_some() {
                        // Prefer UDS when available
                        return self.route_uds(envelope, socket);
                    }
                }
                self.route_mailbox(envelope, &member.agent_name)
            }
        }
    }

    fn route_broadcast(&self, envelope: &MessageEnvelope) -> Result<DeliveryReport, RoutingError> {
        let sender = &envelope.sender.agent_name;
        let mut delivered = 0;
        let mut failed = 0;

        for member in &self.team_config.members {
            if member.agent_name != *sender && member.is_active {
                let mut targeted = envelope.clone();
                targeted.recipient.agent_name = member.agent_name.clone();

                match self.route(&targeted) {
                    Ok(_) => delivered += 1,
                    Err(_) => failed += 1,
                }
            }
        }

        Ok(DeliveryReport { delivered, failed })
    }

    fn route_mailbox(
        &self,
        envelope: &MessageEnvelope,
        agent_name: &str,
    ) -> Result<DeliveryReport, RoutingError> {
        let msg = TeammateMessage {
            from: envelope.sender.agent_name.clone(),
            text: serde_json::to_string(&envelope.payload)?,
            timestamp: envelope.timestamp.to_rfc3339(),
            read: false,
            color: None,
            summary: envelope.summary.clone(),
        };
        self.mailbox.send(agent_name, msg)?;
        Ok(DeliveryReport::single_success())
    }

    fn route_uds(
        &self,
        envelope: &MessageEnvelope,
        socket_path: &str,
    ) -> Result<DeliveryReport, RoutingError> {
        let path = PathBuf::from(socket_path);
        UdsTransport::send_to(&path, envelope)?;
        Ok(DeliveryReport::single_success())
    }
}

The router implements a transport preference hierarchy: UDS when available (zero-latency, bidirectional), falling back to file-based mailbox (universal, resilient). In-process agents bypass both — messages are injected directly into the agent's message queue.

15.6 Task Coordination with File Locking

The shared task list — ~/.claude/tasks/{team-name}/ — is read and written by every agent simultaneously. Without careful coordination, classic concurrency bugs emerge: lost updates, phantom reads, double-claimed tasks.

The Atomic Claim Protocol

use fs2::FileExt;

pub struct TaskCoordinator {
    task_dir: PathBuf,
    high_watermark_path: PathBuf,
}

pub struct Task {
    pub id: String,
    pub subject: String,
    pub description: String,
    pub owner: Option<String>,
    pub status: TaskStatus,
    pub blocks: Vec<String>,
    pub blocked_by: Vec<String>,
}

#[derive(Debug, PartialEq)]
pub enum TaskStatus {
    Pending,
    InProgress,
    Completed,
}

pub enum ClaimResult {
    Success,
    AlreadyClaimed(String),     // by whom
    AlreadyCompleted,
    Blocked(Vec<String>),       // blocking task IDs
    AgentBusy(String),          // task the agent already owns
    LockFailed(String),
}

impl TaskCoordinator {
    /// Claim a task atomically, preventing double-claim
    pub fn claim_task(
        &self,
        task_id: &str,
        claimant: &str,
    ) -> Result<ClaimResult, CoordError> {
        let task_path = self.task_dir.join(format!("task-{}.json", task_id));
        let lock_path = self.task_dir.join(format!("task-{}.lock", task_id));

        // Acquire exclusive lock with exponential backoff
        let lock_file = fs::OpenOptions::new()
            .create(true).write(true).open(&lock_path)?;

        self.acquire_with_backoff(&lock_file, 30, 5, 100)?;

        let result = (|| {
            // Read current task
            let data = fs::read_to_string(&task_path)?;
            let mut task: Task = serde_json::from_str(&data)?;

            // Check: not already claimed by someone else
            if let Some(ref owner) = task.owner {
                if owner != claimant {
                    return Ok(ClaimResult::AlreadyClaimed(owner.clone()));
                }
            }

            // Check: not completed
            if task.status == TaskStatus::Completed {
                return Ok(ClaimResult::AlreadyCompleted);
            }

            // Check: claimant not already busy with another task
            if let Some(busy_task) = self.find_owned_task(claimant)? {
                if busy_task != task_id {
                    return Ok(ClaimResult::AgentBusy(busy_task));
                }
            }

            // Check: all blockers completed
            let unresolved = self.find_unresolved_blockers(&task.blocked_by)?;
            if !unresolved.is_empty() {
                return Ok(ClaimResult::Blocked(unresolved));
            }

            // Claim
            task.owner = Some(claimant.to_string());
            task.status = TaskStatus::InProgress;
            fs::write(&task_path, serde_json::to_string_pretty(&task)?)?;

            Ok(ClaimResult::Success)
        })();

        lock_file.unlock()?;
        result
    }

    fn acquire_with_backoff(
        &self,
        file: &fs::File,
        max_retries: u32,
        min_ms: u64,
        max_ms: u64,
    ) -> Result<(), CoordError> {
        for attempt in 0..max_retries {
            match file.try_lock_exclusive() {
                Ok(()) => return Ok(()),
                Err(_) => {
                    let backoff = std::cmp::min(min_ms * (1 << attempt), max_ms);
                    std::thread::sleep(std::time::Duration::from_millis(backoff));
                }
            }
        }
        Err(CoordError::LockTimeout)
    }

    fn find_owned_task(&self, agent: &str) -> Result<Option<String>, CoordError> {
        for entry in fs::read_dir(&self.task_dir)? {
            let path = entry?.path();
            if path.extension().map(|e| e == "json").unwrap_or(false)
                && path.file_name().map(|n| n.to_string_lossy().starts_with("task-")).unwrap_or(false)
            {
                let data = fs::read_to_string(&path)?;
                let task: Task = serde_json::from_str(&data)?;
                if task.owner.as_deref() == Some(agent)
                    && task.status != TaskStatus::Completed
                {
                    return Ok(Some(task.id));
                }
            }
        }
        Ok(None)
    }

    fn find_unresolved_blockers(&self, blocked_by: &[String]) -> Result<Vec<String>, CoordError> {
        let mut unresolved = Vec::new();
        for blocker_id in blocked_by {
            let path = self.task_dir.join(format!("task-{}.json", blocker_id));
            if path.exists() {
                let data = fs::read_to_string(&path)?;
                let task: Task = serde_json::from_str(&data)?;
                if task.status != TaskStatus::Completed {
                    unresolved.push(blocker_id.clone());
                }
            }
        }
        Ok(unresolved)
    }
}

The lock budget — 30 retries with exponential backoff from 5ms to 100ms — is deliberately generous. From Claude Code's source comment: the critical section does readdir + N reads + writeFile (~50-100ms on slow disks), so the last caller in a 10-way race needs ~900ms. 30 retries gives ~2.6s total wait.

The find_owned_task check prevents a common failure mode in agent swarms: an agent that claims multiple tasks, context-switches between them, and does poorly on all because its context window is split. One task at a time keeps agents focused.

High Water Mark

When a swarm is reset, task files are deleted. But the next swarm's tasks must not reuse IDs. Claude Code stores a high water mark — a simple integer in .highwatermark — and new task IDs start from highWaterMark + 1.

impl TaskCoordinator {
    pub fn next_task_id(&self) -> Result<String, CoordError> {
        let hwm_path = &self.high_watermark_path;
        let current: u32 = if hwm_path.exists() {
            fs::read_to_string(hwm_path)?.trim().parse().unwrap_or(0)
        } else {
            0
        };
        let next = current + 1;
        fs::write(hwm_path, next.to_string())?;
        Ok(next.to_string())
    }
}

15.7 Walkthrough: Multi-CLI Coordination in Action

Let us trace a complete coordination scenario from initial request to delivery.

The request: User types to the Orchestrator: "Build a user authentication API with JWT, rate limiting, and tests."

Phase 1: Team Creation and Task Planning

The Orchestrator creates a team and task DAG:

Team: auth-api-build
Tasks:
  1: "Research JWT libraries"          (no dependencies)
  2: "Design API schema"              (no dependencies)
  3: "Implement auth routes"           (blocked by 1, 2)
  4: "Add rate limiting"              (blocked by 3)
  5: "Security audit"                 (blocked by 3, 4)
  6: "Write integration tests"        (blocked by 3)
  7: "Generate API docs"             (blocked by 3, 4, 5, 6)

The Orchestrator spawns teammates via AgentTool. Each spawned teammate receives its initial instructions via mailbox, not command-line arguments — mailboxes have no length limits or escaping challenges.

Phase 2: Parallel Execution

Tasks 1 and 2 have no dependencies. Two agents claim them simultaneously. The atomic claim protocol ensures no double-claim:

Researcher -> claim_task("1") -> Lock acquired -> No blockers -> SUCCESS
Planner    -> claim_task("2") -> Lock acquired -> No blockers -> SUCCESS

Both work in parallel. When Researcher completes Task 1:

{
  "type": "TaskCompleted",
  "correlation_id": "task_auth-api-build",
  "payload": {
    "task_id": "1",
    "result": "passport-jwt recommended over jsonwebtoken for Express"
  }
}

Phase 3: Dependency Cascade

Task 3 is blocked by Tasks 1 and 2. When both complete, the Coder checks the task list, sees Task 3 is unblocked, and claims it. The claim protocol verifies: blocked_by: ["1", "2"] — both completed. Claim succeeds.

While the Coder works on Task 3, the other agents idle. The idle notification system provides a heartbeat — each agent sends an idle message to the lead after every turn.

Phase 4: Error Recovery

The Coder crashes mid-task (tmux pane dies). The Orchestrator detects the missing heartbeat — no idle notification within 2x the expected turn duration. It broadcasts:

{
  "type": "Broadcast",
  "priority": "High",
  "payload": {
    "alert": "Coder crashed during Task 3. Spawning replacement."
  }
}

The replacement Coder reads the partially completed state from the filesystem and resumes. The task was not completed, so it remains claimable by the new agent.

Phase 5: Graceful Shutdown

When all tasks complete, the Orchestrator initiates shutdown via structured message:

{ "type": "ShutdownRequest", "payload": { "reason": "All tasks completed" } }

Each teammate responds with ShutdownResponse carrying the matching request_id. When all approve, the Orchestrator cleans up the team.

The total message flow for this scenario: 7 task dispatches, ~20 status updates, 1 error broadcast, 1 replacement spawn, 7 completion notifications, 1 shutdown request, 6 shutdown responses — approximately 43 messages across 6 agents. The entire coordination happens without the user seeing anything beyond "Build a user authentication API..."

15.8 Error Propagation

Failures in multi-CLI systems cascade. One CLI's crash orphans tasks, confuses collaborators, and leaves the system in a state no single CLI can diagnose.

Failure Taxonomy

Agent Crash. Process terminates unexpectedly — OOM, segfault, kill -9. Detection: heartbeat timeout (no idle notification within 2x expected turn duration). Recovery: broadcast warning, spawn replacement, reassign orphaned task.

Agent Confusion. The agent is alive but doing the wrong thing — hallucinating a different task, stuck in a loop. Unique to AI agent systems. Detection: task progress monitoring. If an agent has been working too long with no status updates, the Orchestrator sends a query. Recovery: if response is incoherent, force-stop and reassign.

Dependency Deadlock. Circular dependencies in the task graph. Task A blocked by B, B blocked by A. Detection: periodic cycle check in the DAG. Recovery: break one edge, instruct an agent to proceed.

Communication Failure. Messages lost, delayed, corrupted. Detection: acknowledgment timeouts, JSON validation after reads, UDS connection error handlers. Recovery: retry with backoff, escalate to Orchestrator, switch transport.

Error Message Format

15.8 The Complete Communication Stack

From the agent's perspective down to the wire:

+---------------------------------------------------+
| Layer 5: Agent Decision                           |
| LLM decides to call SendMessage tool              |
+---------------------------------------------------+
| Layer 4: Message Envelope                         |
| Structured payload with routing + metadata        |
+---------------------------------------------------+
| Layer 3: Routing                                  |
| Name resolution -> transport selection            |
+---------------------------------------------------+
| Layer 2: Transport                                |
| Mailbox (file) | UDS (socket) | MCP | Bridge     |
+---------------------------------------------------+
| Layer 1: Wire                                     |
| Filesystem I/O | Unix socket | stdio | WebSocket  |
+---------------------------------------------------+

Each layer is independent and replaceable. Swap the transport without changing the envelope. Add message types without changing routing. Change routing without changing the agent's decision-making. This mirrors the OSI model — not because networking theory demands it, but because the same forces that drove network protocol designers to layer their stacks apply equally to multi-agent communication.

Delivery Guarantees by Transport

Transport Guarantee Ordering Failure Signal
Mailbox At-most-once FIFO per sender Silent
UDS Exactly-once FIFO within connection Connection error
MCP (stdio) Exactly-once FIFO Process exit code
MCP (HTTP/SSE) At-most-once FIFO HTTP error code

The mailbox provides at-most-once because there is no acknowledgment. A message written may never be read if the recipient crashes. For critical messages, use UDS or require explicit acknowledgment via response message.

Protocol Versioning

The message envelope includes no explicit version field in Claude Code's current implementation. For production systems, add a protocol_version field to the envelope:

impl MessageEnvelope {
    pub fn check_version_compatibility(
        &self,
        supported_major: u32,
    ) -> Result<(), ProtocolError> {
        let their_major = self.protocol_version
            .split('.')
            .next()
            .and_then(|v| v.parse::<u32>().ok())
            .unwrap_or(1);

        if their_major != supported_major {
            return Err(ProtocolError::VersionMismatch {
                supported: format!("{}.x.x", supported_major),
                received: self.protocol_version.clone(),
            });
        }
        Ok(())
    }
}

Version negotiation follows semantic versioning: patch differences are compatible, minor differences are backward-compatible (new fields ignored by older agents), major differences are incompatible and require a protocol bridge or agent upgrade.

Message Type Registry

The complete registry of message types and their semantics:

Type Direction Response Required Description
TaskAssignment Lead -> Agent No Dispatch work
TaskUpdate Agent -> Lead No Progress report
TaskCompleted Agent -> Lead No Work finished
Query Any -> Any Yes Request information
QueryResponse Any -> Any No Answer to a query
Broadcast Any -> All No System-wide alert
ShutdownRequest Lead -> Agent Yes Graceful shutdown
ShutdownResponse Agent -> Lead No Shutdown ack/nak
PlanApprovalRequest Agent -> Lead Yes Submit plan for review
PlanApprovalResponse Lead -> Agent No Approve/reject
Error Any -> Any Depends on severity Error notification
Heartbeat Lead -> Agent Yes Liveness check
HeartbeatAck Agent -> Lead No Liveness confirmation

Extensibility Points

Three mechanisms for extending the protocol without breaking existing agents:

1. Metadata field. Any message can carry arbitrary key-value metadata. Agents that do not understand a key ignore it. Safest extension mechanism — cannot break existing behavior.

2. New message types. The MessageType enum can be extended. An agent receiving an unknown type responds with an Error message containing UnknownMessageType. The sender falls back to a supported type.

3. Transport plugins. The addressing scheme is extensible. Adding a new transport (mqtt:topic for IoT, grpc:endpoint for microservices) requires: a new address parser, a transport implementation, and registration in the router.

15.9 Non-Engineering Domains

The multi-transport communication stack is not specific to software engineering.

Surgical Teams. A surgical team has a lead surgeon, anesthesiologist, scrub nurse, circulating nurse, and surgical assistant. Communication patterns mirror multi-CLI: the surgeon dispatches tasks (fire-and-forget), the anesthesiologist broadcasts vitals changes (pub/sub), and anyone can call a timeout for safety concerns (request/response with mandatory acknowledgment). The priority system maps directly: "patient desaturating" is Critical, "prep the next instrument" is Normal.

Emergency Response. Incident command systems use the same layered communication: radio (broadcast), phone (request/response), and written situation reports (fire-and-forget status updates). The typed message envelope maps to ICS form types: ICS-201 (incident briefing = TaskAssignment), ICS-209 (situation report = TaskUpdate), ICS-214 (activity log = Heartbeat).

Air Traffic Control. Controllers communicate with aircraft using strict protocols with mandatory readback (request/response with correlation). Frequency changes are broadcast to all aircraft on frequency. Emergency declarations override all other traffic (priority escalation). The deadlock prevention pattern — always respond to the current query before issuing a new one — exists in aviation for the same reason it exists in multi-CLI systems.
