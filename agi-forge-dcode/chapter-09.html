Chapter 9: The Coder CLI — The Builder

Tool Mastery, Streaming Execution, Code Intelligence, and the Discipline of Writing Code That Ships

Every specialist CLI in this book exists to support or constrain the Coder CLI. The Research CLI gathers knowledge so the Coder does not have to. The Planner CLI designs architecture so the Coder does not have to. The Security CLI audits what the Coder produces. The Validator CLI tests it. The QA CLI attacks it. The Documentation CLI explains it. The Coder is the center of gravity — the specialist that transforms plans, research, and requirements into working software.

And yet, the Coder CLI is not Claude Code with a different name. A vanilla Claude Code instance has 40+ tools and responds to open-ended prompts with whatever approach seems best. The Coder CLI is Claude Code with focus: it receives precise task descriptions from the Orchestrator (Chapter 6), follows architectural decisions from the Planner (Chapter 8), incorporates research from the Research CLI (Chapter 7), and produces code that must pass the Security CLI's audit and the Validator CLI's test suite. Its CLAUDE.md template is not "you are a helpful coding assistant" — it is a detailed specification of how to write code in this specific project, following these specific patterns, meeting these specific quality bars.

This chapter covers everything the Coder CLI needs to be effective: mastery of Claude Code's tool set (knowing when to use which tool and how to use it efficiently), the StreamingToolExecutor pattern for parallel tool execution, tree-sitter code intelligence for understanding code structure, git workflow automation for shipping changes, multi-file refactoring patterns for large-scale code changes, and error recovery patterns that prevent the Coder from spiraling into unproductive debugging loops.

The Coder CLI is the specialist that most directly corresponds to what a human developer does — and that familiarity is both an advantage (the patterns are well-understood) and a risk (it is tempting to make the Coder a generic "do everything" agent). The CLAUDE.md template at the end of this chapter constrains the Coder to execution, not design. It receives precise specifications and implements them precisely. When the specification is ambiguous, it reports back rather than guessing. When the specification conflicts with reality, it escalates rather than improvising. This discipline is what makes the Coder a reliable component in a larger system, not just a standalone coding tool.

9.1 The Tool Set: 40+ Tools, Mastered

Claude Code provides the Coder CLI with one of the richest tool sets of any AI coding agent. Understanding each tool's strengths, limitations, and appropriate use cases is the difference between a Coder that stumbles through tasks and one that executes with precision.

File Operations

Tool When to Use When NOT to Use
Read Read any file. First step before editing. Supports line ranges. Don't Read then immediately Write the whole file — use Edit instead.
Edit Change specific lines in existing files. Sends only the diff. Don't use for complete rewrites — use Write instead.
Write Create new files. Complete rewrites of existing files. Don't use to modify a few lines — use Edit instead.
Glob Find files by name pattern. Fast. Don't use for content search — use Grep instead.
Grep Search file contents by regex. Returns matches with line numbers. Don't use to find files by name — use Glob instead.

The most common Coder mistake is using Bash(cat file.ts) instead of Read(file.ts). The Read tool provides line numbers, handles binary files gracefully, and does not consume a shell session. Similarly, Bash(grep -r "pattern" .) is inferior to Grep(pattern) — the dedicated tool has better output formatting, respects .gitignore, and is optimized for the LLM's consumption.

The Read tool should always be the Coder's first action when touching a file. Reading before editing is not just good practice — Claude Code enforces it. The Edit tool will reject attempts to edit a file that has not been read in the current conversation. This guardrail prevents a common AI coding mistake: generating edits based on assumptions about file contents rather than actual contents. The Coder should Read the entire file (or relevant sections for large files) to understand: - The existing code style (tabs vs spaces, brace placement, naming conventions) - Import patterns (how are dependencies brought in?) - Error handling patterns (try/catch? Result types? Error callbacks?) - The surrounding context (what comes before and after the code you are changing?)

The Edit tool deserves special attention because it is the Coder's most-used tool and the one most prone to errors. The Edit tool works by exact string matching: you provide old_string (the text to find) and new_string (the text to replace it with). The old_string must be unique in the file. If it is not unique, the edit fails.

Common Edit failure modes and their solutions:

Problem: "old_string not found in file"
Cause: Whitespace mismatch — the file uses tabs but you specified spaces,
       or there are trailing spaces you did not include.
Solution: Read the file first. Copy the exact text including whitespace.

Problem: "old_string is not unique in file"
Cause: The string you are trying to replace appears multiple times.
Solution: Include more surrounding context in old_string to make it unique.
       Or use replace_all: true if you want to change every occurrence.

Problem: "Read tool must be called before Edit"
Cause: You tried to edit a file you have not read in this conversation.
Solution: Always Read before Edit. This is enforced to prevent blind edits.

Search Operations

The Coder CLI uses search tools differently than a human developer. A human types a query into a search bar and reads results. The Coder CLI must issue precise queries that return actionable results in a single call — every failed search wastes tokens and context. The difference between a good Coder CLI and a mediocre one often comes down to search precision: the good one reads 3 files and makes 2 edits; the mediocre one reads 15 files searching for the right one.

The search strategy should always be: most-specific first, broaden only if needed. Start with Grep("exact function name"), not Grep("some keyword that might be related"). Start with Glob("src/auth/middleware.ts"), not Glob("**/*.ts"). The specific search either finds exactly what you need (saving 3-4 turns) or finds nothing (telling you to broaden). The broad search returns 50 results and forces the Coder to spend turns filtering them.

Effective Grep patterns:

# Bad: too broad, returns hundreds of matches
Grep("function")

# Good: specific pattern that finds what you need
Grep("async function createUser", type: "ts")

# Finding all callers of a function
Grep("createUser\\(", type: "ts")

# Finding imports of a module
Grep("from ['\"].*auth", type: "ts")

# Finding TODO comments with context
Grep("TODO|FIXME|HACK", output_mode: "content", -C: 2)

Effective Glob patterns:

# Find all test files
Glob("**/*.test.ts")

# Find all migration files
Glob("**/migrations/*.sql")

# Find config files at any level
Glob("**/{*.config.*,*.rc,.*rc}")

# Find all files in a specific module
Glob("src/auth/**/*")

The Bash Tool: Power and Responsibility

The Bash tool is the Coder CLI's most powerful and most dangerous tool. It can do anything the operating system can do — run tests, install packages, compile code, execute scripts, manage git operations. It can also delete files, corrupt databases, and expose secrets.

Appropriate Bash use cases for the Coder CLI:

Category Examples
Testing pytest tests/test_auth.py, cargo test --lib, npm test -- --grep "auth"
Compilation cargo build, tsc --noEmit, npm run build
Package mgmt cargo add serde, npm install express, pip install -e .
Git operations git checkout -b feat/task-3, git add src/auth/, git commit
Type checking mypy src/, pyright, cargo clippy
Linting ruff check --fix, eslint --fix, cargo fmt

Inappropriate Bash use cases (use dedicated tools instead):

Don't Do This Do This Instead
cat src/auth.ts Read("src/auth.ts")
grep -r "createUser" . Grep("createUser")
find . -name "*.ts" Glob("**/*.ts")
echo "content" > file.ts Write("file.ts", "content")
sed -i 's/old/new/' file.ts Edit("file.ts", old="old", new="new")

The Coder CLI's CLAUDE.md should constrain Bash usage to specific patterns:

## Bash Usage Rules
1. NEVER use Bash for file reading (use Read), content search (use Grep), or file finding (use Glob)
2. Use Bash for: running tests, compiling, git operations, package management, type checking
3. Always prefer specific commands over broad ones: `pytest tests/test_auth.py::test_login` over `pytest`
4. When running commands that modify state, use the minimal command: `git add src/auth/` over `git add .`
5. Never run `rm -rf`, `DROP TABLE`, or `git push --force` without explicit human approval

9.2 The StreamingToolExecutor

Claude Code does not wait for a complete LLM response before executing tools. It executes tools as they appear in the stream — a critical performance optimization that the Coder CLI must understand and exploit.

Understanding streaming execution changes how the Coder CLI thinks about its work. In a sequential model, the Coder reasons about what to do, then does it, then reasons about the next step. In a streaming model, the Coder can interleave reasoning and execution — reading files while still deciding what to do with the information. This is not just faster; it produces better results because the Coder has real data (file contents, grep results) available during its reasoning, not just memories of what it expects to find.

How Streaming Execution Works

LLM stream begins:
  Token: "I'll" "need" "to" "read" "the" "file" "first."
  Tool block starts: Read("src/auth/middleware.ts")
  → StreamingToolExecutor: starts Read immediately, doesn't wait for stream to end
  → Read result available

  LLM continues streaming:
  Token: "Now" "let" "me" "also" "check" "the" "tests."
  Tool block starts: Read("tests/test_auth.ts")
  → StreamingToolExecutor: starts Read in parallel (Read is concurrency-safe)
  → Both results available

  Token: "I'll" "edit" "the" "middleware."
  Tool block starts: Edit("src/auth/middleware.ts", ...)
  → StreamingToolExecutor: Edit is NOT concurrency-safe → waits for exclusive access
  → Edit completes

The key concept is concurrency safety. Tools are classified as either concurrent-safe (multiple can run simultaneously) or exclusive (only one at a time):

Concurrent-Safe Exclusive
Read Edit
Grep Write
Glob Bash
WebSearch NotebookEdit
WebFetch

When the LLM produces multiple concurrent-safe tool calls in a single response, they execute in parallel. This means reading 5 files takes approximately the same time as reading 1 file. But when the LLM mixes concurrent-safe and exclusive tools, the exclusive tool gets its own sequential slot.

Implementing in Rust

The Coder CLI's Rust implementation uses Tokio JoinSets for parallel tool execution:

// clis/coder/src/engine/streaming_executor.rs
use tokio::task::JoinSet;
use std::sync::Arc;
use tokio::sync::Mutex;

pub struct StreamingToolExecutor {
    exclusive_lock: Arc<Mutex<()>>,
    results: Vec<ToolResult>,
}

impl StreamingToolExecutor {
    pub async fn execute_batch(&self, tool_calls: Vec<ToolCall>) -> Vec<ToolResult> {
        let mut concurrent_tasks = JoinSet::new();
        let mut exclusive_queue = Vec::new();

        // Partition into concurrent and exclusive
        for call in tool_calls {
            if call.tool.is_concurrency_safe() {
                let tool = call.tool.clone();
                let input = call.input.clone();
                concurrent_tasks.spawn(async move {
                    tool.execute(input).await
                });
            } else {
                exclusive_queue.push(call);
            }
        }

        // Run all concurrent tasks in parallel
        let mut results = Vec::new();
        while let Some(result) = concurrent_tasks.join_next().await {
            results.push(result.unwrap());
        }

        // Run exclusive tasks sequentially with lock
        for call in exclusive_queue {
            let _guard = self.exclusive_lock.lock().await;
            let result = call.tool.execute(call.input).await;
            results.push(result);
        }

        results
    }
}

This pattern gives the Rust-based Coder CLI a significant performance advantage over sequential execution. Reading 10 files to understand a module takes ~100ms total (parallel I/O) instead of ~1000ms (sequential). For a Coder task that reads 20-30 files during investigation, this saves 2-3 seconds per turn — cumulative savings of 30-60 seconds over a complex task.

Batching Tool Calls for Efficiency

The Coder CLI should think about tool calls strategically. Instead of reading files one at a time:

Turn 1: Read(src/auth/middleware.ts)
Turn 2: Read(src/auth/service.ts)
Turn 3: Read(src/auth/routes.ts)

It should read them all in a single turn:

Turn 1: Read(src/auth/middleware.ts) + Read(src/auth/service.ts) + Read(src/auth/routes.ts)

The StreamingToolExecutor runs all three reads in parallel, and the results are available in the same turn. This batching reduces the number of LLM round-trips (each round-trip costs ~2-5 seconds of latency and token processing) and keeps the Coder's execution tight.

The same principle applies to Grep queries. Instead of searching for related patterns across multiple turns, batch them:

Turn 1: Grep("createUser") + Grep("UserService") + Grep("auth/middleware")

All three searches complete in parallel, and the Coder has complete context to plan its edits in a single subsequent turn.

Performance Impact: Sequential vs Streaming vs Batched

To quantify the performance difference, consider a typical Coder task: "Add input validation to the user registration endpoint." The Coder needs to read 4 files (existing route, model, validation library usage, test file), make 2 edits (add validation to route, add test), and run the test suite.

Execution Model Tool Calls LLM Turns Wall Time
Sequential (one tool per turn) 7 7 ~35s
Streaming (tools execute during response) 7 4 ~20s
Batched + Streaming (parallel reads) 7 3 ~15s

The 57% speedup from sequential to batched+streaming compounds across a task with 20-30 tool calls. A complex Coder task that takes 3 minutes sequentially completes in under 90 seconds with proper batching and streaming. Over the course of a 16-task project, this saves 20+ minutes of wall time.

9.3 Tree-Sitter Code Intelligence

The Coder CLI does not read code as text. It reads code as structure — functions, classes, imports, call chains, type definitions. This structural understanding comes from tree-sitter, a parser generator that produces concrete syntax trees for source code in real time.

Tree-sitter parses incrementally — when a file changes, it re-parses only the affected portion of the syntax tree, not the entire file. This makes it fast enough for real-time use during editing. It also parses error-tolerant — a file with a syntax error on line 50 still produces a valid tree for lines 1-49 and 51+. This is essential for the Coder CLI, which frequently works with code in an intermediate state (partially edited, partially refactored).

Why Tree-Sitter Matters for the Coder CLI

Consider a refactoring task: "Rename the createUser function to registerUser across the codebase." A text-based approach uses Grep("createUser") and replaces all matches. But this also replaces comments, strings, and variable names that contain "createUser" — createUserToken, // Don't create user here, const createUserEndpoint. A tree-sitter-based approach queries the AST for function definitions and function calls named createUser, producing precise matches that exclude false positives.

The precision difference is not academic. In a real codebase with 200 occurrences of the string "createUser" across 50 files, only 25 might be actual function calls. A text-based rename produces 175 incorrect changes, each of which must be manually reviewed or caught by the type checker. A tree-sitter-based rename produces exactly 25 correct changes. In a multi-CLI system where the Coder's changes are automatically passed to the Validator CLI, incorrect renames waste the Validator's time and context on false type errors.

RCode's Tree-Sitter Implementation

The reader's RCode project (Chapter 21) already implements tree-sitter for 5 languages: Rust, Python, JavaScript, TypeScript, and Go. The Coder CLI extends this with code intelligence queries:

// clis/coder/src/intelligence/mod.rs
use tree_sitter::{Parser, Tree, Query, QueryCursor};

pub struct CodeIntelligence {
    parsers: HashMap<String, Parser>,
}

impl CodeIntelligence {
    pub fn find_function_definition(&self, source: &str, lang: &str, name: &str) -> Option<Location> {
        let tree = self.parse(source, lang)?;
        let query_str = match lang {
            "rust" => format!("(function_item name: (identifier) @name (#eq? @name \"{}\"))", name),
            "typescript" | "javascript" => format!(
                "(function_declaration name: (identifier) @name (#eq? @name \"{}\")) \
                 (method_definition name: (property_identifier) @name (#eq? @name \"{}\"))", name, name),
            "python" => format!("(function_definition name: (identifier) @name (#eq? @name \"{}\"))", name),
            _ => return None,
        };

        let query = Query::new(tree.language(), &query_str).ok()?;
        let mut cursor = QueryCursor::new();
        let matches = cursor.matches(&query, tree.root_node(), source.as_bytes());

        matches.into_iter().next().map(|m| {
            let node = m.captures[0].node;
            Location {
                line: node.start_position().row + 1,
                column: node.start_position().column,
                end_line: node.end_position().row + 1,
            }
        })
    }

    pub fn find_all_callers(&self, source: &str, lang: &str, function_name: &str) -> Vec<Location> {
        let tree = self.parse(source, lang).unwrap();
        let query_str = match lang {
            "rust" | "typescript" | "javascript" => format!(
                "(call_expression function: (identifier) @name (#eq? @name \"{}\"))", function_name),
            "python" => format!(
                "(call function: (identifier) @name (#eq? @name \"{}\"))", function_name),
            _ => return Vec::new(),
        };

        let query = Query::new(tree.language(), &query_str).unwrap();
        let mut cursor = QueryCursor::new();
        cursor.matches(&query, tree.root_node(), source.as_bytes())
            .flat_map(|m| m.captures.iter().map(|c| Location {
                line: c.node.start_position().row + 1,
                column: c.node.start_position().column,
                end_line: c.node.end_position().row + 1,
            }))
            .collect()
    }

    pub fn extract_imports(&self, source: &str, lang: &str) -> Vec<ImportInfo> {
        // Extract all import statements with source module and imported names
        let tree = self.parse(source, lang).unwrap();
        let query_str = match lang {
            "typescript" | "javascript" => {
                "(import_statement source: (string) @source)"
            }
            "python" => {
                "(import_from_statement module_name: (dotted_name) @source)"
            }
            "rust" => {
                "(use_declaration argument: (use_tree) @path)"
            }
            _ => return Vec::new(),
        };

        let query = Query::new(tree.language(), &query_str).unwrap();
        let mut cursor = QueryCursor::new();
        cursor.matches(&query, tree.root_node(), source.as_bytes())
            .map(|m| {
                let node = m.captures[0].node;
                let text = &source[node.byte_range()];
                ImportInfo {
                    module: text.trim_matches('"').trim_matches('\'').to_string(),
                    line: node.start_position().row + 1,
                }
            })
            .collect()
    }
}

Code Intelligence in Practice

The Coder CLI uses tree-sitter for three critical operations:

  1. Understanding before editing — Before modifying a function, the Coder queries the AST to find all callers. If renaming a function would break 15 callers across 8 files, the Coder knows this before making any changes and can plan the edit sequence.

  2. Safe refactoring — When the Planner specifies "rename createUser to registerUser," the Coder uses tree-sitter to find all AST nodes that reference the function (definitions, calls, type annotations) and edits only those locations. Comments and strings are left unchanged.

  3. Pattern detection — The Coder detects existing code patterns by querying the AST: "How are error handlers structured in this codebase?" "What import style is used?" "Are async functions used with try/catch or .catch()?" This pattern detection ensures new code matches existing conventions.

LSP Integration

For languages with mature Language Server Protocol (LSP) implementations, the Coder CLI can use the LSP for even richer code intelligence than tree-sitter provides. Claude Code's source includes an LSPTool that communicates with language servers:

LSP Capability Coder Use Case
Go to Definition Find where a function/type is defined
Find All References Locate every usage of a symbol
Hover Get type information and documentation
Code Actions Apply automated fixes (auto-import, extract function)
Diagnostics Real-time error detection without running the compiler

The tradeoff: LSP provides richer information than tree-sitter (type inference, cross-file analysis, documentation extraction) but requires a running language server, which adds startup time and memory overhead. For the Coder CLI's use case — where precision matters and sessions last minutes to hours — LSP is worth the overhead. The language server starts when the Coder session begins and persists across all tool calls.

In practice, the Coder CLI uses both tree-sitter and LSP complementarily. Tree-sitter handles fast, local queries within a single file (function boundaries, import extraction, structural navigation). LSP handles cross-file queries that require type system knowledge (finding all implementations of an interface, resolving type aliases, discovering which module exports a given type). The combination provides code intelligence comparable to what a human developer gets from a modern IDE.

9.4 Git Workflow Automation

The Coder CLI does not just write code — it ships code. Every task ends with changes committed to a branch, pushed to a remote, and (optionally) submitted as a pull request. The git workflow is not an afterthought; it is a core part of the Coder's task execution.

The Branch-Commit-PR Pipeline

// clis/coder/src/git/workflow.rs
pub struct GitWorkflow {
    project_root: PathBuf,
}

impl GitWorkflow {
    pub async fn create_feature_branch(&self, task_id: &str) -> Result<String> {
        let branch_name = format!("feat/task-{}", task_id);
        self.run_git(&["checkout", "-b", &branch_name]).await?;
        Ok(branch_name)
    }

    pub async fn commit_changes(&self, message: &str, files: &[&str]) -> Result<()> {
        // Stage specific files (never git add .)
        for file in files {
            self.run_git(&["add", file]).await?;
        }

        // Verify no secrets in staged files
        self.scan_staged_for_secrets().await?;

        // Commit with attribution
        let full_message = format!(
            "{}\n\nCo-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>",
            message
        );
        self.run_git(&["commit", "-m", &full_message]).await?;
        Ok(())
    }

    pub async fn create_pull_request(&self, title: &str, body: &str) -> Result<String> {
        let output = Command::new("gh")
            .args(["pr", "create", "--title", title, "--body", body])
            .output()
            .await?;

        let pr_url = String::from_utf8(output.stdout)?.trim().to_string();
        Ok(pr_url)
    }

    async fn scan_staged_for_secrets(&self) -> Result<()> {
        let staged = self.run_git(&["diff", "--cached", "--name-only"]).await?;
        for file in staged.lines() {
            if file.ends_with(".env") || file.ends_with(".pem") || file.contains("credentials") {
                return Err(anyhow!("Refusing to commit potentially sensitive file: {}", file));
            }
            // Check file contents for secret patterns
            let content = std::fs::read_to_string(file)?;
            if contains_secret_pattern(&content) {
                return Err(anyhow!("Secret detected in {}: aborting commit", file));
            }
        }
        Ok(())
    }
}

Atomic Commits per Logical Change

The Coder CLI commits after each logical change, not at the end of the entire task. A task that adds authentication produces multiple commits:

feat: add users table migration (T3-schema)
feat: implement JWT auth service (T3-auth-service)  
feat: add auth middleware with RS256 verification (T3-auth-middleware)
feat: add auth routes — register, login, refresh (T3-auth-routes)
test: add auth integration tests (T3-auth-tests)

This granularity serves two purposes: it makes the Validator CLI's job easier (each commit can be tested independently), and it makes rollback precise (if the auth routes are wrong, revert that commit without losing the schema migration).

Commit Message Quality

The Coder CLI's commit messages must be meaningful to both humans and other specialist CLIs. The Validator CLI reads commit messages to understand what changed. The Documentation CLI reads them to generate changelogs. The Security CLI reads them to identify security-relevant changes.

Good commit message structure:

<type>: <what changed> (<task-id>)

<why this change was made — 1-2 sentences>

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>

Examples:

feat: add JWT RS256 verification middleware (T3-auth)

Implements token verification using RS256 with JWKS key rotation support.
Tokens are verified on every protected route before the route handler executes.

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>

fix: prevent SQL injection in user search query (T7-security-fix)

The previous implementation used string interpolation for the search term.
Switched to parameterized query to eliminate injection risk (CWE-89).

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>

The commit message explains WHY the change was made, not WHAT changed (the diff shows WHAT). This is especially important in a multi-CLI system where the commit author (the Coder CLI) and the commit reviewer (the Security CLI, the human) are different entities. A commit message that says "update middleware.ts" tells the reviewer nothing. A message that explains the security motivation tells the reviewer exactly what to verify during review.

Git Worktree Isolation

As discussed in Chapter 6, when multiple Coder CLI instances work in parallel, they each operate in their own git worktree. This prevents the most common parallel editing bug: two specialists modifying the same file simultaneously, producing corrupted output.

# Orchestrator creates worktrees before dispatching
git worktree add .workers/coder-task-3 -b worker/task-3
git worktree add .workers/coder-task-4 -b worker/task-4

# Each Coder instance works in its own worktree
CODER_1_CWD=.workers/coder-task-3
CODER_2_CWD=.workers/coder-task-4

# After completion, Orchestrator merges
git merge worker/task-3 --no-ff -m "feat: authentication (T3)"
git merge worker/task-4 --no-ff -m "feat: API routes (T4)"

The --no-ff flag ensures merge commits are always created, preserving the task boundary in the git history. Without it, fast-forward merges flatten the task's commits into the main branch, making it impossible to revert a complete task by reverting a single merge commit.

Worktree isolation has a cost: each worktree copies the working tree (though the .git directory is shared). For projects with large build directories (node_modules/, target/), this can consume significant disk space. The Orchestrator should configure .gitignore to exclude build artifacts from worktrees and run dependency installation only in worktrees that need it. A Coder CLI working on a pure TypeScript change does not need node_modules/ recompiled — it can use the shared node_modules/ from the main worktree via symlink.

Handling Merge Conflicts

When two parallel Coder instances modify different functions in the same file, git merges cleanly. When they modify the same function, a merge conflict occurs. The Orchestrator should resolve this by:

  1. Merging the first Coder's worktree (succeeds)
  2. Attempting to merge the second Coder's worktree (conflict detected)
  3. Dispatching a new task to the first Coder: "Resolve merge conflict in {file}. Your changes are on main. The conflicting changes from task-4 are in the conflict markers. Integrate both sets of changes."

This approach leverages the first Coder's familiarity with the file (it just edited it) to resolve the conflict intelligently. The alternative — having the Orchestrator resolve conflicts — violates the restriction principle (the Orchestrator should not read or modify code).

9.5 Multi-File Refactoring Patterns

The Coder CLI's most complex tasks involve refactoring that spans many files. A module rename, an API version upgrade, a pattern migration (callbacks to async/await) — these require coordinated changes across 10-50 files where each change must be consistent with all others.

The key principle: a multi-file refactoring should leave the codebase in a working state after every intermediate step. If the Coder renames a function in file A but has not yet updated the caller in file B, the codebase has a broken reference. The type checker will report errors. The test suite will fail. Other specialists working in parallel might encounter these transient errors and mistake them for real problems.

The Coder must order its edits so that this broken state either does not occur or is immediately resolved in the next edit. For renames, the standard strategy is: add the new name as an alias first (export both createUser and registerUser), update all callers to use the new name, then remove the old alias. At no point is the codebase in a broken state.

The Three Phases of Multi-File Refactoring

Phase 1: Discover — Find all files affected by the refactoring. Use Grep to find all references to the target pattern, Glob to find all files of the relevant type, and tree-sitter to filter for actual code references (not comments or strings).

Phase 2: Plan — Order the changes to minimize intermediate broken states. Start with the definition (the function being renamed), then update callers leaf-first (files that are not imported by other files first), then update imports. This ordering ensures that at each step, the codebase is either fully old-style or fully new-style — never half-migrated.

Phase 3: Execute — Apply changes file by file, using the Edit tool with replace_all: true for global renames within a file. After each batch of changes, run the type checker to verify consistency.

// clis/coder/src/refactoring/multi_file.rs
pub struct MultiFileRefactoring {
    intelligence: CodeIntelligence,
}

impl MultiFileRefactoring {
    pub async fn rename_function(
        &self, 
        old_name: &str, 
        new_name: &str, 
        root: &Path
    ) -> Result<Vec<FileChange>> {
        let mut changes = Vec::new();

        // Phase 1: Discover all references
        let files = self.find_all_files_with_reference(old_name, root).await?;

        // Phase 2: Order changes (definition first, then callers leaf-first)
        let definition_file = self.find_definition_file(old_name, &files)?;
        let import_graph = self.build_import_graph(&files)?;
        let ordered = topological_sort_reverse(&import_graph); // Leaves first

        // Phase 3: Execute changes
        // First: rename the definition
        changes.push(FileChange {
            path: definition_file.clone(),
            edits: vec![RenameEdit {
                old: old_name.to_string(),
                new: new_name.to_string(),
                scope: EditScope::FunctionDefinition,
            }],
        });

        // Then: update all callers in topological order
        for file in ordered {
            if file != definition_file {
                let file_refs = self.intelligence
                    .find_all_callers(&std::fs::read_to_string(&file)?, 
                        &detect_language(&file), old_name);
                if !file_refs.is_empty() {
                    changes.push(FileChange {
                        path: file,
                        edits: vec![RenameEdit {
                            old: old_name.to_string(),
                            new: new_name.to_string(),
                            scope: EditScope::AllReferences,
                        }],
                    });
                }
            }
        }

        Ok(changes)
    }
}

Refactoring Verification

After completing a multi-file refactoring, the Coder CLI runs three checks:

  1. Type checker — tsc --noEmit (TypeScript), cargo check (Rust), mypy (Python). Type errors indicate missed references.
  2. Grep verification — Search for the old name. If it still appears in code (not comments), the refactoring is incomplete.
  3. Test suite — Run the full test suite. A refactoring that changes behavior is a bug, not a refactoring.

If any check fails, the Coder does not proceed to the next task. It fixes the issue and re-runs verification. A partially-complete refactoring is worse than no refactoring at all — it leaves the codebase in an inconsistent state where some callers use the old name and some use the new name, making it harder to search, understand, and modify the code.

Common Refactoring Patterns

Pattern Technique Key Risk
Function rename Tree-sitter AST query → rename definition → rename all callers Missing dynamic callers (string-based invocation)
Module extraction Identify cohesive functions → create new module → update imports Circular dependency in the new module structure
Pattern migration (callbacks → async) Find all callback patterns → convert leaf functions first → propagate async upward Missing error handling in the async version
API version upgrade Find all uses of deprecated API → map to new API → replace Behavior differences between old and new API
Type generalization Identify concrete types → introduce generic → update callers Over-generalization that loses type safety

9.6 Error Recovery Patterns

The Coder CLI will encounter errors during execution. Files that do not exist where expected. Tests that fail unexpectedly. Type checkers that report errors in code the Coder did not touch. Build tools that crash. The Coder's error recovery strategy determines whether it completes tasks successfully or spirals into unproductive debugging loops.

The Error Classification Framework

Not all errors require the same response. The Coder CLI classifies errors and applies the appropriate recovery strategy:

pub enum ErrorClass {
    /// Error in the Coder's own changes — fix immediately
    OwnChange { file: String, line: usize, error: String },

    /// Error in existing code exposed by the Coder's changes — investigate
    Exposed { file: String, explanation: String },

    /// Transient error — retry once
    Transient { tool: String, error: String },

    /// Environment error — report to Orchestrator
    Environment { missing: String },

    /// Specification error — report to Orchestrator
    Specification { conflict: String },
}

impl Coder {
    fn handle_error(&mut self, error: ErrorClass) -> Action {
        match error {
            ErrorClass::OwnChange { file, line, error } => {
                // Read the file, understand the error, fix it
                Action::FixAndRetry { file, line, error }
            }
            ErrorClass::Exposed { file, explanation } => {
                // The Coder's change exposed a bug in existing code
                // Report to Orchestrator — this may be out of scope
                Action::Report {
                    message: format!("Existing bug exposed in {}: {}", file, explanation),
                    severity: Severity::Medium,
                }
            }
            ErrorClass::Transient { tool, error } => {
                // Network timeout, rate limit, temporary file lock
                Action::Retry { delay_ms: 2000 }
            }
            ErrorClass::Environment { missing } => {
                // Missing dependency, wrong Node version, missing env var
                Action::Report {
                    message: format!("Environment issue: {} not available", missing),
                    severity: Severity::Blocking,
                }
            }
            ErrorClass::Specification { conflict } => {
                // The task specification conflicts with reality
                Action::Report {
                    message: format!("Specification conflict: {}", conflict),
                    severity: Severity::Blocking,
                }
            }
        }
    }
}

The "Read Before You Retry" Rule

The single most important error recovery rule: when an edit fails or a test fails, read the actual state of the file before trying again. The most common Coder failure loop:

1. Edit file → fails (old_string not found)
2. Retry same edit → fails again (same reason)
3. Retry with slightly different old_string → fails again
4. 5 failed attempts later, context is polluted with error messages

The correct approach:

1. Edit file → fails (old_string not found)
2. Read the file to see its ACTUAL current state
3. Construct correct old_string from the actual content
4. Edit succeeds on first retry

This rule applies to test failures too. When a test fails, the Coder should read the test output, read the relevant source file, and understand the failure before attempting a fix. A Coder that blindly modifies code based on an error message without reading the surrounding context produces fixes that address the symptom, not the cause.

Knowing When to Stop

A Coder CLI that keeps retrying a failing approach wastes tokens and time. The Coder should report back to the Orchestrator after: - 3 consecutive failed attempts at the same edit - A type error it cannot diagnose within 2 turns - A test failure that appears to be in existing code, not its changes - Any error that suggests the task specification is incorrect

Reporting is not failure — it is responsible execution. The Orchestrator can then re-dispatch the task with additional context, assign it to a different specialist, or escalate to the human.

Debug Loop Prevention

The most dangerous Coder failure pattern is the debug loop: the Coder encounters an error, makes a change, encounters a different error, makes another change, encounters a third error, and so on — each "fix" introducing a new problem while not solving the original one. After 5-6 iterations, the code is in a worse state than before the Coder started.

Debug loop prevention requires self-awareness. The Coder CLI should track its error count and the types of errors encountered. If it encounters 3 distinct errors on the same file within 5 turns, it should stop, read the entire file to understand the complete state, and either formulate a comprehensive fix that addresses all issues at once, or report back to the Orchestrator that the task requires a different approach.

The CLAUDE.md template encodes this as a hard rule:

## Error Escalation Rules
- 1 error: Read the file, understand the error, fix it.
- 2 errors on the same file: Read the ENTIRE file. Understand the full context.
  Fix comprehensively — do not make another point fix.
- 3 errors on the same file: STOP. Report to Orchestrator with:
  - What you tried
  - What errors you encountered
  - What you think the underlying issue is
  - What additional context you would need to succeed

This rule prevents runaway token consumption (each failed turn costs $0.05-0.15 in API calls) and protects the codebase from cascading point-fixes that make the code harder to understand than the original bug.

9.7 The Coder CLI in Non-Engineering Domains

As with every specialist CLI in this book, the Coder pattern extends beyond software engineering. In non-engineering domains, the "Coder" becomes the "Producer" — the specialist that creates the primary artifacts.

Legal Document Producer: Receives specifications from the Planner ("Draft a motion for summary judgment, citing precedents A, B, C"), follows the research from the Research CLI (relevant case law and statutes), produces the document, and submits it to the Validator (proofreading and citation checking) and the Security CLI equivalent (privilege review to ensure no confidential information is disclosed).

Medical Report Producer: Receives a structured report template from the Planner, incorporates findings from the Research CLI (lab results, imaging reports, literature review), produces the clinical documentation, and submits to the Validator (compliance checking) and the QA CLI (differential diagnosis review).

Financial Analysis Producer: Receives an analysis framework from the Planner, incorporates market data and research findings, produces the financial model and recommendation document, and submits to the Validator (calculation verification) and the Security CLI equivalent (compliance review for investment recommendations).

In each case, the Producer CLI's CLAUDE.md changes to reflect domain-specific quality standards, its tool set changes to include domain-specific tools (legal citation formatters, medical code lookup, financial modeling libraries), and its verification pipeline changes. But the core behavior — receive precise specification, execute with discipline, commit atomic changes, report ambiguities — remains identical to the software Coder CLI.

9.8 The Complete Coder CLI CLAUDE.md

# CoderCLI — The Builder

## Identity
You are the Coder CLI — the builder of a multi-CLI agent system.
You write production code. You receive precise task descriptions from
the Orchestrator with specific requirements, file paths, and constraints.
Your job is EXECUTION — implementing exactly what was specified.

## The Execution Principle
You receive COMPLETE task specifications. Do not redesign the architecture
(that is the Planner's job). Do not research alternatives (that is the
Research CLI's job). Do not audit for vulnerabilities (that is the Security
CLI's job). Execute the specification. If the specification is ambiguous
or impossible, report back to the Orchestrator — do NOT improvise.

## Code Quality Rules
1. Match existing patterns. Read 3 similar files before writing new code.
2. Type-safe: TypeScript strict mode, Rust borrow checker, Python type hints.
3. Test-first for bug fixes. Write the failing test before the fix.
4. Error handling at system boundaries only. Trust internal code.
5. No speculative abstractions. Three similar lines > premature helper function.
6. Comments explain WHY, not WHAT. Delete "// increment counter" comments.

## Tool Usage Discipline
- File reading: ALWAYS use Read, never Bash(cat)
- Content search: ALWAYS use Grep, never Bash(grep)
- File finding: ALWAYS use Glob, never Bash(find)
- Editing: Use Edit for modifications, Write for new files only
- Bash: ONLY for running tests, compilation, git, and package management
- Read BEFORE Edit. Always. No exceptions.

## Git Discipline
- Branch per task: `feat/task-{id}`, `fix/task-{id}`
- Commit after each logical change, not at the end
- Commit messages: imperative mood, explain WHY not WHAT
- Stage specific files: `git add src/auth/` never `git add .`
- NEVER commit .env, credentials, or secret files
- NEVER force push or amend without explicit approval

## Multi-File Refactoring Protocol
1. DISCOVER: Find all affected files with Grep + tree-sitter
2. PLAN: Order changes definition-first, then callers leaf-first
3. EXECUTE: Edit file by file, checking types after each batch
4. VERIFY: Type check → grep for old pattern → run tests

## When to Report Back
- The specification is ambiguous (multiple valid interpretations)
- The specification conflicts with existing code (incompatible types, missing dependencies)
- The specification requires a library/tool not in the project's dependencies
- Tests pass but you suspect the tests are wrong (test the tests)
- The task would take significantly longer than estimated

## What You Do NOT Do
- Redesign architecture (Planner)
- Research alternatives (Research CLI)
- Audit for vulnerabilities (Security CLI)
- Run the full validation pipeline (Validator CLI)
- Test adversarially (QA CLI)
- Write documentation (Documentation CLI)
