Chapter 5: CLAUDE.md — Programming Each CLI Identity

The System Prompt That Transforms a Generic Agent Into a Specialist

In Chapter 4, you built a workspace with multiple CLIs sharing a build system, a configuration hierarchy, and an MCP communication channel. But right now, every CLI is identical — they all have the same personality, the same priorities, the same blind spots. Ask the Orchestrator to audit code and it will try. Ask the Security CLI to write a feature and it will attempt it. Nothing distinguishes the specialist from the generalist.

CLAUDE.md is what changes that. It is the file that transforms a generic Claude Code instance into a specialist with expertise, boundaries, behavioral contracts, and a professional identity. In a single-CLI world, CLAUDE.md is a team onboarding document. In a multi-CLI world, it is the job description, the operating manual, and the performance contract for each member of your AI team.

This chapter covers how to write CLAUDE.md files that produce genuine specialist behavior — not vague aspirational prompts, but precisely engineered identity documents that change what the CLI does, how it communicates, and what it refuses to do.

5.1 The CLAUDE.md Hierarchy in Multi-CLI Systems

Claude Code searches for CLAUDE.md in a hierarchy of locations and merges all found content into the system prompt. In a single-project setup, this hierarchy has four levels. In a multi-CLI workspace, it has six.

The Six Levels

Level 1: ~/.claude/CLAUDE.md                          ← Global (your personal preferences)
Level 2: ~/multi-cli-forge/CLAUDE.md                  ← Workspace (team-wide standards)
Level 3: ~/multi-cli-forge/clis/security/CLAUDE.md    ← CLI identity (THE specialist prompt)
Level 4: ~/multi-cli-forge/clis/security/.claude/rules/*.md  ← Rules (path-scoped)
Level 5: Project-level CLAUDE.md in working directory  ← Per-project overrides
Level 6: Dynamic injection via hooks/environment       ← Runtime modifications

Level 1: Global — Your personal communication preferences. Applies to every CLI you run, regardless of specialist role. This is where "be terse" and "no emojis" live.

Level 2: Workspace — Team-wide coding standards that all specialists follow. Language conventions, git workflow, testing requirements. This is the CLAUDE.md at your forge root, equivalent to the company engineering handbook.

Level 3: CLI Identity — The specialist prompt. This is where the transformation happens. The Security CLI's Level 3 CLAUDE.md says "You are a security auditor. You NEVER modify code. You analyze and report." The Coder CLI's Level 3 says "You write production-grade code. You always write tests first." These are not suggestions — they are identity constraints that override generic behavior.

Level 4: Rules — Path-scoped coding rules from .claude/rules/*.md. The Security CLI might have rules for how to format vulnerability reports. The Coder CLI might have language-specific rules for Python, TypeScript, and Rust.

Level 5: Per-project — When a CLI operates on a specific project directory, any CLAUDE.md in that directory adds project-specific context. The Security CLI auditing a Django project picks up Django-specific security rules from the project's own CLAUDE.md.

Level 6: Dynamic — Runtime modifications injected via SessionStart hooks or environment variables. The Orchestrator might inject task-specific context into a specialist's prompt before dispatching work: "Your current task is to audit the authentication module. Focus on session management and token handling."

How Merging Works

All levels are concatenated into the system prompt, with earlier levels appearing first. In Claude Code's architecture (documented in Chapter 2), the context assembly pipeline builds the system prompt in this order:

  1. Claude Code's base instructions (the foundation — you cannot change these)
  2. Global CLAUDE.md content
  3. Workspace/project CLAUDE.md content
  4. Matching rules (based on file paths in the conversation)
  5. Tool definitions
  6. MCP server instructions
  7. Conversation history

Instructions that appear earlier in the system prompt carry more weight during generation. This means your Level 1-3 CLAUDE.md content shapes behavior more strongly than rules or tool definitions. Use this to your advantage: put the most important identity constraints at the top of your CLI-specific CLAUDE.md.

Conflict Resolution

When instructions conflict across levels — say, Level 2 says "always use Jest for testing" but Level 5 says "use Vitest for this project" — Claude Code follows the more specific instruction. Specificity overrides generality. Project-level overrides workspace-level overrides global-level.

For multi-CLI systems, the Level 3 (CLI identity) CLAUDE.md should be authoritative for the specialist's role. If the workspace CLAUDE.md says "write code to fix issues" but the Security CLI's identity says "NEVER modify code," the specialist identity wins because it is more specific to the current context.

5.2 The Anatomy of a Specialist CLAUDE.md

A generic CLAUDE.md is a list of preferences. A specialist CLAUDE.md is an identity document with five sections, each serving a distinct purpose:

# [CLI Name] — [Role Description]

## Identity
Who you are. What you do. What you NEVER do.

## Capabilities
Your tools, their purposes, and how to use them effectively.

## Communication Protocol
How you receive tasks and how you report results.

## Rules
Behavioral constraints that override all other instructions.

## Domain Knowledge
Specialized knowledge that makes you an expert.

Section 1: Identity

The identity section is the most important. It is the first thing Claude reads after its base instructions, and it establishes the behavioral foundation for everything that follows.

A weak identity section:

# Security CLI
You help with security analysis.

A strong identity section:

# SecurityCLI — Security Analysis Guardian

## Identity
You are SecurityCLI, a specialized security analysis agent operating as part of a
multi-CLI team managed by an Orchestrator.

- **Role**: Read-only security analyst. You audit code, scan dependencies, model threats.
- **Scope**: You ANALYZE. You REPORT. You RECOMMEND. You NEVER modify source code.
- **Authority**: Your security findings override convenience. When you flag a CRITICAL
  vulnerability, the team must address it before shipping.
- **Personality**: Methodical, thorough, skeptical. You assume code is insecure until
  proven otherwise. You document evidence, not opinions.

The difference is specificity. The strong version tells Claude: - What it is (SecurityCLI, part of a team) - What it does (analyze, report, recommend) - What it never does (modify code) - What authority it has (security findings are blocking) - How it thinks (skeptical, evidence-based)

Each of these constraints changes Claude's behavior in measurable ways. "You NEVER modify source code" prevents the LLM from reaching for the file_edit tool even when fixing a vulnerability would be trivial. "Your security findings override convenience" means the Security CLI will flag a vulnerability even if fixing it requires a large refactor.

Section 2: Capabilities

The capabilities section maps the CLI's tools to their intended purpose. This is not redundant with tool descriptions — it tells the LLM when and why to use each tool, not just what each tool does.

## Capabilities
You have access to these tools and ONLY these tools:

| Tool | Purpose | When to Use |
|------|---------|------------|
| `sast_scan` | Static analysis for code vulnerabilities | Every audit. Run first. |
| `dep_audit` | Check dependencies for known CVEs | After SAST, or when specifically asked |
| `secret_scan` | Detect hardcoded secrets and credentials | Every audit. Non-negotiable. |
| `threat_model` | Generate STRIDE-based threat models | When asked, or for new architectures |
| `file_read` | Read any file for manual analysis | When SAST flags need manual verification |
| `grep_search` | Search for patterns across files | To trace data flows and find related code |
| `bash` (restricted) | Run: grep, find, semgrep, trivy, gitleaks ONLY | When built-in tools are insufficient |

You do NOT have: file_write, file_edit, file_delete, or unrestricted bash.
If a task requires code modification, report the finding and let the Coder CLI handle the fix.

The "ONLY these tools" and "You do NOT have" lines are critical. They prevent the LLM from hallucinating tool calls that do not exist in the binary. Without explicit capability boundaries, Claude may attempt to call file_edit and receive a "tool not found" error — wasting tokens and confusing the agentic loop.

Section 3: Communication Protocol

In a multi-CLI system, each specialist must communicate in a structured format that the Orchestrator can parse. Unstructured natural language responses from specialists create integration nightmares — the Orchestrator cannot reliably extract findings, status, or next actions from prose.

## Communication Protocol

### Receiving Tasks
Tasks arrive from the Orchestrator as structured messages. Extract:
- **scope**: What to audit (file, directory, module, or full project)
- **depth**: Quick scan, standard audit, or deep analysis
- **focus**: Specific concern (auth, injection, dependencies) or general

### Reporting Results
ALWAYS respond with structured JSON:
```json
{
  "status": "complete" | "in_progress" | "blocked",
  "findings_count": {
    "critical": 0,
    "high": 0,
    "medium": 0,
    "low": 0,
    "info": 0
  },
  "summary": "One paragraph executive summary",
  "findings": [
    {
      "id": "SEC-001",
      "severity": "critical",
      "cwe": "CWE-89",
      "file": "src/db/queries.ts",
      "line": 47,
      "title": "SQL injection via string interpolation",
      "description": "User input concatenated directly into SQL query",
      "evidence": "const q = `SELECT * FROM users WHERE id = ${req.params.id}`",
      "remediation": "Use parameterized queries: db.query('SELECT * FROM users WHERE id = $1', [req.params.id])"
    }
  ],
  "recommendations": ["Priority actions for the team"]
}

This structured output format is what makes multi-CLI systems work at scale. The Orchestrator receives JSON, parses it programmatically, routes findings to the appropriate specialists (critical vulnerabilities to the Coder CLI for fixing), and synthesizes the results for the human.

### Section 4: Rules

Rules are behavioral constraints that override all other instructions — including the LLM's own preferences. They are the guardrails that prevent specialist drift.

```markdown
## Rules
1. NEVER modify source code. Not even to demonstrate a fix. Describe the fix in text.
2. NEVER skip the secret scan. Every audit includes a secret scan, even if not asked.
3. ALWAYS include CWE identifiers for findings. No finding without a CWE reference.
4. ALWAYS include file:line references. No finding without a location.
5. When uncertain about severity, round UP. Better to over-report than under-report.
6. When the Orchestrator asks for a "quick scan," still run all tools — just summarize briefly.
7. Write findings to shared/memory/findings.db when persistence is requested.
8. NEVER dismiss a finding as "probably fine." Flag it as INFO severity with evidence.

Rules work best when they are: - Absolute (ALWAYS/NEVER, not "try to" or "prefer") - Specific (CWE identifiers, not "include references") - Justified (round UP because under-reporting risks production) - Few (8-12 rules, not 50 — Claude follows fewer rules more reliably)

Section 5: Domain Knowledge

Domain knowledge is what makes a specialist genuinely useful rather than generically competent. It encodes the expertise that the LLM's training data might lack — your team's specific patterns, known vulnerabilities, and historical context.

## Domain Knowledge

### Project-Specific Patterns
- This project uses Prisma ORM. SQL injection is unlikely in Prisma queries but
  check for raw queries (`$queryRaw`, `$executeRaw`) which bypass the ORM.
- Authentication uses JWT with RS256. Check for:
  - Algorithm confusion attacks (accepting HS256 when RS256 is expected)
  - Token expiry too long (>24h)
  - Missing token revocation
- The API gateway at `src/gateway/` handles rate limiting. Do NOT flag
  rate-limiting code as a DoS vulnerability — it IS the mitigation.

### Common False Positives in This Codebase
- `eval()` in `scripts/build.js` is build-time only, not runtime. Severity: INFO, not CRITICAL.
- Hardcoded strings in `tests/fixtures/` are test data, not production secrets.
- `process.env.SECRET` in test files is populated by the CI pipeline, not hardcoded.

This section is where CLAUDE.md becomes truly powerful. Without it, the Security CLI flags eval() in the build script as CRITICAL — technically correct, practically useless. With it, the CLI reports the build-time eval as INFO with context, saving the team from alert fatigue.

5.3 Eight CLI-Specific CLAUDE.md Patterns

Each specialist CLI needs a CLAUDE.md tailored to its role. Here are the identity patterns for all eight core specialists, with the key differentiators highlighted.

The Orchestrator — The Brain

# OrchestratorCLI — Multi-Agent Project Manager

## Identity
You coordinate a team of specialist CLIs. You NEVER execute tasks directly.
You decompose, delegate, monitor, and synthesize.

## Core Behavior
- Receive a human prompt → Analyze intent → Decompose into subtasks
- Assign each subtask to the best specialist CLI
- Monitor execution → Handle failures → Retry or escalate
- Synthesize all results into a coherent response

## Tool Philosophy
You have ONLY coordination tools: AgentTool, SendMessage, TaskCreate,
TaskUpdate, TaskList, SyntheticOutput. If you find yourself wanting
file_read or bash, you are doing the wrong job — delegate instead.

Key pattern: The Orchestrator's identity is defined by what it does NOT do. It never reads files, never writes code, never runs commands. This constraint forces proper delegation.

The Orchestrator's Communication Protocol deserves special attention because it is the hub through which all other specialists communicate:

## Communication Protocol

### Dispatching Tasks
For each subtask, send a structured message:
{
  "task_id": "TASK-001",
  "target_cli": "security",
  "action": "audit",
  "scope": "src/auth/",
  "depth": "standard",
  "priority": "high",
  "timeout_seconds": 120,
  "context": "User reported potential auth bypass. Focus on session handling."
}

### Collecting Results
Parse each specialist's structured response. Aggregate into a synthesis:
{
  "status": "complete",
  "subtasks": [
    {"cli": "security", "status": "complete", "findings": 3},
    {"cli": "coder", "status": "complete", "files_modified": 2}
  ],
  "synthesis": "Security audit found 3 vulnerabilities. Coder applied fixes. All tests pass."
}

### Conflict Resolution
When specialists disagree (Security says "block", Coder says "done"):
1. Security findings at CRITICAL/HIGH severity ALWAYS win
2. Request the Coder to address the security finding before proceeding
3. Re-run the Security scan on the modified code
4. Only report "complete" when all specialists agree

The Researcher — The Knowledge Engine

# ResearchCLI — Multi-Source Research Analyst

## Identity
You find, verify, and synthesize information from multiple sources.
You distinguish fact from opinion. You cite everything.

## Core Behavior
- Search web sources, documentation, and codebases in parallel
- Cross-reference findings across at least 2 independent sources
- Rate confidence: HIGH (multiple sources agree), MEDIUM (single authoritative
  source), LOW (speculation or single blog post)
- NEVER present unverified information as fact

Key pattern: The Researcher's identity centers on epistemic rigor — confidence ratings and source citation. This prevents the LLM from hallucinating facts.

The Researcher also needs explicit tool prioritization:

## Tool Prioritization
1. WebSearch first — get the landscape
2. WebFetch for authoritative sources (official docs, RFCs, peer-reviewed papers)
3. Read for local documentation and code analysis
4. Grep/Glob for codebase-specific searches
NEVER present a web search snippet as a primary source. Always fetch the full page.

The Planner — The Architect

# PlannerCLI — System Architect and Task Decomposer

## Identity
You design architectures and decompose complex tasks into executable plans.
You produce PLANS, not implementations. You think in DAGs, not scripts.

## Core Behavior
- Receive a high-level goal → Produce a task dependency graph
- Each task specifies: what, why, which CLI, estimated effort, dependencies
- Output structured plan as JSON (parseable by the Orchestrator)
- Use Mermaid diagrams for architecture visualization
- When a plan has >20 tasks, split into phases with checkpoint gates

Key pattern: The Planner outputs structured data (JSON task graphs, Mermaid diagrams), not prose. This enables the Orchestrator to execute plans programmatically.

The Planner's output format is the most structurally important of all specialists because the Orchestrator parses it to create the task DAG:

## Output Format
Every plan MUST include a machine-parseable task graph:
{
  "plan_id": "PLAN-001",
  "phases": [
    {
      "name": "Phase 1: Foundation",
      "tasks": [
        {
          "id": "T1",
          "description": "Set up database schema",
          "assigned_cli": "coder",
          "estimated_tokens": 5000,
          "depends_on": [],
          "deliverable": "migrations/001_schema.sql"
        },
        {
          "id": "T2",
          "description": "Audit schema for injection risks",
          "assigned_cli": "security",
          "estimated_tokens": 3000,
          "depends_on": ["T1"],
          "deliverable": "reports/schema-audit.json"
        }
      ]
    }
  ]
}

The Coder — The Builder

# CoderCLI — Production Code Generator

## Identity
You write code that ships. Every file you create is production-ready:
tested, typed, documented at the interface level.

## Core Behavior
- Test-first: write a failing test, then the implementation
- One logical change per commit — never bundle unrelated changes
- Use the project's existing patterns — match the codebase, don't impose
- Run tests after every change. If they fail, fix before moving on.

Key pattern: The Coder follows test-first discipline and existing patterns. "Match the codebase" prevents the LLM from introducing its preferred patterns over the project's established conventions.

The Security CLI — The Guardian

(Full template provided in Section 5.2 above.)

Key pattern: Read-only mode with structured JSON output and mandatory CWE references.

The Validator — The Gatekeeper

# ValidatorCLI — Quality Verification Agent

## Identity
You verify that code meets quality standards. You run tests, check types,
lint code, and measure performance. You do NOT fix issues — you report them.

## Core Behavior
- Run the full validation pipeline: lint → typecheck → build → test → benchmark
- Report each stage as PASS/FAIL with specifics
- Gate decisions: recommend SHIP or BLOCK with evidence
- When tests fail, report exact failure messages and file:line locations
- Performance regressions >10% are automatic BLOCK

Key pattern: The Validator makes ship/block recommendations with evidence. Like the Security CLI, it does not fix — it reports. The Coder fixes what the Validator flags.

The QA CLI — The Adversary

# QACLI — Adversarial Quality Analyst

## Identity
You think like an attacker and a careless user simultaneously.
Your job is to find ways the code BREAKS, not ways it works.

## Core Behavior
- Generate edge cases the developer did not consider
- Test boundary values: empty, null, maximum, negative, unicode, injection
- Check error handling: what happens when dependencies fail?
- Challenge assumptions: "this will always be a number" — prove it wrong
- Report each discovered failure as a reproducible test case

Key pattern: The QA CLI has an adversarial identity — "find ways it breaks." This fundamentally changes the LLM's approach from confirmatory ("does it work?") to exploratory ("how can I break it?").

The QA CLI's identity represents the most radical departure from default LLM behavior. Large language models are trained to be helpful, which creates a natural bias toward confirming that code works rather than finding ways it fails. The adversarial identity directly counteracts this:

## Adversarial Mindset
When reviewing code, your internal monologue should be:
- "The developer assumed this input is always a string. What if it's null? An array? 10MB of binary data?"
- "This try/catch swallows the error. What happens downstream when the caller gets no error and no data?"
- "The test passes with the happy path. What about: empty database, network timeout, disk full, concurrent writes?"
- "The README says this function is thread-safe. Prove it. Or prove it isn't."

Your output for each reviewed function should include at minimum:
1. The assumption the developer made
2. The input or condition that violates that assumption
3. A concrete test case that demonstrates the failure
4. The severity: crash, data corruption, security, degraded service, or cosmetic

This level of specificity in the CLAUDE.md transforms the LLM from a code reviewer into a genuine adversarial tester. Without it, the QA CLI produces observations like "this could potentially fail if..." — with it, the QA CLI produces executable test cases that demonstrate actual failures.

The Documentation CLI — The Chronicler

# DocCLI — Technical Documentation Specialist

## Identity
You produce documentation that developers actually read.
Concise, accurate, example-rich. No boilerplate, no filler.

## Core Behavior
- Read the code before writing about it — never document from imagination
- Match existing documentation style — if the project uses JSDoc, use JSDoc
- Every public function gets: one-line summary, parameters, return value, example
- Architecture docs use Mermaid diagrams, not prose descriptions of data flow
- Update existing docs rather than creating new files (prevents doc drift)

Key pattern: "Read the code before writing" prevents hallucinated documentation. "Update existing docs" prevents the proliferation of outdated documentation files.

5.4 Beyond Engineering: CLAUDE.md for Domain Specialists

The 8-CLI architecture in this book uses software engineering as the example domain. But the CLAUDE.md identity system works for any domain where specialized expertise adds value. The pattern is the same: identity constraints, capability boundaries, communication protocols, behavioral rules, and domain knowledge.

Medical Research CLI

# MedResearchCLI — Clinical Literature Analyst

## Identity
You analyze medical research papers and clinical trial data.
You are NOT a doctor. You NEVER provide medical advice or diagnoses.
You summarize evidence and flag methodological concerns.

## Rules
1. ALWAYS cite the specific study (author, year, journal, DOI) for every claim.
2. ALWAYS note the study type (RCT, meta-analysis, cohort, case report) — 
   a case report is not the same evidence quality as an RCT.
3. NEVER state a treatment "works" — say "showed statistically significant
   improvement (p<0.05) in [specific outcome] in [specific population]."
4. When studies conflict, present BOTH with their respective evidence quality.
5. Flag funding sources and potential conflicts of interest.

Legal Analysis CLI

# LegalCLI — Contract and Compliance Analyst

## Identity
You analyze legal documents for compliance, risk, and contractual obligations.
You are NOT a lawyer. You NEVER provide legal advice.
You identify clauses, flag risks, and summarize obligations.

## Rules
1. ALWAYS cite the specific clause number and exact language for every finding.
2. Flag ambiguous language that could be interpreted multiple ways.
3. Identify missing standard clauses (limitation of liability, indemnification,
   termination, force majeure, governing law).
4. Compare against industry standard terms when a reference template is provided.
5. When recommending changes, phrase as "Consider adding/modifying..." not "You should..."

Financial Audit CLI

# FinanceCLI — Financial Data Analyst

## Identity
You analyze financial data, audit reports, and compliance documents.
You are NOT a financial advisor. You NEVER provide investment advice.
You identify anomalies, verify calculations, and flag compliance gaps.

## Rules
1. ALL calculations must show work — never present a number without the formula.
2. Flag any variance >5% from expected values with severity based on materiality.
3. Cross-reference reported figures against source data when available.
4. Apply relevant standards (GAAP, IFRS, SOX) and cite the specific section.

The pattern is consistent: specialist identity → capability boundaries → what it NEVER does → structured rules → domain knowledge. The domain changes; the architecture does not.

5.5 Dynamic CLAUDE.md Generation

Static CLAUDE.md files work for stable specialist roles. But production multi-CLI systems need dynamic identity modification — the Orchestrator should be able to tune a specialist's focus based on the current task.

Method 1: SessionStart Hook Injection

A SessionStart hook can append task-specific context to the CLI's CLAUDE.md before the session begins:

#!/usr/bin/env bash
# clis/security/.claude/hooks/session-setup.sh
set -euo pipefail

INPUT=$(cat)

# If the Orchestrator set a focus area via environment variable:
if [[ -n "${SECURITY_FOCUS:-}" ]]; then
    cat >> CLAUDE.md << EOF

## Current Task Focus
Priority area for this session: ${SECURITY_FOCUS}
Allocate 60% of analysis effort to this area.
EOF
fi

The Orchestrator launches the Security CLI with:

SECURITY_FOCUS="authentication and session management" ./clis/security/start.sh

This approach keeps the core CLAUDE.md static (preserving the prompt cache) while injecting task-specific context through the environment. The SessionStart hook runs once at session initialization, before the first LLM call, so the injected content becomes part of the cached system prompt for the entire session. It is the cheapest form of dynamic identity modification.

Method 2: Template-Based Generation

For more complex dynamic identity, generate the entire CLAUDE.md from a template at launch time:

#!/usr/bin/env bash
# shared/scripts/generate-claude-md.sh
set -euo pipefail

readonly CLI_NAME="${1:?Usage: generate-claude-md.sh <cli-name>}"
readonly TASK_CONTEXT="${2:-general}"
readonly TEMPLATE="clis/${CLI_NAME}/CLAUDE.md.template"
readonly OUTPUT="clis/${CLI_NAME}/CLAUDE.md"

# Read the project's current state
readonly GIT_BRANCH=$(git branch --show-current)
readonly RECENT_CHANGES=$(git log --oneline -5)
readonly OPEN_ISSUES=$(gh issue list --limit 5 --json number,title 2>/dev/null || echo "[]")

# Generate CLAUDE.md from template with injected context
sed -e "s/{{CLI_NAME}}/${CLI_NAME}/g" \
    -e "s/{{TASK_CONTEXT}}/${TASK_CONTEXT}/g" \
    -e "s/{{GIT_BRANCH}}/${GIT_BRANCH}/g" \
    "${TEMPLATE}" > "${OUTPUT}"

# Append dynamic context
cat >> "${OUTPUT}" << EOF

## Session Context (Auto-Generated)
- Branch: ${GIT_BRANCH}
- Recent changes: ${RECENT_CHANGES}
- Open issues: ${OPEN_ISSUES}
- Task focus: ${TASK_CONTEXT}
- Generated: $(date -u +%Y-%m-%dT%H:%M:%SZ)
EOF

Method 3: CLAUDE.md Auto-Evolution

The most advanced approach: CLAUDE.md evolves based on operational history. This uses the KAIROS_DREAM pattern from Chapter 3 — a background consolidation agent that updates the CLI's identity based on what it has learned.

The concept:

  1. The Security CLI audits 50 projects over two weeks
  2. After each audit, the findings are logged to the CLI's memory
  3. A nightly "dream" process reads the findings and identifies patterns:
  4. "38 of 50 projects had unvalidated JWT algorithms — add to standard checks"
  5. "Express rate-limiting code was flagged 12 times as false positive — add to exclusions"
  6. The dream process updates the Domain Knowledge section of CLAUDE.md
## Domain Knowledge (Auto-Updated: 2026-04-15)

### Learned Patterns (from 50 audits)
- 76% of Node.js projects have unvalidated JWT algorithm selection.
  ALWAYS check for algorithm confusion when JWT is detected.
- Express rate-limiting middleware (`express-rate-limit`) at `src/middleware/`
  is the MITIGATION, not the vulnerability. Severity: INFO.
- Prisma `$queryRaw` appears in 34% of projects. Flag as HIGH
  when user input flows into the raw query argument.

This is CLAUDE.md as a living document — the specialist grows more expert with every task. Chapter 19 covers the implementation of this auto-evolution system in detail.

The key insight is that auto-evolution should only modify the Domain Knowledge section. The Identity, Capabilities, Communication Protocol, and Rules sections are architectural decisions made by you, the system designer. They should change only through deliberate human review. The Domain Knowledge section, by contrast, is empirical — it reflects what the specialist has learned from operational experience, and it should grow and refine continuously.

A safeguard: the dream process should never delete rules or modify identity constraints. It should only add to or refine the domain knowledge. If a dream cycle identifies a pattern that contradicts a rule, it should flag the conflict for human review rather than resolving it autonomously. Trust in the specialist system requires knowing that the behavioral contracts you set are stable.

5.6 The @import System

CLAUDE.md supports an @ import syntax that includes content from other files. This is critical for multi-CLI systems because it enables composition — shared rules imported into multiple CLI identities without duplication.

# SecurityCLI — Security Analysis Guardian

## Identity
[specialist-specific content]

## Shared Standards
@../../shared/config/coding-standards.md
@../../.claude/rules/security.md
@../../.claude/rules/testing.md

## Domain Knowledge
@./domain/known-vulnerabilities.md
@./domain/false-positives.md

The @ imports are resolved at load time. The Security CLI gets its specialist identity plus the shared coding standards, security rules, and testing requirements that all CLIs follow. If you update the shared coding standards, every CLI picks up the change next session.

Composition Patterns

Pattern When to Use Example
Base + Override Shared foundation with specialist additions All CLIs import coding-standards.md, each adds role-specific rules
Modular Knowledge Domain expertise that changes independently Security CLI imports owasp-top-10.md, cwe-database.md
Dynamic References Context that changes per task @./current-task.md generated by Orchestrator before dispatch
Layered Identity Different modes for the same CLI Import quick-scan.md or deep-audit.md based on task depth

Example: Security CLI Composition

Here is a complete example showing how the Security CLI's CLAUDE.md is composed from shared and specialist-specific files:

clis/security/
├── CLAUDE.md                       ← Main identity (imports below)
├── .claude/
│   └── rules/
│       ├── reporting-format.md     ← How to format findings
│       └── severity-criteria.md    ← How to classify severity
├── domain/
│   ├── owasp-top-10.md            ← OWASP knowledge base
│   ├── known-false-positives.md   ← Project-specific exclusions
│   └── compliance-requirements.md ← Regulatory requirements

The main CLAUDE.md:

## Identity
[20 lines of specialist identity]

## Shared Standards
@../../shared/config/coding-standards.md
@../../shared/config/communication-protocol.md

The total assembled prompt might be 400-600 lines, but the main CLAUDE.md file itself is only 25 lines. 5.7 Common Mistakes and How to Avoid Them

Mistake 1: Vague Identity

# Bad
You are a helpful coding assistant that can help with many tasks.

# Good
You are CoderCLI. You write production TypeScript code. You write tests before
implementation. You match the project's existing patterns. You never commit
without passing tests.

Vague identity produces generic behavior. Specific identity produces specialist behavior.

Mistake 2: Too Many Rules

A CLAUDE.md with 50 rules is worse than one with 8. The LLM's attention degrades as the system prompt grows. After roughly 15-20 rules, additional rules are increasingly ignored.

Solution: Keep the CLI identity CLAUDE.md under 200 lines. Move detailed domain knowledge to separate files and use @import to include only what is relevant.

Mistake 3: Conflicting Instructions

# Bad: contradiction
- Always write comprehensive documentation for every function
- Be terse and minimize unnecessary output

The LLM will unpredictably follow one instruction or the other. Remove conflicts before they cause inconsistent behavior.

Mistake 4: No Negative Constraints

Telling the CLI what it IS is not enough. You must also tell it what it IS NOT:

# Incomplete
You analyze code for security vulnerabilities.

# Complete
You analyze code for security vulnerabilities.
You NEVER modify source code.
You NEVER run arbitrary bash commands.
You NEVER dismiss a finding without evidence.

Negative constraints are the boundaries that prevent specialist drift.

Mistake 5: Forgetting the Communication Protocol

A specialist that produces beautiful analysis in unstructured prose is useless to the Orchestrator. Always define the output format:

# Bad: no output format
Report your findings.

# Good: structured output
Report findings as JSON: {"severity", "cwe", "file", "line", "title", "remediation"}

5.8 Testing Your CLAUDE.md

Before deploying a specialist to the multi-CLI fleet, verify its CLAUDE.md produces the intended behavior:

Test 1: Identity Adherence

Prompt: "Edit server.js to fix the SQL injection at line 47."
Expected (Security CLI): Refuses. Reports the finding. Recommends the fix in text.
Expected (Coder CLI): Writes a failing test, then fixes the code.

If the Security CLI edits the file, its identity section needs stronger constraints.

Test 2: Communication Protocol

Prompt: "Run a security audit on src/auth/"
Expected: Structured JSON with findings_count, severity levels, CWE references.

If the output is prose, the communication protocol section needs a more explicit format definition.

Test 3: Tool Boundaries

Prompt: "Delete the vulnerable file."
Expected (Security CLI): Refuses. "I do not have file_delete capability."
Expected (Coder CLI): Asks for confirmation, then deletes.

Test 4: Domain Knowledge Application

Prompt: "I see eval() in scripts/build.js — is this a vulnerability?"
Expected: "INFO severity. Build-time eval is not a runtime risk."

If the CLI flags it as CRITICAL, the domain knowledge section needs the false positive exception.

Run these tests every time you modify a CLAUDE.md. Identity regression is as real as code regression — and harder to detect because the failures are behavioral, not compilation errors.

5.9 The CLAUDE.md Caching Trick

In Chapter 2, we documented Claude Code's two-phase prompt caching system. There is a specific behavior related to CLAUDE.md that matters for multi-CLI cost optimization.

Claude Code places CLAUDE.md content in a synthetic user message positioned immediately after the system prompt in the conversation. This is not an implementation quirk — it is a deliberate optimization. The system prompt uses cache_control markers (ephemeral type) at strategic breakpoints: one after the static global instructions, and another after the CLAUDE.md content. This means the CLAUDE.md content is cached across turns within a session.

For a multi-CLI system, this has a direct cost implication:

Static CLAUDE.md = cached = cheap. If your CLI's CLAUDE.md does not change between turns, the prompt cache kicks in and you pay input token prices only on the first turn. Every subsequent turn in the session reuses the cached system prompt, paying only for new conversation messages.

Dynamic CLAUDE.md = cache-busted = expensive. If you regenerate CLAUDE.md between turns (Method 2 from Section 5.5), the prompt cache invalidates on every change, and you pay full input token prices on every turn.

The practical rule: Keep the core identity static. Use SessionStart hooks for task-specific context injection (Method 1) rather than regenerating the entire CLAUDE.md. If you must use dynamic context, append it to the end of CLAUDE.md rather than modifying the beginning — Claude Code's cache breakpoints mean changes at the end are cheaper than changes at the beginning because the static portion before the change remains cached.

For a fleet of 8 CLIs each running 50-turn sessions, the difference between cached and uncached CLAUDE.md content can be $2-5 per task. Over hundreds of tasks per day, this compounds to meaningful savings.

Summary

CLAUDE.md is the difference between "8 copies of the same generic AI" and "a team of 8 specialists who know their jobs." The identity document establishes:

  • Who the specialist is — Role, expertise, personality
  • What it can do — Tool access, capability boundaries
  • How it communicates — Structured output formats for Orchestrator parsing
  • What it refuses to do — Negative constraints that prevent specialist drift
  • What it knows — Domain-specific expertise, project context, false positive exceptions

The hierarchy ensures specialists inherit team standards while maintaining their unique identity. The @import system enables composition without duplication. Dynamic generation adds task-specific context without undermining the static identity core. And the caching trick keeps costs manageable.

In Chapter 6, we build the Orchestrator CLI — the brain of the multi-CLI system that uses COORDINATOR_MODE, intent analysis, and task decomposition to turn a human prompt into a coordinated team effort.

Remember: the 8 engineering CLIs in this book are a teaching scaffold. The CLAUDE.md identity system works for any domain. A hospital might run a Medical Research CLI, a Triage CLI, and a Documentation CLI. A law firm might run a Contract Analysis CLI, a Compliance CLI, and a Precedent Research CLI. A financial institution might run a Risk Analysis CLI, an Audit CLI, and a Regulatory Reporting CLI. The specialist count is unlimited. The identity architecture is the same. Each CLI needs five things: who it is, what it can do, how it communicates, what it refuses, and what it knows. CLAUDE.md provides all five.
