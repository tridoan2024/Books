    query: web::Query<BookQuery>,

    let books = sqlx::query_as!(
        Book,
        r#"
        SELECT id, title, author, isbn, price, stock, description, created_at
        FROM books
        WHERE ($1::text IS NULL OR title ILIKE '%' || $1 || '%')
        ORDER BY created_at DESC
        LIMIT $2 OFFSET $3

    .await

    .fetch_one(pool.get_ref())
    .await
    .map_err(ApiError::Database)?;

    Ok(Json(PaginatedResponse {
        data: books,
        total: total.unwrap_or(0) as u64,
        page: query.page.unwrap_or(1),
        per_page: query.per_page.unwrap_or(20),
    }))

JWT with refresh token rotation:

// src/infrastructure/auth/jwt.rs

pub struct AuthService {
    access_secret: String,
    refresh_secret: String,
    pool: PgPool,
}

impl AuthService {
    pub async fn login(&self, email: &str, password: &str) -> Result<TokenPair, AuthError> {
        let user = sqlx::query_as!(User, "SELECT * FROM users WHERE email = $1", email)
            .fetch_optional(&self.pool)
            .await?
            .ok_or(AuthError::InvalidCredentials)?;

        // Verify password with argon2
        let parsed_hash = PasswordHash::new(&user.password_hash)
            .map_err(|_| AuthError::InvalidCredentials)?;
        Argon2::default()
            .verify_password(password.as_bytes(), &parsed_hash)
            .map_err(|_| AuthError::InvalidCredentials)?;

        // Generate token pair
        let access_token = self.generate_access_token(&user)?;
        let refresh_token = self.generate_refresh_token(&user)?;

        // Store refresh token hash (not the token itself)
        let token_hash = sha256_hash(&refresh_token);
        sqlx::query!(
            "INSERT INTO refresh_tokens (user_id, token_hash, expires_at) VALUES ($1, $2, $3)",
            user.id,
            token_hash,
            Utc::now() + Duration::days(7),
        )
        .execute(&self.pool)
        .await?;

        Ok(TokenPair { access_token, refresh_token })
    }
}

// Resolution: HMAC-SHA256 (1-line change, defense-in-depth)
fn hash_refresh_token(&self, token: &str) -> String {
    assert!(token.len() >= 32, "Refresh token entropy too low");
    let mut mac = HmacSha256::new_from_slice(self.refresh_secret.as_bytes())
        .expect("HMAC key length is valid");
    mac.update(token.as_bytes());
    hex::encode(mac.finalize().into_bytes())
}

// S-004 resolution: Rate limiter for auth endpoints
pub struct RateLimiter {
    attempts: Arc<DashMap<IpAddr, Vec<Instant>>>,
    max_attempts: usize,
    window: Duration,
}

impl RateLimiter {
    pub fn check(&self, ip: IpAddr) -> Result<(), AuthError> {
        let mut attempts = self.attempts.entry(ip).or_default();
        let now = Instant::now();
        attempts.retain(|t| now.duration_since(*t) < self.window);
        if attempts.len() >= self.max_attempts {
            return Err(AuthError::RateLimited {
                retry_after: self.window - now.duration_since(attempts[0]),
            });
        }
        attempts.push(now);
        Ok(())
    }
}

// BUG-004 fix: SELECT ... FOR UPDATE to prevent stock race condition
pub async fn create_order(pool: &PgPool, order: CreateOrder) -> Result<Order, ApiError> {
    let mut tx = pool.begin().await?;

    for item in &order.items {
        let book = sqlx::query_as!(
            Book,
            "SELECT * FROM books WHERE id = $1 FOR UPDATE", // Lock the row
            item.book_id
        )
        .fetch_one(&mut *tx)
        .await
        .map_err(|_| ApiError::NotFound("Book not found"))?;

        if book.stock < item.quantity {
            return Err(ApiError::Conflict(format!(
                "Insufficient stock for '{}': requested {}, available {}",
                book.title, item.quantity, book.stock
            )));
        }

        sqlx::query!(
            "UPDATE books SET stock = stock - $1 WHERE id = $2",
            item.quantity, item.book_id
        )
        .execute(&mut *tx)
        .await?;
    }

    // ... insert order + items, compute total from actual prices
    tx.commit().await?;
    Ok(order)
}

## 24.12 What Went Wrong

No build is perfect. Here is what the forge got wrong:

1. **The frontend's error boundary was missing.** The Frontend CLI generated React components without error boundaries. A runtime error in any component would crash the entire page. The Reviewer CLI caught this during integration review and added `ErrorBoundary` wrappers — an 8-minute fix.

2. **The Docker health check timing was too aggressive.** The API container's health check started at 5-second intervals immediately on startup. PostgreSQL needs 10-15 seconds to initialize. The first few health checks failed, triggering unnecessary restarts. The DevOps CLI adjusted the start period to 15 seconds.

3. **SQLx offline mode was not configured.** The Coder CLI generated `sqlx::query!` macros, which require a live database connection at compile time. For Docker builds, this means the build container needs database access — a CI/CD complication. The fix: generate SQLx offline data (`cargo sqlx prepare`) and check it into the repository.

4. **The Security CLI's rate limit finding was ignored.** The LOW-severity recommendation to add API keys to rate limit keys was logged but not implemented. In a production deployment, this would need addressing before launch.

None of these issues were catastrophic. All were caught before delivery (items 1-3 by the forge itself, item 4 flagged in the security report). The human equivalent would have the same class of issues — missed edge cases, CI/CD configuration bugs, deferred security improvements.

## 24.13 Lessons from the Build

This case study is not just a demonstration. It is a data point. Here are the lessons that generalize beyond this specific build:

**Lesson 1: Model mixing is not optional — it is the economic foundation.** Using Opus for everything would have cost $7+ and taken longer (Opus generates fewer tokens per second). The forge's model routing — Opus for planning and security, Sonnet for implementation, Haiku for docs and devops — optimizes both cost and speed. The Planner and Security CLI *need* Opus because their tasks require deep reasoning about system design and vulnerability analysis. The Coder *needs* Sonnet because it generates the most tokens and Haiku would produce lower-quality code. The DevOps and Documentation CLIs run fine on Haiku because their tasks follow well-established patterns. As discussed in Chapter 22's cost analysis, this mixing strategy saves approximately 48% compared to all-Opus execution. For an enterprise running hundreds of builds per week, this is the difference between a viable tool and a financial sink.

**Lesson 2: The debate protocol catches real bugs that single agents miss.** The HMAC-SHA256 debate at T+3:30 was not busywork. It resulted in a concrete security improvement — defense-in-depth that prevents a class of regression risk. A single-agent system would have either shipped SHA-256 (the Coder's default) or demanded Vault (the Security CLI's maximal position). The debate produced a third option that satisfied both constraints at minimal cost. This pattern repeated with every security finding: the initial finding was often too aggressive, the Coder's pushback was often too permissive, and the negotiated resolution was better than either starting position.

**Lesson 3: Continuous security review is categorically different from post-hoc review.** In a traditional workflow, the security review happens after the code is written. Every finding requires the author to context-switch back to code they wrote hours or days ago. In the forge, Security flagged the JWT secret issue five seconds after the Coder wrote it. The fix took 90 seconds because the code was still fresh in the Coder's context window. Five seconds versus five days. That is the difference between a finding that gets fixed and a finding that gets deferred to "next sprint" and eventually forgotten.

**Lesson 4: Parallelism is the forge's superpower.** The frontend, security audit, and test suite all ran simultaneously between T+6:00 and T+12:00. A sequential system would have taken 12+ additional minutes — more than doubling the wall-clock time. But parallelism only works when the task DAG defines clear boundaries. The Planner's architecture — which separated backend (`src/`), frontend (`frontend/`), tests (`tests/`), and infrastructure (root-level configs) — made zero-conflict parallel execution possible. A bad architecture with unclear boundaries would have produced merge conflicts at T+16:00 that negate the parallel gains.

**Lesson 5: The QA red-green cycle catches bugs that code review misses.** Eight of the forty-four tests failed on first run. Not syntax errors — logic bugs. The TOCTOU race condition in order creation (BUG-004) is the kind of bug that passes every code review because it only manifests under concurrent load. The admin role escalation (BUG-005) is the kind of bug where the developer *thought* they applied the middleware but missed one route. Separate CLIs with separate context windows prevent the "I just wrote this, it's probably fine" blindness.

**Lesson 6: The Orchestrator matters more than any specialist.** The quality of the task DAG — which tasks can run in parallel, which must serialize, which CLIs get which assignments, which file paths belong to which worktree — determines the total throughput. A bad DAG serializes everything and wastes the forge's parallel capacity. A worse DAG assigns overlapping file ownership and creates merge conflicts. The Orchestrator's two-minute investment in architecture and DAG generation prevented twenty-one minutes of potential rework.

**Lesson 7: The first version is not the final version.** The forge produces a working, tested, deployable application in 23 minutes. But it is the *first* version. The human role shifts from building to curating — reviewing the forge's output, adjusting architecture decisions, improving UX, adding the nuanced touches that make software feel crafted rather than generated. The four "What Went Wrong" items above are exactly the kind of issues that a human reviewer catches in 15 minutes. The forge's 23-minute build plus a human's 15-minute review equals a 38-minute delivery that is better than either human-only or forge-only could produce.

// NEW: src/middleware/authorization.ts
export async function authorizePatientAccess(
  req: AuthenticatedRequest, res: Response, next: NextFunction
) {
  const requestedPatientId = req.params.id;
  const userId = req.user.sub;
  const userRole = req.user.role;

  // Clinical staff can access any patient (audited)
  if (['admin', 'doctor', 'nurse'].includes(userRole)) {
    await auditLog('patient_access', { userId, requestedPatientId, role: userRole });
    return next();
  }

  // Patients can only access their own records
  const ownership = await db.query(
    'SELECT 1 FROM patients WHERE id = $1 AND user_id = $2',
    [requestedPatientId, userId]
  );

  if (ownership.rowCount === 0) {
    await auditLog('unauthorized_access_attempt', { userId, requestedPatientId });
    return res.status(403).json({ error: 'Access denied' });
  }
  next();
}

25.10 Lessons from the Engagement

Lesson 1: Separation of Duties Is Not Optional

The Security CLI cannot write code. The Coder CLI cannot run scans. The QA CLI cannot fix what it breaks. This separation is not convenience — it is the architectural property that makes the audit trustworthy. The moment your scanner can also fix, you lose auditability. The moment your fixer can also certify, you lose integrity. The multi-CLI architecture enforces the same separation of duties that financial auditing has required for centuries.

Lesson 2: Attack Chains Matter More Than Finding Counts

SQL-001 as an isolated finding is useful. Proving that SQL-001 chains with no rate limiting and no anomaly detection to exfiltrate 300,000 patient records in under sixty seconds is actionable. The QA CLI's adversarial phase transforms vulnerability counts into risk narratives. Executives do not fund remediation for "47 findings." They fund remediation for "an attacker can steal every patient record in our database."

Lesson 3: Continuous Beats Comprehensive

A $117,000 audit that runs annually is less effective than a $14.73 audit that runs weekly. Security is not a state to achieve. It is a process to maintain. KAIROS transforms security from periodic checkpoint into continuous property.

Lesson 4: Compliance Is a Side Effect of Security

Every compliance gap we closed was closed by fixing an actual vulnerability. We did not implement controls to satisfy a checklist. We fixed real attack vectors and then demonstrated the fixes satisfied compliance requirements. When security is done right, compliance comes for free. When compliance is done first, security often does not follow.

Lesson 5: The Auditor's Expertise Still Matters

The multi-CLI system executed the audit. But the system prompt configuration, the threat model prioritization, the compliance mapping, and the risk narratives all reflect security expertise encoded in the system's design. The Research CLI knows to check OWASP and NVD because a security professional configured it. The QA CLI knows to chain IDOR with JWT algorithm confusion because an adversarial methodology was encoded in its prompt. The machine executes at superhuman speed. The strategy is still human.

This is the pattern described across the preceding twenty-four chapters. The forge is not a replacement for security expertise. It is an amplifier. A single security professional operating the forge produces the output of a ten-person audit team at a fraction of the cost and a fraction of the time. The expertise designs the architecture. The forge executes at scale.

Lesson 6: Infrastructure Defenses Are Not Code Defenses

The SQL injection attack in Chain 1 was blocked from writing data because the application connected to a read-only PostgreSQL replica. That is an infrastructure defense — accidental, not intentional. It prevented data modification but did nothing to prevent data exfiltration. The development team might have pointed to the read-only replica and said "we're protected." They were not. The attacker cannot modify records but can read every single one. Infrastructure defenses are necessary but they are not sufficient. Application-layer defenses — parameterized queries, input validation, authorization checks — are what actually prevent the attack. The multi-CLI audit distinguishes between infrastructure-accidental protections and application-intentional protections. Only intentional protections count toward compliance.
