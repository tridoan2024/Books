Chapter 16: Hooks, Skills, and the Extension Stack

Lifecycle Events, Quality Gates, Reusable Workflows, and Self-Improving Skills

You have built the CLIs. You have wired their memory (Chapter 14) and connected them through communication channels (Chapter 15). But right now, every CLI in the forge is a sealed box — it receives instructions, executes them, reports results. There is no way for the system to react. No way for one CLI's action to automatically trigger another CLI's response. No way for the forge itself to enforce quality gates, inject context, or evolve its own capabilities over time.

Hooks and skills are the extension stack that transforms independent CLIs into a reactive, self-improving system. If the communication protocol from Chapter 15 is the forge's nervous system, hooks are its reflexes and skills are its muscle memory. Hooks fire when things happen: a session starts, a tool executes, a file changes, an agent spawns. Skills encode reusable workflows that any CLI can invoke, share, and — behind the SKILL_IMPROVEMENT feature flag — improve automatically based on outcomes.

In a production multi-CLI setup, you need three things that neither memory nor communication alone provide:

Reactive coordination. When the Coder CLI writes a file, the Validator should automatically run linting. When the Validator finds issues, the Security CLI should scan for secrets. This should not require polling, manual triggering, or explicit orchestration messages. It should happen — reliably, within seconds, every time.

Quality gates. Before any CLI executes a dangerous operation — deploying to production, deleting files, running untrusted code — the system should enforce checks. Not after the fact. As hard gates that block execution until conditions are met.

Capability evolution. When a skill works, the system should reinforce it. When a skill fails, the system should refine its prompt. Static configurations do not survive contact with real codebases. The extension stack must learn.

Claude Code ships with twenty-seven hook lifecycle events, four hook implementation patterns, a full skills system with auto-discovery, and mechanisms for skill auto-improvement and MCP-based skill serving. Most users never touch any of it. This chapter walks through every piece.

16.1 What Hooks Are

A hook is a function that fires when a specific lifecycle event occurs. The system maintains an event bus — implemented in utils/hooks/hookEvents.ts — and when something happens (session starts, tool executes, permission requested), the bus emits an event. Any registered hook function matching that event type receives the payload and can act on it.

The implementation spans several source files:

utils/hooks/hookEvents.ts   -- Event bus, type registry
utils/hooks/hookRunner.ts   -- Executes handlers (bash, HTTP, inline)
schemas/hooks.ts            -- Configuration schema
utils/sessionStart.ts       -- processSessionStartHooks(), processSetupHooks()

Hooks are not blocking by default. They fire asynchronously, and the system continues while hooks execute. The exceptions are specific categories — PreToolUse and permission hooks — where the system must wait because the hook's return value determines whether the operation proceeds. Understanding which hooks block and which do not is the difference between a responsive forge and a deadlocked one.

Hook Configuration: Three Levels

Level Location Scope
User ~/.claude/hooks/ All sessions, all projects
Project .claude/hooks/ (in repo) All sessions in this project
Session Programmatic via SDK Single session only

Project hooks are version-controlled — the entire team shares the same quality gates. User hooks are personal and override project hooks. Session hooks are ephemeral, created programmatically when the Orchestrator spawns a CLI, and destroyed when the session ends.

SessionStart is where you bootstrap multi-CLI coordination. The Coder's SessionStart hook announces itself to the IPC bus, registers capabilities, and pulls queued tasks. SessionEnd deregisters from the bus and flushes pending notifications.

PreToolUse is the single most important hook for multi-CLI coordination. Because it blocks, it is your enforcement point. A Security CLI can register a PreToolUse hook that intercepts every bash tool call across the forge, scans for dangerous patterns, and returns a deny verdict.

PostToolUse is the coordination trigger. When the Coder finishes writing a file, the hook fires with the path. Your hook sends that path to the IPC bus, where Validator and Security CLIs listen. Pure event-driven reactivity.

Event When Blocking?
PermissionRequest Tool needs approval Yes
PermissionDenied Permission denied No
PermissionGranted Permission approved No

In the multi-CLI forge, headless CLIs cannot have interactive prompts. Permission hooks automate the decision: the Orchestrator registers hooks with security policies. Trusted operations auto-approve. Risky operations route to the Security CLI. Dangerous operations deny outright.

TeammateIdle, TaskCreated, and TaskCompleted are the orchestration trinity. When the Coder CLI finishes and becomes idle, TeammateIdle fires. The Orchestrator's hook catches it, checks the queue, and assigns the next task immediately. When a new task is created, the Orchestrator checks for idle agents. This bidirectional matching ensures zero idle time when work is available.

CostThreshold is essential for forge management. Five CLIs in parallel burn tokens fast. Register hooks that report spend to the Orchestrator. When aggregate cost crosses budget, pause non-critical CLIs or switch to cheaper models.

File system events use glob patterns for filtering — register for src/**/*.rs to receive only Rust file changes. This builds targeted pipelines: Rust changes trigger cargo check, config changes trigger validation, migration changes trigger the database CLI.

PreSampling hooks are prompt hooks — they can mutate the system prompt before the API call. The Orchestrator can inject real-time context: "The Security CLI just found a critical vulnerability in auth.rs — prioritize fixing it." The CLI receives this as part of its system prompt without consuming a conversation turn.

16.3 Four Hook Implementation Patterns

Pattern 1: Command Hooks (Shell Scripts)

The simplest pattern. A shell script executes when the event fires. Event payload arrives via stdin as JSON.

pub struct CommandHook {
    pub event: HookEvent,
    pub command: String,
    pub timeout_ms: u64,
    pub matcher: Option<String>, // tool name filter for PreToolUse
}

impl CommandHook {
    pub fn execute(&self, payload: &HookPayload) -> HookResult {
        let payload_json = serde_json::to_string(payload).unwrap();

        let output = std::process::Command::new("bash")
            .arg("-c")
            .arg(&self.command)
            .stdin(std::process::Stdio::piped())
            .stdout(std::process::Stdio::piped())
            .stderr(std::process::Stdio::piped())
            .spawn()
            .and_then(|mut child| {
                if let Some(mut stdin) = child.stdin.take() {
                    stdin.write_all(payload_json.as_bytes()).ok();
                }
                child.wait_with_output()
            });

        match output {
            Ok(output) => {
                match output.status.code() {
                    Some(0) => HookResult::Allow,
                    Some(2) => {
                        let reason = String::from_utf8_lossy(&output.stderr);
                        HookResult::Deny(reason.to_string())
                    }
                    _ => HookResult::Allow, // Non-blocking error
                }
            }
            Err(_) => HookResult::Allow, // Hook failure = allow (fail-open)
        }
    }
}

Exit code 0 means allow. Exit code 2 means deny — the tool call is blocked. Any other exit code is a non-blocking error (logged, continues). This convention from Claude Code's hook system makes security gates simple: write a script that exits 2 when it detects a dangerous pattern.

Pattern 2: HTTP Hooks (Webhooks)

For remote coordination — notifying external systems, calling CI/CD APIs, integrating with monitoring.

pub struct HttpHook {
    pub event: HookEvent,
    pub url: String,
    pub method: String,
    pub headers: HashMap<String, String>,
    pub timeout_ms: u64,
}

HTTP hooks are always fire-and-forget (non-blocking). Using a blocking HTTP hook for PreToolUse would add network latency to every tool call — unacceptable for a responsive forge.

Pattern 3: Prompt Hooks (PreSampling Injection)

Prompt hooks mutate the system prompt before API calls. This is how you inject real-time forge state into every CLI's context:

pub struct PromptHook {
    pub event: HookEvent, // Must be PreSampling
    pub injection_builder: Box<dyn Fn(&ForgeState) -> String>,
}

impl PromptHook {
    pub fn build_injection(&self, state: &ForgeState) -> String {
        let mut lines = Vec::new();

        // What other CLIs are doing
        for cli in &state.active_clis {
            if cli.is_active {
                lines.push(format!("- {}: {}", cli.role, cli.current_task.as_deref().unwrap_or("idle")));
            }
        }

        // Recent findings from other CLIs
        for finding in state.recent_findings.iter().rev().take(5) {
            lines.push(format!("- [{}] {}: {}", finding.severity, finding.source, finding.message));
        }

        // Pending task count
        let pending = state.task_queue.iter().filter(|t| t.status == "pending").count();
        if pending > 0 {
            lines.push(format!("- {} tasks pending in queue", pending));
        }

        if lines.is_empty() {
            return String::new();
        }

        format!("\n<!-- FORGE CONTEXT -->\n{}\n<!-- END FORGE CONTEXT -->", lines.join("\n"))
    }
}

Warning: Prompt hooks add latency to every API call. Keep them under 50ms. Read from local files or shared memory, never from remote APIs. Cache aggressively.

Pattern 4: Agent Hooks (Sub-Agent Lifecycle)

Agent hooks fire when sub-agents are created, complete, or fail. The Orchestrator uses them as its primary coordination mechanism:

pub fn handle_agent_event(event: &AgentEvent, forge: &mut ForgeState) {
    match event {
        AgentEvent::Spawn { agent_id, agent_type, .. } => {
            forge.register_agent(agent_id, agent_type);
        }
        AgentEvent::Complete { agent_id, result, .. } => {
            forge.mark_complete(agent_id, result);
            // Check queue for next task
            if let Some(task) = forge.dequeue_for_type(&forge.agent_type(agent_id)) {
                forge.assign_task(agent_id, &task);
            }
        }
        AgentEvent::Idle { agent_id, .. } => {
            if let Some(task) = forge.dequeue_for_type(&forge.agent_type(agent_id)) {
                forge.assign_task(agent_id, &task);
            } else {
                forge.mark_idle(agent_id);
            }
        }
        AgentEvent::Error { agent_id, error, .. } => {
            forge.handle_failure(agent_id, error);
        }
    }
}

16.4 Hook-Driven Coordination Walkthrough

Trace a single event — the Coder CLI writes a file — through the entire hook chain.

T+0ms: PreToolUse fires. Coder calls file_write for src/middleware/rate_limiter.rs. Before the write executes, the PreToolUse hook fires. The security hook reads the tool input from stdin:

// Simplified PreToolUse handler
fn check_write_permission(payload: &HookPayload) -> HookResult {
    let file_path = payload.tool_input.get("file_path")
        .and_then(|v| v.as_str())
        .unwrap_or("");

    // Coder is restricted to src/ directory
    if !file_path.starts_with("src/") {
        return HookResult::Deny(format!(
            "Coder CLI restricted to src/ directory. Attempted: {}",
            file_path
        ));
    }

    // Block writes to security-critical paths
    let restricted = ["src/auth/keys/", "src/config/secrets/"];
    for prefix in &restricted {
        if file_path.starts_with(prefix) {
            return HookResult::Deny(format!(
                "Write to security-critical path blocked: {}", file_path
            ));
        }
    }

    HookResult::Allow
}

The path is src/middleware/rate_limiter.rs — allowed. Not in a restricted prefix. The write proceeds.

T+15ms: PostToolUse fires. The file write completes. PostToolUse fires with the file path in the payload. The command hook sends the event to the IPC bus:

#!/usr/bin/env bash
set -euo pipefail

INPUT=$(cat)
FILE_PATH=$(echo "${INPUT}" | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_input',{}).get('file_path',''))")

if [[ -z "${FILE_PATH}" ]]; then
  exit 0
fi

# Send to IPC bus
SOCKET="/run/forge/ipc.sock"
if [[ -S "${SOCKET}" ]]; then
  printf '{"event":"file_changed","path":"%s","source":"%s"}' \
    "${FILE_PATH}" "${CLAUDE_SESSION_ID:-unknown}" \
    | socat - UNIX-CONNECT:"${SOCKET}"
fi

T+50ms: Validator CLI receives the event. Spawns a targeted check:

cargo check --message-format=json 2>&1 | head -50
cargo clippy --message-format=json -- -W clippy::all 2>&1 | head -50

Reports results to IPC: pass/fail with diagnostic messages.

T+85ms: Security CLI receives the event simultaneously. Runs three checks on the changed file:

  1. Secret scan — regex patterns for API keys, passwords, tokens (Shannon entropy > 4.5 for strings longer than 8 characters)
  2. Injection check — unsanitized user input flowing into shell commands, SQL queries, or file paths
  3. Unsafe pattern detection — unsafe blocks in Rust without justification comments, eval() in JavaScript, exec() in Python

Reports findings with severity levels to IPC.

T+200ms: Documentation CLI receives the event. Checks if the changed file contains public API functions without doc comments. Marks any affected documentation artifacts as stale.

T+250ms: Results arrive at Orchestrator. Three CLIs reacted to a single file write. The Orchestrator aggregates: - Validator: pass - Security: one INFO finding (unused import, not blocking) - Documentation: one artifact marked stale (queued for regeneration)

No critical findings. The Coder continues uninterrupted.

If Security had found a CRITICAL finding — say, a hardcoded API key — the flow changes: the Orchestrator sends a high-priority message to the Coder CLI: "CRITICAL: Hardcoded secret detected in rate_limiter.rs:23. Remove before proceeding." The Coder receives this in its next PreSampling injection and addresses it immediately.

Total elapsed: 250ms. Three CLIs reacted without polling, manual triggering, or orchestration messages. This is the reactive forge.

16.5 The Skills System

Skills are packaged instructions that teach a CLI specific workflows. When you type /commit, the CLI loads the commit skill's instructions and follows them. When the CLI detects that you are working in API files, the api-conventions skill auto-loads.

Skill Structure

Each skill lives in its own directory with a SKILL.md file containing YAML frontmatter and markdown instructions:

.claude/skills/
  +-- commit/SKILL.md
  +-- security-audit/SKILL.md
  +-- code-review/SKILL.md
  +-- fix-issue/SKILL.md
  +-- api-conventions/SKILL.md   (auto-activating)

pub struct Skill {
    pub name: String,
    pub description: String,
    pub allowed_tools: Option<Vec<String>>,
    pub model: Option<String>,
    pub effort: Option<String>,
    pub paths: Vec<String>,          // Auto-activate globs
    pub user_invocable: bool,
    pub disable_model_invocation: bool, // Only user can trigger
    pub body: String,                // Markdown instructions
}

pub struct SkillRegistry {
    skills: Vec<Skill>,
}

impl SkillRegistry {
    pub fn load(project_root: &Path) -> Self {
        let mut skills = Vec::new();

        // Load project skills
        let project_dir = project_root.join(".claude/skills");
        if project_dir.exists() {
            for entry in walkdir::WalkDir::new(&project_dir) {
                if let Ok(entry) = entry {
                    if entry.file_name() == "SKILL.md" {
                        if let Ok(skill) = Self::parse_skill(entry.path()) {
                            skills.push(skill);
                        }
                    }
                }
            }
        }

        // Load user-level skills
        if let Some(home) = dirs::home_dir() {
            let user_dir = home.join(".claude/skills");
            if user_dir.exists() {
                for entry in walkdir::WalkDir::new(&user_dir) {
                    if let Ok(entry) = entry {
                        if entry.file_name() == "SKILL.md" {
                            if let Ok(skill) = Self::parse_skill(entry.path()) {
                                skills.push(skill);
                            }
                        }
                    }
                }
            }
        }

        Self { skills }
    }

    pub fn find_by_name(&self, name: &str) -> Option<&Skill> {
        self.skills.iter().find(|s| s.name == name)
    }

    /// Find skills that auto-activate for the given file paths
    pub fn auto_matching(&self, touched_paths: &[&str]) -> Vec<&Skill> {
        self.skills.iter()
            .filter(|s| !s.paths.is_empty() && !s.disable_model_invocation)
            .filter(|s| {
                s.paths.iter().any(|pattern| {
                    touched_paths.iter().any(|path| glob_match(pattern, path))
                })
            })
            .collect()
    }

    pub fn list_user_invocable(&self) -> Vec<&Skill> {
        self.skills.iter().filter(|s| s.user_invocable).collect()
    }
}

Skills can include !`command` syntax to run shell commands at load time:

---

Multi-CLI Skill Sharing via MCP_SKILLS

Behind the MCP_SKILLS feature flag, skills can be served over MCP — any CLI can invoke any other CLI's skills as remote tools. The Security CLI's security-audit skill becomes a tool that the Orchestrator can call:

{
  "mcpServers": {
    "security-skills": {
      "command": "rcode",
      "args": ["--mcp", "--skills-only", "--agent", "security"],
      "description": "Security CLI skills served as MCP tools"
    }
  }
}

The Orchestrator calls security-skills.security_audit(target: "src/api/") as a tool call. The Security CLI executes its audit skill and returns the result. Skills become first-class distributed capabilities.

This is powerful because it decouples skill execution from skill definition. The Security CLI defines what a security audit is — the OWASP checks, the CWE references, the severity scoring. Any other CLI can invoke it without knowing any of those details. The skill becomes an opaque capability with a typed interface.

Worked Example: The Commit Skill

The most commonly used skill in practice. Here is a complete commit skill for the multi-CLI forge:

---
name: commit
description: Create a git commit with proper attribution. Use when user
  says "commit", "save changes", "check in", or /commit.
allowed-tools: Bash, Read, Grep, Glob
user-invocable: true
---

# Commit Skill

## Steps

1. Run `git status` to see all changes
2. Run `git diff` to review staged and unstaged changes
3. Run `git log --oneline -5` to see recent commit message style
4. Analyze changes and draft a commit message:
   - Summarize the nature (feat, fix, refactor, etc.)
   - Focus on WHY, not WHAT
   - Keep first line under 72 characters
5. Stage relevant files (never stage .env or credentials)
6. Create the commit with attribution trailer
7. Run `git status` to verify success

## Rules
- NEVER use --no-verify or --amend without explicit user request
- NEVER stage files matching: .env, *.pem, credentials.*, token.json
- Commit message ends with: Co-Authored-By: Claude <noreply@anthropic.com>
- If pre-commit hook fails, fix the issue and create a NEW commit

When the user types /commit, the CLI loads these instructions and follows them step by step. The allowed-tools field means Bash, Read, Grep, and Glob execute without permission prompts during the skill — the user already authorized the workflow by invoking the skill.

The skill tracks outcomes via SKILL_IMPROVEMENT. If users consistently edit the commit message after the skill generates it, the improvement pass analyzes the edits and adjusts the prompt: "Users frequently add issue references — include (fixes #N) when a commit clearly addresses a tracked issue."

16.6 SKILL_IMPROVEMENT: Self-Evolving Workflows

The most ambitious feature in the extension stack. Behind feature('SKILL_IMPROVEMENT'), skills track their own effectiveness and evolve their prompts based on outcomes.

The Feedback Loop

Skill executes -> Outcome observed -> Score computed -> Prompt updated
       ^                                                      |
       |                                                      |
       +------ Next invocation uses improved prompt <---------+

The mechanism:

pub struct SkillImprover {
    skill_name: String,
    outcomes: Vec<SkillOutcome>,
    improvement_threshold: usize, // outcomes before improvement pass
}

pub struct SkillOutcome {
    pub skill_name: String,
    pub timestamp: DateTime<Utc>,
    pub success: bool,
    pub user_accepted: bool,     // Did user accept the result?
    pub tokens_used: u64,
    pub duration_ms: u64,
    pub error: Option<String>,
    pub user_feedback: Option<String>, // "no not that", "perfect", etc.
}

impl SkillImprover {
    pub fn record_outcome(&mut self, outcome: SkillOutcome) {
        self.outcomes.push(outcome);

        if self.outcomes.len() >= self.improvement_threshold {
            self.improve();
        }
    }

    fn improve(&mut self) {
        let total = self.outcomes.len();
        let successes = self.outcomes.iter().filter(|o| o.success && o.user_accepted).count();
        let success_rate = successes as f64 / total as f64;

        // Collect failure patterns
        let failures: Vec<&str> = self.outcomes.iter()
            .filter(|o| !o.success || !o.user_accepted)
            .filter_map(|o| o.error.as_deref().or(o.user_feedback.as_deref()))
            .collect();

        if success_rate >= 0.9 {
            // Skill is working well — no changes
            self.outcomes.clear();
            return;
        }

        // Build improvement prompt for the LLM
        let improvement_prompt = format!(
            "This skill has a {:.0}% success rate over {} invocations.\n\
             Common failure patterns:\n{}\n\n\
             Suggest specific changes to the skill's instructions to address these failures.\n\
             Only modify the parts that are causing failures. Keep what works.",
            success_rate * 100.0,
            total,
            failures.join("\n- ")
        );

        // The improvement prompt is sent to an LLM to generate
        // an updated SKILL.md body. The update is written back
        // to the skill file with a changelog entry.
        self.apply_improvement(&improvement_prompt);
        self.outcomes.clear();
    }
}

The key insight: skills improve based on user acceptance, not just execution success. A skill that runs without errors but produces output the user rejects ("no not that") has a low score. A skill that the user consistently accepts without modification has a high score. User behavior is the ultimate ground truth.

Guardrails

Skill improvement has three safety mechanisms:

  1. Minimum sample size. No improvement pass until at least 5 outcomes are recorded. Small samples produce noisy signals.

  2. Diff review. The improved skill is written with a <!-- IMPROVED: reason --> marker. The user can review the diff and revert if the improvement degraded quality.

  3. Rollback. Previous skill versions are preserved in .claude/skills/{name}/history/. If the improved skill performs worse, it can be rolled back to any previous version.

The Improvement Pipeline in Detail

The improvement pipeline operates as a three-stage process. First, the skill collects outcomes — each invocation records whether it succeeded, whether the user accepted the output, how many tokens it consumed, and any user feedback (corrections, rejections, or explicit praise). Second, when the outcome buffer reaches the threshold, a scoring pass computes the success rate and identifies failure patterns. Third, if the success rate falls below 90%, the system generates an improvement prompt and feeds it to the LLM.

The improvement prompt is carefully constructed. It includes the current skill instructions, the specific failure patterns observed, and examples of what the user changed after the skill ran. The LLM produces a revised skill body that addresses the failures while preserving what works. The revised body replaces the original in SKILL.md, and a timestamped copy of the original is saved to the history directory.

This creates a lineage: you can trace every version of a skill back to its original, see what failures prompted each change, and understand why the skill evolved the way it did. For the multi-CLI forge, skill lineage is especially valuable because multiple CLIs may share the same skill (via MCP_SKILLS), and an improvement on one CLI propagates to all consumers.

16.7 The Hook Execution Engine

The hook execution engine is the runtime that manages hook registration, event matching, and handler execution. In the Rust implementation:

pub struct HookEngine {
    hooks: Vec<RegisteredHook>,
}

pub struct RegisteredHook {
    pub event: HookEvent,
    pub handler: HookHandler,
    pub matcher: Option<String>,    // Tool name filter
    pub timeout_ms: u64,
    pub description: String,
}

pub enum HookHandler {
    Command(String),               // Shell script path
    Http { url: String, method: String },
    Inline(Box<dyn Fn(&HookPayload) -> HookResult>),
}

#[derive(Debug, Clone)]
pub enum HookEvent {
    SessionStart,
    SessionEnd,
    PreToolUse,
    PostToolUse,
    PostToolUseFailure,
    PermissionRequest,
    AgentSpawn,
    AgentComplete,
    TeammateIdle,
    TaskCreated,
    FileChanged,
    PreSampling,
    PostSampling,
    CostThreshold,
    ContextWindowWarning,
    // ... remaining events
}

impl HookEngine {
    pub fn fire(&self, event: HookEvent, payload: &HookPayload) -> Vec<HookResult> {
        let matching_hooks: Vec<_> = self.hooks.iter()
            .filter(|h| h.event == event)
            .filter(|h| {
                // Apply tool name matcher if present
                match (&h.matcher, payload.tool_name.as_ref()) {
                    (Some(matcher), Some(tool)) => tool == matcher,
                    (Some(_), None) => false,
                    (None, _) => true,
                }
            })
            .collect();

        let mut results = Vec::new();

        for hook in matching_hooks {
            let result = match &hook.handler {
                HookHandler::Command(cmd) => {
                    execute_command_hook(cmd, payload, hook.timeout_ms)
                }
                HookHandler::Http { url, method } => {
                    execute_http_hook(url, method, payload, hook.timeout_ms)
                }
                HookHandler::Inline(f) => f(payload),
            };
            results.push(result);

            // For blocking hooks (PreToolUse), stop on first Deny
            if event == HookEvent::PreToolUse {
                if let HookResult::Deny(_) = &results.last().unwrap() {
                    break;
                }
            }
        }

        results
    }

    pub fn register(&mut self, hook: RegisteredHook) {
        self.hooks.push(hook);
    }
}

The engine short-circuits on the first Deny result for PreToolUse hooks. If the destructive command blocker denies a rm -rf, the secret scanner does not need to run — the operation is already blocked. For non-blocking hooks (PostToolUse, FileChanged), all matching hooks execute regardless of individual results.

16.8 Hook Configuration for the Multi-CLI Forge

16.8 Non-Engineering Domains

Manufacturing Quality Control. Each station in a production line is a specialist CLI. The assembly station's PostToolUse hook triggers the inspection station's quality check. The inspection station's PreToolUse hook enforces tolerances — blocking assembly from proceeding on a part that failed QA. The SKILL_IMPROVEMENT pattern maps to continuous process improvement: skills encoding assembly procedures evolve based on defect rates.

Content Publishing. A publishing pipeline has writers, editors, fact-checkers, and layout designers. The writer's FileChanged hook triggers the editor's review. The editor's approval (PermissionGranted) triggers the fact-checker. The CostThreshold hook prevents runaway AI costs during automated content generation. Skills encode house style guides that evolve as editorial feedback accumulates.

Financial Trading. Pre-trade compliance checks map to PreToolUse hooks — blocking trades that violate position limits, regulatory constraints, or risk thresholds. Post-trade hooks notify settlement systems. The orchestration trinity (TeammateIdle, TaskCreated, TaskCompleted) maps to order routing: when an execution agent completes a fill, the portfolio rebalancer checks for the next required trade.
