Chapter 12: The QA CLI — The Adversary

Edge Case Generation, Property-Based Testing, Mutation Analysis, and the Competing Hypothesis Pattern

The Validator CLI (Chapter 11) answers the question: Does this code pass the existing checks? The Validator runs the linter, the type checker, the test suite, and the performance benchmarks — all of which existed before the code was written. If the tests pass, the Validator says green. If they fail, the Validator says red. But there is a problem the Validator cannot solve: what if the tests themselves are incomplete?

Consider a developer who writes a function to validate email addresses. They write four tests: valid email passes, missing @ fails, empty string fails, missing domain fails. All four pass. The Validator runs these tests and reports green. The code ships. Three weeks later, a user enters user@@example.com — two @ signs — and the function crashes with a ValueError because email.split("@") returns three parts but the code unpacks only two. The bug was not in the code the tests covered. The bug was in the code the tests did not cover — the space between what the developer imagined and what users actually do.

This is the QA CLI's domain. Where the Validator runs existing checks, the QA CLI generates new checks — adversarial tests that target the boundaries, edge cases, race conditions, and state transitions that the developer never considered. Where the Validator is a judge evaluating evidence, the QA CLI is a prosecutor building the case. Where the Validator is convergent (applying known rules to reduce uncertainty), the QA CLI is divergent (expanding the test surface to discover unknown failure modes).

The distinction between Validator and QA is architectural, not semantic. The Validator is deterministic — the same code produces the same verdict every time. The QA CLI is creative — it generates different tests based on the code's structure, the change's context, and the historical pattern of defects in the module. The Validator ensures consistency. The QA CLI ensures coverage.

This chapter builds the adversary from four components: the diff-aware targeting system that focuses QA effort on the highest-risk code paths, the edge case generation engine that systematically produces boundary-testing inputs, the property-based and mutation testing infrastructure that catches bugs no finite test list could find, and the competing hypothesis pattern that uses multiple QA agents to argue about correctness.

12.1 The Adversarial Mindset — Encoded in Architecture

The QA CLI's system prompt encodes a cognitive frame that is fundamentally different from every other CLI in the fleet. The Coder is told to build. The Validator is told to verify. The Security CLI is told to audit. The QA CLI is told to break.

This is not anthropomorphism. It is prompt engineering. When you tell a language model "verify this code works," the model's attention gravitates toward evidence of correctness — it will find paths that work and report them. When you tell a model "find a way to make this code fail," the model's attention gravitates toward edge cases, boundary conditions, and unusual inputs — it will find paths that break and report them. The framing determines the output.

The adversarial frame manifests in three operational principles:

Author's blindness. The developer who wrote the code holds a mental model of how the code should behave. Their tests validate that mental model, not the actual behavior. The QA CLI has never seen the developer's mental model. It reads the code cold, with no context about intent, and asks: What inputs could make this produce unexpected results?

Boundary priority. Most bugs live at boundaries — the transitions between valid and invalid input, the edges of integer ranges, the exact moment a token expires, the race between two concurrent requests. The QA CLI systematically targets these boundaries because they yield the highest defect density per test.

Contract testing over implementation testing. The QA CLI tests public interfaces, not internal methods. A test that breaks when the developer refactors internal code without changing behavior is a bad test — it creates friction without catching bugs. A test that breaks when the public behavior changes is a good test — it catches the changes that matter to users.

12.2 Diff-Aware Targeting — Testing What Changed

The QA CLI does not test the entire codebase on every change. That would be wasteful and would dilute its adversarial focus. Instead, it uses diff-aware targeting to concentrate effort on the code paths most likely to contain new defects.

pub struct DiffAwareTargeter {
    repo_root: PathBuf,
    bug_history: HashMap<String, usize>,  // function_name → historical bug count
}

impl DiffAwareTargeter {
    /// Stage 1: Extract changed functions from git diff
    pub fn changed_functions(&self, base_ref: &str) -> Vec<TargetFunction> {
        let diff = git_diff(&self.repo_root, base_ref);
        let mut targets = Vec::new();

        for file_change in &diff.files {
            if !file_change.is_source_code() {
                continue;
            }
            for hunk in &file_change.hunks {
                // Use tree-sitter to find which functions contain changed lines
                let functions = resolve_enclosing_functions(
                    &file_change.path,
                    hunk.new_start,
                    hunk.new_end,
                );
                targets.extend(functions);
            }
        }

        targets.dedup_by(|a, b| a.qualified_name == b.qualified_name);
        targets
    }

    /// Stage 2: Expand to callers and callees (blast radius)
    pub fn blast_radius(&self, targets: &[TargetFunction]) -> Vec<TargetFunction> {
        let mut expanded: HashSet<String> = HashSet::new();
        let mut result = Vec::new();

        for target in targets {
            if expanded.insert(target.qualified_name.clone()) {
                result.push(target.clone());
            }

            // Callers: if hash_password changed, test authenticate() too
            for caller in find_callers(&target.qualified_name) {
                if expanded.insert(caller.qualified_name.clone()) {
                    result.push(caller);
                }
            }

            // Callees: if a dependency changed behavior, test propagation
            for callee in find_callees(&target.qualified_name) {
                if expanded.insert(callee.qualified_name.clone()) {
                    result.push(callee);
                }
            }
        }

        result
    }

    /// Stage 3: Prioritize by risk score
    pub fn prioritize(&self, targets: &mut [TargetFunction]) {
        for target in targets.iter_mut() {
            target.risk_score = self.calculate_risk(target);
        }
        targets.sort_by(|a, b| b.risk_score.partial_cmp(&a.risk_score).unwrap());
    }

    fn calculate_risk(&self, target: &TargetFunction) -> f64 {
        let complexity = (target.cyclomatic_complexity as f64) / 10.0;
        let change_ratio = target.lines_changed as f64 / target.total_lines.max(1) as f64;
        let history = *self.bug_history.get(&target.qualified_name).unwrap_or(&0) as f64 / 5.0;

        // Domain-specific risk multipliers
        let name_lower = target.qualified_name.to_lowercase();
        let auth_factor = if name_lower.contains("auth") || name_lower.contains("login") { 1.5 } else { 1.0 };
        let payment_factor = if name_lower.contains("payment") || name_lower.contains("billing") { 2.0 } else { 1.0 };
        let input_factor = if target.has_external_input { 1.3 } else { 1.0 };

        (0.4 * complexity + 0.3 * change_ratio + 0.3 * history)
            * auth_factor * payment_factor * input_factor
    }
}

The risk calculation is deliberately opinionated. Functions handling authentication get a 1.5x multiplier. Functions handling payments get 2.0x. Functions processing external input get 1.3x. These multipliers reflect the reality that bugs in authentication code cause data breaches, bugs in payment code cause financial losses, and bugs in input-handling code are the most common entry point for attackers. The QA CLI's limited budget should be allocated where defects cause the most damage.

12.3 Edge Case Generation — The Systematic Art of Breaking Things

Edge case generation is the QA CLI's primary weapon. Not random input — that is fuzzing (covered later). Systematic, category-driven generation of inputs that target the boundaries where implementations tend to fail.

The QA CLI maintains an edge case taxonomy — a structured catalogue of boundary conditions organized by data type:

pub struct EdgeCaseTaxonomy {
    pub strings: StringEdgeCases,
    pub numbers: NumberEdgeCases,
    pub collections: CollectionEdgeCases,
    pub temporal: TemporalEdgeCases,
    pub concurrency: ConcurrencyEdgeCases,
}

pub struct StringEdgeCases {
    pub empty: Vec<&'static str>,        // "", " ", "\t", "\n", "\r\n"
    pub boundaries: Vec<String>,          // 1 char, 255, 256, 65535
    pub unicode: Vec<&'static str>,       // "é", "中文", "\u{200B}" (zero-width)
    pub injection: Vec<&'static str>,     // SQL, XSS, command, template
    pub null_bytes: Vec<&'static str>,    // "hello\0world", "\0"
    pub path_traversal: Vec<&'static str>, // "../../../etc/passwd"
}

pub struct NumberEdgeCases {
    pub boundaries: Vec<f64>,     // 0, -1, 1, -0.0, f64::INFINITY, f64::NAN
    pub integer_limits: Vec<i64>, // i32::MAX, i32::MAX + 1, i64::MAX
    pub precision: Vec<f64>,      // 0.1 + 0.2, f64::MIN_POSITIVE, f64::MAX
}

impl EdgeCaseTaxonomy {
    pub fn generate_for_type(&self, type_name: &str) -> Vec<TestInput> {
        match type_name {
            "String" | "str" | "&str" => {
                let mut inputs = Vec::new();
                for s in &self.strings.empty { inputs.push(TestInput::Str(s.to_string())); }
                for s in &self.strings.boundaries { inputs.push(TestInput::Str(s.clone())); }
                for s in &self.strings.unicode { inputs.push(TestInput::Str(s.to_string())); }
                for s in &self.strings.injection { inputs.push(TestInput::Str(s.to_string())); }
                for s in &self.strings.null_bytes { inputs.push(TestInput::Str(s.to_string())); }
                inputs
            }
            "i32" | "i64" | "u32" | "u64" | "f64" => {
                self.numbers.boundaries.iter()
                    .chain(self.numbers.precision.iter())
                    .map(|n| TestInput::Number(*n))
                    .collect()
            }
            _ => vec![],
        }
    }
}

This taxonomy is not static. When a production incident reveals that a service crashed because of a right-to-left Unicode override character (\u{202E}) in a username field, that character gets added to strings.unicode. The taxonomy is the QA CLI's institutional memory of how software fails.

Generating Tests from Taxonomy

For each target function identified by the diff-aware targeter, the QA CLI reads the function's parameter types and generates edge case tests:

pub fn generate_edge_case_tests(
    target: &TargetFunction,
    taxonomy: &EdgeCaseTaxonomy,
) -> Vec<GeneratedTest> {
    let mut tests = Vec::new();

    for param in &target.parameters {
        let edge_inputs = taxonomy.generate_for_type(&param.type_name);

        for input in edge_inputs {
            tests.push(GeneratedTest {
                name: format!(
                    "test_{}_with_{}_{}",
                    target.name, param.name, input.category()
                ),
                target_function: target.qualified_name.clone(),
                input: input.clone(),
                expectation: Expectation::DoesNotCrash,
                // The QA CLI doesn't know the correct output — it only knows
                // that the function should not panic, not corrupt state, and
                // not produce a security vulnerability
                category: TestCategory::EdgeCase,
            });
        }
    }

    tests
}

The expectation is deliberately minimal: DoesNotCrash. The QA CLI does not know what the correct output should be for every edge case input — that requires domain knowledge the QA CLI does not have. What the QA CLI does know is that a function should never panic on unexpected input, never produce undefined behavior, and never corrupt shared state. These are universal properties that apply to every function.

Worked Example: Auth Module Under Adversarial Analysis

The Coder CLI has just implemented a JWT-based authentication module. The developer wrote 12 tests: valid login succeeds, wrong password fails, expired token rejected, and so on. All pass. The Validator says green. But the QA CLI sees this code and generates a different set of tests — tests the developer never imagined:

Input boundary tests: - Empty password: Does the function crash, or return a clean error? - 10MB password: bcrypt truncates at 72 bytes, meaning all passwords longer than 72 characters hash identically. If the function does not reject overly long passwords before hashing, an attacker who knows the first 72 characters of any password can authenticate. - Null bytes in email (admin\x00@evil.com): C library string functions truncate at null bytes, potentially allowing admin to authenticate as a valid user while the display shows a different email. - Unicode normalization bypass: The character ñ can be represented as U+00F1 (single codepoint) or U+006E U+0303 (combining sequence). If the registration endpoint normalizes one way and the login endpoint normalizes differently, an attacker can create a lookalike account.

Token edge cases: - Token verified at the exact second of expiry: Is this valid or expired? The specification must decide, but most implementations do not handle the boundary explicitly. - JWT algorithm confusion: Setting the alg header to "none" produces an unsigned token. If the verification function accepts unsigned tokens, any user can forge any identity. - Refresh token used after password change: Most implementations forget to invalidate refresh tokens when the user changes their password, leaving a window where stolen refresh tokens remain valid.

Concurrency tests: - 50 simultaneous logins for the same user: Does each get a unique session token? Does the rate limiter count correctly under contention? - Rate limiter race condition: If the rate limit is 5 attempts, but 10 requests arrive simultaneously before the counter increments, how many get through? - Two threads refreshing the same token simultaneously: One should succeed and one should get a clear error, not a reused token.

None of these tests duplicate the developer's original 12. Every one targets a specific failure mode that the developer's mental model did not cover. This is what adversarial testing produces at scale.

12.4 Property-Based Testing and Mutation Analysis

Edge case taxonomies are powerful but ultimately finite. The QA CLI augments them with two techniques that generate unbounded test inputs: property-based testing and mutation testing.

Property-Based Testing

Property-based testing inverts the traditional test structure. Instead of specifying inputs and expected outputs, you specify properties — invariants that must hold for all valid inputs — and let the testing framework generate random inputs.

The QA CLI generates property-based tests for common property patterns:

pub enum PropertyPattern {
    /// f(x) produces valid output for all valid x
    InputValidation { function: String, input_strategy: String },

    /// f(f_inverse(x)) == x (roundtrip)
    Roundtrip { encode: String, decode: String },

    /// f(x) is idempotent: f(f(x)) == f(x)
    Idempotent { function: String },

    /// Output ordering matches input ordering
    OrderPreserving { function: String },

    /// Output size is bounded relative to input size
    SizeBounded { function: String, max_ratio: f64 },
}

pub fn generate_property_tests(
    target: &TargetFunction,
    patterns: &[PropertyPattern],
) -> Vec<GeneratedTest> {
    let mut tests = Vec::new();

    // Roundtrip property: encode then decode returns original
    if target.name.starts_with("encode") || target.name.starts_with("serialize") {
        let decode_name = target.name
            .replace("encode", "decode")
            .replace("serialize", "deserialize");

        tests.push(GeneratedTest {
            name: format!("prop_roundtrip_{}", target.name),
            category: TestCategory::Property,
            property: format!(
                "For all valid inputs x: decode(encode(x)) == x. \
                 Test with 500 random inputs using the property-based framework."
            ),
            ..Default::default()
        });
    }

    // Idempotency: normalizing twice produces the same result
    if target.name.contains("normalize") || target.name.contains("sanitize") {
        tests.push(GeneratedTest {
            name: format!("prop_idempotent_{}", target.name),
            category: TestCategory::Property,
            property: format!(
                "For all inputs x: {}({}(x)) == {}(x). \
                 Normalization/sanitization applied twice should equal applied once.",
                target.name, target.name, target.name
            ),
            ..Default::default()
        });
    }

    tests
}

Mutation Testing — Testing the Tests

Mutation testing addresses a meta-problem: how do you know your tests actually catch bugs? A test suite can achieve 100% code coverage and still miss critical defects — because coverage measures which lines executed, not which behaviors were verified.

The QA CLI introduces small bugs (mutations) into the code and checks whether the test suite catches them:

pub struct MutationOperator {
    pub original: &'static str,
    pub mutant: &'static str,
    pub category: &'static str,
}

pub const MUTATION_OPERATORS: &[MutationOperator] = &[
    MutationOperator { original: "==", mutant: "!=", category: "equality_flip" },
    MutationOperator { original: ">=", mutant: ">",  category: "boundary_shift" },
    MutationOperator { original: "<=", mutant: "<",  category: "boundary_shift" },
    MutationOperator { original: "true", mutant: "false", category: "boolean_flip" },
    MutationOperator { original: "&&", mutant: "||", category: "logic_gate_swap" },
    MutationOperator { original: "+ 1", mutant: "",   category: "increment_removal" },
    MutationOperator { original: ".push(", mutant: ".remove(", category: "collection_swap" },
];

pub struct MutationReport {
    pub total_mutations: usize,
    pub killed: usize,         // Test suite caught the mutation
    pub survived: usize,       // Test suite missed the mutation
    pub mutation_score: f64,   // killed / total
    pub survivors: Vec<SurvivedMutation>,
}

pub struct SurvivedMutation {
    pub file: String,
    pub line: usize,
    pub original: String,
    pub mutant: String,
    pub explanation: String,   // "Tests don't verify the return value at line 47"
}

The QA CLI's quality gate for its own tests: a minimum mutation score of 0.85. If more than 15% of mutations survive, the QA CLI generates additional tests to cover the gaps. The adversary tests itself.

Each survived mutation is a specific test gap that the QA CLI must address. Consider a mutation that changes >= to > on the rate limiter threshold check. If the test suite still passes, it means no test verifies the exact boundary — the test checks that 6 attempts are rate-limited, but never checks that exactly 5 are allowed. The survived mutation pinpoints the gap with line-level precision, and the QA CLI generates a targeted test to kill it:

// Survived mutation: line 83, rate_limiter.rs
// Original: if attempts >= MAX_ATTEMPTS
// Mutant:   if attempts > MAX_ATTEMPTS
// Gap: No test verifies the exact boundary (5 attempts allowed, 6th blocked)

// Generated kill test:
#[test]
fn test_rate_limit_exact_boundary() {
    let limiter = RateLimiter::new(max_attempts: 5);

    // Exactly at limit: should still be allowed
    for _ in 0..5 {
        assert!(limiter.check("user").is_ok());
    }

    // One past limit: should be blocked
    assert!(limiter.check("user").is_err());
}

This feedback loop — generate tests, mutate code, identify survivors, generate targeted kill tests — is the QA CLI's mechanism for continuous self-improvement. Each round of mutation testing produces a more robust test suite.

12.5 Competing Hypothesis Testing — When QA Agents Argue

Claude Code's agent-teams pattern enables a QA technique borrowed from scientific methodology: competing hypotheses. Instead of a single QA agent forming a single opinion, three QA sub-agents with different roles analyze the same code independently and then argue about the results.

QA CLI receives code change
  → Spawns 3 sub-agents concurrently:

  Agent 1 ("Optimist"):
    Hypothesis: "This code is correct"
    Task: Construct positive test cases, verify contracts hold

  Agent 2 ("Pessimist"):
    Hypothesis: "This code has a critical bug"
    Task: Adversarial testing, injection, fuzzing, boundary attacks

  Agent 3 ("Skeptic"):
    Hypothesis: "This code works but has subtle issues"
    Task: Race conditions, performance regressions, silent data corruption

The three agents run concurrently (Claude Code supports up to 10 parallel sub-agents). Each produces a report. The QA CLI synthesizes the reports with a resolution algorithm:

pub fn resolve_competing_hypotheses(
    optimist: &QAReport,
    pessimist: &QAReport,
    skeptic: &QAReport,
) -> QAVerdict {
    // Pessimist findings override everything
    if pessimist.has_critical_finding() {
        return QAVerdict {
            result: VerdictResult::Fail,
            confidence: Confidence::High,
            reason: pessimist.critical_findings[0].description.clone(),
            action: Action::BlockAndFix,
        };
    }

    // Optimist couldn't prove correctness — investigate
    if !optimist.proof_complete {
        return QAVerdict {
            result: VerdictResult::Review,
            confidence: Confidence::Medium,
            reason: "Correctness proof incomplete — gaps in test coverage".into(),
            action: Action::HumanReview,
        };
    }

    // Skeptic found subtle issues — proceed with tracking
    if !skeptic.subtle_findings.is_empty() {
        return QAVerdict {
            result: VerdictResult::Warn,
            confidence: Confidence::High,
            reason: format!("{} non-critical issues found", skeptic.subtle_findings.len()),
            action: Action::ProceedWithTracking,
        };
    }

    // All three agree: code is solid
    QAVerdict {
        result: VerdictResult::Pass,
        confidence: Confidence::High,
        reason: "Adversarial analysis complete — no issues found".into(),
        action: Action::Proceed,
    }
}

The power of this pattern is in the disagreement. If the optimist says "correct" but the pessimist says "crashes on null input," the crash wins. The competing hypotheses ensure no single perspective dominates — which is precisely the point of separating QA from development.

Why Three Agents, Not One

A single QA agent with a long system prompt ("be thorough, check edge cases, also look for race conditions, also check performance") produces mediocre results in all categories. The attention diffusion problem — the same phenomenon that makes the Orchestrator a poor substitute for specialized CLIs (Chapter 6) — applies within the QA domain as well.

Three focused agents produce better results because each has a narrow mandate and a deep context. The Pessimist's entire context window is dedicated to finding crashes and security vulnerabilities. It does not waste tokens thinking about performance. The Skeptic's entire context window is dedicated to subtle correctness issues. It does not waste tokens on obvious crashes.

The synthesis step is where the QA CLI adds value beyond what any individual agent could provide. It resolves contradictions ("the Optimist says the rate limiter works; the Pessimist found a race condition in the rate limiter"), aggregates findings into a single structured report, and ensures that the Orchestrator receives one coherent verdict rather than three competing opinions.

The cost of three sub-agents is approximately 3x the API cost of a single QA pass. This is justified when the code under test handles authentication, payments, or user data — domains where a single missed defect can cost orders of magnitude more than the QA budget. For low-risk utility functions, the QA CLI can skip the competing hypothesis pattern and run a single-agent edge case pass. The risk score from the diff-aware targeter (Section 12.2) drives this decision automatically.

12.6 The QA-to-Coder Feedback Loop

The QA CLI does not just find bugs. It produces structured reports that the Orchestrator routes directly to the Coder CLI for automated fixes. The feedback loop is the mechanism that turns QA findings into code improvements without human intervention.

Every QA finding produces a structured report with everything the Coder needs to fix the issue:

pub struct QAFailureReport {
    pub report_id: String,           // "qa-2026-04-02-auth-001"
    pub severity: Severity,          // Critical / High / Medium / Low
    pub category: String,            // "security", "correctness", "performance"
    pub file: String,                // "src/auth/tokens.rs"
    pub function: String,            // "verify_jwt"
    pub line_range: (usize, usize),  // (42, 58)
    pub description: String,         // What the bug is
    pub reproduction: Reproduction,  // How to trigger it
    pub evidence: Evidence,          // Input, expected, actual
    pub suggested_fix: String,       // One-line fix suggestion
    pub cwe_reference: Option<String>, // CWE-327 if applicable
}

pub struct Reproduction {
    pub test_file: String,
    pub test_name: String,
    pub command: String,  // "cargo test test_algorithm_confusion -- --nocapture"
}

pub struct Evidence {
    pub input: String,     // The input that triggered the bug
    pub expected: String,  // What should have happened
    pub actual: String,    // What actually happened
}

The report is machine-readable. The Coder CLI does not need to interpret prose — it receives a structured document with the exact file, function, line range, reproduction command, and suggested fix. This is the difference between "there might be a JWT issue" and "line 47 of tokens.rs is missing an algorithm whitelist, here is the test that proves it, here is the one-line fix."

The full fix-verify cycle:

  1. QA CLI generates adversarial tests and runs them
  2. Failures produce structured QAFailureReports
  3. Reports sent to Orchestrator via shared task list
  4. Orchestrator routes each report to Coder CLI
  5. Coder CLI reads report, applies fix
  6. Orchestrator triggers QA CLI re-verification
  7. QA CLI runs the same adversarial tests against fixed code
  8. If pass: finding marked resolved, proceed to Validator
  9. If fail: back to step 4 with updated context (max 3 retries)
  10. After 3 failed retries: escalate to human review with full history

The retry limit prevents infinite fix-break-fix cycles. The full history — original report, each fix attempt, and why each attempt failed — accompanies the escalation so the human reviewer has complete context.

12.8 Adversarial Memory — The QA CLI That Learns

Unlike the Validator (which is stateless — the same code always produces the same verdict), the QA CLI maintains memory across sessions. This memory serves two purposes: avoiding duplicate work and concentrating effort on historically buggy modules.

pub struct QAMemory {
    /// Historical findings: module → list of past bugs
    pub finding_history: HashMap<String, Vec<HistoricalFinding>>,

    /// Module risk scores: updated after each QA pass
    pub risk_scores: HashMap<String, f64>,

    /// Edge case patterns that found real bugs (promoted to "always test")
    pub proven_patterns: Vec<EdgeCasePattern>,
}

pub struct HistoricalFinding {
    pub module: String,
    pub finding_type: String,   // "boundary", "concurrency", "injection"
    pub date: String,
    pub was_real_bug: bool,     // False positive? Or real bug that was fixed?
    pub fix_attempts: usize,    // How many Coder retries before fix
}

When the QA CLI discovers that authentication modules consistently have JWT algorithm confusion vulnerabilities, it adds "JWT algorithm whitelist check" to the proven_patterns list. On subsequent auth module changes, that pattern is tested first — before the generic edge case taxonomy runs. The proven patterns list is the QA CLI's equivalent of a security researcher's personal knowledge base: patterns that have found real bugs get elevated priority.

The was_real_bug flag is critical for calibrating the QA CLI's aggressiveness. If 90% of the QA CLI's findings are real bugs that the Coder fixes, the QA CLI's adversarial parameters stay unchanged. If only 30% are real bugs and 70% are false positives, the QA CLI tightens its confidence thresholds and reduces the weight of speculative patterns. This self-calibration prevents the "boy who cried wolf" failure mode where the team starts ignoring QA findings because most are noise.

12.9 Non-Engineering Domains

The adversarial QA pattern applies wherever produced artifacts need stress-testing beyond the author's assumptions.

Legal Contract QA: The "Pessimist" reads a contract and asks: What loopholes does this create? What if the counterparty acts in bad faith? What if market conditions change the meaning of "reasonable"? Edge cases include force majeure scenarios, jurisdiction conflicts, and ambiguous defined terms. The "Skeptic" checks for internal contradictions between clauses. The QA finding "Section 7.3 guarantees 99.9% uptime but Section 12.1 limits liability to fees paid — during an extended outage, the client has no meaningful remedy" is structurally identical to a software QA finding that identifies a gap between the specification and the implementation.

Medical Protocol QA: Treatment protocols are tested against edge cases: drug interactions, patient allergies not in the standard allergy database, off-label dosages, concurrent conditions that change treatment priorities. The "Thunderherd" heuristic translates to: what if 50 patients present with the same rare condition simultaneously — does the protocol handle resource contention?

Financial Model QA: Models are tested against boundary conditions: zero interest rates, negative yields, currency hyperinflation, market holidays that shift settlement dates, and the classic "what happens on February 29" in a non-leap year. Mutation testing replaces >= with > in threshold comparisons to verify that boundary handling is correct. The competing hypothesis pattern is particularly valuable here — an optimist who says "the model produces correct valuations" and a pessimist who says "the model fails under extreme market conditions" will together produce a more comprehensive validation than either alone.

In all three domains, the QA CLI's core architecture transfers directly: diff-aware targeting identifies what changed, edge case generation tests the boundaries of the change, property-based testing verifies invariants, mutation testing validates test effectiveness, and competing hypotheses prevent single-perspective blindness. The tools change (legal analysis tools instead of linters, clinical databases instead of test suites), but the adversarial methodology is domain-agnostic.

12.8 Testing the QA CLI

Test 1: Diff-Aware Targeting

Input: Git diff changes 3 functions in auth/login.rs.
Expected: QA CLI targets those 3 functions + their callers.
Functions in unrelated modules are NOT tested.

Test 2: Edge Case Generation

Input: Function accepting String parameter.
Expected: Generated tests include empty string, null bytes,
SQL injection payload, Unicode edge cases, path traversal.
All generated tests compile and run (no syntax errors in generated code).

Test 3: Property-Based Roundtrip

Input: encode/decode function pair.
Expected: Property test verifies decode(encode(x)) == x for 500 random inputs.

Test 4: Mutation Score

Input: QA CLI generates tests for a function. Mutation testing runs.
Expected: Mutation score >= 0.85. If below, additional tests are generated.

Test 5: Competing Hypothesis Verdict

Input: Pessimist finds a crash. Optimist says code is correct.
Expected: Verdict = FAIL (pessimist overrides optimist).
Structured report includes crash reproduction steps.

Test 6: Feedback Loop

Input: QA finding sent to Coder, Coder applies fix.
Expected: QA re-runs the same adversarial test. If fix works, finding
marked as resolved. If fix fails, re-routed with updated context.
Max 3 retry cycles before escalation.
