27.2 The Autonomy Spectrum

Intelligence is not binary. It is a spectrum, and the software engineering domain gives us unusually precise markers along it.

Level 0: Tool

A compiler is a tool. You give it input, it gives you output. No memory, no adaptation, no initiative. rustc main.rs. It does exactly what you tell it, exactly the same way, every time.

GitHub Copilot's inline completions in 2023 were tools. You typed a function signature. It predicted the body. Next keystroke, the slate was blank. Useful — like a fast intern who reads one line and guesses the next. But fundamentally reactive and stateless.

Level 1: Assistant

An assistant maintains conversation context. ChatGPT in 2023-2024 was an assistant: multi-turn conversations, referencing earlier messages, building on previous answers. But it could not act. It could suggest code; it could not run code. It could describe a fix; it could not apply a fix. The gap between suggestion and execution was bridged entirely by the human.

Level 2: Agent

An agent has tools and uses them autonomously. Claude Code in 2024-2025 was an agent. The QueryEngine — that while-true loop at the heart of the system — alternated between LLM inference and tool execution. Read a file. Run a command. Edit code. Run tests. Iterate until done.

The jump from assistant to agent is the most consequential leap in the spectrum. An assistant tells you what to do. An agent does it. The difference is not incremental — it is categorical. Once the model can execute bash commands, read file output, and adjust its next action based on observation, it is operating in a closed feedback loop. That is agency.

Level 3: Multi-Agent System

A multi-agent system is what you have built. Multiple agents, each specialized, communicating through structured protocols, coordinated by an orchestrator, operating in parallel on shared tasks. The forge is not one intelligence doing everything — it is a team of narrow intelligences whose combined capability exceeds what any individual agent could achieve.

The emergent behavior matters. When the Security CLI finds an issue and routes it to the Coder CLI through the shared memory bus, that is not something any individual agent was programmed to do end-to-end. It is coordination that emerges from the system's architecture. No single component understands the full workflow. The system does.

Level 4: Artificial General Intelligence

AGI in the software engineering domain would be a system that can do everything a senior engineering team can do, across any project, without human intervention. Not just executing predefined workflows. Handling novel situations. Making judgment calls under ambiguity. Learning from mistakes and never repeating them. Knowing when to ask for help and when to proceed.

We are not there. The forge is Level 3 with glimpses of Level 4.

27.3 What "AGI for Software Engineering" Means

The term AGI is overloaded to the point of uselessness in popular discourse. We need a concrete, falsifiable definition scoped to our domain.

AGI for software engineering is a system that can, given a natural-language description of a software project, independently produce a production-quality implementation — including architecture, code, tests, security review, documentation, and deployment configuration — across any technology stack, for any domain, at the quality level of a senior engineering team, while correctly handling ambiguous requirements, novel challenges, and changing constraints.

27.4 Honest About the Gaps

If this book is going to be worth reading in two years, it needs to be honest about what does not work today. Hype ages badly. Engineering honesty ages well.

Gap 1: The Context Window Is Still a Cage

We solved the single-agent context problem by distributing work across multiple CLIs. Each agent gets its own 200,000-token window. But the problem is displaced, not eliminated.

A single CLI still runs out of context on sufficiently complex tasks. A Security CLI analyzing a 200-file microservice will consume its entire window in 20-25 turns. When compaction triggers, the same information loss occurs that motivated the multi-CLI architecture in the first place.

The Orchestrator faces a related bottleneck. It receives outputs from all eight CLIs. For a complex project, that is thousands of lines of findings, test results, and documentation that all need synthesis. The Orchestrator's context window becomes the new chokepoint.

The real solution is not bigger windows (though 1M-token context is already in Claude Code's beta headers as context-1m-2025-08-07). The real solution is hierarchical summarization — the same technique that allows human organizations to scale. A VP of Engineering does not read every line of code. They read summaries from team leads, who read summaries from individual contributors. The forge needs the same hierarchical compression.

Gap 2: Hallucination Has Not Gone Away

Models hallucinate — generating plausible-sounding content that is factually wrong. In software engineering, this produces bugs that pass superficial review because the code looks correct.

The forge mitigates hallucination better than a single agent. The Validator CLI catches hallucinated APIs by running the code — if the model invents a function that does not exist, the compiler complains. The Security CLI cross-references findings against CVE databases. The QA CLI stress-tests edge cases that hallucinated implementations tend to get wrong.

But mitigation is not elimination. The model can hallucinate at the architectural level — proposing a design pattern that seems reasonable but is subtly wrong. The Planner might suggest event-driven architecture for a system that needs strong consistency. The Validator can only test what it thinks to test — if the hallucinated architecture creates a category of bug the test suite does not exercise, it ships.

Every production deployment of the forge requires human review of architectural decisions, security findings, and deployment configurations. That requirement is not going away soon.

Gap 3: Reasoning Depth Has Hard Limits

Current models reason well across 3-5 logical steps. Production software often requires 10-20 step reasoning chains. A request through API gateway → authentication → rate limiting → load balancer → service → cache → database → webhook → another service → message queue → worker → different database → CDC → search index. Fifteen steps. Models lose track around step 8.

Extended thinking helps. ULTRATHINK pushes reasoning depth further. But even with 100,000 tokens of thinking budget, the fundamental architecture of transformer attention limits reliable step-chaining. Multi-agent systems help because each agent only reasons about its own domain. But cross-agent reasoning — where a finding in one domain cascades through three others — still requires human-level integration.

Gap 4: Cost Scales Faster Than Value

The forge consumes tokens across eight parallel CLIs. A complex project might use 800,000-1,200,000 tokens. At current Sonnet pricing, that is $5-15 per task. For a 20-task project: $100-300.

That is cheaper than a human team. But the cost curve is sublinear in value: the first $5 gets 80% of the value (initial implementation). The next $10 gets the remaining 20% (edge cases, polish, hardening). The forge is a power tool, not a daily driver. You reach for it when the task justifies the cost. Chapter 22's model mixing strategy (Opus for planning, Sonnet for implementation, Haiku for docs) mitigates this — 48% savings compared to all-Opus — but the fundamental relationship holds.

Gap 5: The Integration Tax

Eight CLIs. Typed IPC. Shared memory bus. DAG scheduler. Git worktrees. Permission hierarchies. CI/CD integration. Monitoring. That is a lot of infrastructure.

When a new Claude Code version ships with breaking changes, every CLI needs updating. When model behavior shifts, system prompts need adjustment. When a new tool is added, someone must decide which CLIs get access. The integration tax is the operational cost of a distributed system. Microservices taught this lesson a decade ago: decomposition buys scalability and isolation but costs operational complexity. The forge is subject to the same trade-off.

We built the forge's IPC layer in Chapter 15 using JSON-over-stdio and a shared memory bus. UDS_INBOX is Claude Code's native approach to the same problem — Unix Domain Socket inter-process messaging. Socket name: claude-swarm-${process.pid}. Peer-to-peer communication without a central broker.

Unix Domain Sockets have near-zero latency (no network stack), kernel-level delivery guarantees, and native process authentication. When this ships, it replaces our custom IPC with a built-in, battle-tested communication channel. The forge's typed message contracts remain valuable — they define what gets sent — but UDS_INBOX replaces how it gets sent.

For the forge, direct peer messaging means the Security CLI can send an urgent finding directly to the Coder CLI without routing through the Orchestrator. The QA CLI can request a specific test fixture from the Coder CLI in real-time. The /peers command lists all active CLIs in the swarm, enabling dynamic discovery rather than hardcoded routing tables.

Feature gate: SKILL_IMPROVEMENT.

SKILL_IMPROVEMENT auto-tunes skill prompts based on outcomes. The agent observes which approaches succeed and which fail, then adjusts its procedures accordingly. This is the feature that edges closest to recursive self-improvement.

A CLI that can modify its own skills based on past performance is not just executing instructions — it is learning. Combine SKILL_IMPROVEMENT with KAIROS_DREAM (autonomous memory consolidation during idle hours) and you have an agent that works, reflects on its work, adjusts its approach, and works again. That is a tight feedback loop approaching genuine adaptation.

The forge already has infrastructure for this: the learning system from Chapter 19 captures cross-session knowledge. What is missing is automatic application of that knowledge to improve CLI behavior. SKILL_IMPROVEMENT provides the mechanism. Consider: the Security CLI runs 200 audits. It notices that 73% of its IDOR findings come from checking authorization middleware first, while only 12% come from endpoint-level checks. SKILL_IMPROVEMENT adjusts its audit procedure to prioritize middleware review. The next 200 audits are 40% faster with no drop in finding quality. That is not human programming. That is learned behavior.

27.6 The Medium-Term Horizon: 2026-2028

Beyond what is visible in the source code, these trends will shape multi-CLI systems over the next two to three years:

Self-Modifying Forges

The forge currently improves through human-curated system prompt updates. When the Security CLI misses a vulnerability class, a human edits its CLAUDE.md to add the pattern. When the Coder CLI produces suboptimal code in a specific framework, a human adjusts the coding rules.

The next step is self-modification: CLIs that update their own system prompts based on performance data. The Security CLI that adds a detection rule after a QA CLI adversarial test exposes a gap it missed. The Coder CLI that adjusts its code patterns after the Validator CLI reports a recurring test failure.

Self-modification requires guardrails — the same kill switch architecture from Chapter 20. A CLI that can modify its own instructions can also degrade them. The SIEGE PROTOCOL from Chapter 23 applies: every self-modification must be reviewed by an independent agent (the Security CLI reviews Coder CLI self-modifications, and vice versa).

Emergent Coordination Protocols

Today, CLI-to-CLI communication follows protocols you designed: the Security CLI sends findings in a specific JSON schema, the Orchestrator routes them using rules you wrote, the Coder CLI consumes them in a format you specified. The coordination is prescribed.

In the medium term, agents develop coordination protocols emergently. Agent teams in Claude Code's source already support direct messaging between teammates through UDS_INBOX, not just through the coordinator. When agents can message each other directly AND learn from outcomes AND persist that learning across sessions, they develop shorthand.

The Security CLI learns that sending compact, pre-prioritized findings directly to the Coder CLI (bypassing the Orchestrator for routine OWASP Top 10 issues) gets faster resolution. The Coder CLI learns that copying the Validator CLI on implementation messages reduces rework by 35% because the Validator starts preparing test scaffolding before the code lands. The Documentation CLI learns to subscribe to Security CLI findings because vulnerability remediations generate the most documentation churn.

This is not anthropomorphizing. It is reinforcement learning applied to communication protocols. Agents that communicate effectively (measured by task completion speed and quality) reinforce those communication patterns. Over hundreds of sessions, the team develops its own internal routing optimizations — a protocol layer optimized for the specific kinds of tasks it handles.

Self-Modifying CLI Configurations

The deepest medium-term evolution is CLIs that modify their own configurations — not just their skills or their memories, but their system prompts, their tool lists, and their permission boundaries.

Consider: the forge runs a task. The Orchestrator notices that the Security CLI consistently takes 40% longer than expected because it runs redundant checks — SAST scanning for injection vulnerabilities that the QA CLI already covers with its adversarial testing suite. The Orchestrator modifies the Security CLI's configuration to skip redundant SAST rules when QA is active, monitors the effect over the next ten tasks, and either keeps the change or reverts it.

Or: the Coder CLI encounters a new framework it has never worked with — say, Leptos for Rust web applications. Instead of struggling with generic approaches, it spawns a temporary Research CLI, ingests the framework's documentation, extracts key patterns, and adds them as new skills to its own configuration. The next time it encounters Leptos, the knowledge is there.

This approaches recursive self-improvement — a system that improves its ability to improve itself. Safety constraints must be immutable: enforced at a level the agent cannot modify, verified by a separate system the agent cannot influence. The permission hierarchy we built does this. The Security CLI's tool restrictions are defined in the Orchestrator's configuration, not the Security CLI's own memory. The agent cannot unlock its own cage.

Model Specialization

Current multi-CLI systems use general-purpose models (Opus, Sonnet, Haiku) with specialization achieved through system prompts. Future systems will use models fine-tuned for specific roles:

  • A code-generation model that has seen 10x more code than general training data, optimized for implementation speed
  • A security-analysis model trained on vulnerability databases, CVE reports, and exploitation techniques
  • An architecture model trained on system design documents, RFC archives, and post-mortem analyses

Role-specific fine-tuning combined with multi-CLI architecture would produce a system where each agent is not just context-specialized but model-specialized — a fundamentally different capability level.

Persistent Agent State

Current CLIs are ephemeral. They spawn, execute, and terminate. Their only persistence is through the shared memory bus. Future CLIs will maintain persistent state — ongoing understanding of the codebase that accumulates across sessions, like a human developer's mental model of the project.

This is the difference between an agent that reads the codebase on every invocation and one that knows the codebase from continuous exposure. The Chapter 19 learning system provides the foundation. True persistent state requires something closer to a continuously running agent that processes every commit, every PR, every issue, and builds an ever-deepening understanding.

Formal Verification Integration

The forge's current verification relies on tests — empirical evidence that specific inputs produce expected outputs. Formal verification provides mathematical proof that code satisfies its specification for all possible inputs.

Integrating formal verification tools (TLA+, Alloy, Lean, Dafny) with the Validator CLI would allow the forge to produce not just tested code but proven code. For safety-critical domains — medical devices, autonomous vehicles, financial systems — this transforms the forge from "good enough for production" to "certified for deployment."

27.7 The Long-Term Horizon: What Changes Everything

Two capabilities, when they arrive, will fundamentally transform the forge from a Level 3 multi-agent system into something approaching Level 4.

Reliable Long-Range Reasoning

When models can reliably chain 20+ logical steps without losing coherence, the Orchestrator's task decomposition becomes optional rather than necessary. A single agent that can reason through a complete system design — from requirements through architecture through implementation through testing — without losing track of assumptions made in step 3 when executing step 17 would be qualitatively different from what exists today.

This does not eliminate the multi-CLI architecture. Even with perfect reasoning, there are benefits to parallel execution, permission isolation, and context specialization. But it does change the nature of the Orchestrator from "the brain that decomposes because agents cannot handle complexity" to "the coordinator that parallelizes because speed matters."

Genuine Self-Assessment

The hardest gap to close is judgment — the ability to assess one's own confidence accurately. When the Coder CLI generates a function, it cannot currently distinguish between "I am 99% confident this is correct" and "I am 60% confident but the code compiles so it looks fine." Both produce the same output.

When models develop reliable confidence calibration — knowing what they know and what they do not know — the forge's human review requirement shrinks. The system could route low-confidence outputs to human review while deploying high-confidence outputs automatically. This is not full autonomy, but it is informed autonomy — a system that knows when it needs help.

The Three Phases of Software Engineering's Future

Follow the trajectory far enough and the human-AI relationship transforms through three distinct phases:

Phase 1: Human-AI Pair Programming (Now). This is where we are. A human engineer and the forge collaborate. The human provides intent, context, judgment, and review. The forge provides speed, breadth of knowledge, tireless iteration, and 1.6 million tokens of working memory. A single engineer with the forge is equivalent to a 3-5 person team. Measured by judgment, the human remains irreplaceable.

Phase 2: AI-AI Collaboration with Human Oversight (Near Future). The forge's architecture already supports agent-to-agent collaboration. The Security CLI reviews the Coder CLI's output. The QA CLI challenges the Planner CLI's design decisions. These are AI-AI interactions — one agent evaluating another's work. Phase 2 extends this to genuine collaboration: two Coder CLIs working on different parts of the same system, sharing context through the memory bus, reviewing each other's code. A Planner and an Architect debating design trade-offs through structured argumentation. The claudecode_team system we studied — with its 4-phase workflow of plan, spawn, debate, synthesize — proved that multi-agent debate produces higher-quality outputs than any single agent. The human's role shifts from pair partner to team lead: setting objectives, reviewing deliverables at the architectural level, making final calls on ambiguous trade-offs.

Phase 3: Autonomous Engineering (Long-Term). The most speculative phase. An autonomous engineering system receives requirements in natural language — or eventually, in the form of business metrics and objectives — decomposes them into projects, staffs those projects with agent teams, executes development across multiple repositories, runs its own CI/CD, monitors production, and fixes issues without human intervention. Every component exists in early form: requirements decomposition (the Orchestrator), team formation (COORDINATOR_MODE), parallel execution (the DAG scheduler), CI/CD (Chapter 23's DevSecOps pipeline), production monitoring (KAIROS), and self-repair (SKILL_IMPROVEMENT). The barriers are reasoning depth, reliable self-assessment, and the non-technical challenges of governance, accountability, and trust.

27.8 The Responsibility Question

When an autonomous agent system writes code, merges it, deploys it, and that code causes a production outage — who is responsible?

This is not a hypothetical. The forge can write code, create branches, open pull requests, and — if configured with CI/CD integration — trigger deployments. If the Security CLI misses a vulnerability and the Coder CLI ships it, who gets the page at 3 AM?

Accountability Cannot Be Delegated to Machines

An AI agent has no legal standing, no professional license, no reputation to lose, no capacity for genuine understanding of consequences. Responsibility flows to the humans who built, configured, deployed, and supervised the system.

In practice:

  • The engineer who configures the forge is responsible for the permission boundaries, the security policies, and the review gates
  • The team lead who approves the forge's PR is responsible for the quality of that code, just as they would be for a junior engineer's PR
  • The organization that deploys the forge is responsible for having audit trails, rollback procedures, and incident response processes that account for AI-generated code

The forge includes these accountability structures by design. Permission hierarchies prevent privilege escalation. Audit trails log every tool call, every file modification, every agent-to-agent message. CI/CD gates require human approval for production deployments. These are not features added for completeness — they are load-bearing walls in the system's safety architecture.

The Recursive Self-Improvement Boundary

A forge that builds better forges approaches recursive self-improvement. This deserves explicit attention.

The system has mild recursive improvement: the learning system captures what works, and future sessions benefit. But there is a crucial distinction between learning from experience (which humans do) and modifying core architecture (which is the step that recursive self-improvement concerns are about).

The forge can improve its skills, its knowledge, its communication patterns. It cannot — and should not, without explicit human authorization — modify its permission model, its safety constraints, or its accountability structures. The line is: improve performance within fixed safety boundaries. If the boundaries themselves need to change, that is a human decision. Chapter 23's SIEGE PROTOCOL enforces this — security constraints are immutable by design, verified by an independent agent, and audited by a system the constrained agent cannot influence.

27.9 The Human Role: From Builder to Curator

The forge does not eliminate software engineers. It changes what they do.

Before the forge, a senior engineer spent 60% of their time writing code, 20% reviewing code, 10% debugging, and 10% on architecture and planning.

With the forge, the distribution shifts: 5% writing code (only the parts the forge cannot handle), 30% reviewing forge output, 15% curating system prompts and rules, 20% on architecture and product decisions, 20% on novel problem-solving the forge cannot automate, and 10% on running and maintaining the forge infrastructure.

The engineer's role shifts from builder to curator. They do not write the code. They define the constraints, review the output, and make the judgment calls that the forge cannot make. This is a more senior role, not a less valuable one. The forge amplifies what a single engineer can deliver by an order of magnitude. But the engineer still needs to understand the code well enough to review it, the architecture well enough to guide it, and the domain well enough to specify it.

The most important skill for a post-forge engineer is not coding speed. It is specification precision. The 43-word prompt in Chapter 24 produced a complete SaaS application. But the quality of the output was directly proportional to the precision of the input. A vague prompt produces vague architecture. A specific prompt produces specific architecture. The human's value is in knowing what to specify and what to leave to the forge's judgment.

B.2 Agent and Orchestration Flags

These are the most important flags for multi-CLI systems. They control how agents spawn, communicate, verify each other's work, and learn from outcomes. Without these flags, Claude Code is a single-agent system. With them, it becomes the infrastructure for the forge.

The COORDINATOR_MODE flag is the keystone. It transforms a general-purpose Claude Code instance into a restricted orchestrator that can only manage other agents — it cannot read files, write code, or run bash. This is the permission model the Orchestrator CLI needs: not a voluntary restriction through system prompt instructions, but an architectural constraint enforced at the tool level. The agent literally cannot violate its specialization because the tools do not exist in its environment.

AGENT_TEAMS and MULTI_TURN_AGENT together enable the forge's collaborative architecture. AGENT_TEAMS creates coordinated groups with shared objectives. MULTI_TURN_AGENT allows the Orchestrator to send follow-up messages to running subagents — "The Security CLI found an issue in your implementation. Address finding #3 before continuing." Without multi-turn capability, subagents are fire-and-forget: you launch them and wait for results. With it, you get genuine dialogue between specialists.

1.3 Agent Management (4 Tools)

These tools enable multi-CLI coordination — the core machinery that transforms a single Claude Code session into the forge. Without these tools, the Orchestrator is just another agent. With them, it becomes the brain of a distributed system.

The Agent tool is the Orchestrator's primary dispatch mechanism. When the Orchestrator decides that a task should go to the Security CLI, it invokes Agent with a task description, an optional agent name (matching a definition in .claude/agents/), and isolation parameters. The subagent spawns with its own context window, its own tool set (potentially restricted by the agent definition), and its own conversation history. It runs to completion and returns a single result message.

The critical subtlety is parallel agent spawning. When Claude issues multiple Agent tool calls in the same turn — and they have no dependencies — they execute concurrently. The Orchestrator can spawn the Security CLI, the QA CLI, and the Documentation CLI simultaneously, each in its own worktree. This is the foundation of the forge's parallel execution model, and it works because Agent is marked as concurrency-safe.

SendMessage enables multi-turn agent conversations. Without it, subagents are fire-and-forget: launch, wait for result, done. With SendMessage, the Orchestrator can redirect a running subagent mid-task: "The Security CLI found an issue with your authentication implementation. Address finding #3 before continuing." This transforms subagents from batch workers into interactive collaborators.

C.2 Slash Command Reference

C.3 Building Custom Tools

Claude Code's tool system is extensible. Custom tools integrate with the permission system, rendering pipeline, and discovery mechanism. There are two approaches: MCP servers for persistent, stateful tools, and skills for workflow-oriented tool compositions.

The Tool Interface

Every tool in Claude Code implements a standard interface. Understanding this interface is essential for building custom tools that behave consistently with built-in ones. The key methods:

  • execute() — The core logic. Receives validated input, returns structured output.
  • isReadOnly() — Determines Plan Mode availability. Read-only tools can be used during planning; write tools cannot.
  • needsPermission() — Determines whether the user is prompted. Permission rules in settings.json can override this.
  • isConcurrencySafe() — Determines parallel execution eligibility. Mark true only if the tool has no side effects that could conflict with concurrent operations.
  • renderToolUse() and renderToolResult() — Control how the tool appears in the user's terminal. Good rendering helps users understand what the agent is doing.

The interface is implemented in TypeScript (simplified from the source):

interface Tool {
  // Identity
  name: string;                          // "my_custom_tool"
  description: string;                   // Shown to the model

  // Schema
  inputSchema: JSONSchema;               // JSON Schema for parameters

  // Execution
  execute(input: ToolInput): Promise<ToolResult>;

  // Permission
  isReadOnly(): boolean;                 // true = available in Plan Mode
  needsPermission(input: ToolInput): boolean;  // true = prompt user

  // Concurrency
  isConcurrencySafe(): boolean;          // true = safe for parallel execution

  // Rendering
  renderToolUse(input: ToolInput): string;     // Human-readable description
  renderToolResult(result: ToolResult): string; // Human-readable output
}

Custom Tool via MCP Server

The simplest and most powerful way to add custom tools is through a Model Context Protocol (MCP) server. An MCP server is a long-running process that registers tools at session start. Claude Code communicates with it via JSON-RPC over stdin/stdout.

MCP servers are ideal for: - Database access (read-only queries, schema inspection) - External API integrations (Jira, Slack, PagerDuty) - Domain-specific analysis tools (static analyzers, profilers) - Persistent state management (caches, indexes, monitoring data)

Each MCP tool automatically inherits Claude Code's permission system — no additional permission logic needed in the server. The user controls access through settings.json allow/deny rules.

// .claude/settings.json
{
  "mcpServers": {
    "my-tools": {
      "command": "python",
      "args": ["tools/mcp_server.py"],
      "env": {}
    }
  }
}

# tools/mcp_server.py
from mcp.server import Server
from mcp.types import Tool, TextContent

server = Server("my-tools")

@server.tool("database_query")
async def database_query(query: str) -> list[TextContent]:
    """Execute a read-only SQL query against the project database."""
    # Validate query is SELECT only (security!)
    if not query.strip().upper().startswith("SELECT"):
        return [TextContent(text="Error: Only SELECT queries allowed")]

    results = await execute_query(query)
    return [TextContent(text=format_results(results))]

server.run()

Tool Selection by CLI

Not every CLI needs every tool. In fact, giving every CLI access to every tool is an anti-pattern — it violates the principle of least privilege and wastes context on tool descriptions the agent will never use.

The forge's permission matrix enforces separation of duties. The Coder CLI can edit files but cannot search the web. The Security CLI can read code but cannot edit it. The Orchestrator can spawn agents but cannot write code. These restrictions are not suggestions — they are architectural constraints that prevent CLIs from short-circuiting the collaborative workflow.

When configuring tools per CLI, follow this principle: each CLI should have exactly the tools it needs to fulfill its role, and no more. A Security CLI with edit access will be tempted to "just fix this one thing" instead of filing a finding. A Coder CLI with web access will spend tokens searching instead of implementing from the Planner's specification.

Appendix D: Protocol Specifications

Every distributed system eventually converges on the same truth: if you do not formalize your protocols, your protocols formalize themselves — as bugs.

The multi-CLI forge communicates through six core protocols. Each protocol defines message shape, state machine, error handling, and timing constraints. This appendix provides the formal specification for all six.

These specifications derive from Claude Code's source code (SendMessageTool, TaskCreateTool, TeamCreateTool schemas), the shared memory architecture from Chapter 14, and the IPC patterns from Chapter 15.

Every TypeScript interface is compilable. Every JSON example validates against its interface.

D.1 Message Envelope Specification

All inter-CLI communication uses a single envelope format. Every message — task assignments, memory operations, health checks, error reports — wraps its domain-specific payload in this envelope. The envelope provides routing, correlation, and lifecycle metadata.

TypeScript Interface

interface MessageEnvelope<T = unknown> {
  /** UUIDv4 — unique per message */
  id: string;

  /** CLI identity of sender (e.g., "coder", "security") */
  sender: string;

  /** CLI identity of recipient, or "*" for broadcast */
  recipient: string;

  /** Discriminated union tag — determines payload shape */
  type: MessageType;

  /** Domain-specific payload — shape determined by `type` */
  payload: T;

  /** Links related messages. All messages in a task share correlationId */
  correlationId: string;

  /** ISO 8601 timestamp of creation */
  timestamp: string;

  /** TTL in milliseconds. 0 = infinite. Default: 30000 (30s) */
  ttl: number;

  /** Message priority: 0 (highest) to 3 (lowest) */
  priority: 0 | 1 | 2 | 3;
}

type MessageType =
  | "task.assign"      // Orchestrator → specialist
  | "task.progress"    // Specialist → Orchestrator (partial)
  | "task.complete"    // Specialist → Orchestrator (final)
  | "task.error"       // Specialist → Orchestrator (failure)
  | "memory.write"     // Any → shared memory bus
  | "memory.read"      // Any → shared memory bus
  | "memory.response"  // Memory bus → requester
  | "finding.report"   // Security/QA → Orchestrator
  | "health.ping"      // Orchestrator → specialist
  | "health.pong"      // Specialist → Orchestrator
  | "consensus.propose" // Any → voting group
  | "consensus.vote"    // Voter → proposer
  | "system.shutdown";  // Orchestrator → all (broadcast)

Routing Rules

  1. Messages with recipient: "*" are delivered to all active CLIs except the sender
  2. Messages with a specific recipient are delivered only to that CLI
  3. Undeliverable messages (recipient not found) are returned to sender with type: "task.error" and error code RECIPIENT_NOT_FOUND
  4. Messages exceeding TTL are silently discarded. The sender must implement retry logic for critical messages

  "recipient": "security",

D.2 Task Lifecycle Protocol

Tasks flow through a defined state machine. Every transition produces a message. The Orchestrator tracks all active tasks and their states.

State Machine

PENDING → ASSIGNED → IN_PROGRESS → COMPLETED
                  ↘ IN_PROGRESS → FAILED → RETRY → IN_PROGRESS
                                         → ABANDONED

State Definitions

State Description Transition Trigger
PENDING Task created but not dispatched Orchestrator creates task
ASSIGNED Task dispatched to specialist task.assign message sent
IN_PROGRESS Specialist is actively working Specialist sends first task.progress
COMPLETED Task finished successfully Specialist sends task.complete
FAILED Task failed (retryable) Specialist sends task.error with retryable: true
RETRY Task being retried Orchestrator re-dispatches (max 3 retries)
ABANDONED Task permanently failed Max retries exceeded or retryable: false

Timeout Handling

Each task has a timeout_minutes field (default: 30). If no task.progress or task.complete message arrives within the timeout, the Orchestrator:

  1. Sends health.ping to the specialist
  2. Waits 30 seconds for health.pong
  3. If no response: marks task as FAILED with error code AGENT_TIMEOUT
  4. Re-dispatches to the same or different specialist (retry logic)

D.3 Shared Memory Protocol

The shared memory bus provides persistent key-value storage accessible to all CLIs. It supports namespacing, TTL, and conflict resolution.

Operations

interface MemoryWrite {
  namespace: string;     // "security", "coder", "shared"
  key: string;           // "auth-findings", "architecture-decision-001"
  value: unknown;        // Any JSON-serializable value
  ttl?: number;          // TTL in seconds. 0 = permanent. Default: 3600
  overwrite?: boolean;   // If false, fails on existing key. Default: true
}

interface MemoryRead {
  namespace: string;
  key: string;
}

interface MemoryResponse {
  found: boolean;
  value?: unknown;
  metadata: {
    created_by: string;    // CLI that wrote the value
    created_at: string;    // ISO 8601
    updated_at: string;    // ISO 8601
    version: number;       // Incremented on each write
    ttl_remaining: number; // Seconds until expiry. -1 = permanent
  };
}

Namespace Conventions

Namespace Owner Access Purpose
orchestrator Orchestrator Read: all. Write: Orchestrator only Task state, execution plans
security Security CLI Read: all. Write: Security only Findings, threat models, CVE data
coder Coder CLI Read: all. Write: Coder only Implementation notes, file ownership
validator Validator CLI Read: all. Write: Validator only Test results, coverage data
qa QA CLI Read: all. Write: QA only Bug reports, stress test results
shared All Read: all. Write: all Cross-cutting data, configuration

Conflict Resolution

When two CLIs write to the same key simultaneously:

  1. Same namespace, different CLIs: Rejected. Only the namespace owner can write
  2. Shared namespace: Last-write-wins with version tracking. The version field increments on each write; readers can detect stale data by comparing versions
  3. Overwrite false: First writer wins. Second write returns error KEY_EXISTS

D.4 Finding Report Protocol

Security and QA CLIs produce findings — structured vulnerability reports that flow to the Orchestrator for triage and to the Coder for remediation.

Finding Schema

interface Finding {
  id: string;                    // "FINDING-001"
  severity: "CRITICAL" | "HIGH" | "MEDIUM" | "LOW" | "INFO";
  cwe?: string;                  // "CWE-89"
  title: string;                 // "SQL Injection in login endpoint"
  location: {
    file: string;                // "src/auth/login.py"
    line_start: number;          // 42
    line_end: number;            // 47
  };
  description: string;           // Detailed explanation
  evidence: string;              // Code snippet or proof
  remediation: string;           // How to fix
  confidence: "CONFIRMED" | "PROBABLE" | "POSSIBLE";
  source: "sast" | "dast" | "manual" | "adversarial";
}

Finding Lifecycle

REPORTED → TRIAGED → ASSIGNED → IN_REMEDIATION → FIXED → VALIDATED
                   → REJECTED (false positive, documented justification)

The Orchestrator triages findings by severity. CRITICAL findings halt in-progress work and redirect the Coder CLI immediately. HIGH findings are queued for the next available remediation slot. MEDIUM and below are batched.

D.5 Health Check Protocol

The Orchestrator monitors all active CLIs through periodic health checks. Health checks use health.ping / health.pong messages.

Timing

Parameter Value Description
Ping interval 60 seconds How often the Orchestrator pings each CLI
Pong timeout 30 seconds How long to wait for response
Failure threshold 3 consecutive Mark CLI as unhealthy after 3 missed pongs
Recovery threshold 1 Mark CLI as healthy after 1 successful pong

Health States

HEALTHY → DEGRADED (1-2 missed pongs) → UNHEALTHY (3+ missed pongs)
UNHEALTHY → HEALTHY (1 successful pong)

When a CLI is marked UNHEALTHY: 1. Orchestrator stops assigning new tasks to it 2. In-progress tasks are marked FAILED with AGENT_UNHEALTHY 3. Orchestrator may restart the CLI process (if configured) 4. Tasks are re-dispatched to healthy CLIs

Pong Payload

interface HealthPong {
  status: "healthy" | "degraded" | "overloaded";
  activeTaskCount: number;
  contextUtilization: number;  // 0.0-1.0 (% of context window used)
  uptime_seconds: number;
  lastTaskCompleted?: string;  // ISO 8601
}

The contextUtilization field is critical for the Orchestrator's load balancing. A CLI at 0.9 context utilization is near compaction — the Orchestrator should avoid assigning complex tasks to it and consider spawning a fresh instance.

D.6 Error Code Reference

All protocol errors use a standardized error code system. Errors are categorized by severity and include machine-readable codes for automated handling.

Code Category Description Recommended Action
RECIPIENT_NOT_FOUND Routing Target CLI not registered Check CLI is running; re-register
MESSAGE_EXPIRED Transport TTL exceeded before delivery Retry with higher TTL or priority
KEY_EXISTS Memory Write to existing key with overwrite=false Read current value; merge or overwrite
NAMESPACE_VIOLATION Memory Write to namespace owned by another CLI Route through correct CLI
AGENT_TIMEOUT Health No response within timeout Health check; restart if unhealthy
AGENT_UNHEALTHY Health 3+ consecutive missed pongs Restart CLI; redistribute tasks
TASK_DEPENDENCY_UNMET Task Task depends on incomplete predecessor Wait or re-plan
MAX_RETRIES_EXCEEDED Task Task failed 3 times Mark abandoned; escalate to user
CONTEXT_EXHAUSTED Resource CLI context window at 95%+ utilization Spawn fresh instance; compact or split task
PERMISSION_DENIED Security CLI attempted unauthorized tool use Log violation; alert Orchestrator
CONSENSUS_TIMEOUT Consensus Voting deadline passed without quorum Re-propose or accept partial consensus
SCHEMA_VALIDATION_FAILED Protocol Message payload does not match expected schema Log and reject; sender must fix payload

Error Severity Levels

  • FATAL: System cannot continue. Requires Orchestrator intervention or user escalation
  • ERROR: Operation failed but system is functional. Retry or alternative approach possible
  • WARNING: Operation succeeded with caveats. No immediate action required but monitor
  • INFO: Expected condition, logged for audit trail
