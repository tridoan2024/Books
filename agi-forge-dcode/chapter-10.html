Chapter 10: The Security CLI — The Guardian

Read-Only by Design, SAST via Tree-Sitter, OWASP Automation, and the Separation of Finding from Fixing

In every engineering organization that takes security seriously, the person who writes the code is not the person who audits it. The developer who just spent three days building an authentication system has an emotional investment in its correctness — they see what it is supposed to do, not what it actually does. The security auditor arrives with fresh eyes, no investment, and a checklist of things that can go wrong. This separation is not about distrust. It is about cognitive independence: the auditor's analysis is clean because it was never contaminated by implementation decisions.

The Security CLI encodes this principle in architecture. It cannot write files. It cannot run commands that modify state. It cannot commit code or push branches. Its entire tool set is read-only — not by convention, not by a prompt instruction that says "please don't modify anything," but by structural enforcement. The Security CLI's tool registry does not include Edit, Write, or any Bash command that could alter the filesystem. If the LLM behind the Security CLI hallucinates a write operation, the tool registry rejects the call. The restriction is at the boundary, not in the prompt.

This chapter builds the Security CLI from five components: the read-only permission architecture (why it matters and how to enforce it), the SAST engine (finding vulnerabilities in code through AST analysis), the six-pass analysis pipeline (systematic coverage of all vulnerability classes), the structured reporting format (producing machine-readable findings that other CLIs consume), and the CLAUDE.md template that constrains the Security CLI to its auditing role.

The Security CLI is arguably the most important specialist in the fleet. A bug in the Coder's output wastes time. A vulnerability in the Coder's output exposes users, data, and the organization to real harm. The Security CLI is the last line of defense before code reaches production — and the only specialist whose correctness depends on never having touched the code it audits.

10.1 The Read-Only Permission Architecture

Why Read-Only Is Non-Negotiable

Consider what happens when a security scanner can also fix what it finds. The scanner detects SQL injection on line 47. It patches line 47 by adding input sanitization. The patch passes the scanner's own re-check — because the scanner and the fixer share the same understanding of what "fixed" means. But the patch introduced a subtle behavior change: it now truncates names containing apostrophes, breaking the application for users named O'Brien. The scanner did not test for this because its job was security, not correctness.

This is not a hypothetical. Automated security remediation tools routinely introduce regressions. Dependabot patches that auto-merge break builds. GitHub's automated security fixes change API behavior. The pattern is always the same: the entity that finds the problem optimizes for eliminating the finding, not for preserving correct behavior. Only a separate specialist (the Coder CLI, with its full context of the application's behavior) can fix a vulnerability while preserving correctness.

In a multi-CLI system, the separation is structural:

Coder CLI (writes code) → commits to branch
    ↓
Security CLI (reads code, produces findings) → structured JSON report
    ↓
Orchestrator (reads findings, decides action) → dispatch
    ↓
Coder CLI (fixes what Security CLI found) → commits fix to same branch
    ↓
Security CLI (re-scans — did the fix actually work?) → updated report

At no point does the finder also fix. At no point does the fixer also verify. The Coder never sees the Security CLI's raw findings — it receives a curated task description from the Orchestrator. The Security CLI never sees the Coder's implementation rationale — it sees only the code as it exists on disk.

Enforcing Read-Only in the Tool Registry

// clis/security/src/tools/mod.rs
pub fn security_tools() -> Vec<Box<dyn Tool>> {
    vec![
        // File reading — full access
        Box::new(ReadTool::new()),
        Box::new(GrepTool::new()),
        Box::new(GlobTool::new()),

        // Bash — read-only commands only
        Box::new(BashTool::with_allowlist(vec![
            "cat", "grep", "rg", "find", "ls", "wc", "head", "tail",
            "jq", "tree", "file", "stat", "diff",
            "git log", "git show", "git diff", "git blame",
            "semgrep", "gitleaks", "npm audit", "pip audit", "cargo audit",
            "node --eval",  // For JSON processing only
        ])),

        // Custom security tools
        Box::new(SASTScannerTool::new()),
        Box::new(SecretScannerTool::new()),
        Box::new(DependencyAuditTool::new()),
        Box::new(OWASPCheckerTool::new()),
        Box::new(ThreatModelerTool::new()),

        // NO Edit, Write, or general Bash access
    ]
}

The BashTool::with_allowlist() pattern is critical. Rather than denying specific dangerous commands (a denylist that can always be circumvented), we allow only specific safe commands (an allowlist that fails closed). Any command not on the list is rejected — even benign commands like echo or date — because the allowlist is designed for the Security CLI's specific needs, not for general-purpose shell access.

The BASH_CLASSIFIER: Lessons from Claude Code

Claude Code's bashPermissions.ts (2,621 lines) and its tree-sitter AST analysis in utils/bash/ast.ts (7,200+ lines combined across the analysis pipeline) represent the most sophisticated bash command security classification system in any AI coding tool. The Security CLI should learn from its design principles:

Fail-closed analysis: When the tree-sitter parser encounters a bash construct it does not recognize — a complex pipeline, a heredoc with embedded commands, a process substitution — it classifies the command as too-complex and requires human approval. It never guesses. This principle protects against novel bypass techniques: an attacker who crafts a command using an unusual bash syntax feature cannot trick the parser into allowing it, because unknown syntax is always denied.

Explicit structural allowlisting: The parser maintains a set of STRUCTURAL_TYPES (program, list, pipeline, redirected_statement) and SEPARATOR_TYPES (&&, ||, |, ;, &). Any AST node type not in these sets triggers rejection. This is the opposite of a traditional regex-based command filter: instead of looking for known-bad patterns in the command string, it requires that the command's structure matches known-good patterns.

Placeholder substitution for nested commands: Command substitutions ($(...)) are replaced with __CMDSUB_OUTPUT__ placeholders, and the inner commands are analyzed separately. This prevents a common bypass: embedding a dangerous command inside a command substitution within an otherwise-safe command (echo $(rm -rf /) looks like an echo command to a naive parser, but the rm -rf / executes before the echo receives its output).

These principles — fail-closed, explicit allowlisting, structural analysis, placeholder isolation — represent the state of the art in command security for AI agents. The reader's existing BASH_CLASSIFIER implementation in the Claude Code source (td_claudecode) implements all four, and the Security CLI's permission system should adopt the same rigor.

The Security CLI adapts these principles for its allowlist enforcement:

// clis/security/src/permissions/bash_classifier.rs
pub fn classify_command(command: &str) -> CommandClassification {
    // Step 1: Parse with tree-sitter-bash
    let tree = parse_bash(command);

    // Step 2: Verify structure uses only allowed node types
    if !all_nodes_allowed(&tree) {
        return CommandClassification::Denied("complex structure not allowed");
    }

    // Step 3: Extract the base command (argv[0])
    let base_cmd = extract_base_command(&tree);

    // Step 4: Check against allowlist
    if !SECURITY_CLI_ALLOWLIST.contains(&base_cmd) {
        return CommandClassification::Denied("command not in security CLI allowlist");
    }

    // Step 5: Check for dangerous flags even on allowed commands
    if has_dangerous_flags(&tree, &base_cmd) {
        return CommandClassification::Denied("dangerous flags detected");
    }

    CommandClassification::Allowed
}

fn has_dangerous_flags(tree: &Tree, cmd: &str) -> bool {
    match cmd {
        "git" => {
            // Allow: log, show, diff, blame
            // Deny: push, commit, merge, rebase, reset, checkout
            let subcmd = extract_git_subcommand(tree);
            !["log", "show", "diff", "blame", "status", "branch"].contains(&subcmd.as_str())
        }
        _ => false,
    }
}

10.2 The Six-Pass Analysis Pipeline

When the Orchestrator dispatches the Security CLI, it runs six analysis passes in sequence. Each pass targets a different vulnerability class, uses different tools, and produces findings in the same structured format. The six passes provide systematic coverage — no vulnerability class is skipped because the scanner got distracted by findings in an earlier category.

Pass 1: SAST (Static Application Security Testing)
  → Tree-sitter AST pattern scanning for injection, XSS, path traversal
  → Custom patterns per language

Pass 2: Secret Scanning
  → Gitleaks for known secret patterns (AWS keys, API tokens)
  → Entropy analysis for unrecognized high-entropy strings
  → Git history scanning (secrets that were committed and then "deleted")

Pass 3: Dependency Audit
  → npm audit / pip audit / cargo audit
  → CVE cross-reference with NVD database
  → License compliance (GPL in a proprietary codebase = problem)
  → Outdated package detection

Pass 4: OWASP Top 10 Checks
  → One detection module per OWASP category
  → A01:Broken Access Control through A10:SSRF
  → Pattern-based detection with LLM-assisted context evaluation

Pass 5: STRIDE Threat Modeling
  → Architecture analysis from Planner CLI's ADRs and diagrams
  → Trust boundary identification
  → Threat enumeration per STRIDE category

Pass 6: Configuration and Infrastructure
  → CORS configuration review
  → TLS/HTTPS enforcement
  → Security header analysis (CSP, HSTS, X-Frame-Options)
  → Environment variable exposure (secrets in .env committed to git)

The six passes are run in order because later passes benefit from earlier results. The STRIDE threat model (Pass 5) uses the SAST findings (Pass 1) to identify which threats have confirmed exploitable evidence. The configuration review (Pass 6) uses the OWASP findings (Pass 4) to prioritize which configurations to check. The dependency audit (Pass 3) informs the SAST pass's context evaluation — if a dependency has a known CVE, code that uses that dependency's affected API gets elevated severity.

Pass 1: SAST via Tree-Sitter

The SAST pass uses the same tree-sitter infrastructure as the Coder CLI (Chapter 9), but with different queries. Instead of finding function definitions and callers, the Security CLI queries for vulnerability patterns:

The context_check function is what elevates tree-sitter SAST above simple regex matching. Finding a format!() macro with SQL keywords is not always a vulnerability — it might be constructing a log message, or using a constant (not user input) for a table name. The context check examines the surrounding code: does the formatted string flow into a database query? Does the variable being interpolated come from user input? This context-aware analysis reduces false positives by 60-80% compared to pattern-only detection.

The difference between traditional SAST tools and the Security CLI's approach is the second evaluation layer. Semgrep and CodeQL produce raw findings based on pattern matches. They cannot reason about whether a finding matters in context. The Security CLI runs the pattern-based scan first (deterministic, fast, exhaustive), then evaluates each finding with LLM reasoning (slower, but able to assess context, intent, and actual exploitability). The combination gives you the coverage of automated tools with the judgment of a human reviewer.

Pass 2: Secret Scanning

Secret scanning catches credentials that should never have been committed to the repository. This is one of the most common and most damaging security failures in real-world software — GitHub's own research found that over 12 million secrets were committed to public repositories in 2023 alone, including AWS keys, database passwords, and API tokens that gave attackers direct access to production systems.

// clis/security/src/secrets/mod.rs
pub struct SecretScanner {
    patterns: Vec<SecretPattern>,
}

pub struct SecretPattern {
    pub name: String,
    pub regex: Regex,
    pub severity: Severity,
    pub entropy_threshold: Option<f64>,
}

impl SecretScanner {
    pub fn default_patterns() -> Self {
        Self {
            patterns: vec![
                SecretPattern {
                    name: "AWS Access Key".to_string(),
                    regex: Regex::new(r"AKIA[0-9A-Z]{16}").unwrap(),
                    severity: Severity::Critical,
                    entropy_threshold: None,
                },
                SecretPattern {
                    name: "GitHub Token".to_string(),
                    regex: Regex::new(r"gh[ps]_[A-Za-z0-9_]{36,}").unwrap(),
                    severity: Severity::Critical,
                    entropy_threshold: None,
                },
                SecretPattern {
                    name: "JWT Secret".to_string(),
                    regex: Regex::new(r#"(?i)(jwt|token|secret|key)\s*[:=]\s*["'][^"']{20,}["']"#).unwrap(),
                    severity: Severity::High,
                    entropy_threshold: Some(4.5), // High entropy confirms it's a real secret
                },
                SecretPattern {
                    name: "Private Key".to_string(),
                    regex: Regex::new(r"-----BEGIN (RSA |EC |OPENSSH )?PRIVATE KEY-----").unwrap(),
                    severity: Severity::Critical,
                    entropy_threshold: None,
                },
                SecretPattern {
                    name: "High-Entropy String".to_string(),
                    regex: Regex::new(r#"["'][A-Za-z0-9+/=]{40,}["']"#).unwrap(),
                    severity: Severity::Medium,
                    entropy_threshold: Some(5.0), // Only flag if entropy is very high
                },
            ],
        }
    }

    pub fn scan_file(&self, path: &str, content: &str) -> Vec<Finding> {
        let mut findings = Vec::new();

        for (line_num, line) in content.lines().enumerate() {
            for pattern in &self.patterns {
                if pattern.regex.is_match(line) {
                    // Entropy check: reduce false positives for generic patterns
                    if let Some(threshold) = pattern.entropy_threshold {
                        let matched = pattern.regex.find(line).unwrap();
                        if shannon_entropy(matched.as_str()) < threshold {
                            continue; // Low entropy = probably not a real secret
                        }
                    }

                    findings.push(Finding {
                        id: format!("SECRET-{}", findings.len() + 1),
                        severity: pattern.severity.clone(),
                        name: pattern.name.clone(),
                        file: path.to_string(),
                        line: line_num + 1,
                        snippet: redact_secret(line),  // Never include the actual secret
                        remediation: "Move secret to environment variable or secrets manager. Rotate the exposed credential immediately.".to_string(),
                    });
                }
            }
        }

        findings
    }
}

fn shannon_entropy(s: &str) -> f64 {
    let mut freq = [0u32; 256];
    for byte in s.bytes() {
        freq[byte as usize] += 1;
    }
    let len = s.len() as f64;
    freq.iter()
        .filter(|&&f| f > 0)
        .map(|&f| {
            let p = f as f64 / len;
            -p * p.log2()
        })
        .sum()
}

fn redact_secret(line: &str) -> String {
    // Replace the middle 80% of any detected secret with asterisks
    // NEVER include actual secrets in findings
    line.chars().enumerate().map(|(i, c)| {
        if i > line.len() / 5 && i < line.len() * 4 / 5 {
            '*'
        } else {
            c
        }
    }).collect()
}

The redact_secret function is critical. The Security CLI's findings are passed to the Orchestrator and then to the Coder CLI. If the finding includes the actual secret value, that secret is now in three different LLM context windows — each of which might be logged, cached, or sent to an API endpoint. Redacting the secret in the finding itself prevents this leakage. The redaction preserves enough of the value (first and last 20% of characters) for a developer to identify which secret is affected, while replacing the core material that an attacker would need.

Shannon entropy deserves a closer look because it is the key to catching secrets that do not match any known pattern. AWS keys and GitHub tokens have distinctive prefixes (AKIA, ghp_) that regex catches easily. But what about a custom API key generated by an internal service, or a database password chosen by an engineer? These have no predictable prefix. What they do have is high randomness — a 40-character alphanumeric string has far more entropy per character than English text, variable names, or even base64-encoded structured data. A threshold of 4.5-5.0 bits per character reliably separates random secrets from structured strings. Below 4.0, the string is almost certainly a hash, encoded value, or constant. Above 5.0, the string is almost certainly generated by a cryptographic random number generator — which is exactly what secret generation tools produce.

Pass 3: Dependency Audit

Your code can be perfect and still be vulnerable. The Log4Shell incident (CVE-2021-44228) demonstrated this with devastating clarity: a single transitive dependency in the Java ecosystem — one that most applications pulled in without ever importing directly — gave attackers remote code execution on hundreds of thousands of servers. The vulnerability was not in anyone's application code. It was in a logging library that a dependency of a dependency of a dependency included automatically.

The dependency audit pass exists because modern applications are 80-90% third-party code. A typical Node.js project with 50 direct dependencies pulls in 800-1,200 transitive dependencies. A Rust project with 30 direct dependencies in Cargo.toml resolves to 200-400 crates. Each of these is an attack surface that the Coder CLI never wrote and probably never read.

The Security CLI integrates with ecosystem-specific audit tools, running whichever apply to the project:

pub async fn audit_dependencies(&self, project_root: &Path) -> Vec<Finding> {
    let mut findings = Vec::new();

    // Detect package managers and run appropriate audits
    if project_root.join("package.json").exists() {
        let output = Command::new("npm")
            .args(["audit", "--json"])
            .current_dir(project_root)
            .output().await?;
        findings.extend(parse_npm_audit(&output.stdout));
    }

    if project_root.join("Cargo.toml").exists() {
        let output = Command::new("cargo")
            .args(["audit", "--json"])
            .current_dir(project_root)
            .output().await?;
        findings.extend(parse_cargo_audit(&output.stdout));
    }

    if project_root.join("requirements.txt").exists() || project_root.join("pyproject.toml").exists() {
        let output = Command::new("pip-audit")
            .args(["--format", "json"])
            .current_dir(project_root)
            .output().await?;
        findings.extend(parse_pip_audit(&output.stdout));
    }

    findings
}

fn parse_npm_audit(output: &[u8]) -> Vec<Finding> {
    let audit: NpmAuditReport = serde_json::from_slice(output)?;
    audit.vulnerabilities.iter().map(|(name, vuln)| {
        Finding {
            id: format!("DEP-NPM-{}", vuln.id),
            severity: map_npm_severity(&vuln.severity),
            cwe: vuln.cwe.first().cloned(),
            name: format!("{} {} — {}", name, vuln.range, vuln.title),
            file: "package.json".to_string(),
            remediation: format!(
                "Upgrade {} to {}. Run: npm audit fix",
                name, vuln.patched_versions.join(" or ")
            ),
            references: vuln.advisories.iter().map(|a| a.url.clone()).collect(),
            ..Default::default()
        }
    }).collect()
}

Beyond known CVEs, the dependency audit checks two often-overlooked dimensions. Outdated package detection identifies dependencies more than two major versions behind — not because old code is inherently insecure, but because old code stops receiving security patches. A package abandoned by its maintainer accumulates disclosed vulnerabilities that will never be fixed upstream. License compliance catches dependencies whose licenses conflict with the project's licensing model. GPL-licensed code in a proprietary codebase is a legal finding, not a security finding, but the Security CLI surfaces it because the Orchestrator needs to know, and the dependency audit is the natural place to detect it.

### Passes 4-6: OWASP, STRIDE, and Configuration

Passes 4-6 are where the Security CLI's LLM reasoning provides its greatest advantage over traditional tools. Pattern matching finds syntactic vulnerabilities — injection, hardcoded secrets, known-bad APIs. But the most dangerous vulnerabilities are *semantic*: flaws in business logic, access control, and system design that look like correct code to any automated scanner.

**Pass 4: OWASP Top 10 Check** — The Security CLI evaluates code against all ten OWASP categories (A01:Broken Access Control through A10:Server-Side Request Forgery), but the evaluation is contextual, not pattern-based. Consider A01:Broken Access Control — the most common web application vulnerability. A traditional scanner might check for the presence of authentication middleware. The Security CLI reads the actual endpoint implementations and asks: *Does this endpoint that retrieves user data by ID verify that the authenticated user has permission to access that specific user's record?* A function like `get_user_profile(user_id)` that takes the ID from the URL path and returns the user's data is a textbook Insecure Direct Object Reference (IDOR) — but only if the endpoint does not verify ownership. The LLM reads the surrounding middleware, the authorization checks (or their absence), and the data flow from URL parameter to database query. No regex can do this.

Each OWASP category has its own evaluation context:

| Category | What the LLM Evaluates |
|----------|----------------------|
| A01: Broken Access Control | Authorization checks per endpoint, IDOR patterns, role enforcement |
| A02: Cryptographic Failures | Algorithm choices, key management, data classification vs. protection |
| A03: Injection | Already covered by Pass 1 SAST — cross-referenced here for completeness |
| A04: Insecure Design | Missing rate limiting, absence of abuse cases in design, trust assumptions |
| A05: Security Misconfiguration | Default credentials, unnecessary features enabled, stack traces exposed |
| A06: Vulnerable Components | Already covered by Pass 3 — cross-referenced here |
| A07: Authentication Failures | Password policies, session management, brute-force protection |
| A08: Data Integrity Failures | Unsigned updates, deserialization of untrusted data, CI/CD pipeline security |
| A09: Logging Failures | Missing audit trails, PII in logs, insufficient monitoring |
| A10: SSRF | URL validation, internal network access from user-supplied URLs |

The cross-references between passes are deliberate. When the OWASP check encounters A03:Injection, it does not re-scan — it references the SAST findings from Pass 1. When it encounters A06:Vulnerable Components, it references the dependency audit from Pass 3. This prevents duplicate findings while ensuring every OWASP category is explicitly addressed.

**Pass 5: STRIDE Threat Modeling** — The Security CLI reads the Planner CLI's architecture diagrams and ADRs (Chapter 8), identifies trust boundaries, and evaluates each component against the STRIDE categories: Spoofing (can an attacker impersonate a legitimate entity?), Tampering (can data be modified in transit or at rest?), Repudiation (can a user deny performing an action?), Information Disclosure (can sensitive data leak?), Denial of Service (can the system be overwhelmed?), and Elevation of Privilege (can a low-privilege user gain higher access?). The output is a structured threat model table that the Orchestrator can prioritize alongside the SAST and OWASP findings.

The STRIDE pass differs from OWASP in scope: OWASP evaluates *implementation* (the code as written), while STRIDE evaluates *architecture* (the system as designed). An application might pass every OWASP check while having a STRIDE-level flaw in its architecture — for example, a microservice that trusts all internal network traffic without authentication, enabling any compromised service to access any other service's data.

**Pass 6: Configuration and Infrastructure** — The final pass reviews configuration files that traditional code scanners ignore entirely. CORS headers set to `Access-Control-Allow-Origin: *` permit any website to make API calls — a finding that exists in a YAML or JSON configuration file, not in application code. Missing `Strict-Transport-Security` headers allow downgrade attacks. Absent `Content-Security-Policy` headers permit XSS exploitation even when the application code is injection-free. Debug mode left enabled in production configuration exposes stack traces, internal paths, and sometimes environment variables to unauthenticated users.

The configuration pass also checks for environment variable exposure: `.env` files committed to git, secrets passed as command-line arguments (visible in process listings), and sensitive values stored in configuration files without encryption.

## 10.3 The Structured Finding Format

Every pass produces findings in the same format, enabling consistent processing by the Orchestrator and the Coder CLI:

```rust
pub struct Finding {
    pub id: String,                // Unique finding ID (SAST-001, SECRET-003, DEP-012)
    pub pass: AnalysisPass,        // Which pass found this
    pub severity: Severity,        // Critical, High, Medium, Low, Info
    pub cwe: Option<String>,       // CWE reference (CWE-89, CWE-79, etc.)
    pub owasp: Option<String>,     // OWASP category (A01, A03, etc.)
    pub name: String,              // Human-readable finding name
    pub description: String,       // What the vulnerability is
    pub file: String,              // Affected file path
    pub line: usize,               // Line number
    pub snippet: String,           // Redacted code snippet
    pub confidence: Confidence,    // High, Medium, Low
    pub remediation: String,       // How to fix it
    pub references: Vec<String>,   // URLs to CWE, OWASP, advisories
}

pub enum Severity {
    Critical,  // Exploitable, data breach risk, must fix before ship
    High,      // Exploitable with effort, should fix before ship
    Medium,    // Risk exists but exploitation requires specific conditions
    Low,       // Minor issue, defense-in-depth improvement
    Info,      // Informational, no immediate risk
}

The Orchestrator uses severity to decide actions:

Severity Orchestrator Action
Critical Block shipment. Dispatch Coder to fix immediately. Re-scan after fix.
High Block shipment. Dispatch Coder to fix. Re-scan after fix.
Medium Add to backlog. Coder may fix in current sprint or defer.
Low Add to backlog. No immediate action required.
Info Include in report for awareness. No action.

10.4 LLM-Augmented Analysis: Beyond Pattern Matching

Traditional SAST tools (Semgrep, CodeQL) are excellent at finding known patterns. They are terrible at finding novel vulnerabilities, understanding business logic flaws, or evaluating whether a finding is a true positive in context. The Security CLI's LLM-augmented analysis fills this gap.

After the tree-sitter SAST pass produces raw findings, the Security CLI uses its LLM to evaluate each finding in context:

## Finding Evaluation Prompt (in Security CLI's system prompt)

For each SAST finding, evaluate:
1. Is this a TRUE POSITIVE? Read 20 lines above and below the flagged line.
   Does the flagged pattern actually connect to user input? Is there
   sanitization nearby that the pattern matcher missed?
2. What is the actual IMPACT? A SQL injection in an admin-only endpoint
   is High (not Critical). A SQL injection in the public login endpoint
   is Critical.
3. Is there a COMPENSATING CONTROL? The code might use a WAF, input
   validation middleware, or ORM that prevents exploitation even though
   the code pattern looks vulnerable.

Adjust severity based on context. Demote false positives to Info with
explanation. Promote findings where context reveals additional risk.

This evaluation reduces false positives by 60-80%. A traditional SAST tool that flags every format!() macro containing SQL keywords produces 50 findings, of which 8 are real. The LLM-augmented evaluation reads the surrounding code and demotes the 42 false positives, leaving 8 true findings that the Coder CLI actually needs to fix. This dramatically reduces the signal-to-noise ratio and prevents the "alert fatigue" that causes teams to ignore security findings entirely.

A Worked Example: SQL Injection Triage

Consider a Rust web application where the tree-sitter SAST pass flags three findings, all matching the format!() + SQL keyword pattern:

Finding 1: src/api/users.rs:47

let query = format!("SELECT * FROM users WHERE id = {}", req.params.get("id").unwrap());
let result = sqlx::query(&query).fetch_one(&pool).await?;

Finding 2: src/db/migrations.rs:12

let create_table = format!("CREATE TABLE {} (id SERIAL PRIMARY KEY, name TEXT)", TABLE_NAME);
sqlx::query(&create_table).execute(&pool).await?;

Finding 3: src/reports/export.rs:89

let query = format!("SELECT {} FROM orders WHERE status = 'completed'", columns.join(", "));
let results = sqlx::query(&query).fetch_all(&pool).await?;

All three match CWE-89 (SQL Injection). A traditional SAST tool reports all three as Critical. The LLM evaluation produces different results:

Finding 1 → CONFIRMED Critical. The id parameter comes directly from req.params (user input from the HTTP request). No sanitization. No parameterized query. This is textbook SQL injection. The remediation is clear: use sqlx::query!("SELECT * FROM users WHERE id = $1", id).

Finding 2 → Demoted to Info. TABLE_NAME is a const defined at the top of the file: const TABLE_NAME: &str = "users";. This is a constant, not user input. The format!() is used for code clarity, not for dynamic SQL construction. No exploitation path exists. The LLM notes this as a style suggestion (prefer static SQL strings) but not a security finding.

Finding 3 → Elevated to Critical with nuance. The columns variable comes from a configuration file, not directly from user input. But the LLM reads the call chain: columns is loaded from a JSON config that an admin user can modify through the application's settings UI. An admin who controls the column list can inject SQL through a crafted column name like *; DROP TABLE orders; --. The finding is real, but the attack requires admin access, so the LLM notes that the severity could be High rather than Critical depending on the organization's threat model.

10.5 The Complete Security CLI CLAUDE.md

# SecurityCLI — The Guardian

## Identity
You are the Security CLI — the guardian of a multi-CLI agent system.
You audit code for vulnerabilities. You find what others miss.
You NEVER modify code. You NEVER commit, push, or write files.
You are READ-ONLY. This is your defining characteristic and your strength.

## The Separation Principle
The entity that finds vulnerabilities must NEVER be the entity that fixes them.
Your findings are clean because you never touched the code.
Your analysis is independent because you have no investment in the code's correctness.
If you could fix what you find, your analysis would be compromised.

## Six-Pass Analysis Pipeline
Run ALL six passes for every audit. Do not skip passes.

### Pass 1: SAST
- Use tree-sitter AST patterns for injection, XSS, path traversal, command injection
- Use context to evaluate each finding (true positive vs false positive)
- Output: Finding with CWE, severity, file:line, remediation

### Pass 2: Secret Scanning
- Scan for AWS keys, API tokens, private keys, JWTs, high-entropy strings
- Scan git history for previously committed secrets
- NEVER include actual secret values in findings — always redact
- Output: Finding with redacted snippet and rotation guidance

### Pass 3: Dependency Audit
- Run ecosystem-specific audit tools (npm audit, cargo audit, pip audit)
- Cross-reference CVEs with NVD database
- Check for outdated packages and license compliance
- Output: Finding with CVE number, affected version, fixed version

### Pass 4: OWASP Top 10
- Evaluate code against all 10 categories (A01-A10)
- Use LLM reasoning for business logic flaws that tools cannot detect
- Output: Finding with OWASP category, description, evidence

### Pass 5: STRIDE Threat Model
- Read architecture diagrams and ADRs from the Planner CLI
- Identify trust boundaries and data flows
- Enumerate threats per STRIDE category
- Output: Threat model table with likelihood, impact, mitigation

### Pass 6: Configuration Review
- Check security headers, CORS, TLS, error handling
- Verify no debug mode, verbose errors, or stack traces in production config
- Output: Configuration findings with remediation

## Finding Format
Every finding MUST include:
- Unique ID (SAST-001, SECRET-003, DEP-012, OWASP-A01-001)
- Severity (Critical/High/Medium/Low/Info)
- CWE reference where applicable
- File path and line number
- Redacted code snippet (NEVER include secrets)
- Confidence level (High/Medium/Low)
- Specific remediation steps (not generic advice)

## Severity Calibration
- CRITICAL: Exploitable by unauthenticated attacker, leads to data breach or RCE
- HIGH: Exploitable with authentication or specific conditions, significant impact
- MEDIUM: Risk exists but exploitation requires unlikely conditions
- LOW: Defense-in-depth improvement, no direct exploitation path
- INFO: Informational only, best practice suggestion

## What You Do NOT Do
- Write, edit, or modify any file
- Commit, push, or create branches
- Run commands that change state (install, build, delete)
- Fix vulnerabilities (that is the Coder's job)
- Make architecture decisions (that is the Planner's job)
- You FIND problems. Others FIX problems.

10.6 Security CLI in Non-Engineering Domains

The Security CLI pattern — independent, read-only auditing with structured findings — applies wherever compliance and correctness matter. The key architectural insight is not specific to code: it is that the entity performing the audit must never have the ability to modify what it audits.

Legal Compliance Auditor: Reviews contracts for regulatory compliance violations (GDPR data handling, SOX controls, HIPAA disclosures). Reads documents but never modifies them. Produces structured findings with regulation references, risk severity, and remediation guidance. The finding "Section 4.2 allows data retention beyond GDPR's 'right to erasure' requirements (Article 17)" is structurally identical to "src/auth/jwt.ts:47 contains hardcoded HMAC secret (CWE-798)." Both have a location, a reference to a standard, a severity, and a remediation. The legal auditor's six-pass pipeline maps naturally: Pass 1 scans for known problematic clause patterns (like open-ended data retention), Pass 2 checks for missing required clauses (like data breach notification timelines), Pass 3 audits referenced third-party agreements for compliance gaps (the dependency audit equivalent), and Passes 4-6 evaluate against regulation-specific checklists (GDPR, CCPA, SOX).

Financial Audit CLI: Reviews transactions, reports, and calculations for discrepancies and compliance violations. Read-only access to financial data. Produces findings with SOX control references. The separation principle is even more critical here — a financial auditor who can modify the books is not an auditor. Sarbanes-Oxley Section 404 explicitly requires that internal controls over financial reporting be evaluated by parties independent of those who operate the controls. The Security CLI's read-only architecture enforces this independence structurally, not procedurally: the Financial Audit CLI cannot modify ledger entries even if the LLM behind it hallucinates a "correction," because the tool registry does not include write operations.

Medical Record Reviewer: Audits clinical documentation for completeness, coding accuracy, and HIPAA compliance. Read-only access to patient records. Findings include ICD-10 code discrepancies, missing informed consent documentation, and privacy violations. The read-only constraint protects patient data integrity — a clinical audit that accidentally modifies a patient's diagnosis code could trigger incorrect treatment decisions. The six-pass model adapts cleanly: Pass 1 checks documentation completeness, Pass 2 scans for PHI exposure in non-clinical fields, Pass 3 audits third-party data sharing agreements, Pass 4 evaluates against CMS quality measures, Pass 5 models data flow trust boundaries (who can access what), and Pass 6 reviews system configuration for access controls and audit logging.

In every domain, the Security CLI's architecture provides the same guarantees: independent analysis, structural read-only enforcement, structured findings, and clear separation between finding problems and fixing them. The six-pass pipeline, the structured finding format, and the severity calibration are domain-agnostic infrastructure. Only the patterns, checklists, and remediation guidance change between engineering and non-engineering applications.

10.7 Testing the Security CLI

The Security CLI's tests verify a different kind of correctness than other specialist CLIs. When the Coder CLI has a bug, it produces wrong code — annoying but fixable. When the Security CLI has a bug, it either misses real vulnerabilities (false negatives that reach production) or drowns the team in phantom alerts (false positives that erode trust in the scanner). Both failure modes are dangerous, but false negatives are existentially worse: a missed SQL injection finding means the vulnerability ships to production where real attackers can exploit it. These six test scenarios cover the Security CLI's critical correctness properties.

Test 1: Read-Only Enforcement

Scenario: Security CLI attempts to use Edit or Write tool.
Expected: Tool registry rejects the call. Security CLI reports it cannot modify files.

Test 2: SQL Injection Detection

Input: File containing `format!("SELECT * FROM users WHERE id = {}", user_id)`
Expected: Finding SAST-R001, CWE-89, Critical severity, with remediation
pointing to sqlx parameterized queries.

Test 3: False Positive Reduction

Input: File containing `format!("SELECT * FROM users WHERE id = {}", 42)` (constant, not user input)
Expected: LLM evaluation demotes to Info or filters out entirely.
Not flagged as Critical.

Test 4: Secret Redaction

Input: File containing `API_KEY = "sk-ant-abc123def456ghi789"`
Expected: Finding includes redacted snippet (first/last chars visible, middle asterisked).
The actual key value NEVER appears in the finding.

Test 5: Multi-Pass Coverage

Input: Project with SQL injection, committed AWS key, outdated dependency with CVE.
Expected: Three separate findings from three different passes (SAST, Secret, Dep).
All six passes run even though findings were found early.

Test 6: Severity Calibration

Input: SQL injection in admin-only endpoint (requires authentication + admin role)
Expected: Severity = High (not Critical), because exploitation requires admin access.
Context-aware calibration, not blind pattern matching.

If all six tests pass, your Security CLI enforces read-only access, detects vulnerabilities accurately, reduces false positives with context, redacts secrets, provides multi-pass coverage, and calibrates severity based on actual exploitability. These tests should be run against known-vulnerable sample projects (deliberately insecure codebases like OWASP's WebGoat or Damn Vulnerable Web Application) to verify detection rates, and against clean projects to verify that the false positive rate stays below 20%. A security scanner that nobody trusts is the same as no security scanner at all.
