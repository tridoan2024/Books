Chapter 7: The Research CLI — Knowledge Engine

Custom Tools, RAG Pipelines, Knowledge Graphs, and the Art of Finding What Matters

Every other specialist CLI in this book produces artifacts — code, tests, documentation, security reports. The Research CLI produces knowledge. It is the specialist you dispatch when the Orchestrator needs facts before making decisions: "What authentication libraries does the JavaScript ecosystem currently recommend?" "What are the known vulnerabilities in this dependency version?" "How does the competitor's API handle pagination?" "What does the latest RFC say about this protocol?"

The Research CLI's value proposition is not that it can search the web — any Claude Code instance can do that with the WebSearch and WebFetch tools. Its value is that it searches systematically, evaluates source credibility, cross-references claims, caches findings for reuse, and produces structured knowledge that other specialists can consume. A Coder CLI that searches the web during implementation gets distracted by tangential results and loses context. A Research CLI that searches the web as its primary function produces curated, verified knowledge that the Coder consumes without distraction.

This chapter builds the Research CLI from four components: the custom tool set (extending Claude Code's built-in search capabilities), the RAG pipeline (Retrieval-Augmented Generation for working with large document collections), the knowledge graph (connecting facts into a navigable structure), and the memory system (persisting research across sessions). By the end, you will have a Research CLI that turns vague research questions into structured, cited, reusable knowledge.

7.1 Why a Dedicated Research Specialist?

Consider what happens when the Orchestrator dispatches a Build task to the Coder CLI: "Build a REST API with rate limiting." The Coder needs to decide which rate limiting library to use. It could search the web, but that search consumes 3-5 tool calls, generates 10,000+ tokens of search results, and fills its context with comparative analysis that has nothing to do with writing code. By the time it starts coding, 15% of its context window is occupied by research it will never reference again.

Now consider the alternative: the Orchestrator dispatches a Research task first: "Research rate limiting approaches for Node.js REST APIs. Compare express-rate-limit, rate-limiter-flexible, and @upstash/ratelimit. Recommend one with rationale." The Research CLI spends its full context on this analysis — reading documentation, checking GitHub stars and maintenance status, comparing APIs, evaluating edge cases. It produces a structured recommendation:

{
  "recommendation": "rate-limiter-flexible",
  "rationale": "Supports Redis and memory backends, sliding window algorithm, per-route configuration. 4.2K GitHub stars, last commit 3 days ago.",
  "alternatives": [
    {"name": "express-rate-limit", "trade_off": "Simpler API but memory-only, no distributed support"},
    {"name": "@upstash/ratelimit", "trade_off": "Serverless-native but vendor lock-in (Upstash Redis)"}
  ],
  "implementation_notes": "Use Redis backend for production. Configure per-route limits in middleware chain.",
  "sources": [
    "https://github.com/animir/node-rate-limiter-flexible",
    "https://npmjs.com/package/rate-limiter-flexible"
  ]
}

The Orchestrator passes this structured recommendation to the Coder CLI in its dispatch prompt. The Coder now has a clear decision — it writes code, not research. Its context stays clean. The total system is faster because the research happened in parallel with other Wave 1 tasks (schema design, environment configuration), and the Coder started with a pre-made decision instead of spending 5 minutes deliberating.

This separation also enables research caching. When the next task needs rate limiting research (perhaps the QA CLI needs to test rate limits), the cached result is available without repeating the web searches. Over the course of a large project, research caching saves significant time and API costs.

The Research CLI adds three custom tools that transform it from a generic searcher into a knowledge engine:

DocumentRAG: Working with Large Document Collections

When researching a topic requires reading more content than fits in a single context window — API documentation spanning 50 pages, a specification with 200 sections, a codebase with thousands of files — the Research CLI needs a way to find the relevant passages without reading everything.

// clis/research/src/tools/document_rag.rs
pub struct DocumentRAGTool {
    index: Arc<DocumentIndex>,
}

pub struct DocumentIndex {
    chunks: Vec<DocumentChunk>,
    embeddings: Vec<Vec<f32>>,    // Pre-computed embeddings
    metadata: Vec<ChunkMetadata>,  // Source, page, section
}

pub struct DocumentChunk {
    pub content: String,
    pub source: String,           // File path or URL
    pub section: String,          // Chapter, heading, or page
    pub chunk_id: usize,
}

impl DocumentRAGTool {
    pub async fn query(&self, question: &str, top_k: usize) -> Vec<RetrievalResult> {
        // Step 1: Embed the question
        let query_embedding = self.embed(question).await;

        // Step 2: Find top-k similar chunks by cosine similarity
        let mut scored: Vec<(usize, f32)> = self.index.embeddings.iter()
            .enumerate()
            .map(|(i, emb)| (i, cosine_similarity(&query_embedding, emb)))
            .collect();
        scored.sort_by(|a, b| b.1.partial_cmp(&a.1).unwrap());

        // Step 3: Re-rank with cross-encoder for precision
        let candidates: Vec<&DocumentChunk> = scored.iter()
            .take(top_k * 3)  // Retrieve 3x for re-ranking headroom
            .map(|(i, _)| &self.index.chunks[*i])
            .collect();

        let reranked = self.cross_encoder_rerank(question, &candidates, top_k).await;

        // Step 4: Return with source attribution
        reranked.into_iter().map(|chunk| RetrievalResult {
            content: chunk.content.clone(),
            source: chunk.source.clone(),
            section: chunk.section.clone(),
            relevance_score: chunk.score,
        }).collect()
    }
}

The RAG pipeline follows the standard four-stage process: embed the query, retrieve candidate chunks by vector similarity, re-rank candidates with a more precise model, and return results with source attribution. The re-ranking step is critical — vector similarity is a blunt instrument that often returns tangentially related content. A cross-encoder that sees both the query and the candidate simultaneously produces much better relevance scores.

For the Research CLI's use case, the documents being indexed are: - API documentation scraped from official sites - Specification documents (RFCs, standards, protocol specs) - Codebase documentation (README files, doc comments, architecture docs) - Previous research results (cached from earlier sessions) - Git commit history (changelogs, release notes, migration guides)

The index is built lazily — documents are indexed when first accessed, not preloaded at startup. A Research CLI analyzing Node.js libraries indexes the Express documentation when the first Express-related query arrives, not when the session starts. This keeps startup time fast and avoids indexing documents that are never queried.

SourceAnalyzer: Evaluating Credibility

Not all sources are equal. A Stack Overflow answer from 2019 about Node.js 12 is not useful for a Node.js 22 project. An npm package with 50 downloads and no TypeScript types is not a safe recommendation. The SourceAnalyzer tool evaluates the credibility and recency of information sources:

// clis/research/src/tools/source_analyzer.rs
pub struct SourceAnalyzerTool;

#[derive(Debug)]
pub struct SourceCredibility {
    pub url: String,
    pub credibility_score: f32,    // 0.0 - 1.0
    pub recency: SourceRecency,
    pub authority: SourceAuthority,
    pub factors: Vec<CredibilityFactor>,
}

pub enum SourceRecency {
    Current,          // Updated within 6 months
    Recent,           // 6-18 months
    Dated,            // 18-36 months
    Stale,            // 3+ years
    Unknown,
}

pub enum SourceAuthority {
    Official,         // Official documentation, RFC, spec
    Authoritative,    // Well-known blog, major publication, core maintainer
    Community,        // Stack Overflow, GitHub discussions, tutorials
    Unknown,          // Unverified source
}

impl SourceAnalyzerTool {
    pub fn evaluate(&self, url: &str, content: &str) -> SourceCredibility {
        let mut factors = Vec::new();
        let mut score: f32 = 0.5; // Neutral baseline

        // Domain authority
        if is_official_docs(url) {
            score += 0.3;
            factors.push(CredibilityFactor::OfficialDocumentation);
        } else if is_known_authority(url) {
            score += 0.2;
            factors.push(CredibilityFactor::KnownAuthorityDomain);
        }

        // Recency from dates in content
        let recency = extract_recency(content);
        match &recency {
            SourceRecency::Current => { score += 0.1; }
            SourceRecency::Stale => { score -= 0.2; }
            _ => {}
        }

        // Version alignment (does the content reference current versions?)
        if references_current_versions(content) {
            score += 0.1;
            factors.push(CredibilityFactor::CurrentVersions);
        }

        SourceCredibility {
            url: url.to_string(),
            credibility_score: score.clamp(0.0, 1.0),
            recency,
            authority: classify_authority(url),
            factors,
        }
    }
}

fn is_official_docs(url: &str) -> bool {
    let official_patterns = [
        "docs.rs", "doc.rust-lang.org", "nodejs.org/api",
        "developer.mozilla.org", "docs.python.org",
        "learn.microsoft.com", "cloud.google.com/docs",
        "docs.aws.amazon.com", "postgresql.org/docs",
    ];
    official_patterns.iter().any(|p| url.contains(p))
}

The SourceAnalyzer prevents a common failure mode: the Research CLI recommending a library based on a 3-year-old blog post when the library has since been deprecated. By scoring sources and flagging staleness, it forces the Research CLI to prefer current, authoritative information.

7.3 The RAG Pipeline in Depth

The DocumentRAG tool introduced in Section 7.2 implements a four-stage pipeline. Each stage has design decisions that significantly affect research quality.

Stage 1: Chunking Documents

Before documents can be searched, they must be split into chunks small enough for embedding. The chunking strategy determines what the Research CLI can find.

Naive chunking splits by character count (e.g., 500 characters per chunk). This is simple but produces terrible results — it splits mid-sentence, separates code from its explanation, and breaks table rows across chunks. The retriever returns half a code example with no context.

Semantic chunking splits on natural boundaries: paragraph breaks, heading boundaries, code block delimiters, and list items. It keeps related content together:

pub fn chunk_document(content: &str, source: &str) -> Vec<DocumentChunk> {
    let mut chunks = Vec::new();
    let mut current_section = String::new();
    let mut current_heading = String::from("Introduction");
    let mut chunk_id = 0;

    for line in content.lines() {
        // Detect heading boundaries
        if line.starts_with('#') {
            // Flush current chunk
            if !current_section.trim().is_empty() {
                chunks.push(DocumentChunk {
                    content: current_section.clone(),
                    source: source.to_string(),
                    section: current_heading.clone(),
                    chunk_id,
                });
                chunk_id += 1;
                current_section.clear();
            }
            current_heading = line.trim_start_matches('#').trim().to_string();
        }

        current_section.push_str(line);
        current_section.push('\n');

        // Split if chunk exceeds max size (but only at paragraph boundaries)
        if current_section.len() > 2000 && line.trim().is_empty() {
            chunks.push(DocumentChunk {
                content: current_section.clone(),
                source: source.to_string(),
                section: current_heading.clone(),
                chunk_id,
            });
            chunk_id += 1;
            current_section.clear();
        }
    }

    // Flush remaining content
    if !current_section.trim().is_empty() {
        chunks.push(DocumentChunk {
            content: current_section,
            source: source.to_string(),
            section: current_heading,
            chunk_id,
        });
    }

    chunks
}

The chunk size of 2,000 characters is a tradeoff. Smaller chunks (500 characters) produce more precise retrieval but lose context — the chunk about JWT token expiry might not include the security warning two paragraphs later. Larger chunks (5,000 characters) preserve context but reduce precision — the chunk about authentication might include irrelevant paragraphs about unrelated middleware. 2,000 characters (roughly 400-500 tokens) is the sweet spot for technical documentation.

Stage 2: Embedding

Each chunk gets embedded into a dense vector representation. The Research CLI uses a local embedding model to avoid sending potentially confidential content to external APIs:

pub struct EmbeddingEngine {
    model: OnnxModel, // Local ONNX-format embedding model
}

impl EmbeddingEngine {
    pub fn embed_batch(&self, texts: &[&str]) -> Vec<Vec<f32>> {
        texts.iter()
            .map(|text| {
                let tokens = self.tokenize(text);
                let output = self.model.run(&tokens);
                self.mean_pool(&output)  // Mean pooling of token embeddings
            })
            .collect()
    }
}

For projects where confidentiality is not a concern, the Anthropic embedding API or OpenAI's embedding API provide higher-quality embeddings without local model overhead. The choice depends on the deployment context — a Research CLI working on proprietary code should use local embeddings; one researching public documentation can use API embeddings.

The embedding dimension matters for storage and performance. Typical models produce 384-dimensional (compact, fast) or 1024-dimensional (precise, slower) vectors. For the Research CLI's use case — indexing thousands of documentation chunks, not millions — 1024 dimensions provide meaningfully better retrieval quality without significant performance impact. Store embeddings in a local SQLite database with the sqlite-vss extension for vector similarity search, or in a flat file if the corpus is small enough to fit in memory.

Stage 3: Retrieval and Re-Ranking

Vector similarity retrieval is fast but imprecise. It finds chunks that are "about the same topic" as the query, but it cannot distinguish between chunks that directly answer the question and chunks that are tangentially related.

Re-ranking fixes this. A cross-encoder model receives both the query and each candidate chunk, and scores how well the chunk actually answers the question — not just whether it is topically similar. The difference is dramatic: vector retrieval for "how to configure JWT token expiry" might return chunks about JWT in general, chunks about token types, and chunks about configuration in other contexts. The re-ranker identifies the one chunk that specifically explains the expiresIn configuration option.

The two-stage approach (fast retrieval then precise re-ranking) is essential for performance. Running the cross-encoder on all 10,000 chunks would take minutes. Running it on the top 30 candidates from vector retrieval takes seconds and produces near-optimal results.

Stage 4: Response Generation with Citations

The Research CLI does not simply return retrieved chunks — that would be a search engine, not a research assistant. It synthesizes chunks into a coherent answer with inline citations pointing back to the original sources. This is the "Generation" in Retrieval-Augmented Generation, and it is where the Research CLI adds value beyond what a vector database provides.

## Research Instructions for Response Generation

When generating a research response from retrieved documents:

1. Synthesize information from multiple chunks into a coherent narrative
2. CITE every factual claim with [Source: url#section]
3. When chunks disagree, present both positions and note the conflict
4. When information is outdated (SourceRecency::Stale), explicitly flag it
5. NEVER fabricate citations — if a fact is not in the retrieved chunks, say "not found in available sources"
6. Confidence levels:
   - "Confirmed" — found in 2+ authoritative sources
   - "Likely" — found in 1 authoritative source
   - "Uncertain" — found only in community sources or conflicting sources

This citation discipline prevents hallucination — the Research CLI's most dangerous failure mode. When it cannot find information, it must say so rather than generating plausible-sounding but unsourced claims. Other specialists depend on the Research CLI's findings being factual; a hallucinated research result propagated to the Security CLI could cause real vulnerabilities to be missed. A hallucinated library recommendation sent to the Coder CLI could introduce a dependency that does not exist or has critical bugs.

The generation step also handles synthesis across multiple chunks. When three chunks discuss the same topic from different angles — one explaining the API, one discussing performance, one warning about edge cases — the Research CLI weaves them into a single coherent narrative rather than presenting three disconnected excerpts. This synthesis is what justifies using an LLM for generation rather than just returning raw search results.

7.4 Knowledge Graph Construction

Individual research findings are useful. Connected findings are powerful. When the Research CLI discovers that library A depends on library B, that library B has a known CVE, and that CVE affects the authentication flow — the connection between these findings is more valuable than any single finding alone.

A knowledge graph connects research findings into a navigable structure:

// clis/research/src/knowledge/graph.rs
pub struct KnowledgeGraph {
    nodes: HashMap<NodeId, KnowledgeNode>,
    edges: Vec<KnowledgeEdge>,
}

pub struct KnowledgeNode {
    pub id: NodeId,
    pub node_type: NodeType,
    pub label: String,
    pub properties: HashMap<String, String>,
    pub sources: Vec<String>,
    pub confidence: Confidence,
    pub created_at: DateTime<Utc>,
}

pub enum NodeType {
    Library { version: String, ecosystem: String },
    Vulnerability { cve: String, severity: Severity },
    Pattern { category: String },
    Technology { domain: String },
    Decision { rationale: String },
    Concept { definition: String },
}

pub struct KnowledgeEdge {
    pub from: NodeId,
    pub to: NodeId,
    pub relation: Relation,
    pub confidence: f32,
}

pub enum Relation {
    DependsOn,
    AffectedBy,         // Library → CVE
    Implements,         // Library → Pattern
    AlternativeTo,      // Library → Library
    Supersedes,         // Library → Library (newer version)
    RecommendedFor,     // Library → Use Case
    ContradictsClaimIn, // Finding → Finding
}

Building the Graph Incrementally

The knowledge graph is not built in one pass. It grows incrementally as the Research CLI conducts investigations. Each research task adds nodes and edges:

impl KnowledgeGraph {
    pub fn add_research_finding(&mut self, finding: &ResearchFinding) {
        // Create node for the main subject
        let subject_id = self.get_or_create_node(&finding.subject);

        // Create nodes and edges for each related entity
        for relation in &finding.relations {
            let related_id = self.get_or_create_node(&relation.entity);
            self.add_edge(KnowledgeEdge {
                from: subject_id,
                to: related_id,
                relation: relation.relation_type.clone(),
                confidence: finding.confidence_score,
            });
        }

        // Check for contradictions with existing knowledge
        let contradictions = self.find_contradictions(&finding);
        for contradiction in contradictions {
            self.add_edge(KnowledgeEdge {
                from: subject_id,
                to: contradiction.node_id,
                relation: Relation::ContradictsClaimIn,
                confidence: 0.5, // Uncertain until resolved
            });
        }
    }

    fn find_contradictions(&self, finding: &ResearchFinding) -> Vec<Contradiction> {
        // Search for existing nodes that make claims incompatible with this finding
        // Example: existing node says "Library X is maintained" but finding says
        // "Library X has not been updated in 2 years"
        self.nodes.values()
            .filter(|node| {
                node.label == finding.subject.label
                    && self.claims_conflict(&node.properties, &finding.properties)
            })
            .map(|node| Contradiction {
                node_id: node.id,
                existing_claim: node.properties.clone(),
                new_claim: finding.properties.clone(),
            })
            .collect()
    }
}

Querying the Graph

The Orchestrator queries the knowledge graph when making dispatch decisions. Before assigning a task to the Coder CLI, it checks: "Does the knowledge graph contain any information about the libraries we chose?"

impl KnowledgeGraph {
    pub fn query_neighborhood(&self, subject: &str, depth: usize) -> GraphView {
        // BFS from the subject node, collecting all nodes within `depth` hops
        let start = self.find_node(subject);
        let mut visited = HashSet::new();
        let mut queue = VecDeque::new();
        let mut result = GraphView::new();

        if let Some(start_id) = start {
            queue.push_back((start_id, 0));
            while let Some((node_id, current_depth)) = queue.pop_front() {
                if current_depth > depth || visited.contains(&node_id) {
                    continue;
                }
                visited.insert(node_id);

                if let Some(node) = self.nodes.get(&node_id) {
                    result.add_node(node.clone());

                    // Traverse edges
                    for edge in self.edges_from(node_id) {
                        result.add_edge(edge.clone());
                        queue.push_back((edge.to, current_depth + 1));
                    }
                }
            }
        }

        result
    }
}

A typical query: query_neighborhood("rate-limiter-flexible", 2) returns the library node, its dependencies, known vulnerabilities, alternative libraries, and the use cases it is recommended for — all within two hops. The Orchestrator includes this subgraph in the Coder CLI's dispatch prompt, giving the specialist complete context about the library it is about to use.

7.5 Research Memory and Caching

Research is expensive. Each web search costs API tokens. Each document fetch costs time. Each LLM analysis costs both. The Research CLI must cache results aggressively to avoid repeating work.

The Research Cache

// clis/research/src/memory/cache.rs
pub struct ResearchCache {
    entries: HashMap<String, CacheEntry>,
    storage_path: PathBuf,
}

pub struct CacheEntry {
    pub query: String,
    pub result: ResearchResult,
    pub sources: Vec<SourceCredibility>,
    pub created_at: DateTime<Utc>,
    pub expires_at: DateTime<Utc>,
    pub access_count: u32,
    pub confidence: Confidence,
}

impl ResearchCache {
    pub fn lookup(&mut self, query: &str) -> Option<&CacheEntry> {
        let key = self.normalize_query(query);

        if let Some(entry) = self.entries.get_mut(&key) {
            // Check expiry
            if Utc::now() > entry.expires_at {
                self.entries.remove(&key);
                return None;
            }

            entry.access_count += 1;
            return Some(entry);
        }

        // Fuzzy match: check if any cached query is semantically similar
        self.fuzzy_lookup(query)
    }

    fn normalize_query(&self, query: &str) -> String {
        // Lowercase, remove stop words, sort remaining words
        // "What are the best JWT libraries for Node.js?" → "best jwt libraries nodejs"
        let stop_words = ["what", "are", "the", "for", "a", "an", "is", "how", "do"];
        query.to_lowercase()
            .split_whitespace()
            .filter(|w| !stop_words.contains(w))
            .collect::<Vec<_>>()
            .join(" ")
    }
}

Cache expiry times vary by content type:

Content Type Default TTL Rationale
Library comparisons 7 days Ecosystem changes slowly
Version-specific docs 30 days Stable within a version
CVE/vulnerability data 1 day New findings emerge daily
API pricing/limits 3 days Providers change pricing
Best practices/patterns 14 days Evolve slowly
News/announcements 6 hours High-frequency changes

Credibility Scores Over Time

The Research CLI maintains credibility scores for recurring sources. A source that consistently provides accurate, current information gets a higher baseline score. A source that has been contradicted by authoritative sources gets penalized:

pub struct SourceReputation {
    pub domain: String,
    pub accuracy_history: Vec<AccuracyRecord>,
    pub current_score: f32,
}

pub struct AccuracyRecord {
    pub research_id: String,
    pub was_accurate: bool,
    pub verification_method: String,
    pub recorded_at: DateTime<Utc>,
}

impl SourceReputation {
    pub fn update_score(&mut self) {
        let recent: Vec<&AccuracyRecord> = self.accuracy_history.iter()
            .filter(|r| Utc::now() - r.recorded_at < Duration::days(90))
            .collect();

        if recent.is_empty() {
            self.current_score = 0.5; // Reset to neutral
            return;
        }

        let accurate_count = recent.iter().filter(|r| r.was_accurate).count();
        self.current_score = accurate_count as f32 / recent.len() as f32;
    }
}

Topic Maps

The Research CLI builds topic maps — hierarchical categorizations of what it knows about each domain area. Topic maps help the Orchestrator decide whether to dispatch a new research task or rely on cached knowledge:

pub struct TopicMap {
    pub topics: HashMap<String, TopicEntry>,
}

pub struct TopicEntry {
    pub name: String,
    pub subtopics: Vec<String>,
    pub research_count: u32,
    pub last_researched: DateTime<Utc>,
    pub coverage: Coverage,
    pub key_findings: Vec<String>,
}

pub enum Coverage {
    Comprehensive,  // Multiple research sessions, cross-referenced
    Partial,        // One research session, basic findings
    Stale,          // Last researched >30 days ago
    None,           // Topic identified but never researched
}

When the Orchestrator receives a prompt about JWT authentication and the topic map shows jwt_authentication: Coverage::Comprehensive, last_researched: 2 days ago, it can skip the Research task and dispatch directly to the Coder with cached findings. When the topic map shows jwt_authentication: Coverage::Stale, the Orchestrator dispatches a Research task to refresh the findings before the Coder starts work.

This feedback loop — Research CLI builds knowledge, topic map tracks coverage, Orchestrator queries topic map before dispatching — is what transforms the Research CLI from a one-shot search tool into a persistent knowledge engine. Over the course of a multi-day project, the topic map fills in, research cache accumulates, and the knowledge graph densifies. The system gets faster and more informed with each task, because research that was needed in Wave 1 of one prompt is already cached when a similar prompt arrives later.

7.6 The Complete Research CLI CLAUDE.md

# ResearchCLI — Knowledge Engine

## Identity
You are the Research CLI — the knowledge engine of a multi-CLI agent system.
You find, evaluate, cross-reference, and structure information.
You NEVER write production code. You NEVER modify files in src/.
You produce KNOWLEDGE: structured findings, comparisons, and recommendations.

## The Credibility Principle
Every claim you make must be traceable to a source.
- Official documentation > blog posts > Stack Overflow > forums
- Current sources (< 6 months) > dated sources (> 18 months)
- Multiple confirming sources > single source
- NEVER fabricate citations. If you cannot find the answer, say so.

## Research Methodology

### For Factual Questions ("What is X?")
1. Search official documentation first (docs.rs, nodejs.org, etc.)
2. Cross-reference with 2+ authoritative sources
3. Note any version-specific caveats
4. Output: Structured answer with [Source: url] citations

### For Comparison Questions ("Which X should I use?")
1. Identify 3-5 candidates based on popularity, maintenance, and fit
2. For each candidate: check GitHub stars, last commit, open issues, TypeScript support
3. Evaluate: API design, performance benchmarks, community support, edge cases
4. Output: Comparison table + recommendation with confidence level + caveats

### For Investigation Questions ("Why does X happen?")
1. Search for known issues, bug reports, changelog entries
2. Check GitHub issues and discussions for similar reports
3. Read relevant source code if available
4. Output: Root cause analysis with evidence chain

### For Trend Questions ("What's the latest on X?")
1. Search with date filters (last 6 months)
2. Check official announcements, release notes, conference talks
3. Cross-reference community sentiment (Twitter, HN, Reddit)
4. Output: Timeline + current status + trajectory

## Output Format
All research outputs follow this structure:

### Summary (2-3 sentences)
### Key Findings (bulleted, with [Source: url] per finding)
### Comparison Table (if comparing options)
### Recommendation (choice + confidence + rationale + caveats)
### Sources (all URLs used, with credibility assessment)
### Cache Metadata (TTL, topic classification)

## Tool Usage
- Use WebSearch for discovery (what exists, what's current)
- Use WebFetch for deep reading (API docs, spec pages, blog posts)
- Use Bash for local investigation (npm info, pip show, git log)
- Use Read for cached research and local documentation
- Use DocumentRAG for searching large document collections
- Use SourceAnalyzer before citing any non-official source
- Use ComparisonBuilder for structured option analysis

## Quality Gates
- Minimum 2 sources per factual claim
- Flag any source older than 18 months
- Flag any library with <100 GitHub stars or no update in 6 months
- Explicitly state confidence level (High/Medium/Low) for all recommendations
- If you find contradictory information, present BOTH sides

## What You Do NOT Do
- Write production code (that is the Coder's job)
- Make architectural decisions (that is the Planner's job)
- Audit security (that is the Security CLI's job)
- You INFORM decisions. You do not MAKE decisions.

7.7 Research Patterns for Non-Engineering Domains

The Research CLI architecture extends naturally beyond software engineering. The same four components — search tools, RAG pipeline, knowledge graph, and caching — serve any domain where decisions depend on external knowledge.

Legal Research CLI: Searches case law databases, retrieves relevant statutes, builds a knowledge graph of precedent relationships (case A cited case B, case B overturned case C). The SourceAnalyzer evaluates whether a case is still good law (not overruled or superseded). The cache stores legal research by topic and jurisdiction.

Medical Research CLI: Searches PubMed and clinical trial databases, retrieves guidelines from authoritative bodies (WHO, CDC, specialist societies). The knowledge graph connects conditions to treatments to contraindications. The SourceAnalyzer prioritizes systematic reviews and meta-analyses over individual case reports. The credibility scoring is inverted from software: older established evidence (large RCTs from 10 years ago) may be more reliable than recent preliminary studies.

Financial Research CLI: Searches SEC filings, earnings transcripts, and market data. The knowledge graph connects companies to industries to economic indicators. The SourceAnalyzer prioritizes primary sources (10-K filings) over secondary analysis (analyst reports). The cache has aggressive short TTLs — market data becomes stale within hours.

Education Research CLI: Searches curriculum databases, pedagogical research, and adaptive learning literature. The knowledge graph connects learning objectives to prerequisite knowledge to assessment methods. The SourceAnalyzer prioritizes peer-reviewed educational research and institutional guidelines. The RAG pipeline indexes textbooks and course materials, enabling the CLI to answer questions like "what prerequisite concepts does this student need before learning calculus" by traversing the knowledge graph.

In each domain, the Research CLI's core behavior remains the same: search systematically, evaluate credibility, cross-reference claims, cache findings, and produce structured knowledge. The CLAUDE.md template changes to reflect domain-specific methodology, the SourceAnalyzer's authority rankings change, and the knowledge graph's relation types change. But the architecture is identical. This is the power of building the Research CLI as a general-purpose knowledge engine with configurable domain-specific overlays rather than a software-specific tool.

7.8 Testing the Research CLI

Test 1: Source Credibility

Query: "Is Express.js still maintained?"
Expected: Research CLI checks GitHub (last commit date, open issues),
npmjs (weekly downloads), and official site. Does NOT cite a 2020 blog post
as evidence. Reports current maintenance status with dates.

Test 2: Comparison Quality

Query: "Compare Prisma, Drizzle, and TypeORM for a new project"
Expected: Structured comparison with GitHub stats, TypeScript support,
query performance, migration story. Recommendation with confidence
level and caveats. All claims sourced.

Test 3: Cache Hit

Run the same query twice within TTL.
Expected: Second query returns cached result without web searches.
Cache metadata shows hit count = 2.

Test 4: Contradiction Detection

Scenario: Source A says "Library X supports PostgreSQL 15."
          Source B says "Library X only supports PostgreSQL up to 14."
Expected: Research CLI flags the contradiction, reports both claims
with sources, and attempts to resolve by checking official docs.

Test 5: Staleness Warning

Query about a library using only sources from 2023.
Expected: Research CLI flags all sources as SourceRecency::Stale
and explicitly warns that findings may be outdated.

Test 6: No Hallucination

Query: "What is the FooBarBaz library's API for authentication?"
(FooBarBaz does not exist)
Expected: Research CLI reports "not found in available sources."
Does NOT generate a plausible-sounding but fabricated API description.
This is the most critical test — a Research CLI that hallucinates is worse
than no Research CLI at all, because its hallucinations will be treated
as facts by other specialists.

Test 7: Integration with Orchestrator

Scenario: Orchestrator dispatches "Research rate limiting options for Node.js"
Expected: Research CLI returns structured ComparisonBuilder output.
Orchestrator can parse the recommendation and include it in the
Coder CLI's dispatch prompt without manual reformatting.

If all seven tests pass, your Research CLI searches systematically, evaluates credibility, caches results, detects contradictions, warns about staleness, refuses to hallucinate, and integrates cleanly with the Orchestrator's dispatch pipeline.
