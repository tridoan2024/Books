Chapter 13: The Documentation CLI — The Chronicler

Code-Driven Documentation, Architecture Decision Records, Changelog Generation, and the Synchronization Problem

Documentation dies the same way every time. A developer writes a README during initial development, when enthusiasm is high and the API is still shifting. The code evolves — endpoints are added, parameters change, entire modules are rewritten — but the documentation stays frozen at the moment of its creation. Three months later, a new team member follows the README's "Getting Started" guide and encounters environment variables that no longer exist, API endpoints that were renamed, and a setup process that references a database migration script that was deleted in sprint 14.

This is not a quality problem. It is a synchronization problem. The documentation exists in a parallel universe that diverges from reality with every commit. No amount of better writing, more thorough style guides, or dedicated technical writers can solve a synchronization problem. Only automation can — specifically, a system that detects code changes and regenerates the affected documentation automatically.

The Documentation CLI is that system. It is the final specialist in the multi-CLI fleet (the Chronicler of what every other CLI produces), and it has a unique relationship with every other specialist: it documents their work. The Coder writes code; the Documentation CLI explains it. The Planner makes architecture decisions; the Documentation CLI records them as ADRs. The Security CLI finds vulnerabilities; the Documentation CLI catalogs remediation history. The QA CLI discovers edge cases; the Documentation CLI turns them into usage warnings.

This chapter builds the Documentation CLI from four components: the code-to-documentation generation engine, the Architecture Decision Record (ADR) lifecycle system, the changelog generator that turns git history into user-facing release notes, and the FileChanged hooks that trigger automatic documentation updates whenever source code changes.

13.1 Documentation as a Synchronization Engine

The Documentation CLI's architecture rests on three pillars:

Source of truth. The code itself — not a wiki, not a Confluence page, not a developer's memory. The actual source files, the actual route handlers, the actual type definitions, the actual test assertions. When the Documentation CLI generates an API reference, it reads the route definitions from the code, not from a manually maintained OpenAPI spec.

Change detection. The hook system (established in the Orchestrator architecture, Chapter 6) provides FileChanged and PostToolUse events. When the Coder CLI modifies a file, the Documentation CLI is notified with the exact files that changed. This replaces the "remember to update the docs" workflow with an automated trigger. The notification is precise — it includes the specific files that changed, not just "something changed in src/" — so the Documentation CLI can regenerate only the affected documentation artifacts rather than rebuilding everything.

Transformation logic. The Documentation CLI reads changed code and produces documentation artifacts — API references, function documentation, architecture diagrams, changelogs, ADRs. The artifacts are accurate by construction because they are generated from the code that just changed, not from a developer's recollection of what they changed. The transformation is not a template fill-in — it uses the LLM to understand the code's semantics, infer the purpose of functions from their names and structure, and produce explanations that would be useful to a developer encountering the code for the first time.

pub struct DocumentationEngine {
    generators: Vec<Box<dyn DocGenerator>>,
    adr_manager: ADRManager,
    changelog_builder: ChangelogBuilder,
    output_dir: PathBuf,
}

pub trait DocGenerator {
    /// Which file patterns trigger this generator
    fn watches(&self) -> &[&str];

    /// Generate documentation from changed files
    fn generate(&self, changed_files: &[ChangedFile], project_root: &Path) -> Vec<DocArtifact>;
}

pub struct DocArtifact {
    pub path: PathBuf,          // Output file path (docs/api/users.md)
    pub content: String,        // Generated markdown
    pub source_files: Vec<String>, // Which source files this was generated from
    pub artifact_type: DocType,
    pub staleness: Staleness,   // Fresh (just generated) or Stale (needs regeneration)
}

pub enum DocType {
    APIReference,
    FunctionDoc,
    ArchitectureDiagram,
    ADR,
    Changelog,
    README,
    Runbook,
}

The staleness field is critical. Every documentation artifact tracks which source files it was generated from. When any of those source files change, the artifact is marked Stale and queued for regeneration. This is the synchronization mechanism — documentation that is out of date is automatically detected and flagged.

13.2 Code-to-Documentation Generation

The Documentation CLI reads source code and generates human-readable documentation. This is not a simple "extract comments" operation — it uses the LLM to understand code structure, infer purpose from names and patterns, and produce documentation that explains why the code exists, not just what it does.

API Reference Generation

For web applications, the Documentation CLI scans route definitions and generates API references:

pub struct APIDocGenerator;

impl DocGenerator for APIDocGenerator {
    fn watches(&self) -> &[&str] {
        &["src/api/**/*.rs", "src/routes/**/*.rs", "src/api/**/*.ts", "src/routes/**/*.ts"]
    }

    fn generate(&self, changed_files: &[ChangedFile], project_root: &Path) -> Vec<DocArtifact> {
        let mut artifacts = Vec::new();

        for file in changed_files {
            let content = read_file(&file.path);
            let routes = extract_routes(&content, &file.path);

            for route in routes {
                let doc = format!(
                    "## {} {}\n\n\
                     {}\n\n\
                     ### Parameters\n\n{}\n\n\
                     ### Response\n\n{}\n\n\
                     ### Errors\n\n{}\n\n\
                     *Source: `{}:{}`*\n",
                    route.method,
                    route.path,
                    route.description,
                    format_parameters(&route.params),
                    format_response(&route.response_type),
                    format_errors(&route.error_types),
                    file.path,
                    route.line,
                );

                artifacts.push(DocArtifact {
                    path: PathBuf::from(format!(
                        "docs/api/{}.md",
                        route.path.replace('/', "_").trim_matches('_')
                    )),
                    content: doc,
                    source_files: vec![file.path.clone()],
                    artifact_type: DocType::APIReference,
                    staleness: Staleness::Fresh,
                });
            }
        }

        artifacts
    }
}

fn extract_routes(content: &str, file_path: &str) -> Vec<RouteInfo> {
    // Use tree-sitter to find route handler patterns
    // Rust: #[get("/users/{id}")]  or  .route("/users", web::get().to(get_user))
    // TypeScript: app.get('/users/:id', handler)  or  @Get('users/:id')
    // Extract: method, path, parameter types, return type
    parse_route_handlers(content, file_path)
}

The source reference (*Source: \src/api/users.rs:47`*`) at the bottom of each generated doc block is intentional. When a developer reads the documentation and wants to understand the implementation, they can navigate directly to the source. When the source changes, the Documentation CLI knows exactly which doc artifact to regenerate because the source reference creates a bidirectional link.

13.3 Architecture Decision Records — The Living Memory

Architecture Decision Records (ADRs) are the Documentation CLI's most valuable output for long-term project health. An ADR captures why a technical decision was made — not just what was decided, but what alternatives were considered, what constraints drove the choice, and what consequences were expected.

In the multi-CLI system, ADRs are generated by the Planner CLI (Chapter 8) and maintained by the Documentation CLI. The Planner creates the initial ADR when making an architecture decision. The Documentation CLI monitors subsequent code changes and updates the ADR's status when the decision's consequences become visible.

pub struct ADR {
    pub number: usize,
    pub title: String,
    pub status: ADRStatus,
    pub date: String,
    pub context: String,        // Why the decision was needed
    pub decision: String,       // What was decided
    pub alternatives: Vec<Alternative>,
    pub consequences: Vec<String>,
    pub references: Vec<String>,
    pub subsequent_constraints: Vec<String>, // Added as code evolves
}

pub enum ADRStatus {
    Proposed,
    Accepted,
    Deprecated,
    Superseded(usize), // Superseded by ADR number
}

pub struct Alternative {
    pub name: String,
    pub description: String,
    pub rejected_reason: String,
}

impl ADR {
    pub fn to_markdown(&self) -> String {
        format!(
            "# ADR-{:04}: {}\n\n\
             **Status:** {}\n\
             **Date:** {}\n\n\
             ## Context\n\n{}\n\n\
             ## Decision\n\n{}\n\n\
             ## Alternatives Considered\n\n{}\n\n\
             ## Consequences\n\n{}\n\n\
             ## Subsequent Constraints\n\n{}\n\n\
             ## References\n\n{}\n",
            self.number,
            self.title,
            self.status.display(),
            self.date,
            self.context,
            self.decision,
            self.alternatives.iter()
                .map(|a| format!("### {}\n{}\n\n**Rejected because:** {}", a.name, a.description, a.rejected_reason))
                .collect::<Vec<_>>().join("\n\n"),
            self.consequences.iter()
                .map(|c| format!("- {}", c))
                .collect::<Vec<_>>().join("\n"),
            self.subsequent_constraints.iter()
                .map(|c| format!("- {}", c))
                .collect::<Vec<_>>().join("\n"),
            self.references.join("\n"),
        )
    }
}

The subsequent_constraints field is what makes ADRs a living document. When ADR-0003 decides to use JWT for authentication, and six months later the team discovers that JWTs cannot be revoked without a blocklist, the Documentation CLI adds "JWT revocation requires a server-side blocklist — see src/auth/blocklist.rs" to the subsequent constraints. The ADR evolves with the code it describes.

Worked Example: ADR Lifecycle

Month 1: The Planner CLI decides to use PostgreSQL for the user database. The Documentation CLI creates:

ADR-0001: Use PostgreSQL for User Data Storage
Status: Accepted
Date: 2026-01-15

Context: The application needs persistent storage for user accounts,
profiles, and preferences. Expected scale: 100K users in year one.

Decision: Use PostgreSQL 16 with connection pooling via PgBouncer.

Alternatives Considered:
- MongoDB: Rejected because user data is highly relational (profiles
  reference organizations, organizations have hierarchical permissions).
- SQLite: Rejected because concurrent write volume exceeds SQLite's
  single-writer limitation.

Consequences:
- Team must maintain PostgreSQL instances in production
- Schema migrations required for data model changes
- ACID transactions available for financial operations

Month 3: The Coder CLI adds a full-text search feature. The Documentation CLI detects that src/search/engine.rs imports from sqlx and constructs tsvector queries, and adds a subsequent constraint:

Subsequent Constraints:
- Full-text search relies on PostgreSQL's tsvector/tsquery engine.
  Migrating to a different database would require replacing the search
  implementation. See src/search/engine.rs.

Month 8: The team decides to add Elasticsearch for search (ADR-0007). The Documentation CLI updates ADR-0001's status:

Status: Accepted (but search responsibility partially moved to ADR-0007)

Subsequent Constraints:
- Full-text search migrated to Elasticsearch (ADR-0007). PostgreSQL
  tsvector queries in src/search/ are deprecated but not yet removed.

This lifecycle — from initial decision to accrued constraints to partial supersession — is what makes ADRs the institutional memory of a project. Without the Documentation CLI maintaining them, ADRs are written once and never updated, becoming yet another fossil in the documentation graveyard.

ADR Lifecycle Management

pub struct ADRManager {
    adr_dir: PathBuf,  // docs/decisions/
    adrs: Vec<ADR>,
}

impl ADRManager {
    /// When a new architecture decision is made
    pub fn create(&mut self, title: &str, context: &str, decision: &str) -> ADR {
        let number = self.adrs.len() + 1;
        let adr = ADR {
            number,
            title: title.to_string(),
            status: ADRStatus::Proposed,
            date: Utc::now().format("%Y-%m-%d").to_string(),
            context: context.to_string(),
            decision: decision.to_string(),
            alternatives: vec![],
            consequences: vec![],
            references: vec![],
            subsequent_constraints: vec![],
        };

        let path = self.adr_dir.join(format!("adr-{:04}-{}.md", number, slugify(title)));
        write_file(&path, &adr.to_markdown());

        self.adrs.push(adr.clone());
        adr
    }

    /// When a decision is superseded by a new one
    pub fn supersede(&mut self, old_number: usize, new_number: usize) {
        if let Some(adr) = self.adrs.iter_mut().find(|a| a.number == old_number) {
            adr.status = ADRStatus::Superseded(new_number);
            self.rewrite_adr(adr);
        }
    }

    /// When code evolution reveals a new constraint
    pub fn add_constraint(&mut self, number: usize, constraint: &str) {
        if let Some(adr) = self.adrs.iter_mut().find(|a| a.number == number) {
            adr.subsequent_constraints.push(constraint.to_string());
            self.rewrite_adr(adr);
        }
    }
}

13.4 Changelog Generation — Git History to User-Facing Narrative

The changelog generator transforms git commit history into user-facing release notes. This is one of the Documentation CLI's most immediately useful outputs, because every release needs a changelog and nobody wants to write one manually.

pub struct ChangelogBuilder {
    categories: HashMap<String, Vec<ChangelogEntry>>,
}

pub struct ChangelogEntry {
    pub category: ChangeCategory,
    pub description: String,      // User-facing description
    pub commit_hash: String,
    pub pr_number: Option<usize>,
    pub breaking: bool,
}

pub enum ChangeCategory {
    Added,       // New features
    Changed,     // Changes to existing functionality
    Fixed,       // Bug fixes
    Security,    // Security-related changes
    Deprecated,  // Soon-to-be-removed features
    Removed,     // Removed features
    Performance, // Performance improvements
}

impl ChangelogBuilder {
    pub fn build_from_git(&mut self, from_ref: &str, to_ref: &str) -> String {
        let commits = git_log_between(from_ref, to_ref);

        for commit in &commits {
            let category = self.categorize_commit(&commit.message);
            let description = self.user_facing_description(&commit.message, &commit.diff_summary);

            self.categories
                .entry(category.display().to_string())
                .or_default()
                .push(ChangelogEntry {
                    category,
                    description,
                    commit_hash: commit.hash.clone(),
                    pr_number: extract_pr_number(&commit.message),
                    breaking: commit.message.contains("BREAKING") || commit.message.contains("!:"),
                });
        }

        self.render()
    }

    fn categorize_commit(&self, message: &str) -> ChangeCategory {
        let prefix = message.split(':').next().unwrap_or("");
        match prefix.trim() {
            "feat" => ChangeCategory::Added,
            "fix" => ChangeCategory::Fixed,
            "perf" => ChangeCategory::Performance,
            "security" | "sec" => ChangeCategory::Security,
            "deprecate" => ChangeCategory::Deprecated,
            "remove" => ChangeCategory::Removed,
            _ => ChangeCategory::Changed,
        }
    }

    fn user_facing_description(&self, message: &str, diff_summary: &str) -> String {
        // Strip conventional commit prefix and scope
        // "feat(auth): add JWT refresh token support" → "Add JWT refresh token support"
        let desc = message
            .split(':')
            .skip(1)
            .collect::<Vec<_>>()
            .join(":")
            .trim()
            .to_string();

        // Capitalize first letter
        let mut chars = desc.chars();
        match chars.next() {
            None => desc,
            Some(c) => c.to_uppercase().to_string() + chars.as_str(),
        }
    }

    fn render(&self) -> String {
        let mut output = String::new();

        // Breaking changes first
        let breaking: Vec<_> = self.categories.values()
            .flat_map(|entries| entries.iter())
            .filter(|e| e.breaking)
            .collect();

        if !breaking.is_empty() {
            output.push_str("### Breaking Changes\n\n");
            for entry in &breaking {
                output.push_str(&format!("- {} ({})\n", entry.description, entry.commit_hash));
            }
            output.push('\n');
        }

        // Then by category
        let category_order = ["Added", "Changed", "Fixed", "Security", "Performance", "Deprecated", "Removed"];
        for cat_name in &category_order {
            if let Some(entries) = self.categories.get(*cat_name) {
                let non_breaking: Vec<_> = entries.iter().filter(|e| !e.breaking).collect();
                if !non_breaking.is_empty() {
                    output.push_str(&format!("### {}\n\n", cat_name));
                    for entry in &non_breaking {
                        let pr_ref = entry.pr_number
                            .map(|n| format!(" (#{n})"))
                            .unwrap_or_default();
                        output.push_str(&format!("- {}{}\n", entry.description, pr_ref));
                    }
                    output.push('\n');
                }
            }
        }

        output
    }
}

The critical transformation is from developer-facing commit messages to user-facing descriptions. "fix(auth): handle race condition in token refresh when concurrent requests arrive" becomes "Fix authentication token refresh failing under concurrent usage." The Documentation CLI strips implementation details (race condition, concurrent requests) and focuses on what the user observes (token refresh failing).

Breaking Change Detection

Breaking changes deserve special attention because they require action from consumers. The changelog builder detects breaking changes through multiple signals:

fn is_breaking_change(commit: &Commit) -> bool {
    // Conventional commit: feat!: or fix!: (exclamation mark)
    let has_bang = commit.message.split(':').next()
        .map(|prefix| prefix.ends_with('!'))
        .unwrap_or(false);

    // Explicit BREAKING CHANGE footer
    let has_footer = commit.message.contains("BREAKING CHANGE:")
        || commit.message.contains("BREAKING-CHANGE:");

    // API signature changes detected from diff
    let has_api_change = commit.diff_summary.iter().any(|d| {
        d.file.contains("api/") && (
            d.removed_lines.iter().any(|l| l.contains("pub fn") || l.contains("pub async fn"))
        )
    });

    has_bang || has_footer || has_api_change
}

When a breaking change is detected, the changelog entry includes a migration guide — a brief description of what consumers need to change. The Documentation CLI generates this by comparing the old and new function signatures from the git diff:

### Breaking Changes

- **POST /projects now requires `visibility` field** — Previously defaulted to 
  "private" when omitted. Now returns 400 if not provided. Update all API clients 
  to include `"visibility": "private"` in request body. (#142)

This level of detail in a changelog entry — the old behavior, the new behavior, and the specific action required — is what separates useful changelogs from the "various improvements and bug fixes" entries that help nobody.

13.5 FileChanged Hooks — Automatic Documentation Sync

The Documentation CLI registers hooks that trigger documentation regeneration when source files change:

pub struct DocSyncHook {
    engine: Arc<DocumentationEngine>,
}

impl DocSyncHook {
    /// Called by PostToolUse when Coder writes or edits a file
    pub async fn on_file_change(&self, changed_files: &[String]) -> Vec<DocAction> {
        let mut actions = Vec::new();

        for generator in &self.engine.generators {
            let matching_files: Vec<ChangedFile> = changed_files.iter()
                .filter(|f| generator.watches().iter().any(|pattern| glob_match(pattern, f)))
                .map(|f| ChangedFile { path: f.clone() })
                .collect();

            if !matching_files.is_empty() {
                let artifacts = generator.generate(&matching_files, &self.engine.output_dir);

                for artifact in artifacts {
                    actions.push(DocAction::Regenerate {
                        artifact_path: artifact.path.clone(),
                        source_files: artifact.source_files.clone(),
                        content: artifact.content,
                    });
                }
            }
        }

        // Mark stale docs that weren't regenerated
        for artifact in self.engine.all_artifacts() {
            if artifact.source_files.iter().any(|s| changed_files.contains(s))
                && !actions.iter().any(|a| matches!(a, DocAction::Regenerate { artifact_path, .. } if artifact_path == &artifact.path))
            {
                actions.push(DocAction::MarkStale {
                    artifact_path: artifact.path.clone(),
                    reason: "Source file changed but generator did not regenerate".into(),
                });
            }
        }

        actions
    }
}

The hook runs with a time budget (10 seconds, matching the PostToolUse constraint from Chapter 11). If regeneration would take longer — for example, regenerating an entire API reference after a major refactor — the hook queues the work for the end of the task rather than blocking the Coder's workflow.

The distinction between immediate regeneration and queued regeneration is important for developer experience. When the Coder edits a single route handler, the API reference for that endpoint can be regenerated in under 2 seconds — well within the hook budget. When the Coder renames a core type that is referenced in 40 documentation artifacts, regenerating all 40 would take 30-60 seconds. The Documentation CLI detects this by counting affected artifacts: if more than 5 artifacts need regeneration, it marks them all as stale and queues a batch regeneration for the end of the current task. The Coder is never blocked, but the staleness is tracked so nothing is silently outdated.

This queuing pattern mirrors how CI/CD systems handle expensive operations. You do not run the full test suite after every keystroke — you run it after the developer signals "I'm done with this change." Similarly, the Documentation CLI runs quick, targeted regeneration per-file-change and comprehensive, batch regeneration per-task-completion.

13.6 The Documentation CLI CLAUDE.md

# Documentation CLI — The Chronicler

## Identity
You are the Documentation CLI. You transform code into documentation.
You generate, you update, you synchronize. You never write source code.

## Core Principle
Documentation is a synchronization problem, not a content problem.
Every doc artifact tracks its source files. When sources change, docs regenerate.
Stale documentation is worse than no documentation.

## Responsibilities
1. API Reference: Generated from route handlers, type definitions, and schemas
2. Function Documentation: Generated from public function signatures and bodies
3. Architecture Decision Records: Created by Planner, maintained by you
4. Changelogs: Generated from git history using conventional commit parsing
5. README Updates: Regenerated when project structure changes
6. Runbooks: Generated from deployment scripts and configuration

## Triggers
- FileChanged hook: source file modified → regenerate affected docs
- PostToolUse hook: Coder writes file → check if docs need updating
- Orchestrator dispatch: explicit documentation task
- Release event: generate changelog for version range

## Output Rules
- Write to docs/ directory only. Never modify source code.
- Include source references (file:line) in all generated documentation.
- User-facing language: explain purpose, not implementation details.
- Mark stale artifacts explicitly rather than leaving them silently outdated.

## Interaction with Other CLIs
- Planner → sends ADR drafts, you format and maintain them
- Coder → changes trigger your hooks, you regenerate affected docs
- Security → findings feed into security documentation and runbooks
- QA → edge case discoveries become usage warnings in API docs

13.7 Non-Engineering Domains

The Documentation CLI's synchronization pattern applies wherever artifacts need to stay current with their source material.

Policy Documentation: Regulatory policies change quarterly. The Documentation CLI tracks policy source documents and regenerates compliance checklists, training materials, and audit procedures when the source policies are updated. The ADR pattern tracks policy decisions — why a particular compliance approach was chosen, what alternatives were rejected, and what constraints emerged during implementation.

Clinical Protocol Documentation: Treatment protocols reference drug databases, clinical guidelines, and formulary lists that update frequently. The Documentation CLI detects when referenced source materials change and flags protocol documentation for review. The changelog pattern tracks protocol revisions with the same structure as software changelogs: what changed, why, and what the clinical impact is. The ADR pattern records why a specific treatment approach was chosen over alternatives — critical information when a protocol is later questioned during a morbidity review.

Legal Contract Templates: Contract templates reference jurisdiction-specific regulations that change annually. The Documentation CLI tracks which regulations each template references and flags templates for legal review when referenced regulations are updated. The staleness tracking ensures no contract template references outdated regulatory language. The breaking change detection pattern applies directly: when a regulation changes in a way that affects existing contracts (a "breaking change" in the legal domain), the changelog entry includes a migration guide specifying which contracts need amendment.
