Chapter 11: The Validator CLI — The Gatekeeper

Five-Stage Pipeline, Quality Gates, Auto-Retry, and the Difference Between Testing and Verification

Every CI/CD pipeline you have ever used is a Validator CLI. Jenkins, GitHub Actions, Azure Pipelines, CircleCI — they all answer the same question: Is this code ready to ship? They check syntax, run tests, verify types, measure coverage, and produce a binary verdict: green or red. But they are dumb. They run the same pipeline regardless of what changed. They report raw error output that humans must parse. They do not categorize failures, suggest fixes, or retry after correction. They are the gatekeeper with a clipboard, not a gatekeeper with a brain.

The Validator CLI is what happens when you give the gatekeeper a brain. It runs the same five-stage pipeline (lint, type-check, build, test, performance), but it understands what failed, why it failed, which specialist should fix it, and whether a retry is worth attempting. When the Coder CLI writes code, the Validator CLI verifies it — not in the same context window where confirmation bias contaminates the judgment, but in a separate process with a separate system prompt that knows nothing about the Coder's intentions and cares only about what the tools report.

This separation is not a nice-to-have. Claude Code's own source code encodes it behind the VERIFICATION_AGENT feature flag: when the main agent completes three or more tasks without a verification step, the system injects a nudge forcing a separate verifier to evaluate the work. Anthropic's engineers discovered what every QA team already knows — the person who writes the code should not be the person who signs off on it.

This chapter builds the Validator CLI from four components: the five-stage validation pipeline with structured output parsing, the quality gate configuration system with configurable thresholds, the failure analysis engine that categorizes and routes failures to the right specialist, and the PostToolUse hooks that provide continuous validation during development rather than a single gate at the end.

11.1 Why Verification Needs Its Own Context Window

When Claude Code writes a function, it holds a mental model of that function's purpose: the inputs it expects, the transformations it applies, the output it produces. If you ask it to verify the function in the same turn, it verifies against its own mental model — not against the type system's constraints, not against the test suite's expectations, not against the edge cases the implementation never considered. This is not a Claude-specific flaw. It is a fundamental property of autoregressive language models: tokens generated later in a sequence are conditioned on tokens generated earlier. The verification is contaminated by the generation because they share context.

The Validator CLI operates in a different process with a different context window. It receives a list of changed files from the Orchestrator and runs them through an automated pipeline. It does not know what the Coder intended. It does not know which approach the Coder considered and rejected. It knows only what the linter reports, what the type checker reports, what the test runner reports, and what the performance benchmarks measure. This is its strength: judgment uncontaminated by implementation decisions.

There is a deeper reason for the separation. Generation and validation are fundamentally different cognitive modes. Generation is divergent — explore the solution space, consider alternatives, make creative leaps. Validation is convergent — apply known rules, check known constraints, reduce uncertainty to a binary outcome. Mixing these modes in a single context window creates interference. The model either becomes too conservative (validating while generating, producing timid code that avoids anything the validator might flag) or too permissive (generating while validating, rubber-stamping its own output because the generation context makes the code feel "obviously correct").

The VERIFICATION_AGENT feature flag in Claude Code's source (tools/TaskUpdateTool/TaskUpdateTool.ts) encodes this insight architecturally. When the main agent completes three or more tasks and none of them was a verification step, the system refuses to let the agent self-certify:

// Structural verification nudge
// Adapted from Claude Code's VERIFICATION_AGENT feature
pub fn check_verification_needed(
    tasks: &[Task],
    feature_flags: &FeatureFlags,
) -> Option<String> {
    if !feature_flags.verification_agent {
        return None;
    }

    let completed = tasks.iter()
        .filter(|t| t.status == TaskStatus::Completed)
        .collect::<Vec<_>>();

    if completed.len() < 3 {
        return None;
    }

    let has_verification = completed.iter()
        .any(|t| t.subject.to_lowercase().contains("verif"));

    if has_verification {
        return None;
    }

    Some(
        "You completed 3+ tasks without verification. \
         Spawn the Validator CLI before writing your summary. \
         You cannot self-assign PARTIAL — only the Validator issues a verdict."
        .to_string()
    )
}

The key insight: "You cannot self-assign PARTIAL by listing caveats in your summary — only the verifier issues a verdict." This is not a suggestion. It is a structural constraint. The main agent literally cannot declare its own work complete without external verification. The Validator CLI formalizes this constraint as a first-class member of the multi-CLI fleet.

11.2 The Five-Stage Validation Pipeline

The Validator runs five stages in sequence. Each stage gates the next — if lint fails, the pipeline does not waste time running the build. If the build fails, tests do not run. This fail-fast design minimizes wasted computation and provides the fastest possible feedback.

Stage 1: Lint        → Static analysis, code style, dead code
Stage 2: Type-Check  → Type correctness, interface compliance
Stage 3: Build       → Compilation, bundling, artifact generation
Stage 4: Test        → Unit tests, integration tests, coverage
Stage 5: Performance → Bundle size, benchmark regression, runtime profiling

Each stage produces a structured result — not raw console output, but a parsed data structure that the Validator evaluates against configured thresholds:

pub struct StageResult {
    pub stage: ValidationStage,
    pub status: StageStatus,       // Pass, Warn, Fail
    pub duration_ms: u64,
    pub error_count: usize,
    pub warning_count: usize,
    pub metrics: HashMap<String, f64>,
    pub errors: Vec<StructuredError>,
    pub suggestions: Vec<FixSuggestion>,
}

pub enum ValidationStage {
    Lint,
    TypeCheck,
    Build,
    Test,
    Performance,
}

pub enum StageStatus {
    Pass,
    Warn,
    Fail,
    Skipped,
}

pub struct StructuredError {
    pub file: String,
    pub line: usize,
    pub column: usize,
    pub code: String,           // ESLint rule, TS error code, etc.
    pub message: String,
    pub severity: ErrorSeverity,
    pub auto_fixable: bool,
    pub fix_command: Option<String>,  // "eslint --fix", "ruff --fix"
}

Stage 1: Lint

Linting catches the problems that do not need a compiler to detect. Unused variables, unreachable code, missing imports, formatting violations. These are cheap to find, cheap to fix, and expensive to ignore — because they compound. A codebase that tolerates lint warnings today tolerates security vulnerabilities tomorrow. The Validator runs the project's configured linter and parses the structured output:

pub async fn run_lint(&self, project_root: &Path) -> StageResult {
    let config = &self.quality_gates.lint;
    let start = Instant::now();

    // Detect language and run appropriate linter
    let output = if project_root.join("package.json").exists() {
        Command::new("npx")
            .args(["eslint", ".", "--format", "json", "--max-warnings", "0"])
            .current_dir(project_root)
            .output().await?
    } else if project_root.join("pyproject.toml").exists() {
        Command::new("ruff")
            .args(["check", ".", "--output-format", "json", "--exit-zero"])
            .current_dir(project_root)
            .output().await?
    } else if project_root.join("Cargo.toml").exists() {
        Command::new("cargo")
            .args(["clippy", "--message-format=json", "--", "-D", "warnings"])
            .current_dir(project_root)
            .output().await?
    } else {
        return StageResult::skipped(ValidationStage::Lint);
    };

    let errors = parse_lint_output(&output.stdout, &output.stderr);
    let error_count = errors.iter().filter(|e| e.severity == ErrorSeverity::Error).count();
    let warning_count = errors.iter().filter(|e| e.severity == ErrorSeverity::Warning).count();

    let status = if error_count > config.max_errors {
        StageStatus::Fail
    } else if warning_count > config.warn_threshold {
        if warning_count > config.fail_threshold {
            StageStatus::Fail
        } else {
            StageStatus::Warn
        }
    } else {
        StageStatus::Pass
    };

    StageResult {
        stage: ValidationStage::Lint,
        status,
        duration_ms: start.elapsed().as_millis() as u64,
        error_count,
        warning_count,
        metrics: HashMap::from([
            ("errors".into(), error_count as f64),
            ("warnings".into(), warning_count as f64),
        ]),
        errors,
        suggestions: generate_lint_suggestions(&errors),
    }
}

The --format json (ESLint), --output-format json (Ruff), and --message-format=json (Clippy) flags are essential. Human-readable lint output is fine for humans. The Validator is not human. It needs structured data it can parse, categorize, and route. JSON output means the Validator can distinguish between a formatting warning (auto-fixable, route to eslint --fix) and a security lint error (no-eval, route to Security CLI for deeper analysis).

Stage 2: Type-Check

Linting catches style violations. Type-checking catches broken contracts. When the Coder declares a function returns Result<User, Error> but the implementation can return a state where the User struct is partially initialized, the type checker catches it. When a trait changes in module A but module B still implements the old trait signature, the type checker catches it.

pub async fn run_type_check(&self, project_root: &Path) -> StageResult {
    let start = Instant::now();

    let output = if project_root.join("tsconfig.json").exists() {
        Command::new("npx")
            .args(["tsc", "--noEmit", "--pretty", "false"])
            .current_dir(project_root)
            .output().await?
    } else if project_root.join("Cargo.toml").exists() {
        // Rust type checking is part of compilation
        Command::new("cargo")
            .args(["check", "--message-format=json"])
            .current_dir(project_root)
            .output().await?
    } else if project_root.join("pyproject.toml").exists() {
        Command::new("pyright")
            .args(["--outputjson"])
            .current_dir(project_root)
            .output().await?
    } else {
        return StageResult::skipped(ValidationStage::TypeCheck);
    };

    let errors = parse_type_errors(&output.stdout, &output.stderr);
    let error_count = errors.len();

    StageResult {
        stage: ValidationStage::TypeCheck,
        status: if error_count > 0 { StageStatus::Fail } else { StageStatus::Pass },
        duration_ms: start.elapsed().as_millis() as u64,
        error_count,
        warning_count: 0,
        metrics: HashMap::from([("type_errors".into(), error_count as f64)]),
        errors,
        suggestions: generate_type_fix_suggestions(&errors),
    }
}

The --pretty false flag (TypeScript) disables ANSI color codes and multi-line formatting that breaks parsing. Every tool in the pipeline follows the same principle: machine-readable output first, human-readable presentation later. The Validator's job is to produce structured data for the Orchestrator, not colorful terminal output for a developer.

Stage 3: Build

Types check. Lint passes. But does the code compile into a runnable artifact? The build stage catches problems invisible to the first two stages: circular dependencies that manifest only at bundle time, missing assets referenced in import statements, environment variables assumed but not provided, and configuration errors that surface only during compilation.

The Validator tracks not just build success but build duration. A build that took 30 seconds last week and takes 90 seconds today signals a problem — an unnecessary dependency, an unoptimized import, a missing tree-shaking configuration. Build time is a metric, not just a pass/fail boolean.

Stage 4: Test

This is where the Validator earns its keep. Lint checks syntax. Types check contracts. Builds check compilation. Tests check behavior. The Validator runs the project's test suite with coverage tracking:

pub async fn run_tests(&self, project_root: &Path) -> StageResult {
    let config = &self.quality_gates.test;
    let start = Instant::now();

    let output = if project_root.join("package.json").exists() {
        Command::new("npx")
            .args([
                "jest", "--coverage",
                "--coverageReporters=json-summary",
                "--json", "--outputFile=test-results.json"
            ])
            .current_dir(project_root)
            .output().await?
    } else if project_root.join("Cargo.toml").exists() {
        Command::new("cargo")
            .args(["test", "--", "--format=json", "-Z", "unstable-options"])
            .current_dir(project_root)
            .output().await?
    } else if project_root.join("pyproject.toml").exists() {
        Command::new("python3")
            .args([
                "-m", "pytest",
                "--json-report", "--json-report-file=test-results.json",
                "--cov=src", "--cov-report=json:coverage.json",
                "-q"
            ])
            .current_dir(project_root)
            .output().await?
    } else {
        return StageResult::skipped(ValidationStage::Test);
    };

    let test_results = parse_test_results(project_root)?;
    let coverage = parse_coverage(project_root)?;

    // Check pass rate
    let pass_rate = if test_results.total > 0 {
        (test_results.passed as f64 / test_results.total as f64) * 100.0
    } else {
        0.0
    };

    // Check coverage regression
    let coverage_regression = if let Some(baseline) = self.load_baseline_coverage() {
        baseline - coverage.lines
    } else {
        0.0
    };

    let status = if test_results.failed > 0 {
        StageStatus::Fail
    } else if coverage.lines < config.coverage_fail_threshold {
        StageStatus::Fail
    } else if coverage_regression > config.regression_threshold {
        StageStatus::Fail
    } else if coverage.lines < config.coverage_warn_threshold {
        StageStatus::Warn
    } else {
        StageStatus::Pass
    };

    StageResult {
        stage: ValidationStage::Test,
        status,
        duration_ms: start.elapsed().as_millis() as u64,
        error_count: test_results.failed,
        warning_count: 0,
        metrics: HashMap::from([
            ("passed".into(), test_results.passed as f64),
            ("failed".into(), test_results.failed as f64),
            ("total".into(), test_results.total as f64),
            ("coverage_lines".into(), coverage.lines),
            ("coverage_branches".into(), coverage.branches),
            ("coverage_regression".into(), coverage_regression),
        ]),
        errors: test_results.failures.iter().map(|f| StructuredError {
            file: f.file.clone(),
            line: f.line,
            column: 0,
            code: f.test_name.clone(),
            message: f.failure_message.clone(),
            severity: ErrorSeverity::Error,
            auto_fixable: false,
            fix_command: None,
        }).collect(),
        suggestions: vec![],
    }
}

The Validator does not just check pass/fail. It tracks coverage regression. If yesterday's coverage was 87% and today's is 84%, something is wrong — either new code was added without tests, or existing tests were removed. Both demand attention. The Validator maintains a coverage baseline and flags any regression beyond the configured threshold.

Stage 5: Performance

The final stage catches the category of problems that the first four stages miss entirely: performance regressions. Code that is correct, well-typed, well-tested, and properly linted can still be catastrophically slow. A database query that returns in 50ms with 100 rows but takes 30 seconds with 10,000 rows. A React component that re-renders 47 times per keystroke. A binary that grew from 10MB to 50MB because someone pulled in an unnecessary dependency that brought in half the crate ecosystem.

The performance stage compares current metrics against stored baselines: bundle size, benchmark execution times, and optionally Lighthouse scores for web projects. A 2% regression in one benchmark might be acceptable if it comes with a 15% improvement in another. The Validator applies configurable thresholds and treats performance as a spectrum — pass, warn, or fail — rather than a binary gate.

11.3 Quality Gates — The Configuration Layer

The five stages are fixed. The thresholds are not. Every team, every project, every phase of development has different quality requirements. A greenfield prototype might accept 50% test coverage. A financial services application demands 95%. The Validator uses a configuration file that defines quality gates with three levels:

#[derive(Deserialize)]
pub struct QualityGates {
    pub lint: LintGate,
    pub type_check: TypeCheckGate,
    pub build: BuildGate,
    pub test: TestGate,
    pub performance: PerformanceGate,
    pub gate_policy: GatePolicy,
}

#[derive(Deserialize)]
pub struct LintGate {
    pub enabled: bool,
    pub max_errors: usize,        // Any error count above this = fail
    pub warn_threshold: usize,    // Warnings above this = warn
    pub fail_threshold: usize,    // Warnings above this = fail
    pub auto_fix: bool,           // Attempt auto-fix before failing
}

#[derive(Deserialize)]
pub struct TestGate {
    pub enabled: bool,
    pub min_pass_rate: f64,          // 100.0 = all tests must pass
    pub coverage_warn_threshold: f64, // < this = warn (e.g., 80.0)
    pub coverage_fail_threshold: f64, // < this = fail (e.g., 70.0)
    pub regression_threshold: f64,    // Coverage drop > this = fail (e.g., 2.0)
    pub timeout_per_test_ms: u64,
}

#[derive(Deserialize)]
pub struct PerformanceGate {
    pub enabled: bool,
    pub bundle_warn_growth_pct: f64,   // > this% growth = warn
    pub bundle_fail_growth_pct: f64,   // > this% growth = fail
    pub benchmark_warn_regression_pct: f64,
    pub benchmark_fail_regression_pct: f64,
}

#[derive(Deserialize)]
pub struct GatePolicy {
    pub fail_action: FailAction,       // Halt pipeline on failure
    pub max_warnings_before_fail: usize, // Cumulative warnings across stages
    pub max_retries: usize,            // Auto-retry attempts before hard fail
}

This configuration is the Validator's contract with the team. It lives in source control. Changes to thresholds require a PR review. No one — not the Coder, not the Orchestrator, not a hurried engineer at 2 AM — can quietly relax a threshold without it appearing in the commit history.

11.4 Failure Analysis and Auto-Retry

When the pipeline fails, the Validator's job is not just to report the failure. It is to analyze the failure, categorize it, and route it to the right specialist. A syntax error goes back to the Coder. A security lint violation routes to the Security CLI for deeper analysis. A performance regression goes back to the Coder with performance-specific guidance. The Validator is a triage nurse, not just a thermometer.

pub enum FailureCategory {
    SyntaxError,          // Parse errors, invalid syntax
    TypeError,            // Type mismatches, missing properties
    BuildError,           // Compilation failures, missing deps
    TestFailure,          // Assertion failures, test errors
    CoverageRegression,   // Coverage dropped below threshold
    PerfRegression,       // Performance degraded beyond threshold
}

pub struct CategorizedFailure {
    pub category: FailureCategory,
    pub stage: ValidationStage,
    pub file: String,
    pub line: Option<usize>,
    pub message: String,
    pub auto_fixable: bool,
    pub fix_suggestion: String,
    pub route_to: SpecialistCLI,
}

pub enum SpecialistCLI {
    Coder,
    Security,
    QA,
}

pub struct FailureAnalyzer;

impl FailureAnalyzer {
    pub fn categorize(&self, result: &StageResult) -> Vec<CategorizedFailure> {
        result.errors.iter().map(|error| {
            let (category, route_to) = match result.stage {
                ValidationStage::Lint => {
                    if is_security_lint_rule(&error.code) {
                        (FailureCategory::SyntaxError, SpecialistCLI::Security)
                    } else {
                        (FailureCategory::SyntaxError, SpecialistCLI::Coder)
                    }
                }
                ValidationStage::TypeCheck => {
                    (FailureCategory::TypeError, SpecialistCLI::Coder)
                }
                ValidationStage::Build => {
                    (FailureCategory::BuildError, SpecialistCLI::Coder)
                }
                ValidationStage::Test => {
                    (FailureCategory::TestFailure, SpecialistCLI::Coder)
                }
                ValidationStage::Performance => {
                    (FailureCategory::PerfRegression, SpecialistCLI::Coder)
                }
            };

            CategorizedFailure {
                category,
                stage: result.stage.clone(),
                file: error.file.clone(),
                line: Some(error.line),
                message: error.message.clone(),
                auto_fixable: error.auto_fixable,
                fix_suggestion: self.suggest_fix(error),
                route_to,
            }
        }).collect()
    }

    fn suggest_fix(&self, error: &StructuredError) -> String {
        match error.code.as_str() {
            c if c.starts_with("TS2345") => {
                "Argument type mismatch. Check function signature and ensure \
                 the passed argument matches the expected parameter type.".into()
            }
            c if c.starts_with("TS7006") => {
                "Add explicit type annotation to the parameter.".into()
            }
            c if c.starts_with("no-unused-vars") || c.starts_with("F841") => {
                "Remove unused variable or prefix with underscore.".into()
            }
            _ => format!("Review error {} and fix the underlying issue.", error.code),
        }
    }

    pub fn generate_fix_request(
        &self,
        failures: &[CategorizedFailure],
    ) -> FixRequest {
        let mut routes: HashMap<SpecialistCLI, Vec<FixItem>> = HashMap::new();

        for failure in failures {
            routes.entry(failure.route_to.clone())
                .or_default()
                .push(FixItem {
                    file: failure.file.clone(),
                    line: failure.line,
                    category: failure.category.clone(),
                    message: failure.message.clone(),
                    suggestion: failure.fix_suggestion.clone(),
                    auto_fixable: failure.auto_fixable,
                });
        }

        FixRequest {
            total_failures: failures.len(),
            auto_fixable_count: failures.iter().filter(|f| f.auto_fixable).count(),
            routes,
            retry_after_fix: true,
        }
    }
}

The Auto-Retry Loop

When the Validator detects failures, it does not just report them. It initiates a retry loop. The fix request goes back to the Coder CLI (via the Orchestrator). The Coder applies fixes. The Validator runs again. This continues up to the configured maximum retries.

pub struct RetryLoop {
    max_retries: usize,
    history: Vec<RetryAttempt>,
}

pub struct RetryAttempt {
    pub attempt: usize,
    pub verdict: Verdict,
    pub failed_stages: Vec<ValidationStage>,
    pub timestamp: String,
}

impl RetryLoop {
    pub async fn run(
        &mut self,
        pipeline: &ValidationPipeline,
        orchestrator: &OrchestratorChannel,
    ) -> FinalVerdict {
        for attempt in 1..=self.max_retries {
            let result = pipeline.run_all_stages().await;

            self.history.push(RetryAttempt {
                attempt,
                verdict: result.verdict.clone(),
                failed_stages: result.failed_stages(),
                timestamp: Utc::now().to_rfc3339(),
            });

            match result.verdict {
                Verdict::Pass => return FinalVerdict {
                    verdict: Verdict::Pass,
                    attempts: attempt,
                    history: self.history.clone(),
                },
                Verdict::Warn if attempt > 1 => return FinalVerdict {
                    verdict: Verdict::Warn,
                    attempts: attempt,
                    history: self.history.clone(),
                },
                Verdict::Fail if attempt == self.max_retries => {
                    return FinalVerdict {
                        verdict: Verdict::Fail,
                        attempts: attempt,
                        history: self.history.clone(),
                    };
                }
                _ => {
                    // Categorize failures and route fix request
                    let analyzer = FailureAnalyzer;
                    let mut all_failures = Vec::new();
                    for stage_result in &result.stage_results {
                        if stage_result.status == StageStatus::Fail {
                            all_failures.extend(analyzer.categorize(stage_result));
                        }
                    }

                    let fix_request = analyzer.generate_fix_request(&all_failures);
                    orchestrator.send_fix_request(fix_request).await;
                }
            }
        }

        FinalVerdict {
            verdict: Verdict::Fail,
            attempts: self.max_retries,
            history: self.history.clone(),
        }
    }
}

Many validation failures are auto-fixable. Missing semicolons, unused imports, formatting violations. The Validator categorizes these, sends specific fix instructions to the Coder, and tries again. Only after exhausting retries does it declare a hard failure. The retry history feeds back to the Orchestrator — "PASS after 3 attempts" tells the Orchestrator this was a difficult change, while "PASS on first attempt" confirms the Coder is producing clean output.

There is a subtle but important constraint on the retry loop: the Validator must detect oscillation. If attempt 1 fails on type errors, the Coder fixes those types but introduces lint errors, and attempt 2 fails on lint, and the Coder fixes the lint but reintroduces the type errors — the system is oscillating, not converging. The Validator tracks the set of failing stages and error codes across attempts. If the same error reappears after being "fixed," the loop terminates with a diagnostic: "Oscillation detected — errors in Stage 2 recurred after being fixed in attempt 1." This prevents infinite fix cycles that waste API tokens without making progress.

The Orchestrator also uses retry history for longer-term optimization. If a particular type of change (say, API endpoint additions) consistently requires 2-3 validation retries, the Orchestrator can learn to give the Coder more specific instructions upfront — or to dispatch a different specialist for that type of work. The validation retry count is a signal about the Coder's reliability on different task types.

11.5 PostToolUse Hooks — Continuous Validation

Running validation after the Coder finishes is good. Running validation after every file write is better. Claude Code's hook system provides PostToolUse events — when the Coder writes a file, a hook fires. The Validator catches it, runs a targeted validation on the changed file, and flags problems before the Coder moves on to the next task.

This is the difference between finding fifty errors at the end of a coding session and finding one error after each file write. The latter is cheaper to fix, less disruptive to flow, and prevents cascading failures where an early mistake propagates through subsequent files.

pub struct ContinuousValidator {
    lint_budget_ms: u64,      // Max time for incremental lint (default: 2000)
    type_budget_ms: u64,      // Max time for incremental type check (default: 5000)
    enabled: bool,
}

impl ContinuousValidator {
    /// Called by PostToolUse hook when a file is written or edited
    pub async fn on_file_change(&self, file_path: &Path) -> Option<QuickResult> {
        if !self.enabled {
            return None;
        }

        // Skip non-source files
        let ext = file_path.extension()?.to_str()?;
        if !["rs", "ts", "tsx", "js", "jsx", "py"].contains(&ext) {
            return None;
        }

        // Skip test files (QA CLI's domain)
        if file_path.to_string_lossy().contains(".test.")
            || file_path.to_string_lossy().contains("_test.")
            || file_path.to_string_lossy().contains("/tests/") {
            return None;
        }

        let mut issues = Vec::new();

        // Quick lint (budget: 2s)
        if let Ok(lint_result) = timeout(
            Duration::from_millis(self.lint_budget_ms),
            run_quick_lint(file_path),
        ).await {
            issues.extend(lint_result.issues);
        }

        // Quick type check (budget: 5s, only if lint passed)
        if issues.is_empty() {
            if let Ok(type_result) = timeout(
                Duration::from_millis(self.type_budget_ms),
                run_quick_type_check(file_path),
            ).await {
                issues.extend(type_result.issues);
            }
        }

        if issues.is_empty() {
            None
        } else {
            Some(QuickResult { file: file_path.to_owned(), issues })
        }
    }
}

The performance budget is critical. A PostToolUse hook that takes 30 seconds defeats the purpose — the Coder is blocked, waiting, the flow is broken. The rule: hooks get a 10-second total budget. Lint a single file takes 1-2 seconds. Incremental type check takes 3-5 seconds. Both fit within budget. Running the full test suite (30-120 seconds) or a build (10-60 seconds) does not. The full five-stage pipeline runs when the Coder signals "task complete." Hooks provide fast, incremental feedback during development. Both mechanisms are necessary. Neither is sufficient alone.

11.6 The Validator CLI CLAUDE.md

# Validator CLI — The Gatekeeper

## Identity
You are the Validator CLI in a multi-CLI AGI system. You produce verdicts:
PASS, WARN, or FAIL. You do not write code. You do not suggest features.
You do not negotiate thresholds.

## Core Protocol

### Five-Stage Pipeline
Execute these stages IN ORDER. Stop on first FAIL.
1. Lint — Run the project's linter with JSON output
2. Type-Check — Run type checker with machine-readable output
3. Build — Compile and track duration + artifact size
4. Test — Run test suite with coverage tracking
5. Performance — Compare metrics against stored baselines

### Quality Gates
Read thresholds from `.validator/quality-gates.toml`.
NEVER override thresholds. NEVER accept arguments to relax them.
The configuration file is the sole source of truth.

### Verdict Format
Always respond with structured JSON:
{
  "verdict": "PASS | WARN | FAIL",
  "stages": { ... per-stage results ... },
  "attempt": 1,
  "timestamp": "ISO-8601"
}

## Failure Handling
When a stage FAILS:
1. Parse error output into structured failures
2. Categorize: syntax | type | build | test | coverage | performance
3. Generate fix suggestions for each failure
4. Route fix request to the appropriate specialist CLI
5. Wait for fixes, then re-run (max 3 retries)

## Rules
1. Never self-certify. You validate what others produce.
2. Never modify source code. You only run tools against it.
3. Never relax thresholds unless the config file changes via PR.
4. Always run all enabled stages. No shortcuts, no skipping.
5. Report structured output. The Orchestrator parses your verdicts.
6. Be fast. Incremental checks in hooks. Full pipeline on task completion.

## Interaction with Other CLIs
- Orchestrator → sends changed files, expects a verdict
- Coder → you send fix requests, Coder sends back fixes
- Security → you flag security lint rules, Security does deep analysis
- QA → you run existing tests, QA writes new tests for uncovered paths

11.7 The Validator in Non-Engineering Domains

The five-stage validation pattern generalizes beyond code. Any domain that produces artifacts requiring quality assurance benefits from a dedicated validation step with structured gates.

Publishing Pipeline: A book manuscript passes through five stages: grammar check (lint), style guide compliance (type-check), rendering to PDF/EPUB (build), reader comprehension tests via focus group feedback (test), and page count / reading time budgets (performance). Each stage gates the next — a manuscript with grammar errors does not proceed to layout. The quality gates are configurable: a technical manual demands different style compliance than a novel.

Data Pipeline Validator: Data engineering teams validate ETL outputs: schema validation (lint), referential integrity (type-check), pipeline execution (build), data quality tests — row counts, null rates, distribution drift (test), and query performance against SLA budgets (performance). A data pipeline where today's row count is 50% below yesterday's baseline triggers the same kind of regression alert as a coverage drop.

Legal Document Review: Contract templates pass through clause completeness checks (lint), regulatory compliance verification (type-check), document generation from template (build), sample scenario testing against edge cases (test), and review turnaround time budgets (performance). The quality gate configuration specifies which regulatory frameworks apply per jurisdiction.

In every domain, the pattern is the same: ordered stages, configurable thresholds, structured output, failure categorization, and auto-retry with routed fix requests. The five stages map to: syntax correctness (lint), semantic correctness (type-check), artifact generation (build), behavioral verification (test), and resource budget compliance (performance).

The Validator pattern also maps directly to CI/CD integration. The five-stage pipeline translates one-to-one into GitHub Actions steps, Azure DevOps stages, or Jenkins pipeline blocks. Each stage produces structured output that CI platforms can parse into check annotations, PR comments, and dashboard metrics. The Validator's structured verdict JSON can be posted as a PR comment, letting human reviewers see the full quality picture at a glance:

Stage Status Details
Lint PASS 0 errors, 3 warnings
Type-Check PASS 0 errors
Build PASS 14.2s (baseline: 12.8s, +10.9%)
Test PASS 147/147 passed, 87.3% coverage
Performance WARN Bundle grew 4.2% (threshold: 5%)
Verdict PASS 2 attempts, 1 auto-fix applied

This table is not a mockup — it is the exact output the Validator produces when integrated into a CI pipeline. The Orchestrator can query the CI system's API to retrieve this verdict programmatically, closing the loop between the multi-CLI system and the team's existing development workflow.

11.8 Validator vs. Security CLI — Complementary, Not Overlapping

A common question when designing the multi-CLI fleet: why do we need both a Validator and a Security CLI? Both check code. Both produce structured findings. Both gate shipment.

The distinction is scope and depth. The Validator catches syntactic and structural problems: lint violations, type errors, test failures, coverage regressions, performance degradation. These are the problems that standard development tools detect. The Security CLI catches semantic security vulnerabilities: injection patterns, hardcoded secrets, dependency CVEs, OWASP Top 10 violations, STRIDE threats. These require specialized analysis tools and domain expertise that the general validation pipeline does not provide.

The interaction between them is precise:

  • The Validator runs the project's standard lint rules. If a lint rule happens to be security-related (like no-eval in ESLint or S301 in Ruff's Bandit integration), the Validator flags it and routes it to the Security CLI for deeper analysis rather than treating it as a simple lint error.
  • The Security CLI's findings do not pass through the Validator's pipeline. They have their own severity calibration and reporting format (Chapter 10). The Orchestrator consumes both outputs independently.
  • The Validator can be configured to require a Security CLI PASS as a sixth stage (a "security gate"), but this is optional and configured per project. Not every commit needs a full six-pass security audit — but every release candidate does.

The Validator is fast, broad, and automated. The Security CLI is deep, focused, and augmented by LLM reasoning. Together they provide both the breadth of standard quality checking and the depth of security-specific analysis.

11.9 Testing the Validator CLI

Test 1: Pipeline Ordering

Input: Project with lint errors AND test failures.
Expected: Pipeline halts at Stage 1 (Lint). Test stage never runs.
Verify: No test execution output in results. Duration reflects lint-only.

Test 2: Quality Gate Enforcement

Input: Test suite passes but coverage is 65%. Gate threshold is 70%.
Expected: Stage 4 (Test) status = FAIL. Verdict = FAIL.
Not: PASS with a warning. Coverage below fail threshold is a hard failure.

Test 3: Auto-Retry Loop

Input: Lint fails with 3 auto-fixable warnings (unused imports).
Expected: Validator routes fix request → Coder runs auto-fix →
Validator retries → PASS on second attempt. History shows 2 attempts.

Test 4: Regression Detection

Input: Baseline coverage 85%. New code drops coverage to 82%.
Regression threshold: 2%.
Expected: Stage 4 FAIL with message "Coverage regressed 3.0% (threshold: 2.0%)".

Test 5: Continuous Validation Hook

Input: Coder writes a .ts file with a type error via FileEditTool.
PostToolUse hook fires.
Expected: Hook detects type error within 5 seconds. Reports file:line.
Coder receives feedback before writing the next file.

Test 6: Structured Output

Input: Multiple failures across lint and test stages.
Expected: Final verdict JSON includes per-stage results, error counts,
coverage metrics, duration per stage, and categorized failures with
route_to fields. Orchestrator can parse without additional processing.

If all six tests pass, the Validator CLI enforces ordered pipeline execution, applies configurable quality gates, retries with routed fix requests, detects regressions, provides continuous feedback via hooks, and produces structured output that the Orchestrator consumes without additional parsing.
