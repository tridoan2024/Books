rcode User Playbook

You Just Opened the Terminal — Now What?

This guide walks you through exactly what to type, when to type it,
and how to get professional-grade results from rcode every single time.

Step 1: Launch rcode

Open your terminal, navigate to your project, and type:

cd ~/my-project
rcode

You'll see the rcode prompt. That's it — you're in.

  rcode v0.2.0 | model: sonnet-4 | effort: medium
  Project: my-project (git: main, clean)
  
  >

What just happened behind the scenes:

• rcode read your CLAUDE.md (if you have one)

• Loaded your saved memories from ~/.rcode/memory/

• Detected your git branch, status, and environment

• Connected to the AI model and is ready to work

Step 2: Your First Conversation

Just talk to it like a colleague. No special syntax needed.

"What does this project do?"

> explain this project to me — what does it do and how is it organized?

rcode reads your project structure, key files, and gives you a summary. This is great when jumping into unfamiliar code.

"Fix this bug"

> the login page shows a blank screen after submitting the form. 
  the error in the browser console is "TypeError: Cannot read property 'token' of undefined"
  the relevant code is in src/pages/Login.tsx

What rcode does:

1. Reads src/pages/Login.tsx

2. Finds the bug

3. Edits the file to fix it

4. Shows you the diff

5. Asks permission before saving (unless you changed permission mode)

"Write a feature"

> add a dark mode toggle to the settings page. 
  use the existing theme system in src/theme/. 
  save the preference to localStorage.

"Run something"

> run the tests and tell me if anything is broken

rcode runs your test command, reads the output, and tells you what passed/failed. If something fails, just say:

> fix the failing tests

Step 4: How to Get Better Answers

Bad vs Good Prompts

Asking for help:

❌  > help me with the database
✅  > the users table query in src/db/queries.py is slow — it takes 3 seconds 
     for 1000 records. optimize it.

Reporting a bug:

❌  > something is broken
✅  > running "npm test" gives this error:
     FAIL src/auth.test.js
     TypeError: jwt.verify is not a function
     fix it

Requesting a feature:

❌  > add caching
✅  > add Redis caching to the /api/products endpoint in src/api/products.py.
     cache for 5 minutes. invalidate when a product is updated.

The Magic Formula

Every great prompt follows this pattern:

WHAT is wrong (or what you want)
WHERE in the code (file path, line number)
HOW to verify (test command, expected behavior)

Example:

> the signup form in src/components/SignupForm.tsx doesn't validate email format.
  add validation that checks for @ and a domain.
  the test in tests/signup.test.tsx should pass after the fix.

Step 5: Real Workflows

Workflow A: "I'm starting my work day"

rcode -c                           # resume yesterday's session

> what were we working on? what's left to do?
> (rcode summarizes and shows remaining tasks)
> let's continue with the API endpoint

Workflow B: "I need to fix a bug"

> I'm getting this error when users upload a file larger than 10MB:
  
  Error: PayloadTooLargeError: request entity too large
  at /src/middleware/upload.js:23
  
  increase the limit to 50MB and add a user-friendly error message
  when they exceed it. run the upload tests after.

What rcode does:

1. Reads the middleware file

2. Increases the limit

3. Adds error handling

4. Runs the tests

5. Reports results

If tests fail:

> fix the failures

Workflow C: "I need to understand this code"

> walk me through how the payment system works.
  start at the checkout button and trace through to the Stripe API call.
  which files are involved?

Then drill deeper:

> now explain the error handling in src/payments/stripe.ts — 
  what happens when a payment fails?

Workflow D: "I need to add a feature"

Start with a plan:

> /plan
> I need to add email notifications when an order ships.
  we use SendGrid for email. the order model is in src/models/order.py.
  the shipping status is updated by a webhook at src/api/webhooks.py.

rcode creates a step-by-step plan. Review it:

> looks good, go ahead and implement it

After implementation:

> /diff                            # review what changed
> run all tests
> commit with message: Add shipping notification emails

Workflow E: "Review my code before I push"

> review all changes since the last commit.
  look for bugs, security issues, and missing error handling.
  rank each finding by severity.

Workflow F: "I'm learning a new codebase"

> I'm new here. give me a tour:
  1. what does this project do?
  2. what's the tech stack?
  3. how do I run it locally?
  4. where are the main entry points?
  5. what are the most important files?

Step 6: Teach rcode Your Preferences

The more rcode knows about you, the better its answers. Tell it things like:

> remember: I prefer TypeScript over JavaScript
> remember: our team uses conventional commits (feat:, fix:, docs:)
> remember: always run "pnpm test" not "npm test" in this project
> remember: I like detailed explanations with code examples
> remember: our API follows REST conventions with /api/v1/ prefix

These memories persist across sessions. Check them with /memory.

Set Up CLAUDE.md (One Time)

Create a file called CLAUDE.md in your project root:

# My Project

## How to run
- Dev server: npm run dev
- Tests: npm test
- Lint: npm run lint
- Build: npm run build

## Rules
- Use TypeScript strict mode
- All functions need type annotations
- Write tests for every new feature
- No console.log in production code — use the logger

## Architecture
src/api/ — Express route handlers
src/services/ — Business logic
src/models/ — Database models
src/utils/ — Shared helpers

rcode reads this EVERY session. It's like onboarding a new developer — but you only do it once.

Step 7: Control Cost and Speed

Quick Reference

Simple question/model haiku (cheapest)
Normal coding/model sonnet (default, balanced)
Hard bug or architecture/model opus + /effort high
Just exploring code/model haiku + /effort low
Need best possible answer/model opus + /effort max
Going too slow/fast (toggle speed mode)
Conversation getting long/compact (compress history)
Quality feels degraded/clear then restate your task

Check your spending

> /cost

Shows: tokens used, USD cost, cache savings.

Step 9: When Things Go Wrong

"rcode isn't understanding me"

Try being more specific. Instead of "fix the bug", paste the exact error message and file path.

"rcode is making too many changes"

Say: "Stop. Revert your last changes. Let's take a different approach."

Or use /rewind to go back to before the changes.

"The conversation feels confused"

> /compact                         # compress history

Or if it's really bad:

> /clear                           # start fresh
> (restate your task clearly)

"rcode keeps asking for permission"

Switch to a more trusting mode:

> /permissions

Choose acceptEdits (auto-approve file changes) or dontAsk (approve everything).

"I'm spending too much"

> /model haiku                     # switch to cheapest model
> /cost                            # check current spending

"rcode forgot what we were doing"

This happens when context gets full. Use /compact proactively, or resume the session with rcode -c.

Step 10: Advanced Moves (When You're Ready)

Use Agents for Big Tasks

> use an explore agent to find all files that import the old API client,
  then update them all to use the new client in src/api/client-v2.ts

Use Plan Mode for Complex Work

> /plan
> refactor the authentication system from session-based to JWT.
  don't break any existing tests. do it in small, safe steps.

Add MCP Servers for Extra Tools

> /mcp

Connect GitHub, Slack, databases, or custom tools.

Create Custom Skills

Put .ts files in ~/.rcode/skills/ for custom slash commands.

CLI Flags for Power Users

rcode -c                           # continue last session
rcode -p "explain src/main.py"     # non-interactive (for scripts)
rcode --model opus --effort max    # maximum intelligence
rcode --permission-mode dontAsk    # no permission prompts
